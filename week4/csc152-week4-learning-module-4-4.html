1. Roles vs. Features: Understanding the Difference

▼

Windows Server's modular architecture allows you to install only the functionality you need. This approach reduces the server's attack surface, minimizes resource consumption, and simplifies management. Understanding the distinction between roles and features is fundamental to effective Windows Server administration.

1.1 What Are Server Roles?

Server Role
A server role is a primary function or set of closely related functions that a server performs for clients or other servers on a network. A role represents the server's main purpose or job in the infrastructure. Examples include File Server, Web Server (IIS), DNS Server, DHCP Server, and Active Directory Domain Services.

Key Characteristics of Roles:

  • Primary Server Function: Roles define what the server does as its main job
  • Client-Facing Services: Roles typically provide services that clients or other servers consume
  • Include Management Tools: Installing a role automatically installs role-specific management tools
  • May Include Role Services: Roles can have sub-components called "role services" that provide specific functionality
  • Resource Intensive: Roles typically consume significant server resources
  • Best Practice - One Primary Role: Generally, dedicate each server to one primary role for optimal performance and security

1.2 What Are Features?

Feature
A feature is supporting software that adds functionality to one or more server roles, or enhances the functionality of the entire server regardless of which roles are installed. Features are typically not standalone server functions but rather components that support roles or provide system-wide capabilities. Examples include .NET Framework, Windows PowerShell, Telnet Client, and Windows Server Backup.

Key Characteristics of Features:

  • Supporting Functionality: Features enhance or support roles but aren't primary server functions
  • System-Wide or Role-Specific: Some features benefit the entire system; others support specific roles
  • Less Resource Intensive: Features generally consume fewer resources than roles
  • Multiple Features Common: Servers typically have multiple features installed
  • May Be Required: Some roles require specific features as dependencies
  • Can Be Standalone: Some features provide useful functionality independent of any role

1.3 Roles vs. Features: Side-by-Side Comparison

1.4 Role Services: Sub-Components of Roles

Many roles include optional role services - sub-components that provide specific functionality within the role. When installing a role, you can choose which role services to include.

Example: File and Storage Services Role

The File and Storage Services role includes several role services:

  • File Server: Basic file sharing functionality
  • BranchCache for Network Files: Caching for remote offices
  • Data Deduplication: Storage space optimization
  • DFS Namespaces: Distributed file system
  • DFS Replication: File replication between servers
  • File Server Resource Manager: Quota and file screening
  • File Server VSS Agent Service: Volume Shadow Copy integration
  • iSCSI Target Server: Block-level storage over network

You don't need all role services - install only what your environment requires.

1.5 The Principle of Least Functionality

A core security principle in server administration is least functionality - install only the roles, role services, and features necessary for the server's intended purpose.

Why This Matters:

  • Security: Fewer installed components = smaller attack surface
  • Performance: Unused components still consume resources
  • Stability: More components = more potential points of failure
  • Maintenance: Fewer components = less patching and updating
  • Troubleshooting: Simpler configurations are easier to diagnose
  • Compliance: Many security frameworks require least functionality

Best Practice: Role-Based Server Design

In enterprise environments, follow these guidelines:

  • Dedicated Servers: One primary role per physical/virtual server
  • Separation of Duties: Don't combine conflicting roles (e.g., Domain Controller + SQL Server)
  • Minimal Installation: Install only required role services and features
  • Regular Review: Audit installed components quarterly; remove unused items
  • Documentation: Document why each role/feature is installed

Exception: Small businesses with limited resources may combine compatible roles on a single server, but this should be a conscious decision with documented justification.

2. Common Server Roles Explained

▼

Windows Server includes numerous roles for different server functions. This section explores the most common roles you'll encounter in enterprise environments and throughout this course.

2.1 File and Storage Services

File and Storage Services

Installed by default. Provides file sharing, storage management, and related services.

Key Role Services:

  • File Server - Basic file sharing
  • DFS Namespaces - Unified namespace
  • DFS Replication - File replication
  • File Server Resource Manager - Quotas and screening

Common Use Cases: Shared network drives, department folders, user home directories, document management

File and Storage Services - Detailed Information ▼

What It Does:

Enables file sharing over the network using SMB (Server Message Block) protocol. Clients can access shared folders, collaborate on documents, and store files centrally.

When You Need It:

  • Users need to access shared files from multiple computers
  • Department or project-based file sharing
  • Centralized document storage and backup
  • User home directories on network

Important Role Services:

  • File Server: Core file sharing functionality
  • DFS Namespaces: Creates unified namespace across multiple file servers (e.g., \\company\shares instead of \\server1\share1)
  • DFS Replication: Replicates folders between servers for redundancy and load distribution
  • File Server Resource Manager (FSRM): Manages quotas, file screening (block certain file types), and storage reports
  • Data Deduplication: Reduces storage space by eliminating duplicate data blocks

Management Tools:

  • Server Manager → File and Storage Services
  • File Server Resource Manager console
  • DFS Management console
  • PowerShell: New-SmbShare, Get-SmbShare

Real-World Example:

A company with 100 employees creates department shares (HR, Finance, IT, Sales) on a file server. Each department has its own folder with appropriate permissions. Users map network drives to access their department's files. FSRM enforces 10 GB quotas per user and blocks .exe files from being stored in shared folders.

2.2 Print and Document Services

Print and Document Services

Centralizes print management, enabling network printing and print server functionality.

Key Role Services:

  • Print Server - Manages network printers
  • Internet Printing - Web-based printing
  • LPD Service - Unix/Linux print support

Common Use Cases: Centralized printer management, network printer sharing, print queue management, print accounting

2.3 Web Server (IIS)

Web Server (IIS)

Internet Information Services - hosts websites, web applications, and web services.

Key Role Services:

  • Web Server - Core HTTP functionality
  • FTP Server - File transfer protocol
  • Application Development - ASP.NET, CGI
  • Management Tools - IIS Manager

Common Use Cases: Company websites, intranet portals, web applications, REST APIs, SharePoint foundation

Web Server (IIS) - Detailed Information ▼

What It Does:

IIS (Internet Information Services) is Microsoft's web server platform. It serves web pages, hosts web applications, and provides web services over HTTP/HTTPS.

When You Need It:

  • Hosting company website or intranet
  • Running ASP.NET web applications
  • Providing REST APIs for applications
  • SharePoint Server (requires IIS)
  • Web-based management interfaces

Important Role Services:

  • Common HTTP Features: Static content, default document, directory browsing, HTTP errors
  • Application Development: ASP.NET, .NET Extensibility, CGI, ISAPI Extensions
  • Security: Request filtering, IP restrictions, authentication methods
  • Performance: Static content compression, dynamic content compression
  • Management Tools: IIS Management Console, scripting tools

Management Tools:

  • IIS Manager (GUI)
  • PowerShell: Get-Website, New-Website, Start-Website
  • Configuration files: applicationHost.config, web.config

Real-World Example:

A company hosts their internal employee portal on IIS. The portal provides access to HR forms, IT helpdesk, company news, and department resources. The site uses ASP.NET for dynamic content and SQL Server for the database backend. SSL certificates ensure secure HTTPS connections.

2.4 DNS Server

DNS Server

Domain Name System - resolves domain names to IP addresses and vice versa.

Core Functionality:

  • Name resolution (forward and reverse)
  • Zone management (primary, secondary)
  • DNS caching and forwarding
  • Active Directory integration

Common Use Cases: Internal name resolution, Active Directory requirement, internet domain hosting, load balancing

2.5 DHCP Server

DHCP Server

Dynamic Host Configuration Protocol - automatically assigns IP addresses and network configuration to clients.

Core Functionality:

  • Automatic IP address assignment
  • Subnet configuration management
  • Lease management
  • Options configuration (DNS, gateway)

Common Use Cases: Automatic client IP configuration, IP address management, reducing manual configuration, mobile device support

DHCP Server Deep Dive

Week 9 Coverage: DHCP Server will be covered in detail in Week 9. You'll configure scopes, reservations, and options. For now, understand that DHCP automates IP address assignment, eliminating manual configuration.

2.6 Active Directory Domain Services (AD DS)

Active Directory Domain Services

Centralized directory service for managing users, computers, groups, and organizational resources in a Windows domain.

Core Functionality:

  • User and computer authentication
  • Centralized user/group management
  • Group Policy application
  • Organizational Units (OUs)
  • Domain and forest management

Common Use Cases: Enterprise user management, single sign-on, centralized security policies, resource access control

Active Directory Deep Dive

Week 11 Coverage: Active Directory Domain Services is a major topic covered extensively in Week 11. You'll install domain controllers, create OUs, manage users and computers, and implement Group Policy. AD DS is one of the most important Windows Server roles.

2.7 Remote Desktop Services

Remote Desktop Services (RDS)

Enables multiple users to connect to remote desktops or applications hosted on a server.

Key Role Services:

  • Remote Desktop Session Host
  • Remote Desktop Connection Broker
  • Remote Desktop Gateway
  • Remote Desktop Web Access
  • Remote Desktop Licensing

Common Use Cases: Virtual desktop infrastructure (VDI), published applications, remote work access, thin client environments

2.8 Hyper-V

Hyper-V

Microsoft's virtualization platform - creates and manages virtual machines on Windows Server.

Core Functionality:

  • Virtual machine creation and management
  • Virtual networking
  • Virtual storage management
  • Live migration (moving VMs between hosts)
  • Snapshots and checkpoints

Common Use Cases: Server consolidation, test/dev environments, disaster recovery, cloud infrastructure, VDI

Hyper-V vs. Proxmox

In this course, we're using Proxmox VE as our virtualization platform. Hyper-V is Microsoft's equivalent:

  • Proxmox: Open-source, Linux-based (KVM), supports both Linux and Windows VMs
  • Hyper-V: Microsoft proprietary, Windows-based, integrates tightly with Windows environments
  • Both: Type 1 hypervisors, enterprise-grade, support live migration, snapshots, and clustering

The concepts you learn with Proxmox translate directly to Hyper-V and other virtualization platforms.

3. Common Features Explained

▼

Features provide supporting functionality that enhances roles or provides system-wide capabilities. Understanding common features helps you build properly configured servers.

3.1 .NET Framework

.NET Framework
A software development platform and runtime environment for building and running applications on Windows. Many server applications and roles require specific .NET Framework versions.

Key Points:

  • Multiple Versions: .NET Framework 3.5, 4.7, 4.8 can coexist
  • Required by Many Roles: Web Server (IIS) with ASP.NET requires .NET
  • Application Dependency: Many server applications require specific .NET versions
  • Includes ASP.NET: Web application framework

When You Need It:

  • Installing applications that require .NET (check application requirements)
  • Running ASP.NET web applications on IIS
  • Using PowerShell Desired State Configuration (DSC)
  • Many Microsoft server applications (Exchange, SharePoint, SQL Server Management Tools)

3.2 PowerShell and PowerShell ISE

Windows PowerShell
Command-line shell and scripting language built on .NET Framework. Essential for automation and remote management.
PowerShell ISE (Integrated Scripting Environment)
Graphical interface for writing, testing, and debugging PowerShell scripts.

Key Points:

  • Installed by Default: PowerShell 5.1 included in Windows Server
  • ISE Optional: PowerShell ISE is a separate feature
  • Essential for Automation: Most administrative tasks can be scripted
  • Remote Management: PowerShell Remoting enables remote administration

When You Need ISE:

  • Writing and testing PowerShell scripts
  • Learning PowerShell (syntax highlighting, IntelliSense)
  • Debugging complex scripts
  • Not needed for running scripts or commands (use PowerShell console)

PowerShell in This Course

Week 5 focuses on PowerShell fundamentals. You'll learn cmdlet syntax, scripting basics, and remote management. PowerShell is used throughout the course alongside GUI tools.

3.3 Windows Server Backup

Windows Server Backup
Built-in backup and recovery solution for Windows Server. Provides full server, system state, and file-level backups.

Key Points:

  • Not Installed by Default: Must be added as a feature
  • Block-Level Backup: Efficient backup using Volume Shadow Copy Service (VSS)
  • Multiple Backup Types: Full server, system state, custom (selected volumes)
  • Scheduled Backups: Can configure automated backup schedules
  • BMR Support: Bare Metal Recovery for disaster recovery

When You Need It:

  • Implementing backup strategy (Week 15 topic)
  • Disaster recovery planning
  • Regulatory compliance requirements
  • System state backups for Active Directory

Backup Deep Dive

Week 15 Coverage: Disaster Recovery & Backup Solutions covers Windows Server Backup in detail. You'll configure backup schedules, perform restorations, and develop backup strategies.

3.4 Telnet Client

Telnet Client
Command-line tool for testing network connectivity to specific TCP ports. Useful for troubleshooting network services.

Key Points:

  • Not Installed by Default: Must be added as a feature
  • Troubleshooting Tool: Tests if a specific port is open and accepting connections
  • Not for Secure Communication: Telnet is unencrypted; use SSH for actual remote access
  • Simple Syntax: telnet hostname port

Common Uses:

  • Test if web server port 80 is accessible: telnet webserver 80
  • Test if mail server port 25 is open: telnet mailserver 25
  • Verify firewall rules are allowing traffic
  • Troubleshoot network service connectivity

Security Note

Telnet Client is for testing connectivity only. Never use Telnet for actual remote administration - it transmits everything in plain text including passwords. Use SSH or PowerShell Remoting for secure remote access.

3.5 BitLocker Drive Encryption

BitLocker Drive Encryption
Full-disk encryption feature that protects data by encrypting entire volumes. Requires TPM (Trusted Platform Module) or USB key for maximum security.

Key Points:

  • Full Volume Encryption: Encrypts entire disk volumes
  • Transparent to Users: Automatic encryption/decryption
  • TPM Support: Uses Trusted Platform Module for key storage
  • Recovery Keys: Provides recovery mechanism if encryption key is lost
  • Performance Impact: Modern CPUs have minimal overhead

When You Need It:

  • Protecting sensitive data on server disks
  • Compliance requirements (HIPAA, PCI-DSS, etc.)
  • Servers in physically insecure locations
  • Protecting against disk theft

3.6 SNMP Service

SNMP (Simple Network Management Protocol) Service
Enables monitoring and management of network devices and servers through SNMP-based management systems.

Key Points:

  • Monitoring Protocol: Standard protocol for network management
  • Read-Only or Read-Write: Configure access levels
  • Community Strings: Simple authentication mechanism
  • MIB (Management Information Base): Defines available metrics

When You Need It:

  • Integration with network monitoring systems (Nagios, PRTG, SolarWinds)
  • Centralized monitoring of multiple servers
  • Automated alerting for server issues
  • Collecting performance metrics over time

3.7 Windows Defender Features

Windows Defender Antivirus
Built-in antivirus and anti-malware protection. Enabled by default on Windows Server.

Key Points:

  • Enabled by Default: Provides baseline protection
  • Real-Time Protection: Scans files as they're accessed
  • Automatic Updates: Definition updates via Windows Update
  • PowerShell Management: Configure via PowerShell cmdlets
  • Can Be Replaced: Third-party antivirus disables Windows Defender

3.8 Group Policy Management

Group Policy Management
Management console for creating and managing Group Policy Objects (GPOs) in Active Directory environments.

Key Points:

  • Installed with AD DS: Automatically included when installing Active Directory
  • Can Install Separately: For managing GPOs from workstations
  • Centralized Configuration: Deploy settings to many computers
  • Security Policies: Enforce password policies, firewall rules, etc.

Group Policy Coverage

Week 11: Group Policy is covered as part of Active Directory Domain Services. You'll create and link GPOs to enforce security policies and configuration settings.

4.3 Using PowerShell for Role and Feature Management

PowerShell provides powerful cmdlets for managing roles and features, especially useful for automation and remote management.

Essential PowerShell Cmdlets:

# View all available roles and features
Get-WindowsFeature

# View only installed roles and features
Get-WindowsFeature | Where-Object {$_.Installed -eq $True}

# View specific role or feature
Get-WindowsFeature -Name Web-Server

# Search for roles/features by name
Get-WindowsFeature -Name *DNS*

# Install a role or feature
Install-WindowsFeature -Name Web-Server

# Install with management tools
Install-WindowsFeature -Name Web-Server -IncludeManagementTools

# Install multiple items at once
Install-WindowsFeature -Name Web-Server, Web-Asp-Net45 -IncludeManagementTools

# Install and restart if needed
Install-WindowsFeature -Name DHCP -IncludeManagementTools -Restart

# Remove a role or feature
Uninstall-WindowsFeature -Name Telnet-Client

# Remove and delete installation files (saves disk space)
Uninstall-WindowsFeature -Name Web-Server -Remove

4.4 Understanding Dependencies and Prerequisites

Many roles require specific features or other roles as dependencies. Windows Server automatically handles most dependencies, but understanding them helps you plan installations.

Common Dependencies:

  • Web Server (IIS): May require .NET Framework depending on applications
  • Active Directory: Automatically installs DNS Server
  • DHCP Server: Requires authorization in Active Directory (if domain-joined)
  • Remote Desktop Services: Requires Remote Desktop Licensing
  • Many Roles: Require specific management features

How Dependencies Are Handled:

  • Server Manager Wizard: Prompts you to install dependencies, shows what will be added
  • PowerShell Default: Installs dependencies automatically
  • PowerShell -IncludeManagementTools: Adds management tools for the role
  • Manual Control: Can use -IncludeAllSubFeature to install all role services

4.5 Post-Installation Configuration

Installing a role is just the first step. Most roles require additional configuration before they're functional.

Common Post-Installation Tasks:

  • Role-Specific Configuration: Each role has unique configuration requirements
  • Firewall Rules: May need to configure Windows Firewall to allow traffic
  • Service Startup: Verify services are running
  • Security Configuration: Apply appropriate security settings
  • Testing: Verify role functionality
  • Documentation: Record configuration decisions

Important: Installation ≠ Configuration

Installing a role makes the software available but doesn't fully configure it. For example:

  • DHCP Server: Must create scopes and authorize server
  • DNS Server: Must create zones and records
  • Web Server: Must create websites and configure bindings
  • Active Directory: Must promote server to domain controller

Each role has specific configuration steps covered in their respective course weeks.

4.6 Removing Roles and Features Safely

Removing roles and features requires care to avoid breaking dependencies or disrupting services.

Safe Removal Process:

  1. Verify Impact: Check if other roles depend on what you're removing
  2. Backup Configuration: Export configuration if you might need it later
  3. Stop Services: Stop role-specific services first
  4. Remove Role: Use Server Manager or PowerShell to remove
  5. Verify Removal: Confirm role is no longer listed
  6. Clean Up: Remove associated data if appropriate
  7. Restart if Required: Some removals require restart

PowerShell Removal Options:

# Standard removal (keeps installation files)
Uninstall-WindowsFeature -Name Web-Server

# Remove installation files (saves ~1-2 GB disk space)
Uninstall-WindowsFeature -Name Web-Server -Remove

# Remove with restart
Uninstall-WindowsFeature -Name DHCP -Remove -Restart

# Remove and see what else would be removed
Uninstall-WindowsFeature -Name Web-Server -WhatIf

Caution: Removing Active Roles

Be extremely careful when removing roles from production servers:

  • Removing Active Directory can break domain authentication
  • Removing DHCP leaves clients without IP addresses
  • Removing DNS breaks name resolution
  • Always have a rollback plan and tested backups

Regularly audit what's installed on your servers as part of security best practices.

Methods to View Installed Items:

5. Best Practices and Design Principles

▼

Effective server role design follows established best practices that improve security, performance, and maintainability.

5.1 The One Primary Role Principle

In enterprise environments, best practice is to dedicate each server to one primary role.

Why One Role Per Server?

  • Performance: Roles don't compete for resources
  • Security: Compromise of one role doesn't affect others
  • Stability: Role-specific issues don't cascade
  • Maintenance: Can patch/restart one role without affecting others
  • Scalability: Can scale individual roles independently
  • Troubleshooting: Simpler to diagnose issues
  • Licensing: Some roles have specific licensing requirements

Common Single-Role Servers:

Typical Single-Role Server Deployments
Server Type Primary Role Why Dedicated
Domain Controller Active Directory Domain Services Security best practice, resource intensive, critical service
Database Server SQL Server (not a Windows role, but application) Resource intensive, performance critical, licensing
Exchange Server Exchange (application) Resource intensive, complex, requires dedicated resources
Hyper-V Host Hyper-V Hypervisor should be minimal, security isolation
Web Server IIS Internet-facing, security isolation, scalability

5.2 When Multi-Role Servers Are Acceptable

While single-role is ideal, multi-role servers are sometimes appropriate:

Acceptable Multi-Role Scenarios:

  • Small Business Servers: Limited budget and resources
    • Example: File Server + Print Server
    • Example: DNS + DHCP on same server
  • Branch Office Servers: Remote locations with minimal IT staff
    • Example: File Server + DNS + DHCP
    • Read-Only Domain Controller + File Server
  • Test/Development Environments: Non-production servers
    • Can combine multiple roles for testing
    • Reduced resource requirements
  • Compatible Lightweight Roles: Roles that complement each other
    • Example: DNS + DHCP (both network infrastructure)
    • Example: File Server + Print Server (both document services)

Roles That Should NEVER Be Combined:

  • Domain Controller + SQL Server: Security and performance concerns
  • Domain Controller + Exchange Server: Microsoft explicitly advises against
  • Hyper-V Host + Other Roles: Hypervisor should be minimal
  • Internet-Facing Web Server + Domain Controller: Extreme security risk

Multi-Role Decision Framework

Before combining roles, ask:

  1. Do both roles have similar resource requirements?
  2. Are both roles similar in criticality?
  3. Is there a security risk in combining them?
  4. Will one role's maintenance affect the other?
  5. Can the server handle both roles' resource needs?
  6. Is there a compelling business reason (cost, space)?

If any answer raises concerns, consider separate servers.

5.3 Minimal Installation Principle

Only install what you need. Every additional component increases attack surface and complexity.

Implementation:

  • Audit Before Installation: Do you really need this role/feature?
  • Role Services: Install only required role services, not everything
  • Features: Only install features required by roles or applications
  • Regular Review: Quarterly audit of installed components
  • Remove Unused: Uninstall roles/features no longer needed
  • Document Justification: Why is each component installed?

Example: Web Server Installation

Instead of installing all IIS role services:

  • Static Website: Only need basic HTTP features
  • ASP.NET Application: Add .NET Framework and ASP.NET
  • Don't Need FTP: Don't install FTP Server role service
  • Don't Need WebDAV: Don't install WebDAV Publishing

5.4 Documentation Standards

Proper documentation is essential for server role management.

What to Document:

  • Server Inventory:
    • Server name and purpose
    • Installed roles and features
    • Role services included
    • Installation date and administrator
  • Configuration Details:
    • Role-specific configuration settings
    • Dependencies and prerequisites
    • Firewall rules created
    • Service accounts used
  • Justification:
    • Why this role was installed
    • Business requirement it fulfills
    • Approval documentation
  • Maintenance Records:
    • Changes made over time
    • Patches applied
    • Issues encountered and resolutions

5.5 Change Management Process

Installing or removing roles should follow a formal change management process in production environments.

Recommended Process:

  1. Request: Document need for role installation/removal
  2. Impact Analysis: Assess impact on existing services
  3. Approval: Get management approval
  4. Planning: Plan installation steps and timing
  5. Testing: Test in non-production environment first
  6. Backup: Backup server before changes
  7. Implementation: Install/remove role during maintenance window
  8. Verification: Test functionality after installation
  9. Documentation: Update server documentation
  10. Review: Post-implementation review

5.6 Security Considerations

Role-Based Security Best Practices:

  • Least Privilege: Roles run with minimum necessary permissions
  • Service Accounts: Use dedicated service accounts, not Administrator
  • Firewall Rules: Only open required ports
  • Regular Updates: Keep roles patched and updated
  • Security Auditing: Enable auditing for role-specific events
  • Network Segmentation: Place servers in appropriate network zones
  • Encryption: Use SSL/TLS for internet-facing roles

Activity 5.1: Server Design Exercise

For each scenario, design an appropriate server deployment following best practices:

Scenario 1: Medium Enterprise (500 employees)

Requirements:

  • Centralized user authentication
  • File sharing for departments
  • Internal website for employee resources
  • Email server
  • Network infrastructure (DNS, DHCP)

Recommended Design:

  • Server 1: Domain Controller (Active Directory)
  • Server 2: Secondary Domain Controller (redundancy)
  • Server 3: File Server (File and Storage Services)
  • Server 4: Web Server (IIS for intranet)
  • Server 5: Email Server (Exchange - separate from DC)
  • Server 6: Infrastructure Server (DNS + DHCP - acceptable combination)

Rationale: Each critical role has dedicated server. DNS and DHCP combined as they're lightweight infrastructure roles. Domain Controllers are redundant for availability.

Scenario 2: Small Business (15 employees)

Requirements:

  • Basic file sharing
  • Printer sharing
  • Backup solution
  • Limited budget

Recommended Design:

  • Server 1: All-in-one server
    • File and Storage Services
    • Print and Document Services
    • Windows Server Backup feature

Rationale: Small business with limited resources. These roles are compatible and lightweight. File and Print services often go together.

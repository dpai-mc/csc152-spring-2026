1.2 Renaming Network Adapters

Default names like "Ethernet" or "Ethernet 2" aren't descriptive. Renaming adapters helps identify their purpose.

  3. Enter descriptive name (e.g., "LAN-Primary", "Management", "DMZ")

# Verify rename

Naming Convention Best Practice

Use descriptive names that indicate adapter purpose:

  • Management: Dedicated management network
  • DMZ: DMZ/internet-facing network
  • Backup: Dedicated backup network
  • iSCSI: Storage network

  • Confirm adapter name before disabling
  • Use -Confirm parameter for safety

2. Static vs. DHCP IP Addressing

â–¼

Understanding when to use static versus DHCP addressing is crucial for server administration. While DHCP is convenient for workstations, servers typically require static IP addresses.

2.1 DHCP (Dynamic Host Configuration Protocol)

DHCP (Dynamic Host Configuration Protocol)
A network protocol that automatically assigns IP addresses and network configuration to devices. A DHCP server maintains a pool of available IP addresses and "leases" them to clients for a specified period.

DHCP Advantages:

  • Automatic Configuration: No manual IP address entry
  • Centralized Management: Change network settings from DHCP server
  • Prevents Conflicts: DHCP server tracks assigned addresses
  • Efficient Use: IP addresses can be reused when devices disconnect
  • Easy for Mobile Devices: Automatically configure when joining network

DHCP Disadvantages for Servers:

  • Address Changes: IP address may change after lease expires
  • Dependency: Requires functioning DHCP server
  • Unpredictable: Can't guarantee specific IP address (without reservation)
  • Service Access: Clients need consistent IP to access server services

2.2 Static IP Addressing

Static IP Address
A manually configured IP address that doesn't change. The IP address, subnet mask, default gateway, and DNS servers are explicitly set and remain constant until manually changed.

Static IP Advantages for Servers:

  • Consistent Address: IP never changes unexpectedly
  • Predictable Access: Clients always know where to find server
  • DNS Stability: DNS records point to unchanging IP
  • No DHCP Dependency: Server works even if DHCP is down
  • Firewall Rules: Easier to configure based on static IPs
  • Documentation: Easier to document and track

Static IP Disadvantages:

  • Manual Configuration: Must configure each server manually
  • Potential Conflicts: Must track used addresses to avoid conflicts
  • Network Changes: Must reconfigure if network settings change
  • More Work: More administrative overhead than DHCP

2.3 When to Use Static vs. DHCP

Static vs. DHCP Decision Matrix
Device Type Recommended Rationale
Servers Static IP Provide services; need consistent addressing
Network Infrastructure Static IP Routers, switches, firewalls need fixed addresses
Printers Static IP or DHCP Reservation Users need consistent address to connect
Workstations DHCP Don't provide services; benefit from automatic config
Mobile Devices DHCP Move between networks; need automatic config
IoT Devices DHCP Reservation Automatic config but consistent address needed

DHCP Reservations: Best of Both Worlds

DHCP reservations provide automatic configuration with consistent IP addresses:

  • DHCP server reserves specific IP for device's MAC address
  • Device always gets same IP from DHCP
  • Centralized management (change settings on DHCP server)
  • Useful for printers, IP phones, and some servers

For this course: We use static IPs on servers for simplicity and learning. DHCP reservations are covered in Week 9.

2.4 Required Network Configuration Elements

Whether using static or DHCP, servers need these network configuration elements:

IP Address
Unique identifier for the server on the network. Must be within the network's IP range and not conflict with other devices. Example: 192.168.1.100
Subnet Mask
Defines which portion of the IP address represents the network and which represents the host. Determines which IP addresses are on the local network. Example: 255.255.255.0 (or /24 in CIDR notation)
Default Gateway
IP address of the router that provides access to other networks and the internet. Packets destined for other networks are sent to the gateway. Example: 192.168.1.1
DNS Server(s)
IP addresses of Domain Name System servers that translate domain names (like google.com) to IP addresses. Essential for internet access and internal name resolution. Example: 8.8.8.8 (Google DNS), 1.1.1.1 (Cloudflare)

How These Work Together:

  1. Local Communication: Server uses subnet mask to determine if destination is local
    • If local: Send directly to destination
    • If remote: Send to default gateway
  2. Name Resolution: When you use a domain name, server queries DNS server to get IP address
  3. Internet Access: Default gateway routes traffic to internet

3.2 PowerShell Configuration Method

Complete Static IP Configuration:

# Step 1: Identify adapter name
Get-NetAdapter

# Step 2: Remove existing IP configuration (if any)
Remove-NetIPAddress -InterfaceAlias "Ethernet" -Confirm:$false
Remove-NetRoute -InterfaceAlias "Ethernet" -Confirm:$false

# Step 3: Configure static IP address
New-NetIPAddress -InterfaceAlias "Ethernet" `
    -IPAddress 192.168.1.100 `
    -PrefixLength 24 `
    -DefaultGateway 192.168.1.1

# Step 4: Configure DNS servers
Set-DnsClientServerAddress -InterfaceAlias "Ethernet" `
    -ServerAddresses ("8.8.8.8","8.8.4.4")

# Step 5: Verify configuration
Get-NetIPConfiguration
ipconfig /all

Understanding PrefixLength

PowerShell uses prefix length (CIDR notation) instead of subnet mask:

  • /24 = 255.255.255.0
  • /16 = 255.255.0.0
  • /8 = 255.0.0.0

The number indicates how many bits are used for the network portion.

3.3 Planning Your Static IP Configuration

Information You Need Before Configuring:

  • Network Range: What IP range does your network use?
    • Common ranges: 192.168.1.0/24, 192.168.0.0/24, 10.0.0.0/8
    • Check your router or existing devices
  • Available IP Address: Choose an IP not in use
    • Check router's DHCP range (avoid that range)
    • Ping the IP first to verify it's not in use
    • Document your choice
  • Subnet Mask: Match your network's subnet mask
    • Most home/small networks: 255.255.255.0
    • Check other devices on your network
  • Default Gateway: Your router's IP address
    • Usually first or last IP in range
    • Common: 192.168.1.1, 192.168.0.1, 10.0.0.1
    • Check your router documentation
  • DNS Servers: Choose reliable DNS servers
    • Public: 8.8.8.8 (Google), 1.1.1.1 (Cloudflare)
    • Your ISP's DNS servers
    • Your router (if it provides DNS)

3.4 Avoiding IP Address Conflicts

IP Address Conflicts

An IP address conflict occurs when two devices on the same network have the same IP address. This causes network connectivity problems for both devices.

Preventing Conflicts:

  1. Check Router's DHCP Range:
    • Log into your router
    • Find DHCP settings
    • Note the DHCP range (e.g., 192.168.1.100-192.168.1.199)
    • Choose static IPs outside this range
  2. Ping Before Assigning: 
    # Test if IP is in use before assigning
ping 192.168.1.100

# If you get replies, IP is in use - choose different one
# If "Request timed out", IP is likely available
  3. Document All Static IPs:
    • Maintain a spreadsheet of static IP assignments
    • Include: IP, hostname, purpose, date assigned
    • Update when making changes
  4. Use Consistent Ranges:
    • Example: .1-.50 for infrastructure (routers, switches)
    • Example: .51-.99 for servers
    • Example: .100-.254 for DHCP pool

4. Windows Network Troubleshooting Tools

â–¼

Windows includes numerous command-line tools for testing and troubleshooting network connectivity. Many have Linux equivalents you learned in Week 3.

4.1 ping - Test Basic Connectivity

ping
Sends ICMP Echo Request packets to test network connectivity and measure round-trip time. Universal tool available on all operating systems.

Basic ping Usage:

# Ping by IP address
ping 192.168.1.1

# Ping by hostname
ping google.com

# Continuous ping (Ctrl+C to stop)
ping -t google.com

# Ping specific number of times
ping -n 10 google.com

# Ping with larger packet size
ping -l 1500 google.com

# Ping with timeout
ping -w 1000 google.com  # 1000ms timeout

Understanding ping Output:

Pinging google.com [142.250.80.46] with 32 bytes of data:
Reply from 142.250.80.46: bytes=32 time=15ms TTL=116
Reply from 142.250.80.46: bytes=32 time=14ms TTL=116
Reply from 142.250.80.46: bytes=32 time=16ms TTL=116
Reply from 142.250.80.46: bytes=32 time=15ms TTL=116

Ping statistics for 142.250.80.46:
    Packets: Sent = 4, Received = 4, Lost = 0 (0% loss),
Approximate round trip times in milli-seconds:
    Minimum = 14ms, Maximum = 16ms, Average = 15ms

What ping Tells You:

  • Reply received: Destination is reachable
  • Time: Network latency (lower is better)
    • <10ms: Excellent (local network)
    • 10-50ms: Good (regional internet)
    • 50-100ms: Acceptable (distant servers)
    • >100ms: High latency (may affect applications)
  • TTL (Time To Live): Number of router hops remaining
    • Starts at 64 or 128 (OS dependent)
    • Decrements at each router
    • Low TTL indicates many hops
  • Packet Loss: Percentage of packets that didn't return
    • 0%: Perfect connection
    • 1-5%: Acceptable for most applications
    • >5%: Network problems

4.2 Test-Connection - PowerShell ping

nslookup
Name Server Lookup - queries DNS servers to resolve domain names to IP addresses or perform reverse lookups. Essential for troubleshooting DNS issues.

nslookup Usage:

# Lookup using specific DNS server

> google.com
> set type=MX
> google.com
> exit

# Query specific record type
nslookup -type=MX google.com
nslookup -type=A google.com
nslookup -type=AAAA google.com  # IPv6

Understanding nslookup Output:

Server:  dns.google
Address:  8.8.8.8

Non-authoritative answer:
Name:    google.com
Addresses:  142.250.80.46
          2607:f8b0:4004:c07::71

Interpreting results:

  • Server: DNS server that answered the query
  • Non-authoritative answer: Result from DNS cache (not from authoritative nameserver)
  • Name: Domain name queried
  • Addresses: IP address(es) for the domain

4.6 tracert - Trace Route

tracert (traceroute)
Traces the path packets take from source to destination, showing each router (hop) along the way. Helps identify where network problems occur.

tracert Usage:

# Trace route to destination
tracert google.com

# Trace without resolving hostnames (faster)
tracert -d google.com

# Trace with maximum hops
tracert -h 20 google.com

Understanding tracert Output:

Tracing route to google.com [142.250.80.46]
over a maximum of 30 hops:

  1    <1 ms    <1 ms    <1 ms  192.168.1.1
  2    10 ms     9 ms    10 ms  10.0.0.1
  3    15 ms    14 ms    15 ms  isp-router.net [203.0.113.1]
  4    20 ms    19 ms    20 ms  142.250.80.46

Trace complete.

What it shows:

  • Each line is a router (hop) along the path
  • Three time measurements (three probes sent)
  • Helps identify where delays or failures occur
  • * * * indicates timeout (router not responding to trace)

4.7 pathping - Combined ping and tracert

Windows-specific tool that combines ping and tracert functionality:

# Run pathping (takes time - tests each hop multiple times)
pathping google.com

Advantages:

  • Shows route like tracert
  • Tests each hop with multiple packets
  • Calculates packet loss per hop
  • Identifies problematic routers in path

5.1 Windows Firewall Overview

Windows Firewall
A host-based firewall that filters network traffic to and from Windows computers. Provides protection by allowing or blocking traffic based on configurable rules.

Key Concepts:

  • Enabled by Default: Windows Firewall is on after installation
  • Stateful Firewall: Tracks connections and allows return traffic
  • Rule-Based: Traffic allowed or blocked based on rules
  • Profile-Based: Different rules for different network types
  • Inbound and Outbound: Separate rules for incoming and outgoing traffic

5.2 Firewall Profiles

Windows Firewall uses different profiles based on network type:

Windows Firewall Profiles
Profile When Used Default Security Level
Domain Computer is joined to Active Directory domain and connected to domain network Moderate (trusts domain environment)
Private Home or work network (user designated as private) Moderate (some services allowed)
Public Public networks like coffee shops, airports (default for unknown networks) High (most incoming traffic blocked)

Viewing Active Profile:

# View firewall profile status
Get-NetFirewallProfile

# View specific profile
Get-NetFirewallProfile -Name Domain

# View current active profile
Get-NetConnectionProfile

5.3 Checking Firewall Status

GUI Method:

  1. Server Manager â†’ Local Server â†’ Click "Windows Firewall"
  2. Or: Control Panel â†’ Windows Defender Firewall
  3. View status for each profile

PowerShell Method:

# Check if firewall is enabled for all profiles
Get-NetFirewallProfile | Select-Object Name, Enabled

# Get firewall state details
Get-NetFirewallProfile | Format-Table Name, Enabled, DefaultInboundAction, DefaultOutboundAction

# Check specific profile

Never Disable Windows Firewall Completely!

Common mistake: Disabling firewall to "fix" connectivity issues.

Instead:

  • Enable specific firewall rules for services you need
  • Create custom rules for applications
  • Keep firewall enabled for security
  • Troubleshoot by checking rules, not disabling firewall

  • Network Discovery: Finding other computers on network

  • Remote Desktop

Enabling/disabling groups is easier than managing individual rules.

6.1 netstat - Network Statistics

netstat

Understanding netstat Output:

Active Connections

  Proto  Local Address          Foreign Address        State           PID
  TCP    0.0.0.0:135            0.0.0.0:0              LISTENING       1044
  TCP    0.0.0.0:3389           0.0.0.0:0              LISTENING       1234
  TCP    192.168.1.100:3389     192.168.1.50:54321     ESTABLISHED     1234

Columns explained:

  • Proto: Protocol (TCP or UDP)
  • Local Address: Your server's IP:Port
  • Foreign Address: Remote computer's IP:Port
  • State: Connection state (LISTENING, ESTABLISHED, etc.)
  • PID: Process ID of application using connection

Common States:

  • LISTENING: Server waiting for connections on this port
  • ESTABLISHED: Active connection
  • TIME_WAIT: Connection closed, waiting for final packets
  • CLOSE_WAIT: Remote side closed, local side closing

Resolve-DnsName google.com -Server 8.8.8.8

# Query specific record type
Resolve-DnsName google.com -Type MX
Resolve-DnsName google.com -Type A
Resolve-DnsName google.com -Type AAAA  # IPv6

# Detailed output
Resolve-DnsName google.com -Type ALL

# Cache only (don't query server)
Resolve-DnsName google.com -CacheOnly

# Clear DNS cache
Clear-DnsClientCache

# View DNS cache
Get-DnsClientCache

Activity 6.1: Network Tools Comparison

Compare Windows and Linux networking tools by completing equivalent tasks:

Task 1: View IP Configuration

Windows: ipconfig /all

Linux: ip addr show or ifconfig

Task 2: Test Connectivity

Windows: Test-NetConnection google.com

Linux: ping google.com

Task 3: DNS Lookup

Windows: Resolve-DnsName google.com

Linux: dig google.com or nslookup google.com

Task 4: View Active Connections

Windows: netstat -ano

Linux: ss -tulpn or netstat -tulpn

Task 5: View Routing Table

Windows: Get-NetRoute or route print

Linux: ip route or route -n

1. Pre-Installation Planning

Proper planning before installation ensures a smooth deployment and optimal server performance. Unlike desktop installations, server deployments require careful consideration of resources, naming conventions, and network configuration.

1.1 Understanding Windows Server Editions

Windows Server Standard
Cost-effective edition suitable for physical or minimally virtualized environments. Includes rights for 2 virtual machines (VMs) per license. Ideal for small to medium businesses with limited virtualization needs.
Windows Server Datacenter
Premium edition designed for highly virtualized and software-defined datacenters. Provides unlimited virtualization rights on the licensed server. Includes advanced features like Storage Spaces Direct, Shielded VMs, and Storage Replica.
Windows Server Evaluation
Full-featured version available for 180-day trial period. Ideal for learning environments and testing. Can be converted to licensed version without reinstallation.

1.2 Installation Options: Desktop Experience vs. Server Core

Comparison of Windows Server Installation Options
Aspect Desktop Experience Server Core
User Interface Full GUI with Start Menu, Desktop, and graphical tools Command-line interface only (PowerShell and cmd)
Disk Space Required ~32 GB minimum (more with updates) ~10 GB (significantly smaller footprint)
Memory Usage Higher (GUI components consume RAM) Lower (minimal overhead)
Attack Surface Larger (more components = more vulnerabilities) Smaller (fewer installed components)
Management Method Local GUI, Remote Desktop, PowerShell Remote PowerShell, Windows Admin Center
Learning Curve Easier for beginners (familiar interface) Steeper (requires command-line proficiency)
Best Use Cases Learning environments, servers requiring GUI applications, Remote Desktop Session Hosts Production environments, cloud deployments, containers, security-focused deployments

For This Course

We will use Desktop Experience for learning purposes. The GUI provides visual feedback and makes initial learning more accessible. However, understanding Server Core is important for real-world scenarios, and we'll explore it conceptually.

Recommended VM Resources for Windows Server

1.4 Network Configuration Planning

Unlike home computers that typically use DHCP, servers require static IP addresses for reliability and consistency.

Static IP Address
A manually configured IP address that does not change. Essential for servers because other systems need a consistent address to access server services. Example: If your DNS server's IP address changed constantly, clients couldn't reliably find it.

2. Creating the Virtual Machine in Proxmox

Before installing Windows Server, we must create a properly configured virtual machine. This process mirrors creating a physical server but with the flexibility of virtualization.

2.1 VM Creation Process

  1. Access Proxmox Web Interface

    Log in to your Proxmox VE web interface using your browser. Navigate to your node in the left sidebar.

  2. Create New VM

    Click the "Create VM" button in the upper right corner. This launches the VM creation wizard.

  3. General Settings (Node Selection)

    Select your Proxmox node and assign a VM ID (Proxmox assigns automatically). Enter your VM name: CSC152-WS-YourName

    Naming Convention: Consistent naming helps identify VMs quickly. Include course, OS type, and your name.

  4. OS Selection

    Select the Windows Server ISO image from your Proxmox storage or upload it if not already available. Set the OS type to "Microsoft Windows" and version to appropriate Windows Server version.

  5. System Configuration

    Configure these critical settings:

    • Graphic card: Default (VGA)
    • Machine: q35 (recommended for modern Windows)
    • BIOS: OVMF (UEFI) - modern firmware interface
    • Add TPM: Optional, can enable for Windows 11-style security
    • SCSI Controller: VirtIO SCSI (best performance)
  6. Disk Configuration

    Configure virtual disk:

    • Storage: Select available storage pool
    • Disk size: 60-80 GB minimum
    • Cache: Write back (better performance)
    • Discard: Enable (allows thin provisioning)
  7. CPU Configuration

    Allocate CPU resources:

    • Sockets: 1
    • Cores: 2-4 (based on your planning)
    • Type: host (best performance, passes through host CPU features)
  8. Memory Configuration

    Allocate RAM:

    • Memory (MiB): 4096-8192 (4-8 GB)
    • Minimum memory: Can set lower limit for ballooning
  9. Network Configuration

    Configure network interface:

    • Bridge: vmbr0 (or your network bridge)
    • Model: VirtIO (paravirtualized - best performance)
    • Firewall: Optional, can enable Proxmox firewall
  10. Confirm and Create

    Review all settings in the confirmation screen. Check "Start after created" if you want to begin installation immediately. Click "Finish" to create the VM.

Important: VirtIO Drivers

Windows Server does not include VirtIO drivers by default. You'll need to load these drivers during installation for optimal disk and network performance. Your instructor will provide the VirtIO driver ISO, or download from the Proxmox/KVM community.

2.2 Understanding VM Hardware Configuration

Why These Hardware Settings Matter ▼

UEFI vs. Legacy BIOS

UEFI (Unified Extensible Firmware Interface) is the modern replacement for traditional BIOS. Benefits include:

  • Support for disks larger than 2 TB
  • Faster boot times
  • Better security features (Secure Boot)
  • Required for modern Windows features

VirtIO Paravirtualization

Traditional virtualization emulates physical hardware, which adds overhead. Paravirtualization allows the guest OS to communicate directly with the hypervisor for better performance:

  • VirtIO SCSI: Optimized disk I/O performance
  • VirtIO Network: Reduced network latency and higher throughput
  • Requires special drivers (not included in Windows by default)

CPU Type: Host

Setting CPU type to "host" passes through your physical CPU's features to the VM, enabling:

  • Hardware virtualization extensions (VT-x/AMD-V)
  • Better performance for nested virtualization
  • Access to modern CPU instructions
Storage Considerations ▼

Thin vs. Thick Provisioning

Thin provisioning (enabled with "Discard") means the virtual disk only uses actual space needed:

  • 60 GB virtual disk might only use 20 GB initially
  • Grows as data is added
  • More efficient use of host storage
  • Requires "Discard" support and TRIM commands

Write-Back Cache

Cache mode affects performance and data safety:

  • Write-back: Faster (acknowledges writes before physical write completes)
  • Write-through: Safer but slower (waits for physical write)
  • For lab environments, write-back is acceptable
  • Production environments may prefer write-through or no cache
Memory Ballooning ▼

Memory ballooning is a technique that allows dynamic memory allocation:

  • Set maximum memory (e.g., 8 GB)
  • Set minimum memory (e.g., 4 GB)
  • Hypervisor can reclaim unused memory from VM
  • Useful when running multiple VMs with varying loads
  • Requires balloon driver in guest OS

For simplicity in this course, we'll use fixed memory allocation initially.

3. Windows Server Installation Process

With the VM created, we can now install Windows Server. The installation process is more straightforward than Linux but includes important decision points.

3.1 Starting the Installation

  1. Start the VM

    In Proxmox, select your VM and click "Start". Open the console (either noVNC web console or use Proxmox VE Console Viewer).

  2. Boot from ISO

    The VM should automatically boot from the Windows Server ISO. You'll see the Windows logo and installation will begin loading.

    If the VM doesn't boot from ISO, check the boot order in VM Options or ensure the ISO is properly attached.

  3. Language and Region Selection

    Select your preferences:

    • Language to install: English (United States) or your preference
    • Time and currency format: Match your location
    • Keyboard or input method: US or your keyboard layout

    Click "Next" to continue.

  4. Install Now

    Click the "Install now" button to begin the installation process.

  5. Product Key (Optional)

    For evaluation versions, you can skip entering a product key by clicking "I don't have a product key". This gives you a 180-day evaluation period.

  6. Select Operating System Edition

    Choose the edition to install. You'll see options like:

    • Windows Server 2022 Standard
    • Windows Server 2022 Standard (Desktop Experience)
    • Windows Server 2022 Datacenter
    • Windows Server 2022 Datacenter (Desktop Experience)

    Select the "Desktop Experience" version of either Standard or Datacenter.

    Important: Once you select an edition, you cannot change it without reinstalling. Desktop Experience includes the GUI; non-Desktop Experience is Server Core (command-line only).

  7. Accept License Terms

    Read and accept the Microsoft Software License Terms by checking the box and clicking "Next".

  8. Installation Type

    Select "Custom: Install Windows only (advanced)" since this is a new installation, not an upgrade.

  9. Disk Partitioning and VirtIO Driver Loading

    This is a critical step where you may need to load VirtIO drivers:

    Loading VirtIO Drivers

    If you configured VirtIO SCSI, Windows won't see your disk initially. You must load drivers:

    1. Click "Load driver"
    2. Browse to the VirtIO driver ISO (should be attached as second CD-ROM)
    3. Navigate to the appropriate folder (e.g., vioscsi\2k22\amd64 for Windows Server 2022)
    4. Select the driver and click "Next"
    5. Repeat for network driver if needed (viostor folder)

    After loading drivers, your disk should appear in the partition list.

    Select the unallocated space and click "Next". Windows will automatically create necessary partitions (System, MSR, Primary).

  10. Installation Progress

    Windows will now copy files, install features, and configure the system. This process takes 15-30 minutes depending on your hardware. The VM will restart several times automatically.

    Tip: This is a good time to take a break or work on documentation. Do not interrupt the installation process.

3.2 Initial Configuration After Installation

  1. Set Administrator Password

    After installation completes and the system restarts, you'll be prompted to set the Administrator password.

    Password Requirements

    Windows Server enforces strong password policies by default:

    • Minimum 8 characters (recommended: 12+ characters)
    • Must contain three of four: uppercase, lowercase, numbers, special characters
    • Cannot contain username or parts of full name
    • Example strong password: CSC152!Server#2026

    Important: Document this password securely! You cannot access the server without it.

  2. First Login

    Press Ctrl+Alt+Delete (in Proxmox console, use the button or Send Key menu) to reach the login screen. Enter the Administrator password you just created.

  3. Server Manager Launches Automatically

    Upon first login, Server Manager opens automatically. This is the central administration interface for Windows Server. We'll explore it in detail in Module 4.3.

4. Essential Post-Installation Configuration

A fresh Windows Server installation requires several critical configuration steps before it's ready for use. These tasks establish the server's identity, network connectivity, and security baseline.

4.1 Changing the Computer Name

The default computer name (like "WIN-RANDOMSTRING") is not descriptive. Changing it to follow your naming convention is essential.

Method 1: Using Server Manager (GUI)

  1. Open Server Manager

    Server Manager should already be open. If not, click the Server Manager icon on the taskbar or search for it in the Start menu.

  2. Navigate to Local Server

    In the left sidebar, click "Local Server". This displays properties and configuration options for the local server.

  3. Click Computer Name

    In the Properties section, find "Computer name" and click the current name (it will be a hyperlink).

  4. Open System Properties

    This opens the System Properties dialog. Click the "Change..." button.

  5. Enter New Name

    In the "Computer name" field, enter your chosen name: CSC152-WS-YourName

    Computer names must follow these rules:

    • Maximum 15 characters for NetBIOS compatibility
    • Letters, numbers, and hyphens only (no spaces or special characters)
    • Cannot start or end with a hyphen
  6. Restart Required

    Click "OK" through the dialogs. You'll be prompted to restart. Click "Restart Now" or "Restart Later" based on your preference.

Method 2: Using PowerShell (Recommended for Automation)

# Open PowerShell as Administrator
# Click Start, type "PowerShell", right-click, "Run as Administrator"

# View current computer name
$env:COMPUTERNAME

# Change computer name (replace with your name)
Rename-Computer -NewName "CSC152-WS-YourName" -Restart

# The -Restart parameter automatically restarts the server
# If you omit -Restart, you must restart manually for the change to take effect

PowerShell Advantage

PowerShell commands can be:

  • Saved in scripts for repeatable deployments
  • Executed remotely on multiple servers
  • Included in automation workflows
  • Documented easily in text format

As you progress in server administration, PowerShell becomes increasingly valuable.

4.2 Configuring Static IP Address

Servers require static IP addresses for consistent accessibility. Unlike DHCP (which assigns addresses dynamically), a static configuration ensures the server always has the same IP address.

Understanding Network Configuration Components

IP Address
Unique identifier for the server on the network. Example: 192.168.1.100. Must be within your network's range and not conflict with other devices.
Subnet Mask
Defines the network portion vs. host portion of the IP address. Common example: 255.255.255.0 (or /24 in CIDR notation). Determines which IP addresses are on the local network.
Default Gateway
Router IP address that provides access to other networks and the internet. Example: 192.168.1.1. Without a correct gateway, the server cannot reach external networks.
DNS Servers
Domain Name System servers that translate names (like google.com) to IP addresses. Example: 8.8.8.8 (Google), 1.1.1.1 (Cloudflare). Essential for internet access and name resolution.

Method 1: Using Server Manager (GUI)

  1. Open Server Manager

    Navigate to Server Manager → Local Server (if not already there).

  2. Click Ethernet Link

    In the Properties section, find "Ethernet" (or your network adapter name) and click it. This opens Network Connections.

  3. Open Adapter Properties

    Right-click your network adapter and select "Properties".

  4. Configure IPv4

    In the list, select "Internet Protocol Version 4 (TCP/IPv4)" and click "Properties".

  5. Enter Static Configuration

    Select "Use the following IP address" and enter:

    • IP address: Your planned static IP (e.g., 192.168.1.100)
    • Subnet mask: Your network's subnet mask (e.g., 255.255.255.0)
    • Default gateway: Your router's IP (e.g., 192.168.1.1)

    Select "Use the following DNS server addresses" and enter:

    • Preferred DNS server: Primary DNS (e.g., 8.8.8.8 or your network's DNS)
    • Alternate DNS server: Secondary DNS (e.g., 8.8.4.4 or 1.1.1.1)
  6. Validate and Apply

    Click "OK" to apply settings. Windows will briefly disconnect and reconnect the network adapter.

  7. Verify Configuration

    Open Command Prompt or PowerShell and run:

    ipconfig /all

    Verify your IP address, subnet mask, gateway, and DNS servers are correctly configured.

Method 2: Using PowerShell

# View current network configuration
Get-NetIPConfiguration

# View network adapter name
Get-NetAdapter

# Set static IP address (replace values with your configuration)
# Replace "Ethernet" with your adapter name if different
New-NetIPAddress -InterfaceAlias "Ethernet" `
    -IPAddress 192.168.1.100 `
    -PrefixLength 24 `
    -DefaultGateway 192.168.1.1

# Set DNS servers
Set-DnsClientServerAddress -InterfaceAlias "Ethernet" `
    -ServerAddresses ("8.8.8.8","8.8.4.4")

# Verify configuration
Get-NetIPConfiguration
Test-NetConnection -ComputerName google.com

Common Pitfall: IP Conflicts

If another device on your network already uses the IP address you choose, you'll get an "IP address conflict" error. To avoid this:

  • Check your router's DHCP range and choose an IP outside that range
  • Ping the IP address before assigning it: ping 192.168.1.100
  • Document all static IP assignments in your lab environment
  • Consider using higher IP addresses (e.g., .100-.200) for servers

4.3 Configuring Windows Update

Keeping Windows Server updated is critical for security and stability. Unlike consumer Windows, server updates require careful management to avoid disrupting services.

Understanding Windows Update in Server Environments

Windows Server updates include:

  • Security updates: Fix vulnerabilities (highest priority)
  • Critical updates: Address serious bugs and stability issues
  • Feature updates: Add new capabilities (less frequent)
  • Quality updates: Cumulative monthly updates
  • Driver updates: Hardware driver updates

Configuring Windows Update

  1. Open Windows Update Settings

    Click Start → Settings → Update & Security → Windows Update

    Or use PowerShell: Start-Process "ms-settings:windowsupdate"

  2. Check for Updates

    Click "Check for updates". Windows will download and install available updates. This may take significant time on a fresh installation (potentially hours).

    First Update Session: The initial update after installation often includes hundreds of updates accumulated since the ISO was created. Be patient and allow sufficient time.

  3. Configure Update Settings

    Click "Advanced options" to configure:

    • Receive updates for other Microsoft products: Enable (includes updates for Office, etc.)
    • Active hours: Set times when you don't want automatic restarts
    • Restart options: Configure notification preferences
  4. Restart if Required

    Many updates require a restart to complete installation. Plan for multiple restarts during initial update process.

PowerShell Update Management

# Install PSWindowsUpdate module (community module for update management)
Install-Module -Name PSWindowsUpdate -Force

# Check for available updates
Get-WindowsUpdate

# Install all available updates
Install-WindowsUpdate -AcceptAll -AutoReboot

# View update history
Get-WUHistory

# Check last update installation time
Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

Production Environment Consideration

In production environments, automatic updates are often disabled in favor of controlled update windows. Enterprise environments use:

  • WSUS (Windows Server Update Services): Centralized update management
  • SCCM/MECM: Microsoft Endpoint Configuration Manager for enterprise-wide updates
  • Maintenance windows: Scheduled times for updates and restarts
  • Testing environments: Test updates before production deployment

For this course, automatic updates are acceptable in our lab environment.

4.4 Time Zone and Regional Settings

Correct time configuration is essential for:

  • Accurate log timestamps
  • Kerberos authentication (requires time synchronization)
  • Scheduled tasks and automation
  • Certificate validity

Setting Time Zone

  1. Using GUI

    Server Manager → Local Server → Click "Time zone" → Select your time zone → Click "OK"

  2. Using PowerShell 
    # View current time zone
Get-TimeZone

# List available time zones

# Set time zone (example: Pacific Standard Time)
Set-TimeZone -Name "Pacific Standard Time"

# Verify

Configuring Time Synchronization

Windows Server uses the Windows Time service (W32Time) to synchronize with time servers:

# Check time service status
Get-Service W32Time

# Start time service if not running
Start-Service W32Time

# Configure NTP server (example: time.nist.gov)
w32tm /config /manualpeerlist:"time.nist.gov" /syncfromflags:manual /reliable:YES /update

# Force immediate synchronization
w32tm /resync

# Check time synchronization status
w32tm /query /status

4.5 Disabling IE Enhanced Security Configuration (ESC)

Internet Explorer Enhanced Security Configuration (IE ESC) is enabled by default on Windows Server to reduce attack surface. However, it significantly restricts web browsing, which can be inconvenient in lab environments.

Security Consideration

Production Servers: Leave IE ESC enabled. Servers shouldn't be used for general web browsing.

Lab Environments: Disabling IE ESC is acceptable for convenience when downloading tools, accessing documentation, etc.

Disabling IE ESC

  1. Open Server Manager

    Navigate to Server Manager → Local Server

  2. Click IE Enhanced Security Configuration

    Find "IE Enhanced Security Configuration" in Properties and click "On"

  3. Disable for Administrators

    In the dialog, set "Administrators" to "Off". You can leave "Users" set to "On" for better security.

  4. Click OK

    Changes take effect immediately; no restart required.

PowerShell Method

# Disable IE ESC for Administrators
$AdminKey = "HKLM:\SOFTWARE\Microsoft\Active Setup\Installed Components\{A509B1A7-37EF-4b3f-8CFC-4F3A74704073}"
Set-ItemProperty -Path $AdminKey -Name "IsInstalled" -Value 0

# Disable IE ESC for Users
$UserKey = "HKLM:\SOFTWARE\Microsoft\Active Setup\Installed Components\{A509B1A8-37EF-4b3f-8CFC-4F3A74704073}"
Set-ItemProperty -Path $UserKey -Name "IsInstalled" -Value 0

# Restart Explorer to apply changes
Stop-Process -Name Explorer -Force

Write-Host "IE Enhanced Security Configuration has been disabled." -ForegroundColor Green

6. Troubleshooting Common Issues

Even with careful planning, issues can arise during installation and configuration. This section covers common problems and their solutions.

Issue: VM Won't Boot from ISO ▼

Symptoms:

  • VM starts but shows "No bootable device"
  • VM attempts to PXE boot
  • VM shows black screen

Solutions:

  1. Verify ISO is attached:
    • In Proxmox, select VM → Hardware → CD/DVD Drive
    • Ensure Windows Server ISO is selected
    • Check "Use CD/DVD disc image file (iso)"
  2. Check boot order:
    • VM → Options → Boot Order
    • Ensure CD-ROM is before Hard Disk
    • Enable CD-ROM in boot order
  3. Verify BIOS settings:
    • For UEFI boot, ensure OVMF (UEFI) is selected in System settings
    • For legacy boot, use SeaBIOS
  4. Try restarting the VM after making changes
Issue: No Disk Visible During Installation ▼

Symptoms:

  • Installation shows "No drives were found"
  • Partition screen shows no disks

Solutions:

  1. Load VirtIO SCSI driver:
    • Click "Load driver" button
    • Browse to VirtIO driver ISO
    • Navigate to vioscsi → 2k22 (or 2k19) → amd64
    • Select Red Hat VirtIO SCSI controller
    • Click Next to load driver
  2. Alternative - Change disk controller:
    • Power off VM
    • Hardware → Hard Disk → Edit
    • Change from VirtIO SCSI to SATA
    • Restart installation
    • Note: SATA has lower performance than VirtIO
Issue: Installation is Extremely Slow ▼

Symptoms:

  • Installation takes several hours
  • Progress bar barely moves
  • System appears frozen but is actually working

Solutions:

  1. Verify sufficient resources:
    • Check Proxmox host isn't overcommitted
    • Ensure VM has at least 4 GB RAM
    • Verify host storage isn't full
  2. Check disk performance:
    • VirtIO SCSI should be faster than IDE/SATA
    • Ensure write-back cache is enabled
    • Check host disk I/O isn't saturated
  3. Be patient:
    • First-time Windows Server installation can take 30-60 minutes
    • Don't interrupt the process
    • Multiple automatic restarts are normal
Issue: Cannot Connect to Network After Installation ▼

Symptoms:

  • No network connectivity
  • Network adapter shows "No internet"
  • Cannot ping gateway or external hosts

Solutions:

  1. Check network adapter driver: 
    # In PowerShell
Get-NetAdapter

# If adapter shows "Disabled" or driver issue:
# Install VirtIO network driver from VirtIO ISO
# Device Manager → Network adapters → Update driver
  2. Verify VM network settings:
    • Proxmox: VM → Hardware → Network Device
    • Ensure connected to correct bridge (vmbr0)
    • Verify "Link State" is up
  3. Check IP configuration: 
    ipconfig /all

# Verify:
# - IP address is in correct range
# - Subnet mask matches network
# - Default gateway is correct
# - DNS servers are configured
  4. Test layer by layer: 
    # Test 1: Can you ping yourself?
ping 127.0.0.1  # Loopback - tests TCP/IP stack

# Test 2: Can you ping your own IP?
ping   # Tests network adapter

# Test 3: Can you ping gateway?
ping   # Tests local network

# Test 4: Can you ping external IP?
ping 8.8.8.8  # Tests internet routing

# Test 5: Can you resolve names?
nslookup google.com  # Tests DNS
  5. Check Windows Firewall: 
    # View firewall status
Get-NetFirewallProfile

# Temporarily disable to test (re-enable after!)
Set-NetFirewallProfile -Profile Domain,Public,Private -Enabled False

# Test connectivity

# Re-enable firewall
Set-NetFirewallProfile -Profile Domain,Public,Private -Enabled True
Issue: Password Doesn't Meet Complexity Requirements ▼

Symptoms:

  • "Password does not meet complexity requirements"
  • Cannot set Administrator password

Solutions:

  1. Understand requirements:
    • Minimum 8 characters (recommend 12+)
    • Must contain THREE of four character types:
      • Uppercase letters (A-Z)
      • Lowercase letters (a-z)
      • Numbers (0-9)
      • Special characters (!@#$%^&*)
    • Cannot contain username or parts of full name
  2. Example strong passwords:
    • CSC152!Server#2026
    • Winter2026@Admin
    • MyServer!Pass123
  3. Password manager recommended:
    • Use password manager to generate and store
    • Document in secure location
    • Never share passwords via unsecure means
Issue: Windows Update Fails or Hangs ▼

Symptoms:

  • Updates download but fail to install
  • Update process hangs at certain percentage
  • "Windows Update error" messages

Solutions:

  1. Run Windows Update Troubleshooter:
    • Settings → Update & Security → Troubleshoot
    • Select "Windows Update" → Run troubleshooter
  2. Reset Windows Update components: 
    # In elevated PowerShell
Stop-Service wuauserv, cryptSvc, bits, msiserver

Remove-Item C:\Windows\SoftwareDistribution -Recurse -Force
Remove-Item C:\Windows\System32\catroot2 -Recurse -Force

Start-Service wuauserv, cryptSvc, bits, msiserver

# Try Windows Update again
  3. Check disk space: 
    Get-Volume

# Ensure C: drive has at least 10-15 GB free
# Clean up if needed:
cleanmgr /d C:
  4. Install updates manually:
    • Visit Microsoft Update Catalog
    • Download specific updates
    • Install manually
  5. Be patient:
    • Initial updates can take several hours
    • Multiple restarts are normal
    • Don't force power off during updates
Issue: Cannot Change Computer Name ▼

Symptoms:

  • Computer name change fails
  • Error message when attempting rename
  • Name doesn't change after restart

Solutions:

  1. Verify name meets requirements:
    • Maximum 15 characters for NetBIOS
    • Letters, numbers, hyphens only
    • Cannot start/end with hyphen
    • No spaces or special characters
  2. Use PowerShell with elevated privileges: 
    # Right-click PowerShell → Run as Administrator
Rename-Computer -NewName "CSC152-WS-YourName" -Force -Restart
  3. Check for domain membership:
    • If server is domain-joined, use different process
    • For workgroup servers, standard rename works
  4. Restart required:
    • Computer name change always requires restart
    • Verify new name after reboot: $env:COMPUTERNAME

General Troubleshooting Approach

When encountering any issue:

  1. Read error messages carefully - they often tell you exactly what's wrong
  2. Check Event Viewer - Windows logs detailed error information
  3. Search for specific error codes - Microsoft documentation and community forums
  4. Verify one change at a time - don't make multiple changes simultaneously
  5. Document what you tried - helps avoid repeating failed solutions
  6. Ask for help - use discussion forums, office hours, or classmate collaboration

1. Understanding PowerShell

▼

PowerShell is Microsoft's powerful command-line shell and scripting language, designed specifically for system administration. It represents a fundamental shift in how Windows systems are managed and automated.

1.1 What is PowerShell?

PowerShell
A task automation and configuration management framework consisting of a command-line shell and associated scripting language. Built on .NET Framework, PowerShell provides access to COM (Component Object Model), WMI (Windows Management Instrumentation), and .NET objects, enabling comprehensive system administration and automation.

Key Characteristics:

  • Object-Oriented: Works with .NET objects, not just text
    • Unlike traditional shells that output text, PowerShell outputs objects
    • Objects have properties and methods you can access
    • Makes data manipulation more powerful and intuitive
  • Consistent Syntax: Verb-Noun naming convention
    • Example: Get-Service, Start-Service, Stop-Service
    • Predictable and discoverable
    • Easy to guess cmdlet names
  • Built on .NET: Access to entire .NET Framework
    • Rich library of functionality
    • Can create complex applications
    • Integration with Windows APIs
  • Extensible: Modules and snap-ins add functionality
    • Microsoft and third parties create modules
    • Import modules for specific tasks
    • Create your own modules
  • Cross-Platform: PowerShell Core runs on Windows, Linux, macOS
    • PowerShell 5.1: Windows-only (Windows PowerShell)
    • PowerShell 7+: Cross-platform (PowerShell Core)
    • This course uses Windows PowerShell 5.1 (included in Windows Server)

1.2 PowerShell vs. Command Prompt (cmd)

PowerShell vs. Command Prompt Comparison
Feature PowerShell Command Prompt (cmd)
Age Modern (2006+) Legacy (1980s DOS heritage)
Output Type Objects (.NET objects) Text only
Scripting Language Full-featured, object-oriented Batch files (limited)
Command Syntax Verb-Noun (Get-Service) Abbreviations (dir, copy, del)
Help System Comprehensive (Get-Help) Limited (command /?)
Pipeline Object pipeline (pass objects) Text pipeline (pass text)
Aliases Supports cmd commands as aliases N/A (native commands)
Tab Completion Intelligent (parameters, paths, cmdlets) Basic (file/folder names only)
Remote Execution PowerShell Remoting (built-in) Limited (psexec third-party tool)
Integration Deep Windows integration, .NET, WMI, COM Basic OS commands
Recommended For All modern Windows administration Legacy scripts, simple tasks

PowerShell is the Future

Microsoft has clearly indicated PowerShell as the future of Windows administration:

  • All new Windows features have PowerShell cmdlets
  • Some features are PowerShell-only (no GUI)
  • Microsoft certifications emphasize PowerShell
  • Job postings increasingly require PowerShell skills
  • Command Prompt is in maintenance mode (no new features)

Learn PowerShell now - it's essential for your career!

Skills Transfer

If you learned bash in Week 3, many concepts transfer to PowerShell:

  • Pipeline Concept: Chain commands together (syntax differs)
  • Variables: Store and manipulate data
  • Loops and Conditionals: Control flow in scripts
  • Functions: Reusable code blocks
  • Scripting Mindset: Automate repetitive tasks

The fundamental concepts are similar; the syntax and approaches differ.

PowerShell ISE
A graphical host application for PowerShell that provides a script editor with syntax highlighting, tab completion, debugging tools, and integrated help. Useful for writing and testing scripts.

ISE Features:

  • Script Pane: Write and edit scripts with syntax highlighting
  • Console Pane: Execute commands interactively
  • Commands Pane: Browse available cmdlets
  • IntelliSense: Auto-completion for cmdlets and parameters
  • Debugging: Set breakpoints, step through code
  • Multiple Tabs: Work on multiple scripts simultaneously

When to Use ISE vs. Console:

  • Use ISE For:
    • Writing scripts
    • Testing and debugging code
    • Learning PowerShell (IntelliSense helps)
    • Complex multi-line commands
  • Use Console For:
    • Quick one-line commands
    • Running existing scripts
    • Remote sessions
    • Faster startup time

  • PowerShell 7: Latest cross-platform version
  • Visual Studio Code: Modern editor with PowerShell extension
  • PowerShell ISE: Still useful but in maintenance mode

2. Cmdlet Structure and Naming Convention

▼

PowerShell uses a consistent, predictable naming convention that makes commands discoverable and easy to remember.

2.1 The Verb-Noun Convention

Cmdlet (Command-let)
A lightweight PowerShell command that performs a specific action. Cmdlets follow the Verb-Noun naming pattern, where the verb indicates the action and the noun indicates what the action affects.

Cmdlet Structure:

Verb-Noun -Parameter1 Value1 -Parameter2 Value2

Examples:
Get-Service -Name W32Time
Stop-Service -Name Spooler
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned
New-Item -Path C:\Temp -ItemType Directory

Common Verbs:

Common PowerShell Verbs
Verb Meaning Example Cmdlets
Get Retrieve information Get-Service, Get-Process, Get-ChildItem
Set Change configuration Set-Service, Set-ExecutionPolicy, Set-Location
New Create something New-Item, New-Service, New-LocalUser
Remove Delete something Remove-Item, Remove-Service, Remove-LocalUser
Start Begin an operation Start-Service, Start-Process, Start-VM
Stop End an operation Stop-Service, Stop-Process, Stop-Computer
Restart Stop and start Restart-Service, Restart-Computer
Test Verify or test Test-Connection, Test-Path, Test-NetConnection
Enable Turn on a feature Enable-PSRemoting, Enable-WindowsOptionalFeature
Disable Turn off a feature Disable-NetAdapter, Disable-WindowsOptionalFeature

Common Nouns:

  • Service: Windows services (Get-Service, Start-Service)
  • Process: Running processes (Get-Process, Stop-Process)
  • Item: File system items (Get-Item, New-Item, Remove-Item)
  • Computer: Computer-level operations (Restart-Computer, Rename-Computer)
  • EventLog: Event logs (Get-EventLog, Clear-EventLog)
  • Content: File contents (Get-Content, Set-Content)
  • NetAdapter: Network adapters (Get-NetAdapter, Disable-NetAdapter)

Discovering Cmdlets

The Verb-Noun pattern makes cmdlets discoverable:

  • Need to start a service? Try: Start-Service
  • Need to get computer info? Try: Get-ComputerInfo
  • Need to test network connection? Try: Test-NetConnection

If you can guess the verb and noun, you can often guess the cmdlet name!

2.2 Parameters and Arguments

Cmdlets accept parameters to modify their behavior or specify what to act upon.

Anatomy of a PowerShell Command:
Get-Service -Name W32Time -ComputerName Server01

Components:
- Get-Service     : The cmdlet
- -Name           : Parameter name
- W32Time         : Parameter value (argument)
- -ComputerName   : Another parameter
- Server01        : Its value

Parameter Types:

  • Named Parameters: Specify parameter name explicitly
    • Example: Get-Service -Name W32Time
    • Clear and explicit
    • Order doesn't matter
  • Positional Parameters: Omit parameter name, rely on position
    • Example: Get-Service W32Time (Name parameter is positional)
    • Shorter to type
    • Must be in correct order
  • Switch Parameters: Boolean flags (on/off)
    • Example: Get-ChildItem -Recurse
    • No value needed - presence means "true"
    • Omit to mean "false"

Parameter Shortcuts:

# Full parameter name
Get-Service -Name W32Time

# Abbreviated (minimum unique characters)
Get-Service -Na W32Time

# Tab completion
Get-Service -N[Tab]  # Completes to -Name

# Positional (no parameter name needed)
Get-Service W32Time

Best Practice: Use Full Parameter Names in Scripts

For interactive commands, abbreviations are fine. In scripts:

  • Use full parameter names for clarity
  • Makes scripts self-documenting
  • Easier for others to understand
  • Prevents ambiguity

2.3 Common Parameter Names

Many cmdlets share common parameter names:

Common PowerShell Parameters
Parameter Purpose Example
-Name Specify name of object Get-Service -Name W32Time
-Path Specify file/folder path Get-Item -Path C:\Temp
-ComputerName Specify remote computer Get-Service -ComputerName Server01
-Filter Filter results Get-ChildItem -Filter *.txt
-Recurse Include subdirectories Get-ChildItem -Recurse
-Force Override restrictions Remove-Item -Force
-WhatIf Show what would happen (don't execute) Remove-Item -WhatIf
-Confirm Prompt before executing Stop-Service -Confirm

Safety Parameters: -WhatIf and -Confirm

These parameters help prevent mistakes:

  • -WhatIf: Shows what would happen without actually doing it
    • Example: Remove-Item C:\Important -Recurse -WhatIf
    • Safe way to test destructive commands
  • -Confirm: Prompts for confirmation before executing
    • Example: Stop-Computer -Confirm
    • Prevents accidental execution

Best Practice: Always use -WhatIf first when testing destructive commands!

3. Getting Help in PowerShell

▼

PowerShell includes a comprehensive help system. Learning to use Get-Help effectively is one of the most important PowerShell skills.

3.1 The Get-Help Cmdlet

Get-Help
The cmdlet that displays help information about PowerShell cmdlets, functions, scripts, and concepts. PowerShell's built-in documentation system.

Basic Get-Help Usage:

# Get help for a specific cmdlet
Get-Help Get-Service

# Get detailed help with examples
Get-Help Get-Service -Detailed

# Get full help (everything)
Get-Help Get-Service -Full

# Get only examples
Get-Help Get-Service -Examples

# Get online help (opens browser)
Get-Help Get-Service -Online

# Search for cmdlets by keyword
Get-Help *service*

3.2 Updating Help Content

PowerShell help files are not fully installed by default. You should update them:

# Update help for all modules (requires internet, run as Administrator)
Update-Help

# Update help for specific module
Update-Help -Module Microsoft.PowerShell.Management

# Update help with error suppression (some modules may fail)
Update-Help -ErrorAction SilentlyContinue

First-Time Setup

Run Update-Help once after installing Windows Server to download the latest help files. This requires internet connection and takes a few minutes.

If you see "Access Denied" errors, ensure you're running PowerShell as Administrator.

3.3 Understanding Help Output

Get-Help output is organized into sections:

Shows the cmdlet name

SYNOPSIS Section ▼

Brief description of what the cmdlet does

SYNTAX Section ▼

Shows all valid ways to use the cmdlet with different parameter combinations

SYNTAX
    Get-Service [[-Name] ] [-ComputerName ] []
    
    Get-Service -DisplayName  [-ComputerName ] []

Reading Syntax:

  • [] around parameter = optional
  • <String[]> = data type (string array)
  • [-Name] in brackets = positional parameter

        Wildcards are permitted.

        Position?                    0
        Default value                None

Practical examples showing how to use the cmdlet (only with -Examples, -Detailed, or -Full)

5. The PowerShell Pipeline

▼

The pipeline is one of PowerShell's most powerful features, allowing you to chain commands together to create sophisticated operations from simple building blocks.

5.1 Understanding the Pipeline

Pipeline
A mechanism that passes the output of one cmdlet as input to another cmdlet. In PowerShell, the pipeline passes complete objects (not just text), making it more powerful than traditional Unix/Linux pipes.

Pipeline Syntax:

Cmdlet1 | Cmdlet2 | Cmdlet3

Example:
Get-Service | Where-Object {$_.Status -eq "Running"} | Sort-Object Name

How It Works:

  1. Get-Service retrieves all services as objects
  2. | (pipe) passes those objects to next cmdlet
  3. Where-Object filters to only running services
  4. | passes filtered objects to next cmdlet
  5. Sort-Object sorts by Name property
  6. Final sorted, filtered list is displayed

5.2 Object-Based Pipeline vs. Text-Based Pipeline

PowerShell Objects vs. Bash Text Streams
Aspect PowerShell (Objects) Bash (Text)
Data Type .NET objects with properties and methods Text strings
Accessing Data Object properties: $service.Name Text parsing: awk, cut, grep
Sorting Sort-Object PropertyName sort (text-based)
Filtering Where-Object {$_.Property -eq Value} grep, awk
Selecting Columns Select-Object Property1, Property2 cut, awk
Formatting Format-Table, Format-List column, manual formatting

Example Comparison:

Task: List running services, sorted by name

PowerShell (Object-Based):

Get-Service | Where-Object {$_.Status -eq "Running"} | Sort-Object Name

Bash (Text-Based):

systemctl list-units --type=service --state=running | awk '{print $1}' | sort

PowerShell Advantage: Direct access to Status property; no text parsing needed

5.3 Common Pipeline Cmdlets

Where-Object (Filtering) ▼

Purpose: Filter objects based on property values

# Filter running services
Get-Service | Where-Object {$_.Status -eq "Running"}

# Filter processes using more than 100MB
Get-Process | Where-Object {$_.WS -gt 100MB}

# Multiple conditions (AND)
Get-Service | Where-Object {$_.Status -eq "Running" -and $_.StartType -eq "Automatic"}

# Multiple conditions (OR)
Get-Process | Where-Object {$_.Name -eq "explorer" -or $_.Name -eq "powershell"}

# Shorthand syntax (PowerShell 3.0+)
Get-Service | Where Status -eq "Running"

Comparison Operators:

  • -eq : Equal to
  • -ne : Not equal to
  • -gt : Greater than
  • -lt : Less than
  • -ge : Greater than or equal
  • -le : Less than or equal
  • -like : Wildcard match
  • -match : Regex match
Select-Object (Selecting Properties) ▼

Purpose: Select specific properties or limit number of objects

# Select specific properties
Get-Service | Select-Object Name, Status, StartType

# Select first 10 objects
Get-Process | Select-Object -First 10

# Select last 5 objects
Get-EventLog -LogName System | Select-Object -Last 5

# Select unique values
Get-Process | Select-Object -Property ProcessName -Unique

# Exclude properties
Get-Service | Select-Object * -ExcludeProperty RequiredServices

# Create calculated properties
Get-Process | Select-Object Name, @{Name="MemoryMB"; Expression={$_.WS / 1MB}}
Sort-Object (Sorting) ▼

Purpose: Sort objects by property values

# Sort by single property
Get-Service | Sort-Object Name

# Sort descending
Get-Process | Sort-Object CPU -Descending

# Sort by multiple properties
Get-Service | Sort-Object Status, Name

# Sort and select top 10
Get-Process | Sort-Object WS -Descending | Select-Object -First 10
Measure-Object (Calculations) ▼

Purpose: Calculate statistics on object properties

# Count objects
Get-Service | Measure-Object

# Sum values
Get-Process | Measure-Object -Property WS -Sum

# Get statistics (min, max, average)
Get-Process | Measure-Object -Property CPU -Minimum -Maximum -Average

# Count lines in file
Get-Content C:\file.txt | Measure-Object -Line
Format Cmdlets (Output Formatting) ▼

Purpose: Control how output is displayed

# Format as table
Get-Service | Format-Table Name, Status, StartType

# Format as list
Get-Service W32Time | Format-List *

# Format as wide list
Get-Process | Format-Wide Name -Column 4

# Auto-size columns
Get-Service | Format-Table -AutoSize

Important: Format cmdlets should be the LAST in pipeline. Once you format, you can't pipe to other cmdlets (formatting destroys object structure).

5.4 Practical Pipeline Examples

Example 1: Find stopped automatic services
Get-Service | 
    Where-Object {$_.Status -eq "Stopped" -and $_.StartType -eq "Automatic"} | 
    Select-Object Name, DisplayName, Status | 
    Format-Table -AutoSize

Explanation: Gets all services, filters to stopped automatic ones, selects specific properties, formats as table

Example 2: Find top 5 memory-consuming processes
Get-Process | 
    Sort-Object WS -Descending | 
    Select-Object -First 5 Name, @{Name="MemoryMB";Expression={[math]::Round($_.WS/1MB,2)}} | 
    Format-Table -AutoSize

Explanation: Gets processes, sorts by memory (WS = Working Set), takes top 5, converts memory to MB, formats as table

Example 3: Count files by extension in a folder
Get-ChildItem C:\Windows -File | 
    Group-Object Extension | 
    Sort-Object Count -Descending | 
    Select-Object Name, Count | 
    Format-Table -AutoSize

Explanation: Gets all files, groups by extension, sorts by count, selects name and count, formats as table

5.5 PowerShell vs. Bash Pipeline Comparison

Task: List processes, filter by name, show top 5 by memory

PowerShell:

Get-Process | 
    Where-Object {$_.Name -like "s*"} | 
    Sort-Object WS -Descending | 
    Select-Object -First 5 Name, WS

Bash:

ps aux | 
    grep "^[^ ]*  *[^ ]*  *[^ ]*  *[^ ]*  *[^ ]*  *s" | 
    sort -k4 -rn | 
    head -5 | 
    awk '{print $11, $4}'

PowerShell Advantage:

  • Direct property access (no text parsing)
  • More readable and maintainable
  • Less error-prone (no regex or awk required)

6.1 What is a PowerShell Script?

PowerShell Script
A text file containing PowerShell commands, saved with a .ps1 extension. Scripts enable automation by executing multiple commands in sequence, with logic, variables, and functions.

Creating a Simple Script:

  1. Open PowerShell ISE or Text Editor

    PowerShell ISE provides best experience for scripting

  2. Write PowerShell Commands 
    # Simple script example
# File: Get-SystemInfo.ps1

Write-Host "System Information Report" -ForegroundColor Green
Write-Host "=========================" -ForegroundColor Green
Write-Host ""

# Computer name
Write-Host "Computer Name: $env:COMPUTERNAME"

# OS Version
$os = Get-CimInstance Win32_OperatingSystem
Write-Host "OS: $($os.Caption)"
Write-Host "Version: $($os.Version)"

# Uptime
$uptime = (Get-Date) - $os.LastBootUpTime
Write-Host "Uptime: $($uptime.Days) days, $($uptime.Hours) hours"

# Memory
$totalRAM = [math]::Round($os.TotalVisibleMemorySize / 1MB, 2)
$freeRAM = [math]::Round($os.FreePhysicalMemory / 1MB, 2)
Write-Host "Total RAM: $totalRAM GB"
Write-Host "Free RAM: $freeRAM GB"

# Disk space
Write-Host ""
Write-Host "Disk Space:"
Get-Volume | Where-Object {$_.DriveLetter} | 
    Select-Object DriveLetter, @{Name="SizeGB";Expression={[math]::Round($_.Size/1GB,2)}}, 
                  @{Name="FreeGB";Expression={[math]::Round($_.SizeRemaining/1GB,2)}} | 
    Format-Table -AutoSize
  3. Save with .ps1 Extension

    File → Save As → Choose location → Name: Get-SystemInfo.ps1

  4. Set Execution Policy (First Time Only) 
    # Check current execution policy
Get-ExecutionPolicy

# Set to allow local scripts (run as Administrator)
Set-ExecutionPolicy RemoteSigned -Force
  5. Run the Script 
    # Navigate to script location
cd C:\Scripts

# Run script
.\Get-SystemInfo.ps1

# Or specify full path
C:\Scripts\Get-SystemInfo.ps1

6.2 Execution Policy

Execution Policy
A safety feature that controls the conditions under which PowerShell loads configuration files and runs scripts. Helps prevent execution of malicious scripts.

Execution Policy Levels:

PowerShell Execution Policies
Policy Description Use Case
Restricted No scripts allowed (default on Windows clients) Maximum security, no scripting
AllSigned Only digitally signed scripts High-security environments
RemoteSigned Local scripts OK, downloaded scripts must be signed Recommended for most environments
Unrestricted All scripts allowed (warns for downloaded) Development/testing
Bypass No restrictions, no warnings Automated systems only

Managing Execution Policy:

# View current policy
Get-ExecutionPolicy

# View policy for all scopes
Get-ExecutionPolicy -List

# Set policy (requires Administrator)
Set-ExecutionPolicy RemoteSigned

# Set for current user only (doesn't require Administrator)
Set-ExecutionPolicy RemoteSigned -Scope CurrentUser

# Bypass policy for single script execution
PowerShell.exe -ExecutionPolicy Bypass -File C:\Scripts\script.ps1

Execution Policy is NOT Security

Important understanding:

  • Execution Policy is a safety feature, not a security boundary
  • Users can bypass it relatively easily
  • Real security comes from NTFS permissions and AppLocker
  • Purpose: Prevent accidental script execution, not malicious attacks

6.3 Variables in PowerShell

Variables store data for use in commands and scripts:

# Create variable ($ prefix)
$computerName = $env:COMPUTERNAME
$services = Get-Service
$number = 42

# Use variable
Write-Host "Computer: $computerName"
Write-Host "Number of services: $($services.Count)"

# Variable types are automatic (but can be explicit)
$name = "Server01"           # String
$count = 100                 # Integer
$price = 19.99               # Double
$isRunning = $true           # Boolean
$date = Get-Date             # DateTime object

# Explicit typing
[string]$name = "Server01"
[int]$count = 100

# Arrays
$servers = @("Server01", "Server02", "Server03")
$servers[0]  # Access first element
$servers.Count  # Number of elements

# Hash tables (key-value pairs)
$serverInfo = @{
    Name = "Server01"
    IP = "192.168.1.100"
    Role = "File Server"
}
$serverInfo.Name  # Access by key
$serverInfo["IP"]  # Alternative access

6.4 Simple Script Structure

Example: Service Status Report Script
# ServiceReport.ps1
# Generates report of service status

# Parameters (optional - Week 5 covers in detail)
param(
    [string]$ComputerName = $env:COMPUTERNAME
)

# Script header
Write-Host "Service Status Report for $ComputerName" -ForegroundColor Cyan
Write-Host "Generated: $(Get-Date)" -ForegroundColor Cyan
Write-Host ""

# Get service counts
$allServices = Get-Service
$runningServices = $allServices | Where-Object {$_.Status -eq "Running"}
$stoppedServices = $allServices | Where-Object {$_.Status -eq "Stopped"}

# Display summary
Write-Host "Summary:" -ForegroundColor Yellow
Write-Host "  Total Services: $($allServices.Count)"
Write-Host "  Running: $($runningServices.Count)"
Write-Host "  Stopped: $($stoppedServices.Count)"
Write-Host ""

# List stopped automatic services (potential issues)
Write-Host "Stopped Automatic Services (Potential Issues):" -ForegroundColor Red
$stoppedAuto = Get-Service | 
    Where-Object {$_.Status -eq "Stopped" -and $_.StartType -eq "Automatic"}

if ($stoppedAuto.Count -gt 0) {
    $stoppedAuto | Format-Table Name, DisplayName -AutoSize
} else {
    Write-Host "  None found - all automatic services are running!" -ForegroundColor Green
}

Write-Host ""
Write-Host "Report Complete" -ForegroundColor Cyan

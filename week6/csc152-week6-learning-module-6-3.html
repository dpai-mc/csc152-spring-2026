Introduction

While individual user accounts provide identity and authentication, groups are the foundation of efficient access control in Linux. Instead of assigning permissions to each user individually, administrators assign permissions to groups and then add users to appropriate groups. This approach dramatically simplifies permission management, especially in organizations with many users.

In this module, you'll learn how Linux groups work, how to create and manage them, and how to design effective group structures for real-world organizational needs.

Learning Objectives

  1. Explain the difference between primary and supplementary groups
  2. Identify and interpret the fields in /etc/group and /etc/gshadow
  3. Create and delete groups using groupadd and groupdel
  4. Modify groups using groupmod
  5. Manage group membership using gpasswd and usermod
  6. Design effective group structures for organizational needs

Why Groups Matter

üìã Scenario: The Permission Problem

Your company has 50 employees in the Engineering department who all need read/write access to the /projects/engineering directory. Without groups, you would need to set individual permissions for all 50 users‚Äîand update them every time someone joins or leaves the department. With groups, you create one "engineering" group, set permissions once, and simply add or remove users from the group as needed.

üí° Key Insight: Groups enable Role-Based Access Control (RBAC)‚Äîusers gain access based on their role (group membership) rather than their individual identity. This is the foundation of scalable permission management.

Understanding Linux Groups CSLO 1

What is a Group?

A group is a collection of user accounts that share common access permissions. Each group has:

  • Group name - Human-readable identifier
  • Group ID (GID) - Numeric identifier used by the kernel
  • Member list - Users who belong to the group
  • Optional password - Rarely used, for temporary group access

Primary vs. Supplementary Groups

Every user has exactly one primary group and can belong to multiple supplementary groups. Understanding this distinction is essential for effective permission management.

üîµ Primary Group
  • Quantity: Exactly one per user
  • Defined in: /etc/passwd (GID field)
  • Purpose: Default group for new files
  • Set with: useradd -g or usermod -g
Example: When user "jsmith" creates a file, it's automatically owned by jsmith's primary group.
üü¢ Supplementary Groups
  • Quantity: Zero or more per user
  • Defined in: /etc/group (member list)
  • Purpose: Additional access permissions
  • Set with: usermod -aG or gpasswd -a
Example: User "jsmith" can access files owned by the "developers" group if they're a supplementary member.

GID Ranges and Conventions

Like UIDs, Group IDs follow specific conventions:

GID Range Group Type Description
0 root The root group (superuser group)
1-999 System Groups Used by system services and daemons
1000+ Regular Groups User-created groups and user private groups

User Private Groups (UPG)

üìñ User Private Group Scheme:

Modern Linux distributions use the User Private Group (UPG) scheme. When you create a user, a group with the same name is automatically created as their primary group. This provides:

  • Better default security (files aren't group-readable by others)
  • Simplified management (one group per user by default)
  • Flexibility to add users to shared groups as needed
Terminal - UPG in action
$ sudo useradd -m jsmith $ id jsmith uid=1001(jsmith) gid=1001(jsmith) groups=1001(jsmith) # Note: A group "jsmith" was automatically created as primary group $ getent group jsmith jsmith:x:1001:

Group Configuration Files CSLO 1

Overview

Linux stores group information in two main files, similar to how user information is split between /etc/passwd and /etc/shadow:

/etc/ ‚îú‚îÄ‚îÄ group # Group definitions and membership (world-readable) ‚îî‚îÄ‚îÄ gshadow # Secure group information (root only)

The /etc/group File

The /etc/group file contains group definitions. Like /etc/passwd, it's readable by all users because many programs need to look up group information.

Format: /etc/group entry
groupname:password:GID:member_list

Field Breakdown

groupname Field 1: The group name (1-32 characters). Must be unique. Used for display and commands.
password Field 2: Group password placeholder. Usually x (password in gshadow) or empty. Group passwords are rarely used.
GID Field 3: Group ID number. The kernel uses this number to identify the group.
member_list Field 4: Comma-separated list of usernames who are supplementary members of this group. Primary group members are NOT listed here.
‚ö†Ô∏è Important: The member list in /etc/group only shows supplementary members. Users whose primary group is this group are NOT listed here‚Äîtheir primary group is recorded in /etc/passwd.

Example Entries

Example: Sample /etc/group entries
root:x:0: daemon:x:1: sudo:x:27:jsmith,asmith www-data:x:33: developers:x:1002:jsmith,asmith,bwilson engineering:x:1003: docker:x:999:jsmith
  • groupname: root
  • password: x (in gshadow)
  • GID: 0
  • members: (empty - the root user has this as primary group)
  • groupname: sudo
  • password: x
  • GID: 27
  • members: jsmith, asmith (these users can use sudo)
üí° Note: On Debian/Ubuntu systems, membership in the "sudo" group grants sudo privileges. On RHEL/Rocky systems, the equivalent group is "wheel".
  • groupname: developers
  • password: x
  • GID: 1002
  • members: jsmith, asmith, bwilson

This is a custom group for developers. All three users have this as a supplementary group, giving them access to files owned by the "developers" group.

  • groupname: engineering
  • password: x
  • GID: 1003
  • members: (empty)

The empty member list doesn't mean no one is in this group! It likely means users have this as their primary group (set in /etc/passwd), not as a supplementary group.

The /etc/gshadow File

The /etc/gshadow file stores secure group information, similar to how /etc/shadow stores secure user information.

Format: /etc/gshadow entry
groupname:password:administrators:members

Field Breakdown

groupname Field 1: Group name (must match /etc/group)
password Field 2: Encrypted group password. ! or * means no password. Rarely used.
administrators Field 3: Users who can manage group membership (add/remove members)
members Field 4: Group members (same as /etc/group)
Example: Sample /etc/gshadow entries
root:*:: sudo:*::jsmith,asmith developers:!:jsmith:jsmith,asmith,bwilson
üí° Group Administrators: The administrators field allows non-root users to manage group membership. In the example above, "jsmith" can add or remove members from the "developers" group using the gpasswd command without needing sudo.

Viewing Group Information

Terminal - Viewing group information
# View all groups $ cat /etc/group # Search for specific group $ getent group developers developers:x:1002:jsmith,asmith,bwilson # Show groups for current user $ groups jsmith sudo developers docker # Show groups for specific user $ groups asmith asmith : asmith sudo developers # Detailed group info with id command $ id jsmith uid=1001(jsmith) gid=1001(jsmith) groups=1001(jsmith),27(sudo),1002(developers),999(docker)

Group Management Commands CSLO 1

Command Overview

Command Purpose Requires Root
groupadd Create a new group Yes
groupmod Modify an existing group Yes
groupdel Delete a group Yes
gpasswd Administer group membership Yes (or group admin)
usermod -aG Add user to group Yes
groups Show user's groups No

Creating Groups with groupadd

Syntax
sudo groupadd [OPTIONS] GROUPNAME

Options

Option Description Example
-g GID Specify group ID -g 2000
-r Create system group (GID < 1000) -r
-f Force (exit successfully if group exists) -f
Terminal - Creating groups
# Create a basic group $ sudo groupadd developers # Verify creation $ getent group developers developers:x:1002: # Create group with specific GID $ sudo groupadd -g 2000 engineering $ getent group engineering engineering:x:2000:
Terminal - Creating system group
# Create system group for a service $ sudo groupadd -r appservice $ getent group appservice appservice:x:998: # Note: GID is below 1000 (system range)
üí° When to use -r: Use the -r option when creating groups for system services or daemons. This keeps system groups organized in the lower GID range.

Modifying Groups with groupmod

Syntax
sudo groupmod [OPTIONS] GROUPNAME

Options

Option Description Example
-n NEWNAME Rename the group -n devteam developers
-g GID Change group ID -g 2500 developers
Terminal - Modifying groups
# Rename a group $ sudo groupmod -n devteam developers $ getent group devteam devteam:x:1002:jsmith,asmith # Change GID (use with caution!) $ sudo groupmod -g 2500 devteam
‚ö†Ô∏è Caution with GID Changes: Changing a group's GID does NOT update file ownerships. Files will show numeric GIDs instead of group names until you run find / -gid OLD_GID -exec chgrp NEW_GROUP {} \;

Deleting Groups with groupdel

Syntax
sudo groupdel GROUPNAME
Terminal - Deleting groups
# Delete a group $ sudo groupdel oldproject # Verify deletion $ getent group oldproject (no output - group deleted)
üö´ Cannot Delete Primary Groups: You cannot delete a group that is any user's primary group. You must first change or delete those users.
$ sudo groupdel jsmith groupdel: cannot remove the primary group of user 'jsmith'

Managing Membership with gpasswd

The gpasswd command is specifically designed for group administration. It's often preferred over usermod for managing group membership.

Syntax
sudo gpasswd [OPTIONS] GROUPNAME

Options

Option Description Example
-a USER Add user to group -a jsmith developers
-d USER Remove user from group -d jsmith developers
-M USER,... Set member list (replaces all) -M alice,bob developers
-A USER,... Set group administrators -A jsmith developers
Terminal - Managing group membership
# Add user to group $ sudo gpasswd -a jsmith developers Adding user jsmith to group developers # Verify $ groups jsmith jsmith : jsmith sudo developers # Remove user from group $ sudo gpasswd -d jsmith developers Removing user jsmith from group developers $ groups jsmith jsmith : jsmith sudo
Terminal - Group administrators
# Make jsmith an administrator of developers group $ sudo gpasswd -A jsmith developers # Now jsmith can add members without sudo jsmith$ gpasswd -a newuser developers Adding user newuser to group developers
üí° Delegation: Group administrators allow you to delegate group management to team leads without giving them full root access.

gpasswd vs usermod -aG

Both commands can add users to groups. Here's when to use each:

Task gpasswd usermod
Add user to one group gpasswd -a user group ‚úì usermod -aG group user ‚úì
Add user to multiple groups Run multiple times usermod -aG g1,g2,g3 user ‚úì
Remove user from group gpasswd -d user group ‚úì Not directly supported ‚úó
Set group admin gpasswd -A admin group ‚úì Not supported ‚úó
‚úì Recommendation: Use gpasswd -a to add users and gpasswd -d to remove users. Use usermod -aG when adding a user to multiple groups at once.

When Group Changes Take Effect

‚ö†Ô∏è Important: Group membership changes don't take effect until the user logs out and back in, or starts a new login shell. To apply changes immediately:
# User can run newgrp to activate new group in current session newgrp developers # Or log out and back in

Group Design Strategies CSLO 1

Why Group Design Matters

A well-designed group structure makes permission management scalable and maintainable. Poor group design leads to permission sprawl, security gaps, and administrative overhead.

Common Group Strategies

Create groups based on organizational departments. This is the most common approach.

Organization
All employees
‚Üì
engineering
GID 2001
marketing
GID 2002
accounting
GID 2003
hr
GID 2004
Commands: Create departmental groups
sudo groupadd -g 2001 engineering sudo groupadd -g 2002 marketing sudo groupadd -g 2003 accounting sudo groupadd -g 2004 hr
‚úì Best for: Traditional organizations with clear departmental boundaries and department-specific resources.

Create groups for projects or teams that span departments. Users can belong to multiple project groups.

project-alpha
Cross-functional team
project-beta
Cross-functional team
project-gamma
Cross-functional team
Commands: Create project groups and add members
sudo groupadd project-alpha sudo groupadd project-beta # Add members from different departments sudo gpasswd -a jsmith project-alpha # Engineer sudo gpasswd -a mwilson project-alpha # Marketing sudo gpasswd -a ajones project-alpha # Accounting
‚úì Best for: Organizations with cross-functional teams, agile environments, or project-based work.

Create groups based on job functions or access levels, regardless of department.

Group Purpose Example Members
sysadmins System administrators IT staff with admin access
developers Software developers Anyone who writes code
managers Management access Department heads, team leads
readonly Read-only access Auditors, interns
Commands: Create role-based groups
sudo groupadd sysadmins sudo groupadd developers sudo groupadd managers sudo groupadd readonly
‚úì Best for: Organizations implementing Role-Based Access Control (RBAC) or with complex permission requirements.

Combine departmental, project, and role-based groups for maximum flexibility. This is the most common real-world approach.

Example: Hybrid group structure
# Departmental groups (primary organization) sudo groupadd engineering sudo groupadd marketing sudo groupadd accounting # Project groups (cross-functional access) sudo groupadd project-website sudo groupadd project-mobile # Role-based groups (functional access) sudo groupadd developers sudo groupadd dbadmins sudo groupadd sudo # Example user setup # Alice: Engineer working on website project with dev access sudo usermod -aG engineering,project-website,developers alice
üí° Design Principle: A user's primary group should reflect their department (for file ownership), while supplementary groups grant access to projects and resources.

Group Naming Conventions

Consistent naming makes groups easier to manage and understand:

Type Convention Examples
Departments Lowercase, no prefix engineering, marketing, hr
Projects project- prefix project-alpha, project-website
Applications app- prefix app-database, app-webserver
Roles role- prefix or descriptive role-readonly, developers, dbadmins

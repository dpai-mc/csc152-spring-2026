Introduction

Throughout this week, you've learned to create users, manage groups, and configure password policies. But there's a critical security question we haven't fully addressed: How do users perform administrative tasks without having permanent administrative access?

This module introduces the Principle of Least Privilege and teaches you how to implement controlled privilege escalation on both Linux (using sudo) and Windows (using UAC and User Rights Assignment).

üîê The Security Dilemma

A system administrator needs to restart a web service, but they shouldn't have permanent root/admin access. A developer needs to install a package for testing, but they shouldn't be able to modify system files. How do you give users the access they need without giving them the keys to the kingdom?

The Principle of Least Privilege CSLO 2

Definition

üõ°Ô∏è Principle of Least Privilege (PoLP):

Users, processes, and systems should have only the minimum permissions necessary to perform their job functions‚Äînothing more.

Why Least Privilege Matters

Benefit Explanation
Reduces Attack Surface Fewer permissions mean fewer ways for attackers to cause damage
Limits Damage A compromised standard user account can't install rootkits
Improves Accountability Elevated actions are logged and traceable
Prevents Accidents Users can't accidentally delete critical files
Supports Compliance Required by PCI-DSS, HIPAA, SOX, and other frameworks

Implementation Strategies

Start with zero permissions and grant only what's explicitly needed.

  • New users should have no special privileges
  • Access is granted through documented requests
  • Each permission grant has a business justification

Assign permissions to groups, then add users to groups.

  • Easier to audit and manage
  • Consistent permissions for similar roles
  • Simple to revoke when user changes roles

Administrators should have two accounts:

  • jsmith - Regular account for email, browsing
  • jsmith-admin - Admin account for server management

Elevated access should be temporary:

  • sudo sessions expire after 15 minutes
  • Just-In-Time (JIT) access for cloud resources

Periodically review and remove unnecessary access:

  • Quarterly access reviews for privileged accounts
  • Immediate revocation when employees leave

Linux: Understanding sudo CSLO 1, 2

What is sudo?

üìñ sudo (Superuser Do):

sudo allows permitted users to execute commands as root (or another user) without logging in as root. It provides granular control over administrative access.

Why sudo Instead of Root Login?

Root Login sudo
‚ùå All commands run as root‚úì Only specified commands elevated
‚ùå No audit trail of who did what‚úì All commands logged with username
‚ùå Shared root password risk‚úì Users authenticate with own password
‚ùå All-or-nothing access‚úì Granular command restrictions
‚ùå No timeout on access‚úì Credentials expire (default 15 min)

Basic sudo Usage

Linux: Basic sudo commands
# Run a single command as root $ sudo systemctl restart nginx [sudo] password for jsmith: ******** # Run command as a different user $ sudo -u www-data whoami www-data # List your sudo privileges $ sudo -l User jsmith may run the following commands on server: (ALL : ALL) ALL # Start a root shell (use sparingly!) $ sudo -i # Invalidate cached credentials $ sudo -k

Configuring /etc/sudoers CSLO 1, 2

The sudoers File

The /etc/sudoers file controls who can use sudo and what commands they can run.

üö® CRITICAL: Always Use visudo!

Never edit /etc/sudoers directly. Always use visudo, which validates syntax before saving and prevents lockouts.

Linux: Using visudo
# Edit the main sudoers file $ sudo visudo # Edit a file in sudoers.d $ sudo visudo -f /etc/sudoers.d/webadmins # Check syntax without editing $ sudo visudo -c

sudoers Syntax

sudoers Rule Format

user host=(runas) commands
user - Who can sudo host - On which hosts (runas) - Run as whom commands - What commands

Common sudoers Examples

sudoers: Full administrative access
# User can run any command as any user jsmith ALL=(ALL:ALL) ALL # Members of sudo group (Debian/Ubuntu) %sudo ALL=(ALL:ALL) ALL # Members of wheel group (RHEL/Rocky) %wheel ALL=(ALL:ALL) ALL
sudoers: Specific service management
# Web admins can manage nginx %webadmins ALL=(root) /bin/systemctl start nginx, \ /bin/systemctl stop nginx, \ /bin/systemctl restart nginx, \ /bin/systemctl reload nginx, \ /bin/systemctl status nginx
sudoers: Skip password prompt
# Service account can run backup without password svc_backup ALL=(root) NOPASSWD: /usr/local/bin/backup.sh # Monitoring can check status without password monitoring ALL=(root) NOPASSWD: /bin/systemctl status *
‚ö†Ô∏è Security Trade-off: Use NOPASSWD only for service accounts or highly restricted commands.

Using sudoers.d Directory

Linux: Creating sudoers.d files
# Create a file for web administrators $ sudo visudo -f /etc/sudoers.d/webadmins # Add rules to the file: # Web Administrators - nginx management %webadmins ALL=(root) /bin/systemctl start nginx, \ /bin/systemctl stop nginx, \ /bin/systemctl restart nginx # Set correct permissions (REQUIRED!) $ sudo chmod 440 /etc/sudoers.d/webadmins # Verify syntax $ sudo visudo -c
üí° Benefits of sudoers.d: Modular configuration, version control friendly, easier automation, less risk of breaking main file.

Windows: UAC and User Rights CSLO 1, 2

User Account Control (UAC)

üìñ User Account Control (UAC):

UAC prevents unauthorized changes to the OS. When administrative privileges are needed, UAC prompts for consent or credentials.

How UAC Works

Even administrators run with standard privileges by default. Administrative tasks trigger a UAC prompt:

Standard User

UAC prompts for admin credentials

Administrator

UAC prompts for consent (Yes/No)

Run as Administrator

PowerShell: Running elevated
# Start elevated PowerShell Start-Process -FilePath "powershell" -Verb RunAs # Check if current session is elevated ([Security.Principal.WindowsPrincipal][Security.Principal.WindowsIdentity]::GetCurrent()).IsInRole([Security.Principal.WindowsBuiltInRole]::Administrator) # Run as different user (cmd) runas /user:Administrator cmd.exe

User Rights Assignment

Windows provides granular control through User Rights Assignment in Local Security Policy (secpol.msc):

User Right Description
Log on locallyAllow interactive logon
Log on through Remote DesktopAllow RDP connections
Shut down the systemAllow system shutdown
Back up files and directoriesBypass file permissions for backup
Take ownership of filesTake ownership of any object

Linux sudo vs Windows UAC

Aspect Linux sudo Windows UAC
GranularityPer-command controlPer-application elevation
Configuration/etc/sudoersGroup membership, User Rights
AuthenticationUser's passwordConsent or admin credentials
Credential caching15 minutes (default)Per-process, no caching
LoggingAll commands loggedEvent log entries
üö® Never Disable UAC on Servers!

Disabling UAC removes a critical security layer. Keep it enabled for audit trail and accident prevention.

Introduction

User account management is one of the most fundamental responsibilities of a Linux system administrator. Every person or service that needs to access a Linux system requires a user account. Understanding how Linux stores, manages, and secures user account information is essential for maintaining system security and ensuring proper access control.

In this module, you will learn the foundational concepts of Linux user accounts, explore the critical system files that store user information, and begin working with the commands used to create and manage users.

Learning Objectives

  1. Explain the purpose and structure of user accounts in Linux
  2. Identify the fields in the /etc/passwd file and explain their purpose
  3. Identify the fields in the /etc/shadow file and explain their security function
  4. Distinguish between regular users, system users, and the root superuser
  5. Use basic user information commands (id, whoami, who)
  6. Understand User ID (UID) and Group ID (GID) numbering conventions

Core Concepts CSLO 1

What is a User Account?

A user account is a mechanism that provides security boundaries and resource allocation for individuals or services interacting with a Linux system. Each user account has:

  • A unique identifier (UID) - A number that the system uses internally
  • A username - A human-readable name for login
  • Authentication credentials - Typically a password (stored securely)
  • A home directory - Personal file storage space
  • A default shell - The command interpreter for interactive sessions
  • Group memberships - For shared access to resources
üìñ Definition: A User Identifier (UID) is a unique positive integer assigned to each user account. The Linux kernel uses UIDs‚Äînot usernames‚Äîto identify users and determine access permissions.

Types of User Accounts

Linux systems have three primary categories of user accounts:

üî¥ Root User (Superuser)
  • UID: Always 0
  • Purpose: Complete system control
  • Access: Unrestricted
  • Risk: High - mistakes can destroy the system

The root account bypasses all permission checks. Use with extreme caution.

üë§ Regular Users
  • UID: 1000+ (modern systems)
  • Purpose: Human users
  • Access: Limited by permissions
  • Home: /home/username

Created for actual people who log into the system.

‚öôÔ∏è System Users
  • UID: 1-999 (typically)
  • Purpose: Run services/daemons
  • Access: Minimal, service-specific
  • Shell: Often /sbin/nologin

Used by services like Apache, MySQL, and mail servers.

UID Ranges and Conventions

Different Linux distributions may use slightly different UID ranges, but the following conventions are common:

UID Range Account Type Description
0 Root Superuser with complete system access
1-200 System (Static) Reserved for system processes by distribution
201-999 System (Dynamic) Assigned dynamically for services
1000-60000 Regular Users Normal user accounts (humans)
65534 Nobody Used for NFS and other special cases
‚ö†Ô∏è Distribution Differences: The exact UID boundaries can vary. Ubuntu and RHEL-based systems typically start regular users at UID 1000, while some older systems used UID 500. Always check /etc/login.defs for your system's configuration.

The Principle of Least Privilege

A fundamental security concept in user management is the Principle of Least Privilege: users should have only the minimum permissions necessary to perform their job functions.

üîí Security Best Practice: Never use the root account for daily tasks. Create regular user accounts and use sudo for administrative tasks. This provides accountability (audit trails) and reduces the risk of accidental system damage.

The /etc/passwd File CSLO 1

Overview

The /etc/passwd file is the primary database for user account information on Linux systems. Despite its name, this file no longer contains actual passwords on modern systems (those are stored in /etc/shadow).

üìñ Key Point: The /etc/passwd file must be readable by all users because many programs need to look up user information (like converting UIDs to usernames). This is why passwords were moved to the protected /etc/shadow file.

File Format

Each line in /etc/passwd represents one user account. Fields are separated by colons (:) with seven fields per line:

Format: /etc/passwd entry structure
username:password:UID:GID:GECOS:home_directory:shell

Field Breakdown

username Field 1: The login name (1-32 characters). Must be unique. Typically lowercase letters, numbers, underscores. Used for login and display purposes.
password Field 2: Historically contained the encrypted password. On modern systems, contains an x indicating the password is stored in /etc/shadow.
UID Field 3: User ID number. The kernel uses this number (not the username) to identify the user. UID 0 is always root.
GID Field 4: Primary Group ID. The default group assigned to files created by this user. References an entry in /etc/group.
GECOS Field 5: Comment field (historically "General Electric Comprehensive Operating System"). Usually contains the user's full name. May include office location, phone numbers separated by commas.
home_directory Field 6: Absolute path to the user's home directory. For regular users, typically /home/username. This is where users land after login.
shell Field 7: The user's default login shell. Common values: /bin/bash, /bin/sh, /sbin/nologin (prevents interactive login).

Example Entries

Let's examine some typical /etc/passwd entries:

Example: Sample /etc/passwd entries
root:x:0:0:root:/root:/bin/bash daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin nobody:x:65534:65534:nobody:/nonexistent:/usr/sbin/nologin jsmith:x:1001:1001:John Smith,Room 101,555-1234:/home/jsmith:/bin/bash mysql:x:27:27:MySQL Server:/var/lib/mysql:/sbin/nologin

Entry Analysis

  • username: root (the superuser)
  • password: x (stored in /etc/shadow)
  • UID: 0 (always 0 for root)
  • GID: 0 (root group)
  • GECOS: root
  • home: /root (not /home/root)
  • shell: /bin/bash (full shell access)
  • username: www-data (web server user)
  • password: x (stored in /etc/shadow)
  • UID: 33 (system user range)
  • GID: 33 (www-data group)
  • GECOS: www-data
  • home: /var/www (web content directory)
  • shell: /usr/sbin/nologin (cannot log in interactively)

Security Note: The nologin shell prevents anyone from logging in as this user, which is appropriate for service accounts.

  • username: jsmith (regular user)
  • password: x (stored in /etc/shadow)
  • UID: 1001 (regular user range)
  • GID: 1001 (user's primary group)
  • GECOS: John Smith,Room 101,555-1234 (full name and contact info)
  • home: /home/jsmith (standard home directory)
  • shell: /bin/bash (interactive shell)

Note: The GECOS field contains comma-separated values: Full Name, Office/Room, Phone Number.

Viewing /etc/passwd

You can view the passwd file using several commands:

Command: View entire passwd file
cat /etc/passwd
Command: Search for a specific user
grep "jsmith" /etc/passwd
Command: View using getent (preferred method)
getent passwd jsmith
üí° Why use getent? The getent command queries the Name Service Switch (NSS) configuration, which means it can retrieve user information from multiple sources (local files, LDAP, NIS, etc.), not just /etc/passwd.

The /etc/shadow File CSLO 2

Overview

The /etc/shadow file stores secure user account information, most importantly the encrypted (hashed) passwords. This file was created to address the security weakness of having password hashes in the world-readable /etc/passwd file.

üîí Security Critical: The /etc/shadow file has permissions 640 or 000 and is owned by root. Only the root user (and the shadow group on some systems) can read this file. This protects password hashes from being copied and cracked offline.

File Format

Each line in /etc/shadow corresponds to a user in /etc/passwd. Fields are separated by colons with nine fields per line:

Format: /etc/shadow entry structure
username:password_hash:lastchange:min:max:warn:inactive:expire:reserved

Field Breakdown

username Field 1: Login name. Must match an entry in /etc/passwd.
password_hash Field 2: The encrypted password. Special values: ! or * means account is locked/no password login. !! means password has never been set. Empty means no password required (dangerous!).
lastchange Field 3: Days since Jan 1, 1970 (epoch) when password was last changed. Value of 0 forces password change at next login.
min Field 4: Minimum days between password changes. Prevents users from immediately changing back to an old password.
max Field 5: Maximum days a password is valid. After this, user must change password. Value of 99999 effectively disables expiration.
warn Field 6: Days before password expires to warn the user. Typically 7 days.
inactive Field 7: Days after password expires until account is disabled. Grace period for password change.
expire Field 8: Days since epoch when account expires completely. Empty means account never expires.
reserved Field 9: Reserved for future use. Currently unused.

Password Hash Format

Modern Linux systems use modular crypt format for password hashes. The hash field contains:

Format: Password hash structure
$algorithm$salt$hash
ID Algorithm Description
$1$ MD5 Legacy, considered weak (do not use)
$5$ SHA-256 Good security, widely supported
$6$ SHA-512 Strong security, modern default
$y$ yescrypt Memory-hard, newest and most secure

Example Entries

Example: Sample /etc/shadow entries (hashes truncated)
root:$6$xyz123...:19500:0:99999:7::: daemon:*:19000:0:99999:7::: jsmith:$6$abc789...:19450:0:90:7:30:: newuser:!!:19500:0:99999:7::: locked:!$6$def456...:19400:0:99999:7:::

Entry Analysis

  • Password: Set (SHA-512 hash starting with $6$)
  • Last changed: Day 19500 since epoch
  • Min days: 0 (can change anytime)
  • Max days: 99999 (never expires)
  • Warn: 7 days
  • Status: Active, can log in
  • Password: * (account cannot authenticate with password)
  • Purpose: System service account
  • Status: Cannot log in with password

The asterisk (*) indicates this account was never intended for interactive login.

  • Password: Set (SHA-512 hash)
  • Max days: 90 (must change every 90 days)
  • Warn: 7 days before expiration
  • Inactive: 30 days grace period
  • Status: Active with password policy
  • Password: !! (password never set)
  • Status: Account exists but cannot log in until password is set

This is common when an account is created with useradd without the -p option.

  • Password: ! prefix before hash (locked)
  • Status: Account is locked, cannot log in
  • Note: The hash is preserved; unlocking restores access

Locked with passwd -l or usermod -L. The ! prefix is prepended to the existing hash.

Viewing Shadow Information

Since /etc/shadow is protected, you need root privileges to view it:

Command: View shadow file (requires root)
sudo cat /etc/shadow sudo grep "jsmith" /etc/shadow sudo getent shadow jsmith

Password Aging Commands

Use the chage command to view and modify password aging settings:

Command: View password aging information
# View aging info for a user sudo chage -l jsmith # Example output: Last password change : Feb 15, 2026 Password expires : May 16, 2026 Password inactive : Jun 15, 2026 Account expires : never Minimum number of days between password change : 0 Maximum number of days between password change : 90 Number of days of warning before password expires : 7

User Information Commands CSLO 1

Overview

Before creating and managing users, you need to know how to query user information. These commands help you understand the current state of user accounts on a system.

Essential Commands

The whoami command displays the username of the current effective user.

Command: whoami
$ whoami jsmith $ sudo whoami root
üí° Note: whoami shows the effective user, which may differ from the logged-in user when using sudo or su.

The id command displays the UID, GID, and group memberships for a user.

Command: id usage examples
# Display current user's information $ id uid=1001(jsmith) gid=1001(jsmith) groups=1001(jsmith),27(sudo),1002(developers) # Display specific user's information $ id root uid=0(root) gid=0(root) groups=0(root) # Display only the UID $ id -u 1001 # Display only the primary GID $ id -g 1001 # Display only group names $ id -Gn jsmith sudo developers

Common Options

Option Description
-u Print only the effective UID
-g Print only the effective GID
-G Print all group IDs
-n Print names instead of numbers (use with -u, -g, -G)

The who command shows who is currently logged into the system.

Command: who usage examples
# Show all logged-in users $ who jsmith pts/0 2026-02-23 09:15 (192.168.1.100) admin pts/1 2026-02-23 08:30 (192.168.1.50) jsmith pts/2 2026-02-23 10:00 (192.168.1.100) # Show current user's terminal $ who am i jsmith pts/0 2026-02-23 09:15 (192.168.1.100) # Show with headers $ who -H NAME LINE TIME COMMENT jsmith pts/0 2026-02-23 09:15 (192.168.1.100)
üí° Output Fields:
  • NAME: Username
  • LINE: Terminal device (pts = pseudo-terminal)
  • TIME: Login time
  • COMMENT: Remote host (if applicable)

The w command shows who is logged in and what they are doing.

Command: w output example
$ w 10:30:15 up 5 days, 3:45, 2 users, load average: 0.15, 0.10, 0.08 USER TTY FROM LOGIN@ IDLE JCPU PCPU WHAT jsmith pts/0 192.168.1.100 09:15 0.00s 0.05s 0.00s w admin pts/1 192.168.1.50 08:30 1:45m 0.10s 0.02s vim config.txt

Output Fields Explained

Field Description
USER Username
TTY Terminal name
FROM Remote host
LOGIN@ Login time
IDLE Idle time
JCPU CPU time used by all processes on TTY
PCPU CPU time used by current process
WHAT Current process/command

The last command shows a listing of last logged-in users.

Command: last usage examples
# Show recent logins $ last jsmith pts/0 192.168.1.100 Mon Feb 23 09:15 still logged in admin pts/1 192.168.1.50 Mon Feb 23 08:30 - 09:00 (00:30) jsmith pts/0 192.168.1.100 Sun Feb 22 14:00 - 18:30 (04:30) reboot system boot 5.15.0-generic Sun Feb 22 10:00 still running # Show logins for specific user $ last jsmith # Show last 5 entries $ last -n 5 # Show bad login attempts $ sudo lastb
üîí Security Tip: Regularly review last and lastb (failed logins) output to detect unauthorized access attempts or suspicious activity.

The getent command retrieves entries from Name Service Switch libraries, including passwd, shadow, and group databases.

Command: getent usage examples
# Get passwd entry for a user $ getent passwd jsmith jsmith:x:1001:1001:John Smith:/home/jsmith:/bin/bash # Get all passwd entries $ getent passwd # Get group entry $ getent group developers developers:x:1002:jsmith,asmith # Get shadow entry (requires root) $ sudo getent shadow jsmith
üí° Why getent? Unlike grep /etc/passwd, getent queries all configured sources (local files, LDAP, NIS, SSSD, etc.) as defined in /etc/nsswitch.conf. This is essential in enterprise environments using centralized authentication.

The finger command displays information about system users. Note: May need to be installed on modern systems.

Command: finger usage
# Install finger (if needed) $ sudo apt install finger # Debian/Ubuntu $ sudo dnf install finger # RHEL/Rocky # Display user information $ finger jsmith Login: jsmith Name: John Smith Directory: /home/jsmith Shell: /bin/bash Office: Room 101 Office Phone: 555-1234 On since Mon Feb 23 09:15 (PST) on pts/0 from 192.168.1.100 2 seconds idle No mail. No Plan.

Key Takeaways

üìù Remember These Points:
  • User accounts provide security boundaries and resource allocation on Linux systems
  • The kernel uses UIDs (not usernames) to identify users and check permissions
  • UID 0 is always root; UIDs 1-999 are typically system accounts; UIDs 1000+ are regular users
  • /etc/passwd stores user account information (world-readable)
  • /etc/shadow stores password hashes securely (root-only access)
  • Modern systems use SHA-512 ($6$) or yescrypt ($y$) for password hashing
  • Use getent instead of grep to query user databases in enterprise environments

Introduction

Having mastered Linux user and group management, you're now ready to learn the Windows approach. While the fundamental concepts are similar‚Äîusers need accounts, accounts need identification, and access is controlled through groups‚ÄîWindows implements these concepts quite differently from Linux.

This module introduces Windows user account fundamentals, focusing on the concepts you need to understand before working with the management tools. You'll learn about account types, Security Identifiers (SIDs), built-in accounts, and account properties.

Linux vs. Windows: Quick Comparison

üêß Linux

Local Accounts

üìñ Definition: A local account is stored in the computer's Security Account Manager (SAM) database and is valid only on that specific computer. Local accounts cannot be used to access resources on other computers in a network.

Key characteristics of local accounts:

  • Stored in C:\Windows\System32\config\SAM (encrypted)
  • Valid only on the local machine
  • Created and managed through Computer Management or PowerShell
  • Used on standalone servers or workgroup computers
  • Each computer maintains its own separate user database
üí° This Week's Focus: In Week 6, we focus on local accounts for standalone Windows Server management. Domain accounts and Active Directory are covered in Weeks 11-12.

The SAM Database

The Security Account Manager (SAM) is a database that stores local user accounts and groups. It's the Windows equivalent of Linux's /etc/passwd and /etc/shadow files combined.

SAM Database Facts
  • Location: %SystemRoot%\System32\config\SAM
  • Format: Binary registry hive
  • Access: Protected by SYSTEM account
  • Cannot be: Directly edited or viewed
  • Managed via: Windows APIs and tools
Linux Comparison
  • Equivalent: /etc/passwd + /etc/shadow
  • Format: Plain text (Linux)
  • Access: Root can read shadow
  • Can be: Edited with text editors
  • Managed via: Commands or direct editing
‚ö†Ô∏è Important Difference: Unlike Linux where you can view /etc/passwd directly, the Windows SAM database is binary and encrypted. You must use Windows tools (GUI, PowerShell, or net commands) to manage users‚Äîthere's no option to edit configuration files directly.

Domain Accounts (Preview)

While we'll cover domain accounts in detail during Weeks 11-12, here's a brief overview:

  • Stored in Active Directory on domain controllers
  • Valid across all computers in the domain
  • Centrally managed from any domain-joined computer
  • Support advanced features: Group Policy, delegation, forest trusts
  • Required for enterprise environments
üí° When to Use Each Type:
  • Local accounts: Standalone servers, test environments, local admin access
  • Domain accounts: Enterprise networks with multiple servers and users
  • Microsoft accounts: Consumer devices, Microsoft 365 integration

Security Identifiers (SIDs) CSLO 1, 2

What is a SID?

A Security Identifier (SID) is a unique value that identifies a security principal (user, group, or computer) in Windows. SIDs are the Windows equivalent of Linux UIDs and GIDs, but with important differences.

üìñ Definition: A Security Identifier (SID) is a variable-length unique value used to identify a trustee (security principal). Each account or group is issued a unique SID that remains constant for the lifetime of that account.

SID Structure

A SID consists of several components that together form a unique identifier:

Example SID Structure

S-1-5-21-3623811015-3361044348-30300820-1013
Prefix S
Revision 1
Authority 5
Domain/Machine ID 21-3623811015-3361044348-30300820
RID 1013

SID Component Breakdown

Component Description Example Value
S Identifies string as a SID S
Revision Level SID specification version (always 1) 1
Identifier Authority Highest level of authority (5 = NT Authority) 5
Domain/Machine ID Unique identifier for domain or computer 21-3623811015-3361044348-30300820
RID (Relative ID) Unique ID for account within domain/machine 1013

Well-Known SIDs

Some SIDs are predefined and consistent across all Windows installations. These are called well-known SIDs:

SID Name Description
S-1-5-18 SYSTEM Local System account (highest privileges)
S-1-5-19 LOCAL SERVICE Local Service account
S-1-5-20 NETWORK SERVICE Network Service account
S-1-5-32-544 Administrators Built-in Administrators group
S-1-5-32-545 Users Built-in Users group
S-1-5-domain-500 Administrator Built-in Administrator account
S-1-5-domain-501 Guest Built-in Guest account

SID vs. UID Comparison

üêß Linux UID
  • Format: Simple integer (e.g., 1001)
  • Length: 32-bit number
  • Root: Always UID 0
  • Uniqueness: Per system only
  • Reuse: Can be reused after user deletion
ü™ü Windows SID
  • Format: Complex string (S-1-5-21-...)
  • Length: Variable, up to 68 bytes
  • Admin: Ends in -500
  • Uniqueness: Globally unique
  • Reuse: Never reused

Why SIDs Matter

üí° Security Implications:
  • Permissions use SIDs: File and registry permissions are stored using SIDs, not usernames
  • Renaming doesn't break access: If you rename a user, their SID stays the same, so permissions remain intact
  • Deleted users can't be recreated: A new user with the same name gets a different SID and won't have the old user's permissions
  • SID history: Used in migrations to preserve access to resources

Viewing SIDs

You can view SIDs using PowerShell or the command line:

PowerShell: View user SID
# Get SID for current user PS> whoami /user USER INFORMATION ---------------- User Name SID ================ ============================================= server1\jsmith S-1-5-21-3623811015-3361044348-30300820-1013 # Get SID using PowerShell cmdlet PS> (Get-LocalUser -Name "Administrator").SID S-1-5-21-3623811015-3361044348-30300820-500 # Get SID for a specific user PS> Get-LocalUser | Select-Object Name, SID Name SID ---- --- Administrator S-1-5-21-3623811015-3361044348-30300820-500 Guest S-1-5-21-3623811015-3361044348-30300820-501 jsmith S-1-5-21-3623811015-3361044348-30300820-1013
‚ö†Ô∏è SID Persistence: When you delete a user and create a new user with the same name, the new user gets a completely different SID. This means:
  • The new user won't have access to the old user's files
  • File ownership shows the old SID as a number (orphaned SID)
  • You cannot "recreate" a user to restore their access

Built-in Windows Accounts CSLO 1, 2

Overview

Windows Server includes several built-in accounts that are created automatically during installation. Understanding these accounts is essential for security and administration.

User Accounts

üî¥ Administrator
SID Ending: -500
Default Status: Enabled (may be disabled on some editions)
Purpose: Complete system administration access
Cannot be: Deleted or removed from Administrators group
Can be: Renamed, disabled
üîí Security Best Practice: Rename the Administrator account and create a decoy "Administrator" account with no privileges. This helps prevent targeted attacks against the known admin account name.
üë§ Guest
SID Ending: -501
Default Status: Disabled
Purpose: Limited temporary access
Cannot be: Deleted
Recommendation: Keep disabled on servers
‚öôÔ∏è DefaultAccount
SID Ending: -503
Default Status: Disabled
Purpose: System-managed account for Windows features
Managed by: Windows (do not modify)

Service Accounts

Windows includes special accounts used by services and the operating system:

Account SID Purpose Network Access
SYSTEM
(LocalSystem)		 S-1-5-18 Highest privilege local account; used by core OS services Yes (as computer account)
LOCAL SERVICE S-1-5-19 Limited privileges; used by services that don't need network access Anonymous only
NETWORK SERVICE S-1-5-20 Limited privileges; used by services needing network access Yes (as computer account)
üí° Linux Comparison: Windows service accounts (SYSTEM, LOCAL SERVICE, NETWORK SERVICE) are similar to Linux system users like www-data, mysql, or nobody. They run services with specific privilege levels rather than full administrative access.

Built-in Groups

Windows also includes built-in groups that provide specific access levels:

Purpose: Full control over the computer

Default Members: Administrator account

Capabilities:

  • Install software and drivers
  • Modify system settings
  • Create and manage user accounts
  • Access all files (with ownership)
  • Manage services and processes
üîí Security: Limit membership to only those who truly need full administrative access.

Purpose: Standard user access

Default Members: Authenticated Users, INTERACTIVE

Capabilities:

  • Run applications
  • Use local and network printers
  • Shut down and lock the workstation
  • Cannot install software or modify system settings

Purpose: Allow remote desktop connections

Default Members: None

Capabilities:

  • Log on remotely via Remote Desktop
  • Does not grant administrative privileges
üí° Note: Administrators can always connect via RDP; this group is for non-admin users who need remote access.

Purpose: Backup and restore files

Default Members: None

Capabilities:

  • Back up and restore files regardless of permissions
  • Log on locally
  • Shut down the system
‚ö†Ô∏è Security Note: This is a privileged group‚Äîmembers can read any file on the system through backup operations.

Purpose: Legacy group from older Windows versions

Current Status: Retained for compatibility but has no special permissions in modern Windows

Recommendation: Do not use; use Administrators or Users groups instead

Password Options

Windows provides several password-related settings for each account:

Purpose: Force user to set their own password on first login

Use case: New accounts with temporary passwords

Linux equivalent: chage -d 0 username or passwd -e username

‚úì Best Practice: Always enable this for new user accounts to ensure users set their own secure passwords.

Purpose: Prevent user from changing their password

Use case: Service accounts, shared accounts, kiosk systems

Linux equivalent: passwd -n 99999 username (set min days very high)

‚ö†Ô∏è Caution: Generally not recommended for regular user accounts as it prevents users from responding to potential compromises.

Purpose: Exempt account from password expiration policy

Use case: Service accounts (with strong, managed passwords)

Linux equivalent: chage -M 99999 username

‚ö†Ô∏è Security Trade-off: While convenient, non-expiring passwords increase risk if compromised. Use only for service accounts with strong passwords and proper monitoring.

Purpose: Prevent all logins to the account

Use case: Suspended employees, inactive accounts, Guest account

Linux equivalent: usermod -L username or passwd -l username

üí° Note: Disabling preserves the account and all its settings; it can be re-enabled later without data loss.

Introduction

Passwords remain the most common form of authentication, and weak password practices are a leading cause of security breaches. As a system administrator, you must implement and enforce password policies that balance security requirements with user convenience.

This module covers password policy configuration on both Linux and Windows platforms, including password aging, complexity requirements, and account lockout policies. These are critical security controls that protect your systems from unauthorized access.

üîê Why Password Policies Matter

In 2023, over 80% of data breaches involved compromised credentials. Strong password policies are your first line of defense against unauthorized access. Without proper policies, users tend to choose weak passwords like "password123" or reuse passwords across multiple systems.

Password Policy Concepts CSLO 2

The Password Security Triad

Effective password policies must balance three factors:

üîí Security

Passwords must be strong enough to resist attacks:

  • Long enough to prevent brute-force
  • Complex enough to resist dictionary attacks
  • Changed often enough to limit exposure
üë§ Usability

Policies must be practical for users:

  • Passwords should be memorable
  • Change frequency shouldn't be excessive
  • Rules shouldn't be overly complex
‚öôÔ∏è Manageability

Policies must be enforceable:

  • Automated enforcement
  • Consistent across systems
  • Auditable compliance

Password Aging Concepts

üìñ Password Aging refers to the set of rules that govern how long a password can be used and when it must be changed. Key parameters include:
  • Maximum age: How long until password expires
  • Minimum age: How soon after changing can user change again
  • Warning period: Days before expiration to warn user
  • Inactive period: Grace period after expiration

Password Aging Timeline

üìÖ Example Timeline (90-day max, 7-day min, 14-day warning):
  • Day 0: User sets new password
  • Days 1-7: Cannot change password (minimum age)
  • Days 8-76: Can change password voluntarily
  • Days 77-90: Warning messages appear at login
  • Day 91+: Password expired, must change to log in

Modern Password Guidance

Password policy recommendations have evolved significantly. The National Institute of Standards and Technology (NIST) updated their guidelines in Special Publication 800-63B:

‚ùå Old Recommendations
  • Change passwords every 30-90 days
  • Require complex character mix
  • Minimum 8 characters
  • Security questions for recovery
‚úì NIST Current Guidance
  • Change only when compromised
  • Focus on length over complexity
  • Minimum 12-14 characters (or passphrases)
  • Check against breach databases
  • Allow password managers
‚ö†Ô∏è Reality Check: While NIST recommends against mandatory password expiration, many organizations still require it for compliance reasons (PCI-DSS, HIPAA, etc.). As an administrator, you'll need to implement whatever policy your organization requires, even if it differs from current best practices.

Traditional complexity requirements include:

Linux Password Policies CSLO 1, 2

Password Policy Configuration Files

Linux password policies are configured through several files:

File Purpose Scope
/etc/login.defs System-wide defaults for new users New accounts only
/etc/shadow Per-user password aging settings Individual users
/etc/pam.d/* Password complexity and quality All users
/etc/security/pwquality.conf Password quality requirements All users

The chage Command

The chage (change age) command is the primary tool for managing password aging on Linux.

üìñ chage Syntax: chage [OPTIONS] USERNAME

chage Options

Option Description Example
-l List password aging information chage -l jsmith
-M Set maximum days between password changes chage -M 90 jsmith
-m Set minimum days between password changes chage -m 7 jsmith
-W Set warning days before expiration chage -W 14 jsmith
-I Set inactive days after expiration chage -I 30 jsmith
-E Set account expiration date chage -E 2026-12-31 jsmith
-d Set last password change date (0 = force change) chage -d 0 jsmith

Viewing Password Aging Information

Linux: View password aging with chage -l
$ sudo chage -l jsmith Last password change : Feb 15, 2026 Password expires : May 16, 2026 Password inactive : Jun 15, 2026 Account expires : never Minimum number of days between password change : 7 Maximum number of days between password change : 90 Number of days of warning before password expires : 14

Setting Password Aging Policies

Linux: Set comprehensive password aging
# Set maximum password age to 90 days sudo chage -M 90 jsmith # Set minimum password age to 7 days sudo chage -m 7 jsmith # Set warning period to 14 days sudo chage -W 14 jsmith # Set inactive period to 30 days sudo chage -I 30 jsmith # All options in one command sudo chage -M 90 -m 7 -W 14 -I 30 jsmith # Verify settings sudo chage -l jsmith
Linux: Force immediate password change
# Method 1: Using chage sudo chage -d 0 jsmith # Method 2: Using passwd sudo passwd -e jsmith # Verify - Last password change shows "password must be changed" sudo chage -l jsmith Last password change : password must be changed
üí° Use Case: Always force password change for new accounts or after resetting a user's password.
Linux: Set account expiration for contractors
# Set account to expire on specific date sudo chage -E 2026-06-30 contractor1 # Set account to expire in 90 days sudo chage -E $(date -d "+90 days" +%Y-%m-%d) contractor1 # Remove account expiration sudo chage -E -1 contractor1 # Verify sudo chage -l contractor1
‚ö†Ô∏è Note: Account expiration (-E) is different from password expiration (-M). An expired account cannot log in at all, while an expired password forces a password change.

System-Wide Defaults: /etc/login.defs

The /etc/login.defs file sets default values for new user accounts. Changes only affect users created after modification.

Linux: Key settings in /etc/login.defs
# Password aging controls PASS_MAX_DAYS 90 # Maximum days password is valid PASS_MIN_DAYS 7 # Minimum days between changes PASS_WARN_AGE 14 # Days warning before expiration # Password length (minimum) PASS_MIN_LEN 12 # Minimum password length # UID/GID ranges UID_MIN 1000 # Minimum UID for regular users UID_MAX 60000 # Maximum UID for regular users GID_MIN 1000 # Minimum GID for regular groups GID_MAX 60000 # Maximum GID for regular groups # Home directory creation CREATE_HOME yes # Create home directory by default UMASK 077 # Default umask for home directories # Password encryption method ENCRYPT_METHOD SHA512 # Use SHA-512 for password hashing
Linux: Modify login.defs (use sudo and text editor)
# Edit the file sudo nano /etc/login.defs # Or use sed for specific changes sudo sed -i 's/^PASS_MAX_DAYS.*/PASS_MAX_DAYS 90/' /etc/login.defs sudo sed -i 's/^PASS_MIN_DAYS.*/PASS_MIN_DAYS 7/' /etc/login.defs sudo sed -i 's/^PASS_WARN_AGE.*/PASS_WARN_AGE 14/' /etc/login.defs

Password Complexity with PAM

Linux uses PAM (Pluggable Authentication Modules) for password complexity. The pam_pwquality module enforces password quality requirements.

Linux: /etc/security/pwquality.conf
# Minimum password length minlen = 12 # Require at least one digit dcredit = -1 # Require at least one uppercase letter ucredit = -1 # Require at least one lowercase letter lcredit = -1 # Require at least one special character ocredit = -1 # Maximum consecutive identical characters maxrepeat = 3 # Reject passwords containing username usercheck = 1 # Number of character classes required (1-4) minclass = 3 # Reject passwords based on dictionary words dictcheck = 1
üí° Credit Values: Negative values mean "require at least N of this type." For example, dcredit = -1 requires at least 1 digit. Positive values give "credit" (reduce required length) for including that character type.

Windows Password Policies CSLO 1, 2

Local Security Policy

Windows Server uses Local Security Policy (or Group Policy in domain environments) to configure password policies. For standalone servers, use Local Security Policy.

ü™ü Accessing Local Security Policy:
  • Run: secpol.msc
  • Or: Server Manager ‚Üí Tools ‚Üí Local Security Policy
  • Navigate to: Account Policies ‚Üí Password Policy

Password Policy Settings

Policy Setting Description Recommended Value
Enforce password history Number of unique passwords before reuse 24 passwords
Maximum password age Days until password expires 60-90 days
Minimum password age Days before password can be changed 1-7 days
Minimum password length Minimum characters required 12-14 characters
Password must meet complexity Require character variety Enabled
Store passwords using reversible encryption Store passwords in recoverable format Disabled (security risk)

GUI Configuration Steps

  1. Open Local Security Policy: Press Win+R, type secpol.msc, press Enter
  2. Navigate to Password Policy: Expand "Account Policies" ‚Üí Click "Password Policy"
  3. Configure each setting:
    • Double-click "Maximum password age" ‚Üí Set to 90 days ‚Üí OK
    • Double-click "Minimum password age" ‚Üí Set to 1 day ‚Üí OK
    • Double-click "Minimum password length" ‚Üí Set to 12 characters ‚Üí OK
    • Double-click "Password must meet complexity requirements" ‚Üí Enable ‚Üí OK
    • Double-click "Enforce password history" ‚Üí Set to 24 passwords ‚Üí OK
  4. Close the console: Changes take effect immediately for new passwords

Windows Password Complexity Requirements

When "Password must meet complexity requirements" is enabled, passwords must:

üîí Windows Complexity Rules:
  • Not contain the user's account name or parts of full name exceeding 2 consecutive characters
  • Be at least 6 characters long (or the minimum length policy)
  • Contain characters from three of these four categories:
    • Uppercase letters (A-Z)
    • Lowercase letters (a-z)
    • Digits (0-9)
    • Special characters (!@#$%^&*()_+-=[]{}|;:'",.<>?/)

PowerShell Configuration

You can also configure password policies using PowerShell and the secedit command or by modifying the local security database.

PowerShell: View current password policy
# View password policy using net accounts PS> net accounts Force user logoff how long after time expires?: Never Minimum password age (days): 1 Maximum password age (days): 90 Minimum password length: 12 Length of password history maintained: 24 Lockout threshold: 5 Lockout duration (minutes): 30 Lockout observation window (minutes): 30 Computer role: SERVER
PowerShell: Modify password policy using net accounts
# Set maximum password age to 90 days PS> net accounts /maxpwage:90 # Set minimum password age to 1 day PS> net accounts /minpwage:1 # Set minimum password length to 12 characters PS> net accounts /minpwlen:12 # Set password history to remember 24 passwords PS> net accounts /uniquepw:24 # Verify changes PS> net accounts
‚ö†Ô∏è Limitation: The net accounts command cannot enable/disable complexity requirements. Use Local Security Policy GUI or secedit for that setting.

Advanced: Using secedit

For complete policy control including complexity, use secedit:

PowerShell: Export, modify, and import security policy
# Export current security policy to file PS> secedit /export /cfg C:\SecurityPolicy.inf # Edit the file (password policy section) # [System Access] # MinimumPasswordAge = 1 # MaximumPasswordAge = 90 # MinimumPasswordLength = 12 # PasswordComplexity = 1 # PasswordHistorySize = 24 # Import modified policy PS> secedit /configure /db C:\Windows\Security\local.sdb /cfg C:\SecurityPolicy.inf /areas SECURITYPOLICY # Refresh policy PS> gpupdate /force

Account Lockout Policies CSLO 2

Why Account Lockout?

Account lockout policies protect against brute-force password attacks by temporarily disabling accounts after multiple failed login attempts.

üîí Brute-Force Attack Defense:

Without lockout policies, an attacker can try thousands of passwords per second. A lockout policy that locks accounts after 5 failed attempts makes brute-force attacks impractical‚Äîthe attacker would need to wait for the lockout to expire after every 5 guesses.

Lockout Policy Components

Component Description Typical Setting
Lockout Threshold Failed attempts before lockout 3-5 attempts
Lockout Duration How long account stays locked 15-30 minutes (or until admin unlocks)
Reset Counter After Time to reset failed attempt counter 15-30 minutes

Platform-Specific Configuration

Linux Account Lockout with PAM

Linux uses the pam_faillock module (or pam_tally2 on older systems) for account lockout.

Linux: Configure account lockout in /etc/security/faillock.conf
# /etc/security/faillock.conf # Number of failed attempts before lockout deny = 5 # Lockout duration in seconds (600 = 10 minutes) unlock_time = 600 # Time window for counting failures (seconds) fail_interval = 900 # Also lock root account (use with caution!) # even_deny_root # Lockout duration for root (if enabled) # root_unlock_time = 60 # Directory to store failure records dir = /var/run/faillock # Audit failed attempts audit # Don't log failed attempts to syslog (reduce noise) silent
Linux: View and manage locked accounts
# View failed login attempts for a user sudo faillock --user jsmith # Reset failed login counter (unlock account) sudo faillock --user jsmith --reset # View all users with failed attempts sudo faillock
‚ö†Ô∏è Root Account: Be very careful about locking the root account. If root is locked and you can't unlock it, you may need to boot into recovery mode. Consider leaving root unlockable or setting a very short lockout duration.

Windows Account Lockout Policy

Configure via Local Security Policy: Account Policies ‚Üí Account Lockout Policy

Policy Description Recommended
Account lockout threshold Invalid logon attempts before lockout 5 attempts
Account lockout duration Minutes account remains locked 30 minutes
Reset account lockout counter after Minutes until counter resets 30 minutes
PowerShell: Configure account lockout
# View current lockout settings PS> net accounts # Set lockout threshold to 5 attempts PS> net accounts /lockoutthreshold:5 # Set lockout duration to 30 minutes PS> net accounts /lockoutduration:30 # Set lockout window to 30 minutes PS> net accounts /lockoutwindow:30
PowerShell: Unlock a locked account
# Check if account is locked PS> Get-LocalUser -Name "jsmith" | Select-Object Name, Enabled, AccountExpires, LastLogon # Unlock account by enabling it PS> Enable-LocalUser -Name "jsmith" # For domain accounts (preview - covered in Week 11) # Unlock-ADAccount -Identity jsmith
üí° Administrator Account: By default, the built-in Administrator account cannot be locked out. This is a safety feature but also a security consideration‚Äîensure the Administrator account has a very strong password.

Lockout Policy Best Practices

‚úì Recommendations:
  • Threshold: 5 attempts is a good balance‚Äîlow enough to stop attacks, high enough to tolerate typos
  • Duration: 15-30 minutes is usually sufficient; longer durations increase help desk calls
  • Monitoring: Log and alert on lockouts‚Äîthey may indicate an attack
  • Service accounts: Consider excluding service accounts from lockout or using very long lockout thresholds
  • Documentation: Document the unlock procedure for help desk staff
üö® Denial of Service Risk:

Account lockout can be weaponized! An attacker who knows usernames can intentionally lock out accounts by submitting bad passwords. Consider:

  • Using CAPTCHA or rate limiting at the application level
  • Implementing progressive delays instead of hard lockouts
  • Monitoring for mass lockout events

§1 – OpenLDAP Components Overview

OpenLDAP is not a single program—it is a suite of related components. Before you install anything, it helps to know what each piece does and where it runs.

OpenLDAP Component Map
Server-Side

slapd – Standalone LDAP Daemon

  • The LDAP server process
  • Listens on ports 389 (LDAP) and 636 (LDAPS)
  • Handles bind, search, add, modify, delete operations
  • Reads its configuration from cn=config

Server Utilities (slap*)

  • slapcat – Export directory to LDIF (offline backup)
  • slapadd – Bulk-import LDIF (offline, slapd must be stopped)
  • slappasswd – Generate hashed passwords
  • slaptest – Validate configuration syntax
Client-Side

LDAP Client Utilities (ldap*)

  • ldapsearch – Query the directory
  • ldapadd – Add new entries (via LDIF)
  • ldapmodify – Modify existing entries
  • ldapdelete – Remove entries
  • ldapwhoami – Test bind credentials

These tools connect to slapd over the network (or locally) and speak the LDAP protocol. They are installed via the ldap-utils package.

💡 Naming Convention Tip Server-side tools start with slap (from standalone ldap daemon). Client-side tools start with ldap. If you remember this pattern, you will always know which side of the connection a tool operates on.
🔗 Key Distinction: slap* vs. ldap* The slap* utilities (like slapadd and slapcat) work directly on the database files—they bypass the LDAP protocol entirely and require slapd to be stopped. The ldap* utilities (like ldapadd and ldapsearch) communicate through the LDAP protocol and work while slapd is running. In day-to-day administration you will use the ldap* tools; the slap* tools are primarily for bulk operations and disaster recovery.

§2 – Installation on Ubuntu Server

The Ubuntu path is the primary installation method used in this course and in the Proxmox lab environment. The process has two phases: package installation and interactive reconfiguration.

Phase 1 – Install the Packages

Two packages are needed: slapd (the server) and ldap-utils (the client tools).

# Update package lists first
sudo apt update

# Install OpenLDAP server and client utilities
sudo apt install slapd ldap-utils -y

During installation, Ubuntu will prompt you for an administrator password. Enter a strong password and remember it—this becomes the password for the cn=admin entry. However, the initial install uses minimal defaults (it auto-generates a domain based on the system hostname). We will override these defaults in Phase 2.

⚠️ Don't Skip Phase 2 The default installation creates a directory base DN derived from the server's hostname, which is often not what you want. Always run dpkg-reconfigure slapd to set your desired domain name and configuration options.

Phase 2 – Reconfigure with dpkg-reconfigure

The dpkg-reconfigure slapd command walks you through a series of prompts that set the foundational parameters for your LDAP directory. Run it with:

sudo dpkg-reconfigure slapd

Use the step-by-step walkthrough below to understand each prompt. Click any step to reveal its explanation.

dpkg-reconfigure slapd — Prompt-by-Prompt Walkthrough

Answer: No

This is a double-negative question that confuses many administrators. Selecting No means "do not omit the configuration"—in other words, proceed with configuration. If you accidentally select Yes, the installer skips all remaining prompts and leaves the default (usually unwanted) settings in place.

Enter your chosen domain, e.g.: lab.local

This value is converted into the base DN for your directory. For example:

  • lab.local → base DN: dc=lab,dc=local
  • mendocino.edu → base DN: dc=mendocino,dc=edu

Choose a domain name that matches your lab environment or project plan. For in-class labs, use lab.local unless your instructor specifies otherwise.

Enter your organization name, e.g.: CSC152 Lab

This is a human-readable label stored in the directory's root entry. It populates the o (organization) attribute. It does not affect the base DN or directory structure—it is purely descriptive.

Enter a strong password (you will be asked to confirm it)

This sets the password for the cn=admin,dc=lab,dc=local entry—the root administrator of your directory. You will use this password every time you bind as admin with the ldap* tools (e.g., ldapadd -D "cn=admin,dc=lab,dc=local" -W).

⚠️ Document This Password If you forget this password, you will need to reconfigure slapd from scratch. Record it in a secure location.

Answer: No

Selecting No preserves your directory data if you ever uninstall the slapd package. This is the safer choice—you can always manually delete the database files later if needed. In a production environment, you would almost never want package removal to silently destroy your directory.

Answer: Yes

If a previous database exists (from the initial install or a prior configuration), this prompt asks whether to move it to /var/backups before creating the new one. Selecting Yes is safe—it backs up the old data rather than deleting it, and creates a clean database with your new settings.

What Happens Behind the Scenes

When dpkg-reconfigure slapd completes, it:

  1. Creates a new cn=config configuration database (under /etc/ldap/slapd.d/)
  2. Creates the MDB data directory (under /var/lib/ldap/)
  3. Populates the root entry (e.g., dc=lab,dc=local) and the admin entry (cn=admin,dc=lab,dc=local)
  4. Restarts the slapd service with the new configuration
💡 Verify Immediately After reconfiguration, always verify the installation (covered in §6 below) before proceeding to populate the directory. Catching configuration errors now saves significant troubleshooting time later.

§3 – Installation on Rocky Linux (Alternative Path)

If your lab environment uses Rocky Linux (or CentOS/RHEL), expand the section below for the alternative installation procedure.

Rocky Linux / RHEL Installation Steps
📋 Note The Ubuntu path is the primary method for this course. The Rocky Linux path is provided for reference and for students who may encounter RHEL-family systems in the workplace. The concepts (slapd, cn=config, MDB backend) are identical—only the package names and initial configuration method differ.

Step 1 – Install Packages

# Install OpenLDAP server and client packages
sudo dnf install openldap-servers openldap-clients -y

# Start and enable the slapd service
sudo systemctl start slapd
sudo systemctl enable slapd

Step 2 – Set the Admin Password

Rocky Linux does not have a dpkg-reconfigure equivalent. You set the admin password by generating a hash and applying it via LDIF:

# Generate a password hash
slappasswd
New password: ********
Re-enter new password: ********
{SSHA}xxxxxxxxxxxxxxxxxxxxxxxxx

Create a file named chrootpw.ldif:

dn: olcDatabase={0}config,cn=config
changetype: modify
add: olcRootPW
olcRootPW: {SSHA}xxxxxxxxxxxxxxxxxxxxxxxxx

Apply it:

sudo ldapadd -Y EXTERNAL -H ldapi:/// -f chrootpw.ldif

Step 3 – Import Base Schemas

sudo ldapadd -Y EXTERNAL -H ldapi:/// -f /etc/openldap/schema/cosine.ldif
sudo ldapadd -Y EXTERNAL -H ldapi:/// -f /etc/openldap/schema/nis.ldif
sudo ldapadd -Y EXTERNAL -H ldapi:/// -f /etc/openldap/schema/inetorgperson.ldif

Step 4 – Configure the Domain and Backend

Create an LDIF file to set the base DN, admin DN, and MDB backend parameters, then apply it with ldapmodify -Y EXTERNAL. The specific LDIF content mirrors the settings that Ubuntu's dpkg-reconfigure handles automatically.

Step 5 – Verify

Use the same verification steps described in §6 below (systemctl status slapd, ldapsearch for naming contexts, port check).

§4 – Understanding cn=config (OLC – On-Line Configuration)

If you have read older LDAP tutorials, you may have encountered references to a file called slapd.conf. This was the original, static configuration file for OpenLDAP. Modern OpenLDAP replaces it with a dynamic system called cn=config, also known as OLC (On-Line Configuration).

Why cn=config Replaced slapd.conf

Static slapd.conf vs. Dynamic cn=config
Aspect slapd.conf (Legacy) cn=config (Modern / OLC)
Configuration format Single flat text file LDIF entries stored in a directory tree under /etc/ldap/slapd.d/
Applying changes Edit file, then restart slapd Apply LDIF with ldapmodify—changes take effect immediately, no restart needed
Service interruption Every configuration change causes downtime Zero downtime for configuration changes
Replication Cannot be replicated Configuration can be replicated to other servers
Status Deprecated in modern OpenLDAP Default and recommended

How cn=config Is Organized

The configuration is itself an LDAP directory—a small DIT stored on disk as LDIF files. The root of this DIT is cn=config, and it contains entries for global settings, backends, databases, overlays, and schemas.

# Simplified cn=config tree structure
cn=config                          # Global settings
├── cn=module{0},cn=config         # Loaded modules
├── cn=schema,cn=config            # Schema definitions
│   ├── cn={0}core                 # Core schema
│   ├── cn={1}cosine               # COSINE schema
│   ├── cn={2}nis                  # NIS schema (posixAccount, etc.)
│   └── cn={3}inetorgperson        # inetOrgPerson schema
├── olcDatabase={-1}frontend       # Frontend (global) database settings
├── olcDatabase={0}config          # The config database itself
└── olcDatabase={1}mdb             # Your data database (MDB backend)

Viewing the Current Configuration

You can query the cn=config DIT using ldapsearch with SASL EXTERNAL authentication (which uses the Unix socket and requires root privileges):

# View the top-level configuration entries
sudo ldapsearch -Y EXTERNAL -H ldapi:/// -b "cn=config" -LLL -s one dn

To see the full configuration of your MDB database:

# View the MDB database configuration
sudo ldapsearch -Y EXTERNAL -H ldapi:/// \
  -b "olcDatabase={1}mdb,cn=config" -LLL
⚠️ Do Not Edit Files in /etc/ldap/slapd.d/ Directly Although cn=config is stored as LDIF files on disk, you should never edit these files with a text editor while slapd is running. Always use ldapmodify to make changes. Direct file editing can corrupt the configuration and cause slapd to fail on restart.

Making Configuration Changes with ldapmodify

To change a configuration setting, you write a small LDIF file describing the modification and apply it. For example, to change the log level:

# File: loglevel.ldif
dn: cn=config
changetype: modify
replace: olcLogLevel
olcLogLevel: stats
# Apply the change (no restart needed)
sudo ldapmodify -Y EXTERNAL -H ldapi:/// -f loglevel.ldif
💡 Connecting to Module 12.1 Notice that cn=config uses the same LDIF format you learned in Module 12.1 (§8). The dn:, changetype:, and attribute-value syntax are identical. Configuring OpenLDAP is literally "using LDAP to manage LDAP."

§5 – MDB Backend Database

MDB (Memory-Mapped Database, based on LMDB – Lightning Memory-Mapped Database) is the modern default backend for OpenLDAP. When you ran dpkg-reconfigure slapd, it automatically created an MDB database for your directory data.

Why MDB?

  • Performance: MDB uses memory-mapped files for extremely fast read operations—ideal for a directory service that is read-heavy.
  • Reliability: Crash-resistant with copy-on-write semantics; no recovery needed after unclean shutdown.
  • Simplicity: No tuning of cache sizes or transaction logs required (unlike the older BDB/HDB backends).
  • Default: MDB has been the recommended backend since OpenLDAP 2.4 and is the only backend actively maintained.

Database Directory

The MDB data files are stored in /var/lib/ldap/. This directory contains:

ls -la /var/lib/ldap/
total 52
drwx------  2 openldap openldap  4096 Apr 13 10:22 .
drwxr-xr-x 42 root     root      4096 Apr 13 10:20 ..
-rw-------  1 openldap openldap 45056 Apr 13 10:22 data.mdb
-rw-------  1 openldap openldap  8192 Apr 13 10:22 lock.mdb
MDB Database Files
File Purpose
data.mdb The actual directory data (all your entries, attributes, and indexes)
lock.mdb Lock file for concurrent access management
⚠️ Ownership Matters The /var/lib/ldap/ directory and its contents must be owned by the openldap user and group. If ownership is incorrect, slapd will fail to start with a "permission denied" error. If you ever need to restore files or troubleshoot, verify ownership with ls -la /var/lib/ldap/ and correct it with sudo chown -R openldap:openldap /var/lib/ldap/.

Index Configuration

Indexes speed up searches on frequently queried attributes. Without indexes, slapd must scan every entry in the database for each search—fine for 10 entries, but unacceptable for 10,000.

The default configuration typically includes indexes on objectClass and cn. For a directory serving as an authentication source, you should also index uid, uidNumber, gidNumber, and memberUid.

Indexes are configured via cn=config by modifying the olcDbIndex attribute on the MDB database entry:

# File: add-indexes.ldif
dn: olcDatabase={1}mdb,cn=config
changetype: modify
add: olcDbIndex
olcDbIndex: uid eq
-
add: olcDbIndex
olcDbIndex: uidNumber eq
-
add: olcDbIndex
olcDbIndex: gidNumber eq
-
add: olcDbIndex
olcDbIndex: memberUid eq
# Apply the index configuration
sudo ldapmodify -Y EXTERNAL -H ldapi:/// -f add-indexes.ldif
💡 Index Types The eq index type supports equality searches (e.g., (uid=jdoe)). Other types include pres (presence: "does this attribute exist?"), sub (substring: (cn=*doe*)), and approx (approximate match). For authentication use cases, eq indexes on key identifiers are the most important.

§6 – Verifying the Installation

After installation and reconfiguration, you should perform three verification checks before moving on: service status, directory query, and listening ports.

Check 1 – Service Status

sudo systemctl status slapd

You should see output indicating the service is active (running):

● slapd.service - OpenLDAP Server Daemon
     Loaded: loaded (/lib/systemd/system/slapd.service; enabled; ...)
     Active: active (running) since Mon 2026-04-13 10:22:15 UTC; 5min ago
   Main PID: 1234 (slapd)
     Tasks: 4 (limit: 4915)
     Memory: 5.2M
     CGroup: /system.slice/slapd.service
             └─1234 /usr/sbin/slapd -h ldap:/// ldapi:/// -g openldap -u openldap
🔍 What to Look For
  • Active: active (running) – The service is up.
  • enabled – The service will start automatically on boot.
  • The -h ldap:/// ldapi:/// flags show that slapd is listening on both the network LDAP port (389) and the local Unix socket (ldapi).

Check 2 – Query Naming Contexts

This command queries the Root DSE (the "top" of the server's information) to confirm what directory databases are available:

ldapsearch -x -LLL -b "" -s base namingContexts

Expected output:

dn:
namingContexts: dc=lab,dc=local

If you see your base DN listed, the directory was configured correctly. If the output is empty or shows a different base DN, re-run dpkg-reconfigure slapd.

💡 Command Breakdown
  • -x – Use simple authentication (anonymous bind, no password needed for this query)
  • -LLL – Produce clean LDIF output (no comments, no version line)
  • -b "" – Start the search at the root (empty base DN = Root DSE)
  • -s base – Search scope is "base only" (just the Root DSE entry itself)
  • namingContexts – The specific attribute to retrieve

Check 3 – Verify Listening Ports

Confirm that slapd is listening on the expected ports:

sudo ss -tlnp | grep slapd

Expected output:

LISTEN  0  2048  0.0.0.0:389  0.0.0.0:*  users:(("slapd",pid=1234,fd=8))
LISTEN  0  2048     [::]:389     [::]:*  users:(("slapd",pid=1234,fd=9))

This confirms slapd is listening on port 389 on all network interfaces (both IPv4 and IPv6).

🔍 About ss -tlnp
  • -t – Show TCP sockets
  • -l – Show only listening sockets
  • -n – Show port numbers (not service names)
  • -p – Show the process using each socket (requires sudo)

Verification Summary Checklist

✅ Installation Verification Checklist
  • ☐ systemctl status slapd shows active (running)
  • ☐ ldapsearch -x -LLL -b "" -s base namingContexts returns your base DN
  • ☐ ss -tlnp | grep slapd shows port 389 listening
  • ☐ Admin password is documented in a secure location

§7 – Troubleshooting Common Installation Errors

Even with careful configuration, things can go wrong. The table below covers the most common installation issues and how to diagnose and fix them.

Common OpenLDAP Installation Issues
Symptom Likely Cause Diagnostic Steps & Fix
slapd fails to start; systemctl status shows "failed" Permission error on /var/lib/ldap/, or corrupted cn=config files 1. Check logs: sudo journalctl -u slapd -e
2. Verify ownership: ls -la /var/lib/ldap/ (must be openldap:openldap)
3. Fix ownership: sudo chown -R openldap:openldap /var/lib/ldap/
4. If config is corrupted, re-run sudo dpkg-reconfigure slapd
ldapsearch returns "Can't contact LDAP server" slapd is not running, or a firewall is blocking port 389 1. Check service: sudo systemctl status slapd
2. Start if needed: sudo systemctl start slapd
3. Check ports: sudo ss -tlnp | grep 389
4. Check local firewall: sudo ufw status (Ubuntu) or pfSense rules
dpkg-reconfigure slapd fails with database errors Old database files conflict with new configuration 1. Stop slapd: sudo systemctl stop slapd
2. Back up existing data: sudo cp -r /var/lib/ldap/ /var/lib/ldap.bak/
3. Remove old database: sudo rm -rf /var/lib/ldap/*
4. Re-run: sudo dpkg-reconfigure slapd
namingContexts query returns wrong base DN dpkg-reconfigure was not run, or the wrong domain was entered Re-run sudo dpkg-reconfigure slapd with the correct DNS domain name. Select "Yes" when asked to move the old database.
"Invalid credentials" when binding as admin Wrong password or wrong admin DN 1. Verify your admin DN matches your base DN: cn=admin,dc=lab,dc=local
2. Test with: ldapwhoami -x -D "cn=admin,dc=lab,dc=local" -W
3. If password is lost, re-run sudo dpkg-reconfigure slapd to set a new one
💡 The Universal First Step: Check the Logs For almost any slapd issue, your first diagnostic command should be: 
sudo journalctl -u slapd -e --no-pager
The -e flag jumps to the end (most recent entries) and --no-pager outputs everything to the terminal. The error messages in the journal are usually specific enough to identify the problem.

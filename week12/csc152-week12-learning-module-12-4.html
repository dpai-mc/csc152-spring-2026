§1 – Why Centralized Authentication Matters

In Modules 12.2 and 12.3 you built an OpenLDAP server and populated it with users and groups. Right now, those entries sit in the directory but no other machine actually uses them to log people in. This module closes that gap: you will configure a separate Linux client VM to authenticate users against the OpenLDAP server, achieving centralized authentication.

The Account-Sprawl Problem

Without centralized authentication every server maintains its own /etc/passwd, /etc/shadow, and /etc/group files. Consider a small network with five Linux servers and 30 users:

  • 150 local accounts to create and maintain (5 servers × 30 users).
  • Password changes must be repeated on every machine—or they drift out of sync.
  • When an employee leaves, the administrator must remember to disable the account on all five servers.
  • Auditing who has access where becomes a manual, error-prone process.

The Centralized Solution

With LDAP-based centralized authentication, user credentials live in one directory. Every server queries that directory at login time:

  • Single source of truth – create a user once, and they can log in to every configured client.
  • Consistent policies – password complexity and expiration rules are enforced in one place.
  • Simplified deprovisioning – disable or delete the LDAP entry and access is revoked everywhere.
  • Audit trail – the directory server logs every bind attempt, giving you a central record of authentication events.
🔗 Week 11 Connection
Active Directory provides the same centralized authentication benefit for Windows environments using Kerberos + LDAP. This module achieves the equivalent on the Linux side using OpenLDAP + SSSD.

§2 – NSS and PAM Overview

Two Linux subsystems work together to make centralized authentication possible: the Name Service Switch (NSS) and Pluggable Authentication Modules (PAM).

Name Service Switch (NSS)

NSS answers the question: "Where does the system look up user and group information?"

  • Configured in /etc/nsswitch.conf.
  • Maps database names (passwd, group, shadow) to sources such as files (local), sss (SSSD), or ldap (direct LDAP).
  • Sources are tried left to right. Example: passwd: files sss means "check local files first, then ask SSSD."

Pluggable Authentication Modules (PAM)

PAM answers the question: "How does the system verify a user's credentials?"

  • A modular framework—each "module" handles one authentication task (password check, account lockout, home directory creation, etc.).
  • Configuration files live in /etc/pam.d/.
  • When SSSD is installed, it registers pam_sss.so so PAM can delegate credential checks to the LDAP server via SSSD.

Login Flow Diagram: login → PAM → SSSD → LDAP

User types
username + password → login / sshd
(PAM-aware app) → PAM stack
pam_sss.so → SSSD
sssd-ldap → OpenLDAP
Server
  1. Application (SSH, login, su) receives the username and password.
  2. PAM walks its module stack. pam_sss.so hands the credentials to SSSD.
  3. SSSD performs an LDAP bind to the OpenLDAP server using the supplied credentials.
  4. OpenLDAP returns success (bind succeeds → user authenticated) or failure (invalid credentials).
  5. PAM reports the result back to the application, which grants or denies the session.
NSS query:

(posixAccount)

NSS uses a similar path to resolve user/group information (UID, GID, home directory, shell) from the LDAP directory.

💡 Key Insight
NSS handles identity (who is this user?), while PAM handles authentication (is the password correct?). Both must be configured for LDAP login to work.

§3 – Configuring SSSD for LDAP Authentication

SSSD (System Security Services Daemon) is the modern, recommended method for integrating Linux clients with LDAP directories. It provides caching, offline authentication support, and a clean configuration model.

Step 1 – Install SSSD Packages

On the client VM (not the OpenLDAP server), install the SSSD LDAP provider:

# Ubuntu / Debian
sudo apt update
sudo apt install sssd-ldap sssd-tools -y
📦 Package Notes
sssd-ldap provides the LDAP identity and authentication back-end. sssd-tools provides helper utilities such as sss_cache for clearing the SSSD cache during testing.

Step 2 – Create /etc/sssd/sssd.conf

SSSD's main configuration file must be created manually. It must be owned by root with permissions 0600—SSSD refuses to start otherwise.

sudo nano /etc/sssd/sssd.conf

Enter the following configuration (adjust the domain and URI to match your lab):

[sssd]
domains = lab.local
config_file_version = 2
services = nss, pam

[domain/lab.local]
# Identity provider: where to look up user/group info
id_provider = ldap

# Authentication provider: where to verify passwords
auth_provider = ldap

# URI of the OpenLDAP server (use your server's IP or hostname)
ldap_uri = ldap://10.0.1.10

# Base DN for searches (must match your DIT)
ldap_search_base = dc=lab,dc=local

# The default LDAP schema type
ldap_schema = rfc2307

# Do not require TLS for lab environment
# (In production, ALWAYS use ldap_id_use_start_tls = true)
ldap_id_use_start_tls = false

# Cache credentials for offline access (optional)
cache_credentials = true

# Enumerate users so getent passwd shows LDAP users
enumerate = true

Directive-by-Directive Explanation

DirectivePurpose
[sssd] → domainsLists the LDAP domains SSSD should manage. Must match a [domain/…] section name below.
config_file_version = 2Required header; SSSD only accepts version 2.
services = nss, pamTells SSSD to act as a back-end for both NSS (identity) and PAM (authentication).
id_provider = ldapUser/group identity information comes from LDAP.
auth_provider = ldapPassword verification is performed via LDAP bind.
ldap_uriThe LDAP server address. Use ldap:// for unencrypted or ldaps:// for TLS (port 636).
ldap_search_baseThe base DN where SSSD begins searching for user/group entries. Must match your DIT root.
ldap_schema = rfc2307Tells SSSD to expect POSIX attributes (uidNumber, gidNumber, etc.) per RFC 2307—the schema used by posixAccount/posixGroup.
ldap_id_use_start_tlsWhether to upgrade the connection to TLS. Set false for lab; set true in production (Week 14 hardening topic).
cache_credentials = trueCaches passwords locally so users can log in even if the LDAP server is temporarily unreachable.
enumerate = trueAllows getent passwd to list all LDAP users. Useful for testing; may be disabled on large directories for performance.

Step 3 – Set Permissions

sudo chmod 0600 /etc/sssd/sssd.conf
sudo chown root:root /etc/sssd/sssd.conf

SSSD enforces strict permissions. If the file is world-readable, the service will refuse to start and log a "permissions too open" error.

Step 4 – Configure /etc/nsswitch.conf

Tell NSS to consult SSSD for user, group, and shadow lookups:

sudo nano /etc/nsswitch.conf

Find the passwd, group, and shadow lines and add sss:

passwd:         files systemd sss
group:          files systemd sss
shadow:         files sss

The files source is listed first so local accounts (root, service accounts) are always resolved locally. SSSD is consulted only if the user is not found in local files.

Step 5 – Enable Automatic Home Directory Creation

LDAP users will not have a home directory on the client until one is created. PAM can do this automatically:

sudo pam-auth-update --enable mkhomedir

This adds pam_mkhomedir.so to the PAM session stack. When an LDAP user logs in for the first time, their home directory (e.g., /home/jdoe) is created automatically.

Step 6 – Restart and Enable SSSD

sudo systemctl restart sssd
sudo systemctl enable sssd
sudo systemctl status sssd

Verify the service is active (running) with no errors. If it fails, check journalctl -u sssd -e for details.

⚠️ Common Pitfall
If SSSD fails to start with "No domains configured," double-check that the domains = lab.local value in [sssd] exactly matches the section header [domain/lab.local]. Even a trailing space can cause a mismatch.

§4 – Alternative: NSS/PAM Direct Configuration

Expand: libnss-ldapd / libpam-ldapd approach (legacy method)

Before SSSD became the standard, Linux systems used libnss-ldapd and libpam-ldapd with the nslcd daemon to connect directly to LDAP. This approach still works and is occasionally seen in older deployments.

Installation

sudo apt install libnss-ldapd libpam-ldapd -y

The installer will prompt for:

  • LDAP server URI: ldap://10.0.1.10
  • Search base: dc=lab,dc=local
  • NSS databases to configure: select passwd, group, shadow

Configuration Files

/etc/nslcd.conf – the nslcd daemon configuration:

uri ldap://10.0.1.10
base dc=lab,dc=local
binddn cn=admin,dc=lab,dc=local
bindpw (admin_password)

/etc/nsswitch.conf – updated automatically during installation:

passwd:         files ldap
group:          files ldap
shadow:         files ldap

Why SSSD Is Preferred

§5 – Testing Client Authentication

After configuring SSSD, run these verification tests on the client VM to confirm LDAP users can be resolved and authenticated.

Test 1: getent passwd

getent passwd

This lists all users from both local files and SSSD. Scroll to the bottom—you should see your LDAP users with their LDAP-assigned UID numbers:

jdoe:*:10001:10001:John Doe:/home/jdoe:/bin/bash
asmith:*:10002:10002:Alice Smith:/home/asmith:/bin/bash

If LDAP users do not appear, SSSD is not connecting to the LDAP server. Check §6 (Troubleshooting).

Test 2: id <username>

id jdoe

Expected output:

uid=10001(jdoe) gid=10001(jdoe) groups=10001(jdoe),10010(developers)

This confirms NSS is resolving the user's UID, GID, and group memberships from LDAP.

Test 3: SSH Login as an LDAP User

ssh jdoe@localhost

Or from another machine:

ssh jdoe@10.0.1.20    # client VM IP

Enter the LDAP user's password. A successful login confirms both NSS (identity resolution) and PAM (authentication) are working.

Test 4: su - <ldapuser>

su - jdoe

This tests PAM authentication via the su command. You should be prompted for jdoe's LDAP password and then dropped into jdoe's shell.

Test 5: Home Directory Auto-Creation

If this is the first login for the LDAP user, pam_mkhomedir should create /home/jdoe automatically:

ls -la /home/jdoe

You should see a freshly created home directory with default skeleton files (.bashrc, .profile, etc.).

💡 Testing Tip
If you make changes to sssd.conf, always clear the cache before retesting: 
sudo sss_cache -E
sudo systemctl restart sssd
SSSD aggressively caches results, and stale cache entries are a frequent source of confusion during troubleshooting.

§6 – Troubleshooting LDAP Client Integration

When LDAP client authentication does not work on the first attempt, use this table to systematically diagnose the issue.

SymptomLikely CauseDiagnostic Tool / CommandResolution
getent passwd shows no LDAP users SSSD not running, wrong ldap_uri, or wrong ldap_search_base systemctl status sssd
journalctl -u sssd -e		 Verify SSSD is running; confirm URI and base DN match the OpenLDAP server configuration
SSSD fails to start: "permission denied" sssd.conf permissions are not 0600 ls -l /etc/sssd/sssd.conf sudo chmod 0600 /etc/sssd/sssd.conf
sudo chown root:root /etc/sssd/sssd.conf
Connection refused on port 389 slapd not running on server, firewall blocking traffic, or wrong IP From client: ldapwhoami -x -H ldap://10.0.1.10
nc -zv 10.0.1.10 389		 Verify slapd is running on the server; check pfSense / ufw rules; verify IP address
Invalid credentials (password rejected) Password hash mismatch, wrong userPassword in LDAP entry ldapwhoami -x -D "uid=jdoe,ou=People,dc=lab,dc=local" -W Re-set the password with slappasswd and update the entry with ldapmodify
TLS handshake failure TLS enabled in SSSD but not configured on the OpenLDAP server journalctl -u sssd -e (look for TLS errors) Set ldap_id_use_start_tls = false for lab, or configure TLS on the server (Week 14 topic)
User found by getent but cannot log in PAM not configured for SSSD, or pam_sss.so missing from PAM stack grep pam_sss /etc/pam.d/common-auth Run sudo pam-auth-update and ensure SSS is enabled
Login succeeds but no home directory pam_mkhomedir not enabled grep mkhomedir /etc/pam.d/common-session sudo pam-auth-update --enable mkhomedir
Stale data after LDAP changes SSSD cache serving old entries sudo sss_cache -E Clear cache and restart: sudo sss_cache -E && sudo systemctl restart sssd
🔍 General Troubleshooting Strategy
  1. Verify network connectivity – Can the client reach the server? (ping, nc -zv)
  2. Verify LDAP connectivity – Can you query LDAP from the client? (ldapsearch -x -H ldap://server -b "dc=lab,dc=local")
  3. Verify SSSD status – Is the service running without errors? (systemctl status sssd, journalctl -u sssd)
  4. Verify NSS – Does getent passwd show LDAP users?
  5. Verify PAM – Can you authenticate? (su - ldapuser)
Work from the bottom of the stack (network) upward (application). Fix the lowest-level failure first.

§7 – Active Directory vs. OpenLDAP: Structured Comparison

Over the past two weeks you have worked hands-on with both Active Directory (Week 11) and OpenLDAP (Week 12). This section synthesizes that experience into a structured comparison across six key dimensions.

DimensionActive Directory (AD DS)OpenLDAP
Protocol & Standards Implements LDAP v3 but extends it with proprietary features. Tightly integrates Kerberos v5 for authentication and DNS for service discovery. Proprietary Microsoft protocol extensions (e.g., LDAP controls for paging, password modification). Standards-compliant LDAP v3 implementation (RFC 4511). No proprietary extensions. Kerberos can be added separately (MIT KDC or Heimdal) but is not built in.
Schema & Object Classes Rich built-in schema with hundreds of predefined classes: user, computer, group, organizationalUnit, groupPolicyContainer, etc. Schema extensions possible but complex and permanent (cannot be deleted once added). Minimal default schema. Administrator explicitly loads schema files (cosine, inetorgperson, nis) as needed. Uses standard classes: inetOrgPerson, posixAccount, posixGroup. Schema is modular and extensible.
Administration Tooling GUI-centric: Active Directory Users and Computers (ADUC), Group Policy Management Console (GPMC), Active Directory Administrative Center (ADAC). PowerShell AD module for scripting and automation. CLI-centric: ldapadd, ldapmodify, ldapsearch, ldapdelete, slapcat. Configuration via LDIF files. Third-party GUIs available (phpLDAPadmin, Apache Directory Studio) but not built in.
Authentication Integration Kerberos + LDAP provide seamless single sign-on (SSO) for Windows domain-joined machines. NTLM as legacy fallback. Domain join is a one-step process. LDAP simple bind for authentication. Kerberos SSO requires a separate KDC setup. Linux client integration via SSSD, NSS/PAM. No built-in "domain join" concept—client configuration is manual.
Enterprise Use Cases Windows-dominant environments. Organizations with Microsoft 365 / Azure AD integration. Enterprises needing Group Policy for centralized desktop management. Requires Windows Server licensing. Linux/Unix environments. Cost-sensitive organizations (open source, no licensing fees). Embedded systems and appliances. Cloud-native environments using LDAP for service authentication. Academic and research institutions.
Replication & Scalability Multi-master replication built in. All domain controllers are writable. Global Catalog servers provide forest-wide search. Sites and site links control replication topology. Scales to millions of objects. Supports syncrepl for replication (provider–consumer and multi-master modes). Configuration is manual and requires more planning. Scales well for read-heavy workloads; write scaling requires careful replication design.

When to Choose Which?

Expand: Decision guidance
  • Choose AD when your environment is primarily Windows, you need Group Policy, you already have Microsoft licensing, or you need tight integration with Azure AD / Microsoft 365.
  • Choose OpenLDAP when your environment is primarily Linux/Unix, licensing cost is a concern, you need a lightweight directory for application authentication, or you need maximum flexibility in schema design.
  • Choose both (hybrid) when your organization has a mixed Windows/Linux environment and needs a single identity source—see §8.

§8 – Hybrid Environments: AD + LDAP Together

Many real-world organizations run both Active Directory and LDAP-based directories. Several integration patterns exist:

Pattern 1: SSSD with AD Back-End

Linux clients can use SSSD with id_provider = ad to authenticate directly against Active Directory—no OpenLDAP needed. AD acts as the single directory, and Linux machines join the AD domain using realm join.

Pattern 2: LDAP Proxy / Pass-Through

An OpenLDAP server can be configured as a proxy that forwards authentication requests to AD. Applications that only speak standard LDAP connect to OpenLDAP, which translates and relays to AD behind the scenes.

Pattern 3: Directory Synchronization

Tools like Fedora Directory Server (389 DS), Microsoft Identity Manager (MIM), or open-source solutions like LSC (LDAP Synchronization Connector) synchronize user entries between AD and OpenLDAP. Changes in one directory are propagated to the other on a schedule.

Pattern 4: FreeIPA as a Bridge

FreeIPA (Identity, Policy, Audit) is a Linux identity management system that integrates LDAP (389 DS), Kerberos, DNS, and certificate management. FreeIPA can establish a cross-realm trust with Active Directory, allowing AD users to access Linux resources and vice versa without duplicating accounts.

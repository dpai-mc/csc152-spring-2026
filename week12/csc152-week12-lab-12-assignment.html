Part A – OpenLDAP Installation & Configuration (6 pts)

Objectives

  • Install the OpenLDAP server (slapd) and LDAP utilities on Ubuntu Server.
  • Reconfigure slapd to set a custom base DN, organization name, and admin password.
  • Verify the LDAP service is running and the base DIT is accessible.
Scenario

Your organization, Example Corp (domain: example.com), needs a centralized directory service for user and group management. You have been tasked with deploying OpenLDAP on an Ubuntu Server and verifying the initial Directory Information Tree (DIT).

1 Update Packages & Install OpenLDAP

Begin by updating your package lists and installing the OpenLDAP server daemon and client utilities:

sudo apt update && sudo apt upgrade -y
sudo apt install -y slapd ldap-utils

During installation you will be prompted to set an administrator password. Choose a strong password and record it securely — you will need it throughout this lab.

2 Reconfigure slapd

The default installation uses a minimal configuration. Reconfigure slapd to set proper values:

sudo dpkg-reconfigure slapd

When prompted, supply the following values:

PromptValue to Enter
Omit OpenLDAP server configuration?No
DNS domain nameexample.com
Organization nameExample Corp
Administrator password(your chosen password)
Database backendMDB
Remove database when slapd is purged?No
Move old database?Yes

3 Verify the Service

Confirm that slapd is running and listening on port 389:

sudo systemctl status slapd
sudo ss -tlnp | grep 389

Then verify the base DIT was created correctly:

ldapsearch -x -LLL -H ldap:/// -b "dc=example,dc=com" dn

You should see at least the base entry dc=example,dc=com and the admin entry.

Hints & Tips – Part A
  • If you need to re-run the reconfiguration at any time, simply use sudo dpkg-reconfigure slapd again.
  • If ldapsearch returns No such object (32), double-check your base DN spelling.
  • Use slapcat as root to dump the raw database and verify entries if ldapsearch gives unexpected results.

Part B – Populating the LDAP Directory (8 pts)

Objectives

  • Design a directory structure with Organizational Units (OUs) for users and groups.
  • Write LDIF files that define OUs, user entries (posixAccount + inetOrgPerson), and group entries (posixGroup).
  • Load the LDIF files into the directory using ldapadd.

1 Plan Your Directory Structure

Create the following DIT hierarchy under dc=example,dc=com:

dc=example,dc=com
├── ou=People
│   ├── uid=jdoe    (John Doe)
│   ├── uid=asmith  (Alice Smith)
│   └── uid=bwhite  (Bob White)
└── ou=Groups
    ├── cn=developers
    └── cn=sysadmins

2 Create the OUs LDIF File

Create a file named base-ous.ldif with the following content:

dn: ou=People,dc=example,dc=com
objectClass: organizationalUnit
ou: People

dn: ou=Groups,dc=example,dc=com
objectClass: organizationalUnit
ou: Groups

Load it into the directory:

ldapadd -x -D "cn=admin,dc=example,dc=com" -W -f base-ous.ldif

3 Create the Users LDIF File

Create a file named users.ldif. Each user entry must include the inetOrgPerson and posixAccount object classes. Use UIDs starting at 10001. Example for the first user:

dn: uid=jdoe,ou=People,dc=example,dc=com
objectClass: inetOrgPerson
objectClass: posixAccount
objectClass: shadowAccount
uid: jdoe
sn: Doe
givenName: John
cn: John Doe
displayName: John Doe
uidNumber: 10001
gidNumber: 10001
userPassword: {SSHA}placeholder
gecos: John Doe
loginShell: /bin/bash
homeDirectory: /home/jdoe

Create similar entries for asmith (UID 10002) and bwhite (UID 10003). You can generate SSHA passwords with:

slappasswd -s yourpassword

Then replace {SSHA}placeholder with the generated hash. Load the file:

ldapadd -x -D "cn=admin,dc=example,dc=com" -W -f users.ldif

4 Create the Groups LDIF File

Create a file named groups.ldif:

dn: cn=developers,ou=Groups,dc=example,dc=com
objectClass: posixGroup
cn: developers
gidNumber: 20001
memberUid: jdoe
memberUid: asmith

dn: cn=sysadmins,ou=Groups,dc=example,dc=com
objectClass: posixGroup
cn: sysadmins
gidNumber: 20002
memberUid: bwhite

Load the file:

ldapadd -x -D "cn=admin,dc=example,dc=com" -W -f groups.ldif
Hints & Tips – Part B
  • LDIF files are sensitive to blank lines — use exactly one blank line between entries and no trailing blank lines.
  • If you receive Already exists (68), the entry was loaded previously. Use ldapdelete to remove it first, or skip re-adding.
  • The shadowAccount object class is needed for password aging attributes used by PAM/SSSD.
  • Make sure uidNumber and gidNumber values do not conflict with local system accounts (stay above 10000).

Part C – Querying & Modifying the Directory (6 pts)

Objectives

  • Construct and execute at least five distinct ldapsearch queries against the directory.
  • Modify an existing user attribute using ldapmodify.
  • Delete a test entry using ldapdelete.

1 Query – List All Users

Return all entries under ou=People:

ldapsearch -x -LLL -H ldap:/// -b "ou=People,dc=example,dc=com" "(objectClass=posixAccount)" uid cn uidNumber

2 Query – Specific User by uid

Find the entry for jdoe:

ldapsearch -x -LLL -H ldap:/// -b "dc=example,dc=com" "(uid=jdoe)"

3 Query – List All Groups

Return all posixGroup entries:

ldapsearch -x -LLL -H ldap:/// -b "ou=Groups,dc=example,dc=com" "(objectClass=posixGroup)" cn gidNumber memberUid

4 Query – Compound Filter

Find all users whose uidNumber is greater than or equal to 10002:

ldapsearch -x -LLL -H ldap:/// -b "ou=People,dc=example,dc=com" "(&(objectClass=posixAccount)(uidNumber>=10002))" uid cn

5 Query – Your Custom Query

Design your own ldapsearch query. Examples: search by sn, search for users whose login shell is /bin/bash, or search for groups that contain a specific memberUid. Document what you are searching for and why.

6 Modify a User Attribute

Change the loginShell for user asmith from /bin/bash to /bin/zsh. Create a file named modify-shell.ldif:

dn: uid=asmith,ou=People,dc=example,dc=com
changetype: modify
replace: loginShell
loginShell: /bin/zsh

Apply the modification:

ldapmodify -x -D "cn=admin,dc=example,dc=com" -W -f modify-shell.ldif

Verify the change with an ldapsearch for asmith.

7 Delete a Test Entry

Delete the user bwhite (you will re-add this user later if needed for Part D testing):

ldapdelete -x -D "cn=admin,dc=example,dc=com" -W "uid=bwhite,ou=People,dc=example,dc=com"

Verify the deletion by searching for bwhite:

ldapsearch -x -LLL -H ldap:/// -b "dc=example,dc=com" "(uid=bwhite)"
Re-add bwhite

If you need bwhite for Part D testing, re-add the entry using ldapadd with the original LDIF or a new file containing just that user's entry.

Hints & Tips – Part C
  • The -LLL flag suppresses LDIF version info and comments for cleaner output.
  • LDAP filters use prefix notation: (&(filter1)(filter2)) for AND, (|(filter1)(filter2)) for OR.
  • If a search returns nothing, check your base DN and filter syntax carefully.
  • Use -D and -W flags when you need authenticated (write) access; anonymous reads work for search if your ACLs allow it.

Part D – LDAP Client Authentication (6 pts)

Objectives

  • Install and configure SSSD on a second Ubuntu VM to authenticate against the OpenLDAP server.
  • Verify that LDAP users are visible via getent and id.
  • Test SSH login as an LDAP user and confirm automatic home directory creation with pam_mkhomedir.
Before You Begin

Ensure your LDAP server VM is reachable from the client VM. Test with ping and ldapsearch from the client, pointing at the server's IP address. Make sure user bwhite exists in the directory (re-add if you deleted it in Part C).

1 Install SSSD and Required Packages on the Client VM

sudo apt update
sudo apt install -y sssd sssd-ldap ldap-utils

2 Configure SSSD

Create or edit /etc/sssd/sssd.conf with the following content (replace SERVER_IP with your LDAP server's actual IP address):

[sssd]
services = nss, pam
domains = example.com

[domain/example.com]
id_provider = ldap
auth_provider = ldap
ldap_uri = ldap://SERVER_IP
ldap_search_base = dc=example,dc=com
ldap_default_bind_dn = cn=admin,dc=example,dc=com
ldap_default_authtok = your_admin_password
ldap_id_use_start_tls = false
cache_credentials = true
enumerate = true

Set proper permissions and restart SSSD:

sudo chmod 600 /etc/sssd/sssd.conf
sudo chown root:root /etc/sssd/sssd.conf
sudo systemctl restart sssd
sudo systemctl enable sssd

3 Enable pam_mkhomedir

This ensures a home directory is automatically created on first login for LDAP users:

sudo pam-auth-update --enable mkhomedir

4 Test – getent and id

Verify LDAP users are visible to the system's name service:

getent passwd jdoe
getent passwd asmith
id jdoe
id asmith

You should see LDAP-sourced entries with the uidNumber and gidNumber you configured.

5 Test – SSH Login

From the client VM (or from another machine that can reach the client), attempt to SSH in as an LDAP user:

ssh jdoe@CLIENT_IP

After logging in, verify the home directory was created and check the user context:

pwd
whoami
id
Hints & Tips – Part D
  • If getent passwd jdoe returns nothing, check: Is SSSD running? (sudo systemctl status sssd) Are there errors in /var/log/sssd/sssd_example.com.log?
  • Common issues: wrong ldap_uri, typo in ldap_search_base, or sssd.conf permissions not set to 600.
  • If SSH prompts for a password but rejects it, verify the user's userPassword hash in LDAP is correct. You can reset it with ldappasswd.
  • You can clear the SSSD cache with sudo sss_cache -E and restart the service to force a fresh lookup.
  • Ensure SSH is installed and running on the client: sudo apt install openssh-server && sudo systemctl enable --now ssh.

Part E – Splunk Enterprise Installation (4 pts)

Objectives

  • Download and install Splunk Enterprise on your Ubuntu Server VM.
  • Access the Splunk web interface and complete initial setup.
  • Enable the receiving port (9997) for future forwarder connections.
  • Configure pfSense firewall rules to allow Splunk traffic.
Why Splunk Now?

Installing Splunk Enterprise in this lab prepares your environment for log aggregation and monitoring tasks in upcoming projects. Getting it installed and verified now means you can focus on configuration and data ingestion later.

1 Download and Install Splunk Enterprise

Download the Splunk Enterprise .deb package from the Splunk website (you will need your Splunk account). Transfer it to your Ubuntu Server VM, then install:

# Example — adjust the filename to match the version you downloaded
sudo dpkg -i splunk-9.x.x-xxxxxxxx-linux-2.6-amd64.deb

Splunk installs to /opt/splunk by default.

2 Start Splunk and Accept the License

sudo /opt/splunk/bin/splunk start --accept-license --answer-yes

You will be prompted to create an admin username and password for the Splunk web UI. Record these credentials securely.

Enable Splunk to start at boot:

sudo /opt/splunk/bin/splunk enable boot-start

3 Access the Splunk Web UI

Open a browser and navigate to:

http://SERVER_IP:8000

Log in with the admin credentials you created. Confirm the dashboard loads successfully.

4 Enable Receiving Port 9997

Configure Splunk to listen on port 9997 for data from Universal Forwarders:

sudo /opt/splunk/bin/splunk enable listen 9997

Verify the port is listening:

sudo ss -tlnp | grep 9997

5 Configure pfSense Firewall Rules

On your pfSense firewall, create rules to allow the following traffic to/from the Splunk server:

PortProtocolPurpose
8000TCPSplunk Web UI
9997TCPSplunk Receiving (Forwarder data)
8089TCPSplunk Management Port

Test connectivity by accessing the Splunk web UI from a machine on a different network segment (through pfSense).

6 Verification Checklist

Confirm each item below. For your submission, indicate Yes or No for each and add any relevant notes.

  • Splunk Enterprise is installed at /opt/splunk
  • Splunk service starts successfully and is enabled at boot
  • Splunk Web UI is accessible at http://SERVER_IP:8000
  • Receiving port 9997 is enabled and listening
  • pfSense rules allow traffic on ports 8000, 9997, and 8089
  • Web UI is reachable through the pfSense firewall from another network segment
Hints & Tips – Part E
  • Splunk Enterprise requires a minimum of 2 GB RAM. If your VM has less, consider increasing it before installation.
  • If the web UI does not load, check that port 8000 is not blocked by ufw on the server itself: sudo ufw allow 8000/tcp.
  • The free Splunk Enterprise license allows up to 500 MB of data ingestion per day, which is sufficient for this course.
  • If enable listen prompts for credentials, use the Splunk admin credentials you set during first start.

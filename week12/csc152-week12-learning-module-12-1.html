Building on Week 11 In Week 11 you installed Active Directory Domain Services and worked with domains, OUs, users, and Group Policy. This week we explore the open-standards counterpart—LDAP and OpenLDAP. Many concepts (hierarchical structure, OUs, centralized authentication) will feel familiar. The key shift is from a GUI-centric, proprietary platform to a CLI-centric, standards-based one.

§1 – What Is a Directory Service?

A directory service is a specialized database designed to store, organize, and provide access to information about network resources—users, groups, computers, printers, and more. Think of it as a combination of:

  • A phone book—you look up a person’s name and get their phone number, email, and office location.
  • A security guard—it verifies who you are (authentication) and what you’re allowed to do (authorization).
  • A policy enforcer—it can push configuration rules to every computer in the organization.

Directory Service vs. Relational Database

Although both store data, they are optimized for very different workloads:

Comparison of directory services and relational databases
CharacteristicDirectory ServiceRelational Database
Data modelHierarchical (tree)Tabular (rows & columns)
Optimized forReads / lookups (read-heavy)Reads and writes (balanced or write-heavy)
SchemaObject-class based; enforced by serverTable-based; enforced by constraints
Query languageLDAP search filtersSQL
Typical dataUsers, groups, devices, certificatesOrders, inventory, transactions
ReplicationBuilt-in multi-site replicationVaries by product
Week 11 Connection Active Directory is Microsoft’s directory service. It uses the LDAP protocol internally. This week, you will work with LDAP directly—without the AD wrapper.

§2 – From X.500 DAP to LDAP

LDAP did not appear out of thin air. It evolved from an earlier, more complex standard:

X.500 Directory Access Protocol (DAP)

  • Developed in the late 1980s by the ITU-T and ISO.
  • Ran over the full OSI protocol stack—heavyweight, complex, and expensive to implement.
  • Defined the concepts we still use today: hierarchical tree, distinguished names, object classes, and attributes.

LDAP – The Lightweight Alternative

  • Created in the early 1990s at the University of Michigan as a simpler front-end to X.500.
  • Runs over TCP/IP instead of the OSI stack—hence “Lightweight.”
  • Current version: LDAP v3, defined in RFC 4511 (2006).
  • LDAP v2 is deprecated and should not be used.

The key takeaway: LDAP kept the powerful data model of X.500 but made it practical by running over the same TCP/IP networks you already manage.

§3 – LDAP Operation Model

LDAP defines a small set of operations that clients use to interact with the directory server. Every LDAP session follows a predictable pattern: connect, authenticate, perform operations, disconnect.

LDAP operations summary
OperationPurposeAnalogy
BindAuthenticate to the directory (provide credentials)Logging in
SearchQuery the directory for entries matching a filterLooking up a contact
AddCreate a new entry in the directoryAdding a new employee record
ModifyChange attributes of an existing entryUpdating a phone number
DeleteRemove an entry from the directoryRemoving a departed employee
CompareCheck whether an entry has a specific attribute value (returns true/false)Asking “Is this person in the Admins group?”
UnbindEnd the LDAP session and close the connectionLogging out

Request–Response Sequence

  LDAP Client                          LDAP Server (slapd)
      │                                       │
      │──── Bind Request (DN + password) ────▶│
      │◀─── Bind Response (success/fail) ─────│
      │                                       │
      │──── Search Request (base, scope,      │
      │     filter, attributes)          ────▶│
      │◀─── Search Result Entry ──────────────│
      │◀─── Search Result Entry ──────────────│
      │◀─── Search Result Done ───────────────│
      │                                       │
      │──── Modify Request ──────────────────▶│
      │◀─── Modify Response ─────────────────│
      │                                       │
      │──── Unbind Request ─────────────────▶│
      │              (connection closed)       │
Figure 2 – Typical LDAP session: Bind → Search → Modify → Unbind
Bind Types: Simple vs. SASL
  • Simple Bind: The client sends a DN and a plaintext password. This is the most common method in lab environments. Always use TLS when using simple bind in production!
  • SASL Bind: The client uses the Simple Authentication and Security Layer framework, which supports mechanisms like GSSAPI (Kerberos), DIGEST-MD5, and EXTERNAL (TLS client certificates). SASL is more secure but more complex to configure.
Anonymous Bind

An anonymous bind sends no credentials at all. The server may allow limited read access to public attributes (like a phone directory). Many production servers disable anonymous binds for security. In OpenLDAP, you control this through Access Control Lists (ACLs).

§4 – LDAP Ports & Transport Security

LDAP ports and transport security
PortProtocolEncryptionNotes
389LDAPNone (plaintext) or StartTLSDefault LDAP port. Can be upgraded to TLS mid-session with StartTLS.
636LDAPSTLS from connection startLDAP over TLS. The entire session is encrypted. Sometimes called “LDAP over SSL” (legacy term).

StartTLS vs. LDAPS

  • StartTLS – The client connects on port 389 in plaintext, then issues a StartTLS command to upgrade the connection to TLS. If the upgrade fails, the client can decide whether to continue unencrypted or abort.
  • LDAPS – The client connects on port 636 and the TLS handshake happens immediately, before any LDAP traffic. The connection is encrypted from the first byte.
Why Encryption Matters Simple bind sends passwords in cleartext. Without TLS, anyone capturing network traffic between the client and server can read those passwords. In production environments, always use either StartTLS or LDAPS. We will configure TLS for LDAP in Week 14 (Server Security & Hardening). For this week’s lab work on an isolated network, plaintext LDAP on port 389 is acceptable.
Week 8–9 Connection Remember your network configuration and firewall work? If your LDAP server and client are on different pfSense subnets, you will need a firewall rule allowing TCP port 389 (and/or 636) between those subnets.

§5 – The Directory Information Tree (DIT)

Every LDAP directory organizes its data in a hierarchical tree structure called the Directory Information Tree (DIT). If you are familiar with a file system (Week 3) or Active Directory’s domain/OU structure (Week 11), the DIT will feel natural.

Key Components

  • Root DSE (Directory Server Entry) – The very top of the tree. It is a special entry with no DN (an empty string). It describes the server’s capabilities, supported LDAP versions, and naming contexts. You queried it in the verification step: ldapsearch -x -LLL -b "" -s base namingContexts.
  • Base DN (Base Distinguished Name) – The top of your directory data. For example: dc=mendocino,dc=edu. This is analogous to the root domain in Active Directory.
  • Organizational Units (OUs) – Containers that group related entries. Common OUs include ou=People, ou=Groups, and ou=Services. These work the same way as AD OUs from Week 11.
  • Leaf Entries – The actual objects: users, groups, devices. They sit at the bottom of the tree.

Sample DIT Diagram

                    Root DSE ("")
                        │
               dc=mendocino,dc=edu          ← Base DN
              ┌─────────┼──────────┐
              │         │          │
         ou=People  ou=Groups  ou=Services
           │   │        │          │
          ▼   ▼        ▼          ▼
     uid=jdoe uid=asmith cn=staff  cn=ldap-proxy
     (user)   (user)   (group)    (service acct)
Figure 3 – A simple LDAP Directory Information Tree

ConceptActive Directory (Week 11)LDAP / OpenLDAP (Week 12)

§6 – Distinguished Names (DNs) & Relative Distinguished Names (RDNs)

Every entry in the DIT has a globally unique identifier called its Distinguished Name (DN). The DN describes the full path from the entry up to the base of the tree—similar to an absolute file path in Linux (Week 3).

DN Syntax

uid=jdoe,ou=People,dc=mendocino,dc=edu

Read this left to right, from the specific entry up to the base:

  • uid=jdoe – the entry itself (the user “jdoe”)
  • ou=People – the OU containing this user
  • dc=mendocino,dc=edu – the base DN (domain components)

Relative Distinguished Name (RDN)

The RDN is the leftmost component of the DN. It uniquely identifies the entry within its parent container.

  • For uid=jdoe,ou=People,dc=mendocino,dc=edu, the RDN is uid=jdoe.
  • For ou=People,dc=mendocino,dc=edu, the RDN is ou=People.

Analogy: DN as a File Path

  Linux file path:    /home/people/jdoe
                       ──── ────── ────
                       base  dir   file   (read left → right)

  LDAP DN:            uid=jdoe, ou=People, dc=mendocino,dc=edu
                      ──────── ────────── ─────────────────────
                       entry    parent OU   base DN   (read left → right = specific → general)
Figure 4 – Comparing LDAP DNs to Linux file paths (note: order is reversed)
Common DN Attribute Types
  • dc = Domain Component (e.g., dc=mendocino)
  • ou = Organizational Unit (e.g., ou=People)
  • uid = User ID (e.g., uid=jdoe)
  • cn = Common Name (e.g., cn=staff for a group)

§7 – Object Classes, Attributes & Schema

Every LDAP entry must belong to one or more object classes. An object class defines which attributes an entry must have (required) and which it may have (optional). The LDAP server enforces these rules—this is called schema enforcement.

Structural vs. Auxiliary Object Classes

TypeDescriptionRule
StructuralDefines the fundamental nature of the entry (what kind of thing it is)Every entry must have exactly one structural class (inherited chains count as one)
AuxiliaryAdds extra attributes to an entry without changing its fundamental typeAn entry may have zero or more auxiliary classes

Common Object Classes for This Course

organizationalUnit – Container for grouping entries

Type: Structural

Purpose: Represents an OU in the DIT (like an AD OU).

AttributeRequired / OptionalDescription
ouRequiredName of the organizational unit
descriptionOptionalHuman-readable description
inetOrgPerson – A person in the organization

Type: Structural (inherits from organizationalPerson → person)

Purpose: Stores standard identity information for a person.

AttributeRequired / OptionalDescription
cnRequiredCommon name (full name)
snRequiredSurname (last name)
uidOptionalUser login ID
mailOptionalEmail address
telephoneNumberOptionalPhone number
userPasswordOptionalHashed password
posixAccount – Linux/UNIX account attributes (Auxiliary)

Type: Auxiliary

Purpose: Adds UNIX account attributes so the LDAP entry can be used for Linux login. This is the LDAP equivalent of an /etc/passwd entry (Week 6 connection).

AttributeRequired / OptionalDescription
uidRequiredLogin username
uidNumberRequiredNumeric user ID (like Linux UID)
gidNumberRequiredPrimary group ID (like Linux GID)
homeDirectoryRequiredPath to home directory (e.g., /home/jdoe)
loginShellOptionalDefault shell (e.g., /bin/bash)
cnRequiredCommon name
Week 6 Callback Remember useradd -m -s /bin/bash -u 1001 -g 1001 jdoe? Those same attributes (UID, GID, home directory, shell) are now stored in LDAP instead of /etc/passwd.
posixGroup – Linux/UNIX group (Structural)

Type: Structural

Purpose: Represents a UNIX group. Equivalent to an /etc/group entry.

AttributeRequired / OptionalDescription
cnRequiredGroup name
gidNumberRequiredNumeric group ID
memberUidOptionalUsername(s) of group members (multi-valued)

How Object Classes Work Together

A typical LDAP user entry for Linux authentication combines a structural class with an auxiliary class:

objectClass: inetOrgPerson      ← structural (identity info)
objectClass: posixAccount        ← auxiliary  (Linux account info)

The server merges the required and optional attributes from both classes. If any required attribute is missing, the server rejects the entry.

§8 – LDIF – LDAP Data Interchange Format

LDIF (LDAP Data Interchange Format) is the standard text format for representing LDAP entries and changes. You will write LDIF files to add, modify, and delete entries in your OpenLDAP directory. Think of LDIF as the “language” you use to talk to the directory when you are not using a GUI.

LDIF Syntax Rules

  1. Each entry starts with a dn: line specifying the Distinguished Name.
  2. Attributes follow as attributeName: value pairs, one per line.
  3. Entries are separated by a blank line.
  4. Lines beginning with # are comments.
  5. Long lines can be wrapped by starting the continuation line with a single space.

LDIF for Adding Entries

The following LDIF file creates an OU and a user:

# Create the People OU
dn: ou=People,dc=mendocino,dc=edu
objectClass: organizationalUnit
ou: People

# Create user jdoe
dn: uid=jdoe,ou=People,dc=mendocino,dc=edu
objectClass: inetOrgPerson
objectClass: posixAccount
uid: jdoe
cn: Jane Doe
sn: Doe
uidNumber: 1001
gidNumber: 1001
homeDirectory: /home/jdoe
loginShell: /bin/bash
mail: jdoe@mendocino.edu
userPassword: {SSHA}hashed_password_here
Generating Hashed Passwords Never put plaintext passwords in LDIF files. Use the slappasswd command to generate a salted hash: 
$ slappasswd
New password: ********
Re-enter new password: ********
{SSHA}W3ipHbBxOzK8rYbAqsIZdOmNOPkCczSy
Copy the output (including the {SSHA} prefix) into your LDIF file as the userPassword value.

To load this file into the directory:

ldapadd -x -D "cn=admin,dc=mendocino,dc=edu" -W -f users.ldif
  • -x – use simple authentication
  • -D – bind DN (the admin account)
  • -W – prompt for password
  • -f – read entries from a file

LDIF for Modifying Entries

Modifications use changetype: modify with add, replace, or delete operations:

# Change Jane Doe's phone number
dn: uid=jdoe,ou=People,dc=mendocino,dc=edu
changetype: modify
replace: telephoneNumber
telephoneNumber: 707-555-1234

Load with ldapmodify:

ldapmodify -x -D "cn=admin,dc=mendocino,dc=edu" -W -f modify.ldif

LDIF for Deleting Entries

# Delete user jdoe
dn: uid=jdoe,ou=People,dc=mendocino,dc=edu
changetype: delete

Or use the command directly:

ldapdelete -x -D "cn=admin,dc=mendocino,dc=edu" -W "uid=jdoe,ou=People,dc=mendocino,dc=edu"
Blank Lines Matter! In LDIF, a blank line signals the end of one entry and the start of the next. An accidental blank line in the middle of an entry will cause a parsing error. Always double-check your LDIF files before loading them.

§9 – Mini-Case: Designing a DIT for Mendocino Widgets, Inc.

Scenario

Mendocino Widgets, Inc. is a small manufacturing company with 60 employees in three departments:

  • Engineering (25 employees) – designs and tests widgets
  • Sales (20 employees) – manages customer accounts and orders
  • Administration (15 employees) – handles HR, finance, and office management

The company also has three shared Linux servers that need centralized authentication. The IT team has chosen the domain widgets.local for their LDAP directory.

Your Task

On paper or in a text editor, design a DIT for Mendocino Widgets, Inc. Your design should include:

  1. The base DN
  2. At least three OUs that logically organize the company’s users and groups
  3. Sample DNs for two users and one group

Take a few minutes to sketch your tree before revealing the suggested solution below.

Reveal Suggested Solution

Recommended DIT Structure

                 dc=widgets,dc=local             ← Base DN
              ┌──────────┼──────────┐
              │          │          │
         ou=People   ou=Groups  ou=Services
         ┌────┼────┐     │
         │    │    │     │
  ou=Eng ou=Sales ou=Admin  cn=engineering
  │       │       │         cn=sales
  │       │       │         cn=admin-staff
uid=mchen uid=jpark uid=lnguyen
Figure 5 – Suggested DIT for Mendocino Widgets, Inc.

Sample DNs

  • User: uid=mchen,ou=Engineering,ou=People,dc=widgets,dc=local
  • User: uid=jpark,ou=Sales,ou=People,dc=widgets,dc=local
  • Group: cn=engineering,ou=Groups,dc=widgets,dc=local

Design Notes

  • ou=People with department sub-OUs keeps users organized and allows different access policies per department.
  • ou=Groups separates group entries from user entries, making searches cleaner.
  • ou=Services is a good practice for service accounts (like an LDAP proxy account) that are not real people.
  • An alternative valid design places all users directly under ou=People (flat structure) and uses group membership to indicate department. Both approaches are acceptable.

Sample LDIF for One User

dn: uid=mchen,ou=Engineering,ou=People,dc=widgets,dc=local
objectClass: inetOrgPerson
objectClass: posixAccount
uid: mchen
cn: Maria Chen
sn: Chen
uidNumber: 1001
gidNumber: 2001
homeDirectory: /home/mchen
loginShell: /bin/bash
mail: mchen@widgets.local
userPassword: {SSHA}example_hash_here

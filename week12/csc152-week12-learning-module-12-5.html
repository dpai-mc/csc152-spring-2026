§3  Splunk Enterprise Installation on Linux

3.1  Splunk Architecture — Quick Review

Before installing, recall the three main Splunk components you learned about in Week 9 and Splunk Fundamentals 1:

ComponentRoleDefault Port(s)
Splunk Enterprise (Indexer + Search Head)Receives, indexes, and searches log data; hosts the web UI8000 (Web), 8089 (Mgmt), 9997 (Receiving)
Universal ForwarderLightweight agent on endpoints; collects and forwards log data9997 (outbound to indexer), 8089 (mgmt)
Syslog InputReceives syslog from network devices (e.g., pfSense)514 (UDP/TCP)

In this module, you install Splunk Enterprise — which acts as both the indexer and search head in our lab environment. The Universal Forwarders will be deployed in Week 13.

You will install Splunk on a Linux VM in your Proxmox lab environment. This can be an existing Ubuntu Server VM on your Servers subnet or a new dedicated VM. Recommended minimum resources:

3.3  Step-by-Step Installation (.deb Package Path)

Step 1: Download the Splunk Enterprise .deb Package

Splunk Enterprise is available as a free download. The free license allows 500 MB/day of indexing, which is sufficient for our lab. Your instructor may provide a direct download link or a local copy on the lab network.

# Navigate to a temporary directory
cd /tmp

# Download Splunk Enterprise .deb package
# (Your instructor will provide the current URL or a local mirror)
wget -O splunk-latest-linux-amd64.deb "https://download.splunk.com/products/splunk/releases/9.2.0/linux/splunk-9.2.0-7c27215a528f-linux-2.6-amd64.deb"
⚠️
Version Note: The filename and URL above are examples. Your instructor will confirm the exact version to download. The installation steps are the same regardless of the minor version number.

Step 2: Install the .deb Package

The dpkg command installs the package to /opt/splunk:

# Install the Splunk Enterprise package
sudo dpkg -i splunk-latest-linux-amd64.deb

Expected output includes lines showing the package being unpacked and installed to /opt/splunk. No errors should appear. If you see dependency errors, run sudo apt --fix-broken install and retry.

Step 3: Start Splunk and Accept the License

The first start requires accepting the license agreement and creating an administrator account:

# Start Splunk for the first time
# --accept-license: accepts the Splunk license agreement
# --answer-yes: answers "yes" to prompts automatically
sudo /opt/splunk/bin/splunk start --accept-license --answer-yes

During this first start, Splunk will prompt you to create an administrator account:

Please enter an administrator username: admin
Password must contain at least:
   * 8 total printable ASCII characters.
Please enter a new password:
Please confirm new password:
🔐
Document Your Credentials! Record the admin username and password immediately. You will need these credentials for the web UI, for configuring forwarders in Week 13, and for Project 3. Store them securely — consider adding them to your lab documentation.

When startup completes, you will see:

Splunk> See your world.  Maybe wish you hadn't.

Checking prerequisites...
	Checking mgmt port [8089]: open
	Checking appserver port [127.0.0.1:8065]: open
	Checking kvstore port [8191]: open
	Checking configuration... Done.
	Waiting for web server at [http://127.0.0.1:8000] to be available.. Done

If you get stuck, we're here to help.
Look for answers here: http://docs.splunk.com

The Splunk web interface is at http://<your-server-ip>:8000

Step 4: Enable Boot-Start

Configure Splunk to start automatically when the VM boots. This ensures your Splunk server survives reboots:

# Enable Splunk to start at boot (creates a systemd unit file)
sudo /opt/splunk/bin/splunk enable boot-start -user splunk --accept-license --answer-yes

This command creates a systemd service unit so that Splunk starts automatically. The -user splunk flag runs the Splunk process as the splunk user (created during installation) rather than root, following the principle of least privilege.

💡
Service Management: After enabling boot-start, you can manage Splunk using standard systemctl commands:
sudo systemctl status Splunkd — Check service status
sudo systemctl restart Splunkd — Restart the service
sudo systemctl stop Splunkd — Stop the service

§4  Accessing the Splunk Web UI

4.1  Connecting to the Web Interface

Splunk's web interface runs on port 8000 by default. Open a web browser on any machine that can reach your Splunk server's IP address and navigate to:

http://<splunk-server-ip>:8000

For example, if your Splunk server's IP is 10.0.1.50:

http://10.0.1.50:8000

Log in with the administrator credentials you created during installation.

⚠️
Cannot Connect? If the web UI does not load, check these common causes:
1. Is the Splunk service running? Run sudo systemctl status Splunkd
2. Is port 8000 open on the Linux VM's local firewall (if ufw is enabled)?
3. Is there a pfSense rule allowing traffic to port 8000? (See §6 below)
4. Are you on the correct network subnet to reach the Servers network?

§5  Enabling the Receiving Port (9997)

5.1  What Is the Receiving Port?

The receiving port is the TCP port on which Splunk Enterprise listens for data from Universal Forwarders. When you deploy forwarders on your Linux and Windows servers in Week 13, those forwarders will send log data to this port.

PortDirectionPurpose
9997Forwarder → Indexer (inbound)Log data transmission from Universal Forwarders
8089BidirectionalSplunk management port (forwarder-to-indexer communication, REST API)
8000Client → Splunk (inbound)Web UI access

Port 9997 is the industry-standard default for Splunk forwarder-to-indexer communication. It is not enabled by default after installation — you must explicitly configure it.

5.2  Enabling via the Web UI

Follow these steps in the Splunk web interface:

  1. Log into the Splunk web UI at http://<splunk-server-ip>:8000
  2. Click Settings in the top navigation bar
  3. Under the Data section, click Forwarding and receiving
  4. In the Receive data section, click Configure receiving
  5. Click the New Receiving Port button (upper right)
  6. Enter 9997 in the port field
  7. Click Save

You should now see port 9997 listed as an active receiving port with a status of Enabled.

5.3  Verifying the Receiving Port

Confirm that Splunk is actually listening on port 9997 from the Linux command line:

# Check if Splunk is listening on port 9997
sudo ss -tlnp | grep 9997

Expected output:

LISTEN  0  128  *:9997  *:*  users:(("splunkd",pid=12345,fd=6))

You can also verify from within Splunk:

# Using the Splunk CLI to check listening ports
sudo /opt/splunk/bin/splunk display listen

Expected output:

Receiving is enabled on port(s): 9997
💡
Why 9997 and Not 514? Port 514 is used for syslog — the protocol pfSense uses to send logs. Port 9997 is Splunk's proprietary forwarder-to-indexer protocol, which is encrypted, compressed, and supports acknowledgment. In Project 3, you will use both: syslog (514) for pfSense and the forwarder protocol (9997) for Linux and Windows servers.

§6  Firewall Configuration for Splunk

Your pfSense firewall (deployed in Week 9 / Project 2) must allow traffic to reach the Splunk server on the required ports. This section applies the firewall rule creation skills you practiced in Module 9.3.

6.1  Required Firewall Rules

Create the following rules on the appropriate pfSense interfaces. The exact interface depends on your network topology from Project 2 (Management, Servers, Clients subnets):

Rule #SourceDestinationPortProtocolPurpose
1Management subnetSplunk server IP8000TCPWeb UI access for administrators
2Servers subnetSplunk server IP9997TCPUniversal Forwarder data from Linux/Windows servers
3Servers subnetSplunk server IP8089TCPForwarder management communication
4Clients subnetSplunk server IP9997TCPUniversal Forwarder data from client machines (if applicable)
5pfSense (self)Splunk server IP514UDPpfSense syslog forwarding (configured in Week 9)
⚠️
Principle of Least Privilege: Only open ports from the subnets that need access. For example, the Clients subnet probably does not need access to port 8000 (Web UI) — only administrators on the Management subnet should access the Splunk dashboard. This is the same principle you applied when designing firewall rules in Project 2.

6.2  Creating the Rules in pfSense

For each rule, navigate in pfSense:

  1. Log into the pfSense web UI
  2. Navigate to Firewall → Rules
  3. Select the appropriate interface tab (e.g., MANAGEMENT, SERVERS, or CLIENTS)
  4. Click Add (up arrow for top, down arrow for bottom)
  5. Set Action to Pass
  6. Set Protocol to TCP (or UDP for syslog)
  7. Set Source to the appropriate subnet or alias
  8. Set Destination to the Splunk server IP (or create an alias)
  9. Set Destination Port Range to the specific port (e.g., 8000, 9997, 8089)
  10. Add a Description (e.g., "Allow Splunk Web UI from Management")
  11. Click Save, then click Apply Changes
💡
Use Aliases: Consider creating a pfSense alias called Splunk_Ports containing 8000, 8089, and 9997. This simplifies rule management — a single rule with the alias covers all three ports from a given subnet. You can also create a Splunk_Server host alias for the server IP.

6.3  Verifying Firewall Rules

After creating the rules, verify connectivity from each relevant subnet:

# From a machine on the Management subnet, test web UI port
nc -zv <splunk-server-ip> 8000

# From a machine on the Servers subnet, test receiving port
nc -zv <splunk-server-ip> 9997

# From a machine on the Servers subnet, test management port
nc -zv <splunk-server-ip> 8089

Successful output looks like:

Connection to 10.0.1.50 9997 port [tcp/*] succeeded!

If the connection fails, check:

  • pfSense rule order (rules are processed top-to-bottom; first match wins)
  • Whether you clicked Apply Changes after adding rules
  • Whether the Linux VM's local firewall (ufw) is also blocking the port
  • pfSense logs: Status → System Logs → Firewall — look for blocked entries matching your Splunk server IP

§7  Verification Checklist

Before moving on, confirm every item below. This checklist ensures your Splunk infrastructure is ready for Universal Forwarder deployment in Week 13 and Project 3:

  • Splunk service is running — sudo systemctl status Splunkd shows active (running)
  • Boot-start is enabled — Splunk will survive VM reboots
  • Web UI is accessible — Navigate to http://<splunk-ip>:8000 from the Management subnet and log in successfully
  • Port 9997 is listening — sudo ss -tlnp | grep 9997 shows Splunk listening
  • Port 8089 is listening — sudo ss -tlnp | grep 8089 shows Splunk management port
  • pfSense rules are in place — Rules for ports 8000, 8089, and 9997 exist on appropriate interfaces
  • Firewall rules are verified — nc -zv tests succeed from relevant subnets
  • Admin credentials are documented — Username and password stored securely in your lab documentation
  • Splunk server IP is documented — Record the IP address for use when configuring forwarders next week

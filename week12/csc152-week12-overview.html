Welcome & Weekly Focus

Welcome to Week 12! This week you shift from the proprietary world of Active Directory to its open-standards counterpart: the Lightweight Directory Access Protocol (LDAP) and its reference implementation, OpenLDAP. You will move from LDAP theory all the way to a running OpenLDAP server with a populated directory and working client authentication—giving you fluency in both major directory-services ecosystems. Along the way, you will also install Splunk Enterprise on a Linux VM, standing up the monitoring infrastructure that Project 3 will build upon starting next week.

Week 12 Learning Goals

  • Explain the LDAP protocol, including its purpose, operation model (bind, search, modify), and relationship to the X.500 standard.
  • Describe the LDAP Directory Information Tree (DIT) structure, including distinguished names (DNs), relative distinguished names (RDNs), and common object classes.
  • Install and perform initial configuration of OpenLDAP (slapd) on a Linux server.
  • Design and populate an LDAP directory with organizational units, user entries, and group entries using LDIF files.
  • Query and modify LDAP directory entries using command-line tools (ldapsearch, ldapadd, ldapmodify, ldapdelete).
  • Configure a Linux client system to authenticate against an OpenLDAP server using SSSD or NSS/PAM integration.
  • Compare and contrast Active Directory and OpenLDAP in terms of schema, tooling, scalability, and enterprise use cases.
  • Install Splunk Enterprise on a Linux VM and confirm web UI access and receiver port configuration.

Key Concepts & Themes

  • Open Standard vs. Proprietary: LDAP is an open, standards-based protocol (RFC 4511); Active Directory extends LDAP with proprietary features. Understanding both gives you cross-platform fluency.
  • Directory Information Tree (DIT): LDAP organizes data in a hierarchical tree—root DSE → base DN → OUs → entries—analogous to a file system or AD’s domain/OU structure.
  • LDIF as the Lingua Franca: The LDAP Data Interchange Format is the text-based format used to add, modify, and export directory entries. If you can read and write LDIF, you can manage any LDAP directory.
  • Centralized Authentication: Moving from local /etc/passwd accounts to LDAP-backed authentication solves password-sync and account-sprawl problems across an organization.
  • NSS, PAM, and SSSD: These Linux subsystems bridge the gap between local identity lookups and remote LDAP directories.
  • Splunk as Security Monitoring Infrastructure: Installing Splunk Enterprise this week creates the receiving server that Universal Forwarders will feed in Week 13—the backbone of your Project 3 monitoring stack.
  • Draw the DIT: Sketch the directory tree on paper before writing LDIF files. A visual map prevents DN syntax errors.
  • Compare side-by-side: Keep your Week 11 AD notes open while studying LDAP. Mapping concepts across both systems deepens understanding.
  • Test incrementally: After each ldapadd command, run ldapsearch to confirm the entries loaded correctly before adding more.
  • Document as you go: Screenshot your terminal output during OpenLDAP and Splunk installation—you will need these for Lab 12 and for Project 3.
  • Start Lab 12 early: Even though it is extra credit, the OpenLDAP and Splunk skills practiced in Lab 12 directly prepare you for Projects 3 and 4.

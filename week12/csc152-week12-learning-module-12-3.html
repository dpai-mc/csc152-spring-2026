Prerequisites This module assumes you have a running slapd instance from Module 12.2 with a base DN of dc=lab,dc=local and a working admin bind DN of cn=admin,dc=lab,dc=local. You should also be comfortable with LDIF syntax and DN notation from Module 12.1.

§1 Designing the Directory Structure

Before you create a single LDIF entry, you need a plan. A well-designed Directory Information Tree (DIT) mirrors the organization it serves and makes future queries, access-control rules, and client integration straightforward.

1.1 Meet the Fictional Organization

Throughout this module we will populate a directory for Mendocino Widgets, Inc.—a small company with three departments:

  • Engineering – builds the widgets
  • Sales – sells the widgets
  • IT – keeps the servers running

We also need a container for POSIX groups that map to these departments (and possibly cross-functional teams).

1.2 Planned DIT

dc=lab,dc=local ├── ou=People │ ├── uid=jdoe (Engineering) │ ├── uid=asmith (Sales) │ └── uid=bwong (IT) ├── ou=Groups │ ├── cn=engineering │ ├── cn=sales │ └── cn=itstaff └── cn=admin (created during slapd setup)

1.3 Design Guidelines

  • Keep it flat at first. Two top-level OUs (People and Groups) are sufficient for most small-to-medium deployments. You can add sub-OUs (e.g., ou=Engineering,ou=People,…) later if needed.
  • Use meaningful RDNs. For people, uid= is the conventional RDN attribute. For groups, use cn=.
  • Plan UID/GID number ranges. Start user uidNumber values at 10000 and group gidNumber values at 20000 to avoid collisions with local Linux accounts.
  • Document your schema choices. We will use inetOrgPerson + posixAccount + shadowAccount for users, and posixGroup for groups.
Week 6 Callback The attributes you assign to LDAP user entries—uidNumber, gidNumber, homeDirectory, loginShell—are the same fields you managed in /etc/passwd during Week 6. LDAP simply stores them centrally instead of on every machine.

§2 Creating Organizational Unit Entries

OUs are the containers that give your DIT its structure. We create them first because user and group entries will live inside them.

2.1 The LDIF File – ous.ldif

ous.ldif 
dn: ou=People,dc=lab,dc=local
objectClass: organizationalUnit
ou: People
description: Container for user accounts

dn: ou=Groups,dc=lab,dc=local
objectClass: organizationalUnit
ou: Groups
description: Container for POSIX groups
LDIF Syntax Reminder Each entry begins with a dn: line, followed by attribute-value pairs (one per line). A blank line separates entries. No trailing spaces after values—they become part of the data!

2.2 Loading with ldapadd

ldapadd -x -D "cn=admin,dc=lab,dc=local" -W -f ous.ldif
FlagMeaning
-xUse simple authentication (not SASL)
-D "…"Bind DN – the admin account you authenticate as
-WPrompt for the bind password interactively
-f ous.ldifRead entries from the specified LDIF file

Expected output on success:

adding new entry "ou=People,dc=lab,dc=local"

adding new entry "ou=Groups,dc=lab,dc=local"

2.3 Verifying

ldapsearch -x -LLL -b "dc=lab,dc=local" "(objectClass=organizationalUnit)" dn

You should see both OU DNs listed.

§3 Creating User Entries

3.1 Required Object Classes & Attributes

Each user entry combines two (or three) object classes:

Object ClassTypeKey Attributes Provided
inetOrgPersonStructuralcn, sn, mail, telephoneNumber, userPassword
posixAccountAuxiliaryuid, uidNumber, gidNumber, homeDirectory, loginShell
shadowAccountAuxiliary (optional)shadowLastChange, shadowMax – password aging

3.2 Generating a Hashed Password

Never store plaintext passwords. Use slappasswd to generate a salted hash:

slappasswd -s P@ssw0rd123

Sample output:

{SSHA}W3ipFGmzS3Khjuv2snm7c+bOBJSxAMDa
Security Note Running slappasswd -s with the password on the command line leaves it in your shell history. In production, run slappasswd without -s so it prompts you interactively. We use -s here only for clarity in examples.

§4 Creating Group Entries

4.1 The posixGroup Object Class

POSIX groups in LDAP mirror /etc/group on Linux. Each group entry needs:

  • cn – group name (also the RDN)
  • gidNumber – numeric GID
  • memberUid – one line per member, listing the uid value (not the full DN)

4.2 The LDIF File – groups.ldif

groups.ldif 
dn: cn=engineering,ou=Groups,dc=lab,dc=local
objectClass: posixGroup
cn: engineering
gidNumber: 20001
memberUid: jdoe

dn: cn=sales,ou=Groups,dc=lab,dc=local
objectClass: posixGroup
cn: sales
gidNumber: 20002
memberUid: asmith

dn: cn=itstaff,ou=Groups,dc=lab,dc=local
objectClass: posixGroup
cn: itstaff
gidNumber: 20003
memberUid: bwong
memberUid: jdoe
Multiple Members To add more than one member to a group, repeat the memberUid: line—once per user. Notice that jdoe is in both engineering and itstaff.

4.3 Loading Groups

ldapadd -x -D "cn=admin,dc=lab,dc=local" -W -f groups.ldif

§5 Bulk Population & Error Handling

5.1 Combining Everything into One File

In practice you will often create a single LDIF file that contains OUs, users, and groups—in that order (parents before children). Here is a condensed example:

directory-seed.ldif 
# --- Organizational Units ---
dn: ou=People,dc=lab,dc=local
objectClass: organizationalUnit
ou: People

dn: ou=Groups,dc=lab,dc=local
objectClass: organizationalUnit
ou: Groups

# --- Users ---
dn: uid=jdoe,ou=People,dc=lab,dc=local
objectClass: inetOrgPerson
objectClass: posixAccount
objectClass: shadowAccount
uid: jdoe
cn: Jane Doe
sn: Doe
uidNumber: 10001
gidNumber: 20001
homeDirectory: /home/jdoe
loginShell: /bin/bash
userPassword: {SSHA}W3ipFGmzS3Khjuv2snm7c+bOBJSxAMDa

dn: uid=asmith,ou=People,dc=lab,dc=local
objectClass: inetOrgPerson
objectClass: posixAccount
objectClass: shadowAccount
uid: asmith
cn: Alex Smith
sn: Smith
uidNumber: 10002
gidNumber: 20002
homeDirectory: /home/asmith
loginShell: /bin/bash
userPassword: {SSHA}xKz9pLmNqR3vTu8Yw1cJd+fAGHiSjKlM

# --- Groups ---
dn: cn=engineering,ou=Groups,dc=lab,dc=local
objectClass: posixGroup
cn: engineering
gidNumber: 20001
memberUid: jdoe

dn: cn=sales,ou=Groups,dc=lab,dc=local
objectClass: posixGroup
cn: sales
gidNumber: 20002
memberUid: asmith
Order Matters! Parent entries must appear before their children in the LDIF file. If you try to add uid=jdoe,ou=People,dc=lab,dc=local before ou=People,dc=lab,dc=local exists, ldapadd will fail with error 32 (No such object).

5.2 The -c (Continue on Error) Flag

When loading many entries, one bad entry can halt the entire import. The -c flag tells ldapadd to skip errors and keep processing:

ldapadd -x -c -D "cn=admin,dc=lab,dc=local" -W -f directory-seed.ldif

5.3 Common Errors

Error CodeMessageLikely CauseFix
32No such objectParent OU does not exist yetAdd parent entries first; check DN spelling
68Already existsDuplicate entryEntry was already loaded; use -c to skip
65Object class violationMissing a required attributeCheck object class requirements (e.g., sn for inetOrgPerson)
21Invalid syntaxMalformed attribute value or extra whitespaceReview LDIF for trailing spaces or blank-line issues

§6 Querying with ldapsearch

6.1 Command Anatomy

ldapsearch [options] -b "base_dn" -s scope "filter" [attributes...]
OptionMeaning
-xSimple authentication
-LLLClean LDIF output (no comments, no version line)
-b "…"Base DN – the starting point of the search
-s base|one|subScope: base = entry itself; one = immediate children; sub = entire subtree (default)
-D "…"Bind DN (omit for anonymous search)
-WPrompt for password

6.2 Scope Visualized

dc=lab,dc=local ← -s base (only this entry) ├── ou=People ← -s one (direct children only) │ ├── uid=jdoe ← -s sub (everything below base) │ ├── uid=asmith │ └── uid=bwong └── ou=Groups ├── cn=engineering ├── cn=sales └── cn=itstaff

6.3 Practical Examples

List all users (subtree search) 
ldapsearch -x -LLL -b "ou=People,dc=lab,dc=local" \
  "(objectClass=posixAccount)" uid cn mail

Returns the uid, cn, and mail attributes for every user under ou=People.

Find a specific user by uid 
ldapsearch -x -LLL -b "dc=lab,dc=local" "(uid=jdoe)"

Searches the entire tree for the entry whose uid equals jdoe. Returns all attributes of that entry.

List all groups 
ldapsearch -x -LLL -b "ou=Groups,dc=lab,dc=local" \
  "(objectClass=posixGroup)" cn memberUid
One-level scope – list only direct children of the base DN 
ldapsearch -x -LLL -b "dc=lab,dc=local" -s one "(objectClass=*)" dn

Returns only ou=People, ou=Groups, and cn=admin—the direct children of the base.

§7 LDAP Search Filters

Filters are the “WHERE clause” of LDAP. They use a prefix-notation syntax defined in RFC 4515.

7.1 Filter Types

TypeSyntaxExampleMatches
Equality(attr=value)(uid=jdoe)Exact match
Presence(attr=*)(mail=*)Any entry that has a mail attribute
Substring(attr=*val*)(cn=*Doe*)Contains “Doe”
Greater-or-equal(attr>=value)(uidNumber>=10002)uidNumber 10002 or above
Less-or-equal(attr<=value)(uidNumber<=10002)uidNumber 10002 or below

7.2 Compound (Boolean) Filters

Combine simple filters using prefix Boolean operators. The operator comes first, inside the parentheses:

OperatorSyntaxExample
AND(&(filter1)(filter2))(&(objectClass=posixAccount)(gidNumber=20001))
OR(|(filter1)(filter2))(|(uid=jdoe)(uid=bwong))
NOT(!(filter))(!(loginShell=/sbin/nologin))

7.3 Filter Examples in Context

All Engineering users (gidNumber = 20001) 
ldapsearch -x -LLL -b "ou=People,dc=lab,dc=local" \
  "(&(objectClass=posixAccount)(gidNumber=20001))" uid cn
Users with uidNumber >= 10002 
ldapsearch -x -LLL -b "ou=People,dc=lab,dc=local" \
  "(uidNumber>=10002)" uid uidNumber
Users who have an email address 
ldapsearch -x -LLL -b "ou=People,dc=lab,dc=local" \
  "(mail=*)" uid mail
Users named jdoe OR bwong 
ldapsearch -x -LLL -b "ou=People,dc=lab,dc=local" \
  "(|(uid=jdoe)(uid=bwong))" uid cn
All users whose shell is NOT /sbin/nologin 
ldapsearch -x -LLL -b "ou=People,dc=lab,dc=local" \
  "(&(objectClass=posixAccount)(!(loginShell=/sbin/nologin)))" uid loginShell

§8 Modifying Entries with ldapmodify

ldapmodify reads an LDIF file that uses changetype: modify directives. Each modification block specifies an operation (add, replace, or delete) on one or more attributes.

8.1 Modification Operations

Replace an Attribute Value

Change Brian Wong’s phone number:

modify-phone.ldif 
dn: uid=bwong,ou=People,dc=lab,dc=local
changetype: modify
replace: telephoneNumber
telephoneNumber: 707-555-0200
ldapmodify -x -D "cn=admin,dc=lab,dc=local" -W -f modify-phone.ldif

Add a New Attribute (or Additional Value)

Add a phone number to Alex Smith (who currently has none), and add jdoe to the sales group:

modify-add.ldif 
dn: uid=asmith,ou=People,dc=lab,dc=local
changetype: modify
add: telephoneNumber
telephoneNumber: 707-555-0101

dn: cn=sales,ou=Groups,dc=lab,dc=local
changetype: modify
add: memberUid
memberUid: jdoe
ldapmodify -x -D "cn=admin,dc=lab,dc=local" -W -f modify-add.ldif

Delete an Attribute Value

Remove jdoe from the sales group (delete a specific memberUid value):

modify-del.ldif 
dn: cn=sales,ou=Groups,dc=lab,dc=local
changetype: modify
delete: memberUid
memberUid: jdoe

To delete the entire attribute (all values), omit the value line:

dn: uid=bwong,ou=People,dc=lab,dc=local
changetype: modify
delete: telephoneNumber

8.2 Multiple Operations in One Block

Use a single dash (-) on its own line to separate multiple operations on the same entry:

modify-multi.ldif 
dn: uid=jdoe,ou=People,dc=lab,dc=local
changetype: modify
replace: mail
mail: jane.doe@lab.local
-
add: telephoneNumber
telephoneNumber: 707-555-0150
The Dash Separator The single - on its own line is required between operations within the same entry block. Forgetting it is one of the most common LDIF errors.

§9 Deleting Entries with ldapdelete

9.1 Deleting a Single (Leaf) Entry

ldapdelete -x -D "cn=admin,dc=lab,dc=local" -W \
  "uid=asmith,ou=People,dc=lab,dc=local"

9.2 Deleting from a File

Create a file with one DN per line:

delete-list.txt 
uid=asmith,ou=People,dc=lab,dc=local
cn=sales,ou=Groups,dc=lab,dc=local
ldapdelete -x -D "cn=admin,dc=lab,dc=local" -W -f delete-list.txt

9.3 Important Constraints

No Subtree Deletes by Default LDAP will not delete a non-leaf entry (an entry that has children). If you try to delete ou=People while it still contains user entries, you will get error 66 (Not allowed on non-leaf). Delete children first, then the parent.

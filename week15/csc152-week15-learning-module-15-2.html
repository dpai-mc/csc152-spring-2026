Objectives & Context

In Module 15.1 you learned why disaster recovery matters and how Recovery Time Objective (RTO) and Recovery Point Objective (RPO) quantify an organization's tolerance for downtime and data loss. This module answers the next logical questions: what types of backups exist, and when should they run?

By the end of this module you will be able to:

  • Define and distinguish full, incremental, and differential backups.
  • Evaluate the tradeoffs of each backup type in terms of backup time, storage consumption, restore time, and restore complexity.
  • Explain industry-standard rotation schemes: Grandfather-Father-Son (GFS) and the 3-2-1 rule.
  • Identify appropriate backup destinations and media for different scenarios.
  • Design a backup schedule that balances RPO requirements against storage and bandwidth constraints.
Key Insight: Your backup strategy is the primary mechanism for achieving your RPO targets. A 24-hour RPO means you need at least daily backups. A 1-hour RPO demands near-continuous data protection. The backup type and schedule you choose directly determine whether you can meet your recovery goals.

2. Full, Incremental, and Differential Backups

2.1 Full Backup

A full backup creates a complete copy of all selected data every time it runs. It is the foundation upon which the other backup types depend.

How It Works

Every file and directory in the backup set is copied to the backup destination regardless of whether it has changed since the last backup. After completion, the archive bit (or equivalent marker) on each file is cleared, indicating it has been backed up.

Advantages

  • Simplest restore: Only a single backup set is needed to recover all data.
  • Self-contained: Each backup is independent — no dependency chain.
  • Fastest restore time: One media set to read.

Disadvantages

  • Longest backup time: Every file is copied every run, even if unchanged.
  • Most storage consumed: Requires the most space per backup cycle.
  • Largest backup window: May not fit within maintenance windows for large datasets.

When to Use

Full backups are ideal for small datasets, critical systems requiring the fastest possible recovery, or as the periodic "baseline" in a combined strategy (e.g., weekly full + daily incremental).

2.2 Incremental Backup

An incremental backup copies only the data that has changed since the last backup of any type (full or incremental). Each incremental captures a small "slice" of changes.

How It Works

After a full backup on Sunday, Monday's incremental backs up only Monday's changes. Tuesday's incremental backs up only Tuesday's changes (not Monday's). Each backup is small and fast, but restoration requires assembling the full chain.

Restore Process

To restore to Saturday's state, you need: Sunday's full + Monday's incremental + Tuesday's incremental + … + Saturday's incremental. All pieces in the chain must be intact and applied in order.

Advantages

  • Fastest backup time: Only changed data is copied each run.
  • Least storage per run: Each incremental is small.
  • Smallest backup window: Ideal for environments with limited maintenance time.

Disadvantages

  • Slowest restore: Must apply the full backup plus every incremental in sequence.
  • Most complex restore: If any single incremental in the chain is missing or corrupted, all subsequent data is lost.
  • Higher risk: Longer dependency chain means more points of failure.

2.3 Differential Backup

A differential backup copies all data that has changed since the last full backup. Unlike incrementals, differentials are cumulative — each one grows larger as the week progresses.

How It Works

After a full backup on Sunday, Monday's differential backs up Monday's changes. Tuesday's differential backs up both Monday's and Tuesday's changes. By Saturday, the differential contains all changes for the entire week.

Restore Process

To restore to any point, you need only two backup sets: the last full backup + the most recent differential. This is significantly simpler and faster than incremental restores.

Advantages

  • Faster restore than incremental: Only two media sets needed (full + latest differential).
  • Simpler restore: No dependency chain beyond the last full.
  • Good balance: Middle ground between full and incremental.

Disadvantages

  • Growing backup size: Each differential is larger than the previous one.
  • More storage than incremental: Cumulative changes mean redundant data across differentials.
  • Longer backup time than incremental: Especially toward the end of the cycle.

4. Backup Rotation Schemes

A rotation scheme defines how long backups are retained and how media are reused over time. Rotation schemes ensure you have recovery points at multiple granularities (daily, weekly, monthly) without consuming unlimited storage.

4.1 Grandfather-Father-Son (GFS)

GFS is the most widely used rotation scheme in enterprise environments. It organizes backups into three tiers:

GFS Rotation Tiers
TierNameFrequencyTypical RetentionPurpose
DailySonEvery business day7–14 daysShort-term recovery (accidental deletion, recent corruption)
WeeklyFatherOnce per week (e.g., Friday)4–5 weeksMedium-term recovery; point-in-time restore from past weeks
MonthlyGrandfatherOnce per month (e.g., last Friday)12 months or longerLong-term archival; compliance; historical recovery

How GFS Works in Practice

  1. Daily (Son): Run an incremental or differential backup each business day. Keep for 1–2 weeks, then overwrite/delete.
  2. Weekly (Father): Promote one daily backup (typically Friday's or run a separate full) to "weekly" status. Keep for 4–5 weeks.
  3. Monthly (Grandfather): Promote one weekly backup (typically the last of the month) to "monthly" status. Keep for 12+ months.

The result: you can restore from yesterday, from any day in the past two weeks, from any week in the past month, or from any month in the past year — all without storing every daily backup indefinitely.

4.2 The 3-2-1 Rule

The 3-2-1 Rule

Maintain at least:

  • 3 copies of your data (the original + 2 backups)
  • 2 different media types (e.g., local disk + cloud, or disk + tape)
  • 1 copy offsite (geographically separated from the primary site)

Why Each Element Matters

3-2-1 Rule Rationale
ElementProtects AgainstExample
3 copiesSingle point of failure; accidental deletionProduction data + local backup + offsite backup
2 media typesMedia-specific failures (e.g., all drives from same batch fail)Internal disk + cloud object storage
1 offsiteSite-wide disasters (fire, flood, theft, ransomware)Cloud backup in a different region; tape stored at a secure facility

The 3-2-1 rule is a principle, not a rigid prescription. Some organizations extend it to 3-2-1-1-0 (add 1 immutable copy and 0 errors verified by testing). The core idea: never let a single event destroy all copies of your data.

4.3 Tower of Hanoi (Awareness)

Tower of Hanoi Rotation — Deeper Dive (Optional)

The Tower of Hanoi scheme uses a mathematical pattern (based on the classic puzzle) to determine media rotation. Media sets are labeled A, B, C, D, etc. Set A is used every other backup, Set B every fourth backup, Set C every eighth, and so on.

Pattern for 5 sets over 16 days: A B A C A B A D A B A C A B A E

Advantage: Provides more recent recovery points while still maintaining some older ones, using fewer media sets than GFS for equivalent coverage.

Disadvantage: Complex to manage manually; primarily used by automated backup software. Most administrators prefer GFS for its simplicity and clarity.

For this course, awareness of the Tower of Hanoi scheme is sufficient. You will not be asked to implement it in Project 3.

5. Backup Destinations & Media

Where you store your backups is as important as what you back up. Each destination has tradeoffs in speed, cost, capacity, and disaster resilience.

Backup Destination Comparison
DestinationSpeedCostCapacityDisaster ResilienceBest For
Local Disk (USB, internal) Fast Moderate Moderate Low (same site) Quick restores; first copy in 3-2-1
NAS / Network Storage Fast (LAN speed) Moderate High (expandable) Low-Moderate (same site unless replicated) Centralized backup for multiple servers
Tape (LTO) Slow (sequential) Low per TB Very High High (portable, offsite storage) Long-term archival; air-gapped protection
Cloud / Offsite Variable (WAN dependent) Ongoing (subscription) Virtually unlimited High (geographic separation) Offsite copy; DR; the "1" in 3-2-1
Deeper Dive: Choosing Destinations for Different Backup Tiers

A well-designed strategy often uses multiple destinations mapped to backup tiers:

  • Daily backups (Son): Local disk or NAS for fast backup and fast restore of recent data.
  • Weekly backups (Father): NAS with replication to a second site, or cloud storage.
  • Monthly backups (Grandfather): Cloud cold storage or offsite tape for long-term retention at low cost.

This tiered approach satisfies the 3-2-1 rule while optimizing cost: fast, expensive storage for frequently accessed recent backups; slow, cheap storage for rarely accessed archives.

RAID Is Not Backup: Remember from Week 10 — RAID provides fault tolerance against disk failure, but it does not protect against accidental deletion, ransomware, corruption, or site-wide disasters. RAID and backups serve complementary but different purposes.

6. Retention Policies

A retention policy defines how long backup data is kept before it is overwritten or deleted. Retention must balance three competing concerns:

  1. Recovery needs: How far back might you need to restore?
  2. Storage costs: Longer retention = more storage consumed.
  3. Compliance requirements: Regulations may mandate minimum retention periods.

Sample Retention Policy

Example Retention Policy for a Small Organization
Backup TierFrequencyRetention PeriodRationale
Daily (Incremental)Every business day14 daysRecover from recent accidental changes
Weekly (Full)Every Sunday4 weeksPoint-in-time restore from past month
Monthly (Full)First Sunday of month12 monthsHistorical recovery; audit support
Annual (Full)January 17 yearsCompliance / legal hold
Deeper Dive: Compliance Considerations

Various regulations specify minimum data retention periods. While the details vary by industry and jurisdiction, awareness of these is important for system administrators:

  • HIPAA (healthcare): Medical records typically retained 6–7 years.
  • PCI-DSS (payment card data): Audit logs retained for at least 1 year, with 3 months immediately available.
  • SOX (financial): Financial records retained 7 years.
  • FERPA (education): Student records retained for the life of the institution in many cases.

For your Project 3 lab environment, you are not bound by these regulations, but your documentation should demonstrate awareness that real-world backup strategies must account for compliance requirements.

7. Designing a Backup Schedule — Guided Exercise

Now let's put everything together. Designing a backup schedule follows a structured process. Walk through these steps for your own lab infrastructure as preparation for your Project 3 deliverable.

Step-by-Step Process

  1. Inventory systems: List all systems and data that need backup.
  2. Assign criticality: Use your BIA from Module 15.1 to rank systems.
  3. Determine RPO: What is the maximum acceptable data loss for each system?
  4. Select backup type: Choose full, incremental, or differential (or a combination) based on RPO, storage, and backup window constraints.
  5. Choose schedule: Define when backups run (time of day, day of week).
  6. Select destination: Where will backups be stored? Apply the 3-2-1 rule.
  7. Define retention: How long will each tier of backup be kept?
  8. Document: Record all decisions in a backup schedule matrix.

Worked Example: Three-Server Lab Environment

Consider an environment similar to your Project 3 lab with a Linux server, a Windows server, and a pfSense firewall/router:

Backup Schedule Matrix — Worked Example
System Backup Type Frequency Time Window Destination Retention
Linux Server (files & configs) Full + Incremental Weekly Full (Sun 2:00 AM); Daily Incremental (Mon–Sat 2:00 AM) 2:00–4:00 AM Local NAS + rsync to offsite Daily: 14 days; Weekly: 4 weeks
Windows Server (System State + Data) Full + Incremental Weekly Full (Sun 3:00 AM); Daily Incremental (Mon–Sat 3:00 AM) 3:00–5:00 AM Local disk + copy to NAS Daily: 14 days; Weekly: 4 weeks
pfSense (Config) Config Export (Full) Weekly (Sun 1:00 AM) 1:00–1:15 AM Local + copy to Linux server 4 versions
Splunk (Configs + Knowledge Objects) Full (tar of $SPLUNK_HOME/etc/) Daily (1:30 AM) 1:30–2:00 AM Local (Linux server) 14 days

Design Rationale

  • Why weekly full + daily incremental? Balances storage efficiency (incrementals are small) with reasonable restore complexity (at most 6 incrementals to apply). Meets a 24-hour RPO.
  • Why 2:00–5:00 AM window? Lowest usage period; minimizes impact on production workloads and avoids conflicts between server backups.
  • Why NAS + offsite? Satisfies the 3-2-1 rule: original data + local backup (NAS) + offsite copy = 3 copies on 2 media types with 1 offsite.
  • Why only config export for pfSense? pfSense configurations are small (KB-sized XML files). A config export captures the entire router/firewall state. No large data volumes to protect.
  • Why daily for Splunk configs? Splunk configurations change as dashboards and alerts are added. Index data is large and protected by retention policies rather than traditional backup.

8. Knowledge Check

Test your understanding with these ungraded questions. Click "Check Answer" to see explanatory feedback.

1. A backup that copies only the files that have changed since the last full backup is called a:

Correct! A differential backup copies all changes since the last full backup. An incremental backup copies changes since the last backup of any type. This distinction is the key difference between the two.
Not quite. The key phrase is "since the last full backup." That describes a differential backup. An incremental copies changes since the last backup of any type (full or incremental). Review Section 2.3 above.

2. A disaster occurs on Thursday evening. You have a full backup from Sunday and daily incremental backups from Monday through Thursday. How many backup sets do you need to restore to Thursday's state?

Correct! With incremental backups, you need the full backup plus every incremental in the chain up to your target restore point. That's Sunday's full + Monday + Tuesday + Wednesday + Thursday = 5 sets total.
Not quite. Incremental restores require the entire chain: the last full backup plus all subsequent incrementals up to the desired restore point. You cannot skip any link in the chain. The answer is 5 sets: Sunday's full + 4 daily incrementals. If this were a differential strategy, you would only need 2 sets (full + Thursday's differential). Review Section 2.2.

3. Your server has 200 GB of data with a 4% daily change rate. Using a weekly full + daily incremental strategy, approximately how much total backup storage is consumed over one week (Sunday full + 6 daily incrementals)?

Correct! Full = 200 GB. Each incremental ≈ 4% of 200 GB = 8 GB. Six incrementals = 48 GB. Total = 200 + 48 = 248 GB. Compare this to daily full backups which would consume 1,400 GB — incremental saves over 80% of storage!
Not quite. Calculate: Full backup = 200 GB. Daily change = 4% × 200 GB = 8 GB per incremental. Six incrementals = 6 × 8 GB = 48 GB. Total = 200 + 48 = 248 GB. Review the worked example in Section 3.

4. The 3-2-1 rule states that you should maintain:

Correct! The 3-2-1 rule: 3 copies of your data, on 2 different media types, with 1 copy stored offsite. This ensures no single event (hardware failure, ransomware, fire) can destroy all copies.
Not quite. The 3-2-1 rule is: 3 total copies of data (original + 2 backups), stored on 2 different media types, with 1 copy offsite. Review Section 4.2.

5. In a Grandfather-Father-Son (GFS) rotation, the "Grandfather" tier typically represents:

Correct! In GFS: Son = daily (short-term), Father = weekly (medium-term), Grandfather = monthly (long-term, typically 12+ months). The grandfather tier provides the longest historical recovery points.
Not quite. GFS tiers: Son = daily backups (1–2 weeks retention), Father = weekly backups (4–5 weeks retention), Grandfather = monthly backups (12+ months retention). The grandfather is the longest-retained tier. Review Section 4.1.

6. A ransomware attack encrypts all data on your server and all files on the directly-attached USB backup drive. Which backup destination would have protected you?

Correct! Ransomware can encrypt any storage the compromised system can reach. An offsite cloud backup with immutable (write-once) storage is isolated from the attack. This is exactly why the 3-2-1 rule requires an offsite copy — and why some organizations extend it to 3-2-1-1 (with an immutable copy).
Not quite. Ransomware encrypts everything accessible to the compromised system — including attached USB drives, local RAID arrays, and local disk partitions. Only a copy that is offsite and inaccessible to the compromised system (such as cloud storage with immutable/write-once settings) would survive. This illustrates why the "1 offsite" in the 3-2-1 rule is critical. Review Section 4.2 and Section 5.

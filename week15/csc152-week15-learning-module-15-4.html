2. Why Verification Matters

Organizations invest significant time and storage in running backups. Yet a surprising number discover their backups are unusable only when disaster strikes and they attempt recovery. At that point, the cost is catastrophic.

Real-World Failure Scenarios

Scenario 1 — Corrupted Media: A company ran nightly tape backups for two years. When ransomware encrypted their file server, they discovered that the tape drive had been writing unreadable data for the past six months due to a worn head. No one had ever tested a restore.
Scenario 2 — Wrong Data Backed Up: An administrator configured backups of /home but the critical application data lived in /opt/appdata. The backups completed successfully every night—backing up the wrong directory.
Scenario 3 — Encryption Key Lost: A team implemented encrypted backups as a security best practice (Module 14.3). When the server failed, they discovered the encryption passphrase was stored only on the failed server itself. The backups were intact but permanently inaccessible.
Scenario 4 — Incomplete Incremental Chain: Incremental backups ran daily, but the monthly full backup failed silently three months ago. The incremental chain cannot be restored without a valid full backup as its base.

The Cost of Late Discovery

When Failure Is Discovered Impact
During routine verification testing Minimal — fix the issue, re-run backup, no data loss
During a disaster recovery attempt Catastrophic — data loss up to full RPO window (or total loss), extended downtime exceeding RTO, potential business failure

The lesson is clear: verification is not optional — it is a required step in the backup process. A backup that has never been tested provides only the illusion of protection.

3. Backup Verification Methods

Effective verification uses multiple complementary methods. No single technique catches all failure modes.

3.1 Checksum Verification

Generate a cryptographic hash of the backup file immediately after creation, then verify it before any restore attempt. If the checksums match, the file has not been corrupted in storage or transit.

Generating Checksums (Linux)

# Generate SHA-256 checksum after backup creation
sha256sum /backup/server1-2026-05-01.tar.gz > /backup/server1-2026-05-01.tar.gz.sha256

# Verify checksum before restore
sha256sum -c /backup/server1-2026-05-01.tar.gz.sha256
# Expected output: /backup/server1-2026-05-01.tar.gz: OK

Using MD5 (faster, less secure — acceptable for integrity, not security)

# Generate MD5 checksum
md5sum /backup/server1-2026-05-01.tar.gz > /backup/server1-2026-05-01.tar.gz.md5

# Verify
md5sum -c /backup/server1-2026-05-01.tar.gz.md5
Best Practice: Use SHA-256 for verification. MD5 is faster but has known collision vulnerabilities. For backup integrity (not cryptographic security), either works, but SHA-256 is the modern standard.

3.2 Backup Log Review

Every backup operation should produce a log. Review logs for:

  • Completion status — Did the backup finish successfully or error out?
  • Warnings and errors — Were any files skipped due to permission issues or locks?
  • File count and total size — Is the backup size consistent with expectations? A sudden drop in size may indicate missing data.
  • Duration — Unexpectedly short backups may indicate the source data is missing.

Example: Checking a tar backup log

# In the backup script (from Module 15.3), redirect output to a log:
tar -czf /backup/home-2026-05-01.tar.gz /home 2>&1 | tee /var/log/backup/home-2026-05-01.log

# Review the log for errors:
grep -i "error\|warning\|cannot" /var/log/backup/home-2026-05-01.log

3.3 File Count and Size Comparison

Compare the number of files and total size in the backup against the source:

# Count files in source directory
find /home -type f | wc -l

# List contents of tar archive and count files
tar -tzf /backup/home-2026-05-01.tar.gz | wc -l

# Compare sizes
du -sh /home
ls -lh /backup/home-2026-05-01.tar.gz
Note: Compressed archive size will be smaller than source size. Track the ratio over time. A significant deviation from the normal ratio signals a problem.

3.4 Non-Destructive Test Restores

The most reliable verification method: actually restore files from the backup to a temporary location and confirm they are intact and usable.

Test restore from tar archive

# Create a temporary directory for the test restore
mkdir /tmp/test-restore

# Extract a specific file to the temp directory
tar -xzf /backup/home-2026-05-01.tar.gz -C /tmp/test-restore home/jdoe/important-doc.txt

# Verify the file is readable and correct
cat /tmp/test-restore/home/jdoe/important-doc.txt

# Clean up
rm -rf /tmp/test-restore

Test restore from rsync backup

# Copy a file from the rsync backup to a temp location
cp /backup/rsync/home/jdoe/important-doc.txt /tmp/test-restore-doc.txt

# Compare with original (if still available)
diff /home/jdoe/important-doc.txt /tmp/test-restore-doc.txt

3.5 Adding Verification to Backup Scripts

Integrate verification directly into the automated backup scripts from Module 15.3:

#!/bin/bash
# Excerpt: verification steps added to backup script

BACKUP_FILE="/backup/home-$(date +%Y-%m-%d).tar.gz"
CHECKSUM_FILE="${BACKUP_FILE}.sha256"
LOG_FILE="/var/log/backup/home-$(date +%Y-%m-%d).log"

# Run backup
tar -czf "$BACKUP_FILE" /home 2>&1 | tee "$LOG_FILE"

# Generate checksum
sha256sum "$BACKUP_FILE" > "$CHECKSUM_FILE"

# Verify checksum immediately
sha256sum -c "$CHECKSUM_FILE" >> "$LOG_FILE" 2>&1

# Check for errors in log
if grep -qi "error" "$LOG_FILE"; then
    echo "BACKUP VERIFICATION FAILED - check $LOG_FILE" | mail -s "Backup Alert" admin@example.com
fi

# Test restore of one file
mkdir -p /tmp/verify-restore
tar -xzf "$BACKUP_FILE" -C /tmp/verify-restore --include="home/testuser/.bashrc" 2>> "$LOG_FILE"
if [ $? -eq 0 ]; then
    echo "Test restore: SUCCESS" >> "$LOG_FILE"
else
    echo "Test restore: FAILED" >> "$LOG_FILE"
fi
rm -rf /tmp/verify-restore

4. System Recovery: Linux

Recovery procedures depend on the scope of the failure. The two primary scenarios are full system recovery (server is completely lost) and file-level recovery (individual files need restoration).

4.1 Full System Recovery

When a Linux server is completely lost (hardware failure, catastrophic corruption), the typical recovery process is:

  1. Reinstall the operating system — Use the same distribution and version as the original.
  2. Apply base configuration — Hostname, network settings, user accounts (from documentation or configuration management).
  3. Restore data from backup — Extract tar archives or rsync from backup storage.
  4. Restore application configurations — Copy configuration files from backup to their correct locations.
  5. Reinstall packages — Use a saved package list to reinstall required software.
  6. Verify services — Start services, confirm functionality, check logs.

Restoring from a tar archive (preserving permissions)

# Restore full /home directory from backup
# The -p flag preserves permissions; must run as root
sudo tar -xzpf /backup/home-2026-05-01.tar.gz -C /

# Restore /etc configuration files
sudo tar -xzpf /backup/etc-2026-05-01.tar.gz -C /

# Verify ownership and permissions
ls -la /home/
ls -la /etc/ssh/
Critical: Always use the -p (preserve permissions) flag when restoring system files. Without it, files may be restored with incorrect ownership or permissions, causing services to fail or creating security vulnerabilities.

Restoring from rsync backup

# Restore /home from rsync backup (preserving attributes)
sudo rsync -avz --progress /backup/rsync/home/ /home/

# Restore /etc from rsync backup
sudo rsync -avz --progress /backup/rsync/etc/ /etc/

4.2 File-Level Recovery

More commonly, you need to restore individual files (accidental deletion, corruption of specific files):

# Extract a single file from tar archive
sudo tar -xzpf /backup/home-2026-05-01.tar.gz -C / home/jdoe/report.docx

# Extract a directory
sudo tar -xzpf /backup/home-2026-05-01.tar.gz -C / home/jdoe/projects/

# From rsync backup — simply copy the file back
sudo cp -p /backup/rsync/home/jdoe/report.docx /home/jdoe/report.docx

4.3 Saving and Restoring Package Lists

# Save installed packages (Ubuntu/Debian)
dpkg --get-selections > /backup/package-list.txt

# Restore packages on new system
sudo dpkg --set-selections < /backup/package-list.txt
sudo apt-get dselect-upgrade

# Save installed packages (RHEL/Rocky Linux)
rpm -qa > /backup/package-list-rpm.txt

4.4 Runbook Documentation Format

Every recovery procedure should be documented in a runbook — a step-by-step guide that anyone on the team can follow under pressure. A good runbook includes:

Section Contents
System Identification Server name, IP, OS version, role/purpose
Prerequisites What you need before starting (install media, backup location, credentials)
Step-by-Step Procedure Numbered steps with exact commands; assume the reader is stressed
Verification Steps How to confirm each step succeeded before moving to the next
Estimated Time How long the full recovery takes (helps set expectations for RTO)
Contact Information Who to call if a step fails
Last Tested Date of most recent successful test of this procedure

5. System Recovery: Windows

Windows Server provides several recovery mechanisms depending on what was backed up and the scope of the failure.

5.1 Bare-Metal Recovery

A bare-metal recovery restores the entire server — operating system, configuration, and data — from a Windows Server Backup image to new or repaired hardware.

Requirements:

  • A bare-metal backup created with wbadmin (from Module 15.3)
  • Windows Server installation media (for booting into recovery)
  • Access to the backup storage location

Process Overview:

  1. Boot from Windows Server installation media.
  2. Select "Repair your computer" instead of "Install now."
  3. Choose Troubleshoot → System Image Recovery.
  4. Select the backup image location (local disk, network share).
  5. Confirm the restore and wait for completion.
  6. Reboot and verify all services are running.

5.2 System State Restore

A system state backup includes critical OS components without the full disk image:

  • Registry
  • Boot files
  • Active Directory database (on Domain Controllers)
  • SYSVOL (on Domain Controllers)
  • Certificate Services database (if installed)
  • COM+ class registration database

Restoring system state with wbadmin:

REM List available backup versions
wbadmin get versions -backupTarget:E:

REM Restore system state (requires reboot)
wbadmin start systemstaterecovery -version:05/01/2026-04:00 -backupTarget:E:
Important: System state restores on a Domain Controller require booting into Directory Services Restore Mode (DSRM) to avoid conflicts with other domain controllers.

5.3 File-Level Recovery

REM Recover specific files from a Windows Server Backup
wbadmin start recovery -version:05/01/2026-04:00 -itemType:File -items:D:\SharedDocs\report.docx -backupTarget:E: -recoveryTarget:D:\Restored\

5.4 Windows Recovery Environment (WinRE) Overview

WinRE is a minimal recovery environment built into Windows Server that provides access to recovery tools when the server cannot boot normally:

  • Startup Repair — Automatically fixes boot problems
  • Command Prompt — Manual repair tools (bootrec, bcdedit, diskpart)
  • System Image Recovery — Bare-metal restore from backup image
  • Uninstall Updates — Remove problematic updates that prevent booting

WinRE can be accessed by booting from installation media or, if configured, from a recovery partition on the server's disk.

6. DR Testing Cadence

A DR plan that has never been tested is a hypothesis, not a plan. Regular testing validates that recovery procedures work, identifies gaps, and builds team confidence.

6.1 Types of DR Tests

Test Type Description Effort Value
Tabletop Exercise Team walks through the DR plan verbally; no actual systems touched. "What would we do if…?" Low Identifies gaps in documentation, unclear roles, missing contacts
Partial Test Restore a single system or subset of data from backup to a test environment Medium Validates that specific backups are restorable; tests individual runbooks
Full DR Test Simulate a complete disaster; recover entire environment from backups at an alternate location or isolated network High Validates end-to-end recovery; measures actual RTO; highest confidence

6.2 Recommended Frequency

  • Tabletop exercises: Quarterly (every 3 months)
  • Partial tests (file/system restore): Monthly
  • Full DR tests: Annually (at minimum); semi-annually for critical environments
  • Automated verification checks: With every backup run (checksums, log review)

6.3 Documenting Test Results

Every DR test should produce a report that includes:

  • Date and scope — What was tested and when
  • Participants — Who was involved
  • Steps performed — What was done (with timestamps)
  • Results — What succeeded and what failed
  • Actual recovery time — Compare to RTO target
  • Issues discovered — Problems encountered during the test
  • Action items — What needs to be fixed or updated
  • DR plan updates — Changes made to the plan based on findings
Key Insight: A DR test that reveals problems is a successful test — you found the issues before a real disaster. The only failed DR test is one that never happens.

7. High Availability Concepts

While backup and DR protect against loss, high availability (HA) and fault tolerance (FT) aim to prevent downtime in the first place. Understanding where each strategy fits helps administrators design appropriate protection for different systems.

7.1 Availability Metrics and SLAs

Availability is typically expressed as a percentage of uptime over a given period (usually one year):

Availability % Common Name Downtime per Year Downtime per Month
99% "Two nines" 3.65 days 7.3 hours
99.9% "Three nines" 8.76 hours 43.8 minutes
99.99% "Four nines" 52.6 minutes 4.38 minutes
99.999% "Five nines" 5.26 minutes 26.3 seconds

Calculating availability:

$$\text{Availability \%} = \frac{\text{Total Time} - \text{Downtime}}{\text{Total Time}} \times 100$$

Service Level Agreements (SLAs) are contractual commitments specifying the availability level a service provider guarantees. Each additional "nine" typically requires exponentially more investment in redundancy and automation.

7.2 The FT → HA → DR Spectrum

These three strategies exist on a spectrum of protection:

Strategy Goal Scope Example
Fault Tolerance (FT) Zero downtime from component failure Single component or system RAID 1 mirror, redundant power supply
High Availability (HA) Minimal downtime (seconds to minutes) Service or application level Clustered servers with automatic failover
Disaster Recovery (DR) Recover from total site/system loss Entire environment Offsite backups, recovery site

Key insight: These strategies are complementary, not alternatives. A well-designed infrastructure uses all three at appropriate levels: FT for critical components, HA for critical services, and DR for catastrophic scenarios.

8. Redundancy & Failover

8.1 Hardware Redundancy

RAID (Review from Week 10)

RAID provides fault tolerance for storage — a disk can fail without data loss. But remember the critical distinction:

RAID is NOT backup. RAID protects against a single disk failure. It does NOT protect against accidental deletion, ransomware, corruption, or multi-disk failure. You still need backups.
  • RAID 1 — Mirroring (survives one disk failure)
  • RAID 5 — Striping with parity (survives one disk failure, better capacity)
  • RAID 10 — Mirrored stripes (survives multiple failures if in different mirrors)

Redundant Power Supplies

Enterprise servers typically have dual power supplies connected to separate circuits or UPS units. If one supply fails, the other continues powering the server with zero interruption.

8.2 Network Redundancy

NIC Teaming / Bonding

Combining multiple network interfaces into a single logical connection provides both redundancy (if one NIC fails, traffic continues on the other) and optionally increased bandwidth.

  • Linux: Network bonding (modes: active-backup, balance-rr, 802.3ad LACP)
  • Windows: NIC Teaming (Switch Independent, LACP)

Redundant Network Paths

Multiple switches, uplinks, and ISP connections ensure that no single network component failure isolates a server from the network.

8.3 Clustering (Active-Active vs. Active-Passive)

Clustering groups multiple servers to provide a single, highly available service:

Type How It Works Use Case
Active-Passive One node handles all traffic; standby node takes over if the primary fails Database servers, file servers where simplicity is valued
Active-Active All nodes handle traffic simultaneously; if one fails, remaining nodes absorb its load Web servers, application servers needing both HA and load distribution

Failover and Failback

  • Failover: The automatic or manual transfer of services from a failed node to a standby or surviving node.
  • Failback: Returning services to the original (repaired) node after the failure is resolved.
8.4 Replication

Replication copies data between systems in real-time or near-real-time:

Type Behavior RPO Trade-off
Synchronous Write is confirmed only after both primary and replica acknowledge it Zero (no data loss) Higher latency; requires low-latency connection between sites
Asynchronous Write is confirmed on primary; replica is updated shortly after Near-zero (seconds to minutes of potential loss) Lower latency; works over longer distances

Replication connects directly to RPO concepts from Module 15.1: synchronous replication achieves RPO = 0, while asynchronous replication provides RPO equal to the replication lag.

8.5 Load Balancing & Geographic Redundancy (Awareness)

Load Balancing

A load balancer distributes incoming requests across multiple servers. If one server fails health checks, the load balancer stops sending traffic to it — providing both performance and availability benefits.

Geographic Redundancy

For protection against site-level disasters (earthquake, flood, extended power outage), organizations maintain infrastructure at geographically separated locations. This is the ultimate DR strategy but carries significant cost and complexity (data synchronization, network latency, regulatory considerations).

9. Splunk Recovery Scenario

In Module 15.3, you learned to back up Splunk configurations ($SPLUNK_HOME/etc/). Now let's walk through what happens when you need to use that backup — recovering a Splunk Enterprise instance from scratch.

Scenario

Your Splunk Enterprise server has suffered a catastrophic disk failure. The operating system and all Splunk data are lost. You have: (1) a recent backup of $SPLUNK_HOME/etc/, (2) your Universal Forwarders on other servers are still running, and (3) your original Splunk installer package.

Recovery Steps

Step 1: Reinstall the Operating System

# Install the same Linux distribution and version as the original
# Configure networking (same IP address so forwarders can reconnect)
# Apply base security hardening (Week 14 content)

Step 2: Reinstall Splunk Enterprise

# Download or retrieve the Splunk Enterprise installer
sudo dpkg -i splunk-9.x.x-linux-amd64.deb    # Debian/Ubuntu
# OR
sudo rpm -i splunk-9.x.x-linux-x86_64.rpm     # RHEL/Rocky

# Accept license and start Splunk
sudo /opt/splunk/bin/splunk start --accept-license

Step 3: Stop Splunk and Restore Configuration

# Stop Splunk before overwriting configurations
sudo /opt/splunk/bin/splunk stop

# Restore the backed-up etc/ directory
sudo tar -xzpf /backup/splunk-etc-2026-05-01.tar.gz -C /opt/splunk/

# Verify the restore
ls /opt/splunk/etc/system/local/
ls /opt/splunk/etc/apps/

Step 4: Restart Splunk and Verify

# Start Splunk with restored configuration
sudo /opt/splunk/bin/splunk start

# Verify receiving port is active
sudo /opt/splunk/bin/splunk display listen

# Check Splunk Web UI is accessible
# Navigate to http://splunk-server:8000

Step 5: Verify Forwarder Reconnection

# On the Splunk server, search for incoming data
# In Splunk Web: Search & Reporting
# Run: index=* | stats count by host

# If forwarders used the server's IP/hostname, they should
# automatically reconnect and resume sending data

Step 6: Reconfigure Forwarders (if needed)

# If the Splunk server IP or hostname changed, update forwarders:
# On each forwarder (Linux example):
sudo /opt/splunkforwarder/bin/splunk add forward-server new-splunk-ip:9997

# Restart the forwarder
sudo /opt/splunkforwarder/bin/splunk restart

Index Data Recovery Considerations

Configuration vs. Index Data:
  • Configurations ($SPLUNK_HOME/etc/) — Small (typically MBs), critical, easy to back up. This is what we restored above.
  • Index data ($SPLUNK_HOME/var/lib/splunk/) — Large (potentially GBs to TBs), may not be practical to back up.

Options for index data:

  • Re-index from sources: If forwarders are still running and source logs exist on the original hosts, Splunk can re-ingest historical data (if inputs.conf is configured with appropriate checkpoints).
  • Accept data loss: For many environments, losing historical index data is acceptable if new data starts flowing immediately after recovery.
  • Restore from backup: If index data was backed up (large storage requirement), restore it alongside the configuration.

Test your understanding with these ungraded questions. Click "Check Answer" to see feedback.

1. Which verification method provides the strongest evidence that a backup can actually be used for recovery?

✓ Correct! A test restore is the most reliable verification because it proves the data can actually be extracted and used. Checksums confirm the file isn't corrupted, and log review confirms the job completed, but only a test restore proves the backup is functionally usable.
✗ Not quite. While checksums and log reviews are valuable (and should be done!), a test restore (option c) provides the strongest evidence because it proves the data can actually be extracted and used in a recovery scenario. The other methods detect specific failure modes but don't confirm end-to-end recoverability.

2. Place these Linux full-system recovery steps in the correct order:

A) Restore data from backup   B) Verify services are running   C) Reinstall the operating system   D) Reinstall required packages

✓ Correct! The order is: Reinstall OS (C) → Restore data including configurations (A) → Reinstall packages using the restored package list (D) → Verify services (B). You need the OS first, then the data/configs that tell you what packages to install, then verification.
✗ Not quite. The correct order is C → A → D → B: Reinstall the OS first (you need a running system), then restore data and configuration files (which include the package list), then reinstall packages, and finally verify everything works. You cannot restore data without an OS to restore it to.

3. A service guarantees 99.9% availability. How much downtime per year does this allow?

✓ Correct! 99.9% availability (three nines) allows 0.1% downtime per year. 0.001 × 8,760 hours/year = 8.76 hours of allowed downtime per year.
✗ Not quite. 99.9% availability allows 8.76 hours of downtime per year. The calculation: 8,760 hours/year × 0.001 = 8.76 hours. Remember: 99.99% = 52.6 min, 99.999% = 5.26 min, and 99% = 3.65 days.

4. An organization needs its web server to continue operating even if one server fails, with automatic transfer of traffic to a standby server. Which concept best describes this requirement?

✓ Correct! This describes High Availability with failover — multiple servers providing the same service with automatic transfer when one fails. Fault tolerance typically refers to component-level redundancy (like RAID or dual power supplies) within a single system. DR addresses recovery from larger-scale disasters, not single-server failures.
✗ Not quite. The correct answer is High Availability with failover (option c). This scenario describes service-level redundancy with automatic transfer — the hallmark of HA. Fault tolerance (b) is component-level (RAID, dual PSU). DR (a) addresses site-level or catastrophic failures. Backup verification (d) is about testing backups, not preventing downtime.

5. When recovering a Splunk Enterprise instance from configuration backups, which directory contains the critical configuration data that should be restored?

✓ Correct! $SPLUNK_HOME/etc/ contains all Splunk configuration files (inputs.conf, outputs.conf, apps, saved searches, dashboards, alerts). This is the critical directory to back up and restore. The var/lib/splunk/ directory contains index data (large, optional to back up), and bin/ contains the Splunk binaries (reinstalled fresh).
✗ Not quite. The correct answer is $SPLUNK_HOME/etc/ (option b). This directory contains all configuration files, apps, saved searches, dashboards, and alerts. var/lib/splunk/ is index data (large, often not backed up). bin/ is the Splunk software itself (reinstalled fresh). var/log/ is Splunk's internal logs.

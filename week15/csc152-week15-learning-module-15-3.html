Linux: Backup with tar

tar (tape archive) is the foundational Linux backup tool. It bundles files and directories into a single archive file, optionally compressed, preserving permissions, ownership, and timestamps.

Creating Compressed Archives

The most common backup operation creates a gzip-compressed archive of a directory:

# Create a gzip-compressed archive of /etc
# -c = create, -z = gzip compression, -f = filename
# Best practice: include a timestamp in the filename
tar -czf /backup/etc-backup-$(date +%Y%m%d-%H%M%S).tar.gz /etc

For better compression (slower but smaller files), use bzip2:

# -j = bzip2 compression (better ratio, slower speed)
tar -cjf /backup/etc-backup-$(date +%Y%m%d-%H%M%S).tar.bz2 /etc

Key Options for Backup Use

OptionPurposeExample
-cCreate a new archivetar -cf archive.tar /data
-zCompress with gzip (.tar.gz)tar -czf archive.tar.gz /data
-jCompress with bzip2 (.tar.bz2)tar -cjf archive.tar.bz2 /data
-fSpecify archive filenameAlways required
-vVerbose (list files processed)Useful for debugging, noisy for cron
-pPreserve permissionsImportant for system backups
--excludeSkip files/directories--exclude='*.tmp'

Extracting Archives (Restore)

# Extract to the current directory
tar -xzf /backup/etc-backup-20260428-020000.tar.gz

# Extract to a specific directory (useful for test restores)
tar -xzf /backup/etc-backup-20260428-020000.tar.gz -C /tmp/restore-test/

# Extract a single file from the archive
tar -xzf /backup/etc-backup-20260428-020000.tar.gz etc/hostname

Verifying Archive Contents

Always verify backups. An archive that cannot be read is worthless.

# List contents without extracting (-t = list)
tar -tzf /backup/etc-backup-20260428-020000.tar.gz

# List with details (permissions, sizes, timestamps)
tar -tzvf /backup/etc-backup-20260428-020000.tar.gz

# Test archive integrity (gzip test)
gzip -t /backup/etc-backup-20260428-020000.tar.gz
echo "Exit code: $?"  # 0 = OK, non-zero = corrupted

Naming Conventions

💡 Best Practice: Timestamp-Based Naming

Always include the date (and optionally time) in backup filenames. This makes rotation and identification trivial:

systemname-backuptype-YYYYMMDD-HHMMSS.tar.gz

Example: webserver-full-20260428-020000.tar.gz

Linux: Backup with rsync

rsync takes a fundamentally different approach from tar. Instead of creating archive files, rsync synchronizes directory trees—copying only the differences (deltas) between source and destination. This makes it extremely efficient for incremental backups.

How rsync Works

  • Delta transfer algorithm: rsync compares files by size and modification time (or checksum) and transfers only changed portions
  • First run: copies everything (like a full backup)
  • Subsequent runs: copies only changes (like an incremental backup)
  • Result: the destination is always a complete, usable mirror

Local rsync Backup

# Basic local sync: mirror /data to /backup/data-mirror
# -a = archive mode (preserves permissions, ownership, timestamps, symlinks)
# -v = verbose, -z = compress during transfer (useful for network)
rsync -av /data/ /backup/data-mirror/

# With --delete: remove files from destination that no longer exist in source
# WARNING: --delete makes destination an exact mirror; deleted source files are gone
rsync -av --delete /data/ /backup/data-mirror/

# Exclude patterns
rsync -av --exclude='*.tmp' --exclude='.cache/' /data/ /backup/data-mirror/

⚠️ Trailing Slash Matters

rsync /data/ /backup/ copies the contents of /data into /backup. rsync /data /backup/ (no trailing slash on source) copies the directory itself, creating /backup/data/. For backups, you usually want the trailing slash on the source.

rsync Over SSH (Remote/Offsite Backups)

# Push local data to a remote backup server
rsync -avz /data/ backupuser@backup-server:/backup/webserver/data/

# Pull data from a remote server to local backup storage
rsync -avz backupuser@production-server:/data/ /backup/production/data/

# Specify SSH port if non-standard
rsync -avz -e 'ssh -p 2222' /data/ backupuser@backup-server:/backup/data/

Space-Efficient Incrementals with --link-dest

The --link-dest option creates hard links to unchanged files from a previous backup, giving you what looks like a full backup for each run while using disk space only for changed files:

# Each daily backup appears complete, but unchanged files are hard-linked
# to the previous backup (using minimal additional space)

BACKUP_BASE="/backup/daily"
TODAY=$(date +%Y%m%d)
YESTERDAY=$(date -d "yesterday" +%Y%m%d)

rsync -av --delete \
  --link-dest="$BACKUP_BASE/$YESTERDAY" \
  /data/ \
  "$BACKUP_BASE/$TODAY/"

tar vs. rsync: When to Use Each

Linux: Automating with cron

A backup that requires manual execution will eventually be forgotten. cron is the standard Linux job scheduler that runs commands at specified times automatically.

cron Syntax

# ┌───────────── minute (0-59)
# │ ┌───────────── hour (0-23)
# │ │ ┌───────────── day of month (1-31)
# │ │ │ ┌───────────── month (1-12)
# │ │ │ │ ┌───────────── day of week (0-7, 0 and 7 = Sunday)
# │ │ │ │ │
# * * * * * command-to-execute

# Examples:
0 2 * * *     # Every day at 2:00 AM
0 2 * * 0     # Every Sunday at 2:00 AM
30 3 1 * *    # 1st of every month at 3:30 AM
0 */6 * * *   # Every 6 hours (0:00, 6:00, 12:00, 18:00)
0 2 * * 1-5   # Weekdays at 2:00 AM

Creating Backup cron Jobs

# Edit the root user's crontab
sudo crontab -e

# Add a daily backup job at 2:00 AM
0 2 * * * /usr/local/bin/backup-script.sh

# View current cron jobs
sudo crontab -l

Environment Considerations

⚠️ cron Runs in a Minimal Environment

cron jobs do not load your interactive shell's environment (no .bashrc, no full PATH). Common pitfalls:

  • PATH: Use full paths to commands (e.g., /usr/bin/tar not just tar) or set PATH at the top of your script
  • Output: Redirect stdout and stderr or cron will attempt to email output (which often fails): command >> /var/log/backup.log 2>&1
  • Working directory: cron starts in the user's home directory; use cd or absolute paths
# Recommended crontab entry with logging
0 2 * * * /usr/local/bin/backup-script.sh >> /var/log/backup-cron.log 2>&1

Verifying cron Execution

# Check system log for cron activity
grep CRON /var/log/syslog | tail -20

# Or on RHEL/CentOS:
grep CRON /var/log/cron | tail -20

Linux: Complete Bash Backup Script

The following annotated script combines everything covered above into a production-ready backup solution. You can adapt this script for your Project 3 deliverable.

#!/bin/bash
#=============================================================
# backup-system.sh - Automated System Backup Script
# Purpose: Create compressed backups with rotation, logging,
#          verification, and basic error handling
# Author:  [Your Name]
# Date:    [Date]
# Usage:   Run via cron: 0 2 * * * /usr/local/bin/backup-system.sh
#=============================================================

#-------------------------------------------------------------
# CONFIGURATION VARIABLES
# Modify these to match your environment
#-------------------------------------------------------------
BACKUP_SOURCE="/etc /home /var/log"        # Directories to back up
BACKUP_DEST="/backup"                       # Where backups are stored
BACKUP_PREFIX="server01-full"               # Naming prefix
RETENTION_DAYS=7                            # Delete backups older than this
MIN_SPACE_MB=500                            # Minimum free space required (MB)
LOG_FILE="/var/log/backup-system.log"       # Log file location
TIMESTAMP=$(date +%Y%m%d-%H%M%S)           # Current timestamp
ARCHIVE_NAME="${BACKUP_PREFIX}-${TIMESTAMP}.tar.gz"
ARCHIVE_PATH="${BACKUP_DEST}/${ARCHIVE_NAME}"

#-------------------------------------------------------------
# LOGGING FUNCTION
# Writes timestamped messages to log file and stdout
#-------------------------------------------------------------
log_message() {
    local MESSAGE="$1"
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $MESSAGE" | tee -a "$LOG_FILE"
}

#-------------------------------------------------------------
# ERROR HANDLING
# Exit cleanly on errors and log the failure
#-------------------------------------------------------------
handle_error() {
    log_message "ERROR: $1"
    log_message "Backup FAILED. Exiting."
    exit 1
}

#-------------------------------------------------------------
# START BACKUP PROCESS
#-------------------------------------------------------------
log_message "========================================="
log_message "Starting backup: $ARCHIVE_NAME"
log_message "Source: $BACKUP_SOURCE"
log_message "Destination: $BACKUP_DEST"

#-------------------------------------------------------------
# PRE-BACKUP CHECK: Verify backup destination exists
#-------------------------------------------------------------
if [ ! -d "$BACKUP_DEST" ]; then
    handle_error "Backup destination $BACKUP_DEST does not exist"
fi

#-------------------------------------------------------------
# PRE-BACKUP CHECK: Verify sufficient disk space
#-------------------------------------------------------------
AVAILABLE_MB=$(df -BM "$BACKUP_DEST" | awk 'NR==2 {print int($4)}')
log_message "Available space: ${AVAILABLE_MB}MB (minimum required: ${MIN_SPACE_MB}MB)"

if [ "$AVAILABLE_MB" -lt "$MIN_SPACE_MB" ]; then
    handle_error "Insufficient disk space: ${AVAILABLE_MB}MB available, ${MIN_SPACE_MB}MB required"
fi

#-------------------------------------------------------------
# EXECUTE BACKUP
#-------------------------------------------------------------
log_message "Creating archive: $ARCHIVE_PATH"

tar -czf "$ARCHIVE_PATH" $BACKUP_SOURCE 2>> "$LOG_FILE"
TAR_EXIT_CODE=$?

if [ $TAR_EXIT_CODE -ne 0 ]; then
    # Exit code 1 = files changed during archive (often acceptable)
    # Exit code 2 = fatal error
    if [ $TAR_EXIT_CODE -eq 2 ]; then
        handle_error "tar encountered a fatal error (exit code: $TAR_EXIT_CODE)"
    else
        log_message "WARNING: tar exited with code $TAR_EXIT_CODE (files may have changed during backup)"
    fi
fi

#-------------------------------------------------------------
# POST-BACKUP VERIFICATION
#-------------------------------------------------------------
# Check that the archive file exists and has a reasonable size
if [ ! -f "$ARCHIVE_PATH" ]; then
    handle_error "Archive file not found after backup: $ARCHIVE_PATH"
fi

ARCHIVE_SIZE=$(stat --format="%s" "$ARCHIVE_PATH")
ARCHIVE_SIZE_MB=$((ARCHIVE_SIZE / 1024 / 1024))
log_message "Archive size: ${ARCHIVE_SIZE_MB}MB ($ARCHIVE_SIZE bytes)"

if [ "$ARCHIVE_SIZE" -lt 1024 ]; then
    handle_error "Archive suspiciously small (${ARCHIVE_SIZE} bytes) - possible failure"
fi

# Verify archive integrity
gzip -t "$ARCHIVE_PATH" 2>> "$LOG_FILE"
if [ $? -ne 0 ]; then
    handle_error "Archive integrity check FAILED for $ARCHIVE_PATH"
fi
log_message "Archive integrity verified (gzip -t passed)"

#-------------------------------------------------------------
# ROTATION: Delete old backups beyond retention period
#-------------------------------------------------------------
log_message "Removing backups older than $RETENTION_DAYS days..."
DELETED_COUNT=$(find "$BACKUP_DEST" -name "${BACKUP_PREFIX}-*.tar.gz" -mtime +$RETENTION_DAYS -delete -print | wc -l)
log_message "Deleted $DELETED_COUNT old backup(s)"

#-------------------------------------------------------------
# COMPLETION SUMMARY
#-------------------------------------------------------------
REMAINING_COUNT=$(find "$BACKUP_DEST" -name "${BACKUP_PREFIX}-*.tar.gz" | wc -l)
log_message "Backup completed successfully"
log_message "Archive: $ARCHIVE_NAME (${ARCHIVE_SIZE_MB}MB)"
log_message "Backups on disk: $REMAINING_COUNT"
log_message "========================================="

exit 0

Making the Script Operational

# 1. Save the script
sudo nano /usr/local/bin/backup-system.sh

# 2. Make it executable
sudo chmod 750 /usr/local/bin/backup-system.sh

# 3. Create the backup destination and log file
sudo mkdir -p /backup
sudo touch /var/log/backup-system.log

# 4. Test manually first (always test before scheduling!)
sudo /usr/local/bin/backup-system.sh

# 5. Check the log
cat /var/log/backup-system.log

# 6. Schedule with cron (daily at 2:00 AM)
sudo crontab -e
# Add: 0 2 * * * /usr/local/bin/backup-system.sh >> /var/log/backup-cron.log 2>&1

Windows: wbadmin & Volume Shadow Copy Service (VSS)

Windows Server Backup (wbadmin)

wbadmin is the command-line interface to Windows Server Backup, the native backup solution included with Windows Server. It supports full server, system state, and file/folder backups.

Installing the Feature

# Install Windows Server Backup feature (PowerShell, run as Administrator)
Install-WindowsFeature Windows-Server-Backup -IncludeManagementTools

Common wbadmin Commands

# Full server backup (bare-metal recovery capable) to a volume
wbadmin start backup -backupTarget:E: -include:C: -allCritical -quiet

# System state backup (AD, registry, boot files, system files)
wbadmin start systemstatebackup -backupTarget:E: -quiet

# Backup specific volumes
wbadmin start backup -backupTarget:E: -include:C:,D: -quiet

# List available backups
wbadmin get versions

# Get details of a specific backup
wbadmin get items -version:05/01/2026-02:00
ParameterPurpose
-backupTarget:Destination volume or network share
-include:Volumes or paths to include
-allCriticalInclude all critical volumes for bare-metal recovery
-quietSuppress prompts (required for scripting)
-vssFullFull VSS backup (updates backup history on files)

Volume Shadow Copy Service (VSS)

VSS solves a fundamental problem: how do you back up files that are open and being written to?

  • The problem: If you copy a database file while the application is writing to it, the copy may be internally inconsistent (corrupted)
  • VSS solution: VSS coordinates with applications ("VSS Writers") to create a point-in-time consistent snapshot before the backup reads the data
  • How it works:
    1. Backup application requests a shadow copy
    2. VSS tells applications to flush buffers and pause writes
    3. A point-in-time snapshot is created (nearly instantaneous)
    4. Applications resume normal operation
    5. Backup reads from the consistent snapshot
  • Key point: wbadmin uses VSS automatically—you get application-consistent backups without stopping services

💡 VSS = Consistent Backups of Open Files

Without VSS, backing up an active SQL Server database or Exchange mailbox by copying files would produce a corrupted backup. VSS ensures the backup captures a consistent state even while the application is running. This is the Windows equivalent of LVM snapshots on Linux.

Windows: PowerShell & Task Scheduler

PowerShell Backup Script

# backup-server.ps1 - Windows Server Backup Script
# Schedule with Task Scheduler for automated execution

# --- Configuration ---
$BackupTarget = "E:"
$LogFile = "C:\BackupLogs\backup-$(Get-Date -Format 'yyyyMMdd-HHmmss').log"
$RetentionDays = 7

# --- Ensure log directory exists ---
$LogDir = Split-Path $LogFile
if (-not (Test-Path $LogDir)) { New-Item -ItemType Directory -Path $LogDir -Force }

# --- Logging function ---
function Write-Log {
    param([string]$Message)
    $Entry = "[$(Get-Date -Format 'yyyy-MM-dd HH:mm:ss')] $Message"
    Add-Content -Path $LogFile -Value $Entry
    Write-Host $Entry
}

# --- Start backup ---
Write-Log "Starting Windows Server Backup to $BackupTarget"

try {
    # Run wbadmin for system state + critical volumes
    $Result = wbadmin start backup -backupTarget:$BackupTarget -allCritical -quiet 2>&1
    $ExitCode = $LASTEXITCODE

    if ($ExitCode -eq 0) {
        Write-Log "Backup completed successfully"
    } else {
        Write-Log "ERROR: wbadmin exited with code $ExitCode"
        Write-Log "Output: $Result"
    }
} catch {
    Write-Log "EXCEPTION: $($_.Exception.Message)"
}

# --- Clean old logs ---
Get-ChildItem -Path $LogDir -Filter "backup-*.log" |
    Where-Object { $_.LastWriteTime -lt (Get-Date).AddDays(-$RetentionDays) } |
    Remove-Item -Force

Write-Log "Backup process complete"

Scheduling with Task Scheduler

Task Scheduler is the Windows equivalent of cron. You can configure it via GUI or PowerShell:

# Create a scheduled task to run the backup script daily at 2:00 AM

$Action = New-ScheduledTaskAction `
    -Execute "PowerShell.exe" `
    -Argument "-NoProfile -ExecutionPolicy Bypass -File C:\Scripts\backup-server.ps1"

$Trigger = New-ScheduledTaskTrigger -Daily -At "2:00AM"

$Settings = New-ScheduledTaskSettingsSet `
    -StartWhenAvailable `
    -DontStopOnIdleEnd `
    -RestartCount 3 `
    -RestartInterval (New-TimeSpan -Minutes 5)

Register-ScheduledTask `
    -TaskName "DailyServerBackup" `
    -Action $Action `
    -Trigger $Trigger `
    -Settings $Settings `
    -User "SYSTEM" `
    -RunLevel Highest `
    -Description "Automated daily server backup via wbadmin"

Run with highest privilegesRequired for system-level backupsRequired for wbadmin

Application-Aware Backups

Standard file-level backups (tar, rsync, file copy) work well for static files—configuration files, documents, web content. But they can produce inconsistent backups for applications that maintain complex internal state, particularly databases.

The Problem: Why File-Level Backups May Be Insufficient

  • A database engine keeps data in memory buffers and writes to disk asynchronously
  • Copying database files while the engine is running may capture a partially-written transaction
  • The resulting backup file may be internally inconsistent and unrestorable
  • This applies to: databases (MySQL, PostgreSQL, SQL Server), email servers, directory services (Active Directory), and any application with write-ahead logs

Solution 1: LVM Snapshots (Linux)

🔗 Connection to Week 10

In Week 10, you learned about Logical Volume Manager (LVM) and how it provides flexible storage management. LVM snapshots leverage that same technology to create point-in-time copies for backup purposes.

An LVM snapshot captures the state of a logical volume at a specific instant. The application continues running while you back up from the frozen snapshot:

# 1. Create a snapshot of the data volume (requires free space in VG)
sudo lvcreate --size 5G --snapshot --name data-snap /dev/vg01/lv-data

# 2. Mount the snapshot read-only
sudo mount -o ro /dev/vg01/data-snap /mnt/snapshot

# 3. Back up from the snapshot (consistent point-in-time copy)
tar -czf /backup/data-snapshot-$(date +%Y%m%d).tar.gz /mnt/snapshot/

# 4. Unmount and remove the snapshot
sudo umount /mnt/snapshot
sudo lvremove -f /dev/vg01/data-snap

Key points:

  • The snapshot is nearly instantaneous (copy-on-write mechanism)
  • The original volume continues operating normally
  • The snapshot only uses space for blocks that change after creation
  • For database consistency, briefly pause writes (flush) before snapshotting

Solution 2: VSS Application-Aware Snapshots (Windows)

As covered in the Windows section, VSS coordinates with application-specific "writers" to ensure consistency:

  • SQL Server VSS Writer: ensures database transactions are complete before snapshot
  • Active Directory VSS Writer: ensures AD database consistency
  • Exchange VSS Writer: ensures mailbox database consistency
  • wbadmin invokes VSS automatically—no extra steps required

Solution 3: Database Dump Utilities (Awareness Level)

Database dump utilities export data in a logical format that is always consistent, regardless of ongoing writes:

# MySQL/MariaDB: export all databases to a SQL dump file
mysqldump --all-databases --single-transaction > /backup/mysql-all-$(date +%Y%m%d).sql

# PostgreSQL: export all databases
pg_dumpall > /backup/postgres-all-$(date +%Y%m%d).sql

# These produce text files containing SQL statements to recreate the data
# --single-transaction ensures a consistent snapshot without locking tables

💡 When to Use Each Approach

  • LVM Snapshots: Fast, works for any application on the volume, good for large databases where dump is too slow
  • VSS (Windows): Automatic with wbadmin, application-aware, no extra scripting needed
  • Database Dumps: Portable, human-readable, easy to restore to different servers, but slower for large databases
  • For Project 3: If you have no databases running, standard file-level backups (tar/rsync) are appropriate. Document why file-level is sufficient for your environment.

Backing Up Splunk Configurations

Your Splunk deployment from Weeks 12–14 is critical infrastructure for Project 3. Losing your Splunk configuration means losing your dashboards, alerts, saved searches, forwarder configurations, and index definitions. This section covers what to back up, how to back it up, and how to recover.

Splunk Data Components: What Needs Backup?

ComponentLocationSizeBackup Priority
Configurations$SPLUNK_HOME/etc/Small (MBs)Critical – always back up
Index data$SPLUNK_HOME/var/lib/splunk/Large (GBs+)Conditional – see below
Splunk binaries$SPLUNK_HOME/bin/, etc.MediumLow – reinstall from package

What lives in $SPLUNK_HOME/etc/?

  • etc/system/local/ – Your custom system-level configurations (inputs.conf, outputs.conf, server.conf, etc.)
  • etc/apps/ – Installed apps, dashboards, saved searches, alert definitions
  • etc/users/ – Per-user settings, private saved searches
  • etc/deployment-apps/ – Configurations pushed to forwarders (if using deployment server)
  • etc/auth/ – Authentication configurations and certificates

Index Data: Back Up or Not?

  • Index data can be very large (gigabytes to terabytes in production)
  • In a lab environment, index data may be small enough to include in backups
  • In production, rely on retention policies and the ability to re-index from sources rather than backing up index data
  • For Project 3: backing up configurations is sufficient; document your reasoning for not backing up index data (if applicable)

How to Back Up Splunk Configurations

# Typical Splunk installation path (Linux)
SPLUNK_HOME="/opt/splunk"

# Create a compressed backup of all Splunk configurations
tar -czf /backup/splunk-etc-$(date +%Y%m%d-%H%M%S).tar.gz \
    -C "$SPLUNK_HOME" etc/

# Or use rsync for incremental configuration backup
rsync -av "$SPLUNK_HOME/etc/" /backup/splunk-etc-mirror/

# For Universal Forwarders (on monitored hosts)
SPLUNK_HOME="/opt/splunkforwarder"
tar -czf /backup/splunk-fwd-etc-$(date +%Y%m%d).tar.gz \
    -C "$SPLUNK_HOME" etc/

💡 Integration with Your Backup Script

Add Splunk configuration backup to the Bash backup script from the previous section. Simply add $SPLUNK_HOME/etc to your BACKUP_SOURCE variable, or create a separate Splunk-specific backup job.

Recovery Scenario: Rebuilding Splunk from Configuration Backup

Imagine your Splunk server's disk fails completely. Here is the recovery procedure using your configuration backup:

Step-by-Step Splunk Recovery Walkthrough
  1. Rebuild the base system: Reinstall the OS (Linux) and apply your standard hardening (documented in Project 3)
  2. Install Splunk Enterprise: Download and install the same version of Splunk Enterprise 
    sudo dpkg -i splunk-enterprise-*.deb
# or: sudo rpm -i splunk-enterprise-*.rpm
  3. Stop Splunk before restoring configurations: 
    sudo /opt/splunk/bin/splunk stop
  4. Restore configuration backup: 
    # Remove the default etc/ that was just installed
sudo rm -rf /opt/splunk/etc

# Restore your backed-up configurations
sudo tar -xzf /backup/splunk-etc-20260428-020000.tar.gz -C /opt/splunk/

# Fix ownership
sudo chown -R splunk:splunk /opt/splunk/etc
  5. Start Splunk: 
    sudo /opt/splunk/bin/splunk start --accept-license
  6. Verify recovery:
    • Log into Splunk Web – confirm dashboards and saved searches are present
    • Check that forwarders reconnect and data begins flowing
    • Verify alerts are active
    • Index data will rebuild as forwarders resend (or from monitored files)

Key insight: Because Splunk configurations are small (megabytes), backing them up is cheap and fast. Recovery is straightforward—reinstall Splunk, restore etc/, and start the service. Your dashboards, alerts, and all customizations return immediately.

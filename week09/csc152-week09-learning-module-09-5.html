🔍 1 · Why Centralized Logging?

Every server, firewall, and network device you manage generates log events — records of what happened, when, and to whom. In a small network with two or three devices, reading logs individually is manageable. In any real organization, it is not.

The Problem with Local Log Review

Imagine your network has 10 servers, a firewall, three switches, and a wireless controller. Each device writes its own log file in its own format. To answer the question "Was there any suspicious activity last night?" you would need to:

  • Log in to each device individually
  • Navigate to the correct log location on each
  • Search through potentially thousands of lines per device
  • Mentally correlate events across systems — without a shared timeline

This approach does not scale, is error-prone, and is far too slow for incident response. Centralized logging solves all of these problems.

The Three Core Benefits

Single Pane of Glass

A centralized log platform aggregates events from every device into one searchable interface. Instead of logging into 10 systems, an administrator runs a single search and sees correlated events from all sources on a unified timeline.

This is sometimes called a Single Pane of Glass — one view that shows the entire environment. Splunk is one of the industry's leading platforms for this purpose. Others include the ELK Stack (Elasticsearch, Logstash, Kibana) and Graylog.

📌 Week 9 Context
In this module you will see pfSense firewall events, DHCP lease events, and general system events all flowing into a single Splunk search interface — that is the single pane of glass in action.

Audit Trails

An audit trail is a chronological record of activity that can be used to verify compliance, reconstruct events, or prove (or disprove) that a specific action occurred. Centralized logging creates tamper-evident audit trails because log data is forwarded off the originating device — even if an attacker compromises a server and deletes local logs, the centralized copy remains.

Audit trails matter for:

  • Compliance frameworks (PCI-DSS, HIPAA, SOC 2) that require log retention and review
  • Change management — who changed what firewall rule and when?
  • HR and legal investigations — documenting user activity on corporate systems
  • Post-incident forensics — reconstructing the sequence of events after a breach

Incident Response

When a security incident occurs, time is critical. Centralized logging dramatically reduces Mean Time to Detect (MTTD) and Mean Time to Respond (MTTR) by allowing analysts to:

  • Search across all systems simultaneously for indicators of compromise (IoCs)
  • Correlate a suspicious IP address seen in firewall logs with failed login attempts in authentication logs
  • Build timelines showing the progression of an attack across multiple systems
  • Set up alerts that fire automatically when specific patterns (e.g., 10 failed logins in 5 minutes) are detected
🔒 CSLO 2 Connection
Centralized logging directly supports CSLO 2 — Apply physical and network data security techniques. A SIEM (Security Information and Event Management) platform like Splunk is a foundational security control in enterprise environments.

Does Centralized Logging Replace Local Log Review?

No — it supplements it. Centralized logging is a powerful force multiplier, but local logs still matter for troubleshooting service-specific issues, for systems that cannot forward logs, and as a cross-check when the centralized platform itself has issues. You will see this concept tested in the Knowledge Check at the end of this module.

📡 2 · Syslog Protocol

Syslog is the standard protocol used by network devices, servers, and appliances to send log messages to a central collector. It was originally defined in RFC 3164 and updated in RFC 5424. Almost every network device you will encounter — routers, switches, firewalls, Linux servers — supports syslog natively.

How Syslog Works

A device configured with a remote syslog destination sends log messages as UDP or TCP datagrams to a syslog server (also called a syslog collector or receiver) listening on port 514.

Syslog Message Structure

Every syslog message contains three key components:

ComponentDescriptionExample
Priority (PRI) Encoded combination of Facility (what generated the message) and Severity (how important it is). Written as a number in angle brackets. <134>
Header Timestamp and hostname of the sending device. Mar 20 14:32:01 pfsense
Message The actual log content — process name, PID, and the event description. filterlog: 5,16,,...,block,in,...

Syslog Severity Levels

Syslog defines 8 severity levels (0 = most critical, 7 = least):

LevelKeywordMeaning
0EMERGSystem is unusable
1ALERTImmediate action required
2CRITCritical condition
3ERRError condition
4WARNINGWarning — may indicate a problem
5NOTICENormal but significant event
6INFOInformational message
7DEBUGDebug-level detail

UDP vs. TCP for Syslog

TransportPortReliable?When to Use
UDP 514 No — messages can be lost if the network is congested Default for most devices; acceptable for high-volume, low-criticality logs
TCP 514 (or 601) Yes — connection-oriented, delivery confirmed Compliance environments where log completeness is required

pfSense as a Syslog Sender

pfSense can forward its firewall filter logs, DHCP events, system events, and more to a remote syslog server. The forwarded messages use the filterlog format for firewall events, which Splunk can parse using the pfsense sourcetype add-on. You will configure this forwarding in Section 5.

💡 Port 514 — Know This
The default syslog port is 514 (UDP or TCP). This is a frequently tested fact in Quiz 9 and on the final exam. It also appears in the Knowledge Check at the end of this module.

Component Descriptions

Forwarders

A Forwarder is a lightweight Splunk agent installed on a data source (e.g., a Linux or Windows server) that collects log data and sends it to an Indexer. The most common type is the Universal Forwarder (UF) — a small, low-overhead agent that does minimal processing locally and forwards raw data.

This week: pfSense does not use a Splunk forwarder. Instead, it uses the built-in syslog protocol to send log messages directly to the Splunk Indexer's syslog listener on port 514. This is a common pattern for network appliances that cannot run software agents.

Project 3: You will install Universal Forwarders on your Linux and Windows Server VMs to forward OS-level logs (auth logs, Windows Event Logs) to your own Splunk instance.

Indexers

The Indexer is the heart of Splunk. It receives incoming data, parses it into individual events, assigns timestamps, extracts fields, and stores the data in compressed, indexed form on disk. The Indexer is what makes searches fast — instead of scanning raw log files line by line, Splunk queries its index structures.

Data on an Indexer is organized into indexes — named buckets of data (similar to a database). Common indexes you will see in this course include network (for firewall and network device logs) and main (the default index).

Search Heads

The Search Head provides the web interface (Splunk Web) where users write SPL queries, view results, and build dashboards. In a small deployment (like your Project 3 instance), the Search Head and Indexer run on the same server. In large enterprise deployments they are separate.

Splunk Web runs on port 8000 by default. When you open the shared Splunk instance in your browser, you are connecting to the Search Head.

Indexes and Sourcetypes

Two concepts are essential for writing SPL searches:

ConceptWhat it isExample
Index A named data store on the Indexer. You specify which index to search with index= in SPL. index=network, index=main
Sourcetype A label that tells Splunk how to parse and interpret the data format. Splunk uses sourcetype to extract fields correctly. sourcetype=pfsense:filterlog, sourcetype=linux_secure
Host The hostname or IP address of the device that sent the data. Used to filter results to a specific device. host=pfsense, host=ubuntu-srv

💻 4 · SPL Basics

SPL (Search Processing Language) is Splunk's query language. It is designed to be readable and pipeline-based — you start with a search that retrieves events, then chain commands together using the pipe character (|) to transform, filter, and summarize the results.

💡 You Don't Need to Memorize SPL
The goal this week is familiarity, not memorization. A reference sheet with the most important SPL commands will be provided as a course handout and will be available during Quiz 9 (open-note). Focus on understanding what each command does and why you would use it.

The SPL Pipeline Model

Think of an SPL query as a pipeline: data flows from left to right, and each |-separated command transforms the data before passing it to the next stage.

📝 SPL Pipeline Concept
-- search terms -- | -- transform -- | -- summarize -- index=network sourcetype=pfsense:filterlog | stats count by src_ip | sort -count

Core SPL Keywords and Commands

Search Terms: index=, sourcetype=, host=

These three are the primary filter keywords that narrow down which events Splunk retrieves. They are placed at the beginning of a search, before any pipe commands.

📝 Filter Keywords
-- Search all events in the "network" index -- index=network -- Filter to pfSense firewall events only -- index=network sourcetype=pfsense:filterlog -- Filter to events from a specific host -- index=network host=pfsense -- Combine multiple filters (AND logic by default) -- index=network sourcetype=pfsense:filterlog action=block

Counting Events: | stats count by <field>

The stats command performs statistical aggregations on your search results. The most common pattern is | stats count by <field>, which groups events by the values of a field and counts how many events belong to each group.

📝 | stats count by
-- How many events came from each host? -- index=network | stats count by host -- How many blocked events per source IP? -- index=network sourcetype=pfsense:filterlog action=block | stats count by src_ip -- Sort results descending by count -- index=network sourcetype=pfsense:filterlog action=block | stats count by src_ip | sort -count

Events Over Time: | timechart count

The timechart command creates a time-series summary — it buckets events into time intervals and counts them. This is the command that produces the line charts and bar charts you see in Splunk dashboards. The span= option controls the bucket size.

📝 | timechart count
-- Count all pfSense events per hour -- index=network sourcetype=pfsense:filterlog | timechart span=1h count -- Count blocked vs. passed events per hour -- index=network sourcetype=pfsense:filterlog | timechart span=1h count by action -- 15-minute buckets for higher resolution -- index=network sourcetype=pfsense:filterlog | timechart span=15m count

Limiting Results: | head N

The head command returns only the first N results from the pipeline. Combined with sort, it gives you "top N" results — for example, the top 10 blocked source IPs.

📝 | head
-- Top 10 blocked source IPs -- index=network sourcetype=pfsense:filterlog action=block | stats count by src_ip | sort -count | head 10

Quick SPL Command Reference

CommandPurposeExample
index= Specify which Splunk index to search index=network
sourcetype= Filter by data source type / format sourcetype=pfsense:filterlog
host= Filter by originating hostname host=pfsense
| stats count by <field> Count events grouped by a field's values | stats count by src_ip
| timechart span=Xh count Count events over time in buckets | timechart span=1h count
| sort -count Sort results by count, descending | sort -count
| head N Return only the first N results | head 10
| table <field1> <field2> Display results as a table with selected fields | table src_ip dest_port count

🔥 5 · Configuring pfSense Syslog Forwarding

🔑 Before You Begin
You will need the shared Splunk instance IP address from the instructor Canvas Announcement before completing this walk-through. Have it ready to enter in Step 6 below.

pfSense includes built-in syslog forwarding. Once configured, pfSense will automatically send copies of its log events to the remote Splunk instance in real time. The configuration is done entirely through the pfSense web UI — no command line is required.

Where pfSense Logs Live

Before configuring forwarding, it helps to know what pfSense is logging locally. Navigate to Status → System Logs in the pfSense web UI to see the following log categories:

TabWhat it containsSplunk sourcetype
Firewall Packet filter decisions — blocked and passed connections, rule matches pfsense:filterlog
DHCP DHCP lease assignments, renewals, releases pfsense:dhcpd
General System events, interface state changes, login activity pfsense:syslog
VPN OpenVPN / IPsec connection and disconnection events pfsense:openvpn

Step-by-Step: Configuring Remote Syslog

  1. 1
    Log in to the pfSense web UI.
    Open a browser on your Management network VM and navigate to https://10.0.1.1 (or your pfSense LAN IP). Log in with your admin credentials.
  2. 2
    Navigate to the System Logs settings.
    From the top menu select Status → System Logs.
  3. 3
    Open the Settings tab.
    Inside the System Logs page, click the Settings tab (it appears as a sub-tab alongside Firewall, DHCP, General, etc.).
  4. 4
    Scroll down to the Remote Logging section.
    Look for the heading Remote Logging Options. You will see a checkbox and fields for remote syslog servers.
  5. 5
    Enable remote syslog.
    Check the box labeled Enable Remote Logging (or Send log messages to remote syslog server in some pfSense versions).
  6. 6
    Enter the Splunk instance address.
    In the Remote log servers field, enter the IP address of the shared Splunk instance followed by :514 (e.g., 10.255.254.50:514). Use the address provided in the instructor Canvas Announcement.
    ⚠️ Use the Correct Address
    Enter the IP address provided by the instructor — do not guess or use a placeholder. Using the wrong address will result in no data appearing in Splunk.
  7. 7
    Select which log categories to forward.
    Under Remote Syslog Contents, check at minimum:
    • Firewall Events — essential for the SPL exercises
    • DHCP service events — useful for tracking lease activity
    • General system events — captures interface changes, login events
  8. 8
    Save the configuration.
    Click Save at the bottom of the Settings page. pfSense will apply the changes immediately — no reboot is required.
  9. 9
    Verify data is flowing.
    Wait 1–2 minutes, then log in to the shared Splunk instance (see Section 6 for navigation instructions) and run:
    📝 Verification Search
    index=network host=pfsense | head 10
    If events appear, your syslog forwarding is working correctly. If no events appear after 5 minutes, double-check the IP address in Step 6 and confirm your pfSense can reach the Splunk instance (try pinging the Splunk IP from the pfSense Diagnostics → Ping tool).

🔬 Exercise 1 — Count Events by Host
Goal: See which devices are sending data to Splunk and how many events each has generated. This is often the first search an analyst runs when they open Splunk to get a situational overview.
  1. 1
    In the Splunk search bar, type or paste the following query exactly as shown:
    📝 Exercise 1 Query
    index=network | stats count by host
  2. 2
    Click the green Search button (or press Enter). Wait for results to appear in the Statistics tab below the search bar.
  3. 3
    You should see a table with two columns: host and count. Each row represents a device that has sent log data. Locate your pfSense entry.
  4. 4
    Interpret the results: Note how many events pfSense has generated compared to other hosts. A high count from pfSense is expected — it logs every connection attempt that matches a firewall rule.
  5. 5
    Take a screenshot of the results table showing at least one row with pfSense data. You will need this for Lab 9 Part D if you are pursuing extra credit.
📌 What You Just Did
You ran your first SPL query. The index=network told Splunk which data store to search. The | stats count by host aggregated all events and grouped them by the host field. No filtering was applied — this is a "show me everything" overview query.
📈 Exercise 2 — Timechart of pfSense Events Over Time
Goal: Visualize how pfSense event volume changes over time. Spikes or drops in event rate can indicate security events, configuration changes, or network problems.
  1. 1
    Clear the search bar and enter the following query:
    📝 Exercise 2 Query
    index=network sourcetype=pfsense:filterlog | timechart span=1h count
  2. 2
    Run the search. Click the Visualization tab (next to Statistics) to see the results as a chart. Splunk will automatically render a line or column chart showing event count per hour.
  3. 3
    Adjust the time range picker to Last 24 hours if you see only a flat line (this means not enough historical data is available in a shorter window).
  4. 4
    Optional extension: Modify the query to split the timechart by action (block vs. pass):
    📝 Extension: Split by Action
    index=network sourcetype=pfsense:filterlog | timechart span=1h count by action
    This produces two series on the chart — one for blocked events and one for passed events.
  5. 5
    Take a screenshot of the Visualization tab showing the timechart. Note whether you can see any interesting patterns (periodic spikes, quiet periods, etc.).
📌 What You Just Did
By adding sourcetype=pfsense:filterlog you narrowed the search to pfSense firewall log events only. The | timechart span=1h count command bucketed those events into 1-hour intervals and counted them — producing the time-series data that Splunk rendered as a chart.
🚫 Exercise 3 — Top 10 Blocked Source IPs
Goal: Identify the source IP addresses responsible for the most blocked traffic. This is a foundational security query — the top blocked IPs are candidates for investigation, threat intelligence lookup, or permanent firewall block rules.
  1. 1
    Enter the following query:
    📝 Exercise 3 Query
    index=network sourcetype=pfsense:filterlog action=block | stats count by src_ip | sort -count | head 10
  2. 2
    Run the search. The Statistics tab will show a table with two columns: src_ip (source IP address) and count (number of times that IP was blocked). The results are sorted from most blocked to least, and only the top 10 are shown.
  3. 3
    Examine the top result. Ask yourself:
    • Is this an internal IP (10.x.x.x, 192.168.x.x) or an external IP?
    • Does the count seem unusually high compared to the others?
    • What might explain repeated blocks from the same source?
  4. 4
    Optional extension: Add destination port to the grouping to see what services the top blocked IP was trying to reach:
    📝 Extension: Add Destination Port
    index=network sourcetype=pfsense:filterlog action=block src_ip=[paste top IP here] | stats count by dest_port | sort -count
  5. 5
    Take a screenshot of the top-10 results table. This screenshot, along with those from Exercises 1 and 2, completes the Lab 9 Part D evidence requirement.
📌 What You Just Did
This query chains four SPL commands. First, it filters to blocked pfSense events. Then | stats count by src_ip groups and counts by source IP. Then | sort -count orders from highest to lowest. Finally | head 10 trims the output to the top 10 rows. This is a complete, production-quality security query used by real SOC analysts.

🛡️ 7 · Connecting Log Data to Security Practice

Raw log data becomes security intelligence when you know what questions to ask. This section connects the SPL skills from Section 6 to real security monitoring use cases you will encounter in Project 3 and beyond.

Top Blocked Source IPs — What They Tell You

The top blocked IP query from Exercise 3 is more powerful than it looks. Here is how a security analyst interprets the results:

ObservationWhat it might meanFollow-up action
External IP with thousands of blocks in an hour Port scan, brute-force attempt, or automated exploit tool targeting your network Add a permanent block rule in pfSense; look up IP in threat intelligence (AbuseIPDB)
Internal IP appearing in blocked traffic Misconfigured service trying to reach a blocked destination; or a compromised internal host Investigate the internal host; check for malware or misconfiguration
Multiple IPs from the same /24 subnet Coordinated scan from a single organization's IP block Consider blocking the entire subnet range using a pfSense alias
Blocks targeting port 22 (SSH) or 3389 (RDP) Automated credential stuffing or brute-force attack against remote access services Ensure SSH/RDP is not exposed on WAN; consider fail2ban or geo-blocking

DHCP Lease Events in Splunk

DHCP logs forwarded from pfSense capture every lease assignment, renewal, and release. This data is valuable for:

  • Device inventory: Which MAC addresses have been on your network and when?
  • Incident response: At 14:32 yesterday, which device had IP 10.0.3.147? Look it up in DHCP lease logs.
  • Rogue device detection: A new MAC address that has never appeared before suddenly obtaining a DHCP lease is worth investigating.
📝 DHCP Lease Events
-- All DHCP events from pfSense -- index=network sourcetype=pfsense:dhcpd | stats count by client_mac client_ip | sort -count

Traffic by Interface

pfSense logs include the interface on which each packet arrived. Searching by interface helps you understand traffic patterns across your network segments — for example, how much traffic is flowing from the Clients subnet to the Servers subnet vs. directly to the WAN.

📝 Traffic by Interface
-- Event count by interface -- index=network sourcetype=pfsense:filterlog | stats count by interface action | sort -count

📊 8 · Mini-Case: The Suspicious Traffic Spike

📊 Scenario: Monday Morning at Mendocino Tech — A Splunk Alert

It is 8:15 AM on Monday. You are the on-call network administrator at Mendocino Tech. You open the Splunk dashboard and notice something unusual in the pfSense firewall timechart. The chart below represents what you see:

pfSense Blocked Events — Last 8 Hours (sourcetype=pfsense:filterlog action=block)
00:0001:0002:00 03:0004:0005:00 06:0007:0008:00
Normal traffic (~10–25 events/30 min)
Spike (03:00–04:30 · 847–340 events/30 min)

You run the top-blocked-IP query (Exercise 3) filtered to the spike window (03:00–04:30) and see that 98% of the blocked events during the spike came from a single source IP: 198.51.100.47 — an external address you do not recognize. The destination ports targeted were 22 (SSH), 3389 (RDP), 445 (SMB), and 23 (Telnet).

❓ Question 1: What does this traffic pattern suggest? What type of activity is most likely occurring?

Likely Activity — Automated Port Scan / Credential Attack: The pattern strongly suggests an automated port scan followed by service exploitation attempts from a single external IP. Key indicators:
  • The sudden spike starting at 03:00 (overnight, when administrators are not watching) is consistent with automated attack tools.
  • The targeted ports — SSH (22), RDP (3389), SMB (445), Telnet (23) — are all remote access or file-sharing services commonly targeted in credential brute-force attacks and ransomware pre-staging.
  • The gradual decline after 04:00 suggests the tool completed its scan range or moved on after finding no open ports.
  • The fact that pfSense blocked all of it means your firewall rules are working — the attacker found no open services on the WAN interface.

❓ Question 2: What additional SPL searches would you run to investigate further?

Follow-Up Searches:
  1. Check all ports targeted by this IP:
    📝 All Ports Targeted
    index=network sourcetype=pfsense:filterlog src_ip=198.51.100.47 | stats count by dest_port | sort -count
  2. Check if this IP appears in any passed traffic (i.e., did anything get through?):
    📝 Check for Passed Traffic
    index=network sourcetype=pfsense:filterlog src_ip=198.51.100.47 action=pass | stats count by dest_port
  3. Check if this IP has appeared before (historical context): Expand the time range to "Last 30 days" and run the same blocked-IP query to see if 198.51.100.47 is a repeat offender.
  4. External lookup: Search the IP in a threat intelligence service (AbuseIPDB, VirusTotal, Shodan) to see if it is a known malicious host.

❓ Question 3: What firewall action would you consider taking, and how does this connect to Module 9.3?

Recommended Firewall Action:
  1. Add an explicit block rule in pfSense for this IP: Navigate to Firewall → Rules → WAN in pfSense and add a rule at the top of the ruleset:
    • Action: Block
    • Source: 198.51.100.47
    • Destination: any
    • Log: Yes (so Splunk continues to record if the IP returns)
    • Description: Block known scanner — detected via Splunk 2026-03-16 03:00 spike
  2. Module 9.3 connection: This action directly applies the firewall rule concepts from Module 9.3 — specifically rule ordering (the explicit block rule must be placed above any allow rules so it matches first), and the use of logging on block rules to maintain the audit trail in Splunk.
  3. Consider blocking the entire subnet (198.51.100.0/24) using a pfSense Alias if threat intelligence confirms the block is a known malicious range. Aliases allow you to maintain a list of IPs/subnets that is easy to update without editing individual rules.
  4. Document the incident — note the time, source IP, targeted ports, block count, and action taken. This is the beginning of an incident report.

This mini-case illustrates the core value of combining a SIEM with a firewall: Splunk detects the pattern, and pfSense enforces the response. In Project 3 you will build automated alerts that can notify you of exactly this type of spike without requiring manual dashboard review.

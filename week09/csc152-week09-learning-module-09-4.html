🔀 1 · TCP vs. UDP — When Each Is Used & Why It Matters

Every network service communicates using one of two transport-layer protocols: TCP (Transmission Control Protocol) or UDP (User Datagram Protocol). Understanding the difference is not just academic — it directly determines how you write firewall rules, because a rule that specifies TCP will not catch UDP traffic on the same port number, and vice versa.

🔵 TCP — Connection-Oriented

  • Establishes a connection via a three-way handshake (SYN → SYN-ACK → ACK) before any data is sent.
  • Guarantees delivery: lost packets are detected and retransmitted.
  • Ordered delivery: data arrives in the sequence it was sent.
  • Flow control & congestion control built in.
  • Higher overhead — connection setup and acknowledgment traffic add latency.
  • Used by: HTTP, HTTPS, SSH, FTP, SMTP, RDP, SMB — any service where data integrity is critical.

🟡 UDP — Connectionless

  • No handshake: packets are sent immediately without establishing a connection first.
  • No guaranteed delivery: packets can be lost, duplicated, or arrive out of order.
  • No retransmission — the application layer must handle errors if needed.
  • Very low overhead — minimal header, no connection state.
  • Used by: DNS queries, DHCP, TFTP, SNMP, syslog, VoIP, video streaming — services where speed matters more than perfection.

Why This Matters for Firewall Rules

Firewall rules specify protocol (TCP, UDP, or both) as part of the rule definition. Specifying the wrong protocol means the rule will silently fail to match real traffic. Consider DNS: it uses both TCP and UDP on port 53. A firewall rule that only allows UDP 53 will block large DNS responses that fall back to TCP, causing intermittent name resolution failures that are notoriously difficult to diagnose.

⚠️ Common Firewall Mistake
Writing a rule for TCP port 53 without a matching UDP port 53 rule (or vice versa) will cause partial DNS failures. Always check the transport protocol column in the port reference table before writing a firewall rule.

Stateful vs. Stateless Inspection — A Quick Reminder

pfSense (and most modern firewalls) use stateful inspection: they track the state of TCP connections and automatically permit the return traffic for an established session. This means you typically only need to write an outbound allow rule — the firewall knows to let the reply back in. Stateless firewalls (like basic iptables without the conntrack module) require explicit rules for both directions.

📌 Ephemeral Ports and Return Traffic
When a client opens a TCP connection to a server's well-known port (e.g., port 443), the client's OS picks a random ephemeral source port (typically 49152–65535) for that session. A stateful firewall tracks this tuple (src IP, src port, dst IP, dst port, protocol) and permits the server's reply automatically. You do not need a rule allowing inbound traffic on all ephemeral ports.

🔢 2 · Port Number Ranges

Port numbers are 16-bit unsigned integers, giving a range of 0 to 65,535. The Internet Assigned Numbers Authority (IANA) divides this range into three categories, each with a distinct administrative meaning.

Well-Known Ports
0 – 1,023
Assigned to core internet services by IANA. Binding to these ports typically requires root/administrator privileges on the server OS. Examples: HTTP (80), HTTPS (443), SSH (22), DNS (53).
Registered Ports
1,024 – 49,151
Registered by IANA for specific applications but do not require elevated privileges to bind. Examples: RDP (3389), MySQL (3306), Splunk Web (8000), Splunk HEC (8088).
Dynamic / Ephemeral
49,152 – 65,535
Assigned automatically by the OS as temporary source ports for outbound connections. Not bound to specific services; you will see these as client-side ports in network captures.

Administrative Significance

  • Well-known port binding requires root: On Linux, a process must run as root (or have CAP_NET_BIND_SERVICE) to listen on ports below 1024. This is a security control — it prevents unprivileged users from impersonating core services.
  • Registered ports in firewall rules: Services like RDP (3389) and Splunk (8000, 8088, 9997) fall in the registered range. These must be explicitly allowed in firewall rules — they are not automatically open.
  • Ephemeral ports and stateful firewalls: You should never need to write explicit allow rules for the ephemeral range on a stateful firewall. If you find yourself doing so, reconsider the rule design.
  • Non-standard ports: Administrators sometimes run services on non-standard ports (e.g., SSH on port 2222 instead of 22) as a minor obscurity measure. This is not a security control — port scanners find these trivially — but it does reduce automated brute-force noise in logs.
📝 Quiz 9 & Final Exam Note
Port number ranges are directly tested. Know the three ranges, their numeric boundaries, and one or two examples from each. The well-known range (0–1023) and the specific ports in Section 3 are the highest priority for both Quiz 9 and the final exam.

🔍 4 · Port Scanning & nmap

Port scanning is the process of probing a host or network to discover which TCP and UDP ports are open — and therefore which services are running and potentially reachable. It is one of the most fundamental tools in both network administration and attacker reconnaissance.

What nmap Does

nmap (Network Mapper) is the industry-standard open-source port scanner. At its core, nmap sends specially crafted packets to a target host and analyzes the responses to determine port state:

  • Open: A service is listening and accepting connections on this port.
  • Closed: The port is reachable but no service is listening (the OS replied with a TCP RST).
  • Filtered: A firewall is blocking the probe — nmap received no response or an ICMP unreachable message.

Common nmap Scan Types

Scan TypeFlagHow It WorksUse Case
TCP SYN Scan (Stealth) -sS Sends SYN; open port replies SYN-ACK; nmap sends RST (never completes handshake). Faster and less likely to be logged by the target application. Default scan; requires root. Standard for most admin use.
TCP Connect Scan -sT Completes the full three-way TCP handshake. Slower; more likely to appear in application logs. When root is not available; more compatible with some targets.
UDP Scan -sU Sends UDP packets; listens for ICMP port-unreachable (closed) or application response (open). Very slow. Finding UDP services like DNS, DHCP, SNMP.
Service Version Detection -sV Probes open ports to identify the running service and version (e.g., OpenSSH 8.9). Vulnerability assessment — knowing the version reveals known CVEs.
OS Detection -O Analyzes TCP/IP stack behavior to guess the target OS. Asset inventory; attacker reconnaissance.

Example nmap Commands

# Scan the most common 1000 TCP ports on a single host nmap 192.168.30.50 # SYN scan with service version detection on a specific host nmap -sS -sV 192.168.30.50 # Scan all 65535 TCP ports (thorough but slow) nmap -p- 192.168.30.50 # Scan a specific set of ports nmap -p 22,80,443,3389 192.168.30.50 # Scan an entire subnet (use with permission only) nmap -sn 192.168.30.0/24 # ping sweep (host discovery only) # Save output to a file for documentation nmap -sV -oN scan-results.txt 192.168.30.50

Legitimate Administrative Use vs. Attack Reconnaissance

The same tool and the same commands serve both purposes. The difference is authorization and intent.

DimensionLegitimate Administrative UseAttack Reconnaissance
Authorization Scanning systems you own or have explicit written permission to scan. Scanning systems without permission — illegal in most jurisdictions under computer fraud laws.
Purpose Verify firewall rules are working; audit open ports; identify misconfigured services; asset inventory. Identify vulnerable services; find attack entry points; map the network before an intrusion attempt.
Scope Defined, documented scope — specific hosts or subnets. Often broad — scanning large IP ranges looking for any open port.
Documentation Results documented, compared to expected baseline, and used to close unnecessary ports. Results used to select targets and exploit vulnerabilities.
⚠️ Legal and Ethical Notice
Port scanning systems you do not own or do not have explicit written permission to scan may violate the Computer Fraud and Abuse Act (CFAA) and equivalent laws in other jurisdictions. In this course, you are authorized to scan only the virtual machines in your own Proxmox lab environment. Do not scan Mendocino College network infrastructure or any external hosts.
💡 The Port Audit Workflow
A professional port audit uses nmap to establish a baseline of open ports, compares that baseline to the expected service profile of the server, identifies any unexpected open ports, and closes them via firewall rules or by disabling the service. This is exactly what you will practice in the Mini-Case in Section 7.

🛡️ 5 · Principle of Least Exposure

The principle of least exposure (also called minimal attack surface) states that a system should expose only the ports, protocols, and services strictly required for its stated function — and nothing more. Every additional open port is an additional potential entry point for an attacker.

The Attack Surface Visualized

The two server profiles below illustrate the difference between an over-exposed server and a properly hardened one. Both servers are intended to serve only web traffic (HTTP/HTTPS) and allow SSH administration.

🔴 Over-Exposed Web Server

12 open ports found by nmap — most are unnecessary:

21 FTP 22 SSH 23 Telnet 25 SMTP 53 DNS 80 HTTP 110 POP3 143 IMAP 389 LDAP 443 HTTPS 445 SMB 3389 RDP

9 of 12 open ports are unnecessary. Each is a potential attack vector.

🟢 Hardened Web Server

Only required ports open — all others closed or filtered:

21 FTP 22 SSH 23 Telnet 25 SMTP 53 DNS 80 HTTP 110 POP3 143 IMAP 389 LDAP 443 HTTPS 445 SMB 3389 RDP

Only 3 ports open — exactly what the stated role (web + SSH) requires.

Applying Least Exposure in Practice

  1. 1
    Define the server's role precisely. "Web server" means HTTP (80) and HTTPS (443). "Web server with SSH administration" adds port 22. Nothing else is implied.
  2. 2
    Start with a deny-all policy. pfSense OPT interfaces start with no rules (implicit deny). This is the correct default. Add only what is explicitly needed.
  3. 3
    Add rules one service at a time, documenting the business justification for each. If you cannot explain why a port needs to be open, it should be closed.
  4. 4
    Restrict source as well as destination. SSH (22) should not be allowed from the internet or the client subnet — only from the management subnet. A rule that allows SSH from any source is significantly weaker than one that restricts it to a specific management IP range.
  5. 5
    Audit periodically. Services are installed and forgotten. Run a port scan against your own servers on a regular schedule to verify that the open port profile matches what you expect.
  6. 6
    Disable the service, not just the port. Closing a firewall port hides a service from external access but does not stop it from running. A compromised host on the same subnet can still reach an unneeded service that is running locally. Disable or uninstall services you do not need.
🔒 CSLO 2 Connection
The principle of least exposure is a foundational network data security technique (CSLO 2). In Project 2, your firewall rule set will be evaluated partly on whether it correctly implements this principle — allowing only the traffic required for each subnet's stated function and blocking everything else.

⚠️ 6 · Common Attack Vectors

Understanding how attackers exploit open ports and insecure protocols helps you make better decisions about what to allow in firewall rules and which services to harden or replace. The following attack vectors are directly connected to the protocols in Section 3.

Cleartext Credential Exposure

Several protocols transmit usernames, passwords, and data with no encryption. Anyone with the ability to capture network traffic on the same segment (or on any router between the client and server) can read these credentials directly.

ProtocolPortWhat Is ExposedMitigation
Telnet 23 Everything — username, password, every command typed, every response. A network capture shows the entire session in plain text. Replace with SSH (port 22). Disable Telnet service. Block port 23 at the firewall.
FTP 20, 21 Username and password sent in cleartext during the control connection (port 21). File contents also transmitted unencrypted. Use SFTP (over SSH, port 22) or FTPS (FTP over TLS, port 990). Block ports 20/21 at the firewall.
HTTP 80 All web content — including form submissions, login credentials, session cookies, and any data the user sends or receives. Redirect all HTTP to HTTPS. Use HSTS headers. Block or redirect port 80 to 443 at the firewall or web server.
POP3 / IMAP 110, 143 Email credentials and message content in cleartext. Use POP3S (995) or IMAPS (993) with TLS. Disable plaintext variants on the mail server.
LDAP 389 Directory queries and bind (authentication) credentials in cleartext. Use LDAPS (port 636) with TLS. Configure Active Directory to require LDAP signing and channel binding.
⚠️ Why This Matters on Internal Networks Too
Many administrators assume cleartext protocols are acceptable on internal networks because "attackers can't get in." This is a dangerous assumption. An attacker who has compromised a single internal host (via phishing, a vulnerable service, or a rogue device) can passively capture all cleartext traffic on the same VLAN or subnet. Defense in depth requires encrypting sensitive traffic everywhere, not just at the perimeter.

Open Port Exploitation

An open port is an invitation. When a service is listening on a port, attackers can interact with it directly — probing for vulnerabilities, attempting authentication, or exploiting known software flaws.

RDP Brute Force (Port 3389)

RDP is one of the most heavily attacked services on the internet. Servers with port 3389 exposed to the internet receive automated brute-force login attempts within minutes of going online. Several high-profile ransomware campaigns gained initial access exclusively through exposed RDP.

⚠️ Firewall Rule: RDP
In pfSense, RDP (TCP 3389) should never be allowed from the WAN interface. Allow it only from the Management subnet to specific Windows Server VMs. This is a required constraint in Project 2.

SMB Exploitation (Port 445)

The 2017 WannaCry ransomware and NotPetya attacks spread globally by exploiting a vulnerability (EternalBlue / MS17-010) in the Windows SMB service on port 445. Organizations with port 445 exposed to the internet were devastated. Even internally, SMB should be restricted to subnets that genuinely need file sharing.

SSH Brute Force (Port 22)

SSH is secure by design but is frequently targeted by automated password-guessing attacks. Mitigations include:

  • Disable password authentication; require SSH key pairs.
  • Restrict source IPs in firewall rules (allow only from management subnet).
  • Use fail2ban to automatically block IPs after repeated failed attempts.
  • Consider moving SSH to a non-standard port as a noise reduction measure (not a security control).

Protocol Downgrade Attacks

A protocol downgrade attack tricks a client and server into negotiating a weaker, older version of a protocol — one that has known vulnerabilities or weaker encryption — even when both endpoints support a more secure version.

TLS Downgrade (HTTPS)

TLS (Transport Layer Security) has evolved through versions 1.0, 1.1, 1.2, and 1.3. Older versions (TLS 1.0, 1.1) have known weaknesses. A man-in-the-middle attacker who can intercept the TLS handshake may force the negotiation to fall back to TLS 1.0, then exploit those weaknesses. Mitigations:

  • Configure web servers to support only TLS 1.2 and TLS 1.3.
  • Disable TLS 1.0 and 1.1 in server configuration (ssl_protocols TLSv1.2 TLSv1.3; in nginx).
  • Implement HTTP Strict Transport Security (HSTS) to prevent browsers from accepting HTTP connections.

SMB Protocol Downgrade

Windows supports multiple versions of SMB (SMBv1, SMBv2, SMBv3). SMBv1 is ancient, has no encryption, and is the version exploited by EternalBlue. An attacker on the network may force a downgrade to SMBv1 if it is still enabled. Mitigation: disable SMBv1 on all Windows systems via PowerShell:

# Disable SMBv1 on Windows Server (run as Administrator) Set-SmbServerConfiguration -EnableSMB1Protocol $false -Force Set-SmbClientConfiguration -EnableSMB1Protocol $false -Force

The Common Thread

All downgrade attacks share a common theme: they exploit backward compatibility. Systems that support both old and new protocol versions to maintain compatibility with legacy clients create an attack surface. The mitigation is always the same: disable old protocol versions and enforce the use of current, secure versions.

🔒 Defense in Depth
No single control stops all attacks. Layered defenses — firewall rules that restrict port access, protocol hardening that disables weak versions, and encryption that protects data in transit — work together. This is the concept of defense in depth, which underpins CSLO 2 throughout this course.

🔎 7 · Mini-Case: The Port Audit

🔎 Scenario: Mendocino Tech — Quarterly Port Audit

You are a junior network administrator at Mendocino Tech. As part of a quarterly security review, you have been asked to audit a server named web-prod-01. According to the server inventory, this server has a single stated role: public-facing web server with SSH administrative access. It runs Ubuntu Server 22.04, hosts the company's public website over HTTPS, and is managed via SSH from the Management subnet only.

You run the following nmap scan from within the Management subnet:

$ nmap -sV -p- 192.168.10.50 Starting Nmap 7.94 ( https://nmap.org ) Nmap scan report for web-prod-01 (192.168.10.50) Host is up (0.0012s latency). PORT STATE SERVICE VERSION 21/tcp open ftp vsftpd 3.0.5 22/tcp open ssh OpenSSH 8.9p1 Ubuntu 23/tcp open telnet Linux telnetd 25/tcp open smtp Postfix smtpd 53/tcp open domain ISC BIND 9.18.12 80/tcp open http Apache httpd 2.4.52 110/tcp open pop3 Dovecot pop3d 143/tcp open imap Dovecot imapd 389/tcp open ldap OpenLDAP 2.6.3 443/tcp open https Apache httpd 2.4.52 445/tcp open microsoft-ds Samba smbd 4.15.13 3389/tcp open ms-wbt-server xrdp 0.9.21 12 ports open.

Green lines are expected. Red lines are unexpected for a server whose role is web + SSH only.

❓ Question 1: Which ports are unnecessary for this server's stated role? List them and explain why each one should not be open.

Unnecessary Open Ports (9 of 12):
  • Port 21 (FTP / vsftpd): File transfer is not part of the server's role. FTP transmits credentials in cleartext. Should be disabled and removed.
  • Port 23 (Telnet): Unencrypted remote access. SSH (already open on 22) is the correct tool. Telnet should never be running on any production server.
  • Port 25 (SMTP / Postfix): This server is not a mail server. An open SMTP service could be exploited for spam relay or as a pivot point.
  • Port 53 (DNS / BIND): This server is not a DNS server. An open DNS service exposes the server to DNS amplification abuse and zone transfer attempts.
  • Port 110 (POP3 / Dovecot): No mail retrieval role. Cleartext protocol. Should not be running.
  • Port 143 (IMAP / Dovecot): Same as POP3 — no mail role. Cleartext by default.
  • Port 389 (LDAP / OpenLDAP): No directory service role. Exposes directory data in cleartext. Should not be running.
  • Port 445 (SMB / Samba): No file sharing role. SMB is a high-value attack target (EternalBlue). Should be disabled immediately.
  • Port 3389 (RDP / xrdp): Linux server managed via SSH — RDP is not needed. xrdp running on a public-facing server is a significant risk.

Ports that are correct and expected: 22 (SSH — management access), 80 (HTTP — redirect to HTTPS), 443 (HTTPS — web service).

❓ Question 2: Write a brief remediation plan. What are the two types of actions you should take for each unnecessary service, and why are both needed?

Remediation Plan — Two Required Actions Per Service:
  1. Action 1 — Disable and uninstall the service: Stop the service, disable it from starting on boot, and uninstall the package if it is not needed elsewhere. Example for vsftpd (FTP):
    sudo systemctl stop vsftpd sudo systemctl disable vsftpd sudo apt remove --purge vsftpd
    Why this is necessary: A firewall rule only hides the port from external access. The service is still running and reachable from any host on the same subnet that has already been compromised. Disabling the service eliminates the vulnerability entirely.
  2. Action 2 — Add a firewall rule to block the port: Even after disabling the service, add an explicit deny rule in pfSense (or ufw / iptables) to block the port. This provides defense in depth — if the service is accidentally re-enabled in the future, the firewall rule prevents exposure.

❓ Question 3: Write the pfSense firewall rules needed to enforce the correct port profile for web-prod-01. The server is on the Servers subnet (192.168.10.0/24). Management subnet is 192.168.1.0/24. The server should be reachable by the public internet (WAN) on ports 80 and 443 only.

pfSense Firewall Rules for web-prod-01:

In pfSense, rules are written on the interface where traffic enters (not exits). Rules are evaluated top-to-bottom; first match wins.

#InterfaceActionProtocolSourceDestinationPort(s)Description
1 WAN Pass TCP any 192.168.10.50 80 Allow public HTTP to web-prod-01
2 WAN Pass TCP any 192.168.10.50 443 Allow public HTTPS to web-prod-01
3 WAN Block any any 192.168.10.50 any Block all other WAN traffic to web-prod-01 (explicit; logged)
4 Management Pass TCP 192.168.1.0/24 192.168.10.50 22 Allow SSH from Management subnet only

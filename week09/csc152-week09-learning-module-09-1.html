🖥️ 1 · What is a DHCP Server?

Dynamic Host Configuration Protocol (DHCP) is a network management protocol that automatically assigns IP addresses and other network configuration parameters to devices on a network. Without DHCP, an administrator would need to manually configure the IP address, subnet mask, default gateway, and DNS servers on every single device — a process that is error-prone and does not scale.

Client Side vs. Server Side

In Week 8 you configured systems as DHCP clients — you told Linux and Windows Server to request an address from whatever DHCP server was available on the network. This week you become the administrator who runs that server.

PerspectiveRoleWhat it doesWeek covered
DHCP ClientEnd device / server NICBroadcasts a request; accepts an IP leaseWeek 8 (Module 8.2)
DHCP ServerDedicated service on a hostListens for requests; manages pools; issues leasesWeek 9 (this module)

Why DHCP Servers Matter in Enterprise Networks

  • Scale: A single DHCP server can serve hundreds or thousands of devices automatically.
  • Consistency: All clients receive correct gateway and DNS settings from one authoritative source.
  • Mobility: Laptops and phones obtain valid addresses whenever they connect, regardless of subnet.
  • Auditability: Lease logs record which MAC address held which IP at what time — valuable for security investigations.
  • Change management: Updating the DNS server address in one DHCP scope propagates to all clients at their next renewal.

In Project 2 Part A you will configure pfSense to act as the DHCP server for your Clients subnet. Everything in this module builds the conceptual foundation for that task. In Project 2 Part B you will also ensure that DNS information is distributed through DHCP options.

🔄 2 · The DORA Process

When a device needs an IP address it follows a four-step exchange with the DHCP server. The acronym DORA describes these four messages: Discover → Offer → Request → Acknowledge.

1
D
DISCOVER
Client has no IP. It broadcasts a DHCPDISCOVER message (UDP src 0.0.0.0:68 → dst 255.255.255.255:67) asking "Is there a DHCP server out there?"
2
O
OFFER
Server responds with a DHCPOFFER — a proposed IP address, subnet mask, lease time, and options (gateway, DNS). May be unicast or broadcast depending on client flags.
3
R
REQUEST
Client broadcasts a DHCPREQUEST formally accepting the offered address (and implicitly declining offers from any other DHCP servers that may have responded).
4
A
ACKNOWLEDGE
Server confirms with a DHCPACK, finalizing the lease. The client now configures its interface with the assigned address and options. A DHCPNAK is sent if the address is no longer available.

Key DORA Details

  • Transport: DHCP uses UDP — port 67 (server) and port 68 (client).
  • Broadcast-based: Discover and Request messages are broadcast because the client has no IP yet and does not know the server's address.
  • Lease renewal: At 50% of the lease time the client sends a unicast DHCPREQUEST directly to the server to renew. At 87.5% it falls back to broadcast if the server did not respond.
  • Release: A client can send DHCPRELEASE to return its address early (e.g., on shutdown with dhclient -r).
💡 Exam Tip
Quiz 9 will ask about the DORA acronym. Remember the order: Discover (client → broadcast), Offer (server → client), Request (client → broadcast), Acknowledge (server → client).

📋 3 · Scopes, Pools, Exclusions & Reservations

Scope

A scope is the administrative definition of a range of IP addresses that a DHCP server can manage for a specific subnet. Each scope is tied to one subnet. A server can host multiple scopes for different subnets (using DHCP relay agents to reach remote subnets).

Example: Scope for the Clients subnet in Project 2 might cover 192.168.30.0/24 with an address pool of 192.168.30.100 – 192.168.30.200.

Address Pool

The address pool (or dynamic range) is the contiguous block of addresses within the scope that the server will actually hand out to clients. Addresses outside the pool but within the subnet can be used for static assignments without conflicting with DHCP.

Exclusions

An exclusion removes specific addresses (or a sub-range) from the pool so the DHCP server will never assign them dynamically. Use exclusions for addresses you intend to assign statically to servers, printers, or network equipment — but that fall within the pool range.

Reservations

A reservation binds a specific IP address to a specific device by its MAC address. When that device sends a DHCPDISCOVER, the server always responds with the same reserved address. The device still goes through the full DORA process — it just always receives the same IP.

📌 Reservation vs. Static IP
A reservation is not the same as a static IP configured on the device itself. With a reservation, the device still uses DHCP — the server simply guarantees a consistent address. With a true static IP, the device never contacts DHCP at all. Reservations are preferred for managed environments because the gateway and DNS options are still distributed centrally.

Lease Duration

Every DHCP assignment is a lease — a temporary grant of an address for a defined period. Common defaults:

  • 8 hours — typical for high-turnover environments (coffee shops, labs)
  • 24 hours — common default for enterprise LANs
  • 7–14 days — used for stable environments with few devices

Shorter leases reclaim addresses faster but increase DHCP traffic. Longer leases reduce traffic but can exhaust the pool if devices leave the network without releasing their address.

⚙️ 4 · DHCP Options

A DHCP server does more than assign an IP address. It also distributes options — additional configuration parameters that tell the client how to use the network. Options are identified by option codes defined in RFC 2132.

Option Code Name Description Example Value
3 Default Gateway (Router) The IP address of the default gateway clients should use to reach other networks. 192.168.30.1
6 DNS Server(s) One or more DNS server addresses for hostname resolution. Listed in preference order. 192.168.10.10, 8.8.8.8
15 Domain Name The DNS domain suffix appended to unqualified hostnames (e.g., server1 → server1.lab.local). lab.local
42 NTP Server(s) Network Time Protocol servers for clock synchronization. 192.168.10.1
51 Lease Time Duration (in seconds) of the IP address lease. 86400 (24 hours)
66 TFTP Server Used for PXE boot environments to direct clients to a boot server. 192.168.10.20

Scope Options vs. Server Options

In Windows Server DHCP, options can be set at three levels:

  • Server Options: Apply to all scopes on the server.
  • Scope Options: Apply only to clients in a specific scope (override server options).
  • Reservation Options: Apply only to a specific reserved client (highest priority).

In isc-dhcp-server the same hierarchy exists through global declarations, subnet blocks, and host blocks in /etc/dhcp/dhcpd.conf.

💡 Project 2 Tip
For your Project 2 Clients subnet, you will need to configure at minimum Option 3 (gateway = pfSense LAN interface IP) and Option 6 (DNS server you configure in Module 9.2). Option 15 (domain name) is recommended for full name resolution to work correctly.

🐧 5 · Configuring DHCP on Linux

Two common tools are used to deploy a DHCP server on Linux: isc-dhcp-server (the traditional, feature-rich choice for Ubuntu Server) and dnsmasq (a lightweight combined DHCP + DNS server popular in small networks and embedded systems). This walk-through focuses on isc-dhcp-server since it maps most directly to enterprise practice and Project 2 concepts.

Step-by-Step: isc-dhcp-server on Ubuntu Server

  1. 1
    Install the package
    sudo apt update && sudo apt install -y isc-dhcp-server
    After installation the service will attempt to start and fail because no scope is configured yet. That is expected.
  2. 2
    Specify the listening interface
    Edit /etc/default/isc-dhcp-server and set the interface on which the server should listen:
    # /etc/default/isc-dhcp-server INTERFACESv4="ens18" # replace with your interface name
    Use ip link show to confirm your interface name.
  3. 3
    Configure the scope in /etc/dhcp/dhcpd.conf
    This is the primary configuration file. Open it with:
    sudo nano /etc/dhcp/dhcpd.conf
    A minimal working configuration for a 192.168.30.0/24 Clients subnet looks like this:
    # /etc/dhcp/dhcpd.conf — Minimal example for CSC 152 Project 2 Clients subnet # Global settings default-lease-time 86400; # 24 hours max-lease-time 172800; # 48 hours authoritative; # This server is THE authority for its subnets # Subnet declaration subnet 192.168.30.0 netmask 255.255.255.0 { range 192.168.30.100 192.168.30.200; option routers 192.168.30.1; # Option 3: default gateway option domain-name-servers 192.168.10.10; # Option 6: DNS server option domain-name "lab.local"; # Option 15: domain suffix } # Reservation example (printer at .102) host printer-01 { hardware ethernet aa:bb:cc:dd:ee:ff; fixed-address 192.168.30.102; }
    ⚠️ authoritative Directive
    Always include authoritative; on a production DHCP server. Without it, the server will not send DHCPNAK messages to clients that have stale leases from another server, causing address conflicts.
  4. 4
    Test the configuration syntax
    sudo dhcpd -t -cf /etc/dhcp/dhcpd.conf
    If no errors are reported, the file is syntactically valid.
  5. 5
    Start and enable the service
    sudo systemctl start isc-dhcp-server sudo systemctl enable isc-dhcp-server sudo systemctl status isc-dhcp-server
    Look for active (running) in the status output.
  6. 6
    Verify leases
    cat /var/lib/dhcp/dhcpd.leases
    After a client connects, its lease entry will appear here with the assigned IP, MAC address, lease start, and expiry times. You can also watch the DHCP log in real time:
    sudo journalctl -u isc-dhcp-server -f

dnsmasq: Lightweight DHCP + DNS

dnsmasq is a single lightweight daemon that provides both DHCP and DNS caching/forwarding. It is commonly used in:

  • Home routers and embedded Linux (OpenWrt)
  • Small office networks needing a simple combined service
  • Lab environments and virtual network setups
  • pfSense uses dnsmasq (or Unbound) internally for DNS forwarding

Basic Installation and Configuration

sudo apt install -y dnsmasq

The primary config file is /etc/dnsmasq.conf. A minimal DHCP-only configuration:

# /etc/dnsmasq.conf — minimal DHCP example # Listen on this interface only interface=ens18 # DHCP range: start, end, lease duration dhcp-range=192.168.30.100,192.168.30.200,24h # Default gateway (option 3) dhcp-option=3,192.168.30.1 # DNS server (option 6) dhcp-option=6,192.168.10.10 # Static reservation by MAC address dhcp-host=aa:bb:cc:dd:ee:ff,printer-01,192.168.30.102

🪟 6 · Configuring DHCP on Windows Server

On Windows Server, DHCP is a server role installed and managed through Server Manager. The GUI-based scope wizard makes initial setup straightforward, and the DHCP console provides ongoing management of scopes, reservations, and options.

Step-by-Step: Adding the DHCP Server Role

  1. 1
    Open Server Manager → click Manage → Add Roles and Features.
  2. 2
    Before You Begin — confirm the server has a static IP address configured. A DHCP server must not itself be a DHCP client (its address must not change).
  3. 3
    Select Role-based or feature-based installation → choose your server → on the Server Roles page check DHCP Server → click Add Features when prompted → click through to Install.
  4. 4
    After installation, click the notification flag in Server Manager and select Complete DHCP configuration. This post-install wizard creates the DHCP Administrators security group and authorizes the server in Active Directory (if domain-joined).
  5. 5
    Open the DHCP console: Tools → DHCP. Expand your server name → expand IPv4.

Step-by-Step: Creating a Scope with the Scope Wizard

  1. 1
    Right-click IPv4 → New Scope… The New Scope Wizard opens.
  2. 2
    Scope Name: Enter a descriptive name (e.g., Clients-Subnet-30) and an optional description.
  3. 3
    IP Address Range: Enter the start and end of your pool (e.g., 192.168.30.100 to 192.168.30.200) and the subnet mask (255.255.255.0).
  4. 4
    Add Exclusions and Delay: Enter any IP addresses or ranges within the pool that should not be assigned dynamically (e.g., 192.168.30.100 if you reserve it for a router).
  5. 5
    Lease Duration: Set the lease time. The default is 8 days; adjust based on your environment (e.g., 24 hours for a lab).
  6. 6
    Configure DHCP Options — Yes, I want to configure these options now. Proceed through:
    • Router (default gateway): 192.168.30.1
    • DNS Servers: your internal DNS server IP
    • Domain Name: lab.local
  7. 7
    Activate Scope — Yes, I want to activate this scope now. Click Finish.

Adding a Reservation

  1. 1
    Expand your scope → right-click Reservations → New Reservation…
  2. 2
    Enter a Reservation name, the IP address to reserve, and the device's MAC address (without dashes or colons, e.g., AABBCCDDEEFF). Select Both for supported types. Click Add.

Viewing Active Leases

In the DHCP console, expand your scope and click Address Leases. You will see all currently active leases with the client IP, name, lease expiry, and MAC address. Right-clicking a lease allows you to delete it or convert it to a reservation.

💡 Authorization Requirement
In an Active Directory domain, a Windows DHCP server must be authorized in AD before it will issue leases. This is a built-in rogue DHCP protection mechanism — only explicitly authorized servers can operate. In a workgroup (non-domain) environment, authorization is not required.

🌐 7 · DHCP in Virtualized Environments & pfSense

Virtualized environments introduce additional DHCP considerations because multiple virtual networks can exist on the same physical host, each potentially needing its own DHCP service.

Proxmox and Virtual Networks

In Proxmox, each VM is connected to a virtual bridge (e.g., vmbr0, vmbr1). VMs on the same bridge are on the same Layer 2 segment and will receive DHCP from whatever server is reachable on that bridge. Key rules:

  • Only one active DHCP server should exist per Layer 2 segment. Multiple servers on the same segment create a "rogue DHCP" situation (see Section 8).
  • Proxmox itself does not provide DHCP by default for VM networks — you must deploy a DHCP server VM or use pfSense.
  • If the Proxmox management interface uses a separate bridge, VMs on lab bridges are isolated from the management DHCP server.

pfSense as a DHCP Server

pfSense includes a built-in DHCP server (powered by dnsmasq/ISC DHCP internally) that can be enabled per interface through the web UI. This is the approach you will use in Project 2.

  1. 1
    Log in to the pfSense web UI → navigate to Services → DHCP Server.
  2. 2
    Select the interface tab for your Clients subnet (e.g., OPT2 or whatever you name it).
  3. 3
    Check Enable DHCP server on [interface] interface.
  4. 4
    Set the Address Pool Range (From / To). The subnet and gateway are automatically inferred from the pfSense interface IP.
  5. 5
    Under Servers, enter your DNS server address. Optionally set the domain name.
  6. 6
    Click Save. pfSense will immediately begin serving DHCP on that interface.

DHCP Relay Agents

DHCP Discover messages are broadcasts and do not cross router boundaries. In a multi-subnet environment, a DHCP relay agent (also called a DHCP helper) is configured on the router/firewall to forward DHCP broadcasts as unicast packets to a central DHCP server. pfSense supports DHCP relay under Services → DHCP Relay. For Project 2, since pfSense is the DHCP server for the Clients subnet and directly connects to that segment, a relay is not needed.

📌 Project 2 Architecture
In your Project 2 topology, pfSense connects directly to three internal subnets (Management, Servers, Clients). You will enable the pfSense DHCP server on the Clients interface only. The Management and Servers subnets use static IP addressing, so no DHCP is needed there.

🚨 8 · Mini-Case: The Rogue DHCP Problem

🚨 Scenario: Monday Morning at Mendocino Tech

It's Monday morning at Mendocino Tech's main office. The helpdesk starts receiving calls at 8:05 AM — users on the second floor cannot reach the internet, internal file servers, or the company intranet. Users on the first floor are unaffected.

A network technician runs ipconfig /all on an affected workstation and notices something odd:

  • IP Address: 192.168.30.147 ✓ (correct subnet)
  • Subnet Mask: 255.255.255.0 ✓
  • Default Gateway: 192.168.30.254 ✗ (should be 192.168.30.1)
  • DNS Servers: 8.8.8.8, 8.8.4.4 ✗ (should be internal DNS 192.168.10.10)
  • DHCP Server: 192.168.30.87 ✗ (the authorized server is 192.168.10.5)

The technician checks the authorized DHCP server's lease log and confirms it has issued no leases to second-floor clients since 7:50 AM. Something on the network answered DHCP Discover messages before the legitimate server could respond.

❓ Question 1: What is happening here, and what is the root cause?

Root Cause — Rogue DHCP Server: An unauthorized DHCP server at IP 192.168.30.87 has appeared on the second-floor network segment. It is responding to DHCP Discover broadcasts faster than the legitimate server, so affected clients accept its DHCP Offer first (DORA — first offer wins). The rogue server is distributing a wrong default gateway (192.168.30.254 — likely a non-existent or attacker- controlled device) and external DNS servers. As a result, clients have no path to internal resources. This could be an accidentally connected consumer router, a misconfigured VM, or a deliberate man-in-the-middle attack.

❓ Question 2: What are the immediate steps to restore service?

Immediate Remediation:
  1. Identify the rogue device: Use the switch's ARP table or MAC address table to find which port 192.168.30.87's MAC address is connected to: show mac address-table | include [MAC] on a managed switch.
  2. Isolate the port: Administratively shut down the switch port to stop the rogue server immediately.
  3. Force lease renewal on clients: Run ipconfig /release then ipconfig /renew (Windows) or sudo dhclient -r && sudo dhclient (Linux) on affected machines to obtain correct settings from the legitimate server.
  4. Investigate the device: Physically locate the device on the identified port and remove or reconfigure it.

❓ Question 3: What long-term controls prevent this from happening again? (Connect to at least one Module 9.3 concept.)

Prevention Controls:
  • DHCP Snooping (Layer 2 switch feature): Configures switch ports as "trusted" (uplinks to authorized DHCP servers) or "untrusted" (access ports). DHCP Offer and Acknowledge messages arriving on untrusted ports are silently dropped, blocking rogue servers at the switch level.
  • Active Directory DHCP Authorization: In Windows domains, only AD-authorized DHCP servers can operate. A rogue Windows DHCP server will detect the authorized server and shut itself down.
  • Firewall rules (Module 9.3): pfSense or perimeter firewall rules can block DHCP traffic (UDP ports 67–68) from unauthorized source IPs or MAC addresses, preventing rogue servers from reaching clients across routed segments.
  • Network Access Control (NAC): 802.1X port authentication prevents unauthorized devices from connecting to the network at all — the rogue device would never get a switch port if NAC were enforced.
  • Monitoring: DHCP server logs and a SIEM (like Splunk — see Module 9.5) can alert on unexpected DHCP Offer messages from unknown source IPs.
🔒 CSLO 2 Connection
The Rogue DHCP scenario directly illustrates CSLO 2 — Apply physical and network data security techniques. DHCP is a critical infrastructure service; unauthorized servers represent both a denial-of-service risk and a potential man-in-the-middle attack vector. The mitigations (DHCP snooping, firewall rules, NAC) are all network security techniques.

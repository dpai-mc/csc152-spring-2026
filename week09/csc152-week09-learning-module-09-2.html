🌐 1 · Why DNS Servers Matter

In Week 8 (Module 8.3) you configured systems as DNS clients — you pointed /etc/resolv.conf or the Windows DNS client settings at a resolver and used nslookup and dig to query it. This week you become the administrator who runs that resolver and the authoritative zones behind it.

The Client vs. Server Perspective

PerspectiveRoleWhat it doesWeek covered
DNS Client Every networked device Sends queries to a resolver; uses answers to connect to services Week 8 (Module 8.3)
DNS Resolver (Recursive) Caching/forwarding server Receives client queries; walks the hierarchy to find answers; caches results Week 9 (this module)
DNS Authoritative Server Zone owner Holds the definitive records for a zone; answers with authority Week 9 (this module)

Why Run Your Own DNS Server?

In a lab or enterprise environment, a locally-hosted DNS server provides several critical capabilities that public resolvers (like 8.8.8.8) cannot:

  • Internal name resolution: Hostnames like fileserver.lab.local only exist inside your network. A public DNS server has no knowledge of them.
  • Reverse lookups: Mapping IP addresses back to hostnames (PTR records) for logging, monitoring, and Active Directory authentication requires a locally-managed reverse zone.
  • Split-horizon DNS: Returning different answers for internal vs. external clients (e.g., internal users reach a server at its private IP; external users reach the same name via a public IP).
  • Performance: A local caching resolver reduces latency for repeated queries and reduces external DNS traffic.
  • Control: You decide which names exist, which records they point to, and how long they are cached.

Project 2 Part B requires you to configure a DNS server with forward and reverse lookup zones for your internal lab.local domain. Every concept in this module directly supports that deliverable. The DHCP server you configured in Module 9.1 will distribute your DNS server's IP address to clients via Option 6 — completing the name resolution chain.

🏛️ 2 · The DNS Hierarchy

DNS is a distributed, hierarchical database. No single server knows every hostname on the internet. Instead, the namespace is divided into a tree of zones, each managed by a different authority. A recursive resolver navigates this tree — starting at the top — to answer any query.

The Five Levels of the DNS Hierarchy

1
Root Zone
.
The top of the DNS tree, represented by a single dot (.). Managed by ICANN/IANA. 13 sets of root server addresses (A–M) distributed globally via anycast. Root servers know the addresses of all TLD nameservers.
▼
2
TLD Servers
.com .edu .org .net
Top-Level Domain servers manage all domains directly beneath them. Generic TLDs (.com, .org, .net) are managed by registries like Verisign. Country-code TLDs (.uk, .ca, .jp) are managed by national authorities. TLD servers know the authoritative nameservers for every registered domain.
▼
3
Authoritative NS
example.com
The authoritative nameserver holds the actual DNS records (A, MX, CNAME, etc.) for a specific domain. This is the server you configure when you run your own DNS zone. It answers queries with the AA (Authoritative Answer) flag set.
▼
4
Recursive Resolver
8.8.8.8 / ISP DNS
The recursive resolver (also called a full-service resolver or caching resolver) does the work of walking the hierarchy on behalf of clients. It caches results for the TTL period to avoid repeating the full lookup. Your local DNS server in Project 2 acts as a recursive resolver for internal clients.
▼
5
DNS Client (Stub)
Your workstation
The client (stub resolver) sends a simple query to its configured resolver and waits for a complete answer. It does not walk the hierarchy itself. This is the DNS client behavior you configured in Week 8.

How a Full DNS Resolution Works: Step by Step

When a client queries www.example.com for the first time (no cached answer available):

  1. 1
    Client → Recursive Resolver: The workstation's stub resolver sends a query for www.example.com to its configured DNS server (e.g., your lab DNS server or 8.8.8.8).
  2. 2
    Resolver → Root Server: The recursive resolver asks a root server: "Who handles .com?" The root server responds with the addresses of the .com TLD nameservers.
  3. 3
    Resolver → TLD Server: The resolver asks a .com TLD server: "Who is authoritative for example.com?" The TLD server responds with the NS records pointing to example.com's authoritative nameservers.
  4. 4
    Resolver → Authoritative Server: The resolver asks the authoritative nameserver for example.com: "What is the A record for www.example.com?" The authoritative server responds with the IP address and the AA flag set.
  5. 5
    Resolver → Client: The resolver returns the answer to the client and caches it for the duration of the record's TTL. Subsequent queries for the same name are answered from cache without repeating steps 2–4.
💡 Internal DNS and Forwarding
In your Project 2 lab, your internal DNS server handles queries for lab.local authoritatively. For all other names (like google.com), it forwards the query to an upstream resolver (e.g., 8.8.8.8) rather than walking the root hierarchy itself. This is called a forwarder configuration and is standard practice for internal DNS servers.

Key DNS Terminology

TermDefinition
ZoneAn administrative unit of the DNS namespace managed by a specific nameserver. A zone contains resource records for a domain or subdomain.
TTL (Time to Live)How long (in seconds) a resolver may cache a DNS record before it must re-query the authoritative server.
FQDNFully Qualified Domain Name — the complete domain name including all labels from the host to the root, e.g., server1.lab.local. (note the trailing dot).
DelegationThe process by which a parent zone hands off authority for a subdomain to a different nameserver, enabling the distributed hierarchy.

📂 3 · Zone Types

A zone is the unit of administrative authority in DNS. DNS servers can host different types of zones, each serving a distinct purpose. Understanding zone types is essential before configuring BIND9 or Windows Server DNS.

🔵 Primary Zone

The read/write master copy of a zone's data. All DNS record changes (adds, edits, deletes) are made on the primary zone. The primary zone file is stored locally on the primary nameserver.

When to use: Your authoritative DNS server for lab.local in Project 2 will host a primary zone.

🟢 Secondary Zone

A read-only copy of a primary zone, obtained via a zone transfer (AXFR/IXFR) from the primary server. Secondary zones provide redundancy and load distribution — if the primary server is unavailable, clients can still resolve names from the secondary.

When to use: Enterprise environments with multiple DNS servers for fault tolerance.

🟠 Stub Zone

Contains only the NS records and SOA for a remote zone — not the full record set. A stub zone tells the local server where to find the authoritative nameservers for a delegated domain, without maintaining a full secondary copy.

When to use: Multi-domain environments where you need to keep NS delegations current automatically.

🟣 Forward Zone

Not a zone in the traditional sense — a forward zone (or conditional forwarder) instructs the DNS server to forward all queries for a specific domain to a designated upstream server, rather than resolving them locally or via the root hierarchy.

When to use: Forwarding queries for a partner organization's domain or for split-horizon DNS configurations.

Active Directory-Integrated Zones (Windows)

In Windows Server environments joined to an Active Directory domain, DNS zones can be stored in the Active Directory database rather than flat zone files. AD-integrated zones provide:

  • Multi-master replication: Any domain controller running DNS can accept record updates (no single primary bottleneck).
  • Secure dynamic updates: Only authenticated domain members can register or update their DNS records.
  • Automatic replication: Zone data replicates via AD replication, not zone transfers.
📌 Lab Scope
For Project 2, you will create a primary zone for lab.local (forward lookup) and a corresponding primary reverse zone. Secondary, stub, and AD-integrated zones are covered here for awareness and Quiz 9 preparation.

Zone Transfer Security

Zone transfers copy the entire contents of a zone from a primary to a secondary server. If left unrestricted, an attacker can perform an unauthorized zone transfer (dig axfr @nameserver domain) and enumerate every hostname and IP address in your network.

⚠️ Best Practice: Restrict Zone Transfers
Always configure zone transfers to be permitted only to known secondary server IP addresses. In BIND9 this is the allow-transfer directive. In Windows Server DNS, right-click the zone → Properties → Zone Transfers tab → allow only to listed servers.

📝 4 · Resource Record Types

A DNS zone file is made up of resource records (RRs) — individual entries that map names to data. Each record has a type that defines what kind of information it carries. Knowing the common record types is essential for both configuring DNS servers and troubleshooting name resolution problems.

Resource Record Format

Every resource record follows this general structure in a zone file:

; name TTL class type rdata server1 3600 IN A 192.168.10.20 @ 3600 IN MX 10 mail.lab.local.
  • name: The hostname or owner of the record. @ refers to the zone apex (the domain itself).
  • TTL: Time to Live in seconds. If omitted, the zone's $TTL default is used.
  • class: Almost always IN (Internet).
  • type: The record type (A, MX, CNAME, etc.).
  • rdata: The record data — the answer to the query.

The Eight Essential Record Types

A Record — IPv4 Address Mapping

The A record (Address record) maps a hostname to an IPv4 address. It is the most fundamental record type and the one you will create most often. When a client queries server1.lab.local, the DNS server returns the A record containing the IPv4 address.

; A records — hostname to IPv4 server1 3600 IN A 192.168.10.20 fileserver 3600 IN A 192.168.10.21 webserver 3600 IN A 192.168.10.22

AAAA Record — IPv6 Address Mapping

The AAAA record (quad-A record) does the same job as an A record but maps a hostname to an IPv6 address. The name comes from the fact that IPv6 is 128 bits — four times the 32 bits of IPv4, hence four A's.

; AAAA record — hostname to IPv6 server1 3600 IN AAAA 2001:db8::10:20
💡 Quiz 9 Focus
The Knowledge Check and Quiz 9 will ask you to identify which record maps a hostname to an IPv4 address. The answer is the A record. AAAA is IPv6; PTR is the reverse (IP → hostname).

PTR Record — Reverse Lookup (IP → Hostname)

The PTR record (Pointer record) is the reverse of an A record — it maps an IP address back to a hostname. PTR records live in a special reverse lookup zone (covered in Section 5) and are queried when a system needs to resolve an IP address to a name — for example, in mail server validation, logging, and Active Directory authentication.

The IP address is written in reverse octet order with the suffix .in-addr.arpa. appended:

; PTR record in the 10.168.192.in-addr.arpa. reverse zone ; IP 192.168.10.20 → reverse notation: 20.10.168.192.in-addr.arpa. 20 3600 IN PTR server1.lab.local. 21 3600 IN PTR fileserver.lab.local.
📌 Trailing Dot
Notice the trailing dot after lab.local. — this marks the name as a Fully Qualified Domain Name (FQDN). Without it, the DNS server appends the zone name, creating an incorrect result. Always use the trailing dot in PTR record targets and NS record targets.

MX Record — Mail Exchanger

The MX record (Mail Exchanger) specifies the mail server responsible for accepting email for a domain. When another mail server needs to deliver email to user@lab.local, it queries the MX record for lab.local to find the correct mail server.

MX records include a priority value (lower number = higher priority). Multiple MX records provide redundancy — if the primary mail server is unavailable, the sending server tries the next-lowest priority.

; MX records — mail routing for lab.local ; @ refers to the zone apex (lab.local itself) @ 3600 IN MX 10 mail1.lab.local. # primary (priority 10) @ 3600 IN MX 20 mail2.lab.local. # backup (priority 20)
⚠️ MX Must Point to an A Record
An MX record must point to a hostname that has an A (or AAAA) record. It cannot point directly to an IP address. If mail1.lab.local has no A record, email delivery will fail even if the MX record exists.

CNAME Record — Canonical Name (Alias)

The CNAME record (Canonical Name) creates an alias — it maps one hostname to another hostname (not directly to an IP address). The resolver follows the CNAME chain until it reaches an A record.

Common uses include:

  • Creating friendly names (www → actual server name)
  • Pointing multiple service names at one host
  • Allowing a hostname to move without updating every service
; CNAME records — aliases www 3600 IN CNAME webserver.lab.local. files 3600 IN CNAME fileserver.lab.local. intranet 3600 IN CNAME webserver.lab.local.
⚠️ CNAME Restrictions
A CNAME record cannot coexist with other records for the same name. In particular, you cannot create a CNAME for the zone apex (@ / the domain itself) — that name must have an SOA and NS record. Never point an MX record target at a CNAME.

NS Record — Nameserver

The NS record (Nameserver record) identifies which DNS servers are authoritative for a zone. Every zone must have at least one NS record at the zone apex. NS records are also used in delegation — a parent zone uses NS records to hand off authority for a subdomain to a child nameserver.

; NS records — authoritative nameservers for lab.local @ 3600 IN NS ns1.lab.local. @ 3600 IN NS ns2.lab.local. ; Corresponding A records (glue records) for the nameservers ns1 3600 IN A 192.168.10.10 ns2 3600 IN A 192.168.10.11

In your Project 2 lab, you will have a single nameserver. Your zone file will have one NS record pointing to that server's hostname, plus an A record for that hostname.

SOA Record — Start of Authority

The SOA record (Start of Authority) is the first record in every zone file. It identifies the zone's primary nameserver, the responsible party's email address, and several timing parameters that control zone replication and caching behavior.

; SOA record — must be the first record in the zone file @ IN SOA ns1.lab.local. admin.lab.local. ( 2026031601 ; Serial — YYYYMMDDNN (increment on every change) 3600 ; Refresh — how often secondary checks for updates (1 hr) 900 ; Retry — how often secondary retries after failure (15 min) 604800 ; Expire — how long secondary serves data without refresh (7 days) 300 ; Minimum TTL / negative cache TTL (5 min) )
FieldPurposeTypical Value

⚠️ Always Increment the Serial
When you modify a zone file, you must increment the serial number. Secondary nameservers compare the serial to decide whether to request a zone transfer. If the serial does not increase, secondaries will not pick up your changes.

SRV Record — Service Location

The SRV record (Service record) specifies the hostname and port of a server providing a specific service. It is used by applications to automatically discover service endpoints without hardcoded IP addresses or port numbers.

SRV records are heavily used by:

  • Active Directory: Clients find domain controllers, Kerberos servers, and LDAP servers via SRV records in the _msdcs subdomain.
  • VoIP (SIP): Phones discover SIP proxy servers.
  • XMPP/Jabber: Chat clients find messaging servers.
; SRV record format: ; _service._protocol.name TTL IN SRV priority weight port target _ldap._tcp.lab.local. 3600 IN SRV 0 100 389 dc1.lab.local. _kerberos._tcp.lab.local. 3600 IN SRV 0 100 88 dc1.lab.local.
📌 SRV and Active Directory
When you promote a Windows Server to a domain controller in a later week, the DNS server is automatically populated with dozens of SRV records. This is how domain-joined clients find authentication services — pure DNS-based service discovery.

↔️ 5 · Forward and Reverse Lookup Zones

DNS lookups travel in two directions. A forward lookup resolves a name to an address; a reverse lookup resolves an address back to a name. Each direction requires its own zone.

Forward Lookup Zone

A forward lookup zone maps hostnames → IP addresses. It is the standard zone type that most people think of when they hear "DNS zone." The zone is named after the domain it serves (e.g., lab.local).

A complete forward zone file for lab.local looks like this:

; /etc/bind/zones/db.lab.local — Forward lookup zone for lab.local $TTL 3600 @ IN SOA ns1.lab.local. admin.lab.local. ( 2026031601 ; Serial 3600 ; Refresh 900 ; Retry 604800 ; Expire 300 ; Minimum TTL ) ; Nameserver records @ IN NS ns1.lab.local. ; A records — hostname to IPv4 ns1 IN A 192.168.10.10 server1 IN A 192.168.10.20 fileserver IN A 192.168.10.21 webserver IN A 192.168.10.22 ; CNAME alias www IN CNAME webserver.lab.local. ; MX record for mail @ IN MX 10 mail.lab.local. mail IN A 192.168.10.25
💡 Verification Command
After creating a forward zone, test resolution from a client with:
nslookup server1.lab.local <dns-server-ip>
or on Linux: dig @192.168.10.10 server1.lab.local A

Reverse Lookup Zone

A reverse lookup zone maps IP addresses → hostnames using PTR records. The zone name is derived from the network address, written in reverse octet order, with .in-addr.arpa appended.

For the 192.168.10.0/24 network:

  • Network: 192.168.10.0/24
  • Reverse zone name: 10.168.192.in-addr.arpa
  • Zone file: /etc/bind/zones/db.192.168.10 (by convention)
; /etc/bind/zones/db.192.168.10 — Reverse lookup zone $TTL 3600 @ IN SOA ns1.lab.local. admin.lab.local. ( 2026031601 ; Serial 3600 ; Refresh 900 ; Retry 604800 ; Expire 300 ; Minimum TTL ) ; Nameserver for this reverse zone @ IN NS ns1.lab.local. ; PTR records — last octet only (zone supplies the rest) 10 IN PTR ns1.lab.local. 20 IN PTR server1.lab.local. 21 IN PTR fileserver.lab.local. 22 IN PTR webserver.lab.local.
💡 Verification Command
Test a reverse lookup with:
nslookup 192.168.10.20 <dns-server-ip>
or: dig @192.168.10.10 -x 192.168.10.20
A successful result returns the PTR record pointing to server1.lab.local.
📌 Why Reverse Zones Matter
Reverse lookup zones are required for:
  • Active Directory domain join and authentication
  • Mail server spam validation (PTR record must match the sending server's hostname)
  • Security logging (log entries show hostnames, not just IP addresses)
  • Some monitoring and management tools that perform reverse lookups on discovered hosts

🐧 6 · Configuring DNS on Linux (BIND9)

BIND9 (Berkeley Internet Name Domain, version 9) is the most widely deployed DNS server software in the world. On Ubuntu Server it is packaged as bind9 and is the standard choice for authoritative and recursive DNS in enterprise Linux environments.

BIND9 Configuration File Structure

BIND9 uses several key files, all located under /etc/bind/:

FilePurpose
/etc/bind/named.confMain entry point — includes the other config files
/etc/bind/named.conf.optionsGlobal server options: forwarders, recursion, listen addresses
/etc/bind/named.conf.localYour custom zone declarations (this is where you add your zones)
/etc/bind/named.conf.default-zonesBuilt-in zones: root hints, localhost, loopback reverse
/etc/bind/zones/db.lab.localYour forward zone file (created by you)
/etc/bind/zones/db.192.168.10Your reverse zone file (created by you)

Step-by-Step: Installing and Configuring BIND9

  1. 1
    Install BIND9
    sudo apt update sudo apt install -y bind9 bind9utils bind9-doc dnsutils
    bind9utils provides named-checkconf and named-checkzone — essential syntax-checking tools. dnsutils provides dig and nslookup for testing.
  2. 2
    Configure global options in named.conf.options
    sudo nano /etc/bind/named.conf.options
    // /etc/bind/named.conf.options options { directory "/var/cache/bind"; // Forward unresolved queries to upstream resolvers forwarders { 8.8.8.8; 8.8.4.4; }; forward only; // Use forwarders exclusively (don't walk the root) // Allow recursive queries from internal networks only recursion yes; allow-recursion { 192.168.0.0/16; 127.0.0.1; }; // Listen on the server's internal IP and loopback listen-on { 127.0.0.1; 192.168.10.10; }; // Disable zone transfers to everyone by default allow-transfer { none; }; dnssec-validation auto; };
    ⚠️ Restrict Recursion
    Setting allow-recursion to your internal subnets only prevents your server from becoming an open resolver — a server that answers recursive queries from anyone on the internet, which is commonly abused in DNS amplification DDoS attacks.
  3. 3
    Declare your zones in named.conf.local
    sudo nano /etc/bind/named.conf.local
    // /etc/bind/named.conf.local // Forward lookup zone zone "lab.local" { type master; file "/etc/bind/zones/db.lab.local"; allow-transfer { none; }; // No secondary servers in this lab }; // Reverse lookup zone for 192.168.10.0/24 zone "10.168.192.in-addr.arpa" { type master; file "/etc/bind/zones/db.192.168.10"; allow-transfer { none; }; };
  4. 4
    Create the zones directory and zone files
    sudo mkdir -p /etc/bind/zones
    Create the forward zone file (content from Section 5):
    sudo nano /etc/bind/zones/db.lab.local
    Create the reverse zone file:
    sudo nano /etc/bind/zones/db.192.168.10
    Use the zone file examples from Section 5 as your templates.
  5. 5
    Check configuration and zone file syntax
    # Check named.conf syntax sudo named-checkconf # Check forward zone file syntax sudo named-checkzone lab.local /etc/bind/zones/db.lab.local # Check reverse zone file syntax sudo named-checkzone 10.168.192.in-addr.arpa /etc/bind/zones/db.192.168.10
    If no errors are reported, the syntax is valid. Both named-checkzone commands should end with OK.
  6. 6
    Start and enable BIND9
    sudo systemctl start bind9 sudo systemctl enable bind9 sudo systemctl status bind9
    Look for active (running). If the service fails, check the log with sudo journalctl -u bind9 -n 30.
  7. 7
    Test resolution from a client
    # Forward lookup — should return 192.168.10.20 dig @192.168.10.10 server1.lab.local A # Reverse lookup — should return server1.lab.local dig @192.168.10.10 -x 192.168.10.20 # Using nslookup (cross-platform) nslookup server1.lab.local 192.168.10.10 nslookup 192.168.10.20 192.168.10.10 # Verify external forwarding works dig @192.168.10.10 google.com A

🪟 7 · Configuring DNS on Windows Server

On Windows Server, DNS is a server role installed through Server Manager and managed through the DNS Manager console (dnsmgmt.msc). The GUI-driven workflow makes zone creation straightforward, and Windows DNS integrates tightly with Active Directory when the server is a domain controller.

Step 1: Install the DNS Server Role

  1. 1
    Open Server Manager → click Manage → Add Roles and Features.
  2. 2
    Confirm the server has a static IP address before proceeding. A DNS server must not use DHCP for its own address.
  3. 3
    Select Role-based or feature-based installation → choose your server → on the Server Roles page check DNS Server → click Add Features → click through to Install.
  4. 4
    After installation, open the DNS Manager: Server Manager → Tools → DNS.

Step 2: Create a Forward Lookup Zone

  1. 1
    In DNS Manager, expand your server name → right-click Forward Lookup Zones → New Zone…
  2. 2
    Zone Type: Select Primary zone. (Uncheck "Store the zone in Active Directory" if the server is not a domain controller.)
  3. 3
    Zone Name: Enter lab.local.
  4. 4
    Zone File: Accept the default filename (lab.local.dns) → click through to Finish.
  5. 5
    Add A records: Right-click the new lab.local zone → New Host (A or AAAA)…
    • Name: server1 | IP: 192.168.10.20 → check Create associated PTR record → Add Host
    • Repeat for each server in your lab
    💡 Create PTR Records Automatically
    Checking Create associated PTR record when adding an A record will automatically add the corresponding PTR record to the reverse lookup zone — but only if that reverse zone already exists. Create the reverse zone first (Step 3 below), then add your A records.

Step 3: Create a Reverse Lookup Zone

  1. 1
    Right-click Reverse Lookup Zones → New Zone… → Primary zone.
  2. 2
    Reverse Lookup Zone Name: Select IPv4 Reverse Lookup Zone → enter the Network ID: 192.168.10. Windows automatically constructs the zone name 10.168.192.in-addr.arpa.
  3. 3
    Accept the default zone file name → click through to Finish.
  4. 4
    Add PTR records manually (if not created automatically): Right-click the reverse zone → New Pointer (PTR)… → enter the host IP (last octet only, e.g., 20) and the FQDN (server1.lab.local).

Step 4: Configure a Conditional Forwarder

A conditional forwarder tells Windows DNS to forward queries for a specific domain to a designated server, rather than resolving them via the root hierarchy. This is useful when you have multiple DNS domains in your environment (e.g., forwarding queries for a partner's domain to their DNS server).

  1. 1
    In DNS Manager, right-click Conditional Forwarders → New Conditional Forwarder…
  2. 2
    DNS Domain: Enter the domain to forward (e.g., partner.local).
  3. 3
    IP addresses of the master servers: Enter the IP of the DNS server authoritative for that domain. Click OK.

Step 5: Configure General Forwarders

General forwarders handle all queries that do not match a local zone or conditional forwarder — typically internet names.

  1. 1
    Right-click your server name in DNS Manager → Properties → Forwarders tab.
  2. 2
    Click Edit… → add forwarder IP addresses (e.g., 8.8.8.8 and 8.8.4.4) → click OK.

Step 6: Verify with nslookup

REM From a Windows command prompt or PowerShell on the DNS server REM Forward lookup nslookup server1.lab.local 192.168.10.10 REM Reverse lookup nslookup 192.168.10.20 192.168.10.10 REM PowerShell alternative Resolve-DnsName server1.lab.local -Server 192.168.10.10 Resolve-DnsName 192.168.10.20 -Server 192.168.10.10 -Type PTR

🔒 8 · DNS Security Overview

DNS was designed in the early 1980s for a small, trusted academic network. Security was not a primary concern. As the internet grew, attackers discovered that manipulating DNS responses is one of the most effective ways to redirect users to malicious destinations without their knowledge. Two topics are essential for every DNS administrator: DNS cache poisoning and DNSSEC.

DNS Cache Poisoning (DNS Spoofing)

DNS cache poisoning (also called DNS spoofing) is an attack in which an attacker injects fraudulent DNS records into a resolver's cache. Once poisoned, the resolver returns the attacker's fake IP address for a legitimate domain name — silently redirecting users to a malicious server without any visible warning.

How It Works

  1. 1
    The attacker sends a query to the target recursive resolver for a domain they control (e.g., evil.com), triggering the resolver to send a query to the authoritative server.
  2. 2
    Before the legitimate response arrives, the attacker floods the resolver with forged responses claiming to be from the authoritative server for a different domain (e.g., bank.com), containing a malicious IP.
  3. 3
    If the forged response matches the resolver's transaction ID and source port (both guessable in older implementations), the resolver accepts and caches the fraudulent record.
  4. 4
    All clients using that resolver are now directed to the attacker's server when they query bank.com, for the duration of the poisoned record's TTL.

The Kaminsky Attack (2008)

Security researcher Dan Kaminsky discovered a critical amplification of the poisoning technique in 2008 — by querying random subdomains of the target domain, an attacker could continuously attempt poisoning without waiting for TTL expiry, making the attack practical against most resolvers of the era. The disclosure led to emergency patches across the DNS ecosystem and accelerated adoption of source port randomization.

Mitigations

  • Source port randomization: Using a random UDP source port (rather than a fixed port 53) for outgoing queries dramatically increases the difficulty of guessing the correct transaction ID + port combination. Modern resolvers implement this by default.
  • DNSSEC: Cryptographic signing of DNS responses makes forged records detectable (covered in the next tab).
  • 0x20 encoding: Randomizing the case of query letters (e.g., BaNk.CoM) and requiring responses to mirror the same case adds another layer of verification.
  • DNS over HTTPS (DoH) / DNS over TLS (DoT): Encrypting DNS queries in transit prevents eavesdropping and man-in-the-middle injection, though they do not directly address cache poisoning of the resolver itself.
🔒 CSLO 2 Connection
DNS cache poisoning is a network data security threat. Mitigations (source port randomization, DNSSEC, DoH/DoT) are network data security techniques — directly aligned with CSLO 2: Apply physical and network data security techniques.

DNSSEC — DNS Security Extensions

DNSSEC (DNS Security Extensions) adds cryptographic signatures to DNS records, allowing resolvers to verify that responses are authentic and have not been tampered with in transit. DNSSEC does not encrypt DNS traffic — it authenticates it.

How DNSSEC Works

DNSSEC uses public-key cryptography. The zone owner generates a key pair:

  • Zone Signing Key (ZSK): Signs the individual resource records in the zone.
  • Key Signing Key (KSK): Signs the ZSK itself, creating a chain of trust.

Each signed record set (RRset) is accompanied by an RRSIG record containing the digital signature. A resolver that supports DNSSEC validation can verify the signature against the public key published in the zone's DNSKEY record.

The Chain of Trust

DNSSEC validation relies on a chain of trust from the root zone downward:

  1. The root zone is signed with a well-known root key (the Root Trust Anchor, managed by ICANN).
  2. Each TLD zone (.com, .org) is signed and its key is validated by the root.
  3. Each domain zone (example.com) is signed and its key is validated by the TLD.
  4. A DNSSEC-validating resolver can follow this chain from root to the queried domain, rejecting any response whose signature does not verify.

New Record Types Introduced by DNSSEC

Record TypePurpose
RRSIGDigital signature for a resource record set
DNSKEYPublic key used to verify RRSIG signatures
DSDelegation Signer — links a child zone's key to the parent zone
NSEC / NSEC3Authenticated denial of existence — proves a name does not exist

DNSSEC in BIND9

In BIND9, enabling DNSSEC validation in named.conf.options is straightforward:

// In named.conf.options — enable DNSSEC validation dnssec-validation auto; // Uses built-in root trust anchors

The auto setting uses BIND's built-in root trust anchor, so no manual key configuration is needed for validation. Signing your own zones requires additional steps (key generation with dnssec-keygen, zone signing with dnssec-signzone) that are beyond the scope of this module but are covered in advanced DNS administration resources.

💡 Lab Scope
For Lab 9 Part B and Project 2 Part B, you are not required to sign your zones with DNSSEC. You should, however, ensure dnssec-validation auto; is present in your named.conf.options so your server can validate responses from DNSSEC-signed external zones.

Other DNS Security Threats

DNS Amplification (DDoS)

DNS amplification attacks exploit open resolvers — DNS servers that answer recursive queries from any source IP. An attacker sends small queries with a spoofed source IP (the victim's IP) to many open resolvers, asking for large record sets (e.g., ANY queries or DNSSEC-signed zones). The resolvers send large responses to the victim, amplifying the traffic volume by a factor of 50–100×.

⚠️ Mitigation: Never Run an Open Resolver
Always restrict recursion to your internal networks using allow-recursion in BIND9 (as shown in Section 6). An open resolver is both a security liability and a potential instrument of DDoS attacks against others.

DNS Hijacking

DNS hijacking redirects DNS queries to a malicious resolver. This can happen at multiple levels:

  • Router hijacking: Attacker changes the DNS server setting on a home or office router, redirecting all clients on that network.
  • Registrar hijacking: Attacker gains access to a domain registrar account and changes the NS records, redirecting all DNS for a domain to attacker-controlled servers.
  • ISP-level interception: Some ISPs intercept DNS queries and redirect NXDOMAIN (non-existent domain) responses to search or advertising pages.

DNS Tunneling

DNS tunneling encodes data inside DNS query and response packets to exfiltrate data or establish covert command-and-control channels through firewalls that permit DNS traffic. Because DNS (port 53) is almost universally allowed outbound, it is a common data exfiltration vector.

Detection methods include monitoring for:

  • Unusually long or high-entropy subdomain labels
  • High query volume from a single host to a single domain
  • Queries for rare or non-existent TLDs

🚨 9 · Mini-Case: Why Can't I Reach the File Server?

🚨 Scenario: Tuesday Morning at Mendocino Tech

It is Tuesday morning at Mendocino Tech's main office. The helpdesk receives three tickets in quick succession — users in the Servers subnet cannot reach the company file server by name. The complaints all look the same:

  • Attempting to connect to \\fileserver or \\fileserver.lab.local in Windows Explorer returns: "Windows cannot access \\fileserver"
  • Attempting ssh fileserver.lab.local from a Linux workstation returns: "ssh: Could not resolve hostname fileserver.lab.local: Name or service not known"

A network technician sits down at one of the affected workstations and runs a quick diagnostic sequence:

# Step 1: Can we reach the file server by IP? C:\> ping 192.168.10.21 Reply from 192.168.10.21: bytes=32 time=1ms TTL=128 ✓ SUCCESS # Step 2: Can we resolve the hostname? C:\> nslookup fileserver.lab.local Server: 192.168.10.10 Address: 192.168.10.10 *** 192.168.10.10 can't find fileserver.lab.local: Non-existent domain ✗ FAIL # Step 3: Is the DNS server responding at all? C:\> nslookup server1.lab.local Server: 192.168.10.10 Address: 192.168.10.10 Name: server1.lab.local Address: 192.168.10.20 ✓ server1 resolves fine # Step 4: Check the DNS zone for the missing record C:\> nslookup -type=A fileserver.lab.local 192.168.10.10 *** 192.168.10.10 can't find fileserver.lab.local: Non-existent domain

The DNS server at 192.168.10.10 is responding — it resolves server1.lab.local correctly. But it has no record for fileserver.lab.local, even though the file server at 192.168.10.21 is clearly online and reachable by IP.

❓ Question 1: What is the root cause of this problem? Be specific about which DNS component is missing and where.

Root Cause — Missing A Record: The forward lookup zone for lab.local on the DNS server at 192.168.10.10 does not contain an A record for fileserver. The file server exists on the network (it responds to ping by IP), but DNS has no mapping from the hostname fileserver.lab.local to the IP address 192.168.10.21. The DNS server correctly returns an NXDOMAIN (Non-Existent Domain) response because the record was never created — possibly because the server was added to the network after the zone was initially configured, or the A record was accidentally deleted.

❓ Question 2: Walk through the exact steps to fix this problem on both a BIND9 (Linux) DNS server and a Windows Server DNS server.

Fix on BIND9 (Linux):
  1. Open the forward zone file: sudo nano /etc/bind/zones/db.lab.local
  2. Add the missing A record: fileserver   IN  A  192.168.10.21
  3. Increment the serial number in the SOA record (e.g., 2026031601 → 2026031602). This step is mandatory — without it, secondary servers will not pick up the change.
  4. Check the zone file syntax: sudo named-checkzone lab.local /etc/bind/zones/db.lab.local
  5. Reload the zone (no full restart needed): sudo rndc reload
  6. Verify: dig @192.168.10.10 fileserver.lab.local A — should return 192.168.10.21.
Fix on Windows Server DNS Manager:
  1. Open DNS Manager → expand your server → expand Forward Lookup Zones → click lab.local.
  2. Right-click in the right pane → New Host (A or AAAA)…
  3. Name: fileserver | IP address: 192.168.10.21
  4. Check Create associated PTR record if the reverse zone exists.
  5. Click Add Host → Done.
  6. Verify: nslookup fileserver.lab.local 192.168.10.10 — should return 192.168.10.21.

❓ Question 3: After adding the A record, a second technician reports that nslookup fileserver.lab.local still fails on their workstation even after the fix was applied five minutes ago. What is the most likely explanation, and how do you resolve it?

Most Likely Explanation — Negative Cache TTL: When the workstation previously queried fileserver.lab.local and received an NXDOMAIN response, the resolver cached that negative result for the duration of the zone's Minimum TTL (the last field in the SOA record — typically 300 seconds / 5 minutes in our example zone). Even though the A record now exists, the workstation's local resolver is still serving the cached NXDOMAIN from before the fix.

Resolution options:
  • Windows: ipconfig /flushdns — clears the local DNS resolver cache immediately. Then retry nslookup fileserver.lab.local.
  • Linux: sudo systemd-resolve --flush-caches (if using systemd-resolved) or sudo rndc flush on the DNS server itself to flush the server-side cache.
  • Wait: The negative cache entry will expire on its own after the Minimum TTL period. For production environments, consider setting a low Minimum TTL (e.g., 300 seconds) to limit the impact of NXDOMAIN caching.

❓ Question 4: The file server is now reachable by hostname. However, a security audit tool reports that the reverse lookup for 192.168.10.21 fails. What additional record is missing, and why does it matter?

Missing Record — PTR Record in the Reverse Zone: The reverse lookup zone (10.168.192.in-addr.arpa) is missing a PTR record for the last octet 21, mapping it back to fileserver.lab.local.

Why it matters:
  • Security logging: Log entries that show IP addresses are harder to read and correlate than those showing hostnames. A missing PTR record means log analysis tools (including Splunk) display raw IPs instead of meaningful hostnames.
  • Active Directory: Some AD operations and Kerberos authentication checks perform reverse lookups; a missing PTR record can cause intermittent authentication failures.
  • Email: If the file server ever sends email, receiving mail servers check that the sending IP resolves to a hostname via PTR — missing PTR records cause mail to be flagged as spam.
Fix (BIND9): Add to the reverse zone file (db.192.168.10): 21  IN  PTR  fileserver.lab.local. Increment the SOA serial, then run sudo rndc reload.

Use the diagnostic sequence from this case (ping by IP → nslookup by name → check zone file → add missing record → reload → flush cache → verify) as your standard DNS troubleshooting workflow.

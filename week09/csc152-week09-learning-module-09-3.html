🛡️ 1 · Why Firewalls Matter

Every network-connected system is a potential target. A firewall is a security control that inspects network traffic and decides — based on a set of rules — whether to allow or block each packet. Without firewalls, every service running on every machine is reachable by anyone on the network, creating an enormous attack surface.

Defense in Depth

Firewalls are one layer in a defense-in-depth strategy. No single control is perfect; the goal is to make an attacker defeat multiple independent controls. A typical layered approach includes:

  • Perimeter firewall — sits at the network boundary, filters traffic entering and leaving the organization (e.g., pfSense on the WAN interface).
  • Internal segmentation firewall — separates internal zones (Management, Servers, Clients) so a compromised client cannot freely reach servers.
  • Host-based firewall — runs on each individual machine (iptables on Linux, Windows Defender Firewall on Windows), providing a last line of defense even if a network firewall is bypassed.

🔍 2 · Firewall Types

Not all firewalls work the same way. Understanding the three main generations helps you choose the right tool and interpret how a given firewall makes its decisions.

Packet Filter (Stateless)

The simplest firewall type. Each packet is evaluated independently against a list of rules based only on information in the packet headers: source IP, destination IP, source port, destination port, and protocol (TCP/UDP/ICMP).

  • No memory of connections — the firewall does not track whether a packet is part of an established session.
  • Fast and low overhead — simple header inspection requires minimal processing.
  • Limitation: Cannot distinguish a legitimate reply packet from an unsolicited inbound packet with the same ports. Requires explicit rules for both directions of a connection.
  • Example: Early ipchains on Linux; basic ACLs on Cisco routers.
⚠️ Stateless Limitation Example
To allow a web server to receive HTTP requests (TCP dst port 80) and send replies, a stateless firewall needs two rules: one allowing inbound TCP port 80, and one allowing outbound TCP from port 80. Without the second rule, responses are blocked. A stateful firewall handles this automatically.

Stateful Inspection

The most common type in modern firewalls, including pfSense. A stateful firewall maintains a connection tracking table (state table) that records the state of every active network connection.

  • Connection states tracked: NEW (first packet of a connection), ESTABLISHED (handshake complete, data flowing), RELATED (associated with an existing connection, e.g., FTP data channel), INVALID (does not match any known state).
  • Automatic reply handling: Once an outbound connection is allowed, reply packets matching that connection are automatically permitted without needing a separate rule.
  • More secure than stateless: Unsolicited inbound packets that do not match an established connection are blocked even if their port numbers would otherwise match a rule.
  • Example: pfSense, iptables with -m conntrack --ctstate, Windows Defender Firewall.
💡 pfSense Uses Stateful Inspection
pfSense is built on FreeBSD's pf (packet filter), which performs full stateful inspection. When you write a rule to allow traffic, pfSense automatically allows the return traffic for that connection — you do not need a separate rule for replies.

Application-Layer Firewall (Proxy / NGFW)

The most sophisticated type. An application-layer firewall (also called a proxy firewall or Next-Generation Firewall / NGFW) understands specific application protocols — HTTP, DNS, SMTP, etc. — and can make filtering decisions based on application content, not just headers.

  • Deep Packet Inspection (DPI): Reads inside the packet payload to identify the application, even if it uses a non-standard port.
  • URL filtering: Can allow or block specific websites or URL categories within HTTP/HTTPS traffic.
  • SSL/TLS inspection: Decrypts HTTPS traffic, inspects it, and re-encrypts it (requires certificate deployment).
  • Intrusion Prevention System (IPS): Detects and blocks known attack patterns within application traffic.
  • Examples: Palo Alto Networks, Fortinet FortiGate, pfSense with Snort/Suricata packages.
📌 Scope for This Course
Project 2 uses pfSense as a stateful packet filter. Application-layer features (Snort, Suricata) are available as pfSense packages but are beyond the scope of this project. Understanding the distinction helps you explain why pfSense rules are based on IP/port rather than application identity.

📋 3 · Rule Components, Ordering & Implicit Deny

Anatomy of a Firewall Rule

Every firewall rule is made up of match criteria (what traffic does this rule apply to?) and an action (what should happen when traffic matches?).

ComponentWhat It SpecifiesExamples
Source Where the traffic originates — IP address, subnet, or alias 10.0.3.0/24, any, CLIENTS_net
Destination Where the traffic is going — IP address, subnet, or alias 10.0.2.10, WAN net, any
Port / Protocol The transport protocol and port number(s) being used TCP/443, UDP/53, TCP/80,443
Action What to do when the criteria match Pass (Allow), Block (Drop), Reject
Interface Which network interface this rule is evaluated on WAN, LAN, SERVERS, CLIENTS
Logging Whether to record matched packets in the firewall log On / Off (recommended On for block rules)

Block vs. Reject: What Is the Difference?

  • Block (Drop): The packet is silently discarded. The sender receives no response and must wait for a timeout. Preferred for inbound WAN traffic — reveals nothing to attackers.
  • Reject: The packet is discarded and an ICMP "destination unreachable" or TCP RST is sent back to the sender. The sender knows immediately the connection was refused. Useful for internal rules where fast failure is more helpful than stealth.

Rule Ordering: First Match Wins

Firewall rules are evaluated top to bottom. The first rule whose criteria match the packet determines the action. Subsequent rules are never evaluated for that packet. This is called the first-match-wins model.

Implicit Deny-All

At the end of every firewall ruleset there is an invisible, unconfigured rule: "deny everything not explicitly permitted." This is the implicit deny-all (also called the default deny policy). You do not see it listed, but it always applies. Its consequence:

  • If a packet does not match any rule, it is blocked.
  • You must explicitly write rules to allow traffic you want.
  • Any service or port you forget to allow is automatically blocked — this is a security feature, not a bug.
⚠️ Rule Order Pitfall
Because the first match wins, a broad allow-all rule placed near the top of a ruleset will match before any specific block rules below it, making those block rules unreachable. Always place more specific rules above more general rules. For example: block SSH from Clients before a general allow-web rule for Clients.

Example: Project 2 Clients Interface Rules (Ordered)

#ActionSourceDestinationPort/ProtoReason
1BlockCLIENTS netLAN (Mgmt) netanySecurity isolation — clients cannot reach management
2BlockCLIENTS netSERVERS netTCP/22No direct SSH from clients to servers
3PassCLIENTS netSERVERS netTCP/53,80,443,445,3389Allow DNS, web, file share, RDP to servers
4PassCLIENTS netWAN netTCP/80,443Web browsing
5PassCLIENTS netWAN netUDP/53External DNS
6BlockCLIENTS netanyanyExplicit deny-all (also catches implicit deny; enables logging)

Rule 1 must come before Rule 3 — if Rule 3 were first, the broad allow would match management-bound traffic before Rule 1 could block it.

🗺️ 4 · Interface Zones: WAN, LAN, DMZ & Management

A zone is a logical grouping of network interfaces or segments that share a common trust level and security policy. Firewalls apply different default behaviors and rule sets to different zones.

Common Zone Definitions

ZoneTrust LevelTypical ContentsDefault Policy
WAN (Wide Area Network) Untrusted Internet, upstream campus network Block all inbound; allow established replies
LAN (Local Area Network) Trusted Internal hosts, workstations (or in pfSense: Management) Allow all outbound by default (pfSense default)
DMZ (Demilitarized Zone) Semi-trusted Public-facing servers (web, mail, DNS) accessible from WAN Allow specific inbound from WAN; restrict access to LAN
Management Highly trusted Admin workstations, jump hosts, monitoring systems Allow full outbound; restrict inbound to management tools only

Why This Segmentation?

  • Management isolation: Only administrators on the Management network (10.0.1.0/24) can access the pfSense web UI. A compromised client VM on 10.0.3.0/24 cannot reach the firewall management interface.
  • Server protection: Clients can only reach specific ports on servers (DNS, HTTP, file sharing). They cannot SSH directly into servers, limiting lateral movement.
  • Blast radius reduction: If the Clients segment is compromised (e.g., malware on a workstation), the attacker cannot freely move to the Servers or Management segments — firewall rules block them.
  • Compliance: Many security frameworks (PCI-DSS, HIPAA, SOC 2) require separation of sensitive systems from general user workstations.
🔒 CSLO 2 Connection
Network segmentation with interface zones is a core network data security technique (CSLO 2). By placing systems into zones and controlling inter-zone traffic with firewall rules, you implement the principle of least privilege at the network layer — each zone can only communicate with what it legitimately needs.

🐧 5 · Linux Host Firewalls: iptables & nftables

Linux includes a powerful built-in firewall framework in the kernel. The two main user-space tools for managing it are iptables (the traditional tool, still widely used) and nftables (the modern replacement, default on Debian/Ubuntu since 20.04 and RHEL/Rocky since 8). Both control the same underlying Netfilter kernel subsystem.

iptables Tables and Chains

iptables organizes rules into tables (by function) and chains (by traffic direction). For host firewall purposes, the filter table is the most important, and it contains three chains:

INPUT
Traffic destined for the local system itself. Controls what reaches services running on this host (e.g., SSH on port 22, a web server on port 80).
OUTPUT
Traffic originating from the local system. Controls what this host is allowed to send out (e.g., outbound HTTP requests, DNS queries).
FORWARD
Traffic passing through this system from one interface to another. Used when the host acts as a router or firewall (like pfSense internally). Not used for traffic to/from the host itself.

Targets (Actions)

Each rule ends with a target that specifies what to do when the rule matches:

ACCEPT — allow the packet through DROP — silently discard (no response to sender) REJECT — discard and send error back to sender

Common iptables Commands

# View current rules (with line numbers) sudo iptables -L -v -n --line-numbers # Allow incoming SSH (TCP port 22) on INPUT chain sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT # Allow incoming HTTP and HTTPS sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT # Allow established/related connections (stateful — required for replies) sudo iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT # Allow loopback interface traffic (always needed) sudo iptables -A INPUT -i lo -j ACCEPT # Set default policy: drop everything not explicitly allowed sudo iptables -P INPUT DROP sudo iptables -P FORWARD DROP sudo iptables -P OUTPUT ACCEPT # allow all outbound by default # Delete a rule by line number (e.g., line 3 in INPUT) sudo iptables -D INPUT 3 # Save rules so they persist after reboot (Ubuntu) sudo apt install iptables-persistent sudo netfilter-persistent save
⚠️ Order Matters in iptables Too
iptables rules are also evaluated top to bottom with first-match-wins. Always add your ESTABLISHED,RELATED accept rule before setting the default DROP policy, and add specific ACCEPT rules before any broad DROP rules.

nftables: The Modern Replacement

nftables replaces iptables with a cleaner syntax, better performance, and atomic rule updates. On Ubuntu 20.04+ and Rocky Linux 8+, nft is the preferred tool, though iptables commands still work via a compatibility layer.

# View current nftables ruleset sudo nft list ruleset # Simple nftables rule file: /etc/nftables.conf table inet filter { chain input { type filter hook input priority 0; policy drop; iif lo accept # loopback ct state established,related accept # stateful replies tcp dport 22 accept # SSH tcp dport { 80, 443 } accept # web } chain output { type filter hook output priority 0; policy accept; } chain forward { type filter hook forward priority 0; policy drop; } }

ufw: Simplified Front-End (Ubuntu)

ufw (Uncomplicated Firewall) is a simplified command-line front-end for iptables/nftables, common on Ubuntu desktop and server installations.

sudo ufw enable sudo ufw allow 22/tcp # allow SSH sudo ufw allow 80/tcp # allow HTTP sudo ufw deny 23/tcp # block Telnet sudo ufw status verbose

🪟 6 · Windows Defender Firewall

Windows Defender Firewall with Advanced Security is the built-in host-based firewall on all modern Windows systems, including Windows Server 2022. It is a stateful firewall that filters both inbound and outbound traffic based on configurable rules.

Network Profiles

Windows Defender Firewall applies different rule sets depending on which network profile is active for a given network adapter. There are three profiles:

ProfileWhen AppliedDefault StanceTypical Use
Domain Computer is joined to an Active Directory domain and can contact a domain controller Moderate — allows domain management traffic Corporate/enterprise networks
Private User marks the network as private (home/work network) Moderate — allows file/printer sharing Home networks, trusted office LANs
Public New or unknown network (default for unrecognized connections) Strict — blocks most inbound connections Coffee shops, airports, untrusted networks
💡 Server 2022 Default Profile
When your Windows Server 2022 VM connects to a Proxmox bridge that is not a domain network, it will likely use the Public profile — the most restrictive. If you find that services are unexpectedly blocked (e.g., file sharing, RDP), check which profile is active and whether you need to add rules to that profile.

Inbound vs. Outbound Rules

  • Inbound rules control traffic arriving at the Windows host from the network. By default, most inbound connections are blocked unless a rule explicitly allows them.
  • Outbound rules control traffic leaving the Windows host. By default, all outbound connections are allowed.

Creating a Custom Inbound Rule (GUI Walk-Through)

  1. 1
    Open Windows Defender Firewall with Advanced Security: Start → search "Windows Defender Firewall with Advanced Security", or open it from Server Manager → Tools.
  2. 2
    In the left pane, click Inbound Rules, then in the right pane click New Rule…
  3. 3
    Rule Type: Select Port (to control by port number) or Program (to control by application). For most server rules, choose Port.
  4. 4
    Protocol and Ports: Select TCP or UDP and enter the specific port number(s), e.g., 443 or 80, 443.
  5. 5
    Action: Choose Allow the connection (or "Allow the connection if it is secure" for IPsec-protected connections).
  6. 6
    Profile: Select which profiles this rule applies to — Domain, Private, and/or Public.
  7. 7
    Name: Give the rule a descriptive name (e.g., Allow HTTPS Inbound). Click Finish.

PowerShell: Managing Firewall Rules

# View all inbound rules Get-NetFirewallRule -Direction Inbound | Select-Object Name, Enabled, Action # Allow inbound HTTPS (TCP 443) on all profiles New-NetFirewallRule -DisplayName "Allow HTTPS Inbound" ` -Direction Inbound -Protocol TCP -LocalPort 443 ` -Action Allow -Profile Any # Block inbound Telnet (TCP 23) New-NetFirewallRule -DisplayName "Block Telnet Inbound" ` -Direction Inbound -Protocol TCP -LocalPort 23 ` -Action Block -Profile Any # Enable an existing rule by name Enable-NetFirewallRule -DisplayName "Remote Desktop - User Mode (TCP-In)" # Check firewall profile status Get-NetFirewallProfile | Select-Object Name, Enabled, DefaultInboundAction
📌 Defense in Depth in Action
Even though pfSense will block SSH from the Clients segment to your Windows Server, you should also ensure Windows Defender Firewall is configured to block unnecessary inbound connections. If pfSense is misconfigured or bypassed, the host-based firewall provides a second layer of protection.

🔥 7 · pfSense 2.7.2 Overview

pfSense Community Edition (CE) 2.7.2 is a free, open-source firewall and router distribution based on FreeBSD. It is one of the most widely deployed open-source firewalls in the world, used in homes, small businesses, enterprises, and labs. In this course it is your virtual firewall appliance for Project 2.

Key Features

VPN Support OpenVPN and IPsec built in Beyond scope of Project 2; awareness only

pfSense Default Behaviors (Important for Project 2)

  • LAN interface: pfSense creates a default allow all rule on the LAN interface. In your topology, LAN = Management (10.0.1.0/24). Management hosts have full outbound access by default.
  • WAN interface: All unsolicited inbound traffic from WAN is blocked by default (implicit deny). Only established reply traffic is allowed.
  • OPT interfaces (OPT1, OPT2): No rules are created by default. All traffic on OPT interfaces is blocked until you explicitly add allow rules. This is why your Servers and Clients VMs will have no connectivity after initial pfSense setup — you must add rules.
  • NAT (Outbound): Automatic outbound NAT is enabled by default, so all internal interfaces can reach the internet through the WAN IP without additional configuration.
⚠️ OPT Interfaces Are Blocked by Default
A very common Project 2 issue: after assigning and enabling OPT1 (SERVERS) and OPT2 (CLIENTS), students find those VMs have no connectivity. This is expected — you must add firewall rules on those interfaces before any traffic is allowed. The LAN (Management) interface is the only one with a default allow rule.

⚙️ 8 · Deploying pfSense 2.7.2 in Proxmox

This section walks through deploying pfSense 2.7.2 as a virtual machine in Proxmox, from ISO upload through interface assignment. Follow these steps carefully — getting the network interface order correct is critical.

Step 1A: Create Virtual Bridges in Proxmox

Before creating the pfSense VM, you need three additional Linux bridges in Proxmox (vmbr0 already exists for WAN/management).

  1. 1
    Log in to the Proxmox web UI. In the left panel, click your node name (e.g., pve).
  2. 2
    Navigate to System → Network. Click Create → Linux Bridge.
  3. 3
    Create vmbr1:
    • Name: vmbr1
    • IPv4/CIDR: leave blank (pfSense will be the gateway)
    • Comment: Management Network 10.0.1.0/24
    • Click Create
  4. 4
    Repeat to create vmbr2 (comment: Servers Network 10.0.2.0/24) and vmbr3 (comment: Clients Network 10.0.3.0/24).

📌 Why Leave Bridge IPs Blank?
Proxmox bridge IPs would place the Proxmox hypervisor itself on those subnets, creating routing conflicts. Since pfSense will be the gateway (10.0.1.1, 10.0.2.1, 10.0.3.1) on each bridge, the Proxmox host should have no IP on these bridges. The bridges function as pure Layer 2 switches connecting VMs.

🗂️ 10 · Network Segmentation Design

With pfSense configured, you have a three-segment internal network. Understanding why the segments are designed the way they are helps you write correct firewall rules and explain your design in Project 2's documentation.

The Three Internal Segments

Management Network — 10.0.1.0/24 (vmbr1)

  • Purpose: Administrative access only. This is where IT administrators connect to manage all other systems, including the pfSense web UI.
  • Who belongs here: Admin workstations, jump hosts, monitoring systems. In the lab: your own machine accessing pfSense via the Proxmox console.
  • IP addressing: Static only — no DHCP. pfSense LAN IP is 10.0.1.1/24.
  • Default firewall behavior: pfSense's default LAN allow-all rule gives Management full outbound access.
  • Security rationale: Servers and Clients cannot initiate connections to Management. If a client or server is compromised, the attacker cannot reach the management plane to reconfigure the firewall or access admin tools.

Servers Network — 10.0.2.0/24 (vmbr2)

  • Purpose: Production servers providing services to the organization — DNS, file sharing, directory services, web applications.
  • Who belongs here: Linux Server (10.0.2.10), Windows Server (10.0.2.11).
  • IP addressing: Static only — servers must have predictable, fixed IPs so clients can reliably reach them by IP (and by DNS name once DNS is configured).
  • Default firewall behavior: OPT1 has no default rules — all traffic blocked until you add rules.
  • Security rationale: Servers can reach the internet for updates (HTTP/HTTPS, DNS) but are blocked from initiating connections to Management. Clients can reach servers only on specific allowed ports — not unrestricted access.

Clients Network — 10.0.3.0/24 (vmbr3)

  • Purpose: End-user workstations and test VMs. The highest-risk zone because end users interact with external content and are most likely to be targeted by phishing or malware.
  • Who belongs here: Linux Client VM (test VM for Project 2 verification).
  • IP addressing: Dynamic via DHCP (10.0.3.100–10.0.3.200). pfSense DHCP server on CLIENTS interface.
  • Default firewall behavior: OPT2 has no default rules — all traffic blocked until you add rules.
  • Security rationale: Clients can reach servers on specific ports (DNS, web, file sharing, RDP) and can browse the internet. They are blocked from Management entirely and blocked from SSH to servers. This limits the damage a compromised client can do.

🚨 11 · Mini-Case: Locked Out After a Rule Change

🚨 Scenario: Friday Afternoon at TechStart Solutions

It is 4:45 PM on a Friday. Alex, the junior IT admin at TechStart Solutions, is tightening security on the pfSense firewall before the weekend. Alex wants to block all traffic from the Servers segment to the Management segment — a sensible security goal.

Alex navigates to Firewall → Rules → LAN (the Management interface) and adds the following rule at the top of the LAN ruleset:

ActionSourceDestinationPortDescription
BlockanyLAN addressany"Block all inbound to management"

Alex clicks Save and Apply Changes. The browser immediately stops loading. Alex can no longer reach the pfSense web UI at https://10.0.1.1. The rule change has locked Alex out of the firewall management interface.

It is now 4:52 PM. The senior admin is unreachable. The firewall is running — it is still routing traffic for users — but nobody can access the management UI to fix the rule.

❓ Question 1: What mistake did Alex make? Explain using rule ordering and implicit deny concepts.

The Mistake — Wrong Rule Placement and Overly Broad Criteria:

Alex added a Block any → LAN address any rule at the top of the LAN interface ruleset. Because pfSense uses first-match-wins ordering, this rule is evaluated before all other rules — including the default allow-all rule that was already there.

The rule matches all traffic destined for the pfSense LAN IP (10.0.1.1), including Alex's own browser traffic from the Management workstation. The intended goal was to block traffic from the Servers segment to Management — but the rule source was set to any, which includes Management hosts themselves.

Two errors combined:

  1. The source should have been SERVERS net, not any — the rule blocked the admin's own traffic.
  2. The rule was placed at the top of the LAN ruleset, overriding the existing default allow rule for Management hosts.

❓ Question 2: How can Alex recover access to the pfSense web UI without physical access to the firewall hardware?

Recovery Options:
  1. Proxmox Console (preferred for a VM): Since pfSense is a VM in Proxmox, Alex can open the Proxmox web UI → pfSense VM → Console tab. The pfSense console menu is accessible regardless of firewall rules — it is a direct connection to the VM's virtual serial port, not a network connection.
    • From the console, select option 8 — Shell.
    • Run: pfctl -d to temporarily disable the packet filter entirely, restoring access to the web UI.
    • Log into the web UI, delete or correct the bad rule, then re-enable pf with pfctl -e.
  2. Alternative — Reset to defaults via console: Console option 4 — Reset to factory defaults will restore pfSense to its out-of-box state, clearing all rules. This is a last resort as it removes all configuration.
  3. Alternative — Restore from snapshot: If a Proxmox snapshot was taken before the rule change, roll back the VM to the snapshot. This is why snapshots before major changes are so important.

❓ Question 3: How should the rule have been written to achieve Alex's original security goal without causing a lockout? Write the corrected rule.

Corrected Rule Design:

The goal is to block the Servers segment from reaching Management — not to block Management from reaching itself. The correct approach is to add the block rule on the SERVERS interface (where Servers-originated traffic enters pfSense), not on the LAN interface.

InterfaceActionSourceDestinationPortDescription
SERVERS Block SERVERS net LAN (Management) net any Block servers from reaching management

Key lessons:

  • In pfSense, rules are applied on the ingress interface — the interface where traffic enters pfSense. Traffic from Servers enters on the SERVERS interface, so block rules for Servers-originated traffic belong there.
  • Never use any as the source when you mean a specific subnet. Be as specific as possible.
  • Test rule changes during business hours with a colleague available to recover access if needed.
  • Always take a snapshot before applying firewall rule changes in a production or lab environment.
🔒 CSLO 2 Connection
This scenario illustrates a critical operational security principle: firewall rule changes must be carefully planned, tested, and reversible. Accidental self-lockout is one of the most common firewall administration mistakes. Defense-in-depth (Proxmox console access, snapshots) provides the recovery path when a rule change goes wrong.

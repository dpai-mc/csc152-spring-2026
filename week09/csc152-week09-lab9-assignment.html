🌐 Part A – Proxmox Network Configuration 6 pts

Before deploying pfSense, you must create three additional Linux bridges in Proxmox. Each bridge becomes one internal network segment. The existing vmbr0 provides WAN (upstream/internet) connectivity; the new bridges will carry internal traffic.

Important: Leave bridge IP fields blank
When creating bridges for pfSense-managed subnets, do not assign IP addresses to the Proxmox bridge interfaces. pfSense will serve as the gateway for each subnet. Assigning IPs to the bridges in Proxmox would create routing conflicts.

Task A-1: Create Linux Bridge vmbr1 (Management)

  1. 1
    Log in to the Proxmox web interface and select your node from the left panel.
  2. 2
    Navigate to Node → Network in the left sidebar.
  3. 3
    Click Create → Linux Bridge.
  4. 4
    Set the name to vmbr1. Leave the IP address and gateway fields blank. Set a comment: Management – 10.0.1.0/24.
  5. 5
    Click Create, then click Apply Configuration at the top of the Network panel.

Task A-2: Create Linux Bridge vmbr2 (Servers)

  1. 1
    Repeat the bridge creation process. Set the name to vmbr2.
  2. 2
    Leave IP/gateway blank. Comment: Servers – 10.0.2.0/24.
  3. 3
    Click Create (no need to apply yet — apply once after all bridges are created).

Task A-3: Create Linux Bridge vmbr3 (Clients)

  1. 1
    Create a third bridge named vmbr3.
  2. 2
    Leave IP/gateway blank. Comment: Clients – 10.0.3.0/24.
  3. 3
    Click Create, then click Apply Configuration. Confirm all three new bridges appear in the network list.

Task C-2: Create Static DHCP Mappings

Static mappings reserve a specific IP for a known device based on its MAC address. Create at least two reservations — for example, your test client VM and one simulated server.

  1. 1
    Scroll to the DHCP Static Mappings section on the same CLIENTS DHCP page.
  2. 2
    Click Add. Enter the MAC address, IP address (must be outside the dynamic pool, e.g., 10.0.3.10 or 10.0.3.11), and a hostname/description.
  3. 3
    Click Save, then repeat for the second reservation.
Finding a VM's MAC address
In Proxmox, select the VM → Hardware → Network Device. The MAC address is listed there. You can also get it from the guest OS with ip link (Linux) or ipconfig /all (Windows).

  3. 3

  4. 4
    In pfSense, navigate to Status → DHCP Leases to confirm the lease appears there as well.

🛡️ Part D – Firewall Rules Implementation 8 pts

pfSense processes firewall rules per interface, in order from top to bottom — the first matching rule wins. OPT interfaces (SERVERS and CLIENTS) have no default rules and block all traffic until you create rules explicitly. The LAN (Management) interface has a default "allow all" rule.

Rule order matters
Place more specific rules above more general rules. A broad "block all" rule at the top will prevent more specific "allow" rules below it from ever matching.

Task D-1: Review Default Rules

  1. 1
    Navigate to Firewall → Rules.
  2. 2
    Click the LAN tab. Note the pre-existing "Default allow LAN to any" rule. This allows full access from the Management segment.
  3. 3
    Click the SERVERS and CLIENTS tabs. Confirm both are empty (no rules = all traffic blocked).

Task D-2: Create SERVERS Network Rules

Navigate to Firewall → Rules → SERVERS and add the following rules in order. For each, click Add (with the down-arrow to add below), fill in the rule details, and click Save. After all rules are added, click Apply Changes.

#ActionProtocolSourceDestinationPortDescription
1PassTCPSERVERS netWAN net80, 443Allow web access from servers
2PassUDPSERVERS netWAN net53Allow DNS queries outbound
3BlockanySERVERS netLAN netanyBlock access to Management segment
4PassanySERVERS netSERVERS netanyAllow inter-server communication

Task D-3: Create CLIENTS Network Rules

Navigate to Firewall → Rules → CLIENTS and add the following rules.

#ActionProtocolSourceDestinationPortDescription
1PassTCPCLIENTS netSERVERS net80, 443, 445, 3389Allow client access to server services
2PassTCPCLIENTS netWAN net80, 443Allow web browsing
3PassUDPCLIENTS netWAN net53Allow DNS queries
4BlockanyCLIENTS netLAN netanyBlock management access from clients
5BlockTCPCLIENTS netSERVERS net22Block SSH to servers from clients
Adding rules in pfSense
When adding multiple rules, use the Add ↓ button (add below current) to build the list top-to-bottom. After adding all rules for an interface, click Apply Changes once to activate them all.

Task D-4: Enable Logging on Block Rules

  1. 1
    Edit each Block rule (click the pencil icon).
  2. 2
    Scroll to the Extra Options section and check Log packets that are handled by this rule.
  3. 3
    Save each rule. Apply changes when all block rules are updated.

Task D-5: Test and Verify Rules

  1. 1
    From your CLIENTS test VM, test an allowed connection: try to reach port 80 on a Servers-network VM or the WAN. Confirm it succeeds.
  2. 2
    Test a blocked connection: attempt to SSH (port 22) from the client to a server VM, or ping the Management segment IP (10.0.1.x). Confirm it times out or is refused.
  3. 3
    Check the firewall log: navigate to Status → System Logs → Firewall. You should see log entries for the blocked connection attempts.
  4. 4
    Note 2–3 example log entries. Each entry shows: timestamp, interface, source IP:port, destination IP:port, protocol, and action.

Include a justification for every rule — this is evaluated in the lab report.

At least 2 entries with explanations are required in the lab report.

💡 Hints & Common Pitfalls

Take snapshots before major changes
In Proxmox, right-click a VM → Snapshots → Take Snapshot. Name it descriptively (e.g., post-install, pre-firewall-rules). If a configuration change breaks something, you can roll back immediately without reinstalling.
Test after each step — not all at once
After completing each task (bridge creation, interface assignment, DHCP scope), verify it works before moving on. Testing incrementally makes it much easier to identify exactly where something went wrong.
pfSense firewall rules: first match wins
Rules are evaluated top-to-bottom; the first rule that matches the traffic wins. Place specific allow rules above broad block rules. If you add a "block all" rule at the top, nothing below it will ever match.
OPT interfaces are blocked by default
Unlike LAN, pfSense OPT interfaces (SERVERS and CLIENTS) have no default allow rule. All traffic is blocked until you explicitly create pass rules. If a VM on an OPT interface cannot reach anything, check whether you have added the necessary allow rules and clicked Apply Changes.
pfSense documentation is comprehensive
The Netgate documentation site (docs.netgate.com) covers every feature in detail. The pfSense community forums are also highly active. Search for your specific question before spending too long troubleshooting on your own.
Apply Changes after every rule edit
pfSense does not activate new or edited rules until you click Apply Changes at the top of the Firewall Rules page. A common mistake is adding rules and then wondering why traffic behavior has not changed — check whether you applied the changes.

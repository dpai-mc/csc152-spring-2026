2.1 The Filesystem Hierarchy Standard (FHS)

Definition: The Filesystem Hierarchy Standard (FHS) defines the directory structure and directory contents in Unix-like operating systems, ensuring consistency across different Linux distributions.

Purpose of FHS:

  • Predictability: Administrators know where to find files across different systems
  • Consistency: Scripts and tools work across distributions
  • Organization: Logical separation of system files, user data, and configuration
  • Maintainability: Easier to backup, secure, and troubleshoot systems
Think of FHS like the Dewey Decimal System in libraries. Just as you know fiction is in the 800s and science is in the 500s, Linux administrators know configuration files are in /etc and logs are in /var/log, regardless of which library (distribution) they're in.

Root Directory Structure:

Everything in Linux starts at the root directory, represented by a single forward slash (/). All other directories branch from here, creating a tree-like structure.

2.2 Essential Directories Explained

Click on each directory to expand and learn more about its purpose, contents, and real-world usage:

/bin - Essential User Binaries

Contains essential command-line utilities needed for system boot and repair
Examples: ls, cp, mv, cat, bash, grep

Why Essential: These commands must be available even if /usr isn't mounted

Typical Size: 10-20 MB

# These commands are in /bin and work even in single-user mode ls /bin/ls # Lists itself cp /bin/ls /tmp/ # Copy works because cp is in /bin

/sbin - System Binaries (Administrator)

System administration commands requiring root privileges
Examples: fdisk, ifconfig, reboot, shutdown, iptables

Access: Typically requires sudo or root user

Typical Size: 15-30 MB

# Regular users can't run these directly ifconfig # Shows network interfaces sudo reboot # Restart system (requires root)

/etc - Configuration Files

System-wide configuration files and scripts
Examples: /etc/passwd, /etc/fstab, /etc/ssh/sshd_config

Characteristics: Text files, human-readable, no binaries

Typical Size: 50-200 MB

# View user accounts cat /etc/passwd # Check SSH configuration sudo less /etc/ssh/sshd_config # Most config files are plain text file /etc/fstab # Output: ASCII text

/home - User Home Directories

Personal directories for regular users
Structure: /home/username/

Contents: User files, documents, personal configurations

Permissions: Each user owns their home directory

Typical Size: Varies (often the largest directory)

# Each user has their own space /home/alice/ # Alice's files /home/bob/ # Bob's files # Users can't access each other's homes by default ls /home/alice/ # Works if you're alice ls /home/bob/ # Permission denied if you're alice

/root - Root User's Home

Home directory for the root (administrator) user
Isolated from /home for security and accessibility

Why Separate: Root's home is under / to be accessible even if /home isn't mounted

Security: Isolated from regular user directories

Typical Size: Usually small (10-100 MB)

/var - Variable Data

Data that changes frequently during system operation
Key subdirectories: /var/log/, /var/www/, /var/mail/

Key Subdirectories:

  • /var/log/ - System and application logs
  • /var/www/ - Web server content
  • /var/mail/ - User mailboxes
  • /var/tmp/ - Temporary files preserved between reboots
  • /var/lib/ - Application state data

Typical Size: Grows over time (can be very large)

# View system logs sudo tail -f /var/log/syslog # Ubuntu sudo tail -f /var/log/messages # Rocky Linux # Check web server content ls /var/www/html/ # Logs grow over time - monitor space du -sh /var/log/

/tmp - Temporary Files

Temporary files created by applications
Cleared on reboot, world-writable

Characteristics:

  • World-writable (any user can write)
  • Cleared on reboot (usually)
  • Not for important data

Typical Size: 100MB-1GB

# Create temporary file echo "test data" > /tmp/mytest.txt # After reboot, file is gone reboot ls /tmp/mytest.txt # File not found

/usr - User Programs

Read-only user data and programs
Subdirectories: /usr/bin/, /usr/sbin/, /usr/local/

Key Subdirectories:

  • /usr/bin/ - User command binaries
  • /usr/sbin/ - System administration binaries
  • /usr/local/ - Locally installed software
  • /usr/share/ - Shared data (docs, icons)

Typical Size: 2-10 GB

/boot - Boot Loader Files

Files needed to boot the system
Kernel, initial ramdisk (initramfs), bootloader config

Contents: Kernel, initial ramdisk (initramfs), bootloader config

Critical: Damage here can make system unbootable. Always be careful when working in /boot!

Typical Size: 200-500 MB

/dev - Device Files

Device files representing hardware
/dev/sda (hard disk), /dev/null (null device), /dev/random

Examples:

  • /dev/sda - First hard disk
  • /dev/sda1 - First partition on first disk
  • /dev/null - Null device (discards all data)
  • /dev/random - Random number generator

Special: These aren't real files; they're interfaces to kernel drivers

# List block devices ls -l /dev/sd* # Send output to nowhere echo "This disappears" > /dev/null # Read random data head -c 10 /dev/random | base64

/proc - Process Information (Virtual)

Virtual filesystem providing process and system information
/proc/cpuinfo, /proc/meminfo, /proc/[PID]/

Characteristics:

  • Not real files on disk
  • Generated by kernel on-the-fly
  • Provides system information
# View CPU information cat /proc/cpuinfo # Check memory usage cat /proc/meminfo # These files don't exist on disk ls -lh /proc/cpuinfo # Shows size as 0

2.3 The "Everything is a File" Philosophy

Concept: In Linux, almost everything is represented as a file, including:

  • Regular files (documents, images)
  • Directories (special files listing other files)
  • Device drivers (hardware interfaces)
  • Sockets (network connections)
  • Pipes (inter-process communication)

Why This Matters:

  • Uniform Interface: Same commands work on different types of "files"
  • Simplicity: One set of tools (read, write, open, close) works for everything
  • Flexibility: Can treat hardware like files, making scripting easier
# Reading from a regular file cat document.txt # Reading from a device file cat /dev/sda1 # Reading from a virtual file cat /proc/cpuinfo # All use the same 'cat' command!

2.4 Absolute vs. Relative Paths

Absolute Path

  • Starts from root directory (/)
  • Complete path from root to target
  • Works from anywhere in the filesystem
  • Example: /home/alice/documents/report.txt

Relative Path

  • Starts from current directory
  • Depends on where you are
  • Uses . (current) and .. (parent)
  • Example: documents/report.txt

Navigation Examples:

# Current location: /home/alice/ # Absolute path - works from anywhere cat /home/alice/documents/report.txt # Relative path - depends on current location cat documents/report.txt # Same file # Go up one directory cd .. # Now in /home/ # Go up two directories cd ../.. # Now in / # Go to home directory cd ~ # Back to /home/alice/ # Go to previous directory cd - # Back to /

üìù Module Summary ‚ñº

Key Takeaways

  • The FHS provides consistency across all Linux distributions
  • /etc contains system configuration files (text-based, human-readable)
  • /var holds variable data like logs, databases, and web content
  • /home contains user personal files and directories
  • /root is the administrator's home directory (separate from /home)
  • Absolute paths start with / and work from anywhere
  • Relative paths depend on your current location
  • "Everything is a file" - devices, processes, and directories are all treated as files
  • /proc and /dev are virtual filesystems (not real files on disk)
  • /tmp is cleared on reboot - never store important data there

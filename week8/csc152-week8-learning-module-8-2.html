§1 — Linux Network Interface Naming Conventions

Before you can configure a network interface, you need to know its name. Linux has used two different naming schemes over the years, and you will encounter both in documentation and in the field.

1.1 Traditional Naming (eth0, eth1, …)

Older Linux kernels assigned interface names sequentially based on detection order: eth0 for the first Ethernet NIC, eth1 for the second, and so on. This scheme is simple but has a critical flaw: if hardware changes or the kernel detects devices in a different order after a reboot, interface names can swap unpredictably.

1.2 Predictable Interface Names

Modern Linux distributions (including Ubuntu Server 20.04+ and Rocky Linux 8+) use systemd predictable network interface names. These names encode information about the hardware location, making them stable across reboots.

Prefix Meaning Example
en Ethernet ens18, enp0s3
wl Wireless LAN wlp2s0
lo Loopback lo

The characters after the prefix indicate the bus type and slot number. For example:

  • ens18 — Ethernet, hotplug slot 18 (common on Proxmox/KVM virtual machines)
  • enp0s3 — Ethernet, PCI bus 0, slot 3 (common on VirtualBox)
💡 Lab Tip In our Proxmox lab environment, your Linux VMs will most likely show ens18 as the primary interface. Always verify the actual name on your system before editing configuration files — never assume it is eth0.

1.3 Windows Interface Names

Windows Server uses friendly names like "Ethernet", "Ethernet 2", or names assigned by the administrator. You can view and rename interfaces in Network and Sharing Center → Change adapter settings or with PowerShell:

Get-NetAdapter

This cmdlet lists all network adapters with their names, status, and link speed.

§2 — Viewing Current Interface Configuration

Before making changes, always inspect the current state. This is your "before" snapshot — essential for troubleshooting if something goes wrong.

🐧 Linux

Show IP addresses on all interfaces:

ip addr show

Show link status (up/down, MAC address):

ip link show

Show only a specific interface:

ip addr show ens18

Key fields in the output:

  • state UP / state DOWN — link status
  • inet 10.0.1.10/24 — IPv4 address with CIDR prefix
  • link/ether — MAC address
🪟 Windows Server

Show all interface details:

ipconfig /all

PowerShell — list adapters:

Get-NetAdapter

PowerShell — show IP configuration:

Get-NetIPAddress -InterfaceAlias "Ethernet"

Key fields in ipconfig /all output:

  • IPv4 Address — current IP
  • Subnet Mask — e.g., 255.255.255.0
  • Default Gateway — router address
  • DHCP Enabled — Yes or No
⚠️ Deprecated Command You may see ifconfig in older documentation. On modern Linux distributions, ifconfig is deprecated and often not installed by default. Always use the ip command suite instead (ip addr, ip link, ip route). 
2: ens18: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP group default qlen 1000
    link/ether bc:24:11:ab:cd:ef brd ff:ff:ff:ff:ff:ff
    inet 10.0.1.10/24 brd 10.0.1.255 scope global ens18
       valid_lft forever preferred_lft forever
    inet6 fe80::be24:11ff:feab:cdef/64 scope link
       valid_lft forever preferred_lft forever

Breaking this down:

  • ens18 — interface name (Ethernet, slot 18)
  • state UP — the link is active
  • link/ether bc:24:11:ab:cd:ef — MAC address
  • inet 10.0.1.10/24 — IPv4 address is 10.0.1.10 with a /24 subnet mask (255.255.255.0)
  • brd 10.0.1.255 — broadcast address for this subnet
  • inet6 fe80::… — auto-configured IPv6 link-local address (recall from Module 8.1)

§3 — Configuring Static IP Addresses on Linux

Linux offers multiple tools for network configuration depending on the distribution and the network management stack in use. We will cover the two most common approaches you will encounter in this course.

3.1 Netplan (Ubuntu Server)

Netplan is the default network configuration tool on Ubuntu Server 18.04 and later. It uses YAML files stored in /etc/netplan/ and renders the configuration to either systemd-networkd or NetworkManager as a backend.

Step-by-Step: Assign a Static IP with Netplan

Step 1: Identify the Netplan configuration file.

ls /etc/netplan/

You will typically see a file like 00-installer-config.yaml or 50-cloud-init.yaml.

Step 2: Open the file with elevated privileges.

sudo nano /etc/netplan/00-installer-config.yaml
⚠️ File Permissions Reminder Netplan YAML files in /etc/netplan/ are owned by root and require sudo to edit. This directly applies the privilege escalation concepts from Week 6.

Step 3: Edit the file to set a static IP. Here is an example configuration:

network:
  version: 2
  ethernets:
    ens18:
      addresses:
        - 10.0.1.10/24
      routes:
        - to: default
          via: 10.0.1.1
      nameservers:
        addresses:
          - 10.0.1.1
          - 8.8.8.8
💡 YAML Syntax Matters YAML is whitespace-sensitive. Use spaces only (no tabs) and maintain consistent indentation (2 spaces per level is standard). A single tab character or misaligned key will cause Netplan to reject the file.

Step 4: Apply the configuration.

sudo netplan apply

Step 5: Verify the new configuration.

ip addr show ens18
Key Purpose Example Value
addresses Static IP(s) in CIDR notation - 10.0.1.10/24
routes Routing entries (default gateway) - to: default / via: 10.0.1.1
nameservers DNS server addresses addresses: [10.0.1.1, 8.8.8.8]
dhcp4 Enable/disable DHCPv4 true or false
dhcp6 Enable/disable DHCPv6 true or false

3.2 nmcli (Rocky Linux / NetworkManager-based Systems)

nmcli (NetworkManager Command-Line Interface) is the primary configuration tool on distributions that use NetworkManager, including Rocky Linux, AlmaLinux, and Fedora Server.

Step-by-Step: Assign a Static IP with nmcli

Step 1: Identify the connection name.

nmcli connection show

Note the NAME column — it might be something like "Wired connection 1" or ens18.

Step 2: Set the IP address, gateway, and DNS.

sudo nmcli connection modify "ens18" \
  ipv4.addresses 10.0.1.10/24 \
  ipv4.gateway 10.0.1.1 \
  ipv4.dns "10.0.1.1 8.8.8.8" \
  ipv4.method manual

Step 3: Activate the changes.

sudo nmcli connection up "ens18"

Step 4: Verify.

ip addr show ens18
ℹ️ Netplan vs. nmcli Both tools accomplish the same goal — persistent static IP configuration. The choice depends on your distribution:
  • Ubuntu Server → Netplan (renders to systemd-networkd)
  • Rocky Linux / RHEL → nmcli (NetworkManager)
In our course labs, you will primarily use Netplan on Ubuntu Server VMs. Knowing nmcli prepares you for RHEL-family systems you will encounter in the workplace.

§4 — Configuring Static IP Addresses on Windows Server

Windows Server provides both a graphical interface and PowerShell for network configuration. As a server administrator, you should be comfortable with both approaches.

4.1 GUI Method (Network and Sharing Center)

  1. Open Server Manager → Local Server and click the link next to Ethernet (or open Control Panel → Network and Sharing Center).
  2. Click Change adapter settings in the left pane.
  3. Right-click the target adapter (e.g., Ethernet) → Properties.
  4. Select Internet Protocol Version 4 (TCP/IPv4) → click Properties.
  5. Select "Use the following IP address" and enter:
    • IP address: 10.0.1.20
    • Subnet mask: 255.255.255.0
    • Default gateway: 10.0.1.1
  6. Under "Use the following DNS server addresses", enter:
    • Preferred DNS server: 10.0.1.1
    • Alternate DNS server: 8.8.8.8
  7. Click OK → Close.
💡 When to Use the GUI The GUI is useful for one-time configuration on a single server. For configuring multiple servers or for repeatable, scriptable deployments, PowerShell is strongly preferred — especially in Project 2 where you will document your configuration steps.

4.2 PowerShell Method

PowerShell provides cmdlets that accomplish the same configuration programmatically. Run PowerShell as Administrator.

Step 1: Identify the interface index.

Get-NetAdapter

Note the ifIndex value for your target adapter (e.g., 6).

Step 2: Remove any existing IP configuration (if needed).

Remove-NetIPAddress -InterfaceIndex 6 -Confirm:$false
Remove-NetRoute -InterfaceIndex 6 -Confirm:$false

Step 3: Assign the static IP address and default gateway.

New-NetIPAddress `
  -InterfaceIndex 6 `
  -IPAddress 10.0.1.20 `
  -PrefixLength 24 `
  -DefaultGateway 10.0.1.1

Step 4: Set the DNS client server addresses.

Set-DnsClientServerAddress `
  -InterfaceIndex 6 `
  -ServerAddresses ("10.0.1.1","8.8.8.8")

Step 5: Verify the configuration.

ipconfig /all

Or in PowerShell:

Get-NetIPConfiguration -InterfaceIndex 6
Cmdlet Key Parameters Purpose
New-NetIPAddress -InterfaceIndex, -IPAddress, -PrefixLength, -DefaultGateway Assigns a static IP and gateway
Set-DnsClientServerAddress -InterfaceIndex, -ServerAddresses Sets preferred and alternate DNS servers
Get-NetIPConfiguration -InterfaceIndex Displays IP, gateway, and DNS for an interface
Get-NetAdapter (none required) Lists all adapters with name, status, index

Note: Use -PrefixLength 24 instead of a subnet mask. This is the CIDR prefix length from Module 8.1 — PowerShell uses it natively.

§5 — Configuring DHCP Client Mode

Not every system needs a static IP. Client workstations and some non-critical servers use DHCP (Dynamic Host Configuration Protocol) to obtain an IP address, subnet mask, default gateway, and DNS servers automatically from a DHCP server. Recall from Module 8.1 that DHCP uses a four-step process: Discover → Offer → Request → Acknowledge (DORA).

ℹ️ When to Use Static vs. DHCP
  • Static IP — Servers, routers, printers, and any device that other systems need to find at a consistent address.
  • DHCP — Client workstations, laptops, and devices that do not host services accessed by name or IP.
In Project 2, your server VMs will use static IPs, while the client subnet will be served by a DHCP server (configured in Week 9).
🐧 Linux — DHCP Client

Netplan (Ubuntu Server):

network:
  version: 2
  ethernets:
    ens18:
      dhcp4: true

Then apply:

sudo netplan apply

nmcli (Rocky Linux):

sudo nmcli connection modify "ens18" \
  ipv4.method auto
sudo nmcli connection up "ens18"

Setting ipv4.method auto tells NetworkManager to use DHCP.

🪟 Windows Server — DHCP Client

GUI: In the IPv4 Properties dialog, select "Obtain an IP address automatically" and "Obtain DNS server address automatically".

PowerShell:

Set-NetIPInterface `
  -InterfaceIndex 6 `
  -Dhcp Enabled

To also reset DNS to DHCP-assigned:

Set-DnsClientServerAddress `
  -InterfaceIndex 6 `
  -ResetServerAddresses

Verify with:

ipconfig /all

Look for "DHCP Enabled: Yes" in the output.

If you switch from static to DHCP (or need to refresh your lease), use these commands:

🐧 Linux
# Release current lease
sudo dhclient -r ens18

# Request a new lease
sudo dhclient ens18

On systems using systemd-networkd, restarting the service or running sudo netplan apply triggers a new DHCP request.

🪟 Windows
ipconfig /release
ipconfig /renew

These commands release and renew the DHCP lease on all DHCP-enabled interfaces.

§6 — Default Gateway Configuration

The default gateway is the IP address of the router interface on your local subnet. When a system needs to communicate with a host outside its own subnet, it forwards the packet to the default gateway, which then routes it toward the destination.

6.1 How the Default Gateway Works

When your system sends a packet, it performs this decision:

  1. Is the destination on my local subnet? (Compare destination IP against my IP and subnet mask.)
  2. If yes → send directly to the destination via the local network (ARP to find the MAC address).
  3. If no → send to the default gateway, which forwards the packet onward.

Without a correctly configured default gateway, a system can communicate with other hosts on the same subnet but cannot reach any remote network, including the internet.

6.2 Verifying the Default Gateway

🐧 Linux
ip route show

Look for a line like:

default via 10.0.1.1 dev ens18

This means: "For any destination not in my local routing table, send to 10.0.1.1 via interface ens18."

🪟 Windows
route print

Or simply check the Default Gateway line in:

ipconfig

6.3 Common Gateway Mistakes

  • Gateway on wrong subnet: The gateway IP must be on the same subnet as the interface. If your IP is 10.0.1.10/24, the gateway must be in the 10.0.1.0/24 range (e.g., 10.0.1.1).
  • Gateway omitted: The system can reach local hosts but nothing beyond the subnet.
  • Wrong gateway IP: Packets are sent to a non-existent or incorrect router — connectivity to remote networks fails.
🔗 Project 2 Connection In Project 2, your pfSense firewall will act as the default gateway for each of the three internal subnets (Management, Servers, Clients). Each server VM must have its default gateway set to the pfSense interface address on its respective subnet. Getting this wrong is one of the most common Project 2 troubleshooting issues.

§7 — Persistent vs. Runtime Configuration

A critical concept for any server administrator: not all configuration changes survive a reboot. Understanding the difference between persistent and runtime changes prevents frustrating troubleshooting sessions where "it worked until I rebooted."

7.1 Runtime (Non-Persistent) Changes

Commands that modify the running configuration immediately but do not write to disk:

# Linux — add an IP address at runtime only
sudo ip addr add 10.0.1.50/24 dev ens18

# Linux — add a default route at runtime only
sudo ip route add default via 10.0.1.1

These changes take effect instantly but are lost on reboot (or when the network service restarts). They are useful for:

  • Quick testing before committing a change
  • Temporary troubleshooting
  • Verifying connectivity before editing configuration files

7.2 Persistent (On-Disk) Changes

Persistent changes are written to configuration files and survive reboots:

Platform Persistent Method Configuration Location
Ubuntu Server Netplan YAML + sudo netplan apply /etc/netplan/*.yaml
Rocky Linux nmcli connection modify + nmcli connection up /etc/NetworkManager/system-connections/
Windows Server GUI (adapter properties) or PowerShell (New-NetIPAddress) Windows Registry / WMI store

7.3 Best Practice Workflow

💡 Recommended Approach
  1. Test at runtime first: Use ip addr add to test a new IP. Verify connectivity with ping.
  2. If it works, make it persistent: Edit the Netplan YAML (or use nmcli modify) and apply.
  3. If it doesn't work: Reboot (or sudo ip addr del) to revert — no configuration files were changed.
This two-step approach is especially valuable in production environments where a misconfiguration could lock you out of a remote server.
⚠️ Conflicting Configurations If you add an IP at runtime with ip addr add and have a different IP in Netplan, running sudo netplan apply will overwrite the runtime change. Always ensure your persistent configuration matches your intended state.

§8 — Guided Practice Activity

ℹ️ About This Activity This guided practice is ungraded and designed to reinforce the skills covered in this module. It mirrors the in-class Zoom lab activity. If you completed these steps during the live session, use this section as a reference and review. If you were unable to attend, complete these steps asynchronously in your Proxmox lab environment.

Part A: Linux Static IP Configuration (Netplan)

Goal: Configure your Ubuntu Server VM with a static IP address and verify connectivity.

  1. Log in to your Ubuntu Server VM via the Proxmox console.
  2. Check your current configuration: 
    ip addr show ens18
ip route show

    Record the current IP, subnet mask, and gateway.

  3. Open the Netplan configuration file: 
    sudo nano /etc/netplan/00-installer-config.yaml
  4. Replace the contents with a static configuration appropriate for your lab subnet. Use the addressing scheme provided by your instructor. Example: 
    network:
  version: 2
  ethernets:
    ens18:
      addresses:
        - 10.0.1.10/24
      routes:
        - to: default
          via: 10.0.1.1
      nameservers:
        addresses:
          - 10.0.1.1
          - 8.8.8.8
  5. Apply the configuration: 
    sudo netplan apply
  6. Verify: 
    ip addr show ens18
ip route show
ping -c 4 10.0.1.1

    You should see your new IP address, the correct default route, and successful ping replies from the gateway.

Part B: Windows Server Static IP Configuration (PowerShell)

Goal: Configure your Windows Server VM with a static IP address using PowerShell and verify connectivity.

  1. Open PowerShell as Administrator on your Windows Server VM.
  2. Identify your adapter: 
    Get-NetAdapter

    Note the ifIndex value.

  3. Check current configuration: 
    Get-NetIPConfiguration
  4. Assign a static IP (adjust the index and addresses for your lab): 
    New-NetIPAddress `
  -InterfaceIndex 6 `
  -IPAddress 10.0.1.20 `
  -PrefixLength 24 `
  -DefaultGateway 10.0.1.1
  5. Set DNS servers: 
    Set-DnsClientServerAddress `
  -InterfaceIndex 6 `
  -ServerAddresses ("10.0.1.1","8.8.8.8")
  6. Verify: 
    ipconfig /all
ping 10.0.1.1

    Confirm the static IP, subnet mask, gateway, and DNS are correct, and that ping succeeds.

Part C: Connectivity Verification

After configuring both VMs, verify cross-platform connectivity:

  1. From the Linux VM, ping the Windows Server VM: 
    ping -c 4 10.0.1.20
  2. From the Windows Server VM, ping the Linux VM: 
    ping 10.0.1.10
💡 Troubleshooting Tip If pings fail between VMs, check these common issues in order:
  1. Are both VMs on the same virtual network/bridge in Proxmox?
  2. Are the IPs on the same subnet? (Both should be in 10.0.1.0/24 in this example.)
  3. Is Windows Firewall blocking ICMP? By default, Windows Server may block incoming ping requests. You can temporarily allow them: 
    New-NetFirewallRule -DisplayName "Allow ICMPv4" `
  -Protocol ICMPv4 -IcmpType 8 `
  -Action Allow

§9 — Module Summary & What's Next

Key Takeaways

  • Modern Linux uses predictable interface names (e.g., ens18) instead of the legacy eth0 scheme.
  • Always inspect current configuration (ip addr show / ipconfig /all) before making changes.
  • On Ubuntu Server, use Netplan YAML files + sudo netplan apply for persistent static IP configuration.
  • On Rocky Linux, use nmcli for persistent configuration via NetworkManager.
  • On Windows Server, use New-NetIPAddress and Set-DnsClientServerAddress in PowerShell (or the GUI for one-off changes).
  • The default gateway must be on the same subnet as the interface — without it, the system cannot reach remote networks.
  • Runtime changes (e.g., ip addr add) are immediate but lost on reboot; persistent changes are written to configuration files.
  • Use the test-then-persist workflow: verify at runtime first, then commit to configuration files.

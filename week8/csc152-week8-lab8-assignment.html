Lab Overview & Learning Objectives

🎯 CSLO 3

Configure systems for network connectivity, including physical and logical elements.

This lab directly assesses your ability to configure and verify network settings on both Linux and Windows Server systems — a core competency for CSLO 3 and a prerequisite for Project 2.

Lab Description

In this extra credit lab, you will perform hands-on network configuration and diagnostic tasks in your Proxmox lab environment. You will work with both your Linux VM (Ubuntu Server or Rocky Linux) and your Windows Server VM to document current network settings, configure static IP addresses, test connectivity using standard diagnostic tools, and troubleshoot common network misconfigurations.

⚠ Extra Credit Lab

Lab 8 is an optional extra credit assignment worth 30 points. It is strongly recommended because the hands-on skills practiced here directly transfer to Project 2 Parts A and D. Students who complete Lab 8 will have a significant head start on Project 2.

Learning Objectives

By completing this lab, you will be able to:

  1. View and document current IP address, subnet mask, default gateway, and DNS configuration on both Linux and Windows
  2. Configure a static IP address on a Linux VM using Netplan or nmcli
  3. Configure a static IP address on a Windows Server VM using both GUI and PowerShell
  4. Verify network connectivity using ping, traceroute/tracert, nslookup/dig, and ip/ipconfig
  5. Diagnose and resolve common network misconfigurations (wrong gateway, incorrect DNS, etc.)
  6. Document network configurations and troubleshooting steps professionally

Prerequisite Knowledge

  • Module 8.1: IP addressing, subnetting, CIDR notation
  • Module 8.2: Static and DHCP interface configuration on Linux and Windows
  • Module 8.3: DNS client configuration and hostname resolution
  • Module 8.4: Network diagnostic tools and structured troubleshooting methodology
  • Weeks 5–7: Command-line proficiency (Bash and PowerShell), sudo usage, file editing

Task 2 — Configure Static IP Addresses

In this task, you will configure static IP addresses on both VMs. This is a critical skill for Project 2, where every server VM must have a predictable, static address on its designated subnet.

⚠ Important: Record Your Original Settings First

Before making any changes, ensure you have documented your current configuration in Task 1. If something goes wrong, you will need those values to restore connectivity. You may also want to take a VM snapshot in Proxmox before proceeding.

Task 2A

Linux VM: Configure a Static IP Address

Choose a static IP address on the same subnet as your current address. Configure it using Netplan (Ubuntu) or nmcli (Rocky Linux).

For Ubuntu Server (Netplan):

Linux Terminal
# 1. Edit the Netplan configuration file
sudo nano /etc/netplan/00-installer-config.yaml

# 2. Example static IP configuration:
network:
  version: 2
  ethernets:
    ens18:
      dhcp4: false
      addresses:
        - 10.0.1.50/24
      routes:
        - to: default
          via: 10.0.1.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4

# 3. Apply the configuration
sudo netplan apply

# 4. Verify
ip addr show ens18
ip route show

For Rocky Linux (nmcli):

Linux Terminal
# 1. Set static IP
sudo nmcli con mod "System ens18" ipv4.method manual \
  ipv4.addresses 10.0.1.50/24 \
  ipv4.gateway 10.0.1.1 \
  ipv4.dns "8.8.8.8 8.8.4.4"

# 2. Restart the connection
sudo nmcli con up "System ens18"

# 3. Verify
ip addr show ens18
ip route show
Task 2B

Windows Server VM: Configure a Static IP Address via PowerShell

Use PowerShell to assign a static IP address to your Windows Server VM. Run the following commands (adjust values for your environment):

PowerShell (Run as Administrator)
# 1. View current adapter information
Get-NetIPConfiguration

# 2. Find your adapter's interface index
Get-NetAdapter

# 3. Remove existing DHCP-assigned address (if applicable)
Remove-NetIPAddress -InterfaceAlias "Ethernet0" -Confirm:$false
Remove-NetRoute -InterfaceAlias "Ethernet0" -Confirm:$false

# 4. Set static IP address
New-NetIPAddress -InterfaceAlias "Ethernet0" `
  -IPAddress 10.0.1.60 `
  -PrefixLength 24 `
  -DefaultGateway 10.0.1.1

# 5. Set DNS server
Set-DnsClientServerAddress -InterfaceAlias "Ethernet0" `
  -ServerAddresses 8.8.8.8,8.8.4.4

# 6. Verify
Get-NetIPConfiguration
ipconfig /all

Task 3 — Connectivity Testing

Now that both VMs have static IP addresses, verify connectivity using standard diagnostic tools. Follow the structured troubleshooting methodology: test from the inside out (local → gateway → remote by IP → remote by name).

Task 3A

Ping Tests (ICMP Connectivity)

Run the following ping tests from your Linux VM and record the results:

Linux Terminal
# 1. Ping your own IP (loopback sanity check)
ping -c 4 127.0.0.1

# 2. Ping your default gateway
ping -c 4 <your-gateway-ip>

# 3. Ping the other VM (Windows Server)
ping -c 4 <windows-server-ip>

# 4. Ping an external IP (Google DNS)
ping -c 4 8.8.8.8

# 5. Ping an external hostname
ping -c 4 google.com
Task 3B

Path Analysis with Traceroute

Run a traceroute from both VMs to an external destination:

Linux Terminal
traceroute 8.8.8.8
PowerShell / CMD (Windows)
tracert 8.8.8.8

Task 4 — DNS Diagnostics

Test and document DNS name resolution using multiple tools. Understanding DNS resolution is essential for Project 2 Part B (DNS server configuration).

Task 4A

DNS Resolution with nslookup and dig

From your Linux VM, run the following commands:

Linux Terminal
# 1. nslookup (basic query)
nslookup www.mendocino.edu

# 2. dig (detailed query)
dig www.mendocino.edu

# 3. Reverse lookup (PTR record)
nslookup 8.8.8.8

# 4. Query a specific DNS server
nslookup www.mendocino.edu 1.1.1.1
Task 4B

Windows DNS Diagnostics

From your Windows Server VM, run the following:

PowerShell
# 1. nslookup
nslookup www.mendocino.edu

# 2. PowerShell Resolve-DnsName
Resolve-DnsName -Name www.mendocino.edu -Type A

# 3. Display DNS client cache
Get-DnsClientCache | Select-Object -First 10

Task 5 — Troubleshooting Scenarios

In this task, you will analyze network misconfigurations and apply the structured troubleshooting methodology from Module 8.4. For each scenario, identify the problem, explain how you would diagnose it, and describe the fix.

📋 Troubleshooting Methodology Reminder
  1. Step 1: Verify physical/virtual connectivity (link state)
  2. Step 2: Verify IP configuration (correct address, mask, gateway)
  3. Step 3: Verify local connectivity (ping gateway)
  4. Step 4: Verify remote connectivity (ping external host by IP)
  5. Step 5: Verify name resolution (ping external host by name)
  6. Step 6: Verify application-layer connectivity (specific port/service)
Scenario 1

Wrong Default Gateway

A Linux server has the following configuration:

  • IP Address: 192.168.1.50/24
  • Default Gateway: 192.168.1.200 (but the actual gateway is 192.168.1.1)
  • DNS Server: 8.8.8.8

The administrator reports: "I can ping other machines on the 192.168.1.0/24 network, but I cannot reach any external websites or IP addresses."

Scenario 2

DNS Misconfiguration

A Windows Server has the following configuration:

  • IP Address: 10.0.2.25/24
  • Default Gateway: 10.0.2.1
  • DNS Server: 10.0.2.99 (this server does not exist on the network)

The administrator reports: "I can ping 8.8.8.8 successfully, but I cannot browse to any website by name. ping google.com fails with 'could not find host'."

Scenario 3

APIPA Address (DHCP Failure)

A Linux workstation is configured for DHCP, but the administrator notices the following when running ip addr show:

inet 169.254.47.132/16 scope link ens18
Scenario 4 (Bonus)

Hands-On: Intentional Misconfiguration & Recovery

⚠ Take a VM Snapshot First!

Before attempting this scenario, take a snapshot of your Linux VM in Proxmox so you can easily revert if needed.

On your Linux VM, intentionally misconfigure the DNS server to a non-existent address (e.g., 10.0.0.99). Then:

  1. Attempt to ping google.com — document the error
  2. Ping 8.8.8.8 — document the result
  3. Use nslookup to confirm DNS is the issue
  4. Fix the DNS configuration and verify resolution works again

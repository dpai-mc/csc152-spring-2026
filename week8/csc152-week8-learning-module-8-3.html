§1 — The Role of DNS in Network Communication

The Domain Name System (DNS) is often described as the "phone book of the internet." It translates human-readable hostnames (like www.google.com) into IP addresses (like 142.250.80.4) that computers use to route traffic. Without DNS, you would need to memorize IP addresses for every server you want to reach.

Why DNS Matters for Server Administrators

DNS is not just a convenience — it is a critical infrastructure service that nearly every other service depends on:

  • Software updates: apt update and Windows Update resolve repository hostnames via DNS
  • Email delivery: Mail servers use DNS MX records to find the recipient's mail server
  • Active Directory: Windows domain-joined machines rely on DNS to locate domain controllers
  • Web services: Every HTTP request begins with a DNS lookup
  • Internal services: In Project 2, your VMs will use DNS to find each other by hostname rather than memorizing IPs
🚨 Common Misconception

"My server has an IP address and can ping the gateway, so networking is done." Not quite. Without DNS configuration, your server can reach other hosts by IP but cannot resolve hostnames. Commands like apt update, curl https://example.com, and even ping google.com will fail. DNS is the final piece of basic network configuration.

How DNS Resolution Works (Simplified)

When your server needs to resolve a hostname, the process follows these steps:

The DNS resolver is the software on your system that handles this lookup. On Linux, this is typically systemd-resolved or the C library's built-in resolver reading /etc/resolv.conf. On Windows, it is the DNS Client service. Your job as an administrator is to tell the resolver which DNS server(s) to query.

§2 — Name Resolution Order

When a system needs to resolve a hostname, it does not immediately contact a DNS server. Instead, it checks several sources in a specific order. Understanding this order is essential for troubleshooting — if a hosts file entry overrides DNS, you need to know where to look.

Linux Resolution Order

On Linux, the name resolution order is controlled by /etc/nsswitch.conf. The default configuration typically specifies:

# /etc/nsswitch.conf (relevant line)
hosts:  files dns

This means:

  1. files → Check /etc/hosts first (local static mappings)
  2. dns → If not found in hosts, query the DNS server(s) listed in /etc/resolv.conf

Windows Resolution Order

Windows follows a similar but slightly more complex order:

  1. Local DNS cache → Previously resolved names are cached in memory
  2. Hosts file → C:\Windows\System32\drivers\etc\hosts
  3. DNS server query → Preferred DNS server, then alternate DNS server
💡 Why This Matters

If you add an entry to /etc/hosts mapping myserver to 10.0.1.50, that mapping takes priority over any DNS response. This is useful for testing (you can override DNS temporarily) but can also cause confusion if you forget about a stale hosts file entry.

Resolution Order Comparison

Name Resolution Order: Linux vs. Windows
Priority Linux Windows
1st (highest) /etc/hosts Local DNS cache
2nd DNS server (via /etc/resolv.conf) C:\Windows\...\hosts
3rd — DNS server (preferred → alternate)
⚠ Windows DNS Cache

Windows aggressively caches DNS responses. If you change a DNS record but Windows still resolves the old IP, flush the cache with: ipconfig /flushdns. Linux with systemd-resolved can be flushed with: sudo resolvectl flush-caches.

§3 — Linux DNS Client Configuration

On Linux, there are multiple ways to configure which DNS servers your system uses. The method depends on your distribution and whether you use systemd-resolved, NetworkManager, or Netplan. We will focus on the two approaches most relevant to your lab environment: direct /etc/resolv.conf editing and Netplan configuration.

Understanding /etc/resolv.conf

The file /etc/resolv.conf is the traditional DNS configuration file on Linux. It tells the system's resolver which DNS servers to query and which domain to append to short hostnames.

/etc/resolv.conf
# DNS servers to query (up to 3)
nameserver 10.0.2.1
nameserver 8.8.8.8

# Domain to append for short hostnames
search techstart.local

Key Directives

/etc/resolv.conf Directives
Directive Purpose Example
nameserver IP address of a DNS server to query (max 3 entries) nameserver 10.0.2.1
search Domain(s) appended to unqualified hostnames search techstart.local
domain Default domain (similar to search but for a single domain) domain techstart.local
⚠ Direct Editing Limitation

On many modern distributions, /etc/resolv.conf is auto-generated by systemd-resolved or NetworkManager. Edits you make directly may be overwritten on reboot or when the network restarts. For persistent changes, configure DNS through Netplan (Ubuntu) or nmcli (Rocky Linux) instead.

Persistent Configuration with Netplan (Ubuntu Server)

In Module 8.2, you configured a static IP using Netplan. DNS settings are added to the same YAML file. The nameservers directive specifies DNS servers and search domains:

/etc/netplan/01-netcfg.yaml
network:
  version: 2
  ethernets:
    ens18:
      addresses:
        - 10.0.2.10/24
      routes:
        - to: default
          via: 10.0.2.1
      nameservers:
        addresses:
          - 10.0.2.1
          - 8.8.8.8
        search:
          - techstart.local

After editing, apply the configuration:

sudo netplan apply

Netplan writes the DNS settings into systemd-resolved (or generates /etc/resolv.conf directly), ensuring your changes survive reboots.

Verifying DNS Configuration on Linux

After configuring DNS, verify the active settings:

Bash
# Check the current resolv.conf contents
cat /etc/resolv.conf

# If using systemd-resolved, check its status
resolvectl status

# Quick test: resolve a hostname
nslookup google.com

systemd-resolved is a system service that provides DNS resolution for local applications. On Ubuntu Server, it typically runs by default and manages /etc/resolv.conf as a symlink to its own stub resolver at 127.0.0.53.

When you see nameserver 127.0.0.53 in /etc/resolv.conf, that is the systemd-resolved stub listener — it forwards queries to the actual upstream DNS servers configured via Netplan or NetworkManager.

Useful commands:

# Show all DNS configuration details
resolvectl status

# Flush the DNS cache
sudo resolvectl flush-caches

# Query a specific hostname through systemd-resolved
resolvectl query www.example.com

For this course, you do not need to configure systemd-resolved directly — Netplan handles it. But understanding this layer helps when troubleshooting unexpected resolv.conf contents.

§4 — Windows Server DNS Client Configuration

On Windows Server, DNS client settings are configured per network adapter — the same place where you set the static IP in Module 8.2. Windows provides both a GUI method and PowerShell commands.

GUI Method

From Module 8.2, you already know how to reach adapter properties. The DNS settings are on the same screen:

  1. Open Network and Sharing Center → Change adapter settings
  2. Right-click the adapter → Properties
  3. Select Internet Protocol Version 4 (TCP/IPv4) → Properties
  4. In the lower section, select "Use the following DNS server addresses"
  5. Enter the Preferred DNS server (e.g., 10.0.2.1)
  6. Enter the Alternate DNS server (e.g., 8.8.8.8)
  7. Click OK → Close
💡 Preferred vs. Alternate

Windows queries the Preferred DNS server first. The Alternate server is used only if the preferred server does not respond (timeout). It is not a secondary lookup — if the preferred server responds with "name not found," Windows does not retry with the alternate server.

PowerShell Method

The Set-DnsClientServerAddress cmdlet configures DNS servers for a specific adapter. This is the same cmdlet you may have seen briefly in Module 8.2:

PowerShell (Run as Administrator)
# Set preferred and alternate DNS servers for the "Ethernet" adapter
Set-DnsClientServerAddress -InterfaceAlias "Ethernet" `
    -ServerAddresses ("10.0.2.1", "8.8.8.8")

# Verify the configuration
Get-DnsClientServerAddress -InterfaceAlias "Ethernet"

To configure the DNS suffix (search domain):

PowerShell
# Set the DNS suffix search list
Set-DnsClientGlobalSetting -SuffixSearchList @("techstart.local")

# Verify
Get-DnsClientGlobalSetting

Verifying DNS Configuration on Windows

# Show full IP and DNS configuration
ipconfig /all

# Show just DNS server addresses
Get-DnsClientServerAddress

# Flush DNS cache (useful after changing DNS settings)
ipconfig /flushdns

# View current DNS cache entries
ipconfig /displaydns

§5 — Testing Name Resolution

After configuring DNS client settings, you must verify that name resolution actually works. Three tools are essential: nslookup (cross-platform), dig (Linux), and Resolve-DnsName (PowerShell). Each provides different levels of detail.

Tool Comparison

DNS Testing Tools
Tool Platform Detail Level Best For
nslookup Linux, Windows Basic Quick lookups; available everywhere
dig Linux Detailed Full DNS query/response analysis; record types, TTLs
Resolve-DnsName Windows (PowerShell) Moderate Scriptable; integrates with PowerShell pipeline

Using nslookup (Cross-Platform)

nslookup queries DNS servers and returns the resolved IP address. It works identically on Linux and Windows:

Bash / PowerShell
# Basic forward lookup (hostname → IP)
nslookup www.google.com

# Query a specific DNS server
nslookup www.google.com 8.8.8.8

# Reverse lookup (IP → hostname)
nslookup 142.250.80.4
$ nslookup www.google.com
Server:         10.0.2.1          ← DNS server that answered
Address:        10.0.2.1#53       ← DNS server IP and port (53 = DNS)

Non-authoritative answer:         ← Answer came from cache, not the
                                     authoritative server for google.com
Name:   www.google.com
Address: 142.250.80.4             ← Resolved IPv4 address

"Non-authoritative answer" means the response came from a caching/recursive DNS server (like your gateway or 8.8.8.8), not from Google's own authoritative DNS server. This is normal for most queries.

Using dig (Linux)

dig (Domain Information Groper) provides much more detailed output than nslookup, including the query type, response flags, TTL values, and the authority section. It is the preferred tool for DNS troubleshooting on Linux.

Bash
# Basic query
dig www.example.com

# Query for a specific record type
dig www.example.com A        # IPv4 address record
dig www.example.com AAAA     # IPv6 address record
dig example.com MX           # Mail exchange record

# Short output (just the answer)
dig +short www.example.com

# Query a specific DNS server
dig @8.8.8.8 www.example.com

# Reverse lookup
dig -x 93.184.216.34
$ dig www.example.com

; <<>> DiG 9.18.18 <<>> www.example.com
;; global options: +cmd
;; Got answer:
;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 45231
;; flags: qr rd ra; QUERY: 1, ANSWER: 1, AUTHORITY: 0

;; QUESTION SECTION:
;www.example.com.            IN      A

;; ANSWER SECTION:
www.example.com.     3600    IN      A       93.184.216.34

;; Query time: 23 msec
;; SERVER: 10.0.2.1#53(10.0.2.1) (UDP)
;; WHEN: Thu Mar 12 14:30:00 UTC 2026
;; MSG SIZE  rcvd: 62

Key fields:

  • status: NOERROR — query succeeded
  • ANSWER SECTION — the resolved record: www.example.com has an A record pointing to 93.184.216.34 with a TTL of 3600 seconds (1 hour)
  • SERVER — which DNS server answered (10.0.2.1)
  • Query time — how long the lookup took (23 ms)

Using Resolve-DnsName (PowerShell)

Resolve-DnsName is the PowerShell-native DNS lookup tool. It returns objects that can be piped to other cmdlets:

PowerShell
# Basic lookup
Resolve-DnsName www.google.com

# Query specific record type
Resolve-DnsName example.com -Type MX

# Query a specific DNS server
Resolve-DnsName www.google.com -Server 8.8.8.8

# Reverse lookup
Resolve-DnsName 142.250.80.4

§6 — The Local Hosts File

Before DNS existed, every hostname-to-IP mapping was maintained in a local file. Today, the hosts file still exists on every operating system and is checked before DNS in the resolution order. It is useful for:

  • Testing: Override DNS temporarily to point a hostname at a test server
  • Local aliases: Give friendly names to lab VMs before you build a DNS server
  • Blocking: Map unwanted hostnames to 127.0.0.1 to block access

Hosts File Locations

Hosts File Path by OS
Operating System File Path Requires Elevated Privileges
Linux /etc/hosts Yes — sudo required
Windows C:\Windows\System32\drivers\etc\hosts Yes — Run as Administrator

Hosts File Format

The format is identical on both platforms — one mapping per line: IP address, then one or more hostnames, separated by spaces or tabs. Lines starting with # are comments.

/etc/hosts
# Loopback entries (do not remove)
127.0.0.1       localhost
127.0.1.1       linux-server

# Lab VM mappings (added for Module 8.3 practice)
10.0.2.10       linux-server.techstart.local    linux-server
10.0.2.11       win-server.techstart.local      win-server
10.0.3.20       webserver.techstart.local        webserver

Edit with: sudo nano /etc/hosts

C:\Windows\System32\drivers\etc\hosts
# Loopback entry
127.0.0.1       localhost

# Lab VM mappings
10.0.2.10       linux-server.techstart.local    linux-server
10.0.2.11       win-server.techstart.local      win-server
10.0.3.20       webserver.techstart.local        webserver

Edit with: Open Notepad as Administrator, then File → Open → navigate to the hosts file. Change the file type filter from "Text Documents" to "All Files" to see it.

🚨 Common Pitfall: nslookup vs. Hosts File

nslookup and dig query DNS servers directly — they do not read the hosts file. If you add an entry to /etc/hosts and test with nslookup, it will appear to fail. Use ping or getent hosts (Linux) to test hosts file entries. This is one of the most common troubleshooting mistakes students make.

§7 — DNS Record Types Preview

In Week 9, you will configure a DNS server and create actual DNS records. For now, you need a conceptual understanding of the most common record types — enough to interpret dig and nslookup output and prepare for Quiz 8.

Common DNS Record Types
Record Type Name Purpose Example
A Address Maps a hostname to an IPv4 address www.example.com → 93.184.216.34
AAAA IPv6 Address Maps a hostname to an IPv6 address www.example.com → 2606:2800:220:1:…
PTR Pointer Reverse lookup — maps an IP back to a hostname 93.184.216.34 → www.example.com
CNAME Canonical Name Creates an alias pointing to another hostname mail.example.com → mx1.example.com
MX Mail Exchange Identifies the mail server for a domain example.com MX → mx1.example.com
NS Name Server Identifies authoritative DNS servers for a domain example.com NS → ns1.example.com
SOA Start of Authority Contains zone metadata (serial number, refresh intervals) example.com SOA ns1.example.com…

Forward vs. Reverse Lookups

The two most common DNS operations are:

  • Forward lookup: hostname → IP address (uses A or AAAA records)
  • Reverse lookup: IP address → hostname (uses PTR records in a special in-addr.arpa zone)

Reverse DNS uses a special domain called in-addr.arpa. The IP address is written in reverse order and appended to this domain. For example:

  • IP: 10.0.2.10
  • Reverse DNS name: 10.2.0.10.in-addr.arpa

A PTR record at that name points back to the hostname. You can test this with:

# dig reverse lookup
dig -x 10.0.2.10

# nslookup reverse lookup
nslookup 10.0.2.10

In Project 2 Part B, you will create a reverse lookup zone on your DNS server for your lab network. Understanding the in-addr.arpa structure now will make that configuration much clearer.

Key Takeaways

  1. DNS translates hostnames to IP addresses and is a critical dependency for nearly every network service.
  2. Name resolution checks the local hosts file before querying DNS servers (on both Linux and Windows).
  3. On Linux, configure DNS persistently through Netplan (nameservers directive), not by editing /etc/resolv.conf directly.

Scenario 1 of 3

Server Can Ping by IP but Not by Hostname

Your Linux server can successfully ping 8.8.8.8 but ping google.com fails with "Temporary failure in name resolution." You check /etc/resolv.conf and find it is empty.

Questions:

  • What is the most likely cause?
  • What command would you run to fix this persistently on Ubuntu Server?
  • What file would you edit?
Your answers — Scenario 1
💡 Hint

If the server can reach an IP address but not a hostname, the network layer is working — the problem is at the DNS resolution layer. An empty /etc/resolv.conf means no DNS servers are configured. On Ubuntu Server, Netplan manages this file.

✅ Solution — Scenario 1

Cause: No DNS nameserver is configured. The empty /etc/resolv.conf means the resolver has no server to query.

Fix: Edit the Netplan configuration file (e.g., /etc/netplan/01-netcfg.yaml) and add the nameservers block under the interface:

      nameservers:
        addresses:
          - 10.0.2.1
          - 8.8.8.8

Then apply: sudo netplan apply

Verify: cat /etc/resolv.conf should now show the nameserver entries, and ping google.com should succeed.

Scenario 2 of 3

nslookup Works but ping by Hostname Fails

You added the following entry to /etc/hosts on your Linux server:

10.0.2.11    win-server

Running ping win-server succeeds (pings 10.0.2.11). But when you try nslookup win-server, it returns "server can't find win-server: NXDOMAIN".

Question: Why does ping work but nslookup fails? Is there actually a problem?

💡 Hint

Re-read the "Common Pitfall" callout in §6. Think about what nslookup queries versus what ping uses for name resolution.

✅ Solution — Scenario 2

There is no problem. This is expected behavior:

  • ping uses the system resolver, which follows the resolution order: /etc/hosts first, then DNS. It finds win-server in the hosts file and resolves it to 10.0.2.11.
  • nslookup queries the DNS server directly — it does not read /etc/hosts. Since there is no DNS record for win-server, the DNS server returns NXDOMAIN (non-existent domain).

This is one of the most common sources of confusion. Remember: nslookup and dig bypass the hosts file. Use ping or getent hosts to verify hosts file entries.

Scenario 3 of 3

Windows DNS Resolution Returns Wrong IP

You changed a DNS record for webserver.techstart.local from 10.0.3.20 to 10.0.3.25 on your DNS server. But when you run nslookup webserver.techstart.local on your Windows Server, it still returns the old IP 10.0.3.20.

Questions:

  • What is the most likely cause?
  • What command would you run to fix it?
  • Where else should you check for stale mappings?
💡 Hint

Windows caches DNS responses aggressively. Also consider: could there be a hosts file entry overriding DNS? Check both the cache and the hosts file.

✅ Solution — Scenario 3

Most likely cause: Windows has cached the old DNS response. The cached entry has not expired yet (its TTL has not elapsed).

Fix: Flush the Windows DNS cache:

ipconfig /flushdns

Then retry nslookup webserver.techstart.local — it should now return 10.0.3.25.

Also check: The hosts file (C:\Windows\System32\drivers\etc\hosts) for a stale entry mapping webserver.techstart.local to the old IP. Since the hosts file is checked before DNS on Windows (after the cache), a stale entry there would also cause this behavior.

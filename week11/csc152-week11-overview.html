Week 11 · Welcome

Welcome to Week 11

Welcome to Week 11! This week marks a significant milestone in the course: you move from managing a single server in isolation to building a centralized directory service that can manage thousands of users, computers, and resources across an entire organization. The technology at the center of that shift is Active Directory Domain Services (AD DS) — the backbone of identity management in virtually every Windows-based enterprise environment you will encounter in your career.

Think of AD DS as three things working together: an enterprise phone book that knows where every user, computer, and resource lives; a security guard that authenticates every login and enforces access rules; and a policy enforcer that pushes configuration settings to every machine in the organization — automatically. This week you will build all three of those capabilities from scratch in your Proxmox lab environment.

Week 11 builds directly on the user and group administration skills from Week 6 and the DNS and network services work from Weeks 8–9. You will install a Domain Controller, design an Organizational Unit hierarchy, create domain users, join a client machine to the domain, and implement Group Policy Objects. By Sunday you will have a functioning AD DS domain — the same foundation that Project 3 will extend over the next four weeks.

  • Project 2: Network Infrastructure & Firewall Implementation is due this week — plan your time accordingly for final testing, documentation, and submission.
  • Project 3: Directory Services & Security Implementation is assigned this week. Read the full specification and begin planning your AD DS domain structure.

1 Explain what a directory service is and describe the role of Active Directory Domain Services in enterprise environments, including how it differs from a workgroup model.
2 Describe the AD DS logical structure — forests, trees, domains, and Organizational Units — and the physical structure — domain controllers, sites, and site links.
3 Explain the relationship between AD DS and DNS, including why DNS must be available before AD DS can be installed and what SRV records do.
4 Install and configure the AD DS role on Windows Server and promote a server to a Domain Controller for a new forest using both the GUI wizard and PowerShell.
5 Verify a Domain Controller installation using dcdiag, nslookup, Event Viewer, and Active Directory Users and Computers (ADUC).
6 Design and create an Organizational Unit hierarchy that reflects real organizational structures and supports efficient GPO application and delegation of control.
7 Create and manage domain user accounts and security groups using ADUC and PowerShell (New-ADUser), and join a Windows client machine to the domain.
8 Explain the Group Policy processing order (LSDOU) and create, link, and verify Group Policy Objects using GPMC, gpupdate /force, and gpresult.
9 Identify key Windows Security Event IDs generated by AD DS (logon events, account changes, lockouts) and explain how Splunk can be used to monitor directory service health.

Week 11 · Key Concepts

Ideas to Pay Attention To

These are the most important ideas woven through all five modules this week. Keep them in mind as you work through the readings, guided practices, and lab activities.

Directory Services as Infrastructure

A directory service is not just a user database — it is the authentication, authorization, and policy engine for an entire organization. AD DS is the dominant implementation in Windows enterprise environments.

AD DS Logical vs. Physical Structure

The logical structure (forest → tree → domain → OU → objects) controls administration and policy scope. The physical structure (domain controllers, sites, site links) controls replication and authentication performance. Both matter for design decisions.

AD DS Depends on DNS

Active Directory cannot function without DNS. Domain controllers register SRV records so clients can locate them. Misconfigured DNS is the most common cause of AD DS failures — always verify DNS first when troubleshooting.

OUs Are for Administration; Groups Are for Access

Organizational Units organize objects for GPO application and delegation of control. Security Groups control access to resources. Confusing the two is a very common mistake — make sure you understand the distinction.

LSDOU — Group Policy Processing Order

Group Policy is applied in a specific order: Local → Site → Domain → OU. Later-applied policies win by default unless a higher-level policy is marked Enforced. Understanding LSDOU is essential for predicting and troubleshooting policy behavior.

Verification Is Not Optional

Every configuration step this week has a corresponding verification tool: dcdiag, nslookup, gpresult, Event Viewer. Professionals always verify — do not skip these steps in your lab work.

Least Privilege and Delegation

AD DS enables fine-grained delegation of control — you can grant a help desk user the ability to reset passwords in one OU without making them a Domain Admin. This is a core security principle you will apply in Project 3.

Monitoring Starts with Event IDs

Windows Security Event Log entries (4624 successful logon, 4625 failed logon, 4740 account lockout, 4720 account created) are the raw data that security monitoring tools like Splunk analyze. Knowing what to look for is the first step toward meaningful monitoring.

PowerShell Is the Professional Path

Every GUI task this week has a PowerShell equivalent. In production environments, PowerShell is preferred for repeatability, automation, and scripted deployments. Practice both — the GUI builds understanding; PowerShell builds skill.

Section 1

Installing the AD DS Role

Before a Windows Server can function as a Domain Controller, the Active Directory Domain Services (AD DS) role must be installed. Installing the role copies the necessary binaries and management tools onto the server — but it does not yet make the server a Domain Controller. That step (promotion) happens next. Think of this as laying the foundation before raising the walls.

ℹ️
Role vs. Promotion — Two Distinct Steps

Installing the AD DS role and promoting the server to a Domain Controller are separate operations. You must complete both. After role installation, Server Manager will display an alert prompting you to run the AD DS Configuration Wizard.

Using the Add Roles and Features Wizard

  1. Open Server Manager. It launches automatically at login. From the Dashboard, click Add roles and features.
  2. Before You Begin. Read the prerequisites notice, then click Next.
  3. Installation Type. Select Role-based or feature-based installation, then click Next.
  4. Server Selection. Confirm your server is selected from the server pool, then click Next.
  5. Server Roles. Scroll down and check Active Directory Domain Services. A dialog will appear listing required features — click Add Features, then Next.
  6. Features. No additional features are needed at this step. Click Next.
  7. AD DS Information Page. Read the important notes about DNS and functional levels. Click Next.
  8. Confirmation. Review the summary. Optionally check Restart the destination server automatically if required. Click Install.
  9. Installation Progress. Wait for the role to install (typically 2–5 minutes). When complete, you will see a link: "Promote this server to a domain controller." Do not close Server Manager yet.
✅
Post-Install Alert

After installation, Server Manager's notification flag (🔔) will show a yellow warning reading "Configuration required for Active Directory Domain Services." Clicking this link launches the AD DS Configuration Wizard covered in Section 2.

Using PowerShell — Install-WindowsFeature

The same role installation can be performed in a single PowerShell command. This is faster, scriptable, and preferred for automated or headless deployments.

PowerShell — Run as Administrator 
# Install the AD DS role and its management tools
Install-WindowsFeature -Name AD-Domain-Services -IncludeManagementTools

# Verify the role is installed
Get-WindowsFeature -Name AD-Domain-Services
ParameterValuePurpose
-NameAD-Domain-ServicesSpecifies the AD DS role to install
-IncludeManagementTools(switch)Installs ADUC, ADAC, Group Policy tools, and AD PowerShell module

Expected output after successful installation:

Success Restart Needed Exit Code      Feature Result
------- -------------- ---------      --------------
True    No             Success        {Active Directory Domain Services, ...}
💡
No Restart Required for Role Installation

Unlike some Windows roles, installing AD DS binaries does not require a restart. The restart occurs later, at the end of the promotion process.

Section 2

Promoting to a Domain Controller

Promotion is the process that transforms an ordinary Windows Server (with the AD DS role installed) into a functioning Domain Controller. During promotion, Windows configures the AD DS database, installs DNS (if not already present), creates the SYSVOL and NETLOGON shares, and then restarts the server. After the reboot, the server is a Domain Controller.

AD DS Configuration Wizard — Step by Step

Launch the wizard from Server Manager's notification flag or from the Tools → Active Directory Domain Services Installation Wizard menu.

Select the deployment operation. For a brand-new environment, choose:

  • Add a new forest — creates the first domain in a new forest (most common for lab and new deployments)
  • Add a domain controller to an existing domain — for adding redundancy to an existing domain
  • Add a new domain to an existing forest — for multi-domain environments

For this course and Project 3, select Add a new forest.

Root domain name: Enter your domain name. For lab use, the instructor-assigned name is corp.local (or as specified in your Project 3 specification). Use a .local suffix for internal-only domains that will not be publicly registered.

⚠️
Domain Name Is Permanent

The root domain name cannot be changed after promotion without rebuilding the domain. Choose carefully. In production, use a subdomain of a registered name (e.g., ad.yourcompany.com) rather than .local to avoid mDNS conflicts.

Forest Functional Level and Domain Functional Level

Functional levels define the minimum version of Windows Server that all Domain Controllers in the forest or domain must run. Setting a higher functional level unlocks advanced AD DS features but prevents adding older Domain Controllers.

Functional LevelMin. DC OS RequiredKey Features Unlocked
Windows Server 2016Server 2016Privileged Access Management (PAM), rolling NTLM secrets
Windows Server 2019Server 2019Azure AD hybrid features, improved security baselines
Windows Server 2022Server 2022Latest feature set; use for new deployments

For this course, accept the default (matches your installed Server version). In production, choose the highest level your environment can support.

Directory Services Restore Mode (DSRM) Password

The DSRM password is a special recovery password used to boot the Domain Controller into a safe mode for AD DS database repair or restoration. It is separate from the local Administrator or domain Administrator password.

🔐
Protect Your DSRM Password
  • Use a strong, unique password (12+ characters, mixed case, numbers, symbols)
  • Store it securely — if you lose it and need to perform AD DS recovery, you will be locked out
  • In production, store in a privileged access management vault (e.g., CyberArk, HashiCorp Vault)
  • For this lab, document it in your lab report alongside your other credentials

Additional options on this page:

  • DNS server: Leave checked — the wizard will install and configure DNS automatically
  • Global Catalog (GC): Leave checked — the first DC in a forest must be a GC server
  • Read only domain controller (RODC): Leave unchecked for a standard writable DC

If no existing DNS infrastructure is detected, the wizard will warn that a delegation for this DNS server cannot be created. In a new lab forest with no parent DNS zone, this warning is expected and safe to ignore. Click Next.

ℹ️
Why AD DS Requires DNS

Active Directory clients locate Domain Controllers using DNS SRV (Service) records. When a client needs to authenticate, it queries DNS for records like _ldap._tcp.corp.local to find a DC. Without DNS, clients cannot find the domain at all. This is why DNS is installed automatically during promotion.

The wizard auto-populates the NetBIOS domain name from your root domain name (e.g., corp.local → CORP). Accept the default unless you have a specific reason to change it. NetBIOS names are limited to 15 characters and are used for backward compatibility with older Windows clients.

The wizard sets default paths for three critical AD DS components:

ComponentDefault PathPurpose
Database (NTDS.dit) C:\Windows\NTDS The AD DS database file — stores all directory objects
Log files C:\Windows\NTDS Transaction logs for AD DS database changes
SYSVOL C:\Windows\SYSVOL Replicated folder containing GPOs and logon scripts

Accept defaults for the lab. In production, placing NTDS and SYSVOL on a separate volume from the OS improves performance and simplifies backup.

The wizard displays a summary of all chosen settings. Review carefully. Click View script to see the equivalent PowerShell command — this is a great way to learn the PowerShell syntax for promotion.

The Prerequisites Check runs automatically before the Install button becomes active. Common warnings (safe to proceed):

  • DNS delegation warning (expected in a new forest)
  • Cryptography algorithm warning (acceptable for lab environments)
  • Windows Firewall is not configured warning (configure after promotion)
⚠️
Errors vs. Warnings

Yellow warnings allow you to proceed. Red errors must be resolved before the Install button will activate. Common errors include: static IP not configured, DNS service not responding, or insufficient disk space.

Click Install. The server will configure AD DS and automatically restart. Do not interrupt this process.

What Happens During Promotion

During the promotion process (after clicking Install), Windows performs these actions automatically:

  • DNS Server role installed — if not already present, DNS is installed and a forward lookup zone for the domain is created
  • AD DS database initialized — NTDS.dit is created at the specified path
  • SYSVOL share created — replicated folder for GPOs and scripts; shared as \\domain\SYSVOL
  • NETLOGON share created — legacy share used by clients for domain logon scripts; shared as \\domain\NETLOGON
  • DNS SRV records registered — records like _ldap._tcp.corp.local and _kerberos._tcp.corp.local are created so clients can locate the DC
  • Kerberos KDC service started — the Key Distribution Center handles authentication tickets
  • Server restarts — mandatory; the server boots as a Domain Controller
🎉
After the Reboot

When the server comes back online, the login screen will show the domain name (e.g., CORP\Administrator). Server Manager will display new tiles for AD DS and DNS. The server is now a functioning Domain Controller.

Section 3

PowerShell Promotion

The GUI wizard is excellent for learning, but in production and automated environments, PowerShell promotion is the standard approach. A single Install-ADDSForest command (or a short script) can replicate everything the wizard does — and it can be version-controlled, peer-reviewed, and run unattended.

Complete PowerShell Promotion Example

The following two-command sequence installs the role and promotes the server to a DC for a new forest:

PowerShell — Run as Administrator 
# Step 1: Install the AD DS role and management tools
Install-WindowsFeature -Name AD-Domain-Services -IncludeManagementTools

# Step 2: Promote to Domain Controller — new forest
Install-ADDSForest `
    -DomainName "corp.local" `
    -DomainNetbiosName "CORP" `
    -ForestMode "WinThreshold" `
    -DomainMode "WinThreshold" `
    -InstallDns:$true `
    -DatabasePath "C:\Windows\NTDS" `
    -LogPath "C:\Windows\NTDS" `
    -SysvolPath "C:\Windows\SYSVOL" `
    -SafeModeAdministratorPassword (ConvertTo-SecureString `
        "P@ssw0rd!DSRM2026" -AsPlainText -Force) `
    -Force:$true
⚠️
DSRM Password in Scripts

The example above uses a hardcoded password for clarity. In real deployments, never hardcode credentials in scripts. Use Read-Host -AsSecureString to prompt interactively, or retrieve the password from a secrets manager.

Key Parameters Explained

ParameterExample ValueDescription
-DomainName "corp.local" The fully qualified domain name (FQDN) of the new forest root domain. This becomes the DNS zone name.
-DomainNetbiosName "CORP" The NetBIOS (pre-Windows 2000) name. Maximum 15 characters. Used by legacy clients and in DOMAIN\User logon format.
-ForestMode "WinThreshold" Sets the Forest Functional Level. WinThreshold = Windows Server 2016 and later. Also accepts "Win2019" or "Win2022" depending on build.
-DomainMode "WinThreshold" Sets the Domain Functional Level. Should match or be lower than -ForestMode.
-InstallDns $true Installs the DNS Server role and creates the domain's DNS zone automatically. Required for AD DS to function.
-DatabasePath "C:\Windows\NTDS" Location for the AD DS database file (NTDS.dit).
-LogPath "C:\Windows\NTDS" Location for AD DS transaction log files.
-SysvolPath "C:\Windows\SYSVOL" Location for the SYSVOL replicated folder (GPOs, logon scripts).
-SafeModeAdministratorPassword (SecureString) The DSRM password. Must be provided as a SecureString object using ConvertTo-SecureString.
-Force $true Suppresses confirmation prompts, allowing the command to run non-interactively in scripts.

Interactive DSRM Password (Safer for Lab Use)

To avoid hardcoding the DSRM password, prompt for it interactively:

PowerShell — Interactive Secure Password Entry 
# Prompt securely — password is not visible when typed
$dsrmPassword = Read-Host -Prompt "Enter DSRM Password" -AsSecureString

Install-ADDSForest `
    -DomainName "corp.local" `
    -DomainNetbiosName "CORP" `
    -ForestMode "WinThreshold" `
    -DomainMode "WinThreshold" `
    -InstallDns:$true `
    -SafeModeAdministratorPassword $dsrmPassword `
    -Force:$true

Why PowerShell Promotion Is Preferred in Production

💡
GUI vs. PowerShell — The Production Reality
  • Repeatability: A PowerShell script deploys identically every time; GUI steps can vary by human error
  • Documentation: The script is the documentation — it shows exactly what was configured
  • Automation: Scripts integrate with CI/CD pipelines, Ansible, or Azure DevOps for Infrastructure-as-Code workflows
  • Remote execution: PowerShell remoting allows DC promotion from a management workstation without RDP
  • Audit trail: Scripts checked into version control provide a change history that GUI actions cannot

Verifying Promotion Completed Successfully (Quick Check)

Immediately after the reboot, run these commands to confirm the server is now a DC:

PowerShell 
# Confirm AD DS domain information
Get-ADDomain

# Confirm forest information
Get-ADForest

# Check that the server is listed as a Domain Controller
Get-ADDomainController

Post-Installation Verification

A successful promotion reboot does not guarantee a healthy Domain Controller. Always verify the installation using the four standard tools below. These same tools are used in production environments to diagnose DC issues, and they are directly tested in Lab 11 and Quiz 11.

dcdiag

Runs a comprehensive suite of diagnostic tests against the Domain Controller. The go-to first step for DC health checks.

nslookup

DNS query tool. Used to confirm that critical AD DS SRV records were registered correctly during promotion.

ADUC

Active Directory Users and Computers. GUI tool for confirming domain objects (default OUs, DC computer object) exist.

Event Viewer

Windows event log viewer. Check the Directory Service and DNS Server logs for errors or warnings after promotion.

dcdiag is a command-line tool that runs a battery of tests against a Domain Controller and reports pass/fail results. Run it from an elevated Command Prompt or PowerShell on the DC.

Command Prompt or PowerShell — Run as Administrator 
# Run all default diagnostic tests
dcdiag

# Run only DNS-related tests (most relevant post-promotion)
dcdiag /test:dns

# Run all tests and write output to a log file
dcdiag /v /f:C:\dcdiag-results.txt

Key tests to look for in the output:

Test NameWhat It Checks
ConnectivityDC can be located via DNS and is reachable on the network
AdvertisingDC is advertising itself as a DC, GC, KDC, and time server
FrsEvent / DFSREventSYSVOL replication is healthy (no errors in event logs)
SysVolCheckSYSVOL share exists and is accessible
NetLogonsNETLOGON share exists and is accessible
DNSDNS SRV records are registered and resolvable

A healthy DC shows "passed" for all tests. Any "failed" result requires investigation — check the specific error message and consult Event Viewer.

AD DS clients find Domain Controllers by querying DNS for SRV (Service Location) records. After promotion, you must confirm these records exist. The most important records to verify are the LDAP and Kerberos SRV records.

Command Prompt — SRV Record Verification 
# Method 1: Interactive nslookup session
nslookup
> set type=SRV
> _ldap._tcp.corp.local
> _kerberos._tcp.corp.local
> exit

# Method 2: Single-command queries (non-interactive)
nslookup -type=SRV _ldap._tcp.corp.local
nslookup -type=SRV _kerberos._tcp.corp.local

Expected successful output for _ldap._tcp.corp.local:

Server:  dc01.corp.local
Address:  192.168.1.10

_ldap._tcp.corp.local   SRV service location:
          priority       = 0
          weight         = 100
          port           = 389
          svr hostname   = dc01.corp.local

dc01.corp.local  internet address = 192.168.1.10
ℹ️
Understanding the SRV Record
  • _ldap._tcp.corp.local — tells clients which server handles LDAP (port 389) for the domain
  • _kerberos._tcp.corp.local — tells clients which server handles Kerberos authentication (port 88)
  • _gc._tcp.corp.local — locates the Global Catalog server (port 3268)

If these records are missing, clients cannot join the domain or authenticate. See the Common Errors section for diagnosis steps.

ADUC is the primary GUI tool for managing AD DS objects. After promotion, open it to confirm the domain structure was created correctly.

How to open ADUC:

  • Server Manager → Tools → Active Directory Users and Computers
  • Run dialog: dsa.msc
  • PowerShell: Start-Process dsa.msc

What to verify in ADUC after promotion:

  • The domain (corp.local) appears in the left tree
  • Default containers exist: Builtin, Computers, Domain Controllers, Users
  • The Domain Controllers OU contains your DC's computer object (e.g., DC01)
  • Default security groups are present in the Builtin and Users containers (Domain Admins, Enterprise Admins, etc.)
PowerShell — Equivalent Verification Commands 
# List all Domain Controllers in the domain
Get-ADDomainController -Filter *

# Verify domain details
Get-ADDomain | Select-Object Name, DNSRoot, NetBIOSName, DomainMode

# Verify forest details
Get-ADForest | Select-Object Name, ForestMode, Domains, GlobalCatalogs

# List default OUs/containers
Get-ADOrganizationalUnit -Filter * | Select-Object Name, DistinguishedName

Event Viewer records detailed log entries for every significant system event. After DC promotion, check these specific logs for errors or warnings:

Log LocationPath in Event ViewerWhat to Look For
Directory Service Applications and Services Logs → Microsoft → Windows → Directory Services AD DS initialization events, replication status, SYSVOL readiness (Event ID 1394, 1168)
DNS Server Applications and Services Logs → Microsoft → Windows → DNS-Server DNS zone creation, SRV record registration, resolver errors
System Windows Logs → System Kerberos KDC startup, Netlogon service startup (Event ID 5805)
Application Windows Logs → Application AD DS database initialization errors (ESENT source)

Key Event IDs to know:

  • Event ID 1394 (Directory Service) — All problems preventing AD DS from running have been cleared
  • Event ID 1000 (Directory Service) — AD DS startup completed successfully
  • Event ID 5805 (System/Netlogon) — Netlogon service started successfully
PowerShell — Query Event Logs for AD DS Events 
# Check Directory Service log for errors in the last 24 hours
Get-WinEvent -LogName "Directory Service" |
    Where-Object { $_.LevelDisplayName -in "Error","Warning" } |
    Select-Object TimeCreated, Id, Message |
    Format-List

# Check DNS Server log for errors
Get-WinEvent -LogName "DNS Server" |
    Where-Object { $_.LevelDisplayName -eq "Error" } |
    Select-Object TimeCreated, Id, Message |
    Format-List

Common Errors Mini-Case

DC promotion failures are common — even for experienced administrators. The three scenarios below represent the most frequently encountered issues. Read each scenario, form your own diagnosis, then reveal the guided steps to compare your thinking.

🔴

Scenario 1 — DNS Not Resolving After Promotion

Situation: Alex has successfully promoted a Windows Server to a DC for corp.local. The server rebooted and Alex can log in with domain credentials. However, when trying to join a Windows 10 client to the domain, the error reads: "An Active Directory Domain Controller (AD DC) for the domain 'corp.local' could not be contacted." Running nslookup _ldap._tcp.corp.local from the client returns "Non-existent domain".

Diagnosis & Resolution Steps
  1. Check the client's DNS server setting. On the Windows 10 client, run ipconfig /all and verify the DNS Servers field points to the DC's IP address (e.g., 192.168.1.10), not to an external DNS server (e.g., 8.8.8.8). External DNS servers do not know about corp.local.
  2. Verify the DC's DNS server is running. On the DC, open Server Manager → DNS or run: Get-Service DNS. If the DNS service is stopped, start it: Start-Service DNS.
  3. Confirm the forward lookup zone exists on the DC. Open DNS Manager (Server Manager → Tools → DNS). Expand the DC → Forward Lookup Zones. You should see a zone named corp.local. If missing, the zone was not created during promotion — re-run dcdiag /test:dns for details.
  4. Check SRV records exist in the zone. In DNS Manager, expand corp.local → _tcp. You should see _ldap, _kerberos, and _gc SRV records. If missing, force re-registration: netlogon /force or restart the Netlogon service.
  5. Update the client DNS and retry. Set the client's DNS to the DC IP, flush DNS cache (ipconfig /flushdns), and retry the domain join.
🔴

Scenario 2 — Promotion Fails: "The password does not meet the length, complexity, or history requirement"

Situation: Jordan is running Install-ADDSForest from PowerShell. The command fails immediately with the error: "The password does not meet the length, complexity, or history requirement of the domain." Jordan used the password admin123 for the DSRM password parameter.

Diagnosis & Resolution Steps
  1. Understand the requirement. The DSRM password is subject to Windows password complexity rules even before the domain is fully created. It must be at least 7 characters and contain characters from three of the following four categories: uppercase letters, lowercase letters, digits (0–9), and special characters (!@#$%^&*). The password admin123 fails because it uses only lowercase letters and digits (two categories).
  2. Choose a compliant DSRM password. Examples of compliant passwords: P@ssw0rd!DSRM, Corp!DC2026#Restore. Use a minimum of 12 characters for security best practice.
  3. Re-run the command with a compliant password. If using the interactive method, use Read-Host -AsSecureString to enter the new password securely without embedding it in the script.
  4. Document the DSRM password securely. Store it in a password manager or your lab documentation. It is needed only for AD DS recovery scenarios, but losing it can be catastrophic.
🔴

Scenario 3 — Prerequisites Check Fails: Functional Level Mismatch

Situation: Morgan is adding a second Domain Controller to an existing corp.local domain. The AD DS Configuration Wizard's prerequisites check fails with: "The domain functional level is higher than the forest functional level." The existing domain was configured at DFL Windows Server 2022, but Morgan accidentally set the Forest Functional Level to Windows Server 2016 in the wizard.

Diagnosis & Resolution Steps
  1. Understand functional level constraints. The Forest Functional Level (FFL) sets a ceiling — no domain within the forest can have a Domain Functional Level (DFL) higher than the FFL. Setting FFL to 2016 while the existing domain is at DFL 2022 is invalid.
  2. Check the current FFL and DFL of the existing domain. On the existing DC, run: 
    Get-ADForest | Select-Object ForestMode
Get-ADDomain | Select-Object DomainMode
    Note the values — your new DC must use a Forest Mode ≥ the existing DFL.
  3. Re-run the wizard (or PowerShell command) with matching or higher functional levels. When adding a DC to an existing domain, the functional level fields in the wizard are typically read-only (inherited from the existing domain/forest). If running Install-ADDSDomainController via PowerShell, do not specify -DomainMode or -ForestMode — these are inherited automatically.
  4. If creating a new forest: Set -ForestMode and -DomainMode to the same value (e.g., both "WinThreshold"). The DFL can never exceed the FFL.

Section 1

What Is a Directory Service?

Definition

A directory service is a specialized database that stores, organizes, and provides access to information about network resources — users, computers, printers, groups, policies, and more. Unlike a general-purpose relational database, a directory service is optimized for frequent reads and hierarchical lookups rather than complex transactional writes.

💡 The Three-Role Analogy
Think of a directory service as three things in one:
  • Phone Book — It stores the names, locations, and attributes of every resource on the network. "Where is the printer in Building B? What is Maria's email address?"
  • Security Guard — It authenticates every user and device that wants to access the network, verifying identity before granting entry.
  • Policy Enforcer — It distributes and enforces configuration policies across thousands of machines simultaneously, ensuring every workstation meets organizational standards.

Why Organizations Depend on Directory Services

  • Centralized authentication — One set of credentials works across all systems (Single Sign-On)
  • Scalability — Manage thousands of users and devices from a single administrative console
  • Policy consistency — Push configuration changes to every machine in the organization instantly
  • Delegation — Grant specific administrative rights to specific people without handing over full control
  • Audit and compliance — Track who accessed what and when, supporting regulatory requirements

A Brief History: X.500 → LDAP → Active Directory

X.500 — The Origin (1988)

In 1988, the International Telecommunication Union (ITU) and ISO published the X.500 standards — a set of protocols defining how a global directory of network resources should be structured and accessed. X.500 introduced the concept of a hierarchical Directory Information Tree (DIT), where entries are organized from a root down through countries, organizations, and individual objects.

X.500 was powerful but complex and required the OSI network stack, which was expensive and difficult to implement. It never saw widespread adoption in its original form, but its data model — the idea of a tree-structured directory with named objects and attributes — became the conceptual foundation for everything that followed.

LDAP — Lightweight Directory Access Protocol (1993–present)

To make X.500's directory model practical for the growing TCP/IP internet, researchers at the University of Michigan developed the Lightweight Directory Access Protocol (LDAP) in 1993. LDAP kept X.500's hierarchical data model and naming conventions but replaced the heavy OSI transport with standard TCP/IP, running on port 389 (or port 636 for LDAPS, the encrypted variant).

LDAP became an open standard (RFC 4511) and is today the universal language for querying and modifying directory services. Both Microsoft Active Directory and open-source OpenLDAP speak LDAP. When you query AD for a user's attributes, you are making an LDAP query — even if you never see the raw protocol.

Key LDAP concepts you will encounter throughout Weeks 11–12:

  • DN (Distinguished Name) — The unique full path to an object, e.g., CN=Maria Garcia,OU=HR,DC=corp,DC=local
  • CN (Common Name) — The name of the object itself
  • OU (Organizational Unit) — A container within the directory tree
  • DC (Domain Component) — A component of the domain name
Active Directory — Microsoft's Implementation (1999–present)

Microsoft introduced Active Directory with Windows 2000 Server in 1999, building a full enterprise directory service on top of LDAP, Kerberos authentication, and DNS. Active Directory is not a replacement for LDAP — it implements LDAP as its access protocol while adding Microsoft-specific extensions, replication mechanisms, Group Policy infrastructure, and deep integration with Windows operating systems.

The specific component we work with in this course is Active Directory Domain Services (AD DS) — the core directory service role that handles authentication, authorization, and policy for a Windows domain. Microsoft has since added other Active Directory services (AD CS, AD FS, AD LDS, AD RMS), but AD DS is the foundation of Windows enterprise environments and the subject of Week 11.

Today, AD DS is used in the vast majority of enterprise Windows environments worldwide. Even organizations moving to cloud-based identity (Azure AD / Microsoft Entra ID) typically maintain an on-premises AD DS instance or synchronize with it.

Workgroup vs. Domain Model

Before deploying Active Directory, it is essential to understand why organizations move to a domain model. The workgroup model works fine for a handful of computers — but it breaks down quickly as organizations grow.

Characteristic Workgroup Model Domain Model
Authentication Local SAM database on each machine Centralized via Domain Controller (Kerberos)
User accounts Separate account on every computer Single domain account works on all computers
Password management Changed separately on each machine Changed once; applies everywhere
Policy enforcement Local Group Policy per machine only Centralized GPOs pushed to all machines
Administration Each machine administered independently Centralized administration from DC
Scalability Practical up to ~10–15 computers Scales to millions of objects
Single Sign-On Not available Full SSO across all domain resources
Security auditing Per-machine logs only Centralized security event logging
Infrastructure required None — peer-to-peer At least one Domain Controller + DNS
Typical use case Home networks, very small offices Any organization with 10+ computers

The Workgroup Model — How It Works

In a workgroup, every Windows computer maintains its own local Security Accounts Manager (SAM) database containing local user accounts and passwords. When a user on Computer A wants to access a shared folder on Computer B, Computer B checks its own SAM — not any central authority — to verify the credentials.

This creates an immediate scaling problem: if you have 50 computers and want a user named "jsmith" to log in to any of them, you must create and maintain a "jsmith" account on all 50 machines. Change the password once? You must change it 50 times.

⚠️ The 10-Computer Rule
Microsoft's own documentation suggests workgroups are practical only for networks of approximately 10–15 computers. Beyond that, the administrative burden becomes unmanageable and security becomes inconsistent.

Workgroup characteristics:

  • All computers are peers — no hierarchy or central authority
  • Each computer name must be unique on the local network
  • Maximum 20 simultaneous connections in Windows home/pro editions
  • No centralized policy enforcement or auditing
  • No Single Sign-On capability

The Domain Model — How It Works

In a domain, a Domain Controller (DC) acts as the central authority for authentication and policy. User accounts, computer accounts, and security policies are stored in the AD DS database on the DC. When a user logs in at any domain-joined workstation, the workstation contacts the DC, which authenticates the user using Kerberos and issues a ticket granting access to network resources.

The user carries their identity with them across the entire domain. Log in at any workstation, access any authorized file server, printer, or application — with the same username and password, managed in one place.

✅ Key Domain Benefits
  • One account, one password — managed centrally
  • Group Policy — push settings to every machine simultaneously
  • Delegation — give help desk staff password-reset rights without making them admins
  • Centralized auditing — know who logged in where and when
  • Software deployment — push applications to machines via GPO

The trade-off: a domain requires infrastructure — at minimum one Domain Controller and a DNS server (often the same machine). This is a worthwhile investment for any organization with more than a handful of computers.

AD DS Logical Structure

The logical structure of AD DS defines how directory objects are organized conceptually — independent of physical hardware or network topology. Understanding this hierarchy is essential before you can make design decisions in Project 3.

AD DS Logical Hierarchy (top = broadest scope)
🌲 Forest Highest boundary of trust and schema
Contains one or more Trees sharing a common schema and Global Catalog
🌳 Tree Contiguous DNS namespace
One or more domains with a contiguous DNS name (e.g., corp.local, sales.corp.local)
🏢 Domain Core administrative boundary
Replication boundary, policy boundary, and security boundary (e.g., corp.local)
📁 Organizational Unit (OU) Administrative container within a domain
Groups objects for GPO application, delegation, and logical organization (e.g., IT, HR, Finance)
👤💻🖨️ Objects Leaf-level directory entries
Users, computers, groups, printers, shared folders — the actual managed resources

Layer-by-Layer Breakdown

🌲 Forest — The Security and Schema Boundary

The forest is the outermost container in AD DS and represents the ultimate security boundary. All domain controllers in a forest share:

  • A common schema — the definition of every object class and attribute that can exist in the directory (e.g., "a User object has these attributes: sAMAccountName, mail, telephoneNumber…")
  • A common Global Catalog — a partial replica of all objects in all domains, enabling forest-wide searches
  • Automatic two-way transitive trusts between all domains in the forest

The first domain created in a forest is called the Forest Root Domain. In most small-to-medium organizations, there is only one forest and one domain — the forest root domain IS the only domain.

⚠️ Important Security Note
The forest is the true security boundary in AD DS — not the domain. A compromised domain administrator in one domain can potentially affect other domains in the same forest. Organizations with strict security separation requirements (e.g., a company and its acquired subsidiary) use separate forests.
🌳 Tree — A Contiguous DNS Namespace

A tree is a collection of one or more domains that share a contiguous DNS namespace. For example:

  • corp.local — root domain of the tree
  • sales.corp.local — child domain, part of the same tree
  • eu.sales.corp.local — grandchild domain, still the same tree

A forest can contain multiple trees with different root DNS names (e.g., corp.local and fabrikam.local). These trees are connected by tree-root trusts.

For most organizations deploying AD DS for the first time, the distinction between a tree and a forest is academic — they will have one tree with one domain. The tree/forest distinction becomes important during mergers, acquisitions, or very large enterprise deployments.

🏢 Domain — The Core Administrative Unit

The domain is the most important unit in day-to-day AD DS administration. It serves as:

  • A replication boundary — full AD DS data replicates between all DCs within a domain; only partial data crosses domain boundaries
  • A policy boundary — domain-level Group Policy (e.g., Default Domain Password Policy) applies to all objects in the domain
  • An administrative boundary — Domain Admins have full control within their domain
  • An authentication boundary — Kerberos tickets are issued within the domain context

Every domain has a unique DNS name (e.g., corp.local) and a NetBIOS name (e.g., CORP) for backward compatibility. The domain is identified in LDAP paths by its DC components: DC=corp,DC=local.

When do you need multiple domains? Multiple domains are justified when you need different password policies at the domain level, different administrative boundaries, or distinct replication topologies. For most organizations with 200–500 users, a single domain is the correct answer — complexity without benefit is a design anti-pattern.

📁 Organizational Unit (OU) — The Administrative Container

An Organizational Unit is a container object within a domain used to organize other objects (users, computers, groups, printers, and even other OUs). OUs serve two primary purposes:

  1. Group Policy application — GPOs are linked to OUs; all objects in the OU receive the policy
  2. Delegation of control — you can grant a non-admin user administrative rights over a specific OU without making them a Domain Admin

Common OU design strategies:

  • By department — IT, HR, Finance, Sales (good for policy differentiation)
  • By geography — NewYork, Chicago, London (good for delegation to site admins)
  • By object type — Users, Computers, Servers (simple; less flexible for GPO)
  • Hybrid — Geography at top level, departments within each location (most common in practice)
⚠️ OUs vs. Security Groups — A Common Confusion
OUs are for organizing objects and applying GPOs/delegation. Security Groups are for granting access to resources (file shares, printers, applications). You cannot assign file share permissions to an OU. You cannot link a GPO to a Security Group. They serve different purposes and are not interchangeable.
👤💻 Objects — The Leaf Entries

Objects are the individual items stored in the directory. Every object belongs to an object class defined in the schema, which determines what attributes the object can have.

Object TypeDescriptionKey Attributes
User Represents a person who can log in to the domain sAMAccountName, userPrincipalName, password, group memberships
Computer Represents a domain-joined machine Computer name, OS version, last logon, group memberships
Group Collection of users/computers for access control Group type (Security/Distribution), scope (Domain Local/Global/Universal)
Printer Published network printer Location, driver, share name
Shared Folder Published network share UNC path, description

Section 4

AD DS Physical Structure

While the logical structure defines how objects are organized conceptually, the physical structure defines how AD DS maps to the actual network infrastructure — servers, network links, and geographic locations. Physical structure decisions directly impact replication efficiency and authentication performance.

Physical Structure Components

🖥️
Domain Controller (DC)
A Windows Server running the AD DS role. Stores a full writable copy of the domain database (NTDS.DIT). Authenticates users and enforces policies. Every domain needs at least one DC; production environments need two or more for redundancy.
🏙️
Site
A logical grouping of IP subnets that share fast, reliable network connectivity. Sites map to physical locations (buildings, campuses, branch offices). AD DS uses site information to route authentication and replication traffic efficiently.
🔗
Site Link
Represents the WAN connection between two sites. Configured with a cost (lower = preferred), a replication schedule, and a replication interval. AD DS uses site links to schedule inter-site replication during off-peak hours.
🌍
Global Catalog (GC)
A special DC that holds a full copy of all objects in its own domain and a partial (read-only) copy of objects from all other domains in the forest. Required for user logon (to resolve Universal Group memberships) and forest-wide object searches.

Why Physical Structure Matters

AD DS Replication

Every change made to the AD DS database — a new user account, a password change, a GPO update — must be replicated to all Domain Controllers in the domain. AD DS uses a multi-master replication model: changes can be made on any writable DC, and those changes propagate to all other DCs.

Intra-site replication (within a site) happens automatically within minutes using the Knowledge Consistency Checker (KCC), which builds an efficient replication topology. Because intra-site connections are assumed to be fast and cheap, replication is triggered immediately on change notification.

Inter-site replication (between sites, across WAN links) is controlled by site link configuration. Administrators schedule replication during off-peak hours (e.g., 2:00 AM) to avoid consuming WAN bandwidth during business hours. The cost assigned to site links determines which path replication traffic takes when multiple paths exist.

Change on DC1 → Notification to DC2 → DC2 Requests Update → Changes Applied

Authentication and Site Awareness

When a user at a workstation presses Ctrl+Alt+Del and enters their domain credentials, the workstation contacts a Domain Controller to authenticate. AD DS is site-aware: it will always attempt to contact a DC in the same site as the workstation first, before trying DCs in remote sites.

This matters enormously for performance. Consider a branch office in Sacramento connected to a main office in San Francisco via a 10 Mbps WAN link. If there is no DC in Sacramento, every single logon attempt must traverse the WAN — creating delays, consuming bandwidth, and potentially failing entirely if the WAN link goes down.

Best practice: Place at least one Domain Controller at every physical location with more than 25–30 users, or at any location where WAN reliability is a concern. This is called placing a local DC for the site.

⚠️ No Local DC = No Logon if WAN is Down
If a branch office has no local DC and the WAN link to the main office fails, users at that branch cannot log in to the domain. (Cached credentials allow some logins, but access to network resources is lost.) This is a critical availability consideration in DC placement decisions.

The Global Catalog Server

The Global Catalog (GC) is a domain controller that has been designated to hold a partial replica of all objects in all domains in the forest. It serves two critical functions:

  1. User logon — When a user logs in, the authenticating DC must contact a GC server to look up the user's Universal Group memberships. Without a reachable GC, logon fails (except for members of the Domain Admins group).
  2. Forest-wide searches — Applications like Microsoft Exchange and Outlook use the GC as a global address book. Queries to the GC (port 3268) can find objects anywhere in the forest.

In a single-domain forest, every DC is automatically a GC server. In multi-domain forests, GC placement must be planned — typically at least one GC per site.

GC vs. regular DC: A regular DC holds the full database for its own domain only. A GC holds the full database for its domain plus a read-only subset of attributes for all other domains. The GC listens on LDAP port 3268 (or 3269 for LDAPS) in addition to the standard port 389.

FSMO Roles — A Brief Introduction

What are FSMO Roles? (Click to expand)

Although AD DS uses multi-master replication for most changes, five operations are single-master — meaning only one DC in the forest or domain can perform them at a time. These are the Flexible Single Master Operations (FSMO) roles, also called Operations Master roles.

RoleScopeResponsibility
Schema MasterForest-wide (1 per forest)Controls all updates to the AD schema
Domain Naming MasterForest-wide (1 per forest)Controls adding/removing domains from the forest
PDC EmulatorPer domainTime synchronization, password changes, account lockouts
RID MasterPer domainAllocates pools of Relative IDs used to create security principals
Infrastructure MasterPer domainUpdates cross-domain object references

In a new single-domain forest, all five FSMO roles are held by the first DC promoted. You will not need to manually manage FSMO roles in this course, but you should know they exist — they appear in Quiz 11 and in real-world troubleshooting.

AD DS and DNS

🔑 The Golden Rule of Active Directory
AD DS requires DNS. DNS is not optional. Without a properly configured DNS server, Active Directory cannot function. This is the single most common source of AD DS installation failures and post-deployment problems.

Why AD DS Requires DNS

Active Directory uses DNS for two fundamental purposes:

  1. Domain name resolution — The domain itself is identified by a DNS name (e.g., corp.local). Every client that wants to join or communicate with the domain must be able to resolve this DNS name.
  2. Service location via SRV records — AD DS registers special Service (SRV) records in DNS that tell clients where to find domain controllers, Kerberos servers, and LDAP servers. Without these records, clients cannot locate the DC — even if the DC is running and healthy.
How a Client Finds a Domain Controller Using DNS
💻 Client Workstation Wants to log in to corp.local
→
🌐 DNS Server Resolves _ldap._tcp.corp.local SRV
→
🖥️ Domain Controller dc01.corp.local (192.168.1.10)

The client queries DNS for the SRV record _ldap._tcp.corp.local to discover the DC's address, then contacts the DC directly for Kerberos authentication.

Key DNS Records Registered by AD DS

Record TypeExamplePurpose
SRV _ldap._tcp.corp.local Locates LDAP service (DC) for the domain
SRV _kerberos._tcp.corp.local Locates Kerberos Key Distribution Center (KDC)
SRV _ldap._tcp.gc._msdcs.corp.local Locates Global Catalog servers
A / AAAA dc01.corp.local → 192.168.1.10 Resolves DC hostname to IP address
NS corp.local NS dc01.corp.local Identifies authoritative DNS server for the domain zone

What Happens When DNS Is Misconfigured

Scenario: DNS Misconfiguration Failure Modes

DNS misconfiguration is responsible for a large proportion of AD DS problems. Here are the most common failure modes:

MisconfigurationSymptomDiagnosis
Client points to wrong DNS server (e.g., ISP DNS instead of DC) Cannot join domain; "domain not found" error nslookup corp.local fails or returns wrong result
SRV records not registered on DC Clients cannot locate DC; logon fails intermittently nslookup -type=SRV _ldap._tcp.corp.local returns no results
DC uses external DNS as forwarder but has no internal zone Internet works but AD DS queries fail DNS zone for corp.local not found on DC
DC's own NIC points to external DNS (not itself) AD DS services fail to start after reboot DC NIC DNS setting should be 127.0.0.1 or its own IP
🛠️ Verification Commands
  • nslookup -type=SRV _ldap._tcp.corp.local — verify DC SRV record
  • dcdiag /test:dns — comprehensive DNS health check for the DC
  • ipconfig /all — verify client DNS server setting
  • nltest /dsgetdc:corp.local — verify DC discovery from a client

AD DS DNS Best Practices

  • Install DNS on the Domain Controller during AD DS promotion (the wizard does this automatically)
  • Configure the DC's own NIC to use itself as the primary DNS server (127.0.0.1 or its own IP)
  • Configure all domain-joined clients and servers to use the DC's IP as their DNS server
  • Use a secondary DNS server (a second DC) for redundancy — never rely on a single DNS server
  • For internet name resolution, configure DNS forwarders on the DC (e.g., forward to 8.8.8.8 for external names)
  • Do not point domain clients to public DNS servers (8.8.8.8, 1.1.1.1) as their primary DNS — they will not be able to find the DC

Section 7

Mini-Case: Greenfield Manufacturing Company

🔗 Connection to Project 3
This mini-case previews the domain design task you will complete in Project 3: Directory Services & Security Implementation. The design thinking you practice here — how many domains, what OU structure, where to place DCs — is exactly what Project 3 Part A asks you to do for your own lab environment.

The Scenario

📋 Scenario Description

Greenfield Manufacturing Co. is a small metal fabrication company with 200 employees spread across three offices:

  • Headquarters (HQ) — Sacramento, CA: 120 employees including executive leadership, HR, Finance, and IT. Connected to the internet via a 1 Gbps fiber connection.
  • Plant Office — Stockton, CA: 55 employees in manufacturing operations, quality control, and plant administration. Connected to HQ via a 50 Mbps MPLS WAN link.
  • Sales Office — San Francisco, CA: 25 employees in sales, marketing, and customer support. Connected to HQ via a 20 Mbps MPLS WAN link.

Currently, all three offices use a workgroup model. Each employee has a local account on their own computer. Sharing files requires setting up individual passwords on each machine. The IT administrator (one person) spends significant time traveling between sites to manage machines individually.

The company has decided to implement Active Directory Domain Services. You have been brought in as the AD DS architect. The company uses Windows Server 2022 and Windows 11 workstations throughout.

Guided Reflection Questions

Before looking at the model answer, consider each question carefully. Use what you have learned in Sections 1–5 to reason through your answers.

1 How many AD DS domains does Greenfield need?
Consider: size, administrative complexity, and whether there is any reason to create separate security or policy boundaries between locations.
2 What OU structure would you recommend?
Consider: the three locations, the departments mentioned, and how GPOs and delegation will need to work for the IT admin.
3 Where should Domain Controllers be placed?
Consider: the WAN link speeds, what happens if a WAN link fails, and the number of users at each location.
4 How many AD DS Sites should be configured?
Consider: the physical network topology and the purpose of Sites in controlling replication and authentication traffic.

✅ Model Answer — Greenfield Manufacturing Co.

1. How Many Domains?

Recommendation: One domain.

With 200 employees across three offices, all under the same company, there is no justification for multiple domains. The company has a single IT administrator, a single set of security policies, and no regulatory or organizational reason to create separate administrative boundaries. Creating multiple domains would add significant complexity (inter-domain trusts, separate domain admin accounts, more complex replication) without any benefit.

Suggested domain name: greenfield.local (or greenfield.com if using a registered public DNS name — best practice for modern deployments).

2. Recommended OU Structure

A geography-first, department-second hybrid structure is appropriate. This allows the IT admin to delegate control to a local point of contact at each site and to apply site-specific GPOs (e.g., different mapped drives, different printer deployments).

  • greenfield.local
    • OU=HQ-Sacramento
      • OU=Executives
      • OU=HR
      • OU=Finance
      • OU=IT
      • OU=Computers-HQ
    • OU=Plant-Stockton
      • OU=Manufacturing
      • OU=QualityControl
      • OU=Computers-Stockton
    • OU=Sales-SanFrancisco
      • OU=Sales
      • OU=Marketing
      • OU=Computers-SF
    • OU=Servers (all server computer objects)
    • OU=ServiceAccounts (service and system accounts)

Separating computer objects into their own sub-OUs (e.g., Computers-HQ) allows computer-targeted GPOs (desktop restrictions, software deployment) to be applied independently of user-targeted GPOs.

3. Domain Controller Placement

Recommendation: At least one DC at each of the three locations.

  • HQ Sacramento: Two DCs (DC01 and DC02) for redundancy. DC01 is the primary DC and holds all FSMO roles. DC02 provides failover and load distribution. Both are Global Catalog servers.
  • Plant Stockton (55 users): One DC (DC03). With 55 users on a 50 Mbps WAN link, a local DC is essential. If the WAN fails, plant employees can still log in and access local resources. DC03 should also be a Global Catalog server.
  • Sales SF (25 users): One DC (DC04). The 20 Mbps WAN link is adequate for light traffic, but 25 users logging in simultaneously would create noticeable delays without a local DC, and a WAN outage would prevent all logins. DC04 should be a Global Catalog server.
4. AD DS Sites Configuration

Recommendation: Three Sites, two Site Links.

  • Site: Sacramento-HQ — subnet covering HQ IP range
  • Site: Stockton-Plant — subnet covering Plant IP range
  • Site: SanFrancisco-Sales — subnet covering SF IP range
  • Site Link: HQ-to-Stockton — cost 100, replicate every 180 minutes, schedule off-peak hours
  • Site Link: HQ-to-SF — cost 150 (lower bandwidth = higher cost), replicate every 180 minutes, schedule off-peak hours

With three properly defined sites, AD DS will automatically ensure that Sacramento users authenticate against DC01/DC02, Stockton users authenticate against DC03, and SF users authenticate against DC04. Replication will be scheduled across the WAN links during off-peak hours to minimize bandwidth consumption.

💡 Key Design Principle Illustrated
Notice that the answer to "How many domains?" was one — even though there are three locations. Location does not equal domain. Location is addressed by Sites and DC placement, not by creating additional domains. This is one of the most important AD DS design principles, and it applies directly to Project 3.

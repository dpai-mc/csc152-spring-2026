Organizational Units, Domain Users & Computers

Populate and organize your Active Directory domain — design OU hierarchies, manage user and computer accounts, and delegate administrative control.

In Module 11.2 you installed Active Directory Domain Services and promoted your server to a domain controller. Now it is time to populate that domain with a logical structure, real user accounts, and computer objects — and to hand off specific administrative tasks without creating security risks.

Section 1 — What Are Organizational Units?

1 / 6

1.1 Definition

An Organizational Unit (OU) is a container object inside an Active Directory domain. You can think of it as a folder — it holds other AD objects such as user accounts, computer accounts, groups, and even other OUs (called child OUs).

OUs serve two primary purposes:

  • Organization — they give administrators a logical way to sort and find objects in a potentially enormous directory.
  • Policy application — Group Policy Objects (GPOs) are linked to OUs. Every object inside an OU inherits the policies linked to that OU (and to parent OUs above it in the hierarchy).
💡 Key Insight OUs are an administrative tool. They exist to make the administrator's job easier. They are not visible to end users, and they do not control what resources a user can access — that is the job of security groups.

1.2 OUs vs. Security Groups — The Classic Confusion

New AD administrators routinely confuse OUs and Security Groups because both can "contain" users. The distinction is fundamental:

📂 Organizational Unit (OU)

  • A container in the directory tree.
  • Used to apply GPOs and to delegate administrative control.
  • A user or computer can belong to only one OU at a time (it lives in exactly one place in the tree).
  • Not used for resource permissions.
  • Not visible in resource ACLs (Access Control Lists).
  • Example: "Put all Marketing laptops in the Marketing OU so they get the Marketing GPO."

👥 Security Group

  • A membership list — not a container in the tree.
  • Used to grant permissions to files, printers, SharePoint sites, mailboxes, etc.
  • A user can belong to many groups simultaneously.
  • Not used to apply GPOs directly.
  • Appears in resource ACLs.
  • Example: "Add Marketing users to the MarketingShare_Read group so they can read the shared drive."
⚠️ Common Mistake Creating an OU called "Finance" and then trying to use it to grant access to a Finance file share will not work. You need a Security Group for permissions. The OU and the group often share the same name — that is fine — but they serve completely different purposes.
📖 Deep Dive: Can a GPO be filtered by group membership?

Yes — and this is where the two concepts intersect. A GPO is linked to an OU (which defines scope), but it also has security filtering: by default, the Authenticated Users group is in the filter, meaning every user and computer in the OU gets the policy.

You can replace Authenticated Users with a specific Security Group to apply a GPO only to members of that group, even though the GPO is linked to a broader OU. This is called security filtering and is an advanced technique covered in Module 11.4 (Group Policy).

The takeaway: OUs determine where a GPO applies; security filtering determines who within that OU actually receives it.

Section 2 — OU Design Strategies

2 / 6

There is no single "correct" OU design. The right structure depends on your organization's size, geographic spread, administrative model, and GPO requirements. Four common strategies are described below.

Geography-Based Design

Top-level OUs represent physical locations (cities, regions, countries). Sub-OUs may then break down by department or object type within each site.

Best for: Organizations with multiple offices where IT administration is decentralized — each site has its own IT staff who manage only their location.

GPO benefit: You can link a "Site-Specific" GPO (e.g., different drive mappings or printer configurations) to a location OU.

🌐 corp.example.com ├── OU=NorthAmerica │ ├── OU=NewYork │ │ ├── OU=Users │ │ └── OU=Computers │ └── OU=LosAngeles │ ├── OU=Users │ └── OU=Computers └── OU=Europe └── OU=London ├── OU=Users └── OU=Computers

Department-Based Design

Top-level OUs represent business departments. This is intuitive for HR and managers because it mirrors the org chart.

Best for: Single-site organizations or those where IT administration is centralized and aligned with business units.

GPO benefit: Department-specific software, desktop settings, or security policies are easy to target.

🌐 corp.example.com ├── OU=Finance │ ├── OU=Users │ └── OU=Computers ├── OU=Marketing │ ├── OU=Users │ └── OU=Computers ├── OU=IT │ ├── OU=Users │ ├── OU=Computers │ └── OU=Servers ← servers get different GPOs └── OU=HR ├── OU=Users └── OU=Computers

Object-Type-Based Design

Top-level OUs separate objects by what they are: Users, Computers, Groups, Servers. This is the simplest structure and is common in small environments.

Best for: Small organizations with a single IT admin, or as a starting point that can be refined later.

GPO caveat: Applying different policies to different departments is harder — you would need to rely on security filtering rather than separate OUs.

🌐 corp.example.com ├── OU=Users ← all domain user accounts ├── OU=Computers ← all workstations ├── OU=Servers ← member servers ├── OU=Groups ← security & distribution groups └── OU=ServiceAccounts

Hybrid Design (Recommended for Most Organizations)

Combines geography and/or department at the top level with object-type sub-OUs. This gives you the GPO flexibility of department-level targeting while keeping users and computers organized separately (important because User Configuration GPOs apply to user objects and Computer Configuration GPOs apply to computer objects).

Best for: Mid-size to large organizations. This is the design you will build in Project 3.

🌐 corp.example.com ├── OU=_Admin ← leading underscore keeps it at top │ ├── OU=AdminUsers │ └── OU=ServiceAccounts ├── OU=Finance │ ├── OU=Users ← link Finance User GPO here │ └── OU=Computers ← link Finance Computer GPO here ├── OU=Marketing │ ├── OU=Users │ └── OU=Computers ├── OU=IT │ ├── OU=Users │ ├── OU=Workstations │ └── OU=Servers ├── OU=HR │ ├── OU=Users │ └── OU=Computers └── OU=Disabled ← staging area for offboarded accounts
✅ Best Practice Always create separate Users and Computers sub-OUs under each department OU. GPOs have separate User Configuration and Computer Configuration sections; keeping object types separate lets you link the right half of a GPO to the right container.

2.1 GPO Scope and OU Inheritance

Understanding how GPOs flow through the OU tree is critical before you design your structure:

  • A GPO linked to a parent OU applies to all objects in that OU and all child OUs below it — this is inheritance.
  • A GPO linked to a child OU applies only to objects in that child OU (and its children).
  • If a setting is defined at both parent and child, the child OU's GPO wins (more specific = higher precedence), unless the parent GPO is marked Enforced.
  • You can block a child OU from receiving parent GPOs using Block Inheritance — use sparingly.
⚠️ Design Implication If you put all users in a flat OU=Users container, you can only apply one set of user policies. The moment you need Finance users to get different desktop restrictions than IT users, you need separate OUs. Design for future GPO needs, not just current organizational convenience.
📖 Creating an OU with PowerShell

You can create OUs from the command line using New-ADOrganizationalUnit. This is useful for scripting an entire OU structure at once.

PowerShell
# Create a top-level OU
New-ADOrganizationalUnit -Name "Finance" -Path "DC=corp,DC=example,DC=com"

# Create a child OU inside Finance
New-ADOrganizationalUnit -Name "Users" -Path "OU=Finance,DC=corp,DC=example,DC=com"

# Protect the OU from accidental deletion (recommended)
New-ADOrganizationalUnit -Name "Computers" `
    -Path "OU=Finance,DC=corp,DC=example,DC=com" `
    -ProtectedFromAccidentalDeletion $true

The -Path parameter uses Distinguished Name (DN) syntax — a comma-separated list of components read from most-specific (left) to least-specific (right).

Section 3 — Managing Domain User Accounts

3 / 6

Domain user accounts live in Active Directory and allow users to authenticate to any domain-joined machine and access domain resources. Unlike local accounts (which only work on one machine), a domain account follows the user across the entire organization.

3.1 Creating Users via ADUC (GUI)

Active Directory Users and Computers (ADUC) is the traditional MMC snap-in for managing AD objects.

  1. Open ADUC: On your domain controller, open Server Manager → Tools → Active Directory Users and Computers, or run dsa.msc.
  2. Navigate to the target OU: Expand your domain, then expand the OU where the user should live (e.g., Finance → Users).
  3. New User Wizard: Right-click the OU → New → User.
  4. Fill in name fields: Enter First name, Last name, and the User logon name (the sAMAccountName — e.g., jsmith). Click Next.
  5. Set password: Enter and confirm a password. Choose password options (e.g., User must change password at next logon). Click Next → Finish.
  6. Edit properties: Double-click the new user to open Properties. Key tabs include: General, Account (logon hours, logon workstations, account expiry), Profile (profile path, logon script, home folder), and Member Of (group membership).
🔍 Key Account Properties Explained

3.2 Creating Users via PowerShell — New-ADUser

The New-ADUser cmdlet (part of the ActiveDirectory PowerShell module) creates a user account from the command line. This is faster for bulk work and is scriptable.

PowerShell — Basic New-ADUser
# Minimum required parameters to create an enabled account
New-ADUser `
    -Name            "Jane Smith" `
    -GivenName       "Jane" `
    -Surname         "Smith" `
    -SamAccountName  "jsmith" `
    -UserPrincipalName "jsmith@corp.example.com" `
    -Path            "OU=Users,OU=Finance,DC=corp,DC=example,DC=com" `
    -AccountPassword (ConvertTo-SecureString "P@ssw0rd!" -AsPlainText -Force) `
    -Enabled         $true `
    -ChangePasswordAtLogon $true
⚠️ Security Note Embedding plain-text passwords in scripts (as shown above for illustration) is a security risk. In production, use Read-Host -AsSecureString to prompt interactively, or retrieve passwords from a secrets vault. Never store scripts with embedded passwords in version control.
PowerShell — Additional Useful Parameters
# Add department, title, manager, and office info
Set-ADUser -Identity "jsmith" `
    -Department  "Finance" `
    -Title       "Accountant" `
    -Office      "Building A, Room 204" `
    -Manager     (Get-ADUser "mwilson")

# Add user to a security group
Add-ADGroupMember -Identity "Finance_Read" -Members "jsmith"

# Verify the account was created
Get-ADUser -Identity "jsmith" -Properties * | Select-Object Name, Enabled, DistinguishedName

3.3 Bulk User Creation via CSV Import

When onboarding many users at once (e.g., a new semester of students, a department reorganization), creating accounts one at a time is impractical. The standard approach is to import from a CSV file.

📄 Example: CSV Import Script

A CSV file (new_users.csv) might look like this:

CSV File — new_users.csv
FirstName,LastName,SamAccountName,Department,OU
Jane,Smith,jsmith,Finance,"OU=Users,OU=Finance,DC=corp,DC=example,DC=com"
Bob,Jones,bjones,Marketing,"OU=Users,OU=Marketing,DC=corp,DC=example,DC=com"
Alice,Lee,alee,HR,"OU=Users,OU=HR,DC=corp,DC=example,DC=com"

And the import script:

PowerShell — CSV Import
Import-Csv -Path "C:\Scripts\new_users.csv" | ForEach-Object {
    New-ADUser `
        -GivenName       $_.FirstName `
        -Surname         $_.LastName `
        -Name            "$($_.FirstName) $($_.LastName)" `
        -SamAccountName  $_.SamAccountName `
        -UserPrincipalName "$($_.SamAccountName)@corp.example.com" `
        -Path            $_.OU `
        -Department      $_.Department `
        -AccountPassword (ConvertTo-SecureString "Welcome1!" -AsPlainText -Force) `
        -Enabled         $true `
        -ChangePasswordAtLogon $true
    Write-Host "Created: $($_.SamAccountName)"
}

This pattern — CSV + ForEach-Object — is the foundation of enterprise user provisioning automation. In production, the CSV would come from HR systems, and the default password would be replaced with a randomly generated one sent securely to the user's manager.

3.4 Account Lifecycle — Enable, Disable, Unlock, Reset

Managing accounts over their lifetime is a daily task for AD administrators. Key operations:

PowerShell — Account Lifecycle Operations
# Disable an account (e.g., employee on leave or offboarded)
Disable-ADAccount -Identity "jsmith"

# Enable an account
Enable-ADAccount -Identity "jsmith"

# Unlock a locked-out account (too many bad password attempts)
Unlock-ADAccount -Identity "jsmith"

# Reset a password
Set-ADAccountPassword -Identity "jsmith" `
    -NewPassword (ConvertTo-SecureString "NewP@ss1!" -AsPlainText -Force) `
    -Reset

# Move a disabled account to the Disabled OU (offboarding best practice)
Move-ADObject -Identity "CN=Jane Smith,OU=Users,OU=Finance,DC=corp,DC=example,DC=com" `
    -TargetPath "OU=Disabled,DC=corp,DC=example,DC=com"
✅ Best Practice — Offboarding When an employee leaves, disable the account rather than deleting it immediately. Deleted accounts lose their SID, which can break permissions on resources the user owned. Keep disabled accounts in a dedicated OU=Disabled for 30–90 days per your organization's retention policy, then delete.

Section 4 — Computer Objects & Domain Join

4 / 6

4.1 What Is a Computer Object?

When a Windows machine joins a domain, Active Directory creates a computer object to represent it. Like a user account, a computer object has:

  • A unique Security Identifier (SID) that identifies the machine to the domain.
  • A computer account password (automatically rotated every 30 days by the machine itself — you do not manage this manually).
  • Group memberships (computers can be members of security groups, just like users).
  • An OU location that determines which Computer Configuration GPOs apply to it.
📌 Why Does This Matter? Computer objects allow the domain to authenticate the machine (not just the user) when it starts up. This enables features like machine-level GPO application before any user logs on, BitLocker key escrow to AD, and certificate auto-enrollment.

4.2 The Default Computers Container

When a computer joins a domain without pre-staging, its computer object lands in the default CN=Computers container (note: CN, not OU — it is a plain container, not an OU). This container cannot have GPOs linked to it. This means newly joined machines receive no Computer Configuration GPOs until you move them to a proper OU.

The fix: pre-stage computer accounts in the correct OU before joining, or move them immediately after joining.

4.3 Joining a Domain — GUI Method

  1. Prerequisites: The machine must be able to reach the domain controller (DNS must resolve the domain name). Set the machine's preferred DNS server to the DC's IP address.
  2. Open System Properties: Right-click This PC → Properties → Advanced system settings → Computer Name tab → Change.
  3. Select Domain and enter the domain name (e.g., corp.example.com). Click OK.
  4. Enter credentials for a domain account with permission to join computers (by default, any domain user can join up to 10 computers; Domain Admins have no limit).
  5. Click OK through the welcome dialog, then restart the machine.
  6. After restart, log on with a domain account: CORP\jsmith or jsmith@corp.example.com.

4.4 Joining a Domain — PowerShell Method

PowerShell — Add-Computer
# Join the domain (will prompt for credentials)
Add-Computer -DomainName "corp.example.com" -Restart

# Join AND place the computer in a specific OU
Add-Computer -DomainName "corp.example.com" `
    -OUPath    "OU=Computers,OU=Finance,DC=corp,DC=example,DC=com" `
    -Credential (Get-Credential) `
    -Restart

# Join using a pre-staged computer account (same cmdlet, AD finds the account)
Add-Computer -DomainName "corp.example.com" `
    -Credential (Get-Credential) `
    -Restart
💡 PowerShell 7 Note Add-Computer is a Windows PowerShell (5.1) cmdlet. In PowerShell 7+, use Add-Computer from the Microsoft.PowerShell.Management module, or use the newer djoin.exe offline join tool for complex scenarios.

4.5 Pre-Staging Computer Accounts

Pre-staging means creating the computer object in AD before the machine joins the domain. This ensures the computer lands in the correct OU automatically, without requiring an admin to move it afterward.

PowerShell — Pre-Stage a Computer Account
# Create the computer account in the Finance Computers OU
New-ADComputer -Name "FIN-PC-001" `
    -Path    "OU=Computers,OU=Finance,DC=corp,DC=example,DC=com" `
    -Enabled $true

# Verify the pre-staged object exists
Get-ADComputer -Identity "FIN-PC-001"

When the physical machine named FIN-PC-001 joins the domain, AD finds the pre-staged account and places it in the correct OU automatically.

4.6 Verifying a Domain Join

After a machine restarts and you log in with a domain account, confirm the join succeeded:

Command Prompt / PowerShell — Verification
# Show current user and all group memberships (look for domain groups)
whoami /all

# Show domain membership
systeminfo | findstr /i "domain"

# PowerShell: check the computer's domain
(Get-WmiObject Win32_ComputerSystem).Domain

# PowerShell: find the computer object in AD (run from DC or admin workstation)
Get-ADComputer -Identity "FIN-PC-001" -Properties * | Select-Object Name, DistinguishedName, Enabled

🎯 Real-World Scenario

A new Finance department workstation arrives. The IT technician pre-stages FIN-PC-042 in OU=Computers,OU=Finance before the machine leaves the imaging room. When the machine is deployed and joins the domain, it automatically receives the Finance Computer GPO (which, for example, maps the Finance shared drive and sets the desktop wallpaper). No manual OU move is needed. This is the workflow at scale.

Section 5 — Delegation of Control

5 / 6

5.1 What Is Delegation of Control?

Delegation of Control is the mechanism in Active Directory that allows you to grant specific administrative permissions over an OU to a user or group — without making them a Domain Admin.

This is a direct application of the principle of least privilege: give people only the permissions they need to do their job, and nothing more.

🚫 The Domain Admin Anti-Pattern A common mistake is making the Help Desk team Domain Admins so they can reset passwords. Domain Admins can do anything in the domain — including deleting OUs, modifying GPOs, or adding themselves to privileged groups. This is a massive security risk. Delegation solves this: give Help Desk only the ability to reset passwords in only the Staff OU.

5.2 The Delegation of Control Wizard

ADUC includes a built-in wizard for common delegation tasks. Here is a walkthrough of the Help Desk password-reset scenario:

  1. Open ADUC and navigate to the OU you want to delegate (e.g., OU=Users,OU=Staff).
  2. Right-click the OU → Delegate Control… — the Delegation of Control Wizard opens.
  3. Click Next on the welcome screen.
  4. Add users or groups: Click Add and search for the group you want to delegate to (e.g., HelpDesk_Staff). Click OK → Next.
  5. Select tasks to delegate: Choose from the common task list. For this scenario, select "Reset user passwords and force password change at next logon". Click Next.
  6. Review the summary and click Finish.
  7. Verify: Log in as a Help Desk user and confirm you can reset a password in that OU but cannot, for example, create new users or modify GPOs.
🔍 Common Delegation Tasks Available in the Wizard
TaskTypical Delegate
Reset user passwords and force password change at next logonHelp Desk
Read all user informationHR staff, managers
Create, delete, and manage user accountsDepartmental IT coordinator
Modify the membership of a groupGroup owners / team leads
Join a computer to the domainDesktop technicians
Manage Group Policy linksSenior IT staff (not Help Desk)
📖 Custom Delegation (Beyond the Wizard)

The wizard covers the most common tasks but is limited. For custom permissions, you can edit the OU's Access Control List (ACL) directly:

  1. In ADUC, go to View → Advanced Features (enables the Security tab on objects).
  2. Right-click the OU → Properties → Security tab → Advanced.
  3. Click Add to add a principal (user or group), then select the specific permissions (e.g., allow Write lockoutTime to let Help Desk unlock accounts).
  4. Set the scope: This object only, Descendant User objects, etc.

This gives you granular control but requires a solid understanding of AD attribute permissions. Document any custom delegations carefully — they are not visible in the wizard and can be difficult to audit later.

5.3 Auditing Delegations

Over time, delegations can accumulate and become difficult to track. Use the following to audit what has been delegated on an OU:

PowerShell — View OU ACL (Delegations)
# View the Access Control List on an OU
(Get-Acl -Path "AD:OU=Users,OU=Staff,DC=corp,DC=example,DC=com").Access |
    Where-Object { $_.IdentityReference -notlike "NT AUTHORITY\*" } |
    Select-Object IdentityReference, ActiveDirectoryRights, AccessControlType
✅ Best Practice Always delegate to security groups, never to individual user accounts. When a Help Desk employee leaves, you simply remove them from the HelpDesk_Staff group — you do not need to touch any OU ACLs. This is the same principle as using groups for resource permissions.

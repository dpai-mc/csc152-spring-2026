💡 Part A – Common Pitfalls & Tips (click to expand)
  • ⚠Static IP required before promotion: Your Windows Server must have a static IP address configured before you promote it to a Domain Controller. If the server uses DHCP, the DC's own DNS records may break after a lease renewal. Set a static IP (e.g., 192.168.x.10) and point the DNS server address to 127.0.0.1 or the server's own static IP.
  • ⚠DSRM password is not your admin password: The Directory Services Restore Mode password is a separate, emergency-use credential. Store it securely — you will not need it during normal operations, but losing it can prevent disaster recovery. Do not leave it blank.
  • ⚠SRV records take a moment to register: Immediately after reboot, DNS SRV records may not yet appear. Wait 2–3 minutes, then run ipconfig /registerdns followed by nslookup -type=SRV _ldap._tcp.techstart.local. If records are still missing, check the DNS Server service is running and the forward lookup zone exists.
  • ⚠dcdiag warnings vs. errors: dcdiag often reports warnings that are not critical (e.g., missing reverse lookup zone). Focus on resolving any lines marked FAIL — warnings can generally be ignored in a lab environment.

Part B – Organizational Unit Structure

6 Points

Design and implement the OU hierarchy for TechStart Solutions. A well-designed OU structure makes Group Policy application, delegation of control, and object management far more efficient. Follow the required structure below and document your design rationale.

1Required OU Hierarchy

Create the following OU structure exactly as shown. The tree represents the techstart.local domain:

techstart.local └── TechStart (top-level OU) ├── IT │ ├── Users │ ├── Computers │ └── Groups ├── Sales │ ├── Users │ ├── Computers │ └── Groups ├── Engineering │ ├── Users │ ├── Computers │ └── Groups ├── Service Accounts └── Servers
2Create the OU Structure

Use Active Directory Users and Computers (ADUC) (Server Manager → Tools → Active Directory Users and Computers) or PowerShell to create the hierarchy.

ADUC GUI Method

  1. In ADUC, right-click the techstart.local domain → New → Organizational Unit
  2. Name it TechStart; leave "Protect container from accidental deletion" checked
  3. Right-click TechStart → New → Organizational Unit → create IT, Sales, Engineering, Service Accounts, and Servers
  4. Inside each department OU, create three child OUs: Users, Computers, and Groups

PowerShell Method

# Create top-level TechStart OU
New-ADOrganizationalUnit -Name "TechStart" `
    -Path "DC=techstart,DC=local" `
    -ProtectedFromAccidentalDeletion $true

# Create department OUs under TechStart
$depts = "IT","Sales","Engineering"
foreach ($dept in $depts) {
    New-ADOrganizationalUnit -Name $dept `
        -Path "OU=TechStart,DC=techstart,DC=local"

    # Create child OUs for each department
    foreach ($child in "Users","Computers","Groups") {
        New-ADOrganizationalUnit -Name $child `
            -Path "OU=$dept,OU=TechStart,DC=techstart,DC=local"
    }
}

# Create Service Accounts and Servers OUs
New-ADOrganizationalUnit -Name "Service Accounts" `
    -Path "OU=TechStart,DC=techstart,DC=local"
New-ADOrganizationalUnit -Name "Servers" `
    -Path "OU=TechStart,DC=techstart,DC=local"

# Verify all OUs were created
Get-ADOrganizationalUnit -Filter "*" | Select-Object Name,DistinguishedName | Sort-Object DistinguishedName

💡 Part B – Common Pitfalls & Tips (click to expand)
  • ⚠OUs vs. Groups — do not confuse them: OUs are containers for organizing AD objects and applying Group Policy. Security Groups are used for assigning permissions to resources. Never use an OU as a substitute for a security group, and never use a group as a substitute for an OU.
  • ⚠Accidental deletion protection: When creating OUs via ADUC, the "Protect from accidental deletion" checkbox is enabled by default — leave it on. If you need to delete an OU later, you must first uncheck this option in ADUC's Advanced Features view (View → Advanced Features → Properties → Object tab).
  • ⚠Distinguished Name path syntax in PowerShell: The -Path parameter in New-ADOrganizationalUnit must use the correct LDAP distinguished name format. A common mistake is reversing the order — it should be OU=TechStart,DC=techstart,DC=local, not DC=local,DC=techstart,OU=TechStart. The path reads from most-specific (left) to least-specific (right).

2Create Security Groups & Implement AGDLP Nesting
📖 AGDLP Explained

AGDLP is a Microsoft best-practice group nesting strategy:

  • Accounts → added to Global groups → added to Domain Local groups → assigned Permissions
  • Global Groups collect users by job function (e.g., all Sales users). They can contain accounts from the same domain.
  • Domain Local Groups are assigned to resources (e.g., a shared folder). They can contain Global groups from any trusted domain, making permission management scalable.

Example: User mjohnson → Global Group GG-Sales-Users → Domain Local Group DL-SalesShare-Read → Read permission on \\DC01\SalesShare

Create the following security groups (or equivalent groups of your own design) and implement AGDLP nesting:

Group NameGroup ScopeGroup TypeOU PathPurpose
GG-IT-AdminsGlobalSecurityIT\GroupsIT department administrators
GG-Sales-UsersGlobalSecuritySales\GroupsAll Sales department users
GG-Engineering-UsersGlobalSecurityEngineering\GroupsAll Engineering department users
DL-SalesShare-ReadDomain LocalSecuritySales\GroupsRead access to Sales shared folder
DL-EngShare-ModifyDomain LocalSecurityEngineering\GroupsModify access to Engineering share
 
# Create Global security groups
New-ADGroup -Name "GG-IT-Admins" `
    -GroupScope Global `
    -GroupCategory Security `
    -Path "OU=Groups,OU=IT,OU=TechStart,DC=techstart,DC=local"

New-ADGroup -Name "GG-Sales-Users" `
    -GroupScope Global `
    -GroupCategory Security `
    -Path "OU=Groups,OU=Sales,OU=TechStart,DC=techstart,DC=local"

New-ADGroup -Name "GG-Engineering-Users" `
    -GroupScope Global `
    -GroupCategory Security `
    -Path "OU=Groups,OU=Engineering,OU=TechStart,DC=techstart,DC=local"

# Create Domain Local security groups
New-ADGroup -Name "DL-SalesShare-Read" `
    -GroupScope DomainLocal `
    -GroupCategory Security `
    -Path "OU=Groups,OU=Sales,OU=TechStart,DC=techstart,DC=local"

New-ADGroup -Name "DL-EngShare-Modify" `
    -GroupScope DomainLocal `
    -GroupCategory Security `
    -Path "OU=Groups,OU=Engineering,OU=TechStart,DC=techstart,DC=local"

# Add users to Global groups (A → G)
Add-ADGroupMember -Identity "GG-IT-Admins" `
    -Members "jsmith","bwilson","alopez"

Add-ADGroupMember -Identity "GG-Sales-Users" `
    -Members "mjohnson","sdavis","cthomas","lmartin"

Add-ADGroupMember -Identity "GG-Engineering-Users" `
    -Members "dwilliams","pbrown","rgarcia"

# Nest Global groups into Domain Local groups (G → DL)
Add-ADGroupMember -Identity "DL-SalesShare-Read" `
    -Members "GG-Sales-Users"

Add-ADGroupMember -Identity "DL-EngShare-Modify" `
    -Members "GG-Engineering-Users"

# Verify group memberships
Get-ADGroupMember -Identity "GG-Sales-Users" | Select-Object Name,objectClass
Get-ADGroupMember -Identity "DL-SalesShare-Read" | Select-Object Name,objectClass,GroupScope

💡 Part C – Common Pitfalls & Tips (click to expand)
  • ⚠Password complexity is enforced immediately: The Default Domain Policy enforces password complexity by default. Passwords like Password1 may be rejected. Use a password that meets the default policy: at least 7 characters, uppercase, lowercase, digit, and/or special character — e.g., Welcome1!.
  • ⚠Group scope limitations: You cannot add a Domain Local group as a member of a Global group — this is by design. The AGDLP flow is one-directional: Accounts → Global → Domain Local → Permissions. Trying to reverse this nesting will produce an error in PowerShell and ADUC.
  • ⚠SamAccountName vs. UserPrincipalName: The SamAccountName (e.g., jsmith) is the legacy logon name used for domain authentication (DOMAIN\jsmith). The UserPrincipalName (e.g., jsmith@techstart.local) is the modern UPN-format logon. Both should be set when creating users — New-ADUser requires -SamAccountName but -UserPrincipalName is a best practice to include.

1GPO 1 – Domain Password & Account Lockout Policy

Create a GPO named TechStart-Password-Policy and link it at the domain level (not to an OU). Password policies that apply to domain user accounts must be linked at the domain root to be effective.

⚠ Password Policy Placement

Password policies (Account Policies → Password Policy and Account Lockout Policy) only take effect for domain user accounts when linked at the domain root level. A password policy GPO linked to an OU will apply to local accounts on computers in that OU, but will not override the domain-level password policy for domain users. This is a very common misconfiguration in the field.

💡 Part D – Common Pitfalls & Tips (click to expand)
  • ⚠Password policy must be linked at the domain root: This is the single most common GPO mistake students make. If you link a password policy GPO to the IT OU, it will configure the local password policy on computers in that OU — but domain user accounts will still use whatever policy is linked at the domain root (usually the Default Domain Policy). Always link password and account lockout policies at the domain level.
  • ⚠gpresult may show "Access Denied" on client VMs: Run gpresult /r in an elevated (Administrator) PowerShell or Command Prompt session. Non-elevated sessions may show incomplete results or access errors, especially for the Computer Settings section.
  • ⚠Policy changes may not apply immediately: After editing a GPO, run gpupdate /force on the target computer to force immediate policy refresh. Without this, changes may take up to 90 minutes (the default background refresh interval) to apply. Always run gpupdate /force before running gpresult to verify your changes.
  • ⚠LSDOU and conflicting settings: If two GPOs define the same setting with different values, the GPO applied last wins (OU-linked GPOs override domain-linked GPOs for most settings). The exception is when a GPO is set to Enforced — enforced GPOs always win regardless of where they are linked in the hierarchy.

Why Monitor Directory Service Events?

In Modules 11.1 through 11.4 you built and managed an Active Directory Domain Services (AD DS) environment — promoting a Domain Controller, creating OUs, users, and groups, and configuring Group Policy. That infrastructure now holds the keys to your organization: user accounts, authentication, and access rights. That makes it a prime target.

Monitoring the Windows Security event log on your Domain Controllers gives you visibility into three critical threat categories:

🔒 Unauthorized Access

Repeated failed logon attempts (brute-force), logins at unusual hours, or successful logins from unexpected source IPs can all signal an intrusion attempt or a compromised credential.

🔐 Account Lockouts

A sudden spike in lockout events may indicate a password-spray attack, a misconfigured service account, or a user whose credentials were stolen and are being actively tested.

⬆ Privilege Escalation

Adding an account to Domain Admins or other privileged groups — especially outside of a change-control window — is one of the most dangerous actions in an AD environment and must be tracked.

Without centralized log collection and alerting, these events sit silently in individual server logs, often overwritten before anyone reviews them. Splunk solves this by aggregating events in real time and enabling search, dashboards, and automated alerts — skills you will develop starting in Week 13.

Compliance Note Many regulatory frameworks — including PCI-DSS, HIPAA, and SOC 2 — explicitly require audit logging of authentication events, privileged account changes, and account lockouts. The Event IDs you will study in the next section directly satisfy those requirements.

Key Windows Security Event IDs for AD DS

Windows logs security events in the Security channel of the Windows Event Log. On a Domain Controller, this channel records authentication, account management, and group membership changes for the entire domain. The table below covers the seven Event IDs you need to know for this course. Click Details under any row for additional monitoring context.

Event ID Name Category Brief Description
4624 Successful Logon Authentication A user or service successfully authenticated to the system.
Why it matters: Baseline logon activity. Look for Logon Type 3 (network) and Type 10 (RemoteInteractive/RDP) events from unexpected source IPs or at off-hours. The event includes the account name, workstation name, source IP, and logon type — all searchable in Splunk.
4625 Failed Logon Authentication An authentication attempt failed.
Why it matters: A single failure is normal; dozens of failures against one account in a short window suggest a brute-force or password-spray attack. Sub-status codes within the event identify why the logon failed (wrong password, account disabled, expired, etc.), making triage much faster.
4720 User Account Created Account Management A new user account was created in the domain.
Why it matters: Unauthorized account creation is a classic persistence technique — attackers create a backdoor account to maintain access. This event records the new account name and the account that performed the creation, enabling rapid attribution.
4740 User Account Locked Out Account Management An account was locked out due to too many failed logon attempts.
Why it matters: Lockout events are generated on the Domain Controller that processed the final bad password. The event includes the caller computer name — the machine where the bad passwords originated — which is the starting point for investigating whether this is a user mistake, a misconfigured service, or an attack.
4728 Member Added to Global Security Group Group Management A member was added to a global security-enabled group.
Why it matters: Global security groups commonly control resource access across the domain. Any unexpected addition — especially to privileged groups like Domain Admins — must be investigated immediately.
4732 Member Added to Local Security Group Group Management A member was added to a local security-enabled group.
Why it matters: On Domain Controllers, local groups like Administrators and Remote Desktop Users grant elevated access. Adding an account to these groups outside a change window is a significant privilege escalation indicator.
4756 Member Added to Universal Security Group Group Management A member was added to a universal security-enabled group.
Why it matters: Universal groups are replicated to the Global Catalog and can span multiple domains in a forest. Unauthorized additions to universal groups can grant cross-domain access, making this event especially important in multi-domain environments.
Audit Policy Prerequisite These events are only generated if the appropriate audit policies are enabled. In a default Windows Server installation, not all of them are active. In Week 13 you will verify audit policy settings on your Domain Controller as part of the Splunk lab setup.

How Splunk Collects Windows Event Logs

You do not need to install anything this week. This section gives you a conceptual preview of the data pipeline you will configure in the Week 13 lab, so the Splunk Fundamentals 1 course makes sense in context.

The Two-Component Architecture

Splunk's Windows log collection relies on two software components working together:

Windows Event Log Security channel
on Domain Controller
→
Universal Forwarder Lightweight agent
on the DC
→
Splunk Enterprise Indexer + Search Head
central server

Windows Event Log (Security channel): The operating system writes security events — logons, account changes, group membership changes — to this channel in real time. On a Domain Controller, this includes domain-wide authentication events. Events are stored locally in a circular buffer; older events are overwritten once the buffer fills.

Splunk Universal Forwarder: A small, low-overhead agent installed on the Domain Controller. It monitors the Windows Event Log channels you configure, reads new events as they are written, and forwards them — in near real time — to Splunk Enterprise over an encrypted TCP connection (default port 9997). The forwarder does not index or search data; it only collects and ships.

Splunk Enterprise (Indexer + Search Head): The central server receives the forwarded events, indexes them for fast retrieval, and provides the web-based interface where you write searches (SPL — Splunk Processing Language), build dashboards, and configure alerts. In the Week 13 lab your Splunk Enterprise instance will be running in a separate VM on your VirtualBox network, receiving events from the DC you built this week.

Week 13 Preview In the Week 13 lab you will install the Universal Forwarder on your Domain Controller VM, configure it to forward the Security event channel to your Splunk Enterprise VM, and then write your first SPL queries to surface Event ID 4625 (failed logons) and 4728/4732/4756 (group membership changes) in real time.

What Is Splunk Fundamentals 1?

Splunk Fundamentals 1 is a free, self-paced online course offered through Splunk's education platform. It is the official starting point for Splunk's certification track and is widely recognized in industry. The course covers:

  • Navigating the Splunk web interface and understanding the data pipeline

  • Working with fields, transforming commands, and statistical functions
  • Building basic reports, charts, and dashboards
  • Saving searches and setting up simple alerts

The skills taught in Fundamentals 1 map directly to what you will do in the Week 13 and Week 14 labs — searching for AD DS event IDs, building a logon-failure dashboard, and configuring a lockout alert.

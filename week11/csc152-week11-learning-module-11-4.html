What Are Group Policy Objects?

The Problem GPOs Solve

Imagine you are the administrator for a company with 400 Windows computers spread across three departments. Without a centralized management tool, enforcing a consistent security configuration would require you to visit—or remotely connect to—every single machine and manually apply the same settings. When a new security requirement arrives (say, passwords must now be at least 14 characters), you would need to repeat that process on all 400 machines. This approach does not scale and is highly error-prone.

Group Policy Objects (GPOs) solve this problem by letting you define configuration settings in one place—on the Domain Controller—and automatically push those settings to all targeted users and computers in the domain. GPOs are the primary mechanism for centralized configuration management in an Active Directory Domain Services (AD DS) environment.

What Can GPOs Control?

CategoryExample Settings
SecurityPassword length, complexity requirements, account lockout thresholds, audit policies
Desktop EnvironmentDisable Control Panel, set a standard wallpaper, restrict access to specific drives
Software DeploymentAutomatically install or remove applications on domain computers
ScriptsRun logon/logoff scripts, startup/shutdown scripts
Network SettingsMap network drives, configure proxy settings, define Windows Firewall rules
Windows UpdateRedirect clients to a WSUS server, set maintenance windows

Where Do GPOs Live?

GPOs are stored in two locations that work together:

  • Group Policy Container (GPC): An object in Active Directory (in the System\Policies container) that stores GPO metadata such as version numbers and status.
  • Group Policy Template (GPT): A folder structure in the SYSVOL share on every Domain Controller (\\domain\SYSVOL\domain\Policies\) that stores the actual policy settings files, scripts, and templates.

Both components must be consistent. When they fall out of sync, GPOs can fail to apply—a common troubleshooting scenario covered in Section 7.

💡

Analogy: Think of a GPO as a rulebook that the Domain Controller hands to every user and computer in its jurisdiction. The rulebook says: "Here is how your computer must be configured and what you are and are not allowed to do." The domain controller does not need to be present 24/7—the client caches the policy and enforces it locally.

Two Default GPOs You Should Know

Every new AD DS domain is created with two built-in GPOs:

  • Default Domain Policy: Linked to the domain root. Contains the domain-wide password and account lockout policies. Best practice: configure password policy here and only here.
  • Default Domain Controllers Policy: Linked to the Domain Controllers OU. Configures audit and user rights assignment for domain controllers.
⚠️

Important: Microsoft recommends that you do not modify the Default Domain Policy or Default Domain Controllers Policy directly. Instead, create new GPOs for your custom settings and link them at the appropriate level. This preserves the defaults as a known-good baseline.

2

LSDOU Processing Order

The LSDOU Mnemonic

When a user logs on or a computer starts up, Windows applies Group Policy in a specific, fixed order. This order is remembered using the acronym LSDOU:

L
Local Policy Applied first. Stored on the local machine. Lowest precedence — easily overridden by any domain policy.
↓
S
Site Policy AD site-level GPOs. Useful for location-specific settings (e.g., different printer mappings in each office).
↓
D
Domain Policy Domain-wide GPOs. The Default Domain Policy lives here. Password policies must be set at this level to take effect.
↓
OU
Organizational Unit Policy OU-level GPOs. Applied last. WINS in conflicts (unless Enforced). Nested OUs process parent-first, child-last.

Processing order: Local → Site → Domain → OU  |  Last applied wins in a conflict

Why Order Matters: The "Last Applied Wins" Rule

Because GPOs are applied sequentially, a setting configured in an OU-level GPO will override the same setting configured in a Domain-level GPO—unless special inheritance controls are in place (see Section 3). This is the "last applied wins" rule.

Example: Password Policy Conflict

Suppose you configure a Domain GPO with a minimum password length of 8 characters. You then create an OU GPO for the IT OU with a minimum password length of 14 characters.

Result: Users in the IT OU get a 14-character minimum because the OU GPO is applied last. Users in all other OUs get the 8-character minimum from the Domain GPO.

⚠️

Exception: Password policies are special. The effective password policy for domain user accounts is always taken from the Domain level, regardless of OU GPOs—unless you use Fine-Grained Password Policies (an advanced topic). OU-level password policies only apply to local accounts on computers in that OU. This is a common source of confusion and is covered in the Troubleshooting section.

Example: Nested OUs — Parent Before Child

Consider this OU hierarchy:

corp.local (Domain)
  └── Staff (OU)
        └── Finance (Child OU)

If GPOs are linked at both Staff and Finance, the processing order for a user in the Finance OU is:

  1. Local Policy
  2. Site Policy (if any)
  3. Domain Policy
  4. Staff OU Policy (parent OU — applied first)
  5. Finance OU Policy (child OU — applied last, wins conflicts)

Multiple GPOs at the Same Level

You can link more than one GPO to the same container. When multiple GPOs are linked at the same level, they are processed in link order—the GPO with the lowest link order number is applied last (and therefore wins). You can adjust link order in the Group Policy Management Console.

🧠

Memory tip: Let Students Do Overrides — Local, Site, Domain, OU. The further down the hierarchy a GPO is linked, the more specific and higher-priority it becomes.

3

GPO Inheritance, Block & Enforce

How Inheritance Works by Default

By default, GPO settings flow down the OU hierarchy. A GPO linked to a parent OU automatically applies to all child OUs and their objects unless something interrupts that flow. This is called GPO inheritance.

For example, a GPO linked at the domain root applies to every user and computer in the domain—including those nested in child OUs—unless inheritance is blocked.

Controlling Inheritance: Two Mechanisms

🚫 Block Inheritance

Set on an OU. Prevents GPOs from parent containers (domain, parent OUs) from flowing into this OU.

Use case: A subsidiary division that manages its own IT policy and does not want corporate-level GPOs interfering.

Risk: Overuse of Block Inheritance makes policy management harder to reason about. Use sparingly.

🔒 Enforced (No Override)

Set on a GPO link. Forces the GPO's settings to apply regardless of any Block Inheritance settings on child OUs. Enforced GPOs always win in conflicts.

Use case: Corporate security baseline that must apply everywhere, even in OUs that have Block Inheritance set.

Note: Enforced GPOs are processed after all other GPOs, so they always take precedence.

⚠️

Interaction between Block Inheritance and Enforced: An Enforced GPO overrides Block Inheritance. If a parent OU has an Enforced GPO and a child OU has Block Inheritance set, the Enforced GPO still applies to the child OU. This is intentional—it allows administrators to guarantee that critical security policies cannot be blocked.

User Configuration vs. Computer Configuration

Every GPO is divided into two main sections. Understanding which section to use for a given setting is essential:

🖥️ Computer Configuration

Settings applied to the computer object when the machine starts up, regardless of who logs on.

  • Password and account lockout policies
  • Windows Firewall rules
  • Startup/shutdown scripts
  • Software installation (machine-wide)
  • Security options (rename Administrator account, etc.)
  • Windows Update / WSUS configuration

👤 User Configuration

Settings applied to the user object at logon, following the user across any computer they log into.

  • Folder redirection
  • Logon/logoff scripts
  • Desktop restrictions (disable Control Panel, etc.)
  • Software installation (per-user)
  • Internet Explorer / Edge settings
  • Drive mappings
💡

Key rule: Password policies (minimum length, complexity, lockout) must be configured under Computer Configuration, not User Configuration. They apply to the computer's local accounts and — when set at the domain level — to all domain user accounts. A password policy set under User Configuration has no effect on domain account passwords.

Advanced: GPO Loopback Processing

By default, User Configuration settings follow the user's OU, not the computer's OU. Loopback Processing is an advanced setting that changes this behavior so that the User Configuration settings of GPOs linked to the computer's OU are applied to users who log on to that computer.

This is commonly used for kiosk machines, lab computers, or terminal servers where you want the desktop environment to be controlled by where the computer is, not where the user is.

Loopback has two modes: Replace (only the computer's User Configuration applies) and Merge (both the user's and computer's User Configuration apply, with the computer's winning conflicts).

Creating & Linking GPOs — GPMC Walkthrough

Opening the Group Policy Management Console

The Group Policy Management Console (GPMC) is the central tool for creating, editing, linking, and managing GPOs. It is available on Domain Controllers and on any management workstation with the Remote Server Administration Tools (RSAT) installed.

Three ways to open GPMC:

  1. Server Manager: Open Server Manager → click Tools in the top-right menu → select Group Policy Management.

  2. Run dialog: Press Win + R, type gpmc.msc, and press Enter.

  3. PowerShell:

PowerShell
gpmc.msc

Navigating the GPMC Console Tree

The left pane of GPMC shows a tree structure. Key nodes to know:

NodePurpose
Forest: corp.localTop-level container for the entire forest
Domains → corp.localYour domain; GPOs linked here apply domain-wide
Group Policy ObjectsCentral repository of all GPOs — create and store GPOs here
SitesLink GPOs at the AD site level
Group Policy ModelingSimulate what policies would apply to a user/computer
Group Policy ResultsView what policies actually applied to a specific user/computer
📋

When you click on a GPO in the console tree, the right pane shows tabs: Scope (where it's linked), Details (status, GUID), Settings (a readable summary of configured settings), Delegation (who can edit/apply it), and Status (replication health).

Creating and Linking a GPO to an OU

There are two approaches: create the GPO in the Group Policy Objects container first and then link it, or right-click an OU and create-and-link in one step. Both are valid; the two-step approach is better practice for reusable GPOs.

Method A: Create First, Then Link (Recommended)

  1. In GPMC, expand Forest → Domains → corp.local → Group Policy Objects.

  2. Right-click Group Policy Objects → New.

  3. Enter a descriptive name (e.g., Password Policy – Domain) → click OK.

  4. The new GPO appears in the list. Right-click it → Edit to open the Group Policy Management Editor and configure settings (see the Password Policy tab).

  5. After editing, close the editor. Now navigate to the OU (or domain) where you want to link it.

  6. Right-click the target OU → Link an Existing GPO.

  7. Select your GPO from the list → click OK. The GPO now appears under the OU with a link icon.

Method B: Create and Link in One Step

  1. Right-click the target OU (or domain) → Create a GPO in this domain, and Link it here…

  2. Enter the GPO name → click OK.

  3. Right-click the new GPO link → Edit to configure settings.

💡

Naming convention tip: Use a consistent naming scheme such as [Setting Type] – [Scope] (e.g., Password Policy – Domain, Desktop Restrictions – Staff OU). This makes it immediately clear what a GPO does and where it applies without opening it.

Configuring a Password Policy GPO

Password and account lockout policies must be set at the domain level (linked to the domain root, not an OU) to affect domain user accounts. Create a GPO named Password Policy – Domain and link it to the domain root.

Navigate to Password Policy Settings

In the Group Policy Management Editor, follow this path:

Remember: Double-click each setting to open it, check "Define this policy setting", set your value, and click OK. A setting that is not defined in a GPO is simply not configured by that GPO — it does not reset the setting to a default value.

Configuring a Desktop Restriction GPO

Create a GPO named Desktop Restrictions – Staff and link it to the Staff OU. This GPO will prevent users in the Staff OU from accessing the Control Panel and Settings app.

Disable Control Panel Access

Navigate to:

User Configuration › Policies › Administrative Templates › Control Panel

Double-click "Prohibit access to Control Panel and PC Settings" → set to Enabled → click OK.

📋

Desktop restriction settings are under User Configuration because they follow the user, not the computer. If a Staff OU user logs onto a computer in the IT OU, these restrictions still apply to that user session.

Managing GPOs with PowerShell

The GroupPolicy PowerShell module provides cmdlets for creating, linking, configuring, and reporting on GPOs — useful for automation and scripted deployments.

View All GPOs in the Domain

PowerShell
# List all GPOs with status and creation time
Get-GPO -All | Select-Object DisplayName, GpoStatus, CreationTime | Format-Table -AutoSize

Create a New GPO

PowerShell
# Create a new GPO (stored in Group Policy Objects container)
New-GPO -Name "Password Policy - Domain" -Comment "Domain-wide password policy"

Link a GPO to an OU or Domain

PowerShell
# Link GPO to the domain root
New-GPLink -Name "Password Policy - Domain" -Target "DC=corp,DC=local"

# Link GPO to a specific OU
New-GPLink -Name "Desktop Restrictions - Staff" -Target "OU=Staff,DC=corp,DC=local"

Generate an HTML Report of a GPO's Settings

PowerShell
# Generate a readable HTML report of the Default Domain Policy
Get-GPO -Name "Default Domain Policy" | `
    Get-GPOReport -ReportType HTML -Path "C:\Reports\DefaultDomainPolicy.html"

# Open the report in the default browser
Start-Process "C:\Reports\DefaultDomainPolicy.html"

View GPO Links on an OU

PowerShell
# Show all GPOs linked to a specific OU
Get-GPInheritance -Target "OU=Staff,DC=corp,DC=local" | Select-Object -ExpandProperty GpoLinks
💡

Note that PowerShell can create and link GPOs but configuring individual policy settings (like specific registry-based Administrative Template settings) is more complex via PowerShell and is typically done through the GUI editor or via direct registry modification. For most production environments, GPO creation and linking is scripted; individual settings are configured in the editor.

Verifying Group Policy Application

After creating and linking a GPO, you need to verify that it is actually being applied to the intended users and computers. Three command-line tools handle this workflow: gpupdate, gpresult, and the GPMC's Group Policy Results wizard.

Tool 1: gpupdate /force — Force Immediate Refresh

By default, Group Policy refreshes automatically every 90 minutes (with a random 30-minute offset) for computers, and at logon/startup. The gpupdate /force command bypasses this wait and immediately re-applies all GPOs.

Command Prompt or PowerShell (run on the client machine)
# Force an immediate refresh of all Group Policies
gpupdate /force

# Force refresh of Computer Configuration only
gpupdate /force /target:computer

# Force refresh of User Configuration only
gpupdate /force /target:user
📋

After running gpupdate /force, you will see output like:
Updating policy...
Computer Policy update has completed successfully.
User Policy update has completed successfully.
A logoff or restart may be required for some settings (particularly those in the Computer Configuration) to take full effect.

Tool 2: gpresult /r — Summary of Applied GPOs

gpresult queries the Resultant Set of Policy (RSoP) — the effective set of policies that actually applied to a user or computer after processing the LSDOU chain.

Command Prompt or PowerShell (run on the target machine)
# Summary of applied GPOs for the current user and computer
gpresult /r

# Summary for a specific user (run as admin)
gpresult /r /user CORP\jsmith

# Verbose output — shows all settings, not just GPO names
gpresult /v

The /r output is divided into two sections:

  • COMPUTER SETTINGS: Lists the domain, site, and all GPOs applied at the computer level, in processing order.
  • USER SETTINGS: Lists GPOs applied at the user level.

Look for the "Applied Group Policy Objects" list and the "The following GPOs were not applied" section (which tells you why certain GPOs were filtered or denied).

Tool 3: gpresult /h report.html — HTML Results Report

For a more detailed and readable output, generate an HTML report. This is especially useful when troubleshooting complex policy interactions.

Command Prompt or PowerShell (run as Administrator)
# Generate an HTML GPO results report for the current user/computer
gpresult /h C:\Reports\gpo-report.html

# Open the report immediately after generating it
gpresult /h C:\Reports\gpo-report.html && start C:\Reports\gpo-report.html

# Generate report for a specific user
gpresult /h C:\Reports\gpo-report-jsmith.html /user CORP\jsmith

The HTML report includes:

  • Computer and user summary information
  • All applied and denied GPOs with reasons
  • Winning and losing GPO settings for each policy area
  • Security group memberships (which affect GPO filtering)

Tool 4: GPMC Group Policy Results Wizard

The GPMC includes a graphical Group Policy Results wizard that performs the same function as gpresult but with a GUI interface. It can query a remote computer across the network.

  1. In GPMC, right-click Group Policy Results → Group Policy Results Wizard.

  2. Select Another computer and enter the target computer name, or use This computer.

  3. Select the user whose policy you want to review.

  4. Review the Summary, Settings, and Policy Events tabs in the results pane.

💡

Quick reference — which tool to use:

  • Need to apply a new GPO immediately? → gpupdate /force
  • Quick check of what GPOs applied? → gpresult /r
  • Detailed report for troubleshooting? → gpresult /h report.html
  • Remote machine, GUI preferred? → GPMC Group Policy Results Wizard

Troubleshooting Mini-Case: GPO Not Applying

🔍 Scenario: Password Policy Not Enforcing

You are the domain administrator for corp.local. Last week you created a GPO named Password Policy – IT and linked it to the IT OU. The GPO is configured with a minimum password length of 14 characters and complexity enabled.

Today, a user in the IT OU successfully changed their password to pass1 — only 5 characters, no complexity. The new password policy is clearly not being enforced. A help desk ticket has been escalated to you.

Your task: Work through the diagnostic steps below (click each one to reveal the explanation and action) to identify the root cause and resolution.

Diagnostic Steps — Click to Reveal

Step 1: Confirm the GPO exists and is linked

Action: Open GPMC. Navigate to the IT OU and check the Linked Group Policy Objects tab. Confirm that Password Policy – IT appears in the list and that its Link Enabled status is Yes.

Also check: In the Group Policy Objects container, click the GPO and check the Details tab. Confirm GPO Status is not set to "All settings disabled" or "Computer configuration settings disabled."

Possible finding: The GPO is linked and enabled. This step rules out a missing or disabled link. ✅

Step 2: Check where the password policy is actually being applied

Root cause identified: This is the most likely cause. Password policies for domain user accounts are a special case in Active Directory. The effective password policy for domain accounts is always taken from the GPO linked to the domain root — not from OU-level GPOs.

A password policy GPO linked to the IT OU only affects the local account password policy on computers in that OU — it has no effect on domain account passwords.

Resolution: Create (or modify) a GPO linked at the domain level (e.g., the Default Domain Policy or a new GPO linked to the domain root) with the desired password settings. The OU-level GPO is not the right place for this setting.

🚨

Important distinction: If you need different password policies for different groups of users (e.g., IT staff require 14-char passwords, regular staff require 10-char passwords), you cannot achieve this with standard GPOs. You must use Fine-Grained Password Policies (configured in Active Directory Administrative Center, not GPMC), which apply to security groups or specific user objects.

Step 3: Run gpresult on the affected user's machine

Action: Log on to (or remotely connect to) a computer in the IT OU. Open an elevated command prompt and run:

Command Prompt (elevated)
gpresult /r /user CORP\affected.user

What to look for:

  • Under COMPUTER SETTINGS → Applied Group Policy Objects: Is Password Policy – IT listed? If not, the GPO is being filtered or not reaching this computer.
  • Under "The following GPOs were not applied": If the GPO appears here, the reason column will tell you why (e.g., "Denied (Security)" means the computer or user account is not in the correct security group for the GPO's security filtering).

Possible finding: The GPO appears under "Applied GPOs" for the computer — confirming it is being processed. This reinforces Step 2's finding: the GPO is applying, but password policies at the OU level don't affect domain account passwords regardless.

Step 4: Check for Block Inheritance on the IT OU

Action: In GPMC, click the IT OU. Check the Group Policy Inheritance tab. If Block Inheritance is set, the OU name in the console tree will show a blue exclamation mark icon.

Why this matters: If Block Inheritance is set on the IT OU, domain-level GPOs (including the Default Domain Policy) are not flowing into the IT OU. This could mean that even a correctly configured domain-level password policy is being blocked.

Resolution if found: Either remove Block Inheritance from the IT OU, or set the domain-level password policy GPO to Enforced so it overrides the Block Inheritance setting.

Step 5: Verify the Default Domain Policy password settings

Action: In GPMC, click Default Domain Policy → Settings tab. Expand Computer Configuration → Windows Settings → Security Settings → Account Policies → Password Policy. Confirm the settings match what you intend to enforce.

PowerShell (alternative verification)
# View current effective password policy for the domain
Get-ADDefaultDomainPasswordPolicy -Identity corp.local

Possible finding: The Default Domain Policy has a minimum password length of only 7 characters (the Windows default). Your new OU-level GPO with 14 characters is not overriding this for domain accounts — it cannot. The Default Domain Policy must be updated (or a new domain-level GPO must be created and linked at the domain root).

Step 6: Apply the fix and verify

Fix:

  1. In GPMC, either edit the Default Domain Policy directly (or, per best practice, create a new GPO named Password Policy – Domain and link it to the domain root).
  2. Set minimum password length to 14, complexity enabled, lockout threshold to 5.
  3. On the client machine, run gpupdate /force.
  4. Run gpresult /r and confirm the domain-level password policy GPO appears under Applied GPOs.
  5. Test by attempting to set a weak password — it should now be rejected by the domain controller.
✅

Lesson learned: Password policies for domain accounts must always be configured at the domain level. OU-level password policy GPOs only affect local accounts on machines in that OU. When in doubt, run gpresult /h report.html and review the full RSoP report — it will show you exactly which settings came from which GPO.

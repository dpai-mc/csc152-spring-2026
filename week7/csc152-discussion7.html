Discussion 7: Permission Challenges

File Systems, Ownership & Permissions — Real-World Scenarios

📅 Initial Post Due: Sunday, March 15, 2026, 11:59 PM 💬 Peer Responses Due: Tuesday, March 17, 2026, 11:59 PM ⭐ 12.5 Points Total

📋 Assignment Overview

Now that you've studied both Linux and Windows permission models, you can apply these concepts to real-world scenarios. Understanding how to diagnose and resolve permission issues — and how to design permission structures that balance security with usability — is one of the most critical skills for any system administrator.

This discussion asks you to analyze a permission scenario, propose a solution, and engage with your classmates' approaches.

12.5 Points Initial Post: Wed Mar 4 Responses: Sun Mar 8 CSLO 1 & CSLO 2

📝 Discussion Prompt

Choose ONE of the following real-world scenarios and write an initial post (minimum 250 words) that addresses all of the required elements listed below.

Scenario Options

Select the scenario that interests you most. Each scenario requires you to apply concepts from Modules 7.1–7.4.

🐧 Scenario A — Linux
The Broken Web Server

You are a system administrator for a small company. A junior admin ran the following command on the production web server:

sudo chmod -R 644 /var/www/html/

Immediately after, the company website went down and the web server (Apache, running as user www-data) is returning "403 Forbidden" errors. Developers (members of the webdev group) are also reporting they cannot cd into subdirectories or run CGI scripts.

Your task: Diagnose the problem, explain exactly what went wrong and why, propose the correct fix with specific commands, and describe what safeguards you would put in place to prevent this from happening again.

🪟 Scenario B — Windows
The Mysterious "Access Denied"

Your company has a shared folder D:\Departments\Finance\ on a Windows file server. The Finance group has Allow Modify permission with inheritance enabled. A new employee, sarah, was added to the Finance group last Friday. On Monday, sarah reports she can browse the Finance folder locally on the server, but when she tries to access \\FileServer\Finance\ from her workstation over the network, she gets "Access Denied."

Investigation reveals that the "Finance" share has share-level permissions set to Everyone: Read.

Your task: Explain the interaction between NTFS and share permissions, diagnose why sarah's network access is limited, propose the fix, and recommend a best-practice approach for configuring share vs. NTFS permissions going forward.

🔄 Scenario C — Cross-Platform
Designing a Secure Shared Project Directory

Your organization has a team of 8 developers who work on both Linux and Windows servers. You need to design a shared project directory structure on BOTH platforms with the following requirements:

  • All team members can read and write project files
  • New files automatically belong to the team group (not individual users' primary groups)
  • Users outside the team have NO access
  • No team member can delete another member's files
  • A separate "releases" subdirectory should be read-only for the team (only the lead developer can modify it)

Your task: Propose the complete permission configuration for both Linux and Windows, including specific commands. Explain which special permissions (SetGID, sticky bit, etc.) you would use and why. Discuss the advantages and limitations of each platform's approach.

Required Elements for Your Initial Post

Regardless of which scenario you choose, your post must include all of the following:

  1. Diagnosis: A clear explanation of what is causing the problem (or what challenges the design requirements present)
  2. Technical Solution: Specific commands or configurations with correct syntax — not just general descriptions. Show the actual chmod, chown, icacls, or PowerShell commands you would use
  3. Conceptual Explanation: Explain why your solution works by referencing specific permission concepts (e.g., "execute on directories enables traversal," "Deny overrides Allow," "SetGID causes group inheritance")
  4. Prevention / Best Practice: Describe what processes, policies, or safeguards you would implement to prevent similar issues in the future or to maintain the configuration over time
  5. Security Consideration: Identify at least one security implication of the scenario and how your solution addresses it
📌 Minimum Length
Your initial post must be a minimum of 250 words. Posts that are purely commands without explanation, or purely explanation without specific commands, will not receive full credit. The best posts integrate both technical detail and conceptual understanding.

💬 Peer Response Requirements

In addition to your initial post, you must respond to at least two classmates' posts. Substantive responses (minimum 75 words each) should do one or more of the following:

  • Offer an alternative approach — Is there a different way to solve the same problem? What are the tradeoffs?
  • Identify a potential issue — Does the classmate's solution have any gaps, edge cases, or security concerns they didn't address?
  • Build on their solution — Add additional commands, configurations, or safeguards that strengthen their approach
  • Ask a thoughtful question — Pose a "what if" scenario that tests the robustness of their solution
  • Connect to real-world experience — If you've encountered a similar situation, share your experience (even from lab work)

💡 Example Approaches (Starting Points Only)

These brief snippets show the type of technical detail expected. Your post should be much more comprehensive.

Example: Scenario A — Key Commands (Do NOT copy — develop your own analysis)

A strong response to Scenario A would include commands like:

# Fix directory permissions (need execute for traversal)
sudo find /var/www/html -type d -exec chmod 750 {} \;

# Fix file permissions (web content needs read by www-data)
sudo find /var/www/html -type f -exec chmod 640 {} \;

# Fix CGI scripts (need execute bit)
sudo find /var/www/html/cgi-bin -type f -name "*.cgi" -exec chmod 750 {} \;

# Ensure correct ownership
sudo chown -R webdev:www-data /var/www/html/

But commands alone are not enough — you must explain why chmod -R 644 broke things (removed execute from directories, preventing cd and file access) and why your fix works.

Example: Scenario B — Key Concepts (Do NOT copy — develop your own analysis)

A strong response to Scenario B would explain:

  • The interaction between NTFS permissions (apply locally and over the network) and share permissions (apply only over the network)
  • The "most restrictive wins" rule for network access
  • Why sarah can access locally (NTFS Modify) but not over the network (share limits to Read)
  • The best practice of setting share permissions broadly and controlling access through NTFS
Example: Scenario C — Key Concepts (Do NOT copy — develop your own analysis)

A strong response to Scenario C would address:

  • Linux: SetGID (2000) for group inheritance, sticky bit (1000) to prevent cross-deletion, umask 002 for group-writable files
  • Windows: NTFS ACLs with inheritance, potentially breaking inheritance on the "releases" subdirectory
  • Comparison of how each platform handles the "no cross-deletion" requirement differently
  • The limitation that Linux cannot natively prevent cross-deletion AND allow group write without the sticky bit

3 Exercise 1: Linux Symbolic Permissions

▼

CSLO 1 CSLO 2

Objective

Configure and verify Linux file and directory permissions using symbolic notation (chmod u+rwx,g+rx,o-rwx) to implement a departmental file structure that enforces the principle of least privilege.

Background

Linux uses a three-tier permission model: owner (u), group (g), and others (o). Each tier can be assigned read (r), write (w), and execute (x) permissions. Symbolic notation uses letters and operators (+, -, =) to modify permissions relative to their current state.

Symbol Meaning Example
uOwner (user)u+w — add write for owner
gGroupg-x — remove execute for group
oOtherso=r — set others to read only
aAll (u+g+o)a+r — add read for everyone
+Add permissiong+rw
-Remove permissiono-rwx
=Set exact permissionu=rwx

Tasks

  1. Create the directory structure: 
    # Create departmental directories
sudo mkdir -p /srv/departments/engineering
sudo mkdir -p /srv/departments/marketing
sudo mkdir -p /srv/departments/shared

# Create test files in each directory
sudo touch /srv/departments/engineering/design-spec.txt
sudo touch /srv/departments/engineering/build-script.sh
sudo touch /srv/departments/marketing/campaign-plan.txt
sudo touch /srv/departments/shared/company-policy.txt
  2. Set ownership: 
    # Set group ownership for department directories
sudo chown -R root:engineering /srv/departments/engineering
sudo chown -R root:marketing /srv/departments/marketing
sudo chown root:root /srv/departments/shared
  3. Apply symbolic permissions: 
    # Engineering: owner full, group read+execute, others none
sudo chmod u=rwx,g=rx,o= /srv/departments/engineering

# Marketing: owner full, group read+execute, others none
sudo chmod u=rwx,g=rx,o= /srv/departments/marketing

# Shared: owner full, all groups read+execute, others read
sudo chmod u=rwx,g=rx,o=rx /srv/departments/shared

# Make build-script.sh executable for group
sudo chmod g+x /srv/departments/engineering/build-script.sh

# Restrict campaign plan to group read only (no execute)
sudo chmod u=rw,g=r,o= /srv/departments/marketing/campaign-plan.txt
  4. Verify permissions: 
    # List all permissions recursively
ls -laR /srv/departments/

# Expected output for engineering directory:
# drwxr-x--- root engineering  ... engineering
# -rw-r----- root engineering  ... design-spec.txt
# -rw-r-x--- root engineering  ... build-script.sh
  5. Test access as different users: 
    # Test as alice (engineering group)
sudo -u alice ls /srv/departments/engineering     # Should SUCCEED
sudo -u alice cat /srv/departments/engineering/design-spec.txt  # Should SUCCEED
sudo -u alice ls /srv/departments/marketing       # Should FAIL (Permission denied)

# Test as charlie (marketing group)
sudo -u charlie ls /srv/departments/marketing     # Should SUCCEED
sudo -u charlie ls /srv/departments/engineering    # Should FAIL (Permission denied)

# Test shared directory access
sudo -u alice ls /srv/departments/shared          # Should SUCCEED
sudo -u charlie ls /srv/departments/shared        # Should SUCCEED

Take screenshots of: (1) the ls -laR output showing all permissions, and (2) each access test showing both successful and denied results. Label each screenshot clearly.

4 Exercise 2: Linux Numeric (Octal) Permissions

▼

CSLO 1 CSLO 2

Objective

Apply the same permission concepts from Exercise 1 using numeric (octal) notation and understand how three-digit and four-digit modes map to permission bits.

Octal Permission Reference

Octal Binary Symbolic Description
7111rwxRead + Write + Execute
6110rw-Read + Write
5101r-xRead + Execute
4100r--Read only
3011-wxWrite + Execute
2010-w-Write only
1001--xExecute only
0000---No permissions

Special Permission Bits (4th digit)

Bit Octal Name Effect on Files Effect on Directories
s (user)4SUIDRuns as file owner—
s (group)2SGIDRuns as file groupNew files inherit group
t1Sticky Bit—Only owner can delete files

Tasks

  1. Create a new project directory structure: 
    # Create project directories
sudo mkdir -p /srv/projects/alpha
sudo mkdir -p /srv/projects/beta
sudo mkdir -p /srv/projects/temp

# Create test files
sudo touch /srv/projects/alpha/readme.md
sudo touch /srv/projects/alpha/deploy.sh
sudo touch /srv/projects/beta/report.docx
sudo touch /srv/projects/temp/scratch.txt
  2. Apply numeric permissions: 
    # alpha: owner rwx, group rx, others none = 750
sudo chmod 750 /srv/projects/alpha

# beta: owner rwx, group rx, others rx = 755
sudo chmod 755 /srv/projects/beta

# deploy.sh: owner rwx, group rx, others none = 750
sudo chmod 750 /srv/projects/alpha/deploy.sh

# readme.md: owner rw, group r, others none = 640
sudo chmod 640 /srv/projects/alpha/readme.md

# report.docx: owner rw, group r, others r = 644
sudo chmod 644 /srv/projects/beta/report.docx
  3. Apply special permission bits: 
    # Set SGID on alpha directory (new files inherit engineering group)
sudo chown root:engineering /srv/projects/alpha
sudo chmod 2750 /srv/projects/alpha

# Set Sticky Bit on temp directory (only owners can delete their files)
sudo chmod 1777 /srv/projects/temp

# Verify special permissions
ls -la /srv/projects/
# Expected:
# drwxr-s--- ... alpha      (SGID shown as 's' in group execute)
# drwxr-xr-x ... beta
# drwxrwxrwt ... temp       (Sticky shown as 't' in others execute)
  4. Test SGID behavior: 
    # Create a file as alice in the SGID directory
sudo -u alice touch /srv/projects/alpha/alice-file.txt

# Check group ownership of new file
ls -la /srv/projects/alpha/alice-file.txt
# Expected: group should be "engineering" (inherited from directory)
  5. Test Sticky Bit behavior: 
    # Create files as different users in temp
sudo -u alice touch /srv/projects/temp/alice-temp.txt
sudo -u bob touch /srv/projects/temp/bob-temp.txt

# Try to delete another user's file
sudo -u alice rm /srv/projects/temp/bob-temp.txt
# Expected: Operation not permitted (sticky bit prevents deletion)
  6. Verify with stat command: 
    # Show detailed permission information
stat /srv/projects/alpha
stat /srv/projects/temp

# The "Access" line shows both symbolic and octal permissions
# Example: Access: (2750/drwxr-s---)  Uid: (0/root)  Gid: (1001/engineering)

ⓘ Symbolic ↔ Octal Conversion

Practice converting between formats. For example: rwxr-x--- → 750, and chmod 2750 → drwxr-s---. Understanding both notations is essential for Quiz 7 and the final exam.

5 Exercise 3: Linux POSIX ACLs

▼

CSLO 1 CSLO 2

Objective

Extend the standard Linux permission model using POSIX Access Control Lists (ACLs) to grant fine-grained permissions to specific users and groups beyond the traditional owner/group/others model.

Background

Standard Linux permissions only allow one owner and one group per file. ACLs solve this limitation by allowing you to define permissions for multiple named users and groups on a single file or directory. ACLs are managed with setfacl (set) and getfacl (view).

💡 ACL Prerequisite

Ensure the acl package is installed and the filesystem is mounted with ACL support. Most modern Linux distributions enable ACLs by default on ext4 and xfs filesystems.

# Verify ACL support
sudo apt install acl        # Debian/Ubuntu
# OR
sudo dnf install acl        # Rocky Linux/RHEL

# Check if filesystem supports ACLs
mount | grep acl
# Or check for the '+' indicator in ls -l output after setting an ACL

Tasks

  1. Scenario: The /srv/departments/engineering directory belongs to the engineering group, but charlie (marketing) needs read-only access to design-spec.txt for a cross-department project. Standard permissions cannot accommodate this without adding charlie to the engineering group.
  2. Set a user-specific ACL: 
    # Grant charlie read access to the engineering directory (for traversal)
sudo setfacl -m u:charlie:rx /srv/departments/engineering

# Grant charlie read access to the specific file
sudo setfacl -m u:charlie:r /srv/departments/engineering/design-spec.txt

# Verify the ACL
getfacl /srv/departments/engineering/design-spec.txt

    Expected getfacl output:

    # file: srv/departments/engineering/design-spec.txt
# owner: root
# group: engineering
user::rw-
user:charlie:r--
group::r--
mask::r--
other::---
  3. Set a group-specific ACL: 
    # Grant marketing group read access to the shared company policy
sudo setfacl -m g:marketing:r /srv/departments/shared/company-policy.txt

# Verify
getfacl /srv/departments/shared/company-policy.txt
  4. Set default ACLs on a directory (inherited by new files): 
    # Set default ACL so new files in engineering inherit charlie's read access
sudo setfacl -d -m u:charlie:r /srv/departments/engineering

# Create a new file and verify it inherits the ACL
sudo touch /srv/departments/engineering/new-document.txt
getfacl /srv/departments/engineering/new-document.txt
# charlie should have r-- access automatically
  5. Test ACL access: 
    # Test charlie's access (should now work via ACL)
sudo -u charlie cat /srv/departments/engineering/design-spec.txt   # Should SUCCEED
sudo -u charlie ls /srv/departments/engineering/                    # Should SUCCEED (rx ACL)

# Verify charlie still cannot write
sudo -u charlie touch /srv/departments/engineering/test.txt         # Should FAIL
  6. View and remove ACLs: 
    # View all ACLs recursively
getfacl -R /srv/departments/engineering

# Remove a specific ACL entry
sudo setfacl -x u:charlie /srv/departments/engineering/design-spec.txt

# Remove all ACLs from a file
sudo setfacl -b /srv/departments/engineering/design-spec.txt

# Verify removal
getfacl /srv/departments/engineering/design-spec.txt
# Should show standard permissions only (no named user entries)

⚠ Important: The ACL Mask

The mask entry defines the maximum effective permissions for all named users and groups (not the owner). If the mask is r--, a named user with rw- ACL effectively only gets r--. Always check the mask when troubleshooting ACL issues. Use setfacl -m m::rwx to set the mask explicitly.

Objective

Configure NTFS permissions on Windows Server to implement a departmental folder structure with appropriate access controls using both the GUI (Security tab) and PowerShell.

NTFS Permission Levels

Permission Abbreviation Allows
Full ControlFAll operations including changing permissions and ownership
ModifyMRead, write, execute, delete
Read & ExecuteRXView contents, run programs
List Folder ContentsLView folder contents (folders only)
ReadRView file contents and attributes
WriteWCreate files/folders, modify attributes

Tasks

  1. Create folder structure using PowerShell: 
    # Create departmental folders
New-Item -ItemType Directory -Path "C:\Departments\Engineering" -Force
New-Item -ItemType Directory -Path "C:\Departments\Marketing" -Force
New-Item -ItemType Directory -Path "C:\Departments\Shared" -Force

# Create test files
New-Item -ItemType File -Path "C:\Departments\Engineering\design-spec.txt" -Force
New-Item -ItemType File -Path "C:\Departments\Marketing\campaign-plan.txt" -Force
New-Item -ItemType File -Path "C:\Departments\Shared\company-policy.txt" -Force

# Add content for testing
Set-Content -Path "C:\Departments\Engineering\design-spec.txt" -Value "Engineering Design Specification v1.0"
Set-Content -Path "C:\Departments\Marketing\campaign-plan.txt" -Value "Q2 Marketing Campaign Plan"
Set-Content -Path "C:\Departments\Shared\company-policy.txt" -Value "Company Policy Document"
  2. Disable inheritance and remove default permissions: 
    # Get current ACL for Engineering folder
$acl = Get-Acl "C:\Departments\Engineering"

# Disable inheritance and remove inherited rules
$acl.SetAccessRuleProtection($true, $false)

# Add Administrators - Full Control
$adminRule = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "Administrators", "FullControl", "ContainerInherit,ObjectInherit", "None", "Allow"
)
$acl.AddAccessRule($adminRule)

# Add Engineering group - Modify
$engRule = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "Engineering", "Modify", "ContainerInherit,ObjectInherit", "None", "Allow"
)
$acl.AddAccessRule($engRule)

# Apply the ACL
Set-Acl "C:\Departments\Engineering" $acl

# Verify
Get-Acl "C:\Departments\Engineering" | Format-List
  3. Configure Marketing folder similarly: 
    $acl2 = Get-Acl "C:\Departments\Marketing"
$acl2.SetAccessRuleProtection($true, $false)

$adminRule2 = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "Administrators", "FullControl", "ContainerInherit,ObjectInherit", "None", "Allow"
)
$acl2.AddAccessRule($adminRule2)

$mktRule = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "Marketing", "Modify", "ContainerInherit,ObjectInherit", "None", "Allow"
)
$acl2.AddAccessRule($mktRule)

Set-Acl "C:\Departments\Marketing" $acl2
  4. Configure Shared folder with read access for all: 
    $acl3 = Get-Acl "C:\Departments\Shared"
$acl3.SetAccessRuleProtection($true, $false)

$adminRule3 = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "Administrators", "FullControl", "ContainerInherit,ObjectInherit", "None", "Allow"
)
$acl3.AddAccessRule($adminRule3)

$usersRule = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "Users", "ReadAndExecute", "ContainerInherit,ObjectInherit", "None", "Allow"
)
$acl3.AddAccessRule($usersRule)

Set-Acl "C:\Departments\Shared" $acl3
  5. Verify using icacls: 
    # View permissions for all department folders
icacls "C:\Departments\Engineering"
icacls "C:\Departments\Marketing"
icacls "C:\Departments\Shared"

# Expected output for Engineering:
# C:\Departments\Engineering BUILTIN\Administrators:(OI)(CI)(F)
#                             YOURSERVER\Engineering:(OI)(CI)(M)
  6. Test access via GUI:
    • Log in as alice (Engineering) → verify access to Engineering folder, denied to Marketing
    • Log in as charlie (Marketing) → verify access to Marketing folder, denied to Engineering
    • Both users should have read access to Shared folder

7 Exercise 5: NTFS ACLs & Troubleshooting

▼

CSLO 1 CSLO 2

Objective

Configure advanced NTFS ACL entries for cross-department access, understand permission inheritance and effective permissions, and troubleshoot common access issues.

Part A: Cross-Department ACL

Scenario: charlie (Marketing) needs Read access to the Engineering folder’s design-spec.txt for a joint project, without granting access to the entire Engineering folder.

  1. Add a specific user ACL entry: 
    # Grant charlie Read access to the specific file only
$fileAcl = Get-Acl "C:\Departments\Engineering\design-spec.txt"

$charlieRule = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "charlie", "Read", "None", "None", "Allow"
)
$fileAcl.AddAccessRule($charlieRule)

Set-Acl "C:\Departments\Engineering\design-spec.txt" $fileAcl

# Also grant charlie List/Traverse on the parent folder (for navigation)
$parentAcl = Get-Acl "C:\Departments\Engineering"

$traverseRule = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "charlie", "ListDirectory,ReadAttributes", "None", "None", "Allow"
)
$parentAcl.AddAccessRule($traverseRule)

Set-Acl "C:\Departments\Engineering" $parentAcl
  2. Verify with icacls: 
    icacls "C:\Departments\Engineering\design-spec.txt"
# Should show charlie with (R) permission

icacls "C:\Departments\Engineering"
# Should show charlie with limited traverse permissions

Part B: Effective Permissions Analysis

  1. Check effective permissions via PowerShell: 
    # View effective access for charlie on the Engineering folder
$engineeringAcl = Get-Acl "C:\Departments\Engineering"
$engineeringAcl.Access | Where-Object {
    $_.IdentityReference -match "charlie|Engineering|Users|Everyone"
} | Format-Table IdentityReference, FileSystemRights, AccessControlType, InheritanceFlags -AutoSize
  2. Check effective permissions via GUI:
    • Right-click C:\Departments\Engineering → Properties → Security → Advanced → Effective Access
    • Select user charlie
    • Screenshot the effective permissions panel

Part C: Troubleshooting Scenarios

Work through each scenario, diagnose the issue, and document your solution:

Scenario 1: Denied Despite Allow Rule

bob has an explicit Allow: Modify on a folder, but receives “Access Denied.” What could cause this?

Diagnostic steps:

# Check for Deny rules (Deny always overrides Allow)
$acl = Get-Acl "C:\Departments\Engineering"
$acl.Access | Where-Object { $_.AccessControlType -eq "Deny" } |
    Format-Table IdentityReference, FileSystemRights, AccessControlType

# Check group memberships (bob might be in a group with Deny)
Get-LocalGroup | ForEach-Object {
    $members = Get-LocalGroupMember -Group $_.Name -ErrorAction SilentlyContinue
    if ($members.Name -match "bob") {
        Write-Host "$($_.Name): bob is a member"
    }
}

Scenario 2: New Files Have Wrong Permissions

Files created in C:\Departments\Shared inherit Full Control for Users instead of Read & Execute. Why?

Diagnostic steps:

# Check inheritance settings
$sharedAcl = Get-Acl "C:\Departments\Shared"
$sharedAcl.Access | Format-Table IdentityReference, FileSystemRights,
    InheritanceFlags, PropagationFlags -AutoSize

# Look for ContainerInherit and ObjectInherit flags
# Fix: Ensure parent folder ACL has correct inheritance flags

Scenario 3: Permission Changes Not Taking Effect

You changed permissions on a parent folder but files inside still show old permissions. What happened?

Answer: Child objects may have inheritance disabled (explicit permissions). Check each child:

# Check if inheritance is enabled on child objects
$childAcl = Get-Acl "C:\Departments\Shared\company-policy.txt"
Write-Host "Are inherited rules protected? $($childAcl.AreAccessRulesProtected)"
# True = inheritance is disabled (explicit only)
# False = inheritance is enabled

# Re-enable inheritance if needed
$childAcl.SetAccessRuleProtection($false, $true)
Set-Acl "C:\Departments\Shared\company-policy.txt" $childAcl

1 Introduction: Linux vs. Windows Permission Models

Windows uses a fundamentally different approach to file permissions than Linux. While Linux uses a compact 12-bit model with three fixed categories, Windows NTFS uses Access Control Lists (ACLs) that can contain any number of entries for any user or group.

1.1 Two Philosophies

🐧 Linux (Modules 7.1–7.2)

  • Three fixed categories: owner, group, others
  • Three permission types: read, write, execute
  • 12 bits total (9 standard + 3 special)
  • One owner, one group per file
  • No built-in "Deny" — absence of permission = denied
  • Managed via chmod, chown, chgrp
vs.

🪟 Windows NTFS (This Module)

  • Unlimited entries for any user or group
  • 13 granular permission types
  • Access Control List (ACL) with variable length
  • Owner + any number of ACL entries
  • Explicit Deny entries override Allow
  • Managed via GUI, icacls, PowerShell

1.2 Key Terminology

NTFS (New Technology File System)
The default file system for Windows Server and modern Windows desktops. NTFS supports file permissions, encryption, compression, disk quotas, and journaling. Permissions are only enforced on NTFS volumes — FAT32 and exFAT do not support permissions.
ACL (Access Control List)
An ordered list of rules (called Access Control Entries, or ACEs) attached to every NTFS object. The ACL defines which users and groups are allowed or denied specific types of access. Every file and folder on an NTFS volume has an ACL.
ACE (Access Control Entry)
A single rule within an ACL. Each ACE specifies: (1) a security principal (user, group, or computer account), (2) an access type (Allow or Deny), and (3) one or more permissions (e.g., Read, Write, Modify). An ACL can contain many ACEs.
Security Principal
Any entity that can be authenticated and assigned permissions: a user account, a group, a computer account, or a service account. Each principal has a unique Security Identifier (SID).
SID (Security Identifier)
A unique, immutable identifier assigned to every security principal in Windows. Even if a user account is renamed, its SID stays the same. Permissions are internally stored using SIDs, not usernames. Example SID: S-1-5-21-3623811015-3361044348-30300820-1013
DACL (Discretionary Access Control List)
The part of the ACL that the file owner can modify. It contains the Allow and Deny ACEs that determine who can access the object. When people say "NTFS permissions," they are usually referring to the DACL.
SACL (System Access Control List)
The part of the ACL used for auditing. It defines which access attempts should be logged to the Security event log. SACLs are configured by administrators and are separate from DACLs. (Covered more in Module 7.4.)

🔄 Comparison: Linux Inode vs. Windows Security Descriptor

In Linux, permissions are stored in the file's inode as a 12-bit field. In Windows, permissions are stored in a Security Descriptor that can be arbitrarily large because the DACL can contain any number of ACEs. This makes Windows permissions more flexible but also more complex.

2 NTFS Permission Fundamentals

NTFS permissions are organized into two tiers: six "basic" permissions that cover most administrative needs, and thirteen "advanced" (granular) permissions that provide fine-grained control.

2.1 The Six Basic NTFS Permissions

Basic permissions are combinations of the underlying advanced permissions, packaged for convenience:

Basic PermissionOn FilesOn Folders
Full ControlRead, write, modify, delete, change permissions, take ownershipAll file permissions + create/delete subfolders and files, change permissions, take ownership
ModifyRead, write, modify, delete the fileRead, write, create files/subfolders, delete the folder and its contents
Read & ExecuteView contents and run executable filesView contents, list folder contents, traverse folders, run executables
List Folder ContentsN/A (folders only)View names of files and subfolders within the folder
ReadView file contents and attributesView folder contents, attributes, and permissions
WriteOverwrite the file, change attributesCreate new files and subfolders, change folder attributes
📌 Permission Hierarchy

Basic permissions are cumulative in a hierarchy. Each higher level includes all permissions below it:

Full Control ⊃ Modify ⊃ Read & Execute ⊃ Read

Write is separate — it does not include Read. A user with only Write permission can create files but cannot view existing ones.

2.2 Allow vs. Deny

Unlike Linux, where the absence of a permission simply means "denied," Windows has explicit Deny entries:

Access TypeBehaviorUse Case
AllowGrants the specified permission to the principalStandard permission assignment (most common)
DenyExplicitly blocks the specified permission, overriding any Allow entriesExceptions — blocking a specific user from a group permission
🚨 Critical Rule: Deny Overrides Allow

If a user receives both an Allow and a Deny for the same permission (e.g., through different group memberships), Deny always wins. This is the single most important rule in NTFS permission evaluation.

Example: Alice is a member of both the Marketing group (Allow Read) and the Contractors group (Deny Read). Result: Alice is denied Read access.

2.3 How Windows Evaluates Effective Access

When a user tries to access an NTFS object, Windows follows these steps:

  1. Collect all applicable ACEs: Gather every ACE that applies to the user directly or through group membership
  2. Check for Deny: If any applicable ACE denies the requested permission, access is denied (Deny wins)
  3. Check for Allow: If at least one applicable ACE allows the requested permission (and no Deny exists), access is granted
  4. Implicit Deny: If no ACE (Allow or Deny) matches the requested permission, access is denied by default

NTFS Permission Evaluation Flow

Collect all ACEs for user + groups
→
Any Deny? → ACCESS DENIED
→
Any Allow? → ACCESS GRANTED
→
No match → IMPLICIT DENY

2.4 Cumulative Permissions from Multiple Groups

When a user belongs to multiple groups, their Allow permissions are cumulative (combined). This is different from Linux, where only one category applies:

Example: Cumulative Permissions

User alice belongs to two groups with permissions on D:\Reports\:

  • Marketing group → Allow Read
  • Managers group → Allow Write

Effective access for alice: Read + Write (cumulative)

Now add a Deny:

  • Marketing group → Allow Read
  • Managers group → Allow Write
  • Contractors group → Deny Write

If alice is also in Contractors:

Effective access: Read only (Deny Write overrides the Allow Write from Managers)

⚠️ Best Practice: Avoid Deny When Possible

Deny entries make permissions harder to troubleshoot and can produce unexpected results. In most cases, you should control access by not granting an Allow permission rather than explicitly denying it. Use Deny only when you need to create an exception to a broad Allow (e.g., blocking one user from a group permission).

💡 When to Use Advanced Permissions

Basic permissions handle 90%+ of real-world scenarios. Use advanced permissions only when you need granular control that basic permissions cannot provide. Examples:

  • Allowing users to create files in a folder but not delete them (grant Create Files, deny Delete)
  • Allowing users to append to a log file but not read or overwrite it (grant Append Data only)
  • Allowing a helpdesk team to change permissions without granting Full Control

4 Permission Inheritance

One of the most powerful features of NTFS permissions is inheritance — child objects (files and subfolders) automatically receive the permissions of their parent folder. This dramatically simplifies permission management in deep directory structures.

Permission Inheritance
The mechanism by which NTFS permissions assigned to a parent folder are automatically propagated to all child objects (files and subfolders) within that folder. Inherited permissions appear grayed out in the GUI, indicating they cannot be changed directly — you must change them at the source (parent) or break inheritance.

4.1 How Inheritance Works

Permission Inheritance Flow

D:\Projects\ (Parent)
Marketing: Allow Modify
↓ inherits
D:\Projects\Reports\ (Child folder)
Marketing: Allow Modify (inherited)
↓ inherits
D:\Projects\Reports\Q1.xlsx (File)
Marketing: Allow Modify (inherited)

4.2 Explicit vs. Inherited Permissions

TypeSourceGUI AppearanceCan Edit Directly?
ExplicitSet directly on this objectCheckboxes are editableYes
InheritedPropagated from a parent folderCheckboxes are grayed outNo — must change at parent or break inheritance

4.3 Breaking Inheritance

Sometimes you need a child object to have different permissions than its parent. You can break (disable) inheritance on any object. When you do, Windows gives you two choices:

  1. Convert inherited permissions into explicit permissions: The current inherited permissions are copied as explicit entries. You can then modify them freely. (Recommended — preserves existing access.)
  2. Remove all inherited permissions: All inherited ACEs are deleted. Only explicit permissions remain. (Caution — can lock everyone out if no explicit permissions exist.)
⚠️ Breaking Inheritance: Think Before You Click

Breaking inheritance creates a "permission island" that no longer updates when the parent changes. This can lead to:

  • Permission drift: Over time, the child's permissions diverge from organizational policy
  • Administrative overhead: Each "island" must be managed independently
  • Troubleshooting difficulty: Harder to trace where permissions come from

Best practice: Keep inheritance enabled whenever possible. Use explicit permissions (added on top of inherited ones) for exceptions rather than breaking inheritance entirely.

4.4 Inheritance and Deny

When combining inheritance with Deny entries, the evaluation order matters:

  1. Explicit Deny (set directly on this object) — highest priority
  2. Explicit Allow (set directly on this object)
  3. Inherited Deny (from parent folder)
  4. Inherited Allow (from parent folder)
📌 Key Insight: Explicit Overrides Inherited

An explicit Allow on a child object overrides an inherited Deny from the parent. This is the one exception to the "Deny always wins" rule — explicit entries take priority over inherited entries of any type. This means you can use explicit Allow entries to create exceptions to inherited Deny rules.

Example: Inheritance Priority

Parent folder D:\Data\:

  • Interns group → Deny Write (inherited by children)

Child file D:\Data\feedback.txt:

  • Interns group → explicit Allow Write (set directly on this file)

Result: Interns can write to feedback.txt because the explicit Allow overrides the inherited Deny.

This is the correct evaluation order: Explicit Allow (priority 2) beats Inherited Deny (priority 3).

5 Managing Permissions via GUI

The Windows GUI provides the Security tab in file/folder Properties for managing NTFS permissions. This is the most common method for interactive permission management.

5.1 Accessing the Security Tab

  1. Right-click the file or folder in File Explorer
  2. Select Properties
  3. Click the Security tab

The Security tab shows two panels:

  • Top panel: List of security principals (users and groups) with permissions on this object
  • Bottom panel: The specific permissions for the selected principal

5.2 Viewing Permissions

In the Security tab:

  • Editable checkboxes = explicit permissions (set on this object)
  • Grayed-out checkboxes = inherited permissions (from parent folder)
  • Allow column = permissions granted
  • Deny column = permissions explicitly blocked

5.3 Modifying Permissions

  1. Click Edit to open the permissions editor
  2. Click Add to add a new user or group
  3. Type the username or group name and click Check Names to validate
  4. Select the desired Allow/Deny checkboxes
  5. Click OK to apply

5.4 Accessing Advanced Permissions

  1. In the Security tab, click Advanced
  2. The Advanced Security Settings dialog shows:
    • Permissions tab: All ACEs with inheritance information
    • Auditing tab: SACL entries (for logging)
    • Owner: Current owner with option to change
    • Effective Access tab: Calculate effective permissions for a specific user
  3. Click Disable inheritance to break inheritance (with copy or remove options)
  4. Click Add to create a new ACE with granular advanced permissions

5.5 Using Effective Access

The Effective Access tab in Advanced Security Settings lets you check what permissions a specific user actually has, accounting for all group memberships, Deny entries, and inheritance:

  1. Click Advanced → Effective Access tab
  2. Click Select a user and enter the username
  3. Click View effective access
  4. Windows displays a grid showing which permissions are granted and denied
💡 Pro Tip: Always Verify with Effective Access

When troubleshooting "access denied" issues, the Effective Access tab is your best friend. It shows the actual permissions after all ACEs, group memberships, and inheritance are evaluated — eliminating guesswork.

6 Managing Permissions via PowerShell & icacls

For scripting, automation, and Server Core installations (no GUI), you need command-line tools. Windows provides two primary approaches: PowerShell cmdlets and the icacls utility.

6.1 Viewing Permissions with icacls

icacls — Integrity Control Access Control Lists
Syntax: icacls <path> [options]
Purpose: View, modify, backup, and restore NTFS permissions from the command line.
Available on: All modern Windows versions (Vista/Server 2008 and later).
REM View permissions on a folder
icacls D:\Projects

REM Sample output:
D:\Projects BUILTIN\Administrators:(OI)(CI)(F)
             NT AUTHORITY\SYSTEM:(OI)(CI)(F)
             YOURSERVER\Marketing:(OI)(CI)(M)
             YOURSERVER\Interns:(OI)(CI)(R)
             BUILTIN\Users:(OI)(CI)(RX)

icacls Permission Abbreviations

icacls Inheritance Flags

FlagMeaning

6.2 Modifying Permissions with icacls

REM Grant Modify permission to Marketing group (with inheritance)
icacls D:\Projects /grant Marketing:(OI)(CI)M

REM Grant Read permission to Interns (with inheritance)
icacls D:\Projects /grant Interns:(OI)(CI)R

REM Deny Write to Contractors (with inheritance)
icacls D:\Projects /deny Contractors:(OI)(CI)W

REM Remove all permissions for a user
icacls D:\Projects /remove OldUser

REM Apply permissions recursively to all children
icacls D:\Projects /grant Marketing:(OI)(CI)M /T

REM Reset permissions to inherited defaults
icacls D:\Projects /reset /T

REM Backup permissions to a file
icacls D:\Projects /save perms_backup.txt /T

REM Restore permissions from a backup
icacls D:\ /restore perms_backup.txt

6.3 Viewing Permissions with PowerShell

# View the ACL of a folder
Get-Acl -Path "D:\Projects" | Format-List

# View just the access rules (ACEs)
(Get-Acl -Path "D:\Projects").Access

# View access rules in a readable table
(Get-Acl -Path "D:\Projects").Access |
  Format-Table IdentityReference, FileSystemRights, AccessControlType, IsInherited -AutoSize

# Sample output:
IdentityReference          FileSystemRights  AccessControlType  IsInherited
-----------------          ----------------  -----------------  -----------
BUILTIN\Administrators     FullControl       Allow              False
NT AUTHORITY\SYSTEM         FullControl       Allow              False
YOURSERVER\Marketing        Modify            Allow              False
YOURSERVER\Interns          Read              Allow              True

6.4 Modifying Permissions with PowerShell

# Step 1: Get the current ACL
$acl = Get-Acl -Path "D:\Projects"

# Step 2: Create a new access rule
# Parameters: IdentityReference, FileSystemRights, InheritanceFlags,
#             PropagationFlags, AccessControlType
$rule = New-Object System.Security.AccessControl.FileSystemAccessRule(
  "Marketing",
  "Modify",
  "ContainerInherit,ObjectInherit",
  "None",
  "Allow"
)

# Step 3: Add the rule to the ACL
$acl.SetAccessRule($rule)

# Step 4: Apply the modified ACL back to the folder
Set-Acl -Path "D:\Projects" -AclObject $acl

6.5 Removing Permissions with PowerShell

# Remove a specific access rule
$acl = Get-Acl -Path "D:\Projects"

# Find the rule to remove
$rule = $acl.Access | Where-Object {
  $_.IdentityReference -eq "YOURSERVER\OldUser"
}

# Remove it
$acl.RemoveAccessRule($rule)
Set-Acl -Path "D:\Projects" -AclObject $acl

6.6 Taking Ownership with PowerShell

# Take ownership of a file (must run as Administrator)
$acl = Get-Acl -Path "D:\LockedFile.txt"
$owner = New-Object System.Security.Principal.NTAccount("Administrator")
$acl.SetOwner($owner)
Set-Acl -Path "D:\LockedFile.txt" -AclObject $acl

🏗️ Activity 4: Real-World Scenarios

Work through each scenario and expand to check your solution.

Scenario 1: Department File Share Setup

Situation: You need to set up D:\Departments\Marketing\ on a Windows file server. Requirements:

  • The Marketing group should have Modify access
  • The IT-Admins group should have Full Control
  • The Interns group should have Read only
  • No one else should have any access
  • All permissions should inherit to subfolders and files
Show Solution

Using icacls:

REM First, disable inheritance and remove inherited permissions
icacls D:\Departments\Marketing /inheritance:r

REM Grant Full Control to IT-Admins
icacls D:\Departments\Marketing /grant IT-Admins:(OI)(CI)F

REM Grant Modify to Marketing
icacls D:\Departments\Marketing /grant Marketing:(OI)(CI)M

REM Grant Read to Interns
icacls D:\Departments\Marketing /grant Interns:(OI)(CI)R

REM Ensure SYSTEM has Full Control (required for OS operations)
icacls D:\Departments\Marketing /grant SYSTEM:(OI)(CI)F

REM Verify
icacls D:\Departments\Marketing

Key points: The /inheritance:r flag removes all inherited permissions, creating a clean slate. Always include SYSTEM with Full Control so the operating system can manage the folder. The (OI)(CI) flags ensure inheritance to files and subfolders.

Scenario 2: Restricting One User from a Group Permission

Situation: The Sales group has Modify permission on D:\SalesData\. However, employee bob (a Sales member) is under investigation and should only be able to read files, not modify them. How do you handle this without affecting other Sales members?

Show Solution 
REM Add explicit Deny Write for bob (blocks Modify's write component)
icacls D:\SalesData /deny bob:(OI)(CI)W

REM Also deny Delete to prevent file deletion
icacls D:\SalesData /deny bob:(OI)(CI)D

Explanation: Bob still receives the Allow Modify from the Sales group, but the explicit Deny Write and Deny Delete override the Allow. His effective access becomes Read & Execute only. Other Sales members are unaffected because the Deny applies only to bob's account. This is one of the few legitimate uses of explicit Deny entries.

Scenario 3: Troubleshooting "Access Denied"

Situation: User carol reports she cannot open files in D:\Projects\Reports\. She is a member of the ProjectTeam group, which has Modify permission on D:\Projects\. What troubleshooting steps should you follow?

Show Solution
  1. Check effective access using the GUI:
    • Right-click D:\Projects\Reports\ → Properties → Security → Advanced → Effective Access
    • Select carol's account and view effective permissions
  2. Check for Deny entries: 
    icacls D:\Projects\Reports

    Look for any Deny ACEs that might apply to carol or her groups

  3. Check inheritance:
    • Was inheritance broken on the Reports subfolder? If so, the ProjectTeam permission from the parent may not have propagated
  4. Check group membership: 
    # Verify carol is actually in ProjectTeam
Get-ADGroupMember -Identity "ProjectTeam" |
  Where-Object { $_.Name -eq "carol" }
  5. Check if carol logged out and back in after being added to the group (group membership is evaluated at logon)
  6. Check share permissions if accessing over the network — NTFS and share permissions combine (most restrictive wins)

Common causes: Broken inheritance, Deny entry from another group, group membership not refreshed (requires re-logon), or share-level permissions more restrictive than NTFS.

✓ Module 7.3 Summary

Key Takeaways

  • Windows NTFS uses Access Control Lists (ACLs) instead of Linux's fixed 12-bit model, allowing unlimited entries for any user or group.
  • Each ACL contains Access Control Entries (ACEs) that specify a security principal, an access type (Allow/Deny), and one or more permissions.
  • There are six basic NTFS permissions (Full Control, Modify, Read & Execute, List Folder Contents, Read, Write) built from thirteen advanced permissions.
  • Deny always overrides Allow — the single most important NTFS permission rule. The one exception: explicit permissions override inherited permissions of any type.
  • Allow permissions from multiple groups are cumulative (unlike Linux, where only one category applies).
  • Inheritance propagates parent folder permissions to children automatically. Breaking inheritance creates management overhead and should be avoided when possible.
  • Permissions can be managed via the GUI Security tab, icacls command-line utility, or PowerShell (Get-Acl, Set-Acl).
  • The Effective Access tab is essential for troubleshooting — it shows actual permissions after all rules are evaluated.

1 Introduction: Ownership & the Permission Puzzle

In Module 7.1, you learned that permissions answer the question "What can be done?" This module answers the equally important questions: "Who owns this?" and "What happens when normal permissions aren't enough?"

Recall from Module 7.1 that every file and directory has two ownership attributes stored in its inode:

  • Owner (UID): The single user account that owns the file
  • Group (GID): The single group associated with the file

These ownership attributes determine which set of permissions (owner, group, or others) applies to any given user. Changing ownership effectively changes who gets which permissions — without modifying the permission bits themselves.

Why Standard Permissions Sometimes Fall Short

The standard 9-bit permission model (rwx for owner, group, others) handles most scenarios. But certain real-world situations require capabilities beyond the basic model:

ProblemStandard Permissions Cannot…Solution
A regular user needs to change their own passwordAllow a user to write to /etc/shadow (owned by root)SetUID on /usr/bin/passwd
A team shares a project directory, but new files belong to individual users' primary groupsForce new files to inherit the directory's groupSetGID on the shared directory
In /tmp, any user can delete any other user's filesRestrict deletion to file owners only in a world-writable directorySticky bit on /tmp

These three problems are solved by the special permission bits — three additional bits stored in positions 9, 10, and 11 of the 12-bit permission field you learned about in Module 7.1.

The Complete 12-Bit Permission Model

12-Bit Permission Field in the Inode

SetUID (bit 11)
SetGID (bit 10)
Sticky (bit 9)
│
Owner r w x
Group r w x
Others r w x

Bits 11–9: Special permissions | Bits 8–0: Standard permissions (Module 7.1)

In numeric notation, special permissions are expressed as a fourth digit prepended to the standard three-digit octal number:

Special PermissionOctal Value4-Digit Example
SetUID40004755
SetGID20002755
Sticky Bit10001777
SetUID + SetGID60006755
None00000755 (same as 755)

2 Changing Ownership with chown & chgrp

Before exploring special permissions, you must master ownership management. The chown and chgrp commands change who owns a file and which group is associated with it.

chown — Change File Owner and Group
Syntax: chown [OPTIONS] OWNER[:GROUP] FILE...
Purpose: Changes the owner and optionally the group of files and directories.
Who can use it: Only root (superuser). Regular users cannot give away ownership of their files.
Man page: man chown
chgrp — Change Group Ownership
Syntax: chgrp [OPTIONS] GROUP FILE...
Purpose: Changes only the group associated with files and directories.
Who can use it: The file owner (if they are a member of the target group) or root.
Man page: man chgrp

2.1 Basic chown Usage

# Change owner only
sudo chown alice report.txt

# Change owner and group simultaneously
sudo chown alice:developers report.txt

# Change group only (note the leading colon)
sudo chown :developers report.txt

# Set owner; group defaults to owner's primary group
sudo chown alice: report.txt
📌 Colon vs. Period
Historically, chown user.group (with a period) was accepted. Modern systems prefer the colon (chown user:group) because usernames can contain periods. Always use the colon syntax.

2.2 Basic chgrp Usage

# Change group ownership (must be owner AND member of target group)
chgrp developers project-plan.md

# As root, change group to anything
sudo chgrp www-data /var/www/html/index.html
⚠️ Why Only Root Can chown?

If regular users could give away file ownership, they could:

  • Exceed disk quotas by transferring large files to other users
  • Create files that appear to be owned by root or other privileged accounts
  • Circumvent accountability and audit trails

2.3 Recursive Ownership Changes (-R)

# Change owner and group for an entire web directory tree
sudo chown -R webdev:www-data /var/www/html/

# Change only the group recursively
sudo chgrp -R developers /opt/project/

# Verbose mode: see each change
sudo chown -Rv alice:developers /shared/project/

2.4 Verifying Ownership Changes

# View ownership with ls -l
ls -l report.txt
# -rw-r----- 1 alice developers 2048 Mar 3 10:00 report.txt
#              ↑      ↑
#            owner   group

# View only numeric UID and GID
ls -ln report.txt
# -rw-r----- 1 1001 1002 2048 Mar 3 10:00 report.txt

2.5 Ownership and Permission Interaction

Changing ownership can immediately change who has access without modifying any permission bits:

Example: Ownership Change Shifts Access 
# File: -rw-r----- alice:developers
# alice=rw, developers=r, others=none

# Change group from developers to sysadmins
sudo chown :sysadmins secret.conf

# Now: -rw-r----- alice:sysadmins
# developers LOST access, sysadmins GAINED read access
# Permission bits didn't change at all!
💡 Common Pattern: Web Server Setup
# Step 1: Set ownership
sudo chown -R webdev:www-data /var/www/html/

# Step 2: Set directory permissions
sudo find /var/www/html -type d -exec chmod 750 {} \;

# Step 3: Set file permissions
sudo find /var/www/html -type f -exec chmod 640 {} \;

3 SetUID (4000) — Running as the File Owner

The Set User ID (SetUID) bit is one of the most powerful — and most dangerous — features of the Linux permission model. When set on an executable file, it causes the program to run with the permissions of the file's owner, not the user who launched it.

SetUID (Set User ID) — Octal 4000
Effect on executable files: The process temporarily gains the privileges of the file's owner. If the file is owned by root, the process runs with root privileges.
Effect on directories: No defined effect on most Linux systems (ignored).
Symbolic notation: chmod u+s file
In ls -l: An s replaces the x in the owner execute position. If execute is not set, it shows as S (uppercase — a misconfiguration).

3.1 The Classic Example: /usr/bin/passwd

ls -l /usr/bin/passwd
# -rwsr-xr-x 1 root root 68208 Mar 14 2023 /usr/bin/passwd
#    ↑
#    's' in owner execute position = SetUID is set

Why does passwd need SetUID?

  • When a user changes their password, passwd must write to /etc/shadow
  • /etc/shadow is owned by root with permissions 640 — only root can write to it
  • Without SetUID, a regular user running passwd would get "Permission denied"
  • With SetUID set and the file owned by root, passwd temporarily runs as root

SetUID Execution Flow: passwd

User alice runs passwd
→
Process runs as root (SetUID)
→
Writes to /etc/shadow
→
Process exits

3.2 Setting and Removing SetUID

# Set SetUID using symbolic notation
sudo chmod u+s /usr/local/bin/myapp

# Set SetUID using numeric notation (4 prepended)
sudo chmod 4755 /usr/local/bin/myapp

# Remove SetUID
sudo chmod u-s /usr/local/bin/myapp

# Remove SetUID with numeric notation
sudo chmod 0755 /usr/local/bin/myapp

3.3 How SetUID Appears in ls -l

Permission StringMeaningNumeric
-rwsr-xr-xSetUID set, owner execute set (correct)4755
-rwSr-xr-xSetUID set, owner execute NOT set (misconfiguration!)4655
-rwxr-xr-xNo SetUID, normal executable0755
🚨 Security Warning: SetUID Risks

SetUID is one of the most commonly exploited permission configurations:

  • Buffer overflow attacks: A vulnerability in a SetUID-root program can give an attacker root access
  • Privilege escalation: A compromised SetUID program gives attacker the file owner's privileges
  • Shell escapes: Some programs allow spawning a shell — if SetUID-root, the shell runs as root

Best practices:

  • Never set SetUID on shell scripts (most Linux systems ignore it for scripts)
  • Minimize the number of SetUID programs on any system
  • Regularly audit for unexpected SetUID files (Section 6)
  • Use sudo instead of SetUID when possible — better logging and granular control

3.4 Common SetUID Programs

ProgramPathWhy SetUID?
passwd/usr/bin/passwdWrites to /etc/shadow (root-owned)
su/usr/bin/suSwitches user identity
sudo/usr/bin/sudoExecutes commands as another user
ping/usr/bin/pingCreates raw network sockets*
mount/usr/bin/mountMounts filesystems
chsh/usr/bin/chshChanges login shell (modifies /etc/passwd)

*Modern Linux often uses capabilities instead of SetUID for programs like ping, granting only the specific privilege needed.

4 SetGID (2000) — Group Inheritance & Execution

The Set Group ID (SetGID) bit has different effects on executable files versus directories. Its most important use case is on directories, where it solves the group collaboration problem.

SetGID (Set Group ID) — Octal 2000
On executable files: The process runs with the GID of the file's group, not the user's primary group.
On directories: New files and subdirectories inherit the directory's group instead of the creator's primary group. Subdirectories also inherit the SetGID bit.
Symbolic notation: chmod g+s file_or_dir
In ls -l: An s replaces the x in the group execute position. If execute is not set, shows as S.

4.1 The Problem SetGID Solves on Directories

Without SetGID, new files inherit the creator's primary group:

# alice's primary group is "alice", bob's is "bob"
# Both are members of "developers" group
# /opt/project is owned by root:developers

# alice creates a file:
touch /opt/project/alice-file.txt
ls -l /opt/project/alice-file.txt
# -rw-r--r-- 1 alice alice 0 Mar 3 10:00 alice-file.txt
#                    ↑ alice's primary group, NOT developers!

4.2 The SetGID Solution

# Set SetGID on the shared directory
sudo chmod g+s /opt/project/
# OR: sudo chmod 2775 /opt/project/

ls -ld /opt/project/
# drwxrwsr-x 2 root developers 4096 Mar 3 10:00 /opt/project/
#        ↑ 's' = SetGID

# Now alice creates a file:
touch /opt/project/alice-file.txt
ls -l /opt/project/alice-file.txt
# -rw-r--r-- 1 alice developers 0 Mar 3 10:00 alice-file.txt
#                    ↑ inherited "developers" from the directory!

4.3 Complete Shared Directory Setup

# Step 1: Create the directory
sudo mkdir -p /opt/project

# Step 2: Set ownership
sudo chown root:developers /opt/project

# Step 3: Set permissions with SetGID (2=SetGID, 770=rwxrwx---)
sudo chmod 2770 /opt/project

# Step 4: Verify
ls -ld /opt/project
# drwxrws--- 2 root developers 4096 Mar 3 10:00 /opt/project
💡 SetGID + umask for Full Collaboration

SetGID ensures correct group, but file permissions still follow the creator's umask. With default umask 022, new files are 644 (no group write). For full collaboration, use umask 002:

# Add to team members' ~/.bashrc
umask 002

4.4 SetGID on Executable Files

# Example: the 'wall' command
ls -l /usr/bin/wall
# -rwxr-sr-x 1 root tty 19024 Mar 14 2023 /usr/bin/wall
#         ↑ SetGID — runs with group 'tty'

4.5 How SetGID Appears in ls -l

Permission StringMeaningNumeric
drwxrwsr-xDirectory: SetGID set, group execute set2775
drwxrwS---Directory: SetGID set, group execute NOT set2760
-rwxr-sr-xFile: SetGID set, group execute set2755

5 Sticky Bit (1000) — Protecting Shared Directories

The sticky bit solves a specific problem: in a world-writable directory, how do you prevent users from deleting each other's files?

Sticky Bit — Octal 1000
On directories: Only the file's owner, the directory's owner, or root can delete or rename files within the directory. Other users with write permission on the directory cannot delete files they don't own.
On files: No meaningful effect on modern Linux.
Symbolic notation: chmod +t directory or chmod o+t directory
In ls -l: A t replaces the x in others' execute position. If execute is not set, shows as T.

5.1 The Problem: World-Writable Directories

# Without sticky bit on /tmp (permissions 777)
# alice creates a temporary file
touch /tmp/alice-temp.txt

# bob can delete alice's file! (he has write on /tmp)
rm /tmp/alice-temp.txt
# Deleted! This is a problem.

5.2 The Sticky Bit Solution

# View /tmp permissions
ls -ld /tmp
# drwxrwxrwt 15 root root 4096 Mar 3 10:00 /tmp
#          ↑ 't' = sticky bit

# With sticky bit: bob CANNOT delete alice's file
rm /tmp/alice-temp.txt
# rm: cannot remove '/tmp/alice-temp.txt': Operation not permitted

5.3 Setting and Removing the Sticky Bit

# Set sticky bit — symbolic
sudo chmod +t /shared/uploads/

# Set sticky bit — numeric
sudo chmod 1777 /shared/uploads/

# Remove sticky bit
sudo chmod -t /shared/uploads/

5.4 How the Sticky Bit Appears in ls -l

Permission StringMeaningNumeric
drwxrwxrwtSticky set, others execute set (e.g., /tmp)1777
drwxrwxrwTSticky set, others execute NOT set1776
drwxrwxrwxNo sticky bit — anyone can delete anything0777

5.5 Sticky Bit Deletion Rules Summary

When the sticky bit is set, a file inside can be deleted or renamed only by:

  1. The file's owner
  2. The directory's owner
  3. The root user
📌 Real-World Sticky Bit Directories
  • /tmp — Temporary files (all users need write, shouldn't delete each other's)
  • /var/tmp — Persistent temporary files (survive reboots)
  • /var/mail — User mailboxes on some systems
  • Any custom shared upload or drop-box directory

6 Finding & Auditing Special Permissions

Security auditing requires the ability to find files with special permissions. The find command with the -perm flag is the primary tool for this task.

6.1 Finding SetUID Files

# Find ALL SetUID files on the system
sudo find / -perm -4000 -type f 2>/dev/null

# With detailed output
sudo find / -perm -4000 -type f -exec ls -l {} \; 2>/dev/null

6.2 Finding SetGID Files and Directories

# Find SetGID files
sudo find / -perm -2000 -type f 2>/dev/null

# Find SetGID directories
sudo find / -perm -2000 -type d 2>/dev/null

6.3 Finding Sticky Bit Directories

sudo find / -perm -1000 -type d 2>/dev/null

6.4 Understanding the -perm Flag

SyntaxMeaningExample
-perm -4000At least these bits set (AND match)Find files where SetUID is set (other bits may be anything)
-perm 4755Exact match onlyFind files with exactly 4755
-perm /6000Any of these bits set (OR match)Find files with SetUID OR SetGID (or both)

6.5 Security Audit Script

#!/bin/bash
# special-perms-audit.sh

echo "===== SetUID Files ====="
find / -perm -4000 -type f -exec ls -lh {} \; 2>/dev/null

echo ""
echo "===== SetGID Files ====="
find / -perm -2000 -type f -exec ls -lh {} \; 2>/dev/null

echo ""
echo "===== SetGID Directories ====="
find / -perm -2000 -type d -exec ls -ld {} \; 2>/dev/null

echo ""
echo "===== World-Writable Dirs (no sticky) ====="
find / -perm -0002 -type d ! -perm -1000 \
  ! -path "/proc/*" ! -path "/sys/*" \
  -exec ls -ld {} \; 2>/dev/null
🚨 Red Flags in Audit Results
  • Unexpected SetUID files — Any SetUID file not in the standard list could indicate a backdoor or compromise
  • SetUID on scripts — Should never exist; likely an attack or misconfiguration
  • SetUID files in user home directories — Highly suspicious; investigate immediately
  • SetUID files in /tmp or /var/tmp — Almost certainly malicious
  • World-writable directories without sticky bit — Any user can delete any file; add sticky bit or restrict write

Scenario 1: Setting Up a Shared Development Directory

Situation: Your team has three developers: alice, bob, and carol. They all belong to the devteam group. You need to create /opt/webapp/ where all three can create and edit files, new files automatically belong to devteam, and no one outside the group has any access.

Task: Write the complete sequence of commands.

Show Solution 
# Create the directory
sudo mkdir -p /opt/webapp

# Set ownership: root owns it, devteam group
sudo chown root:devteam /opt/webapp

# Set permissions: SetGID (2) + rwx for owner and group, none for others (770)
sudo chmod 2770 /opt/webapp

# Verify
ls -ld /opt/webapp
# drwxrws--- 2 root devteam 4096 ... /opt/webapp

# Ensure team members use umask 002 for group-writable files
# Add to each user's ~/.bashrc:
umask 002

Explanation: SetGID (2) ensures new files inherit devteam group. Permissions 770 give owner and group full access, others none. The umask 002 ensures new files are group-writable (664 for files, 775 for directories).

Scenario 2: Creating a Secure Upload Directory

Situation: You need to create /var/uploads/ where all users can upload files, but users should only be able to delete their own files (not each other's).

Show Solution 
sudo mkdir -p /var/uploads
sudo chown root:root /var/uploads
sudo chmod 1777 /var/uploads

ls -ld /var/uploads
# drwxrwxrwt 2 root root 4096 ... /var/uploads

Explanation: Permissions 777 let everyone create files. The sticky bit (1) prevents users from deleting each other's files. This is the same pattern used by /tmp.

Scenario 3: Investigating a Suspicious SetUID File

Situation: During a security audit, you discover an unexpected SetUID file: -rwsr-xr-x 1 root root 45678 /home/bob/.hidden/exploit. What steps should you take?

Show Solution
  1. Immediately remove the SetUID bit to prevent further exploitation: 
    sudo chmod u-s /home/bob/.hidden/exploit
  2. Investigate the file: 
    # Check file type
file /home/bob/.hidden/exploit

# Check timestamps
stat /home/bob/.hidden/exploit

# Check if it matches any known package
dpkg -S /home/bob/.hidden/exploit
  3. Check system logs for related activity: 
    sudo grep "bob" /var/log/auth.log | tail -50
  4. Quarantine the file (move, don't delete, to preserve evidence): 
    sudo mv /home/bob/.hidden/exploit /root/quarantine/
  5. Notify your security team and document the incident
  6. Run a full audit for other suspicious files: 
    sudo find /home -perm -4000 -type f 2>/dev/null

Key point: A SetUID-root binary in a user's home directory is almost certainly malicious. Legitimate SetUID programs are installed in system directories like /usr/bin/.

Scenario 4: Fixing a Web Server Ownership Problem

Situation: A junior admin accidentally ran sudo chown -R root:root /var/www/html/. Now the web server (www-data) cannot read the files and the developer (webdev) cannot edit them. Fix the ownership and permissions.

Show Solution 
# Step 1: Restore correct ownership
sudo chown -R webdev:www-data /var/www/html/

# Step 2: Set directory permissions (owner rwx, group r-x, others none)
sudo find /var/www/html -type d -exec chmod 750 {} \;

# Step 3: Set file permissions (owner rw, group r, others none)
sudo find /var/www/html -type f -exec chmod 640 {} \;

# Step 4: Verify
ls -la /var/www/html/

Explanation: Owner webdev gets full access. Group www-data gets read (and traverse for directories). Others get nothing. Using find with -type ensures different permissions for files vs. directories.

✓ Module 7.2 Summary

Key Takeaways

  • chown changes file owner and/or group; only root can change ownership. chgrp changes only the group; the file owner can use it if they belong to the target group.
  • Both commands support -R for recursive changes across directory trees.
  • Changing ownership shifts who gets which permissions without modifying permission bits.
  • SetUID (4000) on an executable causes it to run as the file owner — critical for programs like passwd but a major security risk if misused.
  • SetGID (2000) on a directory causes new files to inherit the directory's group — essential for team collaboration.
  • The sticky bit (1000) on a directory restricts deletion to the file owner, directory owner, or root — protecting shared directories like /tmp.
  • Uppercase letters (S, S, T) in ls -l indicate the special bit is set without the corresponding execute bit — usually a misconfiguration.
  • Regular security audits using find -perm are essential for detecting unauthorized special permissions.

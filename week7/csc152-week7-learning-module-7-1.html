1 Introduction: Why Permissions Matter

Permissions are the first line of defense in a multi-user operating system. They determine who can read, modify, or execute every file and directory on the system.

Linux is inherently a multi-user operating system. On any given server, dozens or even thousands of user accounts may exist. These include human administrators, application service accounts, and system daemons. Without a robust permission model, any user could read another user's private files, modify critical system configurations, or execute malicious programs.

Real-World Scenarios

  • Web Server: The web server process (running as user www-data) needs to read HTML files but should never be able to write to them — otherwise a vulnerability could allow an attacker to deface the website.
  • SSH Keys: Your private SSH key (~/.ssh/id_rsa) must be readable only by you. SSH will refuse to use a private key file that other users can read.
  • Shared Project Directory: A development team needs a directory where all team members can create and edit files, but users outside the team should have no access at all.
  • Log Files: System log files should be readable by administrators for troubleshooting but not writable by regular users who might try to cover their tracks.
⚠️ Security Principle
Incorrect permissions are one of the most common causes of security vulnerabilities on Linux servers. The principle of least privilege states that every user and process should have only the minimum permissions necessary to perform their function — nothing more.

The Three Questions

Every time Linux checks whether an action is allowed, it answers three questions in order:

  1. Who is trying to perform the action? (Which user and what groups do they belong to?)
  2. What are they trying to do? (Read, write, or execute?)
  3. Which category do they fall into for this particular file? (Owner, group member, or other?)
📌 Connection to Week 6
This module builds directly on your understanding of users and groups from Week 6. The UIDs and GIDs you learned about are exactly what the kernel uses to evaluate file permissions. If you haven't completed Week 6, review user and group concepts before proceeding.

2 The Linux Permission Model

Linux uses a straightforward but powerful model: three types of permissions applied across three categories of users, stored as metadata for every file and directory on the system.

2.1 Permission Types

Read (r)
For files: Allows viewing the contents of the file (e.g., using cat, less, head, or opening in an editor). For directories: Allows listing the names of files within the directory (e.g., using ls). Octal value: 4
Write (w)
For files: Allows modifying, overwriting, or truncating the file's contents. For directories: Allows creating new files, deleting files, and renaming files within the directory. Octal value: 2
Execute (x)
For files: Allows running the file as a program or script. For directories: Allows entering the directory with cd and accessing file metadata (inodes). This is sometimes called the "search" permission. Octal value: 1
💡 Memory Aid
Think of the octal values as a powers-of-two pattern: Read = 4 (2²), Write = 2 (2¹), Execute = 1 (2⁰). Each permission is a single bit in a 3-bit binary number.

2.2 Permission Categories

Owner (u)
The single user account that "owns" the file. By default, this is the user who created the file. The owner is identified by the file's UID (User ID) metadata. The letter u (for "user") is used in symbolic notation.
Group (g)
The single group associated with the file. Every file belongs to exactly one group. All members of that group receive the group permissions. By default, this is the primary group of the user who created the file. The letter g is used in symbolic notation.
Others (o)
Everyone else — any user account on the system that is neither the owner nor a member of the file's group. The letter o is used in symbolic notation. This is sometimes called "world" permissions.
All (a)
A shorthand in symbolic notation representing all three categories simultaneously: owner + group + others. The letter a is used in chmod commands when you want to change permissions for everyone at once.

2.3 How Linux Evaluates Permissions

When a user attempts to access a file, the kernel follows a strict evaluation order:

  1. Is the user root (UID 0)? → If yes, access is granted for read and write operations regardless of permissions. (Root can also execute a file if any execute bit is set.)
  2. Is the user the file's owner? → If yes, only the owner permissions are checked. Group and other permissions are ignored — even if they are more permissive.
  3. Is the user a member of the file's group? → If yes, only the group permissions are checked.
  4. Otherwise → The "others" permissions are checked.
🚨 Critical Concept: No Accumulation
Permissions do not accumulate across categories. If a file has permissions ---rwx---, the owner has no access at all, even though group members have full access. The kernel stops at the first matching category. This is a common source of confusion and a frequent exam topic.
Example: Permission Evaluation Walkthrough

Consider a file with the following attributes:

-rw-r----- 1 alice developers 2048 Mar 2 10:00 report.txt
  • User alice is the owner → owner permissions apply: rw- → can read and write, cannot execute.
  • User bob (member of developers group) → group permissions apply: r-- → can read only.
  • User charlie (not owner, not in developers) → others permissions apply: --- → no access at all.
  • User root → bypasses all permission checks → full access.

Even if alice is also a member of the developers group, she gets the owner permissions (rw-), not the group permissions (r--). The kernel checks owner first and stops there.

2.4 The Permission Bits in Memory

Internally, Linux stores permissions as a 12-bit field in the file's inode (the data structure that stores file metadata). The lower 9 bits represent the standard permissions:

Bit Position876543210
Category Owner Group Others
Permission rwxrwxrwx
Octal Value 421421421

The upper 3 bits (positions 9, 10, 11) store special permissions (SetUID, SetGID, sticky bit), which you will learn about in Module 7.2.

Practice: Read These Permission Strings

1. -rw-r--r--

Show Answer

Regular file. Owner: read + write. Group: read only. Others: read only. Numeric: 644. Typical for configuration files and documents.

2. drwxr-xr-x

Show Answer

Directory. Owner: read + write + execute. Group: read + execute. Others: read + execute. Numeric: 755. Standard for system directories.

3. -rwx------

Show Answer

Regular file. Owner: full access. Group: none. Others: none. Numeric: 700. Appropriate for private scripts.

4. lrwxrwxrwx

Show Answer

Symbolic link. Permissions shown as 777 — but actual access is controlled by the permissions on the target file the link points to.

5. -rw-rw----

Show Answer

Regular file. Owner: read + write. Group: read + write. Others: none. Numeric: 660. Useful for team-shared files.

4 Symbolic & Numeric (Octal) Notation

Linux provides two ways to express permissions: symbolic notation using letters and operators, and numeric (octal) notation using a three- or four-digit number. A competent administrator must be fluent in both.

4.1 Symbolic Notation

ComponentCharactersMeaning
WhouOwner (user)
gGroup
oOthers
aAll (owner + group + others)
Operator+Add permission(s)
-Remove permission(s)
=Set exact permission(s), removing all others
WhatrRead
wWrite
xExecute

Symbolic Notation Examples

# Add execute permission for the owner
chmod u+x script.sh

# Remove write permission from group and others
chmod go-w config.txt

# Set read-only for everyone
chmod a=r readme.txt

# Give owner read+write, group read, others nothing
chmod u=rw,g=r,o= document.txt

# Add execute for owner and group simultaneously
chmod ug+x deploy.sh
💡 When to Use Symbolic Notation
Symbolic notation is ideal when you want to change specific permissions without affecting others. For example, chmod u+x script.sh adds execute for the owner while leaving all other permissions unchanged.

4.2 Numeric (Octal) Notation

Each digit is the sum of the permission values for one category:

PermissionsCalculationOctal Digit
rwx4 + 2 + 17
rw-4 + 2 + 06
r-x4 + 0 + 15
r--4 + 0 + 04
-wx0 + 2 + 13
-w-0 + 2 + 02
--x0 + 0 + 11
---0 + 0 + 00
Worked Example: Converting rwxr-xr-- to Octal

Step 1: Split into three groups:

  • Owner: rwx → 4 + 2 + 1 = 7
  • Group: r-x → 4 + 0 + 1 = 5
  • Others: r-- → 4 + 0 + 0 = 4

Result: 754

4.3 Common Permission Sets

NumericSymbolicTypical Use Case
755rwxr-xr-xExecutable scripts, web directories, system binaries
644rw-r--r--Configuration files, web content, documents
700rwx------Private scripts, user home directories
600rw-------SSH private keys, sensitive configuration files

640rw-r-----Group-readable configuration files, log files
444r--r--r--Read-only files that should not be modified

⚠️ When to Use Numeric Notation
Numeric notation sets all nine permission bits at once, replacing whatever was there before. Use it when you know exactly what the final permissions should be.

5 The chmod Command

The chmod (change mode) command is the primary tool for modifying file and directory permissions on Linux. It supports both symbolic and numeric notation.

chmod — Change File Mode Bits
Syntax: chmod [OPTIONS] MODE FILE...
Purpose: Changes the permission bits of files and directories.
Who can use it: The file owner or root (superuser).
Man page: man chmod

5.1 Using chmod with Symbolic Notation

# Make a script executable by the owner
chmod u+x backup.sh

# Remove write permission from group and others
chmod go-w /etc/config.conf

# Set exact permissions: owner=rwx, group=rx, others=none
chmod u=rwx,g=rx,o= /opt/app/run.sh

# Multiple operations in one command (comma-separated)
chmod u+x,g-w,o-rwx sensitive_script.sh

5.2 Using chmod with Numeric Notation

# Standard web content permissions
chmod 644 /var/www/html/index.html

# Executable script with group read+execute
chmod 750 /opt/scripts/deploy.sh

# Secure SSH private key
chmod 600 ~/.ssh/id_rsa

# Private home directory
chmod 700 /home/alice

5.3 The -R (Recursive) Option

# Apply 755 to a directory and everything inside it
chmod -R 755 /var/www/html/
🚨 Caution with Recursive chmod

Using chmod -R applies the same permissions to both files and directories. This can cause problems because files and directories often need different permissions. For example, chmod -R 644 would remove execute from directories, making them inaccessible via cd.

Better approach — use find to target files and directories separately:

# Set directories to 755
find /var/www/html -type d -exec chmod 755 {} \;

# Set files to 644
find /var/www/html -type f -exec chmod 644 {} \;

5.4 The -v (Verbose) and Other Useful Options

# Verbose: reports each permission change
chmod -v 644 report.txt
# Output: mode of 'report.txt' changed from 0755 (rwxr-xr-x) to 0644 (rw-r--r--)

5.5 Verifying Permission Changes

# Check permissions before
ls -l deploy.sh
# -rw-r--r-- 1 alice developers 512 Mar 2 10:00 deploy.sh

# Add execute for owner
chmod u+x deploy.sh

# Verify after
ls -l deploy.sh
# -rwxr--r-- 1 alice developers 512 Mar 2 10:00 deploy.sh
💡 Pro Tip: stat Command
# View detailed permission info
stat deploy.sh

# Show just octal permissions
stat -c '%a %n' deploy.sh
# Output: 744 deploy.sh

6 File vs. Directory Permissions

The same three permission types — read, write, execute — have fundamentally different meanings when applied to files versus directories. This distinction is one of the most common sources of confusion for new Linux administrators.

6.1 Side-by-Side Comparison

PermissionOn a FileOn a Directory
Read (r) View file contents (cat, less, head, grep) List filenames in the directory (ls). Without execute, you can see names but not details.
Write (w) Modify, overwrite, or truncate the file's contents Create, delete, or rename files within the directory. You can delete a file you cannot read if you have write on the directory.
Execute (x) Run the file as a program or script Enter the directory (cd) and access file metadata (inodes). Sometimes called the "search" or "traverse" permission.

6.2 Critical Directory Permission Combinations

PermissionsCan ls?Can cd?Can Create/Delete?Can Access Files?
r--✅ Names only❌❌❌
-w-❌❌❌❌
--x❌✅❌✅ If you know the filename
r-x✅ Full details✅❌✅
rwx✅ Full details✅✅✅
-wx❌✅✅✅ If you know the filename
⚠️ Key Insight: Execute on Directories

The execute permission on a directory is essential for almost any useful operation. Without it, you cannot cd into the directory, and you cannot access any files inside it. This is why directories almost always have the execute bit set for categories that need access.

Demonstration: Directory Permission Behavior 
# Create a test directory with a file inside
mkdir /tmp/permtest
echo "secret data" > /tmp/permtest/data.txt
chmod 644 /tmp/permtest/data.txt

# Test: read-only directory (r--)
chmod 400 /tmp/permtest
ls /tmp/permtest/
# Shows filenames but with errors — cannot access metadata
cat /tmp/permtest/data.txt
# Permission denied — no execute bit

# Test: execute-only directory (--x)
chmod 100 /tmp/permtest
ls /tmp/permtest/
# Permission denied — cannot list
cat /tmp/permtest/data.txt
# Works! — if you know the exact filename

# Test: read + execute (r-x)
chmod 500 /tmp/permtest
ls -l /tmp/permtest/
# Full listing works

# Clean up
chmod 755 /tmp/permtest
rm -rf /tmp/permtest

6.3 Deleting Files: A Surprising Behavior

Deleting a file depends on the directory's write permission, not the file's permissions. This is because deleting a file is actually modifying the directory (removing the filename entry), not modifying the file itself.

# alice creates a file with no permissions
touch /shared/alice-file.txt
chmod 000 /shared/alice-file.txt

# bob has write permission on /shared/ directory
rm /shared/alice-file.txt  # bob CAN delete the file!

The sticky bit (covered in Module 7.2) was created specifically to address this behavior in shared directories like /tmp.

7 Default Permissions & umask

When you create a new file or directory, Linux does not assign arbitrary permissions. Instead, it uses a system of base permissions and a umask value to calculate the default permissions automatically.

umask — User File-Creation Mode Mask
The umask is a value that specifies which permission bits should be removed (masked off) from the base permissions when creating new files and directories. It acts as a filter, not an assignment.

7.1 Base Permissions

Object TypeBase PermissionsOctalReason
Filesrw-rw-rw-666Files are never created with execute by default (security)
Directoriesrwxrwxrwx777Directories need execute for cd access

7.2 How umask Works

Default Permissions = Base Permissions AND (NOT umask)

umaskNew File (666 base)New Directory (777 base)
022644 (rw-r--r--)755 (rwxr-xr-x)
027640 (rw-r-----)750 (rwxr-x---)
077600 (rw-------)700 (rwx------)
002664 (rw-rw-r--)775 (rwxrwxr-x)
000666 (rw-rw-rw-)777 (rwxrwxrwx)

7.3 Viewing and Setting umask

# View current umask (numeric)
umask
# Output: 0022

# View current umask (symbolic)
umask -S
# Output: u=rwx,g=rx,o=rx

# Set umask for current session
umask 027

# Test: create a file and directory
touch testfile.txt
mkdir testdir
ls -la
# testfile.txt: -rw-r----- (640)
# testdir:      drwxr-x--- (750)

7.4 Making umask Persistent

# For a single user — add to ~/.bashrc or ~/.profile
echo 'umask 027' >> ~/.bashrc

# For all users system-wide — edit /etc/login.defs
sudo nano /etc/login.defs
# Find and modify: UMASK  027
📌 Common umask Values in Practice
  • 022 — Most common default. Files: 644, Dirs: 755. Others can read but not write.
  • 027 — More restrictive. Files: 640, Dirs: 750. Others get no access.
  • 077 — Most restrictive. Files: 600, Dirs: 700. Only the owner has access.
  • 002 — Collaborative. Files: 664, Dirs: 775. Group members can write.
💡 Why Not Just Subtract?

While "subtracting" works for most common cases, it's technically inaccurate. The umask uses a bitwise AND with the complement. Example with umask 033 and file base 666:

  • Subtraction: 666 − 033 = 633 ← Wrong!
  • Bitwise: 666 AND NOT(033) = 644 ← Correct!

The umask can only remove bits, never add them. Since the file base (666) has no execute bit, no umask value can produce execute on new files.

Scenario 1: Securing an SSH Private Key

Situation: Your private key ~/.ssh/id_rsa has permissions -rw-r--r--. SSH refuses to use it, displaying: "Permissions 0644 are too open."

Task: What chmod command fixes this?

Show Solution 
chmod 600 ~/.ssh/id_rsa

SSH private keys must be 600 (rw-------) — readable only by the owner.

Scenario 2: Web Server Document Root

Situation: Configure /var/www/html/ so www-data can read files and traverse directories but not modify content. The developer webdev needs full access.

Show Solution 
sudo chown -R webdev:www-data /var/www/html/

# Directories: 750
sudo find /var/www/html -type d -exec chmod 750 {} \;

# Files: 640
sudo find /var/www/html -type f -exec chmod 640 {} \;

Owner (webdev) gets full access. Group (www-data) gets read + traverse. Others get nothing.

Scenario 3: Making a Script Executable

Situation: backup.sh has -rw-r--r--. Running ./backup.sh gives "Permission denied."

Show Solution 
chmod u+x backup.sh

Adds only execute for the owner. Result: -rwxr--r-- (744). Using symbolic notation preserves existing permissions.

Scenario 4: Calculating umask for Security Policy

Situation: Policy requires new files to be rw-r----- (640) and new directories to be rwxr-x--- (750).

Show Solution

File: 666 − 640 = 026. Directory: 777 − 750 = 027. Use the larger value: umask 027.

umask 027

Verify: 666 AND NOT(027) = 640 ✅ | 777 AND NOT(027) = 750 ✅

✓ Module 7.1 Summary

Key Takeaways

  • Linux uses three permission types (read, write, execute) applied to three categories (owner, group, others).
  • Permissions are stored as 9 bits in the file's inode, plus 3 additional bits for special permissions.
  • The kernel evaluates permissions in strict order: root → owner → group → others. Permissions do not accumulate.
  • Symbolic notation (chmod u+x) is best for targeted changes; numeric notation (chmod 755) sets all bits at once.
  • Permissions mean different things on files versus directories — especially execute.
  • The umask controls default permissions by masking bits from base permissions (666 for files, 777 for directories).
  • Deleting a file requires write permission on the directory, not on the file itself.

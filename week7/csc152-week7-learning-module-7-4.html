1 Systematic Troubleshooting Methodology

"Access Denied" is one of the most common errors system administrators encounter. A systematic approach prevents wasted time and ensures you identify the root cause rather than applying band-aid fixes.

Systematic Troubleshooting
A structured, repeatable process for identifying and resolving problems. For permission issues, this means gathering information, forming hypotheses, testing them methodically, and documenting the resolution — rather than randomly changing permissions until something works.

1.1 The Permission Troubleshooting Flowchart

When a user reports "Access Denied," follow these steps in order:

1 Reproduce the Problem
What exactly is the user trying to do? Which file/folder? What error message? Can you reproduce it?
↓
2 Identify the User Context
Who is the user? What groups are they in? Are they using the correct account? Have they logged out and back in since group changes?
↓
3 Examine Current Permissions
What are the actual permissions on the file/folder? Who owns it? What group is assigned? Is inheritance involved?
↓
4 Check the Full Path
Does the user have execute (Linux) or traverse (Windows) permission on every directory in the path from root to the target?
↓
5 Check for Deny/Override
Linux: Is the user matching the wrong category (owner vs. group)? Windows: Are there explicit Deny ACEs? Broken inheritance?
↓
6 Review Logs
Check system logs for permission-denied events. Logs often reveal exactly what was denied and why.
↓
7 Apply Fix & Verify
Make the minimum change necessary (least privilege). Test that the fix works. Verify no unintended access was granted.
↓
8 Document
Record what the problem was, what caused it, and how it was resolved. This builds institutional knowledge.
⚠️ Anti-Pattern: "chmod 777"

When faced with "Access Denied," the temptation is to run chmod 777 or grant "Everyone: Full Control." This always works — and is always wrong. It removes all access control, violates least privilege, and creates security vulnerabilities. A professional administrator finds the specific permission that's missing and grants only that.

💡 The namei Command

The namei -l command is invaluable for Linux troubleshooting. It shows permissions for every component in a path:

namei -l /opt/project/src/main.py
f: /opt/project/src/main.py
drwxr-xr-x root    root    /
drwxr-xr-x root    root    opt
drwxr-x--- root    devteam project
drwxrwx--- alice   devteam src
-rw-r--r-- alice   devteam main.py

This immediately reveals if any directory in the path is blocking access. In this example, /opt/project has r-x for the group but --- for others — a user not in devteam would be blocked at this level.

2 Linux Permission Troubleshooting

Linux permission problems follow predictable patterns. Learning to recognize these patterns lets you diagnose issues quickly.

2.1 The Five Most Common Linux Permission Problems

Problem 1: Missing Execute on Directories

Symptom
User can see filenames with ls but gets "Permission denied" when trying to cd into the directory or read any file inside it.
# User bob tries to access a file
cd /shared/project/
bash: cd: /shared/project/: Permission denied

# Check the directory permissions
ls -ld /shared/project/
drw-r--r-- 2 alice developers 4096 Mar 3 10:00 /shared/project/
#  ↑ No execute bit on the directory!

# Fix: Add execute for owner, group, and others
sudo chmod a+X /shared/project/
# Note: uppercase X adds execute only to directories, not files

Root cause: Someone used chmod -R 644 which removed execute from directories.

Prevention: Use find -type d and find -type f to set different permissions for directories and files.

Problem 2: Incorrect Ownership

Symptom
User gets "Permission denied" even though the file appears to have the right permissions. The user is falling into the "others" category because they are not the owner and not in the file's group.
# User bob tries to edit a file
nano /shared/config.txt
[ Error writing /shared/config.txt: Permission denied ]

# Check permissions and ownership
ls -l /shared/config.txt
-rw-rw---- 1 alice alice 256 Mar 3 10:00 /shared/config.txt

# Check bob's groups
id bob
uid=1002(bob) gid=1002(bob) groups=1002(bob),1010(developers)
# bob is NOT in group "alice" — he falls into "others" (---)

# Fix: Change the group to one bob belongs to
sudo chgrp developers /shared/config.txt

Problem 3: Missing Group Membership

Symptom
User was recently added to a group but still cannot access group-owned resources. The user has not logged out and back in since the group change.
# Admin added carol to developers group
sudo usermod -aG developers carol

# carol tries to access (in her current session)
cat /opt/project/README.md
cat: /opt/project/README.md: Permission denied

# Check carol's CURRENT groups (in this session)
id
uid=1003(carol) gid=1003(carol) groups=1003(carol)
# "developers" is NOT listed — session hasn't refreshed!

# Fix: carol must log out and log back in
# OR use newgrp to activate the group in current session:
newgrp developers
id
uid=1003(carol) gid=1010(developers) groups=1003(carol),1010(developers)
🚨 Critical: Group Changes Require Re-Login

On Linux, group membership is evaluated at login time. When you add a user to a group with usermod -aG, the change does not take effect until the user starts a new session. For SSH sessions, they must disconnect and reconnect. For GUI sessions, they must log out and log back in. The newgrp command is a temporary workaround for the current shell only.

Problem 4: Owner Category Override

Symptom
The file owner has less access than group members. The user is confused because they "should" have access through the group, but the kernel uses owner permissions instead.
# alice owns a file but can't read it
ls -l report.txt
----rwx--- 1 alice developers 1024 Mar 3 10:00 report.txt
# Owner: --- (no access), Group: rwx (full access)

# alice IS in the developers group, but...
cat report.txt
cat: report.txt: Permission denied
# The kernel checks owner FIRST and stops — no accumulation!

# Fix: Grant appropriate owner permissions
chmod u+rwx report.txt

Key concept: As covered in Module 7.1, Linux permissions do not accumulate. The kernel matches the first applicable category (owner → group → others) and stops.

Problem 5: Parent Directory Blocking Access

Symptom
The target file has correct permissions, but a parent directory in the path blocks access. The user gets "Permission denied" even though they can see the file has the right permissions.
# File has correct permissions
ls -l /data/secret/report.txt
-rw-r--r-- 1 alice developers 512 Mar 3 10:00 report.txt

# But the parent directory blocks access
namei -l /data/secret/report.txt
f: /data/secret/report.txt
drwxr-xr-x root root     /
drwxr-xr-x root root     data
drwx------ root root     secret    ← BLOCKED HERE (700 = owner only)
-rw-r--r-- alice developers report.txt

# Fix: Grant appropriate access to the parent directory
sudo chmod o+x /data/secret/
# Or better: change group and use group permissions
sudo chgrp developers /data/secret/
sudo chmod 750 /data/secret/

3 Windows Permission Troubleshooting

Windows permission issues are often more complex than Linux because of ACL inheritance, Deny entries, and the interaction between NTFS and share permissions. The Effective Access tool is your most powerful diagnostic weapon.

3.1 Common Windows Permission Problems

Problem 1: Deny Overriding Allow

REM User reports "Access Denied" on D:\Reports
REM Check permissions
icacls D:\Reports
D:\Reports YOURSERVER\Marketing:(OI)(CI)(M)
           YOURSERVER\AllStaff:(OI)(CI)(DENY)(W)
           BUILTIN\Administrators:(OI)(CI)(F)

REM User is in both Marketing (Allow Modify) and AllStaff (Deny Write)
REM Deny Write overrides the Allow Modify's write component
REM Effective access: Read & Execute only

Fix: Remove the Deny ACE if it's not needed, or remove the user from the group that has the Deny. Avoid Deny entries whenever possible.

Problem 2: Broken Inheritance

REM Parent folder D:\Projects has Marketing:(OI)(CI)(M)
REM But child folder D:\Projects\Archive does not inherit
icacls D:\Projects\Archive
D:\Projects\Archive BUILTIN\Administrators:(OI)(CI)(F)
                    NT AUTHORITY\SYSTEM:(OI)(CI)(F)
REM Notice: Marketing group is MISSING — inheritance was broken

REM Fix: Re-enable inheritance
icacls D:\Projects\Archive /inheritance:e
REM /inheritance:e = enable (re-inherit from parent)

Problem 3: Share vs. NTFS Permission Conflict

NTFS vs. Share Permissions
When accessing files over the network, both NTFS permissions and share permissions apply. The most restrictive combination wins. NTFS permissions apply locally and over the network. Share permissions apply only over the network. A common mistake is setting generous NTFS permissions but restrictive share permissions (or vice versa).
REM NTFS: Marketing has Modify on D:\Reports
REM Share: "Reports" share grants Everyone: Read
REM Network access: Marketing gets READ ONLY (share is more restrictive)
REM Local access: Marketing gets MODIFY (share doesn't apply locally)

REM Best practice: Set share permissions to broad access
REM and control everything through NTFS permissions
icacls D:\Reports  ← Use this for fine-grained control

Problem 4: Group Membership Not Refreshed

Just like Linux, Windows evaluates group membership at logon time. If a user is added to a group, they must log off and log back on for the change to take effect. For domain accounts, the Kerberos ticket must be refreshed.

# Check current group memberships (in user's session)
whoami /groups

# If the new group is not listed, user must:
# 1. Log off and log back on
# 2. For domain accounts: wait for Kerberos ticket refresh or run:
klist purge

3.2 Using Effective Access (Step-by-Step)

  1. Right-click the file/folder → Properties → Security tab
  2. Click Advanced
  3. Click the Effective Access tab
  4. Click Select a user → enter the username → OK
  5. Click View effective access
  6. Review the results — green checkmarks show granted permissions, red X marks show denied
💡 Effective Access Limitations

The Effective Access tool shows NTFS permissions only. It does not account for:

  • Share permissions (network access restrictions)
  • User Account Control (UAC) elevation requirements
  • Software Restriction Policies or AppLocker rules
  • Dynamic Access Control policies

If Effective Access shows the user should have access but they still get "Access Denied," check share permissions and UAC.

📌 Enabling Object Access Auditing

By default, Windows does not log file/folder access events (Event IDs 4663, 4656). To enable them:

  1. Open Local Security Policy (secpol.msc)
  2. Navigate to Local Policies → Audit Policy
  3. Enable Audit object access (Success and/or Failure)
  4. On the specific folder, go to Security → Advanced → Auditing and add an audit entry

This is covered in detail in Week 14 (Server Security & Hardening) and Project 3.

5.1 The Problem: Logs Are Scattered

In a typical multi-server environment, permission-related events are spread across many systems:

  • Linux Server 1: /var/log/auth.log records sudo usage and SSH logins
  • Linux Server 2: /var/log/auth.log records its own, separate events
  • Windows Server: Security Event Log records logon events and object access
  • Firewall: Logs record allowed/denied network connections

An administrator investigating a security incident must manually log into each system, search through different log formats, and mentally correlate events across timestamps. This is slow, error-prone, and doesn't scale.

5.2 The Solution: Centralized Log Management

SIEM (Security Information and Event Management)
A system that collects, aggregates, and analyzes log data from multiple sources in a centralized location. SIEMs provide real-time monitoring, correlation of events across systems, alerting, and dashboards. Examples include Splunk, Elastic SIEM, Microsoft Sentinel, and IBM QRadar.

Splunk is the SIEM tool you will use in this course. Here's how it connects to the permission concepts you've learned this week:

5.4 Compliance and Audit Requirements

Many regulatory frameworks require organizations to maintain audit logs of access events:

  • PCI DSS (Payment Card Industry): Requires logging all access to cardholder data
  • HIPAA (Healthcare): Requires audit trails for access to protected health information
  • SOX (Sarbanes-Oxley): Requires access controls and audit trails for financial data
  • FERPA (Education): Requires protection of student records with access logging

The permission skills from Modules 7.1–7.3 implement access controls. The logging skills from this module prove those controls are working — which is what auditors need to see.

🏗️ Activity 4: Cross-Platform Troubleshooting Scenarios

Work through each scenario and expand to check your solution.

Scenario 1: Linux — Script Won't Execute

Trouble ticket: User alice created a Bash script /opt/scripts/backup.sh. When she runs ./backup.sh, she gets "Permission denied." She can cat the file and see its contents.

Task: Walk through the troubleshooting steps and identify the fix.

Show Solution 
# Step 1: Check the file permissions
ls -l /opt/scripts/backup.sh
-rw-r--r-- 1 alice developers 2048 Mar 3 10:00 backup.sh
# No execute bit! (644)

# Step 2: Verify alice is the owner (yes — she can read it)

# Step 3: Apply minimum fix — add execute for owner only
chmod u+x /opt/scripts/backup.sh

# Step 4: Verify
ls -l /opt/scripts/backup.sh
-rwxr--r-- 1 alice developers 2048 Mar 3 10:00 backup.sh

# Step 5: Test
./backup.sh
# Script runs successfully

Root cause: The file was created without the execute bit (normal for new files due to umask). The user needed to explicitly add execute permission.

Key principle: The fix was chmod u+x (minimum change), NOT chmod 777.

Scenario 2: Linux — Parent Directory Blocking Access

Trouble ticket: User bob (member of webteam) cannot read /var/www/internal/index.html. The file has permissions -rw-r--r-- root:webteam. Other webteam members can read it fine.

Show Solution 
# Step 1: Check the full path
namei -l /var/www/internal/index.html
f: /var/www/internal/index.html
drwxr-xr-x root root     /
drwxr-xr-x root root     var
drwxr-xr-x root root     www
drwx------ root root     internal   ← BLOCKED (700)
-rw-r--r-- root webteam  index.html

# Step 2: /var/www/internal/ is 700 — only root can enter
# bob hits the wall at this directory regardless of file perms

# Step 3: Fix the directory permissions
sudo chgrp webteam /var/www/internal/
sudo chmod 750 /var/www/internal/

# Step 4: Verify bob can now access
sudo -u bob cat /var/www/internal/index.html

Root cause: The parent directory /var/www/internal/ had permissions 700 (owner only). Even though the file itself was world-readable, the directory blocked traversal.

Key lesson: Always check the entire path, not just the target file.

Scenario 3: Windows — Network Share Access Denied

Trouble ticket: User dave (member of Accounting group) can access D:\Finance\ locally on the server but gets "Access Denied" when accessing \\FileServer\Finance\ over the network.

Show Solution

Diagnosis:

  1. NTFS permissions — Accounting has Modify on D:\Finance\ ✅
  2. Share permissions — The "Finance" share grants Everyone: Read ❌
  3. Effective network access = most restrictive of NTFS and Share = Read only

Fix:

REM Option A: Update share permissions to match NTFS intent
REM In Share Permissions: grant Accounting: Change (or Full Control)

REM Option B (Best practice): Set share to broad access,
REM control everything through NTFS
REM Share: Authenticated Users → Full Control
REM NTFS: Accounting → Modify (already set)

Key lesson: When accessing over the network, both NTFS and share permissions apply. The most restrictive combination wins. Best practice: use broad share permissions and control access through NTFS.

Situation: In the future (Project 3), you will use Splunk to analyze logs like these. For now, interpret the following aggregated data that a Splunk query might produce:

Questions to consider:

  2. What would you investigate first?

Show Analysis

1. Most concerning entries:

  • mallory on server02: 47 + 12 = 59 failed sudo attempts — This is highly suspicious. Normal users might fail once or twice; 59 attempts suggests either a compromised account attempting privilege escalation or an unauthorized user trying to gain root access.
  • admin on web-prod: 8 attempts — Moderate concern. "admin" is a common target for attackers. Verify whether a legitimate admin account exists.

2. Investigate first:

  • Check who mallory is — legitimate user or compromised account?
  • Check what commands were attempted — grep "mallory" /var/log/auth.log
  • Check source IP/terminal — is it from an expected location?
  • Check if mallory should have sudo access at all

3. Immediate actions:

  • Lock mallory's account pending investigation: sudo usermod -L mallory
  • Review all of mallory's recent activity on server02
  • Check for any successful privilege escalation
  • Document findings for incident response

✓ Module 7.4 Summary

Key Takeaways

  • Always follow a systematic troubleshooting approach: reproduce → identify user context → examine permissions → check full path → check for overrides → review logs → fix → verify → document.
  • Never use chmod 777 or "Everyone: Full Control" as a fix — find the specific missing permission and grant only that.
  • The five most common Linux permission problems are: missing execute on directories, incorrect ownership, missing group membership (re-login required), owner category override, and parent directory blocking.
  • Windows troubleshooting must account for Deny overriding Allow, broken inheritance, and share vs. NTFS permission conflicts.
  • Group membership changes require re-login on both Linux and Windows.
  • The namei -l command (Linux) and Effective Access tab (Windows) are your most powerful diagnostic tools.
  • Permission-related events are recorded in /var/log/auth.log (Linux) and the Security Event Log (Windows).
  • Key Windows Event IDs: 4624 (successful logon), 4625 (failed logon), 4663 (object access), 4670 (permission change), 4732 (group member added).
  • Centralized log management with a SIEM like Splunk enables cross-system correlation, alerting, and compliance reporting — skills you'll develop in Project 3 (Weeks 13–15).

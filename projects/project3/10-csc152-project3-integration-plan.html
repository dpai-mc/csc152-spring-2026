2 CIS Control 1 — Service Hardening

🔒 Control 1 2 pts CIS: Disable unnecessary services

Reduce the attack surface by disabling services that are not needed on a headless Ubuntu server. Each unnecessary running service is an additional exposure point.

Before Capture running services

Run this command and save the output for your submission. The diff at the end will show exactly which services were removed.

Terminal — linux-server (10.0.2.10)
# BEFORE — capture this output for submission
systemctl list-units --type=service --state=running > ~/before-services.txt
cat ~/before-services.txt

Identify and disable unnecessary services

The commands below use 2>/dev/null so that services not installed on your system fail silently. Each line confirms whether the service was found and disabled.

Terminal — disable unneeded services
# Check if services exist and are running, then disable:
sudo systemctl disable --now cups.service 2>/dev/null && echo "cups disabled" || echo "cups not found"
sudo systemctl disable --now avahi-daemon.service 2>/dev/null && echo "avahi disabled" || echo "avahi not found"
sudo systemctl disable --now bluetooth.service 2>/dev/null && echo "bluetooth disabled" || echo "bluetooth not found"
sudo systemctl disable --now ModemManager.service 2>/dev/null && echo "ModemManager disabled" || echo "not found"
sudo systemctl disable --now snapd.service 2>/dev/null && echo "snapd disabled" || echo "snapd not found"

After Verify and document changes

Terminal — after state
# AFTER — capture this output for submission
systemctl list-units --type=service --state=running > ~/after-services.txt
diff ~/before-services.txt ~/after-services.txt
Submission tip: Copy and paste the diff output into your submission document. Lines preceded by < are services that were running before hardening but are now stopped.

3 CIS Control 2 — SSH Hardening

🔒 Control 2 3 pts CIS: sshd_config security settings

SSH is the primary remote management vector for linux-server. Hardening /etc/ssh/sshd_config closes common attack vectors including direct root login, brute-force attempts, and idle session persistence.

Before Back up and inspect current config

Terminal — save current SSH config
# BEFORE — save current config
sudo cp /etc/ssh/sshd_config ~/before-sshd_config.bak
cat /etc/ssh/sshd_config | grep -v "^#" | grep -v "^$"

Settings to apply

The table below shows each CIS-recommended directive, its value, and the security rationale.

Setting Value CIS Rationale
PermitRootLogin no Prevents direct root SSH access
MaxAuthTries 3 Limits brute-force attempts per connection
ClientAliveInterval 300 Times out idle sessions (5 minutes)
ClientAliveCountMax 0 Disconnects immediately after timeout
PermitEmptyPasswords no Requires passwords for authentication
X11Forwarding no Reduces attack surface (no GUI needed)
AllowUsers sysadmin Restricts SSH to named users only
LoginGraceTime 60 Limits time allowed to complete authentication
Banner /etc/issue.net Displays legal warning before login

Apply hardening with sed

The sed commands below modify commented-out defaults in-place. The AllowUsers and Banner directives are appended since they typically have no commented placeholder.

Terminal — sshd_config changes
# Modify commented defaults in-place
sudo sed -i 's/#PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config
sudo sed -i 's/#MaxAuthTries.*/MaxAuthTries 3/' /etc/ssh/sshd_config
sudo sed -i 's/#ClientAliveInterval.*/ClientAliveInterval 300/' /etc/ssh/sshd_config
sudo sed -i 's/#ClientAliveCountMax.*/ClientAliveCountMax 0/' /etc/ssh/sshd_config
sudo sed -i 's/#PermitEmptyPasswords.*/PermitEmptyPasswords no/' /etc/ssh/sshd_config
sudo sed -i 's/#X11Forwarding.*/X11Forwarding no/' /etc/ssh/sshd_config
sudo sed -i 's/#LoginGraceTime.*/LoginGraceTime 60/' /etc/ssh/sshd_config

# Append directives that have no commented placeholder
echo "AllowUsers sysadmin" | sudo tee -a /etc/ssh/sshd_config
echo "Banner /etc/issue.net" | sudo tee -a /etc/ssh/sshd_config

# Validate and restart
sudo sshd -t && echo "Config OK" || echo "Config ERROR - fix before restarting"
sudo systemctl restart sshd
⚠
Lock-out risk: Always test SSH access from a second terminal window before closing your current session. A misconfigured sshd_config can lock you out of the server. If that happens, use the Proxmox console (not SSH) to correct the configuration.

After Verify applied settings

Terminal — confirm key directives
sudo grep -E "PermitRootLogin|MaxAuthTries|ClientAliveInterval|PermitEmptyPasswords|X11Forwarding|AllowUsers|LoginGraceTime|Banner" \
  /etc/ssh/sshd_config | grep -v "^#"

4 CIS Control 3 — Host Firewall with UFW

🔒 Control 3 2 pts CIS: Default deny, allow only needed ports

UFW (Uncomplicated Firewall) provides a host-level packet filter. The CIS Benchmark requires a default-deny posture for incoming traffic with explicit allow rules for only the services this server provides.

Before Check current firewall state

Terminal — current UFW status
sudo ufw status

Apply UFW rules

⚠
Add the SSH allow rule before enabling UFW. If you enable UFW first without allowing port 22, you will be immediately locked out of the server and must use the Proxmox console to recover.
Terminal — configure UFW
# Set default policies
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Allow SSH FIRST (before enabling UFW)
sudo ufw allow 22/tcp comment 'SSH'

# Allow DNS (linux-server provides secondary DNS)
sudo ufw allow 53/tcp comment 'DNS TCP'
sudo ufw allow 53/udp comment 'DNS UDP'

# Enable UFW
sudo ufw enable

After Verify firewall rules

Terminal — verify rules in effect
sudo ufw status verbose
Splunk forwarder note: Outbound traffic from linux-server to splunk-vm is already allowed by the default outgoing policy. No additional inbound UFW rule is needed on this server for Splunk forwarding; the receiving rule belongs on splunk-vm.

5 CIS Control 4 — auditd Configuration

🔒 Control 4 2 pts CIS: Audit rules for security events

The Linux Audit subsystem (auditd) records security-relevant kernel events to a tamper-evident log. CIS requires monitoring of authentication files, privilege escalation paths, and user management commands.

Terminal — install, enable, and configure auditd
# Install auditd
sudo apt install -y auditd audispd-plugins

# Enable and start
sudo systemctl enable --now auditd

# Create security rules file
sudo tee /etc/audit/rules.d/security.rules << 'EOF'
# Monitor authentication files
-w /etc/passwd -p wa -k identity
-w /etc/group -p wa -k identity
-w /etc/shadow -p wa -k identity
-w /etc/gshadow -p wa -k identity

# Monitor sudoers
-w /etc/sudoers -p wa -k sudoers
-w /etc/sudoers.d/ -p wa -k sudoers

# Monitor SSH configuration
-w /etc/ssh/sshd_config -p wa -k sshd_config

# Monitor login events
-w /var/log/lastlog -p wa -k logins
-w /var/log/faillog -p wa -k logins

# Monitor cron
-w /etc/crontab -p wa -k cron
-w /etc/cron.d/ -p wa -k cron

# Monitor user modifications
-w /usr/sbin/useradd -p x -k user_modification
-w /usr/sbin/usermod -p x -k user_modification
-w /usr/sbin/userdel -p x -k user_modification
EOF

# Load rules
sudo augenrules --load

# Verify
sudo auditctl -l
sudo ausearch -k identity | head -20
Rule flags explained: -w watches a file or directory. -p wa triggers on write and attribute-change events. -p x triggers on execute. -k assigns a searchable keyword tag to the rule.

6 CIS Control 5 — File Permissions

🔒 Control 5 2 pts CIS: Critical system file permissions

Incorrect permissions on authentication and credential files can allow unprivileged users to read password hashes or modify account data. CIS specifies exact permissions for each file.

Before Document current permissions

Terminal — record current permissions
# BEFORE — document current permissions
stat /etc/passwd /etc/shadow /etc/group /etc/gshadow

Apply correct permissions

Terminal — set CIS-required permissions
# Set correct ownership and permissions
sudo chmod 644 /etc/passwd
sudo chmod 640 /etc/shadow
sudo chmod 644 /etc/group
sudo chmod 640 /etc/gshadow
sudo chown root:shadow /etc/shadow
sudo chown root:shadow /etc/gshadow

# Secure SSH directory and keys (root)
sudo chmod 700 /root/.ssh 2>/dev/null || true
sudo chmod 600 /root/.ssh/authorized_keys 2>/dev/null || true

# Find world-writable files (document for submission)
sudo find / -xdev -type f -perm -0002 -not -path "/proc/*" 2>/dev/null | head -20

After Verify permissions

Terminal — confirm permissions applied
# AFTER — verify permissions
stat /etc/passwd /etc/shadow /etc/group /etc/gshadow
File Mode Owner:Group
/etc/passwd 644 root:root
/etc/shadow 640 root:shadow
/etc/group 644 root:root
/etc/gshadow 640 root:shadow
/root/.ssh 700 root:root
/root/.ssh/authorized_keys 600 root:root

7 CIS Control 6 — Automatic Updates

🔒 Control 6 1 pt CIS: unattended-upgrades

The unattended-upgrades package automatically applies security patches without manual intervention. CIS requires that this service is installed, enabled, and configured to apply at minimum security updates.

Terminal — install and configure unattended-upgrades
# Install unattended-upgrades
sudo apt install -y unattended-upgrades apt-listchanges

# Interactive configuration (select YES when prompted)
sudo dpkg-reconfigure -pmedium unattended-upgrades

# Verify configuration file
cat /etc/apt/apt.conf.d/20auto-upgrades
# Expected output:
# APT::Periodic::Update-Package-Lists "1";
# APT::Periodic::Unattended-Upgrade "1";

# Enable and start service
sudo systemctl enable --now unattended-upgrades

# Test dry run (no changes applied)
sudo unattended-upgrade --dry-run --debug
Dry-run output: The --dry-run --debug flags simulate an upgrade without making changes. Save this output to demonstrate the service is functional and has access to security updates.

8 Additional CIS Controls 7–10

The six required control areas above may implement multiple discrete CIS controls. The following four additional controls bring the total well above the required minimum of 12 and address high-value attack vectors.

Control 7 — Password Quality Policy

CIS: Minimum password complexity requirements
Terminal — libpam-pwquality
sudo apt install -y libpam-pwquality
sudo nano /etc/security/pwquality.conf

# Set these values:
# minlen = 14
# dcredit = -1
# ucredit = -1
# ocredit = -1
# lcredit = -1
# maxrepeat = 3
Setting Value Meaning
minlen 14 Minimum password length of 14 characters
dcredit -1 Require at least 1 digit
ucredit -1 Require at least 1 uppercase letter
ocredit -1 Require at least 1 special character
lcredit -1 Require at least 1 lowercase letter
maxrepeat 3 No more than 3 consecutive identical characters

Control 8 — Disable Root Account

CIS: Lock root; enforce sudo for privilege escalation
Terminal — lock root account
# Lock root account (use sudo instead)
sudo passwd -l root

# Verify — output should show "L" (locked)
sudo passwd -S root
Note: Locking root does not prevent console access in single-user/recovery mode. Physical and Proxmox console security controls are separate concerns.

Control 9 — Legal Login Banner

CIS: Warning banner displayed before authentication
Terminal — create /etc/issue.net
sudo tee /etc/issue.net << 'EOF'
*******************************************************************************
                         AUTHORIZED ACCESS ONLY
This system is the property of TechStart Solutions. Unauthorized access is
prohibited and will be prosecuted to the fullest extent of the law.
All activity on this system is monitored and logged.
*******************************************************************************
EOF
SSH integration: The Banner /etc/issue.net directive added in Control 2 (SSH hardening) ensures this text is displayed to users during SSH authentication, before they are granted access.

Control 10 — Kernel Parameter Hardening (sysctl)

CIS: Network stack and kernel security parameters
Terminal — /etc/sysctl.d/99-hardening.conf
sudo tee /etc/sysctl.d/99-hardening.conf << 'EOF'
# Disable IP forwarding (linux-server is not a router)
net.ipv4.ip_forward = 0

# Disable ICMP redirects
net.ipv4.conf.all.accept_redirects = 0
net.ipv4.conf.all.send_redirects = 0

# Enable SYN flood protection
net.ipv4.tcp_syncookies = 1

# Disable source routing
net.ipv4.conf.all.accept_source_route = 0

# Enable martian packet logging
net.ipv4.conf.all.log_martians = 1

# Protect against time-wait assassination
net.ipv4.tcp_rfc1337 = 1
EOF

# Apply immediately without rebooting
sudo sysctl --system
Parameter Value Security Purpose
net.ipv4.ip_forward 0 Prevents this host from routing packets between networks
accept_redirects 0 Ignores ICMP redirects (used in MITM attacks)
send_redirects 0 Does not send ICMP redirects to other hosts
tcp_syncookies 1 Mitigates SYN flood denial-of-service attacks
accept_source_route 0 Ignores source-routed packets
log_martians 1 Logs packets with impossible source addresses
tcp_rfc1337 1 Protects TCP connections from TIME-WAIT assassination

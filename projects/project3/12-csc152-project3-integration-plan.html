1 Download Splunk Enterprise

SSH into splunk-vm (10.0.2.20) and download the Splunk Enterprise installer package.

bash — splunk-vm
# Create downloads directory
mkdir -p ~/splunk-install && cd ~/splunk-install

# Download Splunk Enterprise 10.2.2 for Linux (64-bit DEB)
wget -O splunk-10.2.2-linux-amd64.deb \
  "https://download.splunk.com/products/splunk/releases/10.2.2/linux/splunk-10.2.2-80b90d638de6-linux-amd64.deb"

# Verify download completed (should be ~280 MB)
ls -lh splunk-*.deb
ℹ
About this download link: The URL above is a direct CDN link to Splunk 10.2.2 and does not require a Splunk account to use. If the link fails (Splunk occasionally changes their download policies), you can get a fresh link by creating a free account at splunk.com, logging in, navigating to the Splunk Enterprise download page, and copying the wget command for the Linux 64-bit .deb package. A free Splunk account also gives you access to Splunk Fundamentals 1 — an excellent free training course worth completing.

2 Install Splunk Enterprise

Install the downloaded .deb package, accept the license, set admin credentials, and configure Splunk to start automatically on boot.

bash — splunk-vm
# Install (Ubuntu/Debian)
sudo dpkg -i splunk-10.2.2-linux-amd64.deb

# Start Splunk and accept the license
sudo /opt/splunk/bin/splunk start --accept-license

# You will be prompted to create an admin username and password
# Username: admin
# Password: (use a strong password, at least 8 chars with mixed case and numbers)

# Enable Splunk to start on boot
sudo /opt/splunk/bin/splunk enable boot-start -user splunk

# Check Splunk status
sudo /opt/splunk/bin/splunk status

Expected Output After Start

If Splunk starts successfully you will see output similar to the following:

expected output
Splunk> Take the sh out of IT.

Checking prerequisites...
  Checking http port [8000]: open
  Checking mgmt port [8089]: open
  ...
Done.
All preliminary checks passed.

Starting splunk server daemon (splunkd)...
Done.
Admin password rules: Splunk requires a password of at least 8 characters containing uppercase letters, lowercase letters, and numbers. Choose a password you will remember — you will use it throughout the remaining integration plans.

3 Access the Web Interface

The Splunk web interface is accessible from any VM that has firewall access to 10.0.2.20:8000. Use mgmt-vm's Firefox browser for the best experience.

  • Open Firefox on mgmt-vm (or any VM with access to the 10.0.2.x network)
  • Navigate to: http://10.0.2.20:8000
  • Login with username admin and the password you set during installation
  • The Splunk Home page loads, showing the search bar and app launcher
What you should see: After login, the Splunk Home page displays the Search & Reporting app icon, a search bar, and navigation menus at the top: Search & Reporting, Settings, and Apps. The home page may also display a "Getting Started" tour prompt — you can dismiss it.
⚠
Do NOT enable Splunk SSL on port 8000 yet. Leave it as HTTP for the lab environment. Enabling HTTPS requires certificate management that is beyond the scope of this project. Changing to HTTPS mid-lab will break the forwarder configurations in Integration Plans 13 and 14.

4 Configure Receiving Port (9997)

Port 9997 is the standard Splunk receiving port. Universal Forwarders installed on linux-server and dc01 (Integration Plans 13 & 14) will send logs to this port.

bash — splunk-vm
# Enable Splunk to receive data from forwarders on port 9997
sudo /opt/splunk/bin/splunk enable listen 9997 -auth admin:yourpassword

# Verify receiver is configured
sudo /opt/splunk/bin/splunk list receiver -auth admin:yourpassword
  • In Splunk Web, click Settings in the top navigation bar
  • Click Forwarding and receiving
  • Under Receive data, click Configure receiving
  • Click New Receiving Port
  • Enter port 9997 and click Save

5 Create Indexes

Create three dedicated indexes to keep log sources organized and searchable. Using separate indexes makes queries faster and access control simpler.

Index Name Purpose Max Size
linux_logs Linux auth, syslog, and audit logs from linux-server 10 GB
windows_logs Windows Security, System, and Application event logs from dc01 10 GB
firewall_logs Reserved for pfSense/firewall logs (future use) 5 GB
bash — splunk-vm
# Create linux_logs index
sudo /opt/splunk/bin/splunk add index linux_logs \
  -maxTotalDataSizeMB 10240 \
  -auth admin:yourpassword

# Create windows_logs index
sudo /opt/splunk/bin/splunk add index windows_logs \
  -maxTotalDataSizeMB 10240 \
  -auth admin:yourpassword

# Create firewall_logs index
sudo /opt/splunk/bin/splunk add index firewall_logs \
  -maxTotalDataSizeMB 5120 \
  -auth admin:yourpassword
  • Click Settings → Indexes → New Index
  • Index Name: linux_logs — Max raw data size: 10 GB → Save
  • Repeat: Index Name: windows_logs — Max size: 10 GB → Save
  • Repeat: Index Name: firewall_logs — Max size: 5 GB → Save

6 Install Splunk Add-ons

Add-ons (also called Technology Add-ons or TAs) provide field extractions and source type definitions that make log data searchable with meaningful field names. Install both add-ons before configuring dashboards.

Required Add-ons

  • Splunk Add-on for Unix and Linux — field extractions for syslog, auth logs, auditd
  • Splunk Add-on for Microsoft Windows — field extractions for Windows Event Logs
  • In Splunk Web, click Apps in the top navigation bar
  • Click Find More Apps
  • Search for Splunk Add-on for Unix and Linux → click Install
  • Enter your Splunk.com account credentials when prompted
  • After the Unix add-on installs, search for Splunk Add-on for Microsoft Windows → click Install
  • Click Restart Now when prompted to activate the add-ons

If the web app browser cannot reach Splunkbase, download the .tgz files from splunkbase.splunk.com on mgmt-vm, then copy them to splunk-vm using scp.

bash — splunk-vm
# Copy downloaded .tgz files to splunk-vm, then install:

# Install Unix add-on
sudo /opt/splunk/bin/splunk install app /tmp/splunk-add-on-for-unix-and-linux.tgz \
  -auth admin:yourpassword

# Install Windows add-on
sudo /opt/splunk/bin/splunk install app /tmp/splunk-add-on-for-microsoft-windows.tgz \
  -auth admin:yourpassword

# Restart Splunk to activate add-ons
sudo /opt/splunk/bin/splunk restart

7 Create Authentication Dashboard B.2 — 4 pts

This dashboard tracks login activity across Linux and Windows systems. It is the primary tool for detecting unauthorized access attempts.

Create the Dashboard

  • In Splunk Web, click Dashboards in the left navigation or app bar
  • Click Create New Dashboard
  • Title: Security - Authentication Events
  • Type: Classic Dashboard → Create

Panel 1 — Failed SSH Logins (Linux)

Shows failed SSH password attempts by source IP and username, sorted by attempt count (highest first).

splunk SPL — panel visualization: Table
index=linux_logs sourcetype=linux_secure "Failed password"
| stats count by src_ip, user
| sort -count
| rename count AS "Attempts", src_ip AS "Source IP", user AS "Username"

Panel 2 — Failed Windows Logins

Queries Windows Security Event 4625 (failed logon) grouped by account name and source IP.

splunk SPL — panel visualization: Bar Chart
index=windows_logs sourcetype=WinEventLog:Security EventCode=4625
| stats count by Account_Name, Source_Network_Address
| sort -count
| rename count AS "Attempts", Account_Name AS "Account", Source_Network_Address AS "Source IP"

Panel 3 — Successful Logins (Windows)

Queries Windows Security Event 4624 (successful logon) for interactive (type 2) and remote interactive (type 10) sessions.

splunk SPL — panel visualization: Pie Chart
index=windows_logs sourcetype=WinEventLog:Security EventCode=4624
  (Logon_Type=2 OR Logon_Type=10)
| stats count by Account_Name
| sort -count

Panel 4 — Sudo Usage (Linux)

Shows sudo command usage by user over time, sorted by most recent activity.

splunk SPL — panel visualization: Timeline
index=linux_logs sourcetype=linux_secure sudo
| stats count by user, _time
| sort -_time
Saving panels: After entering each SPL query, click Add to Dashboard (or Save As → Dashboard Panel) and select the existing Security - Authentication Events dashboard. Choose the visualization type listed above for each panel, then click Save.

8 Create System Events Dashboard B.2 — 4 pts

This dashboard monitors system-level events including Windows service changes, Linux errors, and Linux audit events.

Create the Dashboard

  • Click Dashboards → Create New Dashboard
  • Title: Security - System Events
  • Type: Classic Dashboard → Create

Panel 1 — Windows Service Changes

Event 7045 = new service installed. Event 7040 = service start type changed. Both are common persistence techniques used by attackers.

splunk SPL — panel visualization: Table
index=windows_logs sourcetype=WinEventLog:System
  (EventCode=7045 OR EventCode=7040)
| table _time, Service_Name, Service_Type, Message
| sort -_time

Panel 2 — Linux System Errors

Filters syslog for error and critical-level messages by host, helping identify VMs with recurring problems.

splunk SPL — panel visualization: Table
index=linux_logs sourcetype=syslog (level=err OR level=crit)
| stats count by host, _raw
| sort -count

Panel 3 — Recent Audit Events (Linux)

Summarizes auditd events by audit key, showing which audit rules are triggering most frequently.

splunk SPL — panel visualization: Bar Chart
index=linux_logs sourcetype=linux_audit
| stats count by key
| sort -count

9 Configure Security Alerts B.2 — 4 pts

Alerts run SPL queries on a schedule and notify you when conditions are met. Configure both alerts below. To create an alert: run the query in Search, then click Save As → Alert.

Alert 1 — Multiple Failed Logins (Brute Force Detection)

This alert fires when any single IP address generates more than 5 failed login attempts in a 5-minute window across both Linux and Windows systems.

splunk SPL — brute force alert query
(index=linux_logs sourcetype=linux_secure "Failed password")
OR
(index=windows_logs sourcetype=WinEventLog:Security EventCode=4625)
| stats count by src_ip
| where count > 5

Alert 1 Settings

  • Title: Multiple Failed Logins - Possible Brute Force
  • Schedule: Run every 5 minutes, looking back 5 minutes
  • Trigger condition: Number of Results > 0
  • Actions: Add to Triggered Alerts

Alert 2 — After-Hours Login

This alert detects successful interactive logins to Windows systems outside of business hours (before 6:00 AM or after 6:00 PM).

splunk SPL — after-hours login alert query
index=windows_logs sourcetype=WinEventLog:Security EventCode=4624
  (Logon_Type=2 OR Logon_Type=10)
| eval hour=strftime(_time, "%H")
| where hour >= 18 OR hour < 6
| table _time, Account_Name, Source_Network_Address

Alert 2 Settings

  • Title: After-Hours Successful Login
  • Schedule: Run every 30 minutes, looking back 30 minutes
  • Trigger condition: Number of Results > 0
  • Actions: Add to Triggered Alerts
Viewing triggered alerts: After the alerts are saved you can see them under Activity → Triggered Alerts in Splunk Web. Until log forwarders send data, the alerts will not fire — this is expected behavior.

10 Verify Splunk Setup

Run these commands to confirm Splunk is running correctly, indexes exist, and the receiver is active before moving on to Integration Plan 13.

bash — splunk-vm
# Check Splunk is running
sudo /opt/splunk/bin/splunk status

# List configured indexes (should show linux_logs, windows_logs, firewall_logs)
sudo /opt/splunk/bin/splunk list index -auth admin:yourpassword

# List configured receivers (should show port 9997)
sudo /opt/splunk/bin/splunk list receiver -auth admin:yourpassword

# Check for any errors
sudo tail -100 /opt/splunk/var/log/splunk/splunkd.log | grep ERROR
ℹ
No data yet — this is expected. Until log forwarders are configured in Integration Plans 13 (Linux) and 14 (Windows), the dashboards will show "No results found". Splunk is ready to receive data — no data has been sent yet.

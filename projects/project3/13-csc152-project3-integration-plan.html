Integration 13: Linux Universal Forwarder Configuration

Install and configure Splunk Universal Forwarder on linux-server to forward logs to splunk-vm (Part C.1 — 7 pts)

Source: linux-server (10.0.2.10)
Destination: splunk-vm (10.0.2.20:9997)
Rubric: Part C.1 — 7 pts

✓ Part C.1 Rubric Summary

This integration covers installing and configuring the Splunk Universal Forwarder on linux-server. All four tasks below must be completed and documented for full credit.

Task Points
Install Universal Forwarder 2
Configure Outputs (point to Splunk) 2
Configure Inputs (log sources) 2

Total C.1 7

1 Overview

The Splunk Universal Forwarder (SUF) is a lightweight agent that streams log files to the Splunk indexer. It does not index data itself — it simply forwards. Install on linux-server (10.0.2.10), forward to splunk-vm (10.0.2.20) on port 9997.

The following log sources will be monitored and forwarded to the linux_logs index on splunk-vm:

Log File sourcetype Index Purpose
/var/log/auth.log linux_secure linux_logs Auth events (SSH, sudo)
/var/log/syslog syslog linux_logs System events
/var/log/audit/audit.log linux_audit linux_logs Auditd security events
/var/log/ufw.log ufw linux_logs Host firewall events
Prerequisite: Integration 10 (Linux Server Hardening) must be complete. auditd must be configured and running to produce /var/log/audit/audit.log, and UFW must be enabled to produce /var/log/ufw.log.

2 Download and Install the Universal Forwarder

SSH to linux-server (10.0.2.10) and run the commands below. The forwarder version must match the Splunk Enterprise version running on splunk-vm.

Terminal — linux-server (10.0.2.10)
# Create install directory
mkdir -p ~/splunk-forwarder && cd ~/splunk-forwarder

# Download Splunk Universal Forwarder for Linux
# Get current download URL from: splunk.com/en_us/download/universal-forwarder.html
# The forwarder version MUST match the Splunk Enterprise version on splunk-vm
wget -O splunkforwarder-9.x.x-linux-amd64.deb \
  'https://download.splunk.com/products/universalforwarder/releases/9.x.x/linux/splunkforwarder-9.x.x-linux-amd64.deb'

# Install
sudo dpkg -i splunkforwarder-9.x.x-linux-amd64.deb

# Start and accept license
sudo /opt/splunkforwarder/bin/splunk start --accept-license \
  --answer-yes \
  --no-prompt \
  --seed-passwd 'ForwarderPass1!'

# Enable boot-start
sudo /opt/splunkforwarder/bin/splunk enable boot-start
ℹ
Path note: The Universal Forwarder is installed to /opt/splunkforwarder/, separate from Splunk Enterprise at /opt/splunk/. They use the same binary but different paths. The Forwarder version should match the Enterprise server version.

3 Configure outputs.conf (point to Splunk indexer)

The outputs.conf file tells the forwarder where to send data. Create it in the system/local/ directory, which overrides any default settings.

Terminal — linux-server
# Create the outputs configuration
sudo mkdir -p /opt/splunkforwarder/etc/system/local/

sudo tee /opt/splunkforwarder/etc/system/local/outputs.conf << 'EOF'
[tcpout]
defaultGroup = splunk-indexers

[tcpout:splunk-indexers]
server = 10.0.2.20:9997
compressed = true
sendCookedData = true

[tcpout-server://10.0.2.20:9997]
EOF

echo "outputs.conf created successfully"
cat /opt/splunkforwarder/etc/system/local/outputs.conf
Port 9997: This is Splunk's default receiving port. The Splunk Enterprise instance on splunk-vm must have a TCP input listener configured on port 9997 (set up during Integration 12 — Splunk Deployment). Verify this is enabled in Splunk web under Settings → Forwarding and Receiving.

4 Configure inputs.conf (what to monitor)

The inputs.conf file specifies which log files the forwarder monitors and the sourcetype and index for each. Each stanza defines one log source.

Terminal — linux-server
sudo tee /opt/splunkforwarder/etc/system/local/inputs.conf << 'EOF'
# SSH and authentication events
[monitor:///var/log/auth.log]
disabled = false
sourcetype = linux_secure
index = linux_logs

# System log
[monitor:///var/log/syslog]
disabled = false
sourcetype = syslog
index = linux_logs

# Audit daemon log (requires auditd configured in Build 10)
[monitor:///var/log/audit/audit.log]
disabled = false
sourcetype = linux_audit
index = linux_logs

# UFW host firewall log
[monitor:///var/log/ufw.log]
disabled = false
sourcetype = ufw
index = linux_logs

# Cron log
[monitor:///var/log/cron.log]
disabled = false
sourcetype = cron
index = linux_logs
EOF

echo "inputs.conf created successfully"
cat /opt/splunkforwarder/etc/system/local/inputs.conf
Index must exist: The linux_logs index must be created on splunk-vm before data will be accepted. Create it in Splunk web: Settings → Indexes → New Index → Name: linux_logs. If the index does not exist, events will be silently dropped or sent to the default index.

5 Grant File Read Permissions

The Splunk forwarder runs as the splunk user and needs read access to system log files. Without this, the forwarder starts but cannot read the log files specified in inputs.conf.

Terminal — linux-server
# Add splunk user to adm group (gives read access to /var/log files)
sudo usermod -aG adm splunk

# For audit log specifically (restricted permissions):
sudo setfacl -m u:splunk:r /var/log/audit/audit.log
# If setfacl not available:
sudo apt install -y acl
sudo setfacl -m u:splunk:r /var/log/audit/audit.log

# Verify
sudo -u splunk cat /var/log/auth.log | head -5
sudo -u splunk cat /var/log/audit/audit.log | head -5
⚠
Group membership requires restart: After running usermod -aG adm splunk, the forwarder service must be restarted before the new group membership takes effect. The restart in Section 6 will apply this change.

6 Restart Forwarder and Verify Connection

Terminal — linux-server
# Restart forwarder to apply config
sudo /opt/splunkforwarder/bin/splunk restart

# Check forwarder status
sudo /opt/splunkforwarder/bin/splunk status

# Check connections to Splunk
sudo /opt/splunkforwarder/bin/splunk list forward-server

# View forwarder logs for connection status
sudo tail -50 /opt/splunkforwarder/var/log/splunk/splunkd.log | grep -i "tcpout\|connect\|error"

Expected output from list forward-server:

Expected output: 
Active forwards:
    10.0.2.20:9997
Configured but inactive forwards:
    (none)
If 10.0.2.20:9997 appears under "Configured but inactive forwards" instead, the forwarder cannot reach splunk-vm. Check the network path and verify that Splunk Enterprise on splunk-vm is running and listening on port 9997.

7 Verify Data in Splunk Web

From mgmt-vm Firefox → http://10.0.2.20:8000 → Login with admin credentials:

  1. Go to Search & Reporting → Search bar
  2. Run: index=linux_logs | head 20
  3. Should return log events from linux-server within 1–2 minutes of restarting the forwarder
  4. Check specific log types with the queries below
Splunk Search — verify sourcetypes
index=linux_logs sourcetype=linux_secure | head 10
index=linux_logs sourcetype=linux_audit | head 10

Generate test events to confirm end-to-end

Run the following on linux-server to produce known log entries, then search for them in Splunk to confirm the full forwarding pipeline is working.

Terminal — linux-server — generate test events
# Generate a failed sudo attempt (creates an auth event)
sudo -u nobody whoami 2>/dev/null || true

# Generate a test syslog entry with a unique search string
logger -p auth.info "TEST: Integration 13 forwarder verification - $(date)"
Splunk Search — find the test event
index=linux_logs "Integration 13 forwarder verification"
Success criteria: The logger message should appear in Splunk within 60 seconds. If it does not appear after 2 minutes, check the forwarder log at /opt/splunkforwarder/var/log/splunk/splunkd.log for connection errors, and verify the linux_logs index exists on splunk-vm.

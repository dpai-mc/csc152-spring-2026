1c. Recovery Objectives (RTO / RPO)

D.1 — 1 pt
Objective Target Justification
RTO (Recovery Time Objective) 4 hours Time to rebuild a VM from backup in this lab environment

💡
In a production environment, RTO and RPO would be much tighter (e.g., 1-hour RTO, 1-hour RPO with real-time replication or incremental snapshots). For this lab, daily backups to local disk is a reasonable starting point that demonstrates the core concept of backup planning.

2 Linux Backup Script D.2 — 3 pts

SSH to linux-server (10.0.2.10) and complete all steps below. You will create a backup directory, write an automated script, test it manually, and schedule it with cron.

Step 1 — Create the backup directory

bash — linux-server
# Create backup destination with restricted permissions
sudo mkdir -p /backup
sudo chmod 700 /backup
sudo chown root:root /backup

# Verify disk space available
df -h /

Step 2 — Create the backup script

bash — linux-server
sudo tee /usr/local/bin/backup.sh << 'SCRIPT'
#!/bin/bash
# backup.sh - Automated backup script for linux-server
# Part of CSC 152 Project 3, Integration Plan 15

# Configuration
BACKUP_DIR="/backup"
DATE=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS=7
LOG_FILE="${BACKUP_DIR}/backup.log"

# Ensure backup directory exists
mkdir -p "${BACKUP_DIR}"

echo "$(date '+%Y-%m-%d %H:%M:%S') Backup started" >> "${LOG_FILE}"

# Function for timed backup
backup_item() {
  local name="$1"
  local source="$2"
  local dest="${BACKUP_DIR}/${name}_${DATE}.tar.gz"

  if tar -czf "${dest}" "${source}" 2>/dev/null; then
    echo "$(date '+%Y-%m-%d %H:%M:%S') [OK] ${name}: $(du -sh ${dest} | cut -f1)" >> "${LOG_FILE}"
  else
    echo "$(date '+%Y-%m-%d %H:%M:%S') [WARN] ${name}: completed with warnings" >> "${LOG_FILE}"
  fi
}

# Backup /etc (system configuration)
backup_item "etc" "/etc"

# Backup /home (user home directories)
backup_item "home" "/home"

# Backup Splunk forwarder configuration
if [ -d "/opt/splunkforwarder/etc" ]; then
  backup_item "splunkforwarder_etc" "/opt/splunkforwarder/etc"
fi

# Backup audit logs
if [ -d "/var/log/audit" ]; then
  backup_item "audit_logs" "/var/log/audit"
fi

# Remove backups older than retention period
find "${BACKUP_DIR}" -name "*.tar.gz" -mtime +${RETENTION_DAYS} -delete
echo "$(date '+%Y-%m-%d %H:%M:%S') Old backups cleaned (>${RETENTION_DAYS} days)" >> "${LOG_FILE}"

# Show current backup inventory
echo "$(date '+%Y-%m-%d %H:%M:%S') Backup complete. Files:" >> "${LOG_FILE}"
ls -lh "${BACKUP_DIR}"/*.tar.gz 2>/dev/null >> "${LOG_FILE}"

echo "$(date '+%Y-%m-%d %H:%M:%S') Backup finished" >> "${LOG_FILE}"
echo "---" >> "${LOG_FILE}"
SCRIPT

# Make executable and set ownership
sudo chmod 700 /usr/local/bin/backup.sh
sudo chown root:root /usr/local/bin/backup.sh
echo "Backup script created"

Step 3 — Test the script manually

bash — linux-server
# Run backup script manually (first test)
sudo /usr/local/bin/backup.sh

# Verify output — you should see .tar.gz files
ls -lh /backup/

# Review the log
cat /backup/backup.log
Screenshot required: Capture the output of ls -lh /backup/ showing the archive files and cat /backup/backup.log showing the timestamped log entries. Both are needed for Part D.2 credit.

Step 4 — Schedule with cron

bash — linux-server
# Add cron entry directly (daily at 2 AM)
(sudo crontab -l 2>/dev/null; echo "0 2 * * * /usr/local/bin/backup.sh") | sudo crontab -

# Verify cron entry
sudo crontab -l

The cron expression 0 2 * * * means: minute 0, hour 2, every day of the month, every month, every day of the week. The script will run automatically at 2:00 AM every night.

3 Windows Server Backup (System State) D.2 — 3 pts

On dc01, open PowerShell as Administrator. You will install the Windows Server Backup feature, configure a System State backup policy, run an immediate backup, and verify the result.

Step 1 — Install Windows Server Backup feature

PowerShell — dc01 (Run as Administrator)
# Install Windows Server Backup feature
Install-WindowsFeature Windows-Server-Backup -IncludeManagementTools

# Verify installation
Get-WindowsFeature Windows-Server-Backup | Select Name, InstallState

Step 2 — Create backup directory and configure policy

PowerShell — dc01
# Create a new backup policy
$policy = New-WBPolicy

# Add System State (includes AD DS, DNS, Registry, SYSVOL)
Add-WBSystemState -Policy $policy

# Get the C: volume object — New-WBBackupTarget requires a WBVolume object, not a path string.
# Windows Server Backup will store the backup in a WindowsImageBackup folder on this volume.
# Note: For production use, a separate volume (D:) is recommended. C: is acceptable for lab use.
$targetVolume = Get-WBVolume -AllVolumes | Where-Object { $_.DriveLetter -eq "C" }
$backupTarget = New-WBBackupTarget -Volume $targetVolume -PreserveExistingBackups $true
Add-WBBackupTarget -Policy $policy -Target $backupTarget

# Schedule daily at 3 AM
Set-WBSchedule -Policy $policy -Schedule 03:00

# Save the policy
Set-WBPolicy -Policy $policy

# Verify policy was saved
Get-WBPolicy | Select-Object *

Step 3 — Run an immediate backup to test

PowerShell — dc01
# Run backup now (System State takes approximately 5–15 minutes)
Start-WBBackup -Policy (Get-WBPolicy)

# Monitor job status while it runs
Get-WBJob -Previous 1

Step 4 — Verify the backup completed successfully

PowerShell — dc01
# Check backup history summary
Get-WBSummary

# List backed-up versions
Get-WBBackupSet -BackupTarget (Get-WBBackupTarget -Policy (Get-WBPolicy))
Screenshot required: Capture Get-WBSummary output showing LastBackupResultDetail: The operation completed successfully. and the LastBackupTime timestamp. This is your primary evidence for D.2 credit.
💡
System State backup on a Domain Controller includes the Active Directory database (NTDS.dit), SYSVOL (Group Policy files), AD-integrated DNS zones, and the registry. This is the minimum backup for a DC — losing a DC without a System State backup means rebuilding Active Directory from scratch.

4 pfSense Configuration Backup D.2 — 2 pts

From mgmt-vm, open Firefox and navigate to the pfSense web UI at http://10.0.1.1. Log in as admin. The entire pfSense configuration — firewall rules, NAT, DHCP, interfaces — is stored in a single config.xml file that can be exported in seconds.

Manual backup procedure

  1. Navigate to Diagnostics → Backup & Restore
  2. Under Backup area, select ALL
  3. Encryption: optional (recommended for production — skip for this lab)
  4. Click “Download configuration as XML”
  5. Save the file to mgmt-vm under Documents/pfSense-Backups/

Naming convention for the backup file

Name the downloaded file with an ISO date so backups are easy to sort:

Filename example
pfsense-config-20260421.xml

Store a copy off-system

After saving to mgmt-vm, copy the config.xml to at least one additional location:

  • A shared folder accessible from outside the lab network
  • A USB drive or external disk
  • Cloud storage (Google Drive, OneDrive, etc.)
  • Your personal laptop or desktop — acceptable for this lab
⚠️
A pfSense backup stored only on mgmt-vm (which lives on the same Proxmox infrastructure) is not truly safe. If Proxmox fails, both the firewall and the backup disappear together. In production, copy the backup to an off-site location, cloud storage, or at minimum to your physical laptop/desktop.

5 Backup Verification — Test Restore Procedure D.2 — 2 pts

A backup that has never been tested is a backup you cannot trust. For each system, perform at minimum an integrity check; for Linux, also perform a partial test restore.

5a. Verify Linux backup archives

linux-server
bash — linux-server
# Integrity check — test all .tar.gz files in /backup
for f in /backup/*.tar.gz; do
  echo -n "Testing $f ... "
  if tar -tzf "$f" > /dev/null 2>&1; then
    echo "OK"
  else
    echo "CORRUPTED - investigate immediately"
  fi
done

After confirming integrity, perform a partial test restore to verify the archive can actually be read back:

bash — linux-server (partial restore test)
# Identify the most recent /etc backup
LATEST_ETC=$(ls -t /backup/etc_*.tar.gz | head -1)
echo "Testing: ${LATEST_ETC}"

# List contents (confirm /etc/hostname is inside)
sudo tar -tzf "${LATEST_ETC}" | grep hostname

# Extract /etc/hostname to /tmp as a non-destructive restore test
sudo tar -xzf "${LATEST_ETC}" -C /tmp/ etc/hostname

# Verify the restored file
cat /tmp/etc/hostname
echo "Expected: linux-server"
Screenshot required: Show the loop output (all archives listing "OK") and the cat /tmp/etc/hostname result. This is your restore test evidence.

5b. Verify Windows Server Backup

dc01
PowerShell — dc01
# Check backup set details
$backupSet = Get-WBBackupSet -BackupTarget (Get-WBBackupTarget -Policy (Get-WBPolicy))
$backupSet | Select-Object BackupTime, BackupTarget, RecoveryTargets

# Primary verification — confirm success status
Get-WBSummary | Select-Object LastBackupTime, LastBackupResultDetail, NextBackupTime

The backup is considered verified when:

  • LastBackupResultDetail shows "The operation completed successfully."
  • BackupSet contains System State as a recovery target

5c. Verify pfSense backup file

pfSense

Open the exported config.xml in a text editor on mgmt-vm and confirm:

  • The root element is <pfsense>
  • The file contains <interfaces>, <filter> (firewall rules), and <dhcpd> sections
  • The file is not zero bytes and is not truncated
Optional full restore test: In pfSense, navigate to Diagnostics → Backup & Restore → Restore Configuration, upload the XML, and click "Restore Configuration." Warning: this will reboot pfSense immediately. Only attempt if you can tolerate a brief network outage (approximately 1–2 minutes for this lab environment).

1 Why a Dedicated Splunk VM?

Splunk can technically run on linux-server (10.0.2.10) — and that is a valid approach for the base assignment. However, this environment uses a dedicated VM for three practical reasons:

  • Resource isolation — Splunk is memory- and I/O-hungry. Sharing a VM with linux-server workloads would cause contention and unpredictable performance.
  • Security boundaries — a dedicated VM means the SIEM can be locked down independently, with its own firewall rules and user accounts, without affecting linux-server operations.
  • Storage flexibility — Splunk indexes grow over time. A dedicated 100 GB disk can be expanded or snapshotted independently of other VMs.

The splunk-vm at 10.0.2.20 acts as both the indexer and search head. Universal Forwarders installed on linux-server and dc01 will send log data here on TCP port 9997.

Component VM IP Role
Splunk Enterprise splunk-vm (1006) 10.0.2.20 Indexer + Search Head
Universal Forwarder linux-server (1002) 10.0.2.10 Log collection agent
Universal Forwarder dc01 (1003) 10.0.2.11 Log collection agent

3 Resize VM Resources for Splunk

Select VM 1006 in the Proxmox left panel, then click the Hardware tab. Make the following changes while the VM is still powered off:

Memory Double-click Memory → change to 8192 MB (8 GB) → OK
CPU Cores Double-click Processors → set Cores to 4 → OK
Disk resize Select the disk entry (e.g., virtio0 or scsi0) → click Disk Action → Resize → enter 80 in the "Size Increment" field → Resize Disk
Disk math: The Ubuntu template starts with a 20 GB disk. Adding 80 GB gives a total of 100 GB. The Proxmox resize operation only expands the block device — the partition and filesystem inside the VM must be expanded separately after first boot (Section 4).

4 First Boot Configuration

Start VM 1006 and log in as sysadmin via the Proxmox console. Run the following to set the hostname and expand the disk inside the VM to match the Proxmox resize performed in Section 3:

Step 1 — Set hostname
# Set hostname
sudo hostnamectl set-hostname splunk-vm
Step 2 — Expand disk partition and filesystem
# ── Option A: Non-LVM (standard partition) ──
# Use this if the template used a plain partition layout
sudo growpart /dev/vda 2
sudo resize2fs /dev/vda2

# ── Option B: LVM (logical volume) ──
# Use this if the template used LVM (check with: lsblk)
sudo growpart /dev/vda 3
sudo pvresize /dev/vda3
sudo lvextend -l +100%FREE /dev/ubuntu-vg/ubuntu-lv
sudo resize2fs /dev/mapper/ubuntu--vg-ubuntu--lv

# Verify total disk is now ~100 GB
df -h /
Which option to use? Run lsblk to inspect the disk layout. If you see an entry like ubuntu--vg-ubuntu--lv, the template uses LVM (Option B). If you see only vda1, vda2 as plain partitions with no LVM layer, use Option A.

5 Configure Static IP

The splunk-vm must have a fixed IP address so that Universal Forwarders on other VMs can reliably connect to it on port 9997. Configure a static IP via Netplan:

Step 1 — Check interface name
# Find your network interface name (commonly ens18 in Proxmox VMs)
ip link show
Step 2 — Open Netplan config
sudo nano /etc/netplan/00-installer-config.yaml

Replace the file contents with the following. If your interface name is not ens18, substitute the correct name found in Step 1:

Netplan config — /etc/netplan/00-installer-config.yaml
network:
  version: 2
  ethernets:
    ens18:
      dhcp4: false
      addresses:
        - 10.0.2.20/24
      routes:
        - to: default
          via: 10.0.2.1
      nameservers:
        addresses:
          - 10.0.2.11
          - 8.8.8.8
Step 3 — Apply and verify
sudo netplan apply
ip addr show ens18

6 Pre-Splunk System Preparation

Splunk Enterprise has specific OS-level requirements for reliable performance. Run the following commands to tune kernel settings and create the dedicated splunk OS user that will own the Splunk process:

ℹ
Deferred to Integration 08. splunk-vm has no internet access at this stage because pfSense has not been configured yet. apt update && apt upgrade will be run in Integration 08 · pfSense Configuration & Network Integration once internet routing is in place.
Step 1 — Set system limits for Splunk (open file descriptors)
# Splunk recommends a high open-file limit for indexing performance
sudo bash -c 'cat >> /etc/security/limits.conf << EOF
splunk  soft  nofile  65535
splunk  hard  nofile  65535
EOF'
Step 2 — Disable Transparent Huge Pages (THP)
# Disable THP immediately (takes effect without reboot)
echo 'never' | sudo tee /sys/kernel/mm/transparent_hugepage/enabled
echo 'never' | sudo tee /sys/kernel/mm/transparent_hugepage/defrag

# Make THP setting persistent across reboots via /etc/rc.local
sudo nano /etc/rc.local

In the rc.local file, enter the following content (create the file if it does not exist):

/etc/rc.local — persistent THP disable
#!/bin/bash
echo never > /sys/kernel/mm/transparent_hugepage/enabled
echo never > /sys/kernel/mm/transparent_hugepage/defrag
exit 0
Step 3 — Make rc.local executable and create the splunk user
# Make rc.local executable
sudo chmod +x /etc/rc.local

# Create the splunk system user (no login shell needed at this stage)
sudo useradd -m -r -s /bin/bash splunk
Why disable Transparent Huge Pages? THP is a Linux kernel feature that dynamically manages large memory pages. Splunk's memory access patterns conflict with THP, causing latency spikes and degraded indexing throughput. Disabling it is a standard Splunk tuning requirement documented in Splunk's official installation guide.

7 Open Firewall Ports (UFW)

Configure UFW to allow SSH administration access and the two Splunk-specific ports: 8000/tcp (Splunk Web UI) and 9997/tcp (Universal Forwarder receiving). All other incoming traffic will be blocked by default.

UFW configuration — run as sysadmin
# Enable UFW
sudo ufw enable

# Default policies — block inbound, allow outbound
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Allow SSH admin access
sudo ufw allow 22/tcp comment 'SSH admin access'

# Allow Splunk Web UI (browser-based search and dashboards)
sudo ufw allow 8000/tcp comment 'Splunk Web UI'

# Allow Splunk receiving from Universal Forwarders on other VMs
sudo ufw allow 9997/tcp comment 'Splunk Universal Forwarder input'

# Verify all rules are in place
sudo ufw status verbose
Port / Protocol Service Purpose
22/tcp SSH Admin shell access from mgmt-vm
8000/tcp Splunk Web Browser-based UI for search and dashboards
9997/tcp Splunk forwarder input Receives log data from Universal Forwarders on linux-server and dc01

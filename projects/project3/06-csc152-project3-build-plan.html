1 Overview

The linux-client VM is a lightweight Xubuntu desktop on the Clients network (vmbr3). It represents an end-user workstation in the TechStart Solutions environment — the kind of machine a typical employee would use day-to-day.

Project 3 does not actively use this VM. It is built now so it is available for two future purposes:

  • Firewall rule testing — used in Integration Plan 08 to verify pfSense is correctly segmenting Clients from Servers and Management networks.
  • Project 4 — the client VM will be used for end-to-end testing of domain authentication, log forwarding, and network services.

Because the VM is cloned from the Xubuntu minimal template (9002), no ISO download or OS installation is required. The entire build takes only a few minutes.

2 Clone from Template 9002

In the Proxmox web UI, right-click template 9002 (xubuntu-minimal-template) and select Clone. Use the following settings:

Mode Full Clone — creates an independent disk copy, not linked to the template
VM ID 1005
Name linux-client
Target Storage local-lvm
Click Clone and wait for the task to complete (~1–2 minutes)

After cloning, verify the VM resources. No changes are needed — the template defaults match the target spec:

  • CPU: keep 1 vCPU (no change needed)
  • RAM: keep 2 GB (no change needed)
  • Disk: keep 15 GB (no change needed)
  • Network: confirm the bridge is vmbr3 (Clients) — not vmbr2 (Servers). If it shows vmbr1 or vmbr2, edit the network device and change it to vmbr3 before proceeding.
Why vmbr3? The Clients network (vmbr3 / 10.0.3.0/24) is intentionally separated from the Servers network (vmbr2 / 10.0.2.0/24) by pfSense firewall rules. Placing a client workstation on vmbr3 correctly models real-world network segmentation.

3 Configure linux-client (DHCP)

Start VM 1005 and log in as sysadmin. Open a terminal (right-click the desktop → Terminal Emulator) and run the following commands:

Step 1 — Set hostname
# Set hostname
sudo hostnamectl set-hostname linux-client
Step 2 — Identify network interface name
# List network interfaces to find the correct name
ip link show
# Look for an interface starting with 'ens' or 'eth' (commonly ens18)
# This VM cloned from the same template as mgmt-vm, so the name is likely identical
Step 3 — Open Netplan config (verify DHCP is set)
# Configure DHCP via Netplan (should already be default, but verify)
sudo nano /etc/netplan/01-netcfg.yaml

The file should contain the following. If it does not, replace its contents with the DHCP configuration below. If your interface name differs from ens18, substitute it in the ethernets key:

Netplan config — /etc/netplan/01-netcfg.yaml
network:
  version: 2
  ethernets:
    ens18:
      dhcp4: true
Step 4 — Apply and verify
sudo netplan apply

# Verify — expect an IP in 10.0.3.100-200 range after pfSense is configured
ip addr show ens18
ℹ
No IP yet — that is normal. The DHCP lease from pfSense will not work until pfSense is fully configured in Integration Plan 08. At this stage the VM will show no IP or a 169.254.x.x link-local address. This is expected behavior.

4 Domain Join Optional for Project 3

Joining the linux-client to the techstart.local Active Directory domain is not required for Project 3. These steps are provided for reference and will become relevant in Project 4 once Active Directory is fully configured (Build 09).

Domain join — run after Active Directory is configured (Build 09)
# Install sssd and realmd for domain join
sudo apt install -y realmd sssd adcli samba-common-bin

# Discover the domain (after AD is configured in Build 09)
realm discover techstart.local

# Join the domain
sudo realm join -U ts.admin techstart.local
Prerequisites before domain join: Active Directory must be running and reachable on dc01 (10.0.2.11), DNS must resolve techstart.local, and pfSense firewall rules must allow traffic from the Clients network (vmbr3) to reach the Servers network (vmbr2). None of these are true yet at this stage.

5 Power Off and Store

Once the hostname and Netplan configuration are confirmed, shut the VM down cleanly:

Shut down VM 1005
sudo poweroff

Leave VM 1005 powered off. No further action is needed for this build. The VM is ready and waiting for use in:

  • Integration Plan 08 — power on temporarily to test pfSense firewall rules from the Clients network
  • Project 4 — used as the primary end-user client for testing domain authentication, network services, and log collection

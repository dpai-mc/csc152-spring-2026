Integration 09: Active Directory & DNS Setup

Promote dc01 to domain controller, create the techstart.local domain, configure DNS, build the OU structure, and add users and groups from Project 2

Server: dc01 (10.0.2.11)
Domain: techstart.local
Forest/Domain Level: Windows Server 2016
⚠
Take a Proxmox snapshot of dc01 (pre-AD-baseline) before proceeding if you haven't already. AD promotion installs services and modifies the registry in ways that are not easily reversed. A clean snapshot lets you roll back if something goes wrong.

1 Overview

This integration step rebuilds the Project 2 Active Directory environment from scratch on dc01. You will promote dc01 to the first domain controller in a new forest, configure AD-integrated DNS, and recreate the OU structure, users, and groups established in Project 2.

Prerequisites: pfSense is fully configured (Integration 08) and dc01 has network connectivity to the Servers network (10.0.2.0/24).

After completing this integration:

  • Domain: techstart.local (Windows Server 2016 functional level)
  • DNS: AD-integrated, authoritative for techstart.local and reverse zone 2.0.10.in-addr.arpa
  • Users: ts.admin, john.smith, maria.garcia, robert.chen, sarah.jones
  • Groups: IT_Admins (security, global scope)
  • OUs: TechStart → IT_Admins, Employees, Computers
DC=techstart,DC=local
├─ OU=TechStart
├─ OU=IT_Admins
└─ ts.admin
├─ OU=Employees
├─ john.smith  ├─ maria.garcia
├─ robert.chen  └─ sarah.jones
└─ OU=Computers

2 Install AD DS Role

On dc01, open PowerShell as Administrator (right-click the Start button → Windows PowerShell (Admin)). Run the following commands to install the Active Directory Domain Services role and management tools:

PowerShell — dc01 (Run as Administrator)
# Install Active Directory Domain Services role
Install-WindowsFeature -Name AD-Domain-Services -IncludeManagementTools

# Verify installation
Get-WindowsFeature AD-Domain-Services

The installation will take a few minutes. When complete, the InstallState column should show Installed. The server does not need to reboot yet — promotion in the next step handles that.

3 Promote to Domain Controller

Run the following PowerShell commands to promote dc01 as the first domain controller in a new forest. The WinThreshold mode corresponds to Windows Server 2016 functional level.

PowerShell — dc01 (Run as Administrator)
# Promote dc01 as the first DC in a new forest
Import-Module ADDSDeployment

Install-ADDSForest `
  -DomainName "techstart.local" `
  -DomainNetbiosName "TECHSTART" `
  -ForestMode "WinThreshold" `
  -DomainMode "WinThreshold" `
  -InstallDns:$true `
  -DatabasePath "C:\Windows\NTDS" `
  -LogPath "C:\Windows\NTDS" `
  -SysvolPath "C:\Windows\SYSVOL" `
  -NoRebootOnCompletion:$false `
  -Force:$true
DSRM Password prompt: You will be prompted to enter a Directory Services Restore Mode (DSRM) password. Choose a strong password that is different from the Administrator account password. Record it securely — it is needed for AD recovery operations and is not easily reset.

WinThreshold = Windows Server 2016 functional level. This matches the environment you configured in Project 2.

After promotion completes, the server will reboot automatically. Log in as TECHSTART\Administrator after the restart.

4 Verify AD DS and DNS

After dc01 reboots and you log in as TECHSTART\Administrator, open PowerShell as Administrator and run these verification commands:

PowerShell — dc01
# Verify AD DS services are running
Get-Service ADWS, DNS, Netlogon, KDC | Select Name, Status

# Verify domain information
Get-ADDomain

# Verify DNS is serving the zone
Resolve-DnsName dc01.techstart.local -Server 127.0.0.1
Resolve-DnsName techstart.local -Server 127.0.0.1 -Type SOA

All four services (ADWS, DNS, Netlogon, KDC) should show Running. The Get-ADDomain output confirms the domain name, functional level, and forest root. The Resolve-DnsName commands confirm the DNS server is answering queries for its own zone.

5 Configure DNS Zones

AD DS promotion automatically creates the techstart.local forward lookup zone and adds the essential SRV, A, and NS records for the domain controller. You now need to add A records for the remaining environment hosts, CNAME aliases, and the reverse lookup zone with PTR records.

5a — Add A Records and CNAME Aliases

Open DNS Manager (Server Manager → Tools → DNS) or use PowerShell:

PowerShell — dc01 · A Records & CNAMEs
# Add A records for all environment hosts
Add-DnsServerResourceRecordA -ZoneName "techstart.local" `
  -Name "pfsense-mgmt" -IPv4Address "10.0.1.1"

Add-DnsServerResourceRecordA -ZoneName "techstart.local" `
  -Name "linux-server" -IPv4Address "10.0.2.10"

Add-DnsServerResourceRecordA -ZoneName "techstart.local" `
  -Name "splunk-vm" -IPv4Address "10.0.2.20"

Add-DnsServerResourceRecordA -ZoneName "techstart.local" `
  -Name "mgmt-vm" -IPv4Address "10.0.1.10"

# Add CNAME records (aliases)
Add-DnsServerResourceRecordCName -ZoneName "techstart.local" `
  -Name "dns" -HostNameAlias "dc01.techstart.local."

Add-DnsServerResourceRecordCName -ZoneName "techstart.local" `
  -Name "ldap" -HostNameAlias "dc01.techstart.local."

Add-DnsServerResourceRecordCName -ZoneName "techstart.local" `
  -Name "splunk" -HostNameAlias "splunk-vm.techstart.local."

Reference table for all expected DNS records in the techstart.local zone:

Name Type Value
dc01A10.0.2.11 (auto-created by AD DS)
pfsense-mgmtA10.0.1.1
linux-serverA10.0.2.10
splunk-vmA10.0.2.20
mgmt-vmA10.0.1.10
dnsCNAMEdc01.techstart.local.
ldapCNAMEdc01.techstart.local.
splunkCNAMEsplunk-vm.techstart.local.

5b — Create Reverse Lookup Zone and PTR Records

PowerShell — dc01 · Reverse Zone & PTR Records
# Create reverse zone for 10.0.2.x (Servers network)
Add-DnsServerPrimaryZone -NetworkID "10.0.2.0/24" `
  -ReplicationScope "Domain"

# Add PTR records
Add-DnsServerResourceRecordPtr -ZoneName "2.0.10.in-addr.arpa" `
  -Name "1" -PtrDomainName "pfsense-opt1.techstart.local."

Add-DnsServerResourceRecordPtr -ZoneName "2.0.10.in-addr.arpa" `
  -Name "10" -PtrDomainName "linux-server.techstart.local."

Add-DnsServerResourceRecordPtr -ZoneName "2.0.10.in-addr.arpa" `
  -Name "11" -PtrDomainName "dc01.techstart.local."

Add-DnsServerResourceRecordPtr -ZoneName "2.0.10.in-addr.arpa" `
  -Name "20" -PtrDomainName "splunk-vm.techstart.local."
Note: The reverse zone 2.0.10.in-addr.arpa covers the 10.0.2.x Servers network. The Management network (10.0.1.x) does not require a reverse zone for this project, but you may add one using the same pattern (1.0.10.in-addr.arpa) if desired.

6 Update pfSense DNS

Now that dc01 is the authoritative DNS server for techstart.local, update pfSense so that DNS queries from all network segments are forwarded to dc01 first.

  • From mgmt-vm, open Firefox and navigate to https://10.0.1.1 (pfSense web UI)
  • Go to System → General Setup → DNS Servers
  • Set DNS Server 1: 10.0.2.11 (dc01)
  • Set DNS Server 2: 8.8.8.8 (Google — internet fallback)
  • Click Save
Also update: Netplan DNS on linux-server and splunk-vm should also be updated to use 10.0.2.11 as their primary DNS. This is covered in Section 9 below for linux-server; repeat the same steps for splunk-vm.

7 Create Organizational Unit Structure

Run the following PowerShell commands on dc01 to create the OU hierarchy. This mirrors the structure you built in Project 2.

PowerShell — dc01
# Create top-level OU
New-ADOrganizationalUnit -Name "TechStart" `
  -Path "DC=techstart,DC=local" `
  -ProtectedFromAccidentalDeletion $true

# Create sub-OUs inside TechStart
New-ADOrganizationalUnit -Name "IT_Admins" `
  -Path "OU=TechStart,DC=techstart,DC=local"

New-ADOrganizationalUnit -Name "Employees" `
  -Path "OU=TechStart,DC=techstart,DC=local"

New-ADOrganizationalUnit -Name "Computers" `
  -Path "OU=TechStart,DC=techstart,DC=local"

The -ProtectedFromAccidentalDeletion $true flag on the top-level OU prevents it from being deleted through Active Directory Users and Computers without first unchecking the protection option. The sub-OUs inherit the domain-level deletion confirmation by default.

8 Create Users and Groups

Create the IT_Admins security group, the ts.admin administrator account, and four standard employee accounts. All passwords shown below are temporary — students are required to change them at first logon.

PowerShell — dc01
# Create security group
New-ADGroup -Name "IT_Admins" `
  -GroupScope Global `
  -GroupCategory Security `
  -Path "OU=IT_Admins,OU=TechStart,DC=techstart,DC=local"

# Create admin user (ts.admin)
New-ADUser -Name "TS Admin" `
  -GivenName "TS" -Surname "Admin" `
  -SamAccountName "ts.admin" `
  -UserPrincipalName "ts.admin@techstart.local" `
  -Path "OU=IT_Admins,OU=TechStart,DC=techstart,DC=local" `
  -AccountPassword (ConvertTo-SecureString "P@ssw0rd123!" -AsPlainText -Force) `
  -Enabled $true `
  -PasswordNeverExpires $false `
  -ChangePasswordAtLogon $true

# Add ts.admin to IT_Admins group and Domain Admins
Add-ADGroupMember -Identity "IT_Admins" -Members "ts.admin"
Add-ADGroupMember -Identity "Domain Admins" -Members "ts.admin"

# Create employee users
$employees = @(
  @{First="John";   Last="Smith";  Sam="john.smith"},
  @{First="Maria";  Last="Garcia"; Sam="maria.garcia"},
  @{First="Robert"; Last="Chen";   Sam="robert.chen"},
  @{First="Sarah";  Last="Jones";  Sam="sarah.jones"}
)
foreach ($emp in $employees) {
  New-ADUser -Name "$($emp.First) $($emp.Last)" `
    -GivenName $emp.First -Surname $emp.Last `
    -SamAccountName $emp.Sam `
    -UserPrincipalName "$($emp.Sam)@techstart.local" `
    -Path "OU=Employees,OU=TechStart,DC=techstart,DC=local" `
    -AccountPassword (ConvertTo-SecureString "TempP@ss1!" -AsPlainText -Force) `
    -Enabled $true `
    -ChangePasswordAtLogon $true
}

# Verify users were created
Get-ADUser -Filter * -SearchBase "OU=TechStart,DC=techstart,DC=local" |
  Select Name, SamAccountName, Enabled
ℹ
Password policy reminder: The temporary passwords above (P@ssw0rd123! for ts.admin, TempP@ss1! for employees) meet default Windows Server complexity requirements. Students should change them immediately after first logon, or you can set them to something else before class. Never use production passwords in lab documentation.

9 Update linux-server DNS

SSH to linux-server (10.0.2.10) and update its Netplan configuration to use dc01 as the primary DNS server. Repeat these steps for splunk-vm (10.0.2.20) as well.

bash — linux-server (SSH from mgmt-vm)
# Open the Netplan configuration file
sudo nano /etc/netplan/00-installer-config.yaml

# In the nameservers block, update addresses to:
#   addresses: [10.0.2.11, 8.8.8.8]

# Apply the change
sudo netplan apply

# Test DNS resolution against dc01
nslookup dc01.techstart.local
nslookup techstart.local

The nslookup commands should return the correct IP addresses, confirming linux-server is resolving names through dc01. If resolution fails, verify that dc01 is reachable on port 53 from linux-server:

bash — linux-server · connectivity check
# Confirm dc01 is responding on UDP 53
nc -zvu 10.0.2.11 53

10 Verify Complete

Run a final verification pass on dc01 to confirm AD, DNS, OUs, users, and groups are all correctly configured:

PowerShell — dc01 · Final Verification
# Verify AD OU structure
Get-ADOrganizationalUnit -Filter * |
  Select Name, DistinguishedName

# Verify all users
Get-ADUser -Filter * |
  Select Name, SamAccountName, Enabled

# Verify groups
Get-ADGroup -Filter * |
  Select Name, GroupScope, GroupCategory

# Verify DNS forward resolution
nslookup linux-server.techstart.local 127.0.0.1
nslookup splunk-vm.techstart.local 127.0.0.1
nslookup mgmt-vm.techstart.local 127.0.0.1

# Verify DNS reverse resolution
nslookup 10.0.2.10 127.0.0.1
nslookup 10.0.2.11 127.0.0.1
nslookup 10.0.2.20 127.0.0.1

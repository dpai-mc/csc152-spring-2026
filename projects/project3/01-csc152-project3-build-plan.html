Build 01: Proxmox Network Bridge Configuration

Configure virtual network bridges on the Proxmox host before building any VMs

Course: CSC 152 Project: 3 Type: Build Plan Scope: Host Config (no VMs yet)

1 Overview

Proxmox VE uses Linux bridges as virtual switches — the same concept as a managed switch in physical networking, but implemented entirely in software on your Proxmox host. Each bridge (vmbr0, vmbr1, vmbr2, vmbr3) acts as an isolated Layer 2 segment. VMs attach their virtual NICs to whichever bridge corresponds to the network they should live on.

Before any virtual machines are created, the host must have the correct bridge topology in place. This build plan configures all four bridges that Project 3 requires:

  • vmbr0 — WAN bridge (linked to your physical NIC, uplink to home router)
  • vmbr1 — Management network (admin access, pfSense LAN side)
  • vmbr2 — Servers network (linux-server, dc01, splunk-vm)
  • vmbr3 — Clients network (linux-client, DHCP pool)

Network Topology Diagram 

Internet / Home Router (192.168.50.1)
         |
    [Physical NIC]
         |
    vmbr0 (WAN) ────────────── pfSense WAN interface
    192.168.50.0/24                DHCP from home router

    vmbr1 (Management) ─────── pfSense LAN ──── mgmt-vm
    10.0.1.0/24                    10.0.1.1        10.0.1.10

    vmbr2 (Servers) ────────── pfSense OPT1 ─── linux-server · dc01 · splunk-vm
    10.0.2.0/24                    10.0.2.1          .10    .11     .20

    vmbr3 (Clients) ────────── pfSense OPT2 ─── linux-client (DHCP)
    10.0.3.0/24                    10.0.3.1        10.0.3.100–200

Key point: Only vmbr0 connects to a physical NIC. The other three bridges are internal only — they carry no physical port and are isolated from your home network by design. pfSense will route between them once it is built in Build 02.

2 Prerequisites

Confirm the following before starting:

  • Proxmox VE 8.x installed and accessible at https://192.168.50.10:8006
  • Logged in as root or a user with the System.Modify privilege on the node
  • Physical NIC connected to your home network (192.168.50.0/24) and visible in Proxmox
  • No VMs have been created yet — bridges should be configured before any guest is built
Note on vmbr0: In most Proxmox installations, vmbr0 is created automatically during the OS installer and is already linked to your physical NIC. Verify it exists in the Network panel before creating it. You only need to create vmbr1, vmbr2, and vmbr3 from scratch.

3 Step-by-Step: Proxmox Web UI Method

Use the Proxmox web interface to create each bridge. This is the recommended approach for students new to Proxmox — changes are applied atomically and can be reviewed before committing.

  1. 1

    Log into the Proxmox web interface

    Open a browser and navigate to https://192.168.50.10:8006. Accept the self-signed certificate warning. Log in as root (PAM authentication).

  2. 2

    Navigate to your node's Network panel

    In the left-hand resource tree, click your node name (e.g., pve). In the center panel, select Network from the node menu. You will see a list of existing network interfaces.

  3. 3

    Verify vmbr0 exists

    Confirm that vmbr0 appears in the list and is linked to your physical NIC (e.g., eno1, eth0, or similar). It should show the 192.168.50.10/24 address assigned during Proxmox installation.

    If vmbr0 does not exist, click Create → Linux Bridge, select your physical NIC as the bridge port, and assign 192.168.50.10/24 with gateway 192.168.50.1.

  4. 4

    Create vmbr1 — Management Network

    Click Create → Linux Bridge and fill in the following fields:

    • Name: vmbr1
    • IPv4/CIDR: leave blank (pfSense owns this subnet)
    • Bridge ports: leave empty (internal only — no physical NIC)
    • Comment: Management Network 10.0.1.0/24

    Click Create to add it to the pending changes list.

  5. 5

    Create vmbr2 — Servers Network

    Click Create → Linux Bridge again:

    • Name: vmbr2
    • IPv4/CIDR: leave blank
    • Bridge ports: leave empty
    • Comment: Servers Network 10.0.2.0/24

    Click Create.

  6. 6

    Create vmbr3 — Clients Network

    Click Create → Linux Bridge one more time:

    • Name: vmbr3
    • IPv4/CIDR: leave blank
    • Bridge ports: leave empty
    • Comment: Clients Network 10.0.3.0/24

    Click Create.

  7. 7

    Apply Configuration

    After creating all three bridges, click the Apply Configuration button at the top of the Network panel. This writes the changes to /etc/network/interfaces and brings the new bridges up — without requiring a reboot.

    A progress dialog will appear briefly. Once it closes, all four bridges (vmbr0–vmbr3) should show as active.

4 CLI Method (Alternative — SSH to Proxmox Host)

If you prefer the command line, or the web UI is unavailable, you can edit /etc/network/interfaces directly via SSH. This is equivalent to what the web UI does behind the scenes.

Open /etc/network/interfaces in a text editor (e.g., nano) and append the following stanzas after the existing vmbr0 block:

/etc/network/interfaces — additions 
auto vmbr1
iface vmbr1 inet manual
    bridge-ports none
    bridge-stp off
    bridge-fd 0
    # Management Network 10.0.1.0/24

auto vmbr2
iface vmbr2 inet manual
    bridge-ports none
    bridge-stp off
    bridge-fd 0
    # Servers Network 10.0.2.0/24

auto vmbr3
iface vmbr3 inet manual
    bridge-ports none
    bridge-stp off
    bridge-fd 0
    # Clients Network 10.0.3.0/24

Why inet manual? Setting the address family to manual means the Proxmox host itself holds no IP address on these bridges. pfSense will own those subnets — the host just provides the virtual switch layer.

After saving the file, activate the new configuration with either command:

Bash — activate bridges 
# Preferred (Proxmox/Debian tool — reloads only changed interfaces)
ifreload -a

# Alternative (restarts all networking — may briefly drop SSH)
systemctl restart networking
⚠
Use ifreload -a when possible. Running systemctl restart networking will briefly drop all network connections, including your SSH session. If you are connected over SSH, prefer ifreload -a, which reloads only the changed interfaces without interrupting existing connections.

5 Verification

After applying the configuration (via web UI or CLI), confirm that all four bridges exist and are operational. Run the following commands on the Proxmox host:

Bash — verify bridges 
# List all bridges and their member ports
brctl show

# Alternative: list bridges using the ip command
ip link show type bridge

# Check each bridge individually
ip link show vmbr1
ip link show vmbr2
ip link show vmbr3

Expected Output

All four bridges (vmbr0, vmbr1, vmbr2, vmbr3) should appear. Their state will show as either UP or UNKNOWN.

Expected output — brctl show 
bridge name    bridge id               STP enabled    interfaces
vmbr0          8000.xxxxxxxxxxx        no             eno1
vmbr1          8000.000000000000       no
vmbr2          8000.000000000000       no
vmbr3          8000.000000000000       no
UNKNOWN state is normal. A bridge with no active VM ports attached will report UNKNOWN rather than UP in some kernel versions. This is expected behavior — the bridge is operational and ready to accept VM interfaces. Once pfSense or any VM is attached to a bridge, the state will show UP.

6 Bridge Summary Table

Reference this table when configuring VM network adapters in subsequent build plans. The pfSense Interface column shows the virtual NIC name that pfSense will see inside its guest OS (VirtIO naming).

Bridge Role Subnet pfSense Interface Notes
vmbr0 WAN 192.168.50.0/24 vtnet0 Physical NIC; DHCP from home router; Proxmox host IP here
vmbr1 Management 10.0.1.0/24 vtnet1 (LAN) Admin access only; mgmt-vm at 10.0.1.10
vmbr2 Servers 10.0.2.0/24 vtnet2 (OPT1) linux-server (.10), dc01 (.11), splunk-vm (.20)
vmbr3 Clients 10.0.3.0/24 vtnet3 (OPT2) linux-client; DHCP pool 10.0.3.100–200
⚠
Do not assign IP addresses to vmbr1, vmbr2, or vmbr3 on the Proxmox host. pfSense will own those subnets entirely. Assigning host IPs to these bridges creates routing conflicts — the Proxmox host would compete with pfSense as a router on the same segment, causing unpredictable behavior and potential network loops.

What Comes Next

With all four bridges in place, the Proxmox host is ready for VM creation. The next step is Build 02: pfSense VM Build, which will create and configure the virtual firewall/router that ties all four bridges together into a routed multi-segment network.

Integration 14: Windows Universal Forwarder Configuration

Install and configure Splunk Universal Forwarder on dc01 to forward Windows Event Logs to splunk-vm (Part C.2 — 8 pts)

Source: dc01 (10.0.2.11)
Destination: splunk-vm (10.0.2.20:9997)
Rubric: Part C.2 — 8 pts

✓ Part C.2 Rubric Summary

This integration covers installing and configuring the Splunk Universal Forwarder on dc01 (the Domain Controller). All four tasks below must be completed and documented for full credit.

Task Points
Install Universal Forwarder 2
Configure Outputs (point to Splunk) 2
Configure Inputs (Windows Event Logs) 3
Verify data appearing in Splunk 1
Total C.2 8

1 Overview

Windows Event Logs are forwarded from dc01 (the Domain Controller) to splunk-vm using the Splunk Universal Forwarder for Windows. Security, System, and Application logs will be indexed in the windows_logs index. As a Domain Controller, dc01 generates rich security events for authentication, account management, and service changes.

Key Event IDs that will become searchable in Splunk:

Event ID Category Description
4624 Logon Successful logon
4625 Logon Failed logon
4634 Logoff Account logged off
4720 Account Mgmt User account created
4722 Account Mgmt User account enabled
4740 Account Mgmt Account locked out
7045 Service New service installed
7040 Service Service start type changed
Prerequisite: Integration 12 (Splunk Deployment) must be complete, with a TCP receiving input configured on port 9997 and the windows_logs index created on splunk-vm before events will be accepted.

2 Download the Windows Universal Forwarder

On dc01, open a browser and download the Windows MSI installer from the Splunk website. A free Splunk account login is required to access the download page and generate a valid download link.

⚠
Version note: Version 10.2.2 does not exist — Splunk skipped from 10.2.1 directly to 10.2.3. Download version 10.2.3, which matches the Linux Universal Forwarder installed in Integration 13. The download page provides a wget command, but a direct browser download works equally well.
How to download:
  1. Log in to your Splunk account at splunk.com
  2. Navigate to Downloads → Universal Forwarder
  3. Select version 10.2.3, then choose 64-bit — Windows 10, 11, Server 2019, 2022, 2025 (not the 32-bit Windows 10 option)
  4. Click the download button or copy and use the wget command shown
The file downloaded is: splunkforwarder-10.2.3-4d61cf8a5c0c-windows-x64.msi

3 Install via GUI Wizard or Command Line

GUI Wizard (recommended for first-time setup)

1 Run as Administrator
Right-click the MSI file → Run as Administrator
2 Accept License
Check the License Agreement checkbox → Next
3 Deployment type
Select "An on-premises Splunk Enterprise instance" → Next
4 Create credentials
Username: admin   Password: ForwarderPass1! — use this exact password so it matches the rest of this guide
5 Receiving Indexer
Hostname or IP: 10.0.2.20   Port: 9997
6 Complete
Click Install → wait for completion → Click Finish

Command Line (silent installation)

Run from an Administrator PowerShell prompt in the same directory as the MSI. This method sets the receiving indexer and enables the three main Windows Event Log channels automatically.

PowerShell — silent install (Administrator)
# Silent install with predefined receiving indexer
msiexec.exe /i splunkforwarder-10.2.3-4d61cf8a5c0c-windows-x64.msi SPLUNKUSERNAME=admin `
  SPLUNKPASSWORD="ForwarderPass1!" `
  RECEIVING_INDEXER="10.0.2.20:9997" `
  WINEVENTLOG_SEC_ENABLE=1 `
  WINEVENTLOG_SYS_ENABLE=1 `
  WINEVENTLOG_APP_ENABLE=1 `
  AGREETOLICENSE=Yes /quiet

# Wait for installation to complete, then verify
Get-Service SplunkForwarder
Install path: The forwarder installs to C:\Program Files\SplunkUniversalForwarder\. All configuration files are under ...\etc\system\local\. Do not modify files in the default\ subdirectory — always place customizations in local\.

4 Configure inputs.conf

After installation, create or update inputs.conf to specify which Windows Event Log channels to monitor. Two methods are provided — use whichever fits your workflow.

Method 1 — Notepad (easiest when typing manually)

Open Notepad directly to the target path. Notepad will create the file if it does not already exist.

PowerShell — dc01 (Administrator)
notepad "C:\Program Files\SplunkUniversalForwarder\etc\system\local\inputs.conf"

In Notepad, type the following content exactly, then save (File → Save) and close:

inputs.conf — type this in Notepad
[WinEventLog://Security]
disabled = 0
index = windows_logs
evt_resolve_ad_obj = 1

[WinEventLog://System]
disabled = 0
index = windows_logs

[WinEventLog://Application]
disabled = 0
index = windows_logs

[WinEventLog://Microsoft-Windows-Sysmon/Operational]
disabled = 0
index = windows_logs
renderXml = true

Method 2 — PowerShell (when copy-paste is available)

PowerShell — dc01 (Administrator)
$confPath = "C:\Program Files\SplunkUniversalForwarder\etc\system\local\inputs.conf"

@(
  "[WinEventLog://Security]",
  "disabled = 0",
  "index = windows_logs",
  "evt_resolve_ad_obj = 1",
  "",
  "[WinEventLog://System]",
  "disabled = 0",
  "index = windows_logs",
  "",
  "[WinEventLog://Application]",
  "disabled = 0",
  "index = windows_logs",
  "",
  "[WinEventLog://Microsoft-Windows-Sysmon/Operational]",
  "disabled = 0",
  "index = windows_logs",
  "renderXml = true"
) | Set-Content -Path $confPath -Encoding UTF8

Verify the file was written

PowerShell — dc01 (Administrator)
Test-Path "C:\Program Files\SplunkUniversalForwarder\etc\system\local\inputs.conf"
Get-Content "C:\Program Files\SplunkUniversalForwarder\etc\system\local\inputs.conf"
Channel Key Events Note
WinEventLog://Security 4624, 4625, 4720, 4740, … evt_resolve_ad_obj=1 resolves AD SIDs to names
WinEventLog://System 7045, 7040, hardware errors System and service events
WinEventLog://Application Application-level errors and info App crashes, .NET runtime events
WinEventLog://…Sysmon/Operational Process creation, network connections Only if Sysmon is installed on dc01
Sysmon stanza: The Sysmon channel stanza is included but will only produce data if Sysmon is installed on dc01. If Sysmon is not installed, this stanza is harmless — the forwarder will skip it.

5 Verify outputs.conf

If the receiving indexer was set correctly during installation, outputs.conf will already exist in the local\ directory. Navigate there and check.

PowerShell — dc01 (Administrator)
# Navigate to the forwarder local config directory
cd "C:\Program Files\SplunkUniversalForwarder\etc\system\local"

# List files — confirm outputs.conf is present and non-zero size
ls

# Open outputs.conf to view and verify the contents
notepad outputs.conf
Verify the server line: In Notepad, confirm the file contains server = 10.0.2.20:9997. A typo in the IP address is a common cause of a forwarder that installs correctly but never sends data.
If outputs.conf is missing: The installer did not set a receiving indexer. Create the file in Notepad with the following content, save it to the local\ directory, and proceed to Section 6.
outputs.conf — type this in Notepad if file is missing
[tcpout]
defaultGroup = splunk-indexers

[tcpout:splunk-indexers]
server = 10.0.2.20:9997

6 Restart and Verify Forwarder Service

PowerShell — dc01 (Administrator)
# Restart the SplunkForwarder service
Restart-Service SplunkForwarder

# Check service status
Get-Service SplunkForwarder | Select Name, Status, StartType

# Verify connection to Splunk indexer via CLI (requires forwarder admin password)
& "C:\Program Files\SplunkUniversalForwarder\bin\splunk.exe" list forward-server `
  -auth admin:ForwarderPass1!
Expected output
Active forwards:
    10.0.2.20:9997
Configured but inactive forwards:
    (none)
If 10.0.2.20:9997 appears under "Configured but inactive forwards," the forwarder cannot reach splunk-vm on port 9997. Check the Windows Firewall on dc01 (ensure outbound TCP 9997 is allowed) and verify that Splunk Enterprise on splunk-vm is running.
CLI login fails? The list forward-server command requires the forwarder's management password, but the actual log forwarding does not — it runs entirely from outputs.conf. If the service shows Running, skip the CLI check and verify by running index=windows_logs | head 20 in Splunk web (Section 7). If that returns events, forwarding is confirmed working and the CLI password issue can be ignored. You can also confirm the connection via the forwarder log without any password: 
Get-Content "C:\Program Files\SplunkUniversalForwarder\var\log\splunk\splunkd.log" |
  Select-String "tcpout|connect|forward" |
  Select-Object -Last 20
Look for lines containing Connected to idx or Forwarding to as confirmation the connection is established.
⚠
Start type: Confirm the StartType column shows Automatic. If it shows Manual, the forwarder will not start after a reboot. Fix with: Set-Service SplunkForwarder -StartupType Automatic
Login failed on list forward-server? If the GUI wizard was used and a different password was set in Step 4, the -auth admin:ForwarderPass1! flag will fail. Reset the password using user-seed.conf:
  1. Run Stop-Service SplunkForwarder in Administrator PowerShell
  2. Run notepad "C:\Program Files\SplunkUniversalForwarder\etc\system\local\user-seed.conf"
  3. Type the following, then save and close Notepad: 
    [user_info]
USERNAME = admin
PASSWORD = ForwarderPass1!
  4. Run Start-Service SplunkForwarder — Splunk reads the file on startup, resets the password, and deletes it automatically
  5. Retry: & "C:\Program Files\SplunkUniversalForwarder\bin\splunk.exe" list forward-server -auth admin:ForwarderPass1!

7 Verify Data in Splunk Web

From mgmt-vm → Firefox → http://10.0.2.20:8000 → Search & Reporting:

  1. Run index=windows_logs | head 20 — should return Windows Event Log entries within 1–2 minutes of restarting the service
  2. Test specific sourcetype queries below
  3. Verify Security events with specific EventCode filters
Splunk Search — verify Windows sourcetypes
index=windows_logs sourcetype=WinEventLog:Security | head 10
index=windows_logs sourcetype=WinEventLog:System | head 10
index=windows_logs EventCode=4624 | head 5

Generate test events on dc01 to verify forwarding

The PowerShell snippet below attempts to authenticate with a nonexistent account, which generates a 4625 (Failed Logon) event in the Security log. Then search Splunk to confirm end-to-end delivery.

PowerShell — dc01 — generate a 4625 event
# Generate a test security event (failed login attempt)
$cred = New-Object System.Management.Automation.PSCredential(
  "nonexistentuser",
  (ConvertTo-SecureString "wrongpassword" -AsPlainText -Force)
)
# This will fail and generate EventCode 4625 in the Security log

# Check Event Viewer locally to confirm the event exists
Get-EventLog -LogName Security -Newest 5 | Select EventID, Message
Splunk Search — confirm 4625 event received
index=windows_logs EventCode=4625 | head 5
Success criteria: The 4625 event should appear in Splunk within 60 seconds of the failed logon attempt on dc01. If it does not appear after 2 minutes, verify the SplunkForwarder service is running, the Security channel is enabled in inputs.conf, and the windows_logs index exists on splunk-vm.

8 Validate Both Forwarders Together

With both Integration 13 (Linux) and Integration 14 (Windows) complete, run the unified query below from Splunk web to confirm data is flowing from both sources simultaneously.

Splunk Search — http://10.0.2.20:8000
# Both sources active
index=linux_logs OR index=windows_logs
| stats count by index, host

The results table should show both sources with event counts:

index host count
linux_logs linux-server > 0 events
windows_logs dc01 > 0 events
Submission documentation: Take a screenshot of this query result showing both linux_logs from linux-server and windows_logs from dc01 with non-zero event counts. This screenshot is the primary evidence for Part C (both C.1 and C.2) verification.
Congratulations: With both forwarders active and confirmed in Splunk, your SIEM pipeline is complete. linux-server and dc01 are continuously shipping logs to splunk-vm, enabling centralized security monitoring across your entire lab environment. Integration 15 (Backup Strategy) is the final step.

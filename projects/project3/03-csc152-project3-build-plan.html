1

Why Use a Template?

A Proxmox template is a VM that has been converted to a read-only, non-bootable master image. Once converted, you can clone the template to produce fully independent VMs in seconds, without repeating a full OS installation each time.

In this project, we create one template (ID 9001) based on Ubuntu Server 22.04 LTS. That single template is then cloned twice: once for linux-server (this build) and again later for splunk-vm (Build 07). Every clone starts from the same validated, cleaned-up baseline, which means:

  • Consistency — every Ubuntu VM has the same package versions, tools, and configuration starting point.
  • Speed — cloning takes seconds; a full install takes 10–20 minutes.
  • Cleanliness — the template is prepared with a cleared machine-id and SSH host keys, so each clone gets its own unique identity on first boot.
  • Safety — the template itself is never started after conversion, so it cannot drift from its known-good state.

The template VM uses modest resources (1 socket, 2 cores, 2 GB RAM, 20 GB disk). After cloning, you adjust each clone’s resources to match what that VM actually needs.

Download Ubuntu Server 22.04 LTS ISO

Option A — Download via browser, then upload to Proxmox
  1. Go to ubuntu.com/download/server and download Ubuntu Server 22.04.x LTS (the .iso file).
  2. In the Proxmox web UI, navigate to:
    Datacenter → pve → local → ISO Images → Upload
  3. Select the downloaded ISO and click Upload. Wait for the upload to complete.
Option B — Download directly on the Proxmox host (CLI)

SSH into the Proxmox host and run:

bash
cd /var/lib/vz/template/iso/
wget https://releases.ubuntu.com/22.04/ubuntu-22.04.5-live-server-amd64.iso

The ISO will appear under local → ISO Images in the Proxmox UI automatically.

3

Create the Template Base VM (ID 9001)

Step 1 — Create VM in Proxmox

In the Proxmox web UI, click Create VM (top-right) and fill in each tab:

  • General: VM ID: 9001 — Name: ubuntu-2204-template
  • OS: Select the Ubuntu Server 22.04 ISO — Guest OS: Linux, version 6.x – 2.6 Kernel
  • System: SCSI Controller: VirtIO SCSI Single — Machine: q35
  • Disks: Bus: VirtIO Block — Storage: local-lvm — Size: 20 GB — enable Discard
  • CPU: Sockets: 1 — Cores: 2 — Type: host
  • Memory: 2048 MB
  • Network: Bridge: vmbr0 (WAN — temporary for internet access) — Model: VirtIO — uncheck Firewall
  • Confirm: Uncheck “Start after created” — then click Finish
⚠
Why vmbr0 instead of vmbr2? At this point in the build sequence, pfSense has been installed but not yet configured (that happens in Integration Plan 08). vmbr2 (Servers network) has no gateway, no DHCP server, and no internet access yet. If we attach the template VM to vmbr2 now, the Ubuntu installer cannot reach a package mirror, and post-install commands like apt update and apt install qemu-guest-agent will fail.

The solution: build the template on vmbr0, which connects to your home router (192.168.50.1) and has full internet access. After all post-install preparation is complete and before converting to a template, we switch the NIC to vmbr2. Every clone then inherits the correct Servers network.
Step 2 — Enable QEMU Guest Agent in Proxmox Options

The qemu-guest-agent is a small service installed inside the VM that allows Proxmox to report the VM’s IP address and perform clean shutdowns. Before the service can run, Proxmox must be told to expose the VirtIO serial channel to the VM — otherwise the service will fail with a dependency error on first start.

  1. With VM 9001 stopped, select it in the left panel.
  2. Click Options (in the VM detail tabs).
  3. Double-click QEMU Guest Agent.
  4. Check Enabled, then click OK.
⚠
Do this before starting the VM. If the VM is already running when you enable this option, the VirtIO port will not appear until the VM is fully powered off and restarted (not just rebooted from inside). Skipping this step causes qemu-guest-agent.service to fail with “Dependency failed” even after the package is installed.
Step 3 — Start VM 9001 and open the console

Select VM 9001 in the left panel, click Start, then click Console to open the noVNC console. The Ubuntu Server installer will boot from the ISO.

Ubuntu Server 22.04 Installation Walkthrough

Walk through the Ubuntu Server installer screens in order:

  1. Language → English
  2. Installer update → Continue without updating
    (Saves time in the lab; packages will be updated after install.)
  3. Keyboard configuration → English (US) / English (US)
  4. Network configuration → Leave as DHCP. Your home router (192.168.50.1) will assign an address automatically because the VM is on vmbr0.

  5. Proxy → Leave blank, press Enter.
  6. Mirror address → Accept the default or use:
    http://archive.ubuntu.com/ubuntu
  7. Storage configuration → Use an entire disk.
    LVM is acceptable; keep defaults. This is a template — storage layout does not need to be elaborate.
  8. Storage summary → Review, then select Done and confirm the destructive action.
  9. Profile setup — enter the following:
    • Your name: sysadmin
    • Your server’s name: ubuntu-template
    • Pick a username: sysadmin
    • Choose a password: Choose a strong password and record it. Students will change this after cloning.
  10. SSH Setup → Check Install OpenSSH server: Yes.
    Import SSH identity: No.
  11. Featured server snaps → Skip all. Do not select any snaps.
  12. Installation complete → Select Reboot Now.
    When prompted, press Enter to remove the installation media (Proxmox unmounts the ISO automatically).

Post-Install Template Preparation

After VM 9001 reboots, log in via the Proxmox console as sysadmin. Run the following commands to prepare a clean template. Read the inline comments carefully — each block serves a specific purpose.

bash — run on VM 9001 as sysadmin
# Update all packages to their latest versions
sudo apt update && sudo apt upgrade -y

# Install qemu-guest-agent (required for Proxmox IP reporting and clean shutdown)
sudo apt install -y qemu-guest-agent
# Note: Ubuntu starts qemu-guest-agent automatically after install via udev/socket
# activation — "systemctl enable" is not needed and will produce a harmless warning.
# Verify it is running:
sudo systemctl status qemu-guest-agent

# Install common administrative tools
sudo apt install -y curl wget net-tools vim htop

# Clean up apt cache to reduce the template's disk footprint
sudo apt clean
sudo apt autoremove -y

# Clear machine-id so each clone gets a unique ID on first boot
sudo truncate -s 0 /etc/machine-id
sudo truncate -s 0 /var/lib/dbus/machine-id

# Clear SSH host keys so each clone generates its own unique key pair
sudo rm -f /etc/ssh/ssh_host_*

# Clear bash history (both root and sysadmin)
history -c
sudo sh -c 'cat /dev/null > /root/.bash_history'
cat /dev/null > ~/.bash_history

# Shut down cleanly before converting to template
sudo poweroff
ℹ
Why clear machine-id? The /etc/machine-id file is a unique identifier used by systemd and other services. If every clone shares the same machine-id, DHCP servers and some applications will treat them as the same machine. Truncating it (not deleting it) causes systemd to regenerate a new unique ID on the first boot of each clone.
5b

Switch NIC from vmbr0 to vmbr2 (Before Templating)

VM 9001 is now powered off. Before converting it to a template, switch its network bridge from vmbr0 (temporary WAN used for internet access during setup) to vmbr2 (Servers network). Every clone — linux-server and splunk-vm — will then automatically inherit the correct bridge.

Step 1 — Open VM 9001 Hardware settings

In the Proxmox web UI, select VM 9001 in the left panel, then click Hardware.

Step 2 — Edit the Network Device
  1. Double-click the Network Device (net0) entry in the hardware list.
  2. In the Bridge dropdown, change vmbr0 → vmbr2.
  3. Leave all other settings (Model: VirtIO, Firewall: unchecked) unchanged.
  4. Click OK.
Step 3 — Verify the change

The Hardware list should now show:

Proxmox Hardware tab — expected entry
Network Device (net0)    virtio=XX:XX:XX:XX:XX:XX,bridge=vmbr2

Confirm vmbr2 appears before proceeding to Section 6.

Step 4 — Enable QEMU Guest Agent in Proxmox Options
  1. With VM 9001 still selected, click the Options tab.
  2. Double-click QEMU Guest Agent in the options list.
  3. Check Enabled → click OK.
ℹ️
This Proxmox-side flag adds the virtio-serial channel the agent uses to communicate with the hypervisor. Without it, Proxmox cannot display the VM's IP in the Summary tab, perform clean shutdowns, or freeze the filesystem for consistent snapshots — even if qemu-guest-agent is installed inside the VM. Because this setting is stored in the template config, every clone (linux-server, splunk-vm) inherits it automatically.
⚠
Do not start VM 9001 again after this switch. On vmbr2 there is no gateway yet (pfSense OPT1 is configured in Integration Plan 08), so the VM would have no network. There is no need to boot it — go directly to Section 6 and convert to template.
6

Convert VM 9001 to a Template

VM 9001 is powered off and its NIC is now set to vmbr2. Convert it to a template:

  1. In the Proxmox left panel, right-click on VM 9001.
  2. Select Convert to Template.
  3. Confirm the dialog that appears.
  4. VM 9001 will now show a template icon (a stack of pages) in the left panel instead of a normal VM icon.
⚠
Template is now read-only — do not try to start it directly. Once converted, VM 9001 cannot be started or modified. It exists only as a cloning source. If you later need to update the template (e.g., to apply newer packages), clone it, make your changes, power it off, and convert the clone to a new template. You can then delete or archive the old one.

Clone Template to Create linux-server (ID 1002)

Step 1 — Clone the template
  1. Right-click template 9001 → Clone.
  2. Set the following options:
    • Mode: Full Clone — not Linked Clone. A full clone is completely independent of the template.
    • Target Node: pve
    • VM ID: 1002
    • Name: linux-server
    • Target Storage: local-lvm
  3. Click Clone and wait for the task to complete (watch the task log at the bottom of the screen).
Step 2 — Adjust resources for linux-server

The clone inherits the template’s minimal specs. Increase them for linux-server:

  1. Select VM 1002 → Hardware.
  2. Memory: double-click → change to 4096 MB (4 GB).
  3. Processors: already set to 2 cores — leave as-is.
  4. Disk resize: select the disk (e.g., virtio0) → click Disk Action → Resize → add 20 GB (brings total to 40 GB).

After the first boot, expand the filesystem to use the new space (see Section 8).

8

First Boot Configuration of linux-server

Start VM 1002 in Proxmox, then open the console and log in as sysadmin.

Set the hostname
bash
sudo hostnamectl set-hostname linux-server
Check the network interface name

Interface names vary. Identify yours before editing Netplan:

bash
ip link show

Look for the primary Ethernet interface — it will be ens18, ens3, enp6s18, or similar. Use that name in the Netplan file below. The Netplan file uses ens18 as the example — if your interface is named anything else, replace every occurrence of ens18 in that file with your actual interface name before saving.

Configure static IP via Netplan
bash
sudo nano /etc/netplan/00-installer-config.yaml

Replace the file contents with the following (adjust the interface name if needed):

yaml — /etc/netplan/00-installer-config.yaml
network:
  version: 2
  ethernets:
    ens18:
      dhcp4: false
      addresses:
        - 10.0.2.10/24
      routes:
        - to: default
          via: 10.0.2.1
      nameservers:
        addresses:
          - 10.0.2.11
          - 8.8.8.8

Apply the configuration:

bash
sudo netplan apply
Verify IP address assignment
bash
ip addr show ens18

Confirm that 10.0.2.10/24 appears on the interface. Gateway connectivity (ping 10.0.2.1) is verified in Integration 08 after pfSense is fully configured.

Expand the filesystem to use the resized disk (if needed)

If you used LVM during installation:

bash — LVM layout
# Expand the partition
sudo growpart /dev/vda 3

# Resize the logical volume and filesystem
sudo lvextend -l +100%FREE /dev/mapper/ubuntu--vg-ubuntu--lv
sudo resize2fs /dev/mapper/ubuntu--vg-ubuntu--lv

If you used a simple partition (no LVM):

bash — simple partition layout
sudo growpart /dev/vda 2
sudo resize2fs /dev/vda2

Confirm the result:

bash
df -h /
9

Verify VM Identity

Run these commands to confirm linux-server has the correct identity and address:

bash
# Confirm hostname
hostname

# Confirm IP address assignment
ip addr show

# Confirm machine-id is populated (not empty)
# A non-empty machine-id means systemd regenerated a unique ID on first boot
cat /etc/machine-id

Expected results:

  • hostname returns linux-server
  • ip addr show shows 10.0.2.10/24 on the primary interface
  • cat /etc/machine-id returns a 32-character hex string (not empty)

If all three checks pass, linux-server is complete. This VM will later receive DNS secondary configuration, Samba file sharing, and the Splunk Universal Forwarder. Those services are configured in their respective Integration Plan builds (08–13).

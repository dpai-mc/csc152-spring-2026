1 Pre-Hardening Baseline Snapshot

Before applying any changes, take a Proxmox snapshot of dc01 (VM 1003) and capture the current state of all running services. This gives you a rollback point and satisfies the rubric requirement to document the before state.

Step 1 — Proxmox snapshot

In the Proxmox web UI: VM 1003 → Snapshots → Take Snapshot

  • Name: pre-hardening-baseline
  • Description: Before CIS hardening (Part A.2)
  • Leave "Include RAM" unchecked unless the VM is powered on and you want a live snapshot.

Step 2 — Document current state via PowerShell

Run this on dc01 to export a CSV of all currently running services for your submission:

PowerShell — dc01
# Take Proxmox snapshot of VM 1003 before hardening
# In Proxmox UI: VM 1003 → Snapshots → Take Snapshot
# Name: pre-hardening-baseline
# Then in PowerShell — document current state:
Get-Service | Where-Object {$_.Status -eq 'Running'} |
  Select-Object Name, DisplayName |
  Export-Csv "$env:USERPROFILE\before-services.csv" -NoTypeInformation
Submission tip: Save the CSV file as before-services.csv and include it with your project submission along with an after-hardening export to demonstrate the before/after comparison.

2 CIS Control 1 — Service Hardening

⚙ Control 1 2 pts CIS Level 1 — Windows Services

Disable Windows services that are unnecessary on a domain controller. Fewer running services means a smaller attack surface. The script below checks for each service before attempting to disable it, so it is safe to run even if a service is not installed.

Before

Before
PowerShell — document running services
# Document running services
Get-Service | Where-Object {$_.Status -eq 'Running'} | Select Name, DisplayName

Apply — Disable unnecessary services

Check each service for existence before disabling. These are common on Windows Server 2022 but may not all be present depending on your install:

PowerShell — disable unnecessary services
$servicesToDisable = @(
  "Fax",              # Fax service
  "XblAuthManager",  # Xbox Live Auth
  "XblGameSave",     # Xbox Live Game Save
  "XboxNetApiSvc",   # Xbox Live Networking
  "WMPNetworkSvc",   # Windows Media Player Network Sharing
  "RemoteRegistry",  # Remote Registry (security risk)
  "Browser",         # Computer Browser (deprecated)
  "SSDPSRV",         # SSDP Discovery (UPnP)
  "upnphost",        # UPnP Device Host
  "WerSvc"           # Windows Error Reporting
)

foreach ($svc in $servicesToDisable) {
  $service = Get-Service -Name $svc -ErrorAction SilentlyContinue
  if ($service) {
    Set-Service -Name $svc -StartupType Disabled -ErrorAction SilentlyContinue
    Stop-Service -Name $svc -Force -ErrorAction SilentlyContinue
    Write-Host "Disabled: $svc" -ForegroundColor Green
  } else {
    Write-Host "Not found: $svc" -ForegroundColor Yellow
  }
}

After

After
PowerShell — verify running services
Get-Service | Where-Object {$_.Status -eq 'Running'} | Select Name, DisplayName
What to look for: The services listed in $servicesToDisable should no longer appear in the Running state. Export the after list to after-services.csv for your submission.

3 CIS Control 2 — Windows Defender Firewall

🔒 Control 2 2 pts CIS Level 1 — Firewall & Network Protection

Enable Windows Defender Firewall on all three profiles (Domain, Private, Public) and set the default to block all inbound traffic. Then add explicit allow rules for the protocols that a domain controller legitimately needs to accept.

⚠
Critical: Always add the AD DS and RDP allow rules before setting the default inbound action to Block. If you set Block first and lose your RDP session, you will need to use the Proxmox console to recover.

Before

Before
PowerShell — check current firewall state
Get-NetFirewallProfile | Select Name, Enabled, DefaultInboundAction, DefaultOutboundAction

Apply — Enable firewall and configure rules

PowerShell — configure Windows Defender Firewall
# Enable firewall on all profiles
Set-NetFirewallProfile -Profile Domain,Public,Private -Enabled True

# Set default inbound block, outbound allow
Set-NetFirewallProfile -Profile Domain,Public,Private `
  -DefaultInboundAction Block `
  -DefaultOutboundAction Allow

# Allow AD DS traffic (Domain Controller — essential)
$adRules = @(
  @{Name="AD-LDAP";        Port=389;  Protocol="TCP"},
  @{Name="AD-LDAPS";       Port=636;  Protocol="TCP"},
  @{Name="AD-Kerberos-TCP";Port=88;   Protocol="TCP"},
  @{Name="AD-Kerberos-UDP";Port=88;   Protocol="UDP"},
  @{Name="AD-DNS-TCP";     Port=53;   Protocol="TCP"},
  @{Name="AD-DNS-UDP";     Port=53;   Protocol="UDP"},
  @{Name="AD-GC";          Port=3268; Protocol="TCP"},
  @{Name="AD-RPC";         Port=135;  Protocol="TCP"},
  @{Name="RDP";            Port=3389; Protocol="TCP"},
  @{Name="WinRM";          Port=5985; Protocol="TCP"}
)
foreach ($rule in $adRules) {
  New-NetFirewallRule -DisplayName $rule.Name `
    -Direction Inbound `
    -Protocol $rule.Protocol `
    -LocalPort $rule.Port `
    -Action Allow `
    -ErrorAction SilentlyContinue
}

# Allow Splunk forwarder outbound
New-NetFirewallRule -DisplayName "Splunk-Forwarder-Out" `
  -Direction Outbound -Protocol TCP `
  -RemotePort 9997 -RemoteAddress 10.0.2.20 `
  -Action Allow

# Verify
Get-NetFirewallProfile | Select Name, Enabled, DefaultInboundAction

After — Expected output

After
Profile Enabled DefaultInboundAction DefaultOutboundAction
DomainTrueBlockAllow
PrivateTrueBlockAllow
PublicTrueBlockAllow

4 CIS Control 3 — Account Policies

👤 Control 3 3 pts CIS Level 1 — Account & Lockout Policy

Because dc01 is the Domain Controller for techstart.local, the account policy is set on the Default Domain Policy GPO and applies to all domain accounts. Use Set-ADDefaultDomainPasswordPolicy to configure both password and lockout settings in a single command.

Prerequisite — verify ADWS is running. The Set-ADDefaultDomainPasswordPolicy cmdlet requires the Active Directory Web Services (ADWS) service. Check it before running the policy command:
PowerShell — check and start ADWS
# Check ADWS status
Get-Service ADWS

# If Status is not Running, start it and set to automatic
Set-Service ADWS -StartupType Automatic
Start-Service ADWS
If ADWS shows Running but the policy command still fails with ADServerDownException, the -Server parameter in the command below resolves this by explicitly targeting dc01.
PowerShell — configure domain password and lockout policy
# Configure domain password policy
Set-ADDefaultDomainPasswordPolicy `
  -Identity "techstart.local" `
  -Server "dc01.techstart.local" `
  -MinPasswordLength 14 `
  -PasswordHistoryCount 24 `
  -MaxPasswordAge (New-TimeSpan -Days 60) `
  -MinPasswordAge (New-TimeSpan -Days 1) `
  -ComplexityEnabled $true `
  -ReversibleEncryptionEnabled $false `
  -LockoutThreshold 5 `
  -LockoutDuration (New-TimeSpan -Minutes 30) `
  -LockoutObservationWindow (New-TimeSpan -Minutes 30)

# Verify
Get-ADDefaultDomainPasswordPolicy -Server "dc01.techstart.local"

Policy Settings — CIS Reference

6 CIS Control 5 — Security Options

🔐 Control 5 2 pts CIS Level 1 — Local Security Policy

These settings harden local security policy options that are not covered by other controls. They can be applied via secpol.msc (Local Security Policy) or directly via registry edits in PowerShell. The registry approach below is repeatable and scriptable.

PowerShell — apply security option registry settings
# Do not display last username on login screen
Set-ItemProperty -Path "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System" `
  -Name "DontDisplayLastUserName" -Value 1

# Require Ctrl+Alt+Del for login
Set-ItemProperty -Path "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System" `
  -Name "DisableCAD" -Value 0

# UAC — notify but don't dim desktop (balanced for server use)
Set-ItemProperty -Path "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System" `
  -Name "ConsentPromptBehaviorAdmin" -Value 1

# Disable anonymous SID/Name translation
Set-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa" `
  -Name "RestrictAnonymousSAM" -Value 1
Set-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa" `
  -Name "RestrictAnonymous" -Value 1

# Disable LM hash storage
Set-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa" `
  -Name "NoLMHash" -Value 1

# Set NTLMv2 only (LmCompatibilityLevel 5)
Set-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa" `
  -Name "LmCompatibilityLevel" -Value 5

Settings Reference

Setting Value Why It Matters
DontDisplayLastUserName 1 Prevents attackers from seeing the last logged-in account name at the login screen
DisableCAD 0 Requires Ctrl+Alt+Del before login, preventing credential-stealing spoofed login screens
ConsentPromptBehaviorAdmin 1 UAC prompts for credentials on the secure desktop for admin actions
RestrictAnonymousSAM / RestrictAnonymous 1 Blocks unauthenticated enumeration of SAM accounts and shares
NoLMHash 1 Stops Windows from storing the weak LAN Manager password hash
LmCompatibilityLevel 5 Accept only NTLMv2 responses; refuse LM and NTLM. Mitigates pass-the-hash attacks.

7 CIS Control 6 — Windows Update

🔁 Control 6 1 pt CIS Level 1 — Patch Management

Configure Windows Update to automatically download and install patches. Because this lab environment does not use WSUS, the registry settings below point directly at Microsoft Update and schedule installs daily at 3 AM.

PowerShell — configure automatic updates
# Configure automatic updates via registry (WSUS-free, direct Microsoft Update)
$wuPath = "HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate\AU"
New-Item -Path $wuPath -Force | Out-Null
Set-ItemProperty -Path $wuPath -Name "NoAutoUpdate"         -Value 0
Set-ItemProperty -Path $wuPath -Name "AUOptions"            -Value 4  # 4 = auto download and schedule install
Set-ItemProperty -Path $wuPath -Name "ScheduledInstallDay"  -Value 0  # 0 = every day
Set-ItemProperty -Path $wuPath -Name "ScheduledInstallTime" -Value 3  # 3 AM

# Verify current update status
Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

AUOptions values reference

Value Behavior
2Notify before download
3Auto download, notify before install
4Auto download and schedule install (recommended)
5Let local admin choose

+ Additional Controls Beyond Required 6 Categories

The three controls below are not in the rubric's six named categories, but they each address a distinct CIS Benchmark recommendation. Applying them ensures you exceed the 12-control minimum and adds real security value to dc01.

Control 7 — Disable SMBv1

🔒 Control 7 CIS — SMB Protocol Hardening

SMBv1 is the protocol exploited by WannaCry, NotPetya, and EternalBlue. It has been disabled by default in modern Windows Server builds, but explicitly disabling it via configuration ensures it cannot be re-enabled accidentally.

PowerShell — disable SMBv1
# Disable SMBv1
Set-SmbServerConfiguration -EnableSMB1Protocol $false -Force

# Verify
Get-SmbServerConfiguration | Select EnableSMB1Protocol

Expected output: EnableSMB1Protocol : False

Control 8 — Configure Event Log Sizes

📄 Control 8 CIS — Event Log Management

The default Security event log size is too small for an active domain controller. Increasing log capacity prevents security events from being overwritten before Splunk can forward them.

Use wevtutil instead of Limit-EventLog. Limit-EventLog is deprecated and rejects size values on Windows Server 2022 even when the values are valid. wevtutil sl is the modern replacement and accepts the size in bytes directly. The values below are exact multiples of 64 KB as required: 512 MB = 536,870,912 bytes  |  128 MB = 134,217,728 bytes.
PowerShell — increase event log sizes
# Security log: 512 MB
wevtutil sl Security /ms:536870912

# System log: 128 MB
wevtutil sl System /ms:134217728

# Application log: 128 MB
wevtutil sl Application /ms:134217728

# Verify all three
wevtutil gl Security    | Select-String "maxSize"
wevtutil gl System      | Select-String "maxSize"
wevtutil gl Application | Select-String "maxSize"

Control 9 — Windows Defender Antivirus

🛡 Control 9 CIS — Antivirus / Endpoint Protection

Confirm that Windows Defender real-time protection is running and ensure it has not been disabled. On Windows Server 2022, Defender is the built-in antivirus and should be active by default.

PowerShell — verify and enable Windows Defender
Get-MpComputerStatus | Select AMRunningMode, RealTimeProtectionEnabled, AntivirusEnabled

# Ensure real-time monitoring is on
Set-MpPreference -DisableRealtimeMonitoring $false

Expected: RealTimeProtectionEnabled : True and AntivirusEnabled : True

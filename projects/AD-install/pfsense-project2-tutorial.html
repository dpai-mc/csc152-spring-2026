pfSense Firewall Installation & Configuration Tutorial

A step-by-step guide to deploying pfSense 2.7.2 on Proxmox, configuring network segments, DHCP, DNS, and firewall rules for Project 2.

📋 Project 2 – Network Infrastructure ⏱️ Est. 3–4 hours 🔢 pfSense CE 2.7.2 🖥️ Proxmox Environment
★

Introduction & Prerequisites

+
What this tutorial covers: Deploying pfSense 2.7.2 as a virtual firewall on your Proxmox server to create a segmented network for TechStart Solutions. When done, you will have four separate network segments controlled by pfSense, with DHCP, DNS, and firewall rules enforcing your security policy.

Prerequisites – What You Should Already Have

This tutorial assumes you completed Project 1. Before starting, confirm the following:

  • Linux Server VM running (Ubuntu Server or Rocky Linux) – hostname techstart-srv01, accessible via Proxmox console
  • Windows Server 2022 VM running – hostname TECHSTART-SRV01, accessible via Proxmox console
  • Both servers can ping each other on the existing network
  • You can log in to the Proxmox web interface at your assigned URL
  • The pfSense ISO (pfSense-CE-2.7.2-RELEASE-amd64.iso) is available in Proxmox ISO storage — see Get the pfSense ISO below if it is not yet uploaded

What We Will Build

You will reconfigure both Project 1 servers to sit on the Servers (vmbr2) network, create a new Linux client VM on the Clients (vmbr3) network, and place pfSense in the middle to control all traffic flows.

⚠️
Before You Start

Take a snapshot of your Linux Server and Windows Server VMs before making any network changes. In Proxmox, select the VM → Snapshots → Take Snapshot. Name it pre-project2. This lets you roll back if needed.

Network Segments Explained

SegmentBridgepfSense InterfaceSubnetWho Lives Here
WAN vmbr0 vtnet0 10.255.254.0/24 Campus internet uplink
MANAGEMENT vmbr1 vtnet1 (LAN) 10.0.1.0/24 Your admin access point to pfSense web GUI
SERVERS vmbr2 vtnet2 (OPT1) 10.0.2.0/24 Linux Server (.10), Windows Server (.11)
CLIENTS vmbr3 vtnet3 (OPT2) 10.0.3.0/24 Linux Client VM (DHCP assigned)
ℹ️
Why separate networks?

Network segmentation is a core security practice. Separating management traffic from server traffic and client traffic means that a compromised client cannot directly reach your servers or your management interface. pfSense's firewall rules enforce exactly what traffic is allowed between segments.

Why no IP on the bridge?

The Proxmox bridge acts as a virtual switch — it just passes frames between VMs. pfSense will have the gateway IP (10.0.1.1) on its own interface. Putting an IP on the bridge itself would create routing confusion and is not part of this design.

A.3

Install pfSense

+
Goal: Boot the pfSense VM from the ISO and complete the installation to disk. This installs the pfSense operating system. The whole process takes about 5–10 minutes.

Start the VM and Open Console

1

Start the pfSense VM

Select pfsense-fw in Proxmox → click Start. Then immediately click Console (or the >_ Console button) to open the VM's display.

2

Wait for the Boot Menu

pfSense will show a boot loader menu briefly. Do nothing — it will auto-continue in a few seconds and begin loading the installer.

3

Installation Screens

4

5

6

Choose Partition Scheme – Auto (ZFS)

Select Auto (ZFS) – Guided Root-on-ZFS using the arrow keys. Press Enter.

💡
Why ZFS?

ZFS is recommended for pfSense because it provides data integrity checking and built-in compression, which helps on virtual disks. It's also the same file system Proxmox itself uses.

7

Proceed with Installation

The ZFS configuration screen appears. Select Proceed with Installation. Press Enter.

8

Choose "Stripe – No Redundancy"

Since we have one virtual disk, select Stripe – No Redundancy. Press Enter.

9

Select the Virtual Disk

You will see the virtual disk listed (e.g., vtbd0 or da0). Press Space to select it (an asterisk [*] will appear next to it). Press Enter to continue.

10

Confirm Disk Destruction

A warning asks "Are you sure you want to destroy the current contents?" Use the arrow key to highlight YES and press Enter. This erases the virtual disk to install pfSense.

11

Wait for Installation to Complete

Installation takes 2–5 minutes. You will see progress output scrolling. When it finishes, a screen asks about manual configuration. Select No (skip manual config), then select Reboot. Press Enter.

12

Remove the ISO After Reboot

As the VM reboots, quickly go to Proxmox → pfsense-fw → Hardware → double-click the CD/DVD Drive → change it to Do not use any media. Click OK. This prevents it from booting the installer again.

D.1

Understanding pfSense Firewall Rules

+
Goal: Before creating rules, understand how pfSense processes firewall rules. Getting the order and logic wrong is the most common source of errors in this project.

Key Concepts

  • Rules are processed per interface, on incoming traffic: Firewall rules on an interface apply to traffic entering pfSense through that interface. Rules on the CLIENTS tab apply to traffic from client VMs coming into pfSense.
  • First match wins: pfSense processes rules from top to bottom. The first rule that matches the packet is applied. Rules below it are ignored for that packet.
  • Default deny: If no rule matches, traffic is blocked. This means you must explicitly allow everything you want to permit.
  • Stateful firewall: Return traffic for established connections is automatically allowed. You only need rules for the initiating direction.

Traffic Flow Matrix for Project 2

Table 2: Required Traffic Flows – Implement these as firewall rules

Management (LAN)AnyAnyPASSAdmins need full access from management network
ServersWAN (internet)80, 443PASSServers need web access for updates
ServersWAN (internet)53PASSServers need DNS to resolve external names

Rule Order Is Critical

In pfSense, BLOCK rules must come before broader PASS rules for the same traffic. If you put "Allow all traffic from Clients to Servers" before "Block SSH from Clients to Servers," the block rule will never be reached.

Troubleshooting Guide

+

Use this section when things don't work as expected. Work through issues systematically.

SymptomLikely CauseSteps to Fix
pfSense web GUI not accessible Wrong interface, network, or IP Verify your management VM is on vmbr1; confirm LAN IP is 10.0.1.1 from console; try https://10.0.1.1
DHCP not assigning addresses to client DHCP service not enabled, wrong interface Verify DHCP is enabled on CLIENTS tab; verify client VM is on vmbr3; check DHCP logs in pfSense
Client can't ping gateway (10.0.3.1) OPT2/CLIENTS interface not enabled Go to Interfaces → CLIENTS; confirm "Enable interface" is checked; confirm IP is 10.0.3.1/24
Client can't reach internet Missing CLIENTS rules, WAN NAT issue Check CLIENTS firewall rules exist; verify NAT: Firewall → NAT → Outbound; confirm "Automatic outbound NAT" is selected
Servers can't reach internet Missing SERVERS rules, NAT not covering vmbr2 Verify SERVERS rules allow outbound traffic; check NAT outbound rules cover 10.0.2.0/24
DNS not resolving internal names DNS server not configured, wrong DHCP DNS option Verify DNS server is installed on Windows/Linux Server; confirm DHCP gives 10.0.2.10 or .11 as DNS
Firewall blocking legitimate traffic Rule order wrong, missing rule Check rule order (first match wins); review pfSense firewall log to see what's being blocked; add specific allow rule before any catch-all block
Interface wrong (vtnet mapping off) Adapters added in wrong order Use pfSense console option 1 to re-assign interfaces; plug and unplug adapters from console to identify by MAC
WAN interface has no IP IPv6 preference issue Go to Interfaces → WAN; set IPv6 type to None; Save → Apply; reboot pfSense
NAT not working Outbound NAT missing rule for new networks Firewall → NAT → Outbound → Switch to "Automatic" if on Manual; or manually add rules for 10.0.2.0/24 and 10.0.3.0/24 via WAN

Useful pfSense Diagnostic Tools

🔍

Packet Capture

Diagnostics → Packet Capture — Capture traffic on a specific interface to see exactly what packets are arriving and what pfSense is doing with them. Invaluable for debugging firewall rules.

🌐

Ping from pfSense

Diagnostics → Ping — Ping from pfSense itself to test if pfSense has connectivity. Select the source interface to test routing from each segment.

📊

Firewall States

Diagnostics → States — View active firewall connection states. Look for your connections here to confirm they're being permitted or to see if states are building up unexpectedly.

📝

System Logs

Status → System Logs → Firewall — View blocked packet log entries from your logging-enabled block rules. This is the primary evidence for your firewall testing documentation.

💡
Pro Tip: Verify NAT is Enabled

One common setup issue is that NAT (Network Address Translation) isn't properly configured for the new networks. In pfSense, go to Firewall → NAT → Outbound. Make sure it's set to "Automatic Outbound NAT rule generation." This automatically creates NAT rules for all connected networks to masquerade through the WAN interface when accessing the internet.

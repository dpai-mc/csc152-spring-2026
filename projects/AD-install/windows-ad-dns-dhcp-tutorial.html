Windows Server: Active Directory, DNS & DHCP

Configure Windows Server 2022 as the TechStart Solutions domain controller — providing Active Directory authentication, internal DNS for techstart.local, and coordinated DHCP with pfSense.

📋 Project 2 – Parts B & C ⏱️ Est. 60–90 minutes 🪟 Windows Server 2022 🌐 techstart.local domain
★

Introduction & Role in Project 2

+
What this tutorial builds: Windows Server 2022 is promoted to the domain controller for techstart.local. It hosts Active Directory Domain Services (AD DS), the authoritative DNS server for the internal zone, and a coordinated DHCP setup with pfSense. Your Linux client VM will be able to resolve internal hostnames and optionally join the domain.

How the Services Fit Together

The three services configured in this tutorial work together as a chain. Understanding the flow before you begin will help you troubleshoot if something doesn't work.

  CLIENT VM (10.0.3.x)
       │
       │  DNS query: "linux.techstart.local?"
       ▼
  pfSense CLIENTS interface (10.0.3.1)
       │  DHCP option: DNS server = 10.0.2.11
       │  Routes DNS queries to Windows Server
       ▼
  Windows Server DC01 (10.0.2.11)
  ┌────────────────────────────────────────┐
  │  Active Directory Domain Services       │
  │  Domain: techstart.local               │
  │                                        │
  │  DNS Server (role)                     │
  │  ├─ Zone: techstart.local (primary)    │
  │  │   ├─ dc01.techstart.local → .11     │
  │  │   ├─ linux.techstart.local → .10    │
  │  │   └─ pfsense.techstart.local → .1   │
  │  └─ Forwarder: 8.8.8.8 / 1.1.1.1      │
  │     (for external names like google.com)│
  └────────────────────────────────────────┘
       │
       │  External query: "google.com?" → forwards to 8.8.8.8

Prepare Windows Server

+
Goal: Before installing Active Directory, verify the server has a static IP address and rename the computer to DC01. Active Directory is very particular about the server name — it gets embedded into the domain metadata and is difficult to change afterward.

Verify Static IP Address

1

Confirm IP Configuration in PowerShell

Open PowerShell (right-click the Start button → Windows PowerShell) and run:

PowerShell – Verify IP configuration 
Get-NetIPConfiguration

Confirm the output shows:

  • IPv4 Address: 10.0.2.11
  • IPv4 Default Gateway: 10.0.2.1

If these were set in the pfSense tutorial Part B.3, you are ready. If not, set them now via Network and Sharing Center → adapter settings → IPv4 properties before continuing.

⚠️
Do NOT proceed with DHCP enabled on this adapter

Active Directory domain controllers must have a static IP address. A DHCP-assigned address can change on reboot, breaking DNS and authentication for all domain members. Verify the IP is static before continuing.

2

Set DNS to Point to Itself (127.0.0.1)

After AD DS is installed, this server will be its own DNS server. Set that up now so DNS works correctly during and after promotion.

PowerShell – Set DNS server to loopback 
# Find your network adapter name first
Get-NetAdapter

# Set DNS (replace "Ethernet0" with your adapter name)
Set-DnsClientServerAddress -InterfaceAlias "Ethernet0" `
    -ServerAddresses ("127.0.0.1","8.8.8.8")
💡
Why 127.0.0.1 first?

Setting the DNS server to its own loopback address tells Windows to use the DNS Server role running locally. The secondary 8.8.8.8 provides a fallback for external resolution during the brief window before the DNS forwarder is configured.

Rename the Server to DC01

3

Rename Using PowerShell (Fastest Method)

PowerShell – Rename computer 
Rename-Computer -NewName "DC01" -Restart

The server will reboot automatically to apply the new name. Wait for it to restart and log back in as Administrator before continuing.

Alternatively, via GUI: right-click Start → System → Rename this PC → enter DC01 → Restart Now.

4

Verify the New Name After Reboot

PowerShell – Confirm hostname 
hostname

Output should be DC01. Server Manager also shows the hostname in the top-left header.

Configure DNS Forwarder

The forwarder tells this DNS server where to send queries for names it cannot resolve from its own zones (i.e., anything outside techstart.local, like google.com).

5

Add Forwarders in DNS Manager

In DNS Manager, right-click DC01 (the server root, not the zone) → Properties. Click the Forwarders tab → Edit.

Add each of the following IP addresses, pressing Enter after each:

Forwarder IPProviderPurpose
8.8.8.8GooglePrimary external DNS
1.1.1.1CloudflareSecondary external DNS

Click OK → Apply → OK.

ℹ️
Why not forward to pfSense (10.0.2.1)?

You could forward to pfSense instead of directly to 8.8.8.8. Either works. Forwarding directly to a public resolver is simpler and slightly faster for a lab. If you want to route all DNS through pfSense for logging/filtering, use 10.0.2.1 as your forwarder instead.

Update pfSense DHCP to Use This DNS Server

6

Update pfSense DHCP DNS Option

Now that the Windows Server DNS is configured, update pfSense to hand out 10.0.2.11 (Windows Server) as the DNS server to CLIENTS via DHCP.

In pfSense web GUI: Services → DHCP Server → CLIENTS tab. In the DNS Servers field change the value to 10.0.2.11. Click Save → Apply Changes.

On the Linux Client VM, renew the DHCP lease to pick up the new DNS server:

Bash (Linux Client) – Renew DHCP lease 
sudo dhclient -r ens18 && sudo dhclient ens18
resolvectl status | grep "DNS Servers"

The DNS Server should now show 10.0.2.11.

Verify DNS from the Linux Client

DHCP: pfSense vs. Windows Server

+
Decision point: Project 2 uses pfSense to provide DHCP for the CLIENTS network. This section explains that design choice and what Windows Server's DHCP role would look like — for reference and for understanding how AD-integrated DHCP works in production.

Why pfSense Handles DHCP in This Project

In a real enterprise, either the domain controller or a dedicated DHCP server typically provides DHCP. In Project 2, pfSense handles DHCP for the CLIENTS network (10.0.3.0/24) for several reasons:

  • pfSense's DHCP server is already configured and functional from Part B.1
  • It avoids a DHCP conflict between two servers on the same segment
  • It demonstrates network-layer DHCP (router-based) vs. server-based DHCP — both are common in real networks
  • The SERVERS network (10.0.2.0/24) where Windows Server lives uses static IPs, so no DHCP is needed there
ℹ️
You do NOT need to install Windows Server DHCP for Project 2

The DHCP role on Windows Server is not required to satisfy any Project 2 deliverable. pfSense is already handling DHCP for the CLIENTS network. Installing a second DHCP server on the same network segment would create IP address conflicts.

How Windows Server DHCP Would Integrate with AD (Reference)

In production environments, Windows Server DHCP integrates tightly with Active Directory through DHCP Failover, Dynamic DNS Updates, and DHCP Authorization. Key concepts:

Table 3: Windows DHCP + AD Integration Features (Reference)
FeatureWhat It DoesRelevance to Project 2
DHCP AuthorizationAD requires DHCP servers to be authorized before they can hand out leases — prevents rogue DHCP serversNot needed; pfSense is the DHCP server and is not AD-aware
Dynamic DNS UpdateWindows DHCP automatically adds/removes A records in DNS when leases are assignedNot used; you manually create DNS A records in this project
DHCP FailoverTwo Windows DHCP servers share a scope for redundancyNot needed in a single-server lab
DHCP Scope OptionsLike pfSense, can distribute DNS server, gateway, domain name via DHCPpfSense handles this for CLIENTS network
💡
Want to explore Windows DHCP?

If you'd like to see how Windows Server DHCP works, you can install the role on your Windows Server and configure a scope for a different subnet — just make sure it doesn't overlap with pfSense's CLIENTS range (10.0.3.100–200). This won't conflict as long as the Windows DHCP scope is for a range not served by pfSense.

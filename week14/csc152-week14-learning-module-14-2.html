§1 - What Is Server Hardening?

Definition

Server hardening is the process of reducing a server’s attack surface—the total number of points where an unauthorized user could try to enter or extract data—by removing unnecessary software, disabling unneeded services, tightening configurations, and applying security best practices.

The Principle of Least Functionality

Every running service, open port, and installed package is a potential entry point for an attacker. The principle of least functionality states that a system should be configured to provide only the capabilities required for its designated role—nothing more. A web server does not need a print spooler; a database server does not need a GUI desktop environment.

Default Installation vs. Hardened Installation

Operating systems ship with default settings designed for broad compatibility, not security. The table below highlights the contrast:

⚠️ Default Installation

  • Many services enabled “just in case”
  • Root/Administrator remote login allowed
  • Firewall permissive or disabled
  • Audit logging minimal or off
  • Legacy protocols enabled (e.g., SMBv1)
  • Default banners reveal OS version
  • Unnecessary packages installed

✅ Hardened Installation

  • Only role-required services running
  • Root/Administrator remote login disabled
  • Firewall default-deny; explicit allow rules
  • Comprehensive audit logging enabled
  • Legacy protocols removed or disabled
  • Legal warning banner displayed
  • Minimal package footprint
💡 Key Takeaway
Hardening is not a one-time event. It is a process that begins at installation, follows a checklist, and is verified and maintained throughout the server’s lifecycle.

§2 - Hardening Frameworks & Benchmarks

Rather than inventing your own checklist, security professionals rely on established frameworks. Three of the most widely used are described below.

CIS Benchmarks (Center for Internet Security)

CIS publishes free, consensus-based configuration guides for over 100 technologies. Each benchmark document is organized into numbered recommendations with a Profile Level:

  • Level 1 - Practical settings that can be applied broadly with minimal impact on functionality.
  • Level 2 - Defense-in-depth settings that may reduce functionality; intended for high-security environments.

Each recommendation includes a description, rationale, audit procedure (how to check), and remediation (how to fix). Learning to read a CIS Benchmark document is a transferable skill you will use throughout your career.

DISA STIGs (Security Technical Implementation Guides)

Published by the U.S. Defense Information Systems Agency, STIGs are mandatory for Department of Defense systems and widely adopted in government. Each finding is rated by severity: CAT I (high), CAT II (medium), or CAT III (low).

NIST SP 800-123 - Guide to General Server Security

This NIST Special Publication provides a high-level framework for securing servers throughout their lifecycle: planning, installation, configuration, maintenance, and disposal. It is less prescriptive than CIS or DISA but excellent for understanding why hardening steps matter.

How to Select and Apply a Checklist

  1. Identify the OS and version of the server you are hardening.
  2. Download the matching benchmark (e.g., “CIS Ubuntu Linux 22.04 LTS Benchmark”).
  3. Choose a profile level appropriate for your environment (Level 1 for most lab work).
  4. Work through each recommendation: audit the current state, apply the remediation if needed, and document the change.
  5. Verify using automated tools (covered in §5).

§3 - Linux Server Hardening

This section walks through key hardening steps for an Ubuntu Server (commands also apply to Rocky Linux with minor package-manager differences noted). Use the accordion items below to explore each hardening category in detail.

List all active services, then disable those not required by the server’s role:

# List all enabled services
systemctl list-unit-files --type=service --state=enabled

# Example: disable the cups (printing) service
sudo systemctl stop cups
sudo systemctl disable cups

# Mask a service to prevent it from being started even manually
sudo systemctl mask cups

disable prevents a service from starting at boot. mask goes further by linking the unit file to /dev/null, preventing any start attempt. Use mask for services you are certain are never needed.

Common candidates for disabling on a server: cups (printing), avahi-daemon (mDNS), bluetooth, ModemManager.

Every installed package increases the attack surface. Remove packages that are not needed:

# Ubuntu: list installed packages
dpkg --list | less

# Remove an unnecessary package (example: telnet client)
sudo apt purge telnet
sudo apt autoremove
# Rocky Linux equivalent
rpm -qa | less
sudo dnf remove telnet
sudo dnf autoremove

purge removes both the package and its configuration files. autoremove cleans up orphaned dependencies.

SSH is the primary remote-access method for Linux servers. Hardening it is critical:

# Back up the original configuration first
sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.bak

# Edit the SSH daemon configuration
sudo nano /etc/ssh/sshd_config

Apply these key settings:

# Disable root login over SSH
PermitRootLogin no

# Restrict SSH to specific users or groups
AllowUsers sysadmin deployer
# OR restrict by group:
# AllowGroups ssh-users

# Disable password authentication (after deploying SSH keys - see Module 14.3)
# PasswordAuthentication no

# Use SSH Protocol 2 only (default on modern systems)
Protocol 2

# Set idle timeout (disconnect after 5 minutes of inactivity)
ClientAliveInterval 300
ClientAliveCountMax 0

# Limit maximum authentication attempts
MaxAuthTries 3

# Disable X11 forwarding (not needed on servers)
X11Forwarding no
# Validate the configuration before restarting
sudo sshd -t

# Restart the SSH service to apply changes
sudo systemctl restart sshd
⚠️ Warning
Do not disable PasswordAuthentication until you have confirmed SSH key-based login works. Otherwise you may lock yourself out. Module 14.3 covers SSH key setup.

This extends the firewall fundamentals from Week 9. The goal is a default-deny posture with explicit allow rules:

# UFW (Uncomplicated Firewall) — Ubuntu
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Allow SSH (required for remote management)
sudo ufw allow 22/tcp

# Allow HTTPS if this is a web server
sudo ufw allow 443/tcp

# Enable the firewall
sudo ufw enable

# Verify rules
sudo ufw status verbose
# iptables equivalent (works on any distro)
sudo iptables -P INPUT DROP
sudo iptables -P FORWARD DROP
sudo iptables -P OUTPUT ACCEPT

# Allow established connections
sudo iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Allow loopback
sudo iptables -A INPUT -i lo -j ACCEPT

# Allow SSH
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Save rules (Ubuntu)
sudo apt install iptables-persistent
sudo netfilter-persistent save

Principle: Only open the ports your server’s role requires. Document every allowed port and its justification.

Restrict what can happen on certain mount points by adding security options in /etc/fstab:

# Example /etc/fstab entry for /tmp with hardening options
tmpfs  /tmp  tmpfs  defaults,noexec,nosuid,nodev  0  0
OptionEffect
noexecPrevents execution of binaries on this partition
nosuidIgnores SetUID/SetGID bits (prevents privilege escalation)
nodevPrevents interpretation of device files on this partition

Common targets: /tmp, /var/tmp, /dev/shm. After editing /etc/fstab, remount:

sudo mount -o remount /tmp

The kernel exposes tunable parameters through /proc/sys/. Persist changes in /etc/sysctl.d/:

# Create a hardening configuration file
sudo nano /etc/sysctl.d/99-hardening.conf
# Disable IP forwarding (unless this is a router)
net.ipv4.ip_forward = 0

# Enable SYN cookies (protects against SYN flood attacks)
net.ipv4.tcp_syncookies = 1

# Ignore ICMP broadcast requests
net.ipv4.icmp_echo_ignore_broadcasts = 1

# Disable source routing
net.ipv4.conf.all.accept_source_route = 0

# Enable reverse-path filtering (anti-spoofing)
net.ipv4.conf.all.rp_filter = 1

# Log suspicious packets (martians)
net.ipv4.conf.all.log_martians = 1
# Apply changes immediately
sudo sysctl --system

The Linux Audit Daemon records security-relevant events. Install and enable it:

# Install auditd (Ubuntu)
sudo apt install auditd audispd-plugins

# Enable and start
sudo systemctl enable auditd
sudo systemctl start auditd

Add rules to monitor critical files and events:

# Watch for changes to user/group files
sudo auditctl -w /etc/passwd -p wa -k user-modify
sudo auditctl -w /etc/shadow -p wa -k shadow-modify
sudo auditctl -w /etc/group  -p wa -k group-modify

# Watch for changes to SSH configuration
sudo auditctl -w /etc/ssh/sshd_config -p wa -k sshd-config

# Make rules persistent: add them to /etc/audit/rules.d/hardening.rules
# Search audit logs
sudo ausearch -k user-modify

Restrict cron and at to authorized users:

# Only allow specific users to use cron
sudo bash -c 'echo "sysadmin" > /etc/cron.allow'

# Only allow specific users to use at
sudo bash -c 'echo "sysadmin" > /etc/at.allow'

If /etc/cron.allow exists, only users listed in it may use crontab. All others are denied.

Legal warning banners display a notice before login, establishing legal standing for monitoring:

# Set the pre-login banner
sudo nano /etc/issue.net
*********************************************************************
WARNING: This system is for authorized use only. All activity may be
monitored and recorded. Unauthorized access is prohibited and will
be prosecuted to the fullest extent of the law.
*********************************************************************
# Enable the banner in SSH
# In /etc/ssh/sshd_config, set:
Banner /etc/issue.net

# Restart SSH
sudo systemctl restart sshd

§4 - Windows Server Hardening

This section covers key hardening steps for Windows Server. Where possible, both GUI and PowerShell methods are shown. Use the accordion items to explore each category.

GUI: Open Server Manager → Manage → Remove Roles and Features. Uncheck any role or feature not required by the server’s function.

PowerShell - list and disable services:

# List all running services
Get-Service | Where-Object {$_.Status -eq 'Running'} | Sort-Object DisplayName

# Example: disable the Print Spooler on a non-print server
Stop-Service -Name Spooler -Force
Set-Service -Name Spooler -StartupType Disabled

# Remove an unnecessary Windows feature
Uninstall-WindowsFeature -Name Windows-Defender-GUI   # example

Common services to evaluate: Print Spooler, Xbox services (if present), Fax, Bluetooth Support.

GUI: Open Windows Defender Firewall with Advanced Security (wf.msc). Set all profiles (Domain, Private, Public) to Block inbound by default.

PowerShell:

# Set default inbound to block on all profiles
Set-NetFirewallProfile -Profile Domain,Public,Private -DefaultInboundAction Block

# Allow RDP only from a management subnet (example)
New-NetFirewallRule -DisplayName "Allow RDP - Mgmt Subnet" `
    -Direction Inbound -Protocol TCP -LocalPort 3389 `
    -RemoteAddress 192.168.10.0/24 -Action Allow

# View current firewall rules
Get-NetFirewallRule | Where-Object {$_.Enabled -eq 'True'} |
    Select-Object DisplayName, Direction, Action |
    Sort-Object DisplayName

This extends the Group Policy content from Week 11. Open secpol.msc (Local Security Policy) or use Group Policy for domain-joined servers.

User Rights Assignment

Restrict powerful rights (e.g., “Log on locally,” “Allow log on through Remote Desktop Services”) to only the accounts and groups that need them.

PowerShell - export current security policy for documentation:

# Export local security policy to a file (useful for before/after comparison)
secedit /export /cfg C:\Hardening\before-policy.inf

SMBv1 is vulnerable to exploits such as EternalBlue (used in the WannaCry ransomware attack). It should be disabled on all modern servers:

# Check if SMBv1 is enabled
Get-WindowsOptionalFeature -Online -FeatureName SMB1Protocol

# Disable SMBv1
Disable-WindowsOptionalFeature -Online -FeatureName SMB1Protocol -NoRestart

# Also disable via server configuration
Set-SmbServerConfiguration -EnableSMB1Protocol $false -Force

Other legacy protocols to evaluate: LLMNR, NetBIOS over TCP/IP, NTLMv1.

Ensure the Security event log is large enough to capture meaningful history and that key events are being recorded:

# Increase Security log maximum size to 1 GB
wevtutil sl Security /ms:1073741824

# Verify the setting
wevtutil gl Security

Limit who can connect via RDP and from where:

# Restrict RDP to members of a specific group
# In Local Security Policy → User Rights Assignment:
#   "Allow log on through Remote Desktop Services" → only "Remote Desktop Users" group

# Enable Network Level Authentication (NLA)
Set-ItemProperty -Path 'HKLM:\SYSTEM\CurrentControlSet\Control\Terminal Server\WinStations\RDP-Tcp' `
    -Name UserAuthentication -Value 1

# Use Windows Firewall to limit RDP source IPs (see §4.2)

Windows Defender:

# Verify Windows Defender is enabled and up to date
Get-MpComputerStatus | Select-Object AntivirusEnabled, RealTimeProtectionEnabled,
    AntivirusSignatureLastUpdated

# Run a quick scan
Start-MpScan -ScanType QuickScan

# Update definitions
Update-MpSignature

Remove unnecessary features:

# List installed features
Get-WindowsFeature | Where-Object {$_.Installed -eq $true}

# Remove an unnecessary feature (example: PowerShell v2 — legacy, potential bypass)
Uninstall-WindowsFeature -Name PowerShell-V2

§5 - Hardening Verification

After applying hardening steps, you must verify that they are in effect. Manual checklists work, but automated tools save time and catch items you may have missed.

Lynis - Security Auditing Tool for Linux

Lynis is an open-source security auditing tool that scans a Linux system and produces a hardening index along with specific recommendations.

# Install Lynis (Ubuntu)
sudo apt install lynis

# Rocky Linux
sudo dnf install lynis

# Run a full system audit
sudo lynis audit system

Lynis produces output organized into sections (Boot & Services, Kernel, Networking, SSH, File Systems, etc.). At the end, you will see:

  Hardening index : 67 [#############       ]
  Tests performed : 256
  Suggestions     : 42
  Warnings        : 3

Interpreting the output:

  • Hardening index: A score out of 100. Higher is better. A freshly installed server typically scores 55-65; a well-hardened server scores 80+.
  • Suggestions: Specific recommendations with CIS/NIST references. Each includes a test ID you can look up.
  • Warnings: High-priority items that should be addressed immediately.
# View the detailed log
sudo cat /var/log/lynis.log

# View only suggestions
sudo grep "suggestion" /var/log/lynis-report.dat
📋 Project 3 Tip
Run Lynis before hardening (capture the score), apply your hardening steps, then run it again. The before/after comparison is excellent evidence for your Project 3 documentation.

Microsoft Security Compliance Toolkit (Awareness Level)

The Microsoft Security Compliance Toolkit (SCT) includes tools and GPO baselines for hardening Windows Server. Key components:

  • Policy Analyzer - compares your current GPO settings against Microsoft’s recommended baselines.
  • LGPO - command-line tool for applying Local Group Policy settings from a baseline.
  • Security Baselines - pre-built GPO packages for each Windows Server version.

For this course, awareness of SCT is sufficient. For Project 3, you can document Windows hardening using the secedit /export method shown in §4.3 and manual verification of each setting.

§6 - Mini-Case: Hardening a Lab Server

Read the scenario below, then review the identified hardening steps and the before/after comparison.

📄 Scenario

You have been given a freshly installed Ubuntu Server 22.04 VM that will serve as a web server (Apache/Nginx) in your Project 3 lab environment. A quick audit reveals the following default state:

  1. Root SSH login is permitted.
  2. The cups (printing) and avahi-daemon (mDNS) services are running.
  3. UFW firewall is inactive.
  4. /tmp has no noexec mount option.
  5. IP forwarding is enabled in the kernel (net.ipv4.ip_forward = 1).
  6. No audit logging is configured (auditd not installed).
  7. No legal warning banner is displayed before login.

Your task: identify and apply hardening steps to address each finding.

Hardening Steps Identified

#FindingHardening ActionReference
1Root SSH login permittedSet PermitRootLogin no in sshd_configCIS 5.2.10

3Firewall inactiveEnable UFW; default deny inbound; allow 22, 80, 443CIS 3.5.1
4/tmp missing noexecAdd noexec,nosuid,nodev to /tmp in /etc/fstabCIS 1.1.x
5IP forwarding enabledSet net.ipv4.ip_forward = 0 in sysctlCIS 3.2.1
6No audit loggingInstall & enable auditd; add rules for critical filesCIS 4.1.x
7No legal bannerCreate /etc/issue.net; set Banner in sshd_configCIS 1.7.x

💡 Reflection
The Lynis hardening index improved from 58 to 79 with seven targeted changes. Notice that the number of suggestions dropped from 51 to 28 and warnings from 5 to 1. This kind of before/after evidence is exactly what you should capture for Project 3.

§1 - Authentication Fundamentals

The AAA Model

Network and system security is organized around three interrelated functions known as AAA:

FunctionQuestion It AnswersExample
AuthenticationWho are you?Entering a username and password; presenting an SSH key
AuthorizationWhat are you allowed to do?File permissions (Week 7); sudo rules (Week 6); AD group membership (Week 11)
AccountingWhat did you do?Audit logs (auditd, Windows Security Event Log); Splunk dashboards (Module 14.4)

Authentication is the first gate. If authentication is weak, authorization and accounting cannot protect the system because an attacker is already operating under a legitimate identity.

Authentication Factors

Authentication credentials fall into three categories, often called factors:

FactorDescriptionExamples
Something you knowA secret stored in your memoryPassword, PIN, security question
Something you haveA physical or digital object you possessSSH private key, hardware token (YubiKey), authenticator app (TOTP code), smart card
Something you areA biometric characteristicFingerprint, retina scan, facial recognition

Multi-Factor Authentication (MFA)

MFA requires credentials from two or more different factor categories. Using two passwords (both “something you know”) is not MFA—the factors must be from different categories.

Why Passwords Alone Are Insufficient
  • Brute-force & dictionary attacks - automated tools can test millions of combinations.
  • Credential stuffing - reused passwords leaked from one breach are tried on other services.
  • Phishing - users can be tricked into revealing passwords.
  • Keyloggers - malware can capture keystrokes.

SSH key-based authentication (§2) replaces passwords with a cryptographic “something you have” factor, dramatically raising the bar for attackers.

🔗 Connection to Prior Weeks

In Week 6 you configured password policies (minimum length, complexity, expiration). Those policies strengthen the “something you know” factor but cannot eliminate its inherent weaknesses. The techniques in this module go further by adding or replacing that factor entirely.

§2 - SSH Key-Based Authentication

How Public/Private Key Pairs Work

SSH key-based authentication uses asymmetric cryptography. A mathematically linked pair of keys is generated:

KeyLocationPurposeSecrecy
Private keyClient machine (your workstation)Proves your identity by signing a challengeMust never be shared
Public keyServer (~/.ssh/authorized_keys)Verifies the signature created by the private keySafe to share freely
Simplified SSH Key Authentication Handshake
Client sends
connection request → Server sends
random challenge → Client signs challenge
with private key → Server verifies
with public key → Access
Granted ✓

The private key never leaves your machine and is never transmitted over the network. Even if an attacker captures the entire network conversation, they cannot reconstruct the private key.

Step-by-Step: Generating and Deploying SSH Keys

Step 1 - Generate a Key Pair

Run this on the client (your workstation):

# Generate an Ed25519 key pair (recommended, modern, fast)
ssh-keygen -t ed25519 -C "yourname@yourhost"

# You will be prompted:
#   Enter file in which to save the key (/home/you/.ssh/id_ed25519): [Enter]
#   Enter passphrase (empty for no passphrase): [type a strong passphrase]
#   Enter same passphrase again: [confirm]

# Result: two files created
#   ~/.ssh/id_ed25519       (private key - KEEP SECRET)
#   ~/.ssh/id_ed25519.pub   (public key  - safe to share)
🔒 Always Use a Passphrase

A passphrase encrypts the private key file on disk. If your workstation is compromised, the attacker still cannot use the key without the passphrase. This effectively creates two-factor authentication: something you have (the key file) + something you know (the passphrase).

Step 2 - Deploy the Public Key to the Server

The easiest method is ssh-copy-id:

# Copy your public key to the remote server
ssh-copy-id -i ~/.ssh/id_ed25519.pub username@server-ip

# This appends your public key to the server's
#   ~/.ssh/authorized_keys  file
# You will be prompted for your PASSWORD one last time

If ssh-copy-id is not available, you can do it manually:

# Manual method: copy the public key content
cat ~/.ssh/id_ed25519.pub

# On the server, append it to authorized_keys
mkdir -p ~/.ssh
chmod 700 ~/.ssh
echo "paste-your-public-key-here" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

Step 3 - Test Key-Based Login

# Connect using your key (no password prompt should appear)
ssh -i ~/.ssh/id_ed25519 username@server-ip

# If you set a passphrase, you will be prompted for it
# (this is the KEY passphrase, not the server password)

Step 4 - Disable Password Authentication

Once key-based login is confirmed working, disable password authentication on the server to eliminate password-based attacks entirely. Edit /etc/ssh/sshd_config:

# Edit the SSH daemon configuration
sudo nano /etc/ssh/sshd_config

# Find and set these directives:
PasswordAuthentication no
ChallengeResponseAuthentication no
UsePAM no

# Save the file, then restart the SSH service
sudo systemctl restart sshd
⚠️ Critical Warning

Always verify key-based login works before disabling password authentication. If you disable password auth and your key is not properly deployed, you will be locked out of the server. Keep an existing SSH session open while testing.

Step 5 - SSH Agent (Optional but Recommended)

The SSH agent caches your decrypted private key in memory so you do not have to re-enter the passphrase for every connection during a session:

# Start the SSH agent
eval "$(ssh-agent -s)"

# Add your key to the agent
ssh-add ~/.ssh/id_ed25519
# Enter your passphrase once; subsequent connections
# in this session will not prompt again
🔗 Connection to Module 14.2

In Module 14.2 you hardened SSH by disabling root login and restricting access to specific users via /etc/ssh/sshd_config. Disabling password authentication here is the natural next step—together these settings ensure that only authorized users with valid key pairs can access the server.

§3 - Centralized Authentication Review

In a production environment with many servers, managing local accounts on each machine is impractical and error-prone. Centralized authentication solves this by providing a single point of identity management and policy enforcement. You studied two major systems earlier in the course:

FeatureActive Directory / Kerberos (Week 11)OpenLDAP (Week 12)
ProtocolKerberos (ticket-based)LDAP (directory query)
Platform focusWindows-centric (cross-platform possible)Linux/cross-platform
Authentication flowUser → KDC → TGT → service ticketUser → LDAP bind (simple or SASL)
Policy enforcementGroup Policy Objects (GPOs)LDAP attributes + PAM/sudo rules
Authorization modelAD groups → permissions/GPOsGroups → sudo rules / PAM

How Centralized Auth Supports Hardening

  • Single point of policy enforcement - Password complexity, account lockout, and expiration policies are set once and applied to all systems.
  • Rapid account revocation - Disabling one account in AD or LDAP immediately blocks access across every connected server.
  • Role-Based Access Control (RBAC) - AD groups or LDAP groups map to specific permissions and sudo rules, enforcing least privilege consistently.
  • Audit trail - Centralized authentication events are logged in one place, making them easier to monitor with Splunk (Module 14.4).
💡 Key Takeaway

Local accounts should be reserved for emergency “break-glass” access. Day-to-day authentication should flow through a centralized directory service so that policies are uniform and auditable.

§4 - Encryption: Data in Transit

Data in transit is any data moving across a network—between a client and server, between two servers, or across the internet. Without encryption, this data can be intercepted and read by anyone with access to the network path.

Quick Review: Symmetric vs. Asymmetric Encryption

TypeKeysSpeedUse Case
SymmetricOne shared secret keyFastBulk data encryption (AES)
AsymmetricPublic + private key pairSlowKey exchange, digital signatures (RSA, Ed25519)

In practice, most protocols use both: asymmetric encryption to securely exchange a symmetric session key, then symmetric encryption for the actual data transfer. This is exactly how TLS works.

SSL/TLS Fundamentals

TLS (Transport Layer Security) is the successor to SSL (Secure Sockets Layer). While “SSL” is still used colloquially, modern systems use TLS 1.2 or TLS 1.3. SSL versions are deprecated and insecure.

IPSec Overview (Awareness Level)

IPSec (Internet Protocol Security) operates at the network layer (Layer 3), encrypting IP packets themselves rather than individual application connections. It is the foundation of most VPN (Virtual Private Network) tunnels. Unlike TLS, which protects a single application session, IPSec can protect all traffic between two endpoints.

  • Transport mode - encrypts only the payload; used for host-to-host communication.
  • Tunnel mode - encrypts the entire original IP packet; used for site-to-site VPNs.

In Week 9 you configured firewalls that control which traffic flows; encryption controls whether that traffic can be read. In Week 12, you could have configured LDAPS to encrypt directory queries. These are all applications of the same principle: encrypt data in transit.

§5 - Encryption: Data at Rest

Data at rest is any data stored on disk—files, databases, logs, backups. If an attacker gains physical access to a drive (or steals a backup), encryption at rest ensures the data remains unreadable without the decryption key.

Linux: LUKS / dm-crypt

LUKS (Linux Unified Key Setup) is the standard for Linux full-disk and partition encryption. It uses the dm-crypt kernel subsystem.

Key Concepts

  • Encrypts an entire block device (partition or disk).
  • Supports multiple key slots (up to 8 passphrases or key files per device).
  • Transparent to applications—once unlocked, the filesystem operates normally.

Basic LUKS Setup Commands (Reference)

# 1. Initialize LUKS on a partition (DESTROYS existing data)
sudo cryptsetup luksFormat /dev/sdb1

# 2. Open (unlock) the encrypted partition
sudo cryptsetup luksOpen /dev/sdb1 secure_data
# Creates /dev/mapper/secure_data

# 3. Create a filesystem on the unlocked device
sudo mkfs.ext4 /dev/mapper/secure_data

# 4. Mount and use
sudo mkdir /mnt/secure
sudo mount /dev/mapper/secure_data /mnt/secure

# 5. When finished, unmount and close
sudo umount /mnt/secure
sudo cryptsetup luksClose secure_data
💡 Connection to Week 10 (Storage)

In Week 10 you learned about partitioning, LVM, and RAID. LUKS can be layered on top of LVM logical volumes or RAID arrays, adding encryption to any storage architecture you already know how to build.

Encrypted Backups

Backups are a copy of your data—if the original data is sensitive, the backup is equally sensitive. Encryption at rest should extend to backup media:

  • Use encrypted backup tools (e.g., gpg-encrypted tar archives, BitLocker-encrypted backup drives).
  • Encrypt backup data before transferring it off-site.
  • This topic connects directly to Week 15 (Disaster Recovery & Backup Solutions).

Secure Storage Disposal (Brief Review)

From Module 14.1, recall that decommissioned drives must be securely erased to prevent data recovery. NIST SP 800-88 defines three levels:

LevelMethodWhen to Use
ClearOverwrite with zeros/patternsReuse within organization
PurgeCryptographic erase, degaussingReuse outside organization
DestroyShredding, incineration, disintegrationHighest-sensitivity data

If a drive was LUKS- or BitLocker-encrypted, securely destroying the encryption key effectively renders the data unrecoverable (cryptographic erase).

§6 - Patch Management Strategy

Why Patching Matters

Every piece of software has vulnerabilities. When a vulnerability is discovered, it is assigned a CVE (Common Vulnerabilities and Exposures) identifier—a standardized, public reference number (e.g., CVE-2024-3094).

Vulnerability Lifecycle
Vulnerability
Discovered → CVE
Assigned → Vendor
Releases Patch → Admin
Tests Patch → Patch
Deployed ✓
⚠️ Real-World Impact

The WannaCry ransomware attack (2017) exploited a Windows SMB vulnerability (CVE-2017-0144) for which Microsoft had released a patch two months earlier. Organizations that had not applied the patch were devastated. The Equifax breach (2017) exploited an Apache Struts vulnerability (CVE-2017-5638) that had been patched months before the breach. Patching is not optional—it is a core operational discipline.

Linux Patch Management

Debian/Ubuntu (APT)

# Update the package index (list of available updates)
sudo apt update

# Upgrade all installed packages
sudo apt upgrade -y

# Apply only security updates
sudo apt-get install --only-upgrade $(apt list --upgradable 2>/dev/null | grep -i security | cut -d/ -f1)

# Full distribution upgrade (includes kernel, may remove obsolete packages)
sudo apt full-upgrade -y

# Configure unattended security upgrades
sudo apt install unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades

RHEL/Rocky/CentOS (DNF)

# Check for available updates
sudo dnf check-update

# Apply all updates
sudo dnf update -y

# Apply security-only updates
sudo dnf update --security -y

# Configure automatic updates
sudo dnf install dnf-automatic
sudo systemctl enable --now dnf-automatic-install.timer

Kernel Updates and Reboots

Kernel updates require a reboot to take effect. Plan kernel updates during maintenance windows. Check if a reboot is needed:

# Debian/Ubuntu: check if reboot is required
cat /var/run/reboot-required 2>/dev/null && echo "Reboot needed" || echo "No reboot needed"

# RHEL/Rocky: check running vs. installed kernel
uname -r                          # running kernel
rpm -q kernel | tail -1           # latest installed kernel

Windows Patch Management

Windows Update (Standalone)

Individual servers can be updated through Settings → Update & Security → Windows Update or via PowerShell:

# PowerShell: check for and install updates
Install-Module PSWindowsUpdate -Force
Get-WindowsUpdate
Install-WindowsUpdate -AcceptAll -AutoReboot

WSUS (Windows Server Update Services) - Overview

In enterprise environments, WSUS provides centralized update management:

  • Downloads updates from Microsoft once; distributes to internal servers.
  • Administrators approve updates before deployment.
  • Supports computer groups for staged rollouts (test → staging → production).
  • Integrates with Group Policy (Week 11) to assign servers to WSUS and control update behavior.

Rollback Procedures

Not every patch goes smoothly. Always have a rollback plan:

  • VM snapshots (Week 13) - Take a snapshot before patching; revert if problems occur.
  • Package manager rollback - sudo apt install package=previous-version or sudo dnf history undo last.
  • Windows - Uninstall updates via Settings → Update History → Uninstall Updates or wusa /uninstall /kb:KBNUMBER.
  • Backups - Full system backup before major patch cycles (connects to Week 15).

Maintenance Windows and Change Management

  • Schedule patching during low-usage periods (e.g., weekends, overnight).
  • Document every change: what was patched, when, by whom, and the outcome.
  • Change management ensures accountability and provides an audit trail.

Vulnerability Scanning (Awareness Level)

Vulnerability scanners automatically check systems for known vulnerabilities and missing patches:

  • Nessus - Commercial scanner widely used in enterprise environments.
  • OpenVAS - Open-source alternative with a large vulnerability database.

Scan results produce a prioritized list of vulnerabilities, helping administrators decide which patches to apply first. You are not expected to install or operate these tools in this course, but you should understand their role in a patch management strategy.

§7 - Check for Understanding

These questions are ungraded—use them to test your understanding before moving to Module 14.4. Click Check Answer for immediate feedback.

1. Which is the correct order of steps for deploying SSH key-based authentication?

✓ Correct! You must generate the key pair first, deploy the public key, verify it works, and only then disable password authentication. Disabling password auth before confirming key access would lock you out.
✗ Not quite. The correct order is: Generate key pair → Copy public key to server → Test key-based login → Disable password auth. You must always verify key access works before disabling password authentication.

2. A company uses LUKS to encrypt its database server’s data partition. What type of encryption is this?

✓ Correct! LUKS encrypts data stored on disk (at rest). The data is decrypted only when the partition is unlocked and mounted. If the physical drive is stolen, the data remains unreadable.
✗ Not quite. LUKS is encryption at rest—it protects data stored on disk. Encryption in transit (like TLS) protects data moving across a network. Hashing is a one-way function, not encryption.

3. An administrator configures Apache to use a Let’s Encrypt certificate so that all web traffic uses HTTPS. What type of protection does this provide?

✓ Correct! HTTPS uses TLS to encrypt data traveling between the client’s browser and the web server—this is encryption in transit. The Let’s Encrypt certificate enables the server to prove its identity and establish the TLS session.
✗ Not quite. HTTPS protects data in transit between the browser and server using TLS. The certificate from Let’s Encrypt enables the TLS handshake.

4. A user logs in with a password and then enters a six-digit code from a phone authenticator app. Which statement is true?

✓ Correct! The password is “something you know” and the authenticator app code is “something you have” (you must possess the phone with the enrolled app). Two different factor categories = MFA.
✗ Not quite. The password is “something you know” and the TOTP code from the authenticator app is “something you have” (the phone). These are two different factor categories, making this true MFA.

5. A critical CVE is published affecting your web server software. What is the correct patch management sequence?

✓ Correct! Even for critical patches, the disciplined approach is: test first, then staged rollout, then full production, then verify and document. For critical CVEs the timeline is compressed (hours/days, not weeks), but the steps remain the same.
✗ Not quite. The correct sequence is: Test → Staged rollout → Full production → Verify & document. Applying untested patches to all production servers risks outages. Waiting months for a critical CVE leaves you exposed. Disabling the service may not be feasible.

6. In the AAA model, which function is responsible for tracking what actions a user performs after they have been granted access?

✓ Correct! Accounting (the third “A”) is responsible for logging and tracking user actions—what they accessed, when, and what changes they made. This is what tools like auditd and Splunk support.
✗ Not quite. Accounting tracks user actions after access is granted. Authentication verifies identity; Authorization determines permissions. Accounting provides the audit trail.

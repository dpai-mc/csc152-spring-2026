§1 - The Hardening-Monitoring Feedback Loop

Why Hardening Without Monitoring Is Incomplete

Imagine locking every door and window in a building but never installing an alarm system or checking whether the locks still work. That is what server hardening looks like without monitoring. You may have:

  • Disabled root SSH login—but how do you know an attacker is not still trying?
  • Configured firewall rules to block unauthorized ports—but are those rules actually dropping traffic?
  • Enabled account lockout policies—but has any account actually been locked out?

Monitoring answers these questions. Log data generated by hardened systems provides evidence that controls are functioning. When you collect and analyze those logs in a centralized platform, you gain visibility across your entire infrastructure.

Splunk as a SIEM Tool

A SIEM (Security Information and Event Management) system collects, normalizes, and correlates log data from multiple sources to provide a unified security view. In your lab environment, Splunk Enterprise serves as your SIEM:

  • Week 12: You installed Splunk Enterprise on a Linux server and enabled the receiving port (9997).
  • Week 13: You deployed Universal Forwarders on Linux and Windows servers, configured inputs.conf and outputs.conf, and verified data ingestion.
  • This week: You use Splunk to validate hardening and detect suspicious activity.

The Feedback Loop

The hardening-monitoring feedback loop is a continuous cycle. Each step informs the next:

The Hardening-Monitoring Feedback Loop

① Apply Hardening
(e.g., disable password-based SSH, enable firewall rule)
↓
② Log Event Generated
(auth.log records rejected password attempts; iptables logs blocked packets)
↓
③ Splunk Search & Dashboard
(SPL query surfaces the events; dashboard visualizes trends)
↓
④ Evidence & Insight
(0 failed SSH password logins confirms key-only auth works; spike in firewall blocks reveals scanning activity)
↓
⑤ Adjust & Improve
(Add new firewall rule for scanning source; tighten alert thresholds)
↓
🔄 Return to Step ① — the cycle continues
✅ Key Takeaway
Hardening and monitoring are two halves of the same security strategy. Hardening reduces the attack surface; monitoring proves it and catches what slips through. Your Project 3 submission must demonstrate both.

§2 - SPL Searches for Security Events

This section provides guided SPL (Search Processing Language) examples targeting the most important security events. Each example includes the search, an explanation of what it detects, and why it matters for security validation. You will adapt these searches to your own lab environment for Project 3.

⚠️ Adapt to Your Environment
The examples below use common default index names, source paths, and host values. Replace index=main, host=linux-srv, host=win-srv, and source paths with the actual values from your lab. Use index=* host=<your-hostname> to discover your data first.

2.1 - Failed Login Attempts (Linux)

What it detects: Failed SSH or console login attempts recorded in /var/log/auth.log (Ubuntu) or /var/log/secure (Rocky/CentOS).

Why it matters: After hardening SSH (disabling password auth, restricting users), you should see these events drop to zero for password-based failures. Any remaining failures may indicate brute-force attempts or misconfiguration.

index=main source="/var/log/auth.log" "Failed password"
| stats count by host, src_ip, user
| sort -count

Explanation: Searches for the literal string "Failed password" in auth.log, then counts occurrences grouped by host, source IP, and target user. Results are sorted in descending order so the most-targeted accounts appear first.

2.2 - Failed Logon Attempts (Windows)

What it detects: Failed logon events in the Windows Security Event Log (Event ID 4625).

Why it matters: After configuring account lockout policies and restricting Remote Desktop access, monitoring Event ID 4625 confirms those controls are blocking unauthorized attempts.

index=main source="WinEventLog:Security" EventCode=4625
| stats count by host, Account_Name, Source_Network_Address
| sort -count

Explanation: Filters Windows Security events to Event Code 4625 (failed logon), then groups by host, the account targeted, and the source network address. This reveals which accounts are being targeted and from where.

2.3 - Firewall Blocks (Linux iptables / UFW)

What it detects: Packets blocked by iptables or UFW firewall rules.

Why it matters: Validates that firewall hardening rules are actively dropping unauthorized traffic. A high volume of blocks from a single IP may indicate scanning or attack activity.

index=main source="/var/log/syslog" "UFW BLOCK"
| rex field=_raw "SRC=(?<src_ip>\d+\.\d+\.\d+\.\d+).*DPT=(?<dest_port>\d+)"
| stats count by src_ip, dest_port
| sort -count

Explanation: Searches syslog for UFW block entries, uses regex to extract the source IP and destination port, then counts blocks by source and port. Replace "UFW BLOCK" with "iptables DENY" or your custom log prefix if you are using raw iptables rules.

2.4 - Firewall Blocks (Windows Firewall)

What it detects: Connections dropped by Windows Firewall with Advanced Security.

Why it matters: Confirms that Windows Firewall rules configured during hardening (Module 14.2) are actively blocking unauthorized inbound connections.

index=main source="WinEventLog:Security" EventCode=5157 Action="Block"
| stats count by host, src_ip, dest_port
| sort -count

Explanation: Event ID 5157 logs Windows Filtering Platform connection blocks. If your Windows Firewall logging writes to a text-based log file instead, adjust the source to that file path and search for the DROP action keyword.

2.5 - Sudo / Privilege Escalation Events (Linux)

What it detects: Successful and failed sudo commands, indicating privilege escalation activity.

Why it matters: After restricting sudo access (Module 14.2), monitoring these events confirms that only authorized users are escalating privileges. Failed sudo attempts may indicate an attacker testing compromised credentials.

index=main source="/var/log/auth.log" "sudo"
| rex field=_raw "(?<sudo_user>\w+)\s+:\s+.*COMMAND=(?<command>.+)"
| stats count by sudo_user, command
| sort -count

Explanation: Searches auth.log for sudo entries, extracts the user who ran the command and the command itself, then counts by user and command. Look for unexpected users or commands.

2.6 - Account Lockouts (Windows)

What it detects: Account lockout events in Active Directory or local security policy (Event ID 4740).

Why it matters: Account lockout policies are a key hardening control (Module 14.2). Monitoring lockout events shows the policy is working and helps identify accounts under attack.

index=main source="WinEventLog:Security" EventCode=4740
| table _time, Account_Name, ComputerName, Source_Network_Address
| sort -_time

Explanation: Filters for Event ID 4740 (account lockout) and displays the time, locked account, the computer where the lockout was recorded, and the source address that triggered it.

2.7 - Service State Changes

What it detects: Services starting or stopping unexpectedly on Linux or Windows.

Why it matters: After disabling unnecessary services during hardening, a service restarting unexpectedly could indicate tampering, misconfiguration, or a persistence mechanism.

Linux (systemd via syslog)

index=main source="/var/log/syslog" ("Started" OR "Stopped" OR "Failed")
| rex field=_raw "(?<action>Started|Stopped|Failed)\s+(?<service_name>.+)\."
| stats count by host, action, service_name
| sort -count

Windows (Event IDs 7036 / 7040)

index=main source="WinEventLog:System" (EventCode=7036 OR EventCode=7040)
| table _time, host, EventCode, Message
| sort -_time

Explanation: Event ID 7036 logs service state changes (started/stopped); Event ID 7040 logs service start-type changes (e.g., from automatic to disabled). Both are relevant to verifying that hardening-disabled services remain off.

Start broad, then narrow. Begin with index=* host=<hostname> to see all data from a host, then add keywords and filters to isolate specific security events. This is especially useful when you are not sure which source or sourcetype contains the events you need.

§3 - Building Security Dashboards

🎯 Project 3 Requirement
Project 3 requires at least three meaningful dashboards. Each dashboard should surface security-relevant information and include a written explanation of what it reveals. The walk-through below builds one complete dashboard; you will create at least two more for your submission.

Dashboard Design Principles

  • Purpose-driven: Every panel should answer a specific security question (e.g., “Are there failed login attempts?” “Which ports are being blocked?”).
  • Appropriate visualization: Choose panel types that match the data:
    • Timechart - trends over time (e.g., failed logins per hour)
    • Table - detailed event data (e.g., lockout events with timestamps)
    • Single Value - key performance indicators (e.g., total blocked connections today)
    • Column/Bar Chart - comparisons (e.g., top 10 blocked source IPs)
  • Readable: Use clear panel titles, appropriate time ranges, and avoid clutter.
  • Actionable: A viewer should be able to look at the dashboard and decide whether action is needed.

Walk-Through: Building a “Security Overview” Dashboard

Follow these steps to create a multi-panel dashboard in Splunk Web. This dashboard will contain four panels covering different security event categories.

1

Create a New Dashboard

In Splunk Web, navigate to Dashboards (from the top menu bar) and click Create New Dashboard. Enter the following:

  • Title: Security Overview
  • Description: Multi-panel dashboard for security event monitoring and hardening validation
  • Permissions: Private (you can share later)
  • Dashboard Type: Classic Dashboard

Click Create. You will be taken to the dashboard editor.

2

Panel 1 - Failed Logins Over Time (Timechart)

Click Add Panel → New → Timechart. Enter the following search:

index=main (source="/var/log/auth.log" "Failed password")
    OR (source="WinEventLog:Security" EventCode=4625)
| timechart span=1h count by host

Panel title: Failed Login Attempts Over Time

Time range: Last 7 days

What it shows: A line chart with one series per host, showing failed login frequency per hour. Spikes indicate brute-force attempts or misconfiguration. A flat zero line after disabling password-based SSH confirms key-only authentication is working.

Click Add to Dashboard.

3

Panel 2 - Total Firewall Blocks (Single Value)

Click Add Panel → New → Single Value. Enter the following search:

index=main ("UFW BLOCK" OR "iptables" OR (source="WinEventLog:Security" EventCode=5157 Action="Block"))
| stats count as total_blocks

Panel title: Total Firewall Blocks (Last 24h)

Time range: Last 24 hours

What it shows: A single number representing all firewall-blocked connections across Linux and Windows hosts. A non-zero number confirms firewall rules are actively enforcing policy.

Click Add to Dashboard.

4

Panel 3 - Privilege Escalation Events (Table)

Click Add Panel → New → Statistics Table. Enter the following search:

index=main source="/var/log/auth.log" "sudo"
| rex field=_raw "(?<sudo_user>\w+)\s+:\s+.*COMMAND=(?<command>.+)"
| table _time, host, sudo_user, command
| sort -_time
| head 50

Panel title: Recent Sudo / Privilege Escalation Events

Time range: Last 7 days

What it shows: A table listing the 50 most recent sudo commands with timestamps, hosts, users, and commands. Review for unexpected users or suspicious commands.

Click Add to Dashboard.

5

Panel 4 - Account Lockouts (Table)

Click Add Panel → New → Statistics Table. Enter the following search:

index=main source="WinEventLog:Security" EventCode=4740
| table _time, Account_Name, ComputerName, Source_Network_Address
| sort -_time

Panel title: Windows Account Lockout Events

Time range: Last 7 days

What it shows: A table of all account lockout events. If your lockout policy is working, this panel will populate when an account exceeds the failed login threshold. An empty table after hardening may mean no lockouts have been triggered—which is also useful evidence.

Click Add to Dashboard.

6

Save the Dashboard

Click Save in the dashboard editor. Your “Security Overview” dashboard now has four panels covering failed logins, firewall activity, privilege escalation, and account lockouts.

§4 - Configuring Splunk Alerts

🎯 Project 3 Requirement
Project 3 requires at least one alert configuration. This section walks you through creating a meaningful security alert.

What Are Splunk Alerts?

A Splunk alert is a saved search that runs on a schedule (or in real time) and triggers one or more actions when specified conditions are met. Alerts automate the detection of security events so you do not have to watch dashboards constantly.

When to Use Alerts

  • Threshold-based detection: Alert when a count exceeds a limit (e.g., >5 failed logins in 10 minutes)
  • Anomaly detection: Alert when an unexpected event occurs (e.g., a disabled service starts)
  • Compliance monitoring: Alert when a required condition is not met (e.g., no log data received from a host in 1 hour)

Alert Actions (Conceptual)

ActionDescriptionLab Applicability
Log EventWrites an entry to Splunk’s internal index✅ Works in all environments
Email NotificationSends an email to specified recipientsRequires mail server configuration
WebhookSends an HTTP POST to an external service (e.g., Slack, PagerDuty)Conceptual awareness
Run a ScriptExecutes a custom script on the Splunk serverAdvanced; conceptual awareness

Walk-Through: Creating a Failed Login Alert

This example creates a scheduled alert that triggers when more than 5 failed login attempts from a single source IP occur within a 10-minute window.

1

Write and Test the Search

First, run this search in the Splunk search bar to verify it returns results:

index=main (source="/var/log/auth.log" "Failed password")
    OR (source="WinEventLog:Security" EventCode=4625)
| stats count by src_ip
| where count > 5

This search counts failed logins by source IP and filters to only those with more than 5 failures. Adjust the threshold as appropriate for your lab environment.

2

Save As Alert

Click Save As → Alert. Configure the following settings:

  • Title: High Failed Login Rate from Single IP
  • Description: Triggers when a single source IP generates more than 5 failed login attempts in 10 minutes
  • Alert Type: Scheduled
  • Schedule: Every 10 minutes (Cron: */10 * * * *)
  • Time Range: Last 10 minutes (earliest=-10m latest=now)
3

Set Trigger Conditions

  • Trigger condition: Number of results is greater than 0
  • Throttle: Suppress triggering for 30 minutes (prevents alert fatigue from the same event)
4

Configure Alert Action

  • Select Log Event as the alert action (this works in all lab environments without mail server setup).
  • Optionally add Email if your lab has a mail relay configured.

Click Save.

5

Test and Verify

To test your alert:

  1. Generate failed login attempts from a test machine (e.g., use ssh baduser@target repeatedly).
  2. Wait for the alert schedule to run (up to 10 minutes).
  3. Check Activity → Triggered Alerts in Splunk Web to confirm the alert fired.
  4. Take a screenshot of the triggered alert for your Project 3 documentation.
📋 Other Alert Ideas for Project 3
You only need one alert, but consider these alternatives or additions:
  • Account lockout alert: Trigger on any Event ID 4740
  • Service state change alert: Trigger when a hardening-disabled service restarts
  • No data received alert: Trigger when a host stops sending logs for >1 hour (indicates forwarder failure)
  • Firewall block spike: Trigger when firewall blocks exceed a threshold in a time window

Your Project 3 submission must include comprehensive documentation of your Splunk monitoring setup. This section provides a checklist and guidance for capturing the evidence you need.

Capturing Dashboard Screenshots

  • Open each dashboard in Splunk Web and set the time range to show meaningful data (e.g., last 7 days or last 24 hours).
  • Take a full-page screenshot that shows all panels, the dashboard title, and the time range selector.
  • If a dashboard has more panels than fit on one screen, take multiple screenshots or scroll-capture.
  • Save screenshots with descriptive filenames (e.g., dashboard-security-overview.png).

Writing Dashboard Explanations

For each dashboard, write a brief explanation that covers:

  1. Purpose: What security question does this dashboard answer?
  2. Panels: What does each panel show? What SPL search powers it?
  3. Findings: What does the data reveal? Are hardening controls working?
  4. Action: What would you do if the data showed something unexpected?

Connecting Monitoring to Hardening Decisions

The strongest Project 3 submissions explicitly connect Splunk outputs to hardening actions. Here are examples of effective documentation statements:

  • “The Account Lockout table is empty, indicating no accounts have been locked out since enabling the lockout policy. This is expected in a lab environment with controlled access, but the policy is ready to trigger if brute-force attempts occur.”

⚠️ Common Pitfalls
  • Empty dashboards: If your dashboards show no data, verify your forwarders are running and data is being indexed. Run index=* host=<hostname> | head 10 to check.
  • Generic searches: Avoid dashboards that just show “all events.” Each panel should target a specific security event type.
  • Missing explanations: Screenshots alone are not sufficient. Always include a written explanation of what the dashboard shows and how it connects to your hardening work.
  • Untested alerts: Always test your alert by generating the triggering condition and capturing evidence that it fired.

2. You want to display the total number of firewall-blocked connections in the last 24 hours as a single prominent number on a dashboard. Which panel type is most appropriate?

✅ Correct! A Single Value panel displays one prominent number, making it ideal for KPIs like “total blocked connections.” A timechart would show trends over time, a table would show individual events, and a bar chart would compare categories.

3. You configure a Splunk alert to trigger when more than 5 failed logins from a single IP occur in 10 minutes. Which trigger condition is correct?

✅ Correct! The SPL search itself filters to IPs with count > 5 using a where clause. The alert trigger condition only needs to check whether the search returned any results (number of results > 0), because any result that appears has already exceeded the threshold.
❌ Not quite. The correct answer is B. The threshold logic (| where count > 5) is built into the SPL search itself. The alert trigger only needs to check whether the filtered search returned any results at all. If it returns > 0 results, at least one IP exceeded the threshold.

4. After disabling password-based SSH authentication and enforcing key-only login, your Splunk dashboard shows zero “Failed password” events over the past week. What does this indicate?

✅ Correct! When password authentication is disabled in sshd_config, the SSH server rejects password attempts at the protocol level, so “Failed password” entries are no longer generated. ❌ Not quite. The correct answer is C. When password authentication is disabled, SSH rejects password attempts at the protocol level before generating a “Failed password” log entry. Zero events is the expected result and confirms the hardening is working. To rule out forwarder issues, verify that other events from the same host are appearing in Splunk.

5. Which Windows Security Event ID would you search for to detect account lockout events?

✅ Correct! Event ID 4740 is logged when a user account is locked out. Event ID 4625 is a failed logon, 7036 is a service state change (System log, not Security), and 5157 is a Windows Filtering Platform connection block.
❌ Not quite. The correct answer is B - Event ID 4740. This event is logged when an account lockout occurs. Event ID 4625 = failed logon, 7036 = service state change, 5157 = firewall connection block.

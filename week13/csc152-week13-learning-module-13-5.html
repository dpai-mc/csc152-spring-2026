1. Splunk Forwarder Architecture

Why Forwarders?

In a production Splunk deployment you almost never install a full Splunk Enterprise instance on every server you want to monitor. Instead, you deploy lightweight forwarders that collect log data at the source and ship it to a central Splunk instance for indexing and search. This keeps resource consumption low on monitored hosts and centralizes data management.

Forwarder Types Compared

FeatureUniversal Forwarder (UF)Heavy Forwarder (HF)Full Splunk Enterprise
Footprint~50 MB RAM; minimal CPU~1 GB+ RAM; moderate CPUFull installation; significant resources
Parsing / IndexingNo local parsing or indexingCan parse, filter, route before forwardingFull parsing, indexing, and search
Web UINone — CLI and config files onlyOptional — can enable Splunk WebFull Splunk Web interface
Typical Use CaseCollect & forward logs from endpoints and serversIntermediate tier: filter sensitive data, route to multiple indexersIndexing, searching, dashboards, alerts

For CSC 152, we use Universal Forwarders on every monitored host and a single Splunk Enterprise instance as the receiver/indexer.

Key Configuration Files

Two files control what a Universal Forwarder collects and where it sends data:

  • outputs.conf — Tells the forwarder where to send data (your Splunk Enterprise server's IP address and port 9997).
  • inputs.conf — Tells the forwarder what to collect (which log files or event log channels to monitor).

Both files live in the forwarder's local configuration directory. We will create and annotate them in the sections that follow.

2. Installing the Universal Forwarder on Linux

The steps below install the Splunk Universal Forwarder on an Ubuntu Server VM. The installer package is provided in your lab environment — refer to the Lab 13 instructions for the exact file location.

💡 Tip

Replace splunkforwarder-<version>-linux-amd64.deb with the actual filename provided in your lab files. The version number may differ.

1
Download / copy the installer to your Linux VM.

Your instructor has provided the .deb package. Transfer it to your server (for example, via scp or a shared folder).

2
Install the package. 
sudo dpkg -i splunkforwarder-<version>-linux-amd64.deb

This installs the forwarder to /opt/splunkforwarder.

3
Accept the license and start the forwarder for the first time. 
sudo /opt/splunkforwarder/bin/splunk start --accept-license

You will be prompted to create an admin username and password for the forwarder. Remember these credentials.

4
Enable the forwarder to start at boot. 
sudo /opt/splunkforwarder/bin/splunk enable boot-start

This creates a systemd unit so the forwarder starts automatically when the server reboots.

5
Verify the forwarder is running. 
sudo /opt/splunkforwarder/bin/splunk status

You should see output indicating splunkd is running.

📝 Note

The Universal Forwarder has no web interface. All configuration is done through config files and the CLI. This is by design — the UF is meant to be lightweight.

3. Configuring outputs.conf (Linux)

The outputs.conf file tells the Universal Forwarder where to send collected data. Create this file in the forwarder's local configuration directory:

sudo nano /opt/splunkforwarder/etc/system/local/outputs.conf

Add the following content:

# outputs.conf — Splunk Universal Forwarder (Linux)
# Tells the forwarder where to send data

[tcpout]
# Default output group — all data goes to this group
defaultGroup = splunk-server

[tcpout:splunk-server]
# IP address (or hostname) and receiving port of your
# Splunk Enterprise server from Module 12.5
server = <SPLUNK_SERVER_IP>:9997

# Optional: compressed transmission (reduces bandwidth)
# compressed = true

Directive Explanations

Stanza / DirectivePurpose
[tcpout]Global TCP output settings. The defaultGroup directive specifies which output group receives all data by default.
defaultGroup = splunk-serverNames the output group. Must match the [tcpout:<name>] stanza below.
[tcpout:splunk-server]Defines the target server group named splunk-server.
server = <IP>:9997The IP address and port of your Splunk Enterprise receiver. Port 9997 is the default receiving port you enabled in Module 12.5.
⚠ Common Mistake

Make sure the defaultGroup name exactly matches the name after the colon in [tcpout:<name>]. A mismatch means no data is forwarded — and no error message is displayed.

After saving the file, restart the forwarder to apply the change:

sudo /opt/splunkforwarder/bin/splunk restart

4. Configuring inputs.conf (Linux)

The inputs.conf file tells the forwarder what to monitor. Create it in the same local directory:

sudo nano /opt/splunkforwarder/etc/system/local/inputs.conf

Add the following content to monitor syslog and authentication logs:

# inputs.conf — Splunk Universal Forwarder (Linux)
# Defines which log files to monitor and forward

[monitor:///var/log/syslog]
# Monitor the main system log
disabled = false
# sourcetype tells Splunk how to parse this data
sourcetype = syslog
# index specifies which Splunk index receives this data
# Use "main" (default) or a custom index you created
index = main

[monitor:///var/log/auth.log]
# Monitor the authentication log — login attempts,
# sudo usage, SSH sessions, and PAM events
disabled = false
sourcetype = linux_secure
index = main

Directive Explanations

Stanza / DirectivePurpose
[monitor://<path>]Tells the forwarder to continuously read new data appended to the specified file. The triple slash (///) represents monitor:// + absolute path starting with /.
disabled = falseExplicitly enables this input. If set to true, the stanza is ignored.
sourcetype = syslogAssigns a sourcetype label so Splunk knows how to parse the data. Use recognized sourcetype names for best results.
index = mainRoutes data to the main index. In production, you might create dedicated indexes (e.g., linux_logs).
💡 Why auth.log?

/var/log/auth.log records every login attempt (successful and failed), sudo command usage, and SSH session. This is the single most valuable log file for security monitoring on a Linux server — and it is the data source for the dashboard panel you will build in Section 8.

After saving, restart the forwarder:

sudo /opt/splunkforwarder/bin/splunk restart
📝 Permissions Note

The Splunk forwarder process must have read access to the log files specified in inputs.conf. On Ubuntu, /var/log/syslog and /var/log/auth.log are typically readable by the adm group. If the forwarder runs as the splunkfwd user, you may need to add that user to the adm group: sudo usermod -aG adm splunkfwd, then restart the forwarder.

5. Installing the Universal Forwarder on Windows

The Windows Universal Forwarder uses an MSI installer with a graphical wizard. The installer is provided in your lab environment — refer to the Lab 13 instructions for the file location.

1
Run the MSI installer.

Double-click the splunkforwarder-<version>-x64-release.msi file. Accept the license agreement.

2
Choose installation options.
  • Install as Local System — this gives the forwarder permission to read Windows Event Logs.
  • Set an admin username and password when prompted.
3
Optionally configure the receiving indexer during installation.

The wizard may prompt you for a deployment server or receiving indexer. You can enter your Splunk Enterprise server's IP and port 9997 here, or skip this and configure outputs.conf manually (covered in Section 6).

4
Complete the installation and verify the service.

Open Services (services.msc) and confirm that SplunkForwarder Service is listed with a status of Running and startup type of Automatic.

5

cd "C:\Program Files\SplunkUniversalForwarder\bin"
.\splunk.exe status

You should see splunkd is running.

📝 Installation Path

The default Windows installation path is C:\Program Files\SplunkUniversalForwarder. Configuration files are located under ...\etc\system\local\, just like the Linux forwarder.

6. Configuring outputs.conf & inputs.conf (Windows)

The configuration files on Windows follow the same syntax as Linux — only the file paths and input types differ. Navigate to the local config directory:

C:\Program Files\SplunkUniversalForwarder\etc\system\local\

outputs.conf (Windows)

Create or edit outputs.conf in the directory above. The content is identical to the Linux version:

# outputs.conf — Splunk Universal Forwarder (Windows)

[tcpout]
defaultGroup = splunk-server

[tcpout:splunk-server]
server = <SPLUNK_SERVER_IP>:9997

inputs.conf (Windows)

Windows servers use Windows Event Log channels instead of flat log files. Create or edit inputs.conf:

# inputs.conf — Splunk Universal Forwarder (Windows)
# Monitor Windows Event Log channels

[WinEventLog://Security]
# The Security log contains logon/logoff events,
# privilege use, account management, and policy changes
disabled = false
index = main
# evt_resolve_ad_obj = true  (optional: resolve AD SIDs to names)

[WinEventLog://System]
# The System log contains service start/stop, driver
# failures, and hardware events
disabled = false
index = main

[WinEventLog://Application]
# The Application log contains software-level events
# from installed applications and services
disabled = false
index = main

Stanza Explanation

Stanza / DirectivePurpose
[WinEventLog://<channel>]Monitors a Windows Event Log channel. The channel name must match exactly (e.g., Security, System, Application).
disabled = falseEnables collection from this channel.
index = mainRoutes events to the main index on the Splunk Enterprise server.

After saving both files, restart the forwarder service. Open an Administrator Command Prompt:

cd "C:\Program Files\SplunkUniversalForwarder\bin"
.\splunk.exe restart

Alternatively, restart the SplunkForwarder Service in services.msc.

⚠ Case Sensitivity

Windows Event Log channel names are case-sensitive in inputs.conf. Use Security, not security. An incorrect channel name will silently produce no data.

7. Verifying Data Ingestion

After configuring forwarders on both platforms, switch to your Splunk Enterprise web interface (http://<SPLUNK_SERVER_IP>:8000) to confirm data is arriving.

Step-by-Step Verification

1
Search for all data from your hosts.

In the Splunk search bar, enter:

index=* host=<linux-hostname> OR host=<windows-hostname>

Set the time range to Last 15 minutes. You should see events from both hosts. If you see results, your forwarders are working.

2
Check event counts by host and sourcetype. 
index=* | stats count by host, sourcetype

This shows you which hosts are sending data and what types of data are arriving. You should see entries like:

hostsourcetypecount
linux-srv01syslog342
linux-srv01linux_secure87
win-srv01WinEventLog:Security1,204
win-srv01WinEventLog:System156
win-srv01WinEventLog:Application93
3
Check Splunk internal logs for forwarder connections. 
index=_internal sourcetype=splunkd "Connected to" OR "accepted from"

This searches Splunk's own internal logs for connection events. You should see messages indicating forwarders have connected on port 9997.

Troubleshooting Tips

If you do not see data from a forwarder, work through this checklist:

SymptomLikely CauseFix
No data at all from a host Firewall blocking port 9997 Check pfSense rules: allow TCP 9997 from the forwarder's subnet to the Splunk server. Verify with telnet <splunk-ip> 9997 from the forwarder host.
No data at all from a host outputs.conf typo — wrong IP, wrong port, or defaultGroup name mismatch Double-check outputs.conf. The defaultGroup value must exactly match the [tcpout:<name>] stanza name.
No data at all from a host Forwarder service not running Linux: sudo /opt/splunkforwarder/bin/splunk status. Windows: check services.msc for SplunkForwarder Service.
Host appears but specific sourcetype missing inputs.conf stanza error — wrong path, wrong channel name, or disabled = true Review inputs.conf carefully. On Linux, verify the log file exists and the forwarder user can read it. On Windows, verify the channel name is spelled correctly with correct capitalization.
Receiver port not listening on Splunk Enterprise Port 9997 was not enabled in Module 12.5 In Splunk Web: Settings → Forwarding and Receiving → Configure Receiving → verify port 9997 is listed and enabled.

8. Creating Your First Dashboard Panel

Now that data is flowing, you will create a dashboard panel that visualizes failed login attempts over time. This is a fundamental security monitoring view and one of the dashboard deliverables for Project 3.

Platform-Specific SPL Searches

The SPL query differs depending on which host's data you are visualizing. Use the tabs below to see the appropriate search for your platform:

Linux — Failed SSH/login attempts from auth.log:

index=main sourcetype=linux_secure "Failed password"
| timechart span=1h count AS "Failed Logins"

This searches auth.log data for the string "Failed password" (generated by SSH and PAM on failed login attempts) and charts the count per hour.

Windows — Failed logon events (Event ID 4625):

index=main sourcetype="WinEventLog:Security" EventCode=4625
| timechart span=1h count AS "Failed Logins"

This searches the Windows Security Event Log for Event ID 4625 (failed logon) and charts the count per hour. Recall from Module 11.5 that Event ID 4625 is the key indicator for failed authentication attempts.

Building the Dashboard

1
Run your search.

Paste the appropriate SPL query into the Splunk search bar. Set the time range to Last 24 hours (or Last 4 hours if your forwarder was just installed). Click Search.

2
Switch to the Visualization tab.

After results appear, click the Visualization tab above the results. Select a Line Chart or Column Chart visualization type. You should see failed logins plotted over time.

3
Save as a Dashboard Panel.

Click Save As → Dashboard Panel.

  • Dashboard Title: CSC 152 Security Monitoring (or a descriptive name of your choice)
  • Panel Title: Failed Login Attempts Over Time
  • Panel Type: keep the default (inline search)

Click Save. Splunk creates a new dashboard with your panel.

4
View your dashboard.

Click View Dashboard to see the finished result. You now have a working security monitoring dashboard that updates in real time.

10. Configuration Review Activity

A fellow student asks you to review their forwarder configuration files because data is not appearing in Splunk. Examine each configuration below, identify the error, and click the option that describes the problem. Immediate feedback is provided.

Scenario 1: Linux outputs.conf

The student's outputs.conf on their Linux server contains:

[tcpout]
defaultGroup = my-splunk

[tcpout:splunk-server]
server = 10.0.1.50:9997
A. The port number should be 8000, not 9997.
B. The defaultGroup name (my-splunk) does not match the stanza name (splunk-server).
C. The [tcpout] stanza should be [tcp].
✓ Correct! The defaultGroup value is my-splunk, but the output group stanza is named [tcpout:splunk-server]. These must match. The fix is to change either the defaultGroup to splunk-server or rename the stanza to [tcpout:my-splunk].
✗ Not quite. Look at the defaultGroup value and the [tcpout:...] stanza name. Do they match? The defaultGroup tells Splunk which output group to use — if the name does not match any defined group, data goes nowhere.

Scenario 2: Windows inputs.conf

The student's inputs.conf on their Windows server contains:

[WinEventLog://security]
disabled = false
index = main

[WinEventLog://System]
disabled = false
index = main
A. The index = main directive is wrong; it should be index = wineventlog.
B. The first stanza uses security (lowercase) instead of Security (capitalized).
C. The WinEventLog:// prefix should be monitor:// on Windows.
✓ Correct! Windows Event Log channel names are case-sensitive in inputs.conf. The channel is named Security (capital S), not security. With the wrong case, the forwarder silently collects nothing from that channel — no error message is displayed.
✗ Not quite. The WinEventLog:// prefix is correct for Windows Event Logs, and index = main is valid. Look carefully at the capitalization of the channel name in the first stanza. Windows Event Log channel names are case-sensitive.

Scenario 3: Linux inputs.conf

The student's inputs.conf on their Linux server contains:

[monitor:///var/log/syslog]
disabled = true
sourcetype = syslog
index = main

[monitor:///var/log/auth.log]
disabled = false
sourcetype = linux_secure
index = main
A. The syslog stanza has disabled = true, which prevents it from being monitored.
B. The sourcetype for auth.log should be syslog, not linux_secure.
C. Both stanzas should use index = linux instead of index = main.
✓ Correct! The disabled = true directive in the syslog stanza tells the forwarder to ignore this input. The student likely copied a template and forgot to change it to disabled = false. Meanwhile, auth.log data is being collected because its stanza correctly has disabled = false.
✗ Not quite. The sourcetype and index values are fine. Look at the disabled directive in the first stanza. What does disabled = true do?

11. Knowledge Check

This ungraded self-assessment covers the key concepts from this module. Select your answer and click Check Answer for immediate feedback.

1. What is the primary difference between a Splunk Universal Forwarder and a full Splunk Enterprise instance?

✓ Correct! The Universal Forwarder is a lightweight agent that collects log data and forwards it to a Splunk Enterprise instance (or indexer) for parsing, indexing, and search. It has no web UI and does not index data locally, which keeps its resource footprint minimal.
✗ Not quite. The Universal Forwarder runs on both Linux and Windows. Its key characteristic is that it does not index data or provide a search interface — it only collects and forwards. This is what makes it lightweight enough to deploy on every monitored host.

2. What is the purpose of the outputs.conf file on a Universal Forwarder?

✓ Correct! outputs.conf tells the forwarder where to send data. It contains the IP address and port (typically 9997) of the Splunk Enterprise receiver. The file that specifies what to collect is inputs.conf.
✗ Not quite. Remember the two key config files: outputs.conf = where to send data (the receiver's address and port). inputs.conf = what to collect (log files and event channels). The correct answer describes the "where" function.

3. Which inputs.conf stanza correctly monitors the Windows Security Event Log?

✓ Correct! The proper stanza for Windows Event Logs uses the [WinEventLog://<ChannelName>] format, and the channel name is case-sensitive. Security (capital S) is the correct name. Option C would fail silently because of the lowercase "s."
✗ Not quite. Windows Event Logs are monitored using the [WinEventLog://] prefix (not [monitor://] or [eventlog://]). The channel name must be capitalized exactly as Windows defines it: Security, System, Application.

4. After installing forwarders on two hosts, you want to confirm that data is arriving in Splunk. Which SPL search would show event counts grouped by host and sourcetype?

✓ Correct! index=* | stats count by host, sourcetype searches all indexes and produces a table showing how many events each host has sent, broken down by sourcetype. This is the go-to verification search after deploying forwarders.
✗ Not quite. The | table command lists individual events (not counts). The | timechart command shows data over time but does not group by sourcetype. The _internal index contains Splunk's own logs, not your forwarded data. The command that counts events grouped by fields is | stats count by <field1>, <field2>.

Defining VM Specifications

Before you click "New" in any hypervisor, you need to know what resources your VM requires. Over-provisioning wastes host resources; under-provisioning causes poor performance. The three primary resources to size are vCPU, vRAM, and virtual disk.

vCPU (Virtual CPU)

  • A vCPU is a virtual processor core assigned to a VM. The hypervisor schedules vCPU time on the host's physical cores.
  • General rule: Start with the OS minimum and add cores based on workload. A lightly loaded server (DNS, DHCP) may need only 1 vCPU; a database server may need 4+.
  • Overcommitment: You can assign more total vCPUs across all VMs than the host has physical cores — the hypervisor time-shares. However, excessive overcommitment degrades performance.

vRAM (Virtual Memory)

  • Memory allocated to a VM is reserved from host RAM while the VM is running.
  • General rule: Always meet the OS minimum, then add based on workload. Memory is typically the first bottleneck in virtualized environments.
  • Host reserve: Always leave enough RAM for the host OS itself (at least 2 GB for a Type 2 hypervisor host, more if running multiple VMs).

Virtual Disk

  • The virtual disk stores the guest OS, applications, and data. You choose both the size and the allocation type (dynamically allocated vs. fixed — covered in Section 5).
  • General rule: Size for the OS plus expected data growth. You can often expand a virtual disk later, but shrinking is difficult.

⚠️ Host Resource Budget

If your lab host has 16 GB RAM and you plan to run three VMs simultaneously (e.g., a Windows Server at 4 GB, a Linux server at 1.5 GB, and a Splunk server at 2 GB), that's 7.5 GB for VMs plus 2–4 GB for the host OS. Always calculate your total VM footprint before creating new machines.

Creating a VM — Platform Walkthroughs

The following walkthroughs cover the VM creation process in VirtualBox (our primary lab platform) and Hyper-V (an alternative for students on Windows Pro/Education). Expand the platform you're using.

🖥️ VirtualBox — Step-by-Step VM Creation

These steps use VirtualBox 7.x. The interface may vary slightly with other versions.

  1. Open VirtualBox Manager — Launch VirtualBox. You'll see the main window listing any existing VMs on the left.
  2. Click "New" — In the toolbar, click the New button (or press Ctrl+N) to start the Create Virtual Machine wizard.
  3. Name and Operating System — Enter a descriptive Name (e.g., Ubuntu-Server-Lab). Choose the Folder where VM files will be stored. Select the ISO Image if you want VirtualBox to auto-detect the OS type. Otherwise, manually set Type (e.g., Linux) and Version (e.g., Ubuntu 64-bit). If you check Skip Unattended Installation, you'll do a manual install.
  4. Hardware — Memory and CPU — Set Base Memory (vRAM) using the slider or type a value (e.g., 2048 MB). Set Processors (vCPU count). Stay within the green zone on the slider to avoid overcommitting host resources.
  5. Virtual Hard Disk — Choose Create a Virtual Hard Disk Now. Set the Disk Size (e.g., 25 GB). By default VirtualBox uses VDI format and dynamically allocated storage. You can change these under the Hard Disk File Type and Variant options (more on this in Section 5).
  6. Review and Finish — The wizard shows a summary. Verify your settings and click Finish. The new VM appears in the Manager's list.
  7. Attach Installation Media (if not done in Step 3) — Select the VM → Settings → Storage → click the empty disk icon under the IDE controller → click the disk icon on the right → Choose a disk file → browse to your ISO.
  8. Start the VM — Click Start. The VM boots from the ISO and the OS installer launches. Proceed with installation as you did in Weeks 3–4.
💡 VBoxManage CLI Alternative

You can create a VM entirely from the command line:
VBoxManage createvm --name "Ubuntu-Server-Lab" --ostype Ubuntu_64 --register
VBoxManage modifyvm "Ubuntu-Server-Lab" --memory 2048 --cpus 2
VBoxManage createmedium disk --filename "Ubuntu-Server-Lab.vdi" --size 25600
This is useful for scripting and automation.

🪟 Hyper-V — Step-by-Step VM Creation

Hyper-V is available on Windows 10/11 Pro, Education, and Enterprise editions, and as a server role on Windows Server. These steps use Hyper-V Manager.

  1. Open Hyper-V Manager — Search for "Hyper-V Manager" in the Start menu, or run virtmgmt.msc.
  2. New → Virtual Machine — In the Actions pane (right side), click New → Virtual Machine to launch the wizard.
  3. Specify Name and Location — Enter a Name (e.g., Ubuntu-Server-Lab). Optionally change the storage location.
  4. Specify Generation — Choose Generation 1 (BIOS, broad compatibility) or Generation 2 (UEFI, Secure Boot — required for modern Windows, optional for Linux). For Ubuntu Server, Generation 2 is recommended.
  5. Assign Memory — Enter the startup memory (e.g., 2048 MB). Optionally enable Dynamic Memory, which lets Hyper-V adjust RAM allocation based on demand.
  6. Configure Networking — Select a Virtual Switch from the dropdown. If none exist, you'll create one later (covered in Module 13.3).
  7. Connect Virtual Hard Disk — Choose Create a virtual hard disk. Set the Name, Location, and Size (e.g., 25 GB). Hyper-V creates a VHDX file (dynamically expanding by default).
  8. Installation Options — Select Install an operating system from a bootable image file and browse to your ISO.
  9. Summary and Finish — Review settings and click Finish.
  10. Set vCPU Count — Right-click the VM → Settings → Processor → set Number of virtual processors (e.g., 2). The wizard doesn't prompt for CPU count, so this is a post-creation step.
  11. Start and Connect — Right-click the VM → Connect (opens the console window) → Start. The VM boots from the ISO.
💡 PowerShell Alternative

New-VM -Name "Ubuntu-Server-Lab" -MemoryStartupBytes 2GB -NewVHDPath ".\Ubuntu-Server-Lab.vhdx" -NewVHDSizeBytes 25GB -Generation 2
Set-VMProcessor -VMName "Ubuntu-Server-Lab" -Count 2
Add-VMDvdDrive -VMName "Ubuntu-Server-Lab" -Path "C:\ISOs\ubuntu-22.04-server.iso"

VM Configuration Settings — Deep Dive

After creating a VM shell, you can fine-tune dozens of settings. This section covers the most important categories you'll encounter in both VirtualBox and Hyper-V.

Processor Settings

  • vCPU Count: Adjustable post-creation. Increasing vCPUs helps multi-threaded workloads but adds scheduling overhead on the host.
  • Execution Cap (VirtualBox): Limits the percentage of host CPU time a VM can use. Useful for preventing a runaway VM from starving the host. Default is 100%.
  • Nested Virtualization: Allows a VM to run its own hypervisor (e.g., running Hyper-V inside a VirtualBox VM). Must be explicitly enabled; needed for certain advanced labs.
  • PAE/NX: Physical Address Extension and No-Execute bit. Typically enabled by default; required by some 32-bit Linux kernels to access more than 4 GB RAM.

Memory Settings

  • Base Memory: The amount of RAM allocated when the VM starts. Changes require the VM to be powered off.
  • Dynamic Memory (Hyper-V): Allows the hypervisor to adjust RAM between a minimum and maximum value based on demand. VirtualBox does not have a direct equivalent (memory is statically assigned).
  • Ballooning (Advanced): A technique where the hypervisor reclaims unused memory from a guest. Supported by Guest Additions in VirtualBox.

Display Settings

  • Video Memory: Amount of VRAM allocated for the virtual display adapter. Increase for GUI-based guests (128 MB recommended for desktop environments).
  • Graphics Controller (VirtualBox): Options include VBoxVGA (legacy), VMSVGA (recommended for Linux), and VBoxSVGA (recommended for Windows guests).
  • Remote Display: VirtualBox supports VRDP (VirtualBox Remote Desktop Protocol) for remote console access; Hyper-V uses Enhanced Session Mode with RDP.

USB Pass-Through

  • USB Controller Type: USB 1.1 (OHCI), USB 2.0 (EHCI), or USB 3.0 (xHCI). Higher versions require the VirtualBox Extension Pack.
  • Device Filters: Define which host USB devices are automatically attached to the VM when plugged in. Useful for USB drives, serial adapters, or hardware keys.
  • Hyper-V Enhanced Session Mode: Provides USB redirection through the RDP connection rather than direct pass-through.

Most hardware-level settings (vCPU count, base memory, storage controllers, firmware type) can only be changed when the VM is powered off. Display settings and USB filters can sometimes be adjusted at runtime, but it's safest to shut down first.

Virtual Disk Types & Formats

A virtual disk is a file (or set of files) on the host that appears as a physical hard drive to the guest OS. Two key decisions affect performance, portability, and storage efficiency: the allocation type and the disk format.

Allocation Type: Dynamically Allocated vs. Fixed-Size

Dynamically Allocated (Thin Provisioning)
  • How it works: The disk file starts small and grows as the guest writes data, up to the maximum size you defined.
  • Pros: Conserves host disk space; quick to create; allows overcommitting storage.
  • Cons: Slight write performance overhead (file must expand); fragmentation over time; does not shrink automatically when guest deletes files.
  • Best for: Lab environments, development VMs, situations where host disk space is limited.
Fixed-Size (Thick Provisioning)
  • How it works: The entire disk space is allocated on the host at creation time. A 40 GB virtual disk immediately consumes ~40 GB on the host.
  • Pros: Better write performance (no expansion overhead); predictable host space usage; less fragmentation.
  • Cons: Slower to create; consumes full space immediately even if guest uses little of it.
  • Best for: Production servers, performance-sensitive workloads, hosts with ample disk space.

Disk Formats

Different hypervisors use different virtual disk file formats. Understanding these is important for portability and migration.

VDI — VirtualBox Disk Image

  • Native to: Oracle VirtualBox
  • Supports: Dynamic and fixed allocation; snapshots (differencing images)
  • Portability: VirtualBox only (though VirtualBox can convert VDI ↔ VMDK ↔ VHD)
  • File extension: .vdi
  • When to use: Default choice when working exclusively in VirtualBox

VMDK — Virtual Machine Disk

  • Native to: VMware products (ESXi, Workstation, Fusion)
  • Supports: Dynamic (thin) and fixed (thick) provisioning; split into 2 GB files for FAT32 compatibility
  • Portability: VMware ecosystem; also supported by VirtualBox (read/write)
  • File extension: .vmdk
  • When to use: When you need compatibility with VMware platforms or plan to migrate VMs to ESXi

VHD / VHDX — Virtual Hard Disk

  • Native to: Microsoft Hyper-V
  • VHD (legacy): Max size 2 TB; 512-byte sectors; compatible with older Hyper-V and VirtualBox
  • VHDX (modern): Max size 64 TB; 4 KB sectors for better performance on large disks; resilient to power failure corruption; Hyper-V only
  • File extensions: .vhd, .vhdx
  • When to use: VHD for cross-platform compatibility; VHDX for Hyper-V production workloads

🔄 Converting Between Formats

VirtualBox includes a built-in conversion tool:
VBoxManage clonemedium disk source.vdi target.vmdk --format VMDK
You can also use qemu-img convert on Linux for broader format support. Conversion creates a new file; the original is unchanged.

Snapshots & Cloning

Snapshots and clones are two of the most powerful VM management tools. They serve different purposes: snapshots let you roll back in time; clones let you duplicate a VM.

Snapshots

A snapshot captures the exact state of a VM at a point in time — including disk contents, memory state (if the VM is running), and VM configuration. After taking a snapshot, all subsequent disk writes go to a differencing disk (delta file), leaving the original disk unchanged.

Snapshot Lifecycle

OperationWhat HappensUse Case
Create (Take)A new differencing disk is created. The current disk becomes read-only. All new writes go to the delta.Before installing software, applying updates, or making risky configuration changes.
Restore (Revert)The VM's disk and (optionally) memory state are rolled back to the snapshot point. Changes since the snapshot are discarded.When an update breaks the system, or you want to return to a known-good state.
Delete (Discard)The differencing disk is merged back into the parent. This does not undo changes — it consolidates them.When you're satisfied with changes and want to clean up snapshot overhead.
🚨 Snapshot Best Practices
  • Don't keep snapshots long-term. Differencing disks grow over time and degrade performance. Use snapshots as temporary safety nets, not permanent backups.
  • Limit snapshot depth. Avoid creating chains of more than 2–3 snapshots. Deep chains compound performance penalties.
  • Name your snapshots descriptively (e.g., "Pre-AD-Install" or "Clean-OS-with-Guest-Additions") so you can identify them later.
  • Snapshots are NOT backups. They depend on the original disk file. If the base disk is corrupted, all snapshots are lost.

Cloning

A clone creates a copy of an existing VM. Unlike snapshots, clones are independent (or semi-independent) VMs that can run alongside the original.

Full Clone
  • How it works: Creates a complete, independent copy of all VM files (disk, configuration, etc.).
  • Pros: Fully independent — can be moved to another host; no dependency on the original VM.
  • Cons: Takes longer to create; consumes as much disk space as the original.
  • Best for: Creating a production copy, distributing VMs to other machines, long-term use.
Linked Clone
  • How it works: Creates a new VM that shares the original's base disk (via a snapshot). Only differences are stored in a new differencing disk.
  • Pros: Very fast to create; minimal additional disk space.
  • Cons: Depends on the original VM's snapshot — if deleted, the linked clone breaks. Cannot be moved independently.
  • Best for: Rapid provisioning of temporary or test VMs from a template, classroom lab environments.

OVA / OVF — Export & Import

The Open Virtualization Format (OVF) is an industry standard for packaging VMs for distribution. An OVA file is simply a single-file archive (TAR) containing the OVF descriptor, virtual disks, and metadata.

  • Export: In VirtualBox, File → Export Appliance; in Hyper-V, use Export (creates a folder structure, not OVA — you'd need a conversion tool for true OVA).
  • Import: In VirtualBox, File → Import Appliance → select the OVA file → review settings → import. The VM is recreated with the packaged configuration.
  • Use cases: Distributing pre-configured lab VMs, migrating VMs between hosts, sharing appliances (e.g., a firewall VM template).
💡 Snapshot vs. Clone — Quick Decision Guide

Need to undo a change? → Snapshot.
Need a separate, parallel copy? → Clone.
Need a quick, temporary test copy? → Linked Clone.
Need to share a VM with someone? → Full Clone or OVA Export.

🎯 Scenario: Snapshot or Clone?

You have a working Windows Server VM with Active Directory configured. You need to:

  1. Test a Group Policy change that might break login functionality.
  2. Give a classmate an identical copy of the AD server for their own lab.

For each task, which approach is most appropriate?

Model Answer — B. Task 1 is a temporary, reversible test — a snapshot is perfect. Take a snapshot before applying the GPO, test it, and restore the snapshot if login breaks. Task 2 requires giving someone an independent copy they can run on their own machine. A full clone or OVA export creates a standalone VM with no dependency on your original. A linked clone (Option D for Task 1) would work but is overkill — snapshots are simpler for rollback. A snapshot alone (Option A) can't give your classmate a separate VM.

Guest Additions & Integration Tools

Out of the box, a guest OS running inside a VM has limited interaction with the host. Guest Additions (VirtualBox) and Integration Services (Hyper-V) are software packages installed inside the guest that dramatically improve the VM experience.

Installing Guest Additions (VirtualBox)

🐧 Linux Guest
  1. Install prerequisites:
    sudo apt update && sudo apt install -y build-essential dkms linux-headers-$(uname -r)
  2. Insert the Guest Additions CD: In the VirtualBox menu bar, click Devices → Insert Guest Additions CD image…
  3. Mount and run the installer:
    sudo mount /dev/cdrom /mnt
    sudo /mnt/VBoxLinuxAdditions.run
  4. Reboot the VM:
    sudo reboot
  5. Verify: After reboot, the VM window should auto-resize. Check with:
    lsmod | grep vboxguest
🪟 Windows Guest
  1. Insert the Guest Additions CD: In the VirtualBox menu bar, click Devices → Insert Guest Additions CD image…
  2. Run the installer: Open File Explorer → navigate to the CD drive → double-click VBoxWindowsAdditions.exe.
  3. Follow the wizard: Accept defaults. Allow driver installation prompts (click "Install" when Windows asks about unsigned drivers).
  4. Reboot when prompted.
  5. Verify: The VM window should auto-resize. Check the system tray for the VirtualBox Guest Additions icon.

Hyper-V Integration Services

Modern Windows guests include Integration Services automatically. For Linux guests on Hyper-V:

  • Most modern Linux kernels (4.x+) include Hyper-V integration modules (hv_vmbus, hv_storvsc, hv_netvsc, etc.) built in.
  • Verify with: lsmod | grep hv_
  • Enhanced Session Mode (for GUI access) requires xrdp installed on the Linux guest.

💡 Always Install Guest Tools

Guest tools should be one of the first things you install after the OS. Without them, you'll fight with mouse capture, low screen resolution, and no clipboard sharing — all of which slow down your lab work significantly.

Knowledge Check

Test your understanding of Module 13.2 with these ungraded questions. Click an answer to see immediate feedback.

1. In the VM creation workflow, at which step should you install Guest Additions or Integration Services?

✅ Correct! Guest tools are installed after the OS is running (Step 4) but before final configuration (Step 6). This is Step 5 in our six-step workflow.
❌ Not quite. Guest tools require a running OS to install into, so they come after OS installation (Step 4). The correct answer is Step 5. Review the workflow chart in Section 1.

2. A dynamically allocated virtual disk starts at 2 GB on the host and the guest has written 15 GB of data. The maximum disk size was set to 40 GB. Approximately how much host disk space is consumed?

✅ Correct! A dynamically allocated disk grows as data is written. With 15 GB of guest data, the host file is approximately 15 GB (plus minor metadata overhead). It will never exceed the 40 GB maximum.
❌ Not quite. Dynamically allocated disks grow to match the amount of data actually written by the guest. They don't stay at their initial size (A), and they don't pre-allocate the full maximum (C). The host file is approximately 15 GB. Review Section 5.

3. Which virtual disk format is native to Hyper-V and supports a maximum size of 64 TB?

✅ Correct! VHDX is the modern Hyper-V format supporting up to 64 TB, 4 KB sectors, and improved resilience. VHD (legacy) is limited to 2 TB.
❌ Not quite. VDI is native to VirtualBox; VMDK is native to VMware; VHD is the older Hyper-V format (max 2 TB). VHDX is the modern Hyper-V format supporting up to 64 TB. Review the disk format tabs in Section 5.

4. What happens when you delete a snapshot in VirtualBox?

✅ Correct! Deleting a snapshot merges the differencing disk into the parent. This consolidates changes — it does not undo them. To undo changes, you would restore the snapshot instead.
❌ Not quite. Deleting a snapshot does not revert the VM (that's restore) and does not lose data. It merges the differencing disk into the parent, preserving all changes made since the snapshot. Review the Snapshot Lifecycle table in Section 6.

5. A classmate needs an independent copy of your VM to run on their own laptop. Which approach is best?

✅ Correct! A full clone or OVA export creates a completely independent VM package that can be transferred and imported on another machine. Linked clones depend on the original, and snapshot files alone don't contain the base disk.
❌ Not quite. Snapshots depend on the base disk (A is incomplete). Linked clones depend on the original VM (B breaks if moved). Sharing just the VDI (D) lacks the VM configuration. A full clone or OVA export (C) is the correct approach for independent portability. Review Section 6.

6. On a Linux guest in VirtualBox, shared folders are not visible after installing Guest Additions. What is the most likely fix?

✅ Correct! VirtualBox shared folders are mounted with ownership by the vboxsf group. If your user isn't in that group, you can't access them. Run sudo usermod -aG vboxsf $USER and log out/in. Review the troubleshooting table in Section 7.
❌ Not quite. Shared folder access on Linux requires the user to be a member of the vboxsf group. This is a permissions issue, not a resource or disk format issue. The fix: sudo usermod -aG vboxsf $USER then log out and back in. Review Section 7's troubleshooting table.
📋 Module 13.2 Summary
  • VM creation follows a six-step workflow: Define Specs → Create Shell → Attach Media → Install OS → Guest Tools → Configure & Harden.
  • Size VMs appropriately using the reference table — balance guest needs against host resources.
  • Dynamically allocated disks save host space; fixed-size disks offer better performance.
  • VDI (VirtualBox), VMDK (VMware), and VHD/VHDX (Hyper-V) are the major disk formats — understand portability implications.
  • Snapshots are temporary rollback points; clones create independent (full) or dependent (linked) VM copies.
  • OVA/OVF is the standard format for exporting and sharing VMs across platforms.
  • Guest Additions / Integration Services should be installed immediately after the OS for the best VM experience.

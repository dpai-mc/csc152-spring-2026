Virtual Network Adapters

Every physical server has one or more Network Interface Cards (NICs). In a virtual environment, the hypervisor emulates NICs in softwareвҖ”these are called virtual NICs (vNICs). From the guest OS's perspective, a vNIC looks and behaves exactly like a physical NIC.

Virtual NIC Types

Hypervisors typically offer several emulated NIC models. The choice affects compatibility and performance:

Virtual NIC types comparison
NIC TypeEmulatesPerformanceBest For
Intel PRO/1000 (e1000)Intel Gigabit NICGood вҖ“ widely supportedGeneral-purpose; broad guest OS compatibility
PCnet-FAST III (Am79C973)AMD PCnet 10/100ModerateLegacy OSes (older Linux, DOS)
Paravirtualized (virtio-net)None вҖ“ guest-aware driverBest вҖ“ near-nativeLinux guests with virtio drivers; KVM/VirtualBox
Hyper-V Network AdapterSynthetic (VMBus)Best вҖ“ near-nativeWindows/Linux guests on Hyper-V with Integration Services
рҹ’Ў Performance Tip

Paravirtualized adapters skip hardware emulation entirely. The guest OS "knows" it's running inside a hypervisor and uses optimized drivers to communicate directly with the hypervisor's networking stack. Always prefer paravirtualized NICs when your guest OS supports them.

MAC Address Assignment

Just like physical NICs, every vNIC has a MAC address. Hypervisors auto-generate MAC addresses using a vendor-specific OUI prefix:

  • VirtualBox: 08:00:27:xx:xx:xx
  • Hyper-V: 00:15:5D:xx:xx:xx
  • VMware: 00:0C:29:xx:xx:xx or 00:50:56:xx:xx:xx

You can manually set a MAC address in the VM's network adapter settings. This is useful when:

  • A DHCP server reserves IPs by MAC address.
  • Software licensing is tied to a specific MAC.
  • You clone a VM and need to avoid MAC conflicts on the same network segment.
рҹ”— Connection to Week 8

In Week 8 you configured network interfaces using ip addr on Linux and Get-NetAdapter in PowerShell. Those same commands work inside a VMвҖ”the guest OS sees the vNIC as a normal interface. The MAC addresses you see with ip link show will match the hypervisor's assignment.

Adding Multiple NICs

VMs can have multiple vNICs, each attached to a different virtual network. This is how you build multi-homed VMsвҖ”for example, a firewall VM with one NIC on an external network and another on an internal network. VirtualBox supports up to 8 network adapters per VM; Hyper-V supports up to 12 synthetic adapters.

NAT Mode

NAT (Network Address Translation) is the default network mode in VirtualBox and the simplest way to give a VM internet access. The hypervisor acts as a NAT router between the VM and the host's physical network.

Figure 2-1: NAT mode вҖ“ the VM's traffic is translated through the host's IP address

Key behaviors:

  • The VM receives an IP (typically 10.0.2.15) from VirtualBox's built-in DHCP server.
  • Outbound traffic from the VM is translated to the host's IPвҖ”external devices see only the host.
  • Inbound connections from the external network cannot reach the VM unless you configure port forwarding.
  • Each VM gets its own isolated NAT networkвҖ”two VMs in NAT mode cannot see each other by default.
рҹ”— Connection to Week 9

NAT in a hypervisor works on the same principle as the NAT you studied on physical routers in Week 9. The hypervisor's NAT engine maintains a translation table mapping VM source ports to host ports, just like a home router does for your LAN devices.

  • Quick testing where the VM only needs outbound internet access (e.g., downloading packages).
  • Situations where you do not want the VM exposed on the physical network.
  • Single-VM setups where inter-VM communication is not needed.

Bridged Mode

In Bridged mode, the hypervisor uses a net filter driver to connect the VM's vNIC directly to the host's physical NIC. The VM appears as a separate device on the physical networkвҖ”it gets its own IP from the network's DHCP server (or you assign one statically).

Because a bridged VM is fully exposed on the physical network, you must apply the same firewall and hardening practices you would to a physical server. This connects directly to the firewall rules you configured in Week 9 and the hardening you'll perform in Week 14.

When to Use Bridged Mode

  • The VM runs a server that other physical or virtual machines must reach (web server, Splunk indexer, DNS).
  • You need the VM to participate in Active Directory or DHCP/DNS on the physical network.
  • Testing network services in a realistic environment that mirrors production.

Internal & Host-Only Modes

Both of these modes create isolated networks, but they differ in whether the host can participate.

Host-Only Mode

Virtual Switches

A virtual switch (vSwitch) is a software-based Layer 2 switch that connects vNICs within a hypervisorвҖ”exactly as a physical switch connects physical NICs. Every network mode you've learned about is implemented through virtual switches, even if the hypervisor abstracts that detail away.

VirtualBox Internal Networking as a vSwitch

When you assign multiple VMs to the same Internal Network name (e.g., intnet-lab), VirtualBox creates an implicit virtual switch connecting those VMs. Key points:

  • The switch is created automatically when the first VM with that network name starts.
  • It operates at Layer 2вҖ”frames are switched by MAC address, just like a physical switch.
  • There is no configuration interface for the switch itself (no VLANs, no STP). It's a simple flat switch.
  • The Host-Only adapter (vboxnet0) is also a vSwitchвҖ”but one where the host has a port.

Hyper-V provides more explicit control over virtual switches. You create and manage them through Hyper-V Manager вҶ’ Virtual Switch Manager or PowerShell.

Virtual Disk Management

Just as virtual NICs replace physical NICs, virtual disks replace physical hard drives. A virtual disk is a file on the host filesystem that the hypervisor presents to the guest OS as a block device. From the guest's perspective, it looks like a real hard drive.

Virtual disk management is the virtualized equivalent of DASвҖ”the virtual disk file is "directly attached" to the VM. Later in Section 9, you'll see how NAS/SAN concepts extend into the virtual world with shared storage.

Adding a Virtual Disk

Adding a new disk to an existing VM is a common taskвҖ”for example, when a server needs more storage for logs or databases. The process:

  1. Power off the VM (or hot-add if the hypervisor and guest OS support it).
  2. Open VM Settings вҶ’ Storage.
  3. Select a storage controller (SATA, SCSI, etc.) and click Add Hard Disk.
  4. Choose Create New Disk вҶ’ select format (VDI) вҶ’ choose dynamic or fixed вҶ’ set size.
  5. Start the VM.
  6. Inside the guest OS, the new disk appears as an unformatted block device (e.g., /dev/sdb on Linux or Disk 1 in Windows Disk Management).
  7. Partition, format, and mount the disk using the same commands from Weeks 4вҖ“5 and Week 10.

Resizing a Virtual Disk

If a VM's disk runs low on space, you can resize it without recreating the VM:

  1. VirtualBox CLI: VBoxManage modifyhd "disk.vdi" --resize 51200 (size in MB)
  2. Hyper-V PowerShell: Resize-VHD -Path "disk.vhdx" -SizeBytes 50GB
  3. After resizing the virtual disk file, you must extend the partition and filesystem inside the guest OS:
    • Linux: Use growpart + resize2fs (ext4) or xfs_growfs (XFS). If using LVM, use pvresize вҶ’ lvextend вҶ’ resize2fs.
    • Windows: Use Disk Management вҶ’ Extend Volume, or Resize-Partition in PowerShell.
вҡ пёҸ Important

Resizing the virtual disk file only changes the container. The guest OS doesn't automatically use the new spaceвҖ”you must extend the partition and filesystem inside the VM. This is the same two-step process (physical expansion + logical expansion) you learned with LVM in Week 10.

Performance Considerations

Dynamically allocated disks start small and grow as data is written. They save host disk space but have a slight write penalty because the hypervisor must allocate new blocks on the fly.

Fixed-size disks allocate the full space upfront. They offer better I/O performance (no allocation overhead) but consume the full disk space on the host immediately.

Recommendation: Use dynamic for development/testing; use fixed for production workloads or I/O-intensive VMs (databases).

The performance of virtual disks is ultimately limited by the host's physical storage. A virtual disk stored on a host SSD will dramatically outperform one stored on a spinning HDD. If you run multiple VMs, consider:

  • Placing virtual disk files on an SSD for frequently accessed VMs.
  • Separating VM disk files across different physical drives to reduce I/O contention.
  • Avoiding placing all VM disks on the same drive as the host OS.

Snapshots create differencing disksвҖ”new writes go to the snapshot file while the base disk becomes read-only. Each snapshot adds a layer that the hypervisor must traverse for reads. Too many snapshots degrade performance significantly.

Best practice: Use snapshots for short-term rollback (before risky changes). Delete snapshots you no longer need. Never use snapshots as a long-term backup strategy.

Shared Storage for VMs

iSCSI (Internet Small Computer Systems Interface) transmits SCSI commands over TCP/IP networks. In a virtual environment:

  • An iSCSI target (storage server) exports block-level storage over the network.
  • A VM acts as an iSCSI initiator, connecting to the target and mounting the storage as a local block device.
  • This is the virtualized equivalent of a SANвҖ”the VM gets block-level access to remote storage.

Use case: When multiple VMs need access to a shared storage pool, or when storage must be centralized for management and backup.

рҹ”— Connection to Week 10

iSCSI is the SAN technology you studied in Week 10. The difference here is that both the initiator (VM) and potentially the target (another VM or physical NAS) operate within the virtualized infrastructure.

NFS (Network File System) provides file-level shared storage over the network. In virtualization:

  • An NFS server exports directories that VMs can mount as network drives.
  • Unlike iSCSI (block-level), NFS operates at the file levelвҖ”multiple VMs can mount the same NFS share simultaneously for read/write.
  • Hypervisors like VMware ESXi can use NFS datastores to store VM disk files centrally.

Use case: Shared file repositories, centralized VM storage for hypervisor clusters, home directories.

рҹ”— Connection to Week 10

NFS is the NAS protocol you learned about in Week 10. Here it serves double duty: VMs can use NFS for shared file access, and the hypervisor itself can use NFS to store virtual disk files on centralized storage.

Live migration is the ability to move a running VM from one physical host to another with zero downtime. This is a cornerstone of enterprise virtualization and requires shared storage.

How it works:

  1. Both hosts connect to the same shared storage (iSCSI SAN or NFS datastore).
  2. The VM's disk files live on shared storageвҖ”accessible from either host.
  3. During migration, the VM's memory state is transferred from Host A to Host B over the network.
  4. Once the memory is synchronized, the VM switches to Host B seamlessly.

Technologies: Hyper-V Live Migration, VMware vMotion, KVM/libvirt live migration.

Why it matters: Live migration enables hardware maintenance without downtime, load balancing across hosts, and disaster avoidance. It's a key reason enterprises invest in shared storage infrastructure.

Module Summary

In this module you learned how virtualization extends into networking and storage:

  • Virtual NICs emulate physical network adapters; paravirtualized drivers offer the best performance.
  • Four network modesвҖ”NAT, Bridged, Host-Only, and InternalвҖ”provide different levels of connectivity and isolation.
  • Virtual switches connect VMs at Layer 2, with Hyper-V offering explicit External, Internal, and Private switch types.
  • Multi-NIC designs allow VMs to span network tiers, mirroring real-world server architecture.
  • Virtual disks can be added, resized, and tuned for performance; the guest OS must extend partitions after a resize.
  • Storage controllers (IDE, SATA, SCSI, NVMe) affect performance and device limits.
  • Shared storage (iSCSI, NFS) enables advanced features like live migration and maps directly to the SAN/NAS concepts from Week 10.

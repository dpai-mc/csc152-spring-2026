Introduction — RAID Goals & Hardware vs. Software

▼

RAID stands for Redundant Array of Independent Disks. It is a technology that combines two or more physical disks into a single logical unit to achieve one or more of three goals: improved performance, improved fault tolerance (redundancy), or increased capacity. No single RAID level maximizes all three simultaneously — every design involves trade-offs.

📍 Where We Are in Week 10
This is the third of five modules in Week 10. You should have completed Module 10.1 (Storage Architecture) and Module 10.2 (Disk Partitioning) before starting here. The partitioning skills from Module 10.2 — particularly setting the Linux RAID partition type (fd) with fdisk — are a direct prerequisite for the mdadm workflow in Section 4 of this module.

The Three RAID Goals

Goal What It Means RAID Levels That Prioritize It
Performance Reads and/or writes are distributed across multiple disks (striping), increasing throughput and IOPS RAID 0, RAID 10 (read/write), RAID 5 (read)
Fault Tolerance The array continues operating and data remains accessible even if one or more drives fail RAID 1, RAID 5, RAID 10
Capacity All raw disk space is available for data — no capacity is "lost" to redundancy overhead RAID 0 only (no redundancy overhead)

Hardware RAID vs. Software RAID vs. Firmware RAID

RAID can be implemented at three different layers of the system stack. Understanding the differences matters for both design decisions and troubleshooting.

🔩 Hardware RAID

A dedicated RAID controller card (or built-in controller on enterprise motherboards) manages all RAID operations in hardware. The OS sees a single logical disk — it has no knowledge of the underlying RAID structure.

  • Pros: Best performance (dedicated processor and cache); OS-independent; battery-backed write cache for power-loss protection
  • Cons: Expensive; if the controller fails, you need an identical replacement to read the array; vendor lock-in
  • Typical use: Production servers, enterprise environments

💻 Software RAID

The operating system kernel manages RAID operations using standard disk I/O. Linux mdadm and Windows Storage Spaces are both software RAID implementations.

  • Pros: No additional hardware cost; portable (array metadata travels with the disks); flexible; open-source (mdadm)
  • Cons: Uses CPU cycles for parity calculations (RAID 5/6); performance ceiling lower than hardware RAID
  • Typical use: Lab environments, small servers, cost-sensitive deployments — including your VMs in this course
📌 Firmware RAID (a.k.a. "Fake RAID")
A third category — firmware RAID or "host-based RAID" — is implemented in the motherboard's storage controller firmware (common on consumer motherboards as Intel RST). It uses the CPU for processing (like software RAID) but presents itself as hardware RAID to the OS. It is generally considered the worst of both worlds for server use and is not covered in depth in this course.

RAID 0 — Striping (Performance Only)

How it works: Data is split into chunks (stripes) and written across all disks simultaneously. Disk 1 gets stripe A1, Disk 2 gets A2, and so on. Both reads and writes benefit because multiple disks work in parallel. There is zero redundancy — if any single disk fails, all data in the array is lost.

  • Video editing scratch disks where speed matters and data is expendable
  • Gaming storage (non-critical data)
  • Temporary processing workloads where data can be regenerated
⚠ Never Use RAID 0 for Production Data
With N disks in RAID 0, the probability of array failure equals 1 − (1 − Pfail)N where Pfail is the individual disk failure probability. More disks = higher array failure risk. RAID 0 is a performance tool, not a storage solution for data you care about.

RAID 1 — Mirroring (Redundancy Focus)

How it works: Every write is duplicated identically to all disks in the mirror set. With two disks, Disk 2 is an exact, real-time copy of Disk 1. If one disk fails, the other contains a complete, current copy of all data. Reads can be served from either disk (some controllers split reads for a performance benefit).

RAID 5 — Striping with Distributed Parity

How it works: Data is striped across all disks (like RAID 0) but one stripe's worth of space per stripe set is reserved for parity — an XOR-calculated value that allows any single missing disk's data to be reconstructed. Critically, parity is distributed across all disks (not on a dedicated parity disk), so there is no single bottleneck disk. If one disk fails, the array continues operating in a degraded state; the failed disk's data is reconstructed from parity when a replacement is added.

📌 The RAID 5 Write Penalty
Every write to a RAID 5 array requires 4 disk operations: read old data, read old parity, write new data, write new parity. This "write penalty" makes RAID 5 a poor choice for write-intensive workloads like busy transaction databases. It excels in read-heavy environments.

RAID 10 — Striped Mirror (Performance + Redundancy)

How it works: RAID 10 (also written RAID 1+0) combines mirroring and striping. Drives are first organized into mirror pairs (RAID 1), then data is striped across those pairs (RAID 0). This gives you both the redundancy of mirroring and the performance of striping. It is the most expensive RAID level in terms of disk overhead but delivers the best combination of performance and fault tolerance.

💡 RAID 10 vs. RAID 5: The Classic Trade-off
With 4 × 1 TB drives: RAID 10 gives you 2 TB usable with excellent write performance. RAID 5 gives you 3 TB usable but with a write penalty. The choice comes down to: do you need capacity efficiency (RAID 5) or write performance and simpler fault tolerance (RAID 10)?

RAID Level Quick-Reference Summary

Level Min Disks Usable Capacity Fault Tolerance Read Perf. Write Perf. Best For
RAID 0 2 N × size None Excellent Excellent Scratch / temp / non-critical speed
RAID 1 2 1 × size 1 disk Good Moderate OS drives, small critical datasets
RAID 5 3 (N−1) × size 1 disk Good Moderate (write penalty) File servers, NAS, read-heavy
RAID 10 4 (N/2) × size ≥1 disk (cross-pair) Excellent Excellent Databases, VMs, write-intensive

RAID Capacity Math — Formulas & Worked Examples

▼

RAID capacity calculations appear on Quiz 10. Memorize the formulas, then practice with the worked examples and the interactive calculator below.

Formulas

RAID Level Formula Variables
RAID 0 Usable = N × S N = number of disks
S = size of one disk
(all disks must be the same size; if mixed, use the smallest)
RAID 1 Usable = S  (only 1 copy is usable data)
RAID 5 Usable = (N − 1) × S
RAID 10 Usable = (N / 2) × S

Worked Example 1 — Four 1 TB Drives

Given: 4 drives × 1 TB each. Calculate usable capacity for each RAID level.

RAID 0

1N = 4, S = 1 TB
2Usable = N × S = 4 × 1 TB
Usable = 4 TB  (0% overhead)

RAID 1

1N = 4, S = 1 TB
2With 4-disk RAID 1, all 4 drives are mirrors of each other. Usable = S = 1 TB
Usable = 1 TB  (75% overhead — rarely used with 4 disks; 2-disk mirror is standard)

RAID 5

1N = 4, S = 1 TB
2Usable = (N − 1) × S = (4 − 1) × 1 TB = 3 × 1 TB
Usable = 3 TB  (1 TB overhead for parity)

RAID 10

1N = 4, S = 1 TB
2Usable = (N / 2) × S = (4 / 2) × 1 TB = 2 × 1 TB
Usable = 2 TB  (2 TB overhead for mirroring)

Worked Example 2 — Six 2 TB Drives

Given: 6 drives × 2 TB each. Calculate usable capacity for each RAID level.

RAID 0

1N = 6, S = 2 TB
2Usable = 6 × 2 TB
Usable = 12 TB

RAID 5

1N = 6, S = 2 TB
2Usable = (6 − 1) × 2 TB = 5 × 2 TB
Usable = 10 TB  (2 TB overhead — only 1 disk equivalent regardless of array size)

RAID 10

1N = 6, S = 2 TB
2Usable = (6 / 2) × 2 TB = 3 × 2 TB
Usable = 6 TB  (6 TB overhead — 50% always)

Software RAID on Linux — mdadm Workflow

▼

mdadm (Multiple Device Admin) is the standard Linux utility for creating and managing software RAID arrays. It is included in virtually every Linux distribution. The workflow below walks through creating a RAID 1 array — the same task required in Lab 10 Part B.

🔗 Lab 10 Connection
The commands in this section map directly to Lab 10 Part B deliverables. If you are pursuing the optional extra credit, follow this workflow on your Linux VM after completing the disk partitioning steps from Module 10.2. You will need two partitions with type set to Linux RAID (fd) before starting.

Step 1 — Verify Disks and Partitions

Before creating the array, confirm both partitions are visible and have the correct type.

bash — verify partitions lsblk # list all block devices and partitions NAME MAJ:MIN RM SIZE RO TYPE MOUNTPOINT sdb 8:16 0 10G 0 disk └─sdb1 8:17 0 10G 0 part sdc 8:32 0 10G 0 disk └─sdc1 8:33 0 10G 0 part fdisk -l /dev/sdb # confirm partition type is fd (Linux RAID)

Step 2 — Create the RAID 1 Array

The mdadm --create command assembles the array. Each flag is explained inline.

bash — create RAID 1 array mdadm --create /dev/md0 \ --level=1 \ # RAID level: 1 (mirror) --raid-devices=2 \ # number of active member devices /dev/sdb1 /dev/sdc1 # the two member partitions mdadm: Defaulting to version 1.2 metadata mdadm: array /dev/md0 started.

Step 3 — Monitor Array Creation

RAID 1 performs an initial synchronization (resync) to ensure both mirrors are identical. Monitor progress with:

bash — monitor sync progress cat /proc/mdstat Personalities : [raid1] md0 : active raid1 sdc1[1] sdb1[0] 10475520 blocks super 1.2 [2/2] [UU] [==========>.........] resync = 52.4% (5492736/10475520) finish=0.4min speed=183424K/sec unused devices: <none> mdadm --detail /dev/md0 # detailed array status

In the [UU] status indicator, each U represents a healthy member disk. A failed disk shows as _.

Step 4 — Format and Mount the Array

bash — format and mount mkfs.ext4 /dev/md0 # create ext4 filesystem on the array mkdir -p /mnt/raid1 # create mount point mount /dev/md0 /mnt/raid1 # mount the array df -h /mnt/raid1 # verify mount and available space Filesystem Size Used Avail Use% Mounted on /dev/md0 9.8G 24M 9.3G 1% /mnt/raid1

Step 5 — Make the Mount Persistent (/etc/fstab)

Without an fstab entry, the mount will not survive a reboot. Use the array's UUID (not device name, which can change) for reliability.

bash — get UUID and add to fstab blkid /dev/md0 # get the UUID /dev/md0: UUID="a1b2c3d4-e5f6-7890-abcd-ef1234567890" TYPE="ext4" # Add this line to /etc/fstab (replace UUID with your actual value): UUID=a1b2c3d4-e5f6-7890-abcd-ef1234567890 /mnt/raid1 ext4 defaults 0 2

Step 6 — Save the RAID Configuration

bash — save mdadm configuration mdadm --detail --scan >> /etc/mdadm/mdadm.conf # This ensures the array is reassembled automatically on reboot

Step 7 — Simulate a Disk Failure

You can mark a member disk as faulty without physically removing it. This is how Lab 10 asks you to test fault tolerance.

bash — simulate failure and observe degraded state mdadm --manage /dev/md0 --fail /dev/sdb1 mdadm: set /dev/sdb1 faulty in /dev/md0 cat /proc/mdstat md0 : active raid1 sdc1[1] sdb1[0](F) 10475520 blocks super 1.2 [2/1] [_U] # [_U] = one disk failed, one healthy; array still running (degraded) mdadm --detail /dev/md0 # confirm degraded state

Step 8 — Remove and Re-Add Disk to Rebuild

bash — remove failed disk and rebuild mdadm --manage /dev/md0 --remove /dev/sdb1 mdadm: hot removed /dev/sdb1 from /dev/md0 mdadm --manage /dev/md0 --add /dev/sdb1 mdadm: added /dev/sdb1 cat /proc/mdstat md0 : active raid1 sdb1[2] sdc1[1] 10475520 blocks super 1.2 [2/1] [_U] [===========>.........] recovery = 57.3% (6002688/10475520) finish=0.3min speed=195432K/sec # Array is rebuilding; [UU] will appear when complete
✅ Key Takeaway
The RAID 1 array remained accessible throughout the failure and rebuild process. This is fault tolerance in action — the data was never unavailable even with one disk completely failed. This is the core value proposition of redundant RAID levels.

Windows Storage Spaces — GUI & PowerShell

▼

Storage Spaces is Windows Server's software-defined storage feature. It allows you to group physical disks into a storage pool, then carve that pool into one or more virtual disks (also called spaces) with configurable resiliency. From the OS perspective, a virtual disk looks and behaves like a regular disk drive.

Key Concepts

Concept Definition Analogy
Storage Pool A collection of physical disks grouped together as a single capacity reservoir The raw ingredients (all your disks combined)
Virtual Disk (Space) A logical disk created from pool capacity with a chosen resiliency type A dish made from those ingredients — you choose the recipe (resiliency)
Resiliency Type The fault tolerance model applied to the virtual disk (Simple, Mirror, Parity) The recipe's cooking method
Volume A formatted partition on the virtual disk, assigned a drive letter The final plated dish, ready to serve

Resiliency Types Mapped to RAID Equivalents

Resiliency Type RAID Equivalent Min Disks Fault Tolerance Use Case
Simple RAID 0 1 None Maximum capacity/performance; non-critical data
Mirror (2-way) RAID 1 2 1 disk failure OS volumes, critical data with moderate capacity
Mirror (3-way) RAID 1 (3 copies) 5 2 disk failures Highest redundancy requirements
Parity RAID 5 3 1 disk failure File servers, archival, capacity-efficient redundancy
Dual Parity RAID 6 7 2 disk failures Large arrays where simultaneous failures are a concern

GUI Workflow — Server Manager

The GUI approach uses Server Manager and is the most accessible starting point.

  1. Open Server Manager → click File and Storage Services in the left panel.
  2. Click Storage Pools → in the Storage Pools pane, click TASKS → New Storage Pool…
  3. Follow the wizard: name the pool, select the physical disks to include, and click Create.
  4. After the pool is created, click TASKS → New Virtual Disk…
  5. Select the pool, name the virtual disk, choose the resiliency type, configure size, and click Create.
  6. The New Volume Wizard launches automatically. Format as NTFS or ReFS, assign a drive letter, and click Create.

PowerShell Workflow

PowerShell is the preferred method for scripted or automated deployments. The commands below create a 2-way mirror virtual disk from two disks.

powershell — create storage pool and virtual disk # Step 1: View available physical disks (not yet in a pool) Get-PhysicalDisk -CanPool $true # Step 2: Create a storage pool named "DataPool" from two disks $disks = Get-PhysicalDisk -CanPool $true New-StoragePool -FriendlyName "DataPool" \ -StorageSubsystemFriendlyName "Windows Storage*" \ -PhysicalDisks $disks # Step 3: Create a 2-way mirror virtual disk (RAID 1 equivalent) New-VirtualDisk -StoragePoolFriendlyName "DataPool" \ -FriendlyName "MirrorDisk" \ -ResiliencySettingName "Mirror" \ -NumberOfDataCopies 2 \ -UseMaximumSize # Step 4: Initialize the disk, create a partition, and format it Get-VirtualDisk -FriendlyName "MirrorDisk" | Get-Disk | Initialize-Disk -PassThru | New-Partition -AssignDriveLetter -UseMaximumSize | Format-Volume -FileSystem NTFS -NewFileSystemLabel "MirrorVol" -Confirm:$false
📌 ReFS vs. NTFS with Storage Spaces
Windows Server supports formatting Storage Spaces volumes with either NTFS or ReFS (Resilient File System). ReFS is designed to work with Storage Spaces and provides additional data integrity features (checksumming, automatic corruption repair). For new Storage Spaces deployments on Windows Server 2016 and later, ReFS is generally preferred for data volumes.

RAID Is Not a Backup

▼
⚠ Critical Concept — Read This Section Carefully
This is one of the most important and most commonly misunderstood distinctions in storage administration. Confusing RAID with backup has caused real data loss at real organizations. Do not skip this section.

RAID provides availability — the ability to keep operating when a disk fails. It does not provide protection against the scenarios that actually cause most data loss events.

What RAID Protects Against

  • A single physical disk failing (hardware failure) — with RAID 1, 5, or 10
  • Service interruption due to disk failure — the array keeps running
  • The need to immediately replace a failed disk before data is accessible

What RAID Does NOT Protect Against

  • Accidental deletion — if you delete a file, it is deleted on all mirrors instantly
  • Ransomware / malware — if malware encrypts your files, all mirrors are encrypted
  • Logical corruption — a corrupted database or filesystem is mirrored identically
  • Controller failure — if the RAID controller fails, the array may be inaccessible
  • Catastrophic events — fire, flood, theft, or power surge can destroy all disks simultaneously
  • Human error — misconfigured RAID rebuild, wrong disk removed, accidental format
  • Multiple simultaneous disk failures — RAID 5 cannot survive two simultaneous failures

The 3-2-1 Backup Rule

RAID is one component of a storage strategy, not a complete strategy. The industry-standard framework is the 3-2-1 rule:

📋 3-2-1 Backup Rule
  • 3 copies of your data
  • 2 different storage media types (e.g., disk + tape, or disk + cloud)
  • 1 copy stored offsite (physically or in the cloud)

RAID may satisfy the "2 copies on disk" portion of this rule, but it does not satisfy the offsite requirement. A complete data protection strategy requires both RAID and regular backups.

Mini-Case — RAID Level Selection for a Small Business

Work through this scenario step by step. Calculate usable capacity, compare the results, and make a recommendation. Click Reveal Answer to check your work after completing each step. This activity is ungraded and directly previews the storage design decisions in Project 4.

📋 Scenario: Riverside Accounting Partners

Riverside Accounting Partners is a small firm with 12 employees. They need a new file server to store client financial documents, tax records, and internal reports. Their requirements:

  • Shared file access for all 12 workstations (Windows and one Mac)
  • Fault tolerance — they cannot afford to lose client data
  • Budget allows for exactly four 2 TB drives
  • The workload is primarily read-heavy (staff retrieves files more than they write)
  • They want to maximize usable storage capacity while maintaining redundancy

Step 1 — Calculate Usable Capacity for RAID 5

Using the formula Usable = (N − 1) × S, calculate the usable capacity of a RAID 5 array built from four 2 TB drives.

Work this out on paper before revealing the answer.

✅ Step 1 Answer

N = 4, S = 2 TB

Usable = (4 − 1) × 2 TB = 3 × 2 TB = 6 TB usable

Overhead = 1 disk equivalent = 2 TB (used for distributed parity)

Step 2 — Calculate Usable Capacity for RAID 10

Using the formula Usable = (N / 2) × S, calculate the usable capacity of a RAID 10 array built from the same four 2 TB drives.

✅ Step 2 Answer

N = 4, S = 2 TB

Usable = (4 / 2) × 2 TB = 2 × 2 TB = 4 TB usable

Overhead = 50% = 4 TB (used for mirroring)

Step 4 — Make Your Recommendation

Based on the scenario requirements (read-heavy workload, fault tolerance required, maximize capacity, four 2 TB drives), which RAID level would you recommend for Riverside Accounting Partners? Write a brief justification before revealing the suggested answer.

✅ Suggested Recommendation: RAID 5

Recommended RAID level: RAID 5

Justification: RAID 5 is the better fit for this scenario for three reasons:

  1. Capacity: RAID 5 provides 6 TB usable vs. RAID 10's 4 TB — a 50% increase in usable storage with the same four drives. For a document storage workload that grows over time, this matters.
  2. Workload match: The workload is described as read-heavy. RAID 5's write penalty is a significant concern only for write-intensive workloads. For file retrieval (reads), RAID 5 performs well.
  3. Fault tolerance is sufficient: RAID 5 tolerates one disk failure, which is adequate for a 4-disk array in a small business context. The firm should pair this with regular offsite backups for complete data protection.

Caveat: If the firm's write volume grows significantly, or if they later add more drives and want better rebuild performance, RAID 10 becomes a stronger candidate. A well-reasoned argument for RAID 10 (prioritizing write performance and simpler fault tolerance) is also valid — what matters is that the recommendation is justified by the requirements.

Introduction — Why LVM?

▼

Traditional disk partitioning is rigid. When you create a partition at install time and later discover it is too small, your options are painful: unmount the filesystem, boot from rescue media, resize the partition, hope nothing goes wrong, and then bring the service back up. On a production server running a database or a web application, that downtime is unacceptable.

Logical Volume Management (LVM) solves this problem by inserting an abstraction layer between physical disks and the filesystems that live on them. Instead of filesystems sitting directly on partitions, they sit on logical volumes — flexible, software-defined storage containers that can be grown, shrunk, snapshotted, and moved across physical disks while services remain online.

💡 The Core Promise of LVM
With LVM you can add a new disk to a running server and extend a mounted filesystem in minutes — without unmounting it, without rebooting, and without interrupting any running service. This is not a special case; it is routine production administration.

What LVM Gives You That Partitions Cannot

  • Online resize: Grow (and in many cases shrink) filesystems while they are mounted and in use.
  • Span multiple disks: A single logical volume can draw storage from multiple physical disks transparently.
  • Snapshots: Capture a point-in-time consistent copy of a volume for backup or testing without stopping the application.
  • Thin provisioning: Allocate more logical space than physically exists, consuming real space only as data is written.
  • Flexible naming: Name volumes by purpose (/dev/vg_data/lv_mysql) rather than cryptic device numbers (/dev/sdb3).

Where LVM Fits in Week 10

You have already studied storage architectures (Module 10.1), partitioning (Module 10.2), and RAID (Module 10.3). LVM builds on all three: LVM physical volumes are typically created on top of partitions or RAID arrays, and the logical volumes LVM creates are where you actually build filesystems. LVM Architecture — Three Layers

▼

LVM organizes storage into three distinct layers. Each layer has a specific role and a specific set of commands. Understanding this hierarchy is the key to understanding every LVM operation.

Layered Architecture Diagram

Layer 1 — Physical Volumes (PVs)

A Physical Volume is a real storage device — a whole disk (/dev/sdb), a disk partition (/dev/sdb1), or a RAID array (/dev/md0) — that has been initialized for use by LVM. Initialization writes LVM metadata to the device, divides it into fixed-size Physical Extents (PEs) (default 4 MB each), and registers it as available storage.

  • PVs are the raw material — the actual physical storage.
  • A PV can only belong to one Volume Group at a time.
  • You can add PVs to a VG at any time, including while the VG is in use.

Layer 2 — Volume Groups (VGs)

A Volume Group aggregates one or more Physical Volumes into a single named storage pool. All the Physical Extents from every PV in the group are pooled together. The VG is the unit of administration — you add disks to the VG, and the VG makes that space available to Logical Volumes.

  • Think of the VG as a "virtual disk" assembled from real disks.
  • VG names are administrator-defined (e.g., vg_data, ubuntu-vg).
  • Expanding a VG is as simple as adding another PV with vgextend.

Layer 3 — Logical Volumes (LVs)

A Logical Volume is a slice of the Volume Group's storage pool — the LVM equivalent of a partition. You format an LV with a filesystem (ext4, XFS, etc.) and mount it just like a traditional partition. Unlike a partition, an LV can be resized online, can span multiple physical disks, and can be snapshotted.

  • LV names are administrator-defined (e.g., lv_data, lv_mysql).
  • The full device path is /dev/<vg-name>/<lv-name>, e.g., /dev/vg_data/lv_mysql.
  • LVs draw Logical Extents (LEs) from the VG pool; each LE maps to a PE on a PV.

Physical Extents — The Unit of Allocation

The Physical Extent (PE) is the smallest allocatable unit in LVM. By default, PEs are 4 MB. When you create a 20 GB logical volume, LVM allocates 5,120 PEs (20 GB ÷ 4 MB) from the VG pool and maps them to the LV. When you extend the LV by 2 GB, LVM adds 512 more PEs.

✅ The Mental Model
PV = raw disk → VG = storage pool → LV = usable volume. Data flows: filesystem → LV → VG → PV → physical disk. Administration flows in the opposite direction: you add PVs to grow the VG, then extend LVs to use the new space.

Quick-Reference: Layer Summary

Layer Object Example Name Analogous To Key Property
1 — Bottom Physical Volume (PV) /dev/sdb A physical disk Initialized with pvcreate; divided into 4 MB PEs
2 — Middle Volume Group (VG) vg_data A virtual disk pool Aggregates PVs; named by admin; expandable at runtime
3 — Top Logical Volume (LV) lv_mysql A partition Formatted with a filesystem; mounted; resizable online

Core LVM Commands

▼

LVM commands follow a consistent naming pattern: a two-letter prefix identifies the layer (pv, vg, lv) followed by a verb (create, display, extend, remove). Once you know the pattern, the entire command set becomes predictable. Use the tabs below to explore commands by layer.

⚠ Root Privileges Required
All LVM commands require root (or sudo) privileges. In the examples below, the # prompt indicates a root shell. In your lab environment, prefix each command with sudo.

Physical Volume Commands

pvcreate — Initialize a device as a PV

Writes LVM metadata to the device and marks it as available for use in a Volume Group. The device must not be currently mounted or in use.

# Initialize a single disk as a PV # pvcreate /dev/sdb # Initialize multiple disks at once # pvcreate /dev/sdb /dev/sdc /dev/sdd # Initialize a partition instead of a whole disk # pvcreate /dev/sdb1

pvs — Brief PV listing

Displays a compact, one-line-per-PV summary. Use this for a quick status check.

# pvs PV VG Fmt Attr PSize PFree /dev/sdb vg_data lvm2 a-- 100.00g 0 /dev/sdc vg_data lvm2 a-- 100.00g 30.00g

pvdisplay — Detailed PV information

Shows full metadata for one or all PVs, including PE size, total PEs, and allocated PEs.

# pvdisplay /dev/sdb --- Physical volume --- PV Name /dev/sdb VG Name vg_data PV Size 100.00 GiB Allocatable yes (but full) PE Size 4.00 MiB Total PE 25600 Free PE 0 Allocated PE 25600

Volume Group Commands

vgcreate — Create a Volume Group

Creates a new VG from one or more initialized PVs. The VG name is administrator-defined and should be descriptive.

# Create vg_data from two PVs # vgcreate vg_data /dev/sdb /dev/sdc # Create a VG from a single PV # vgcreate vg_logs /dev/sdd

vgextend — Add a PV to an existing VG

Adds one or more PVs to an existing VG, immediately making their space available to LVs. The VG does not need to be taken offline.

# Add /dev/sde to the existing vg_data (online operation) # vgextend vg_data /dev/sde Volume group "vg_data" successfully extended

vgs — Brief VG listing

# vgs VG #PV #LV #SN Attr VSize VFree vg_data 2 3 0 wz--n- 199.99g 30.00g

vgdisplay — Detailed VG information

# vgdisplay vg_data --- Volume group --- VG Name vg_data System ID Format lvm2 VG Size 199.99 GiB PE Size 4.00 MiB Total PE 51198 Alloc PE / Size 43518 / 170.00 GiB Free PE / Size 7680 / 30.00 GiB

Logical Volume Commands

lvcreate — Create a Logical Volume

Allocates space from a VG and creates a new LV. Use -L for an absolute size or -l for a count of extents or a percentage of the VG.

# Create an 8 GB LV named lv_data in vg_data # lvcreate -L 8G -n lv_data vg_data # Create a 4 GB LV for logs # lvcreate -L 4G -n lv_logs vg_data # Create lv_backup using ALL remaining free space in the VG # lvcreate -l 100%FREE -n lv_backup vg_data # After creating an LV, format it (ext4 example) # mkfs.ext4 /dev/vg_data/lv_data # Mount it # mount /dev/vg_data/lv_data /mnt/lvm/data

lvextend — Grow a Logical Volume

Increases the size of an LV. Use -L +2G to add 2 GB to the current size, or -L 10G to set an absolute new size. After extending the LV, the filesystem must also be resized (see Section 4).

# Extend lv_data by 2 GB (LV must already exist) # lvextend -L +2G /dev/vg_data/lv_data Size of logical volume vg_data/lv_data changed from 8.00 GiB to 10.00 GiB. Logical volume vg_data/lv_data successfully resized. # Extend lv_data to use ALL remaining free space in the VG # lvextend -l +100%FREE /dev/vg_data/lv_data

lvs and lvdisplay — LV status

# lvs LV VG Attr LSize Pool Origin Data% lv_backup vg_data -wi-ao---- 6.00g lv_data vg_data -wi-ao---- 10.00g lv_logs vg_data -wi-ao---- 4.00g # lvdisplay /dev/vg_data/lv_data --- Logical volume --- LV Path /dev/vg_data/lv_data LV Name lv_data VG Name vg_data LV Size 10.00 GiB Current LE 2560 Segments 1 Allocation inherit Read ahead sectors auto

Complete Workflow: Disk to Mounted Volume

The following example shows the full sequence from a bare disk to a mounted, usable filesystem — the exact workflow you will follow in Lab 10 Part C.

# Step 1: Initialize the new disk as a Physical Volume # pvcreate /dev/sdb # Step 2: Create a Volume Group named vg_lab # vgcreate vg_lab /dev/sdb # Step 3: Create three Logical Volumes # lvcreate -L 8G -n lv_data vg_lab # lvcreate -L 4G -n lv_logs vg_lab # lvcreate -l 100%FREE -n lv_backup vg_lab # Step 4: Format each LV with a filesystem # mkfs.ext4 /dev/vg_lab/lv_data # mkfs.ext4 /dev/vg_lab/lv_logs # mkfs.xfs /dev/vg_lab/lv_backup # Step 5: Create mount points and mount # mkdir -p /mnt/lvm/{data,logs,backup} # mount /dev/vg_lab/lv_data /mnt/lvm/data # mount /dev/vg_lab/lv_logs /mnt/lvm/logs # mount /dev/vg_lab/lv_backup /mnt/lvm/backup # Step 6: Verify with pvs, vgs, lvs, and df # pvs && vgs && lvs # df -h /mnt/lvm/*

Online Volume Extension — Growing Without Downtime

▼

Extending a Logical Volume is a two-step process. The first step (lvextend) grows the LVM block device. The second step resizes the filesystem that sits on top of that block device. Both steps can be performed while the filesystem is mounted and in active use — no unmounting, no service interruption.

⚠ Two-Step Process — Both Steps Are Required
lvextend grows the LVM layer. The filesystem does not automatically know about the extra space until you also run resize2fs (ext4) or xfs_growfs (XFS). Skipping the filesystem resize leaves the new space invisible to the OS and any applications.

Step 1 — Extend the Logical Volume

# Add 5 GB to lv_data (filesystem remains mounted throughout) # lvextend -L +5G /dev/vg_data/lv_data Size of logical volume vg_data/lv_data changed from 20.00 GiB to 25.00 GiB. Logical volume vg_data/lv_data successfully resized.

Step 2A — Resize an ext4 Filesystem (resize2fs)

For ext4 filesystems, use resize2fs. When no size is specified, it grows the filesystem to fill the entire LV.

# Grow the ext4 filesystem to fill the newly extended LV # resize2fs /dev/vg_data/lv_data resize2fs 1.46.5 Filesystem at /dev/vg_data/lv_data is mounted at /var/lib/mysql; on-line resizing required old_desc_blocks = 3, new_desc_blocks = 4 The filesystem on /dev/vg_data/lv_data is now 6553600 (4k) blocks long.

Step 2B — Resize an XFS Filesystem (xfs_growfs)

XFS uses a different tool: xfs_growfs. It takes the mount point, not the device path, as its argument. XFS filesystems can only be grown (not shrunk) online.

# Grow the XFS filesystem — use the MOUNT POINT, not the device # xfs_growfs /mnt/lvm/backup meta-data=/dev/vg_data/lv_backup isize=512 agcount=4, agsize=393216 blks = sectsz=512 attr=2, projid32bit=1 data = bsize=4096 blocks=1572864, imaxpct=25 = sunit=0 swidth=0 blks realtime =none extsz=4096 blocks=0, rtextents=0 data blocks changed from 1572864 to 2097152

Shortcut — Extend LV and Resize ext4 in One Command

The -r flag on lvextend calls resize2fs automatically after extending the LV. This works for ext4 and some other filesystem types.

# Extend AND resize filesystem in a single command (ext4 only) # lvextend -L +5G -r /dev/vg_data/lv_data

Before and After: Disk Usage Verification

⚠ Before Extension
$ df -h /var/lib/mysql Filesystem Size Used Avail Use% Mounted on /dev/mapper/vg_data-lv_data 20G 17G 3.0G 85% /var/lib/mysql

Filesystem at 85% capacity — approaching critical threshold. The database will fail to write new data when the disk fills completely.

✓ After Extension (+5 GB)
$ df -h /var/lib/mysql Filesystem Size Used Avail Use% Mounted on /dev/mapper/vg_data-lv_data 25G 17G 8.0G 68% /var/lib/mysql

Filesystem now at 68% — comfortable headroom restored. The database continued running throughout. Zero downtime.

Filesystem Resize Command Reference

Filesystem Resize Command Argument Online? Shrink?
ext4 resize2fs <device> Device path Yes Yes (unmounted)
XFS xfs_growfs <mountpoint> Mount point Yes No — grow only
Btrfs btrfs filesystem resize max <mountpoint> Mount point Yes Yes (online)

Snapshots and Thin Provisioning

▼

Two advanced LVM capabilities — snapshots and thin provisioning — are introduced here at a conceptual level. You will not implement these in Lab 10, but understanding them is important for Quiz 10 and for Project 4 infrastructure planning.

LVM Snapshots

An LVM snapshot is a point-in-time, read-consistent copy of a Logical Volume, created instantly without stopping the application that is writing to the original LV. Snapshots use a copy-on-write (CoW) mechanism: the original data blocks are only copied to the snapshot's reserved space when the original LV writes new data that would overwrite them.

How Copy-on-Write Works

  • At snapshot creation time: no data is copied. The snapshot just records the current state of the LV's block map.
  • When the original LV writes to a block: LVM first copies the old version of that block to the snapshot's reserved area, then allows the write to proceed on the original LV.
  • The snapshot therefore always represents the original LV's state at the moment of snapshot creation.

Creating a Snapshot

# Create a 5 GB snapshot of lv_data named lv_data_snap # The 5 GB is the CoW reserve — size it based on expected write activity # lvcreate -L 5G -s -n lv_data_snap /dev/vg_data/lv_data Logical volume "lv_data_snap" created. # Mount the snapshot read-only for backup # mount -o ro /dev/vg_data/lv_data_snap /mnt/snap # Run your backup from /mnt/snap (consistent, point-in-time data) # Then unmount and remove the snapshot when done # umount /mnt/snap # lvremove /dev/vg_data/lv_data_snap
💡 Why Snapshots Matter for Backup
Backing up a live database directly risks capturing an inconsistent state — a file that was partially written mid-transaction. A snapshot captures a consistent state in an instant, then the backup tool reads from the snapshot while the database continues writing to the live LV. This is how production database backups are performed on LVM systems.

Snapshot Sizing Guidance

  • Size the snapshot's CoW reserve based on how much data the original LV will change during the backup window — not the total LV size.
  • If the CoW reserve fills up, the snapshot becomes invalid and is automatically removed by LVM.
  • Monitor snapshot usage with lvs — the Data% column shows CoW reserve utilization.

Thin Provisioning

Thin provisioning allows you to create Logical Volumes that are larger than the physical storage currently available, with the expectation that actual data usage will grow gradually over time. Real physical space is consumed only when data is actually written.

The Concept

Imagine you have a 100 GB Volume Group and need to provision storage for 10 virtual machines, each expecting a 50 GB disk. Normally, that would require 500 GB — far more than your 100 GB VG. With thin provisioning, you create 10 × 50 GB thin LVs, knowing that in practice each VM might only use 8–12 GB initially. The VG's 100 GB is consumed only as the VMs actually write data.

⚠ Over-Provisioning Risk
Thin provisioning requires careful monitoring. If actual usage exceeds the physical capacity of the thin pool, writes will fail. Administrators must monitor thin pool utilization and add physical storage before the pool is exhausted. Tools like lvs and alerting scripts are essential in thin-provisioned environments.

Thin Provisioning in LVM: The Thin Pool

LVM implements thin provisioning through a thin pool — a special LV that acts as the backing store for multiple thin LVs. The thin pool has a fixed physical size; the thin LVs that draw from it can collectively exceed that size.

# Create a 50 GB thin pool named lv_thinpool in vg_data # lvcreate -L 50G --thinpool lv_thinpool vg_data # Create a thin LV of 100 GB backed by the 50 GB pool # lvcreate -V 100G --thin -n lv_thin_data vg_data/lv_thinpool # Check actual utilization of the thin pool # lvs vg_data LV VG Attr LSize Pool Data% lv_thinpool vg_data twi-aotz-- 50.00g 12.45 lv_thin_data vg_data Vwi-aotz-- 100.00g lv_thinpool 6.23
📌 Thin Provisioning and Project 4
When planning storage for your Project 4 infrastructure, consider whether thin provisioning is appropriate for any of your VM datastores. If you are provisioning multiple VMs that will not all be heavily used simultaneously, thin provisioning can significantly reduce the physical storage required. Windows Parallel — Dynamic Disks and Storage Spaces

▼

Windows provides two technologies that parallel LVM's capabilities: Dynamic Disks (the older approach) and Storage Spaces (the modern, recommended approach introduced in Windows Server 2012). Module 10.5 covers Storage Spaces in depth; this section establishes the conceptual bridge.

Dynamic Disks (Legacy)

Dynamic Disks allow Windows to create volumes that span multiple physical disks — similar to LVM Volume Groups. Dynamic Disks support several volume types:

  • Simple volumes: Single disk, equivalent to a basic partition.
  • Spanned volumes: Span across multiple disks (no redundancy — like LVM linear).
  • Striped volumes: Data striped across disks for performance (like RAID 0).
  • Mirrored volumes: Redundant copy on two disks (like RAID 1).
  • RAID-5 volumes: Striped with parity across three or more disks.
⚠ Dynamic Disks Are Legacy
Microsoft considers Dynamic Disks a legacy technology. New deployments should use Storage Spaces instead. Dynamic Disks are not supported on Windows Server Core or in Storage Spaces configurations.

Storage Spaces (Modern)

Storage Spaces is Microsoft's modern software-defined storage layer, introduced in Windows Server 2012. It is conceptually similar to LVM but uses Windows-native terminology:

🐧 Linux LVM
  • Physical Volume (PV) — initialized disk
  • Volume Group (VG) — storage pool
  • Logical Volume (LV) — usable volume
  • pvcreate, vgcreate, lvcreate
  • Configured via CLI (lvm tools)
🪟 Windows Storage Spaces
  • Physical Disk — added to pool
  • Storage Pool — aggregated disk pool
  • Virtual Disk / Storage Space — usable volume
  • Server Manager GUI or PowerShell (New-StoragePool, etc.)
  • Configured via GUI or PowerShell

Thin Provisioning in Storage Spaces

Storage Spaces supports thin provisioning through the provisioning type setting when creating a virtual disk. Setting provisioning to Thin creates a virtual disk that consumes physical pool space only as data is written — identical in concept to LVM thin provisioning.

# PowerShell: Create a thin-provisioned virtual disk in a storage pool PS> New-VirtualDisk -StoragePoolFriendlyName "LabPool" ` -FriendlyName "ThinData" ` -Size 100GB ` -ProvisioningType Thin ` -ResiliencySettingName Mirror

Feature Comparison Summary

Feature Linux LVM Windows Storage Spaces
Storage Pooling Volume Groups (VGs) Storage Pools
Logical Volumes Logical Volumes (LVs) Virtual Disks / Spaces
Online Resize Yes (lvextend + resize2fs) Yes (GUI or PowerShell)
Snapshots Yes (lvcreate -s) Via VSS (Volume Shadow Copy)
Thin Provisioning Yes (thin pools) Yes (ProvisioningType Thin)

Mini-Case — "The Growing Database"

▼
📝 Ungraded Activity — Think Before You Reveal
Read the scenario and work through the steps in your head (or on paper) before clicking "Reveal Solution." There is no grade for this activity. It is designed to prepare you for the scenario-based questions in Quiz 10 and for Lab 10 Part C.

🗄 The Scenario

You are the Linux systems administrator for a regional e-commerce company. The production MySQL database server runs on a VM with the following storage configuration:

  • Volume Group: vg_prod
  • Logical Volume: lv_mysql — 200 GB, ext4, mounted at /var/lib/mysql
  • Current usage: 170 GB used (85% full)
  • The VG currently has no free space

Your monitoring system has alerted that /var/lib/mysql will reach 95% capacity within 72 hours based on current write rates. A 500 GB spare disk has been installed in the server as /dev/sdc. The database must remain online — the business cannot tolerate any downtime.

Your Task

Describe the exact sequence of LVM commands needed to:

  1. Add the new 500 GB disk to the existing vg_prod Volume Group.
  2. Extend lv_mysql by 300 GB (leaving some VG space for future growth).
  3. Resize the ext4 filesystem to use the new space — without unmounting or stopping MySQL.
  4. Verify that the filesystem has grown successfully.

Step-by-Step Solution

1

Initialize the new disk as a Physical Volume

The 500 GB disk (/dev/sdc) must be initialized before LVM can use it. This writes LVM metadata and divides the disk into Physical Extents.

# pvcreate /dev/sdc Physical volume "/dev/sdc" successfully created.
2

Add the new PV to the existing Volume Group

vgextend adds the 500 GB PV to vg_prod immediately. The VG now has 500 GB of free space. The database continues running — this operation is completely transparent to MySQL.

# vgextend vg_prod /dev/sdc Volume group "vg_prod" successfully extended # Verify: vg_prod should now show ~500 GB free # vgs vg_prod VG #PV #LV Attr VSize VFree vg_prod 2 1 wz--n- 699.99g 499.99g
3

Extend the Logical Volume by 300 GB

lvextend grows the LVM block device. We add 300 GB (leaving ~200 GB free in the VG for future growth). MySQL is still running; this operation does not affect it.

# lvextend -L +300G /dev/vg_prod/lv_mysql Size of logical volume vg_prod/lv_mysql changed from 200.00 GiB to 500.00 GiB. Logical volume vg_prod/lv_mysql successfully resized.
4

Resize the ext4 filesystem online

The LV is now 500 GB, but the ext4 filesystem still thinks it is 200 GB. resize2fs expands the filesystem to fill the LV while it is mounted. MySQL continues reading and writing data throughout this operation.

# resize2fs /dev/vg_prod/lv_mysql resize2fs 1.46.5 Filesystem at /dev/vg_prod/lv_mysql is mounted at /var/lib/mysql; on-line resizing required old_desc_blocks = 25, new_desc_blocks = 63 The filesystem on /dev/vg_prod/lv_mysql is now 131072000 (4k) blocks long.
5

Verify the new filesystem size

df -h confirms the filesystem is now 500 GB with approximately 63% utilization — comfortable headroom. MySQL never went offline.

# df -h /var/lib/mysql Filesystem Size Used Avail Use% Mounted on /dev/mapper/vg_prod-lv_mysql 493G 170G 299G 37% /var/lib/mysql
✅ Result
Four commands — pvcreate, vgextend, lvextend, resize2fs — expanded a production database filesystem from 200 GB to 500 GB with zero downtime. This is the power of LVM in production server administration.

Lab Objectives

In this lab you will gain hands-on experience with core storage management technologies used in enterprise Linux and Windows environments. By completing this lab you will be able to:

  • Create and manage disk partitions using both MBR (fdisk) and GPT (parted) partition schemes on Linux.
  • Implement software RAID 1 using mdadm, simulate a drive failure, and perform an array rebuild.
  • Configure Logical Volume Management (LVM) — including physical volumes, volume groups, logical volumes, and online volume extension.
  • Create a Storage Spaces pool and virtual disk on Windows Server using both the GUI and PowerShell.

Part A: Disk Partitioning 6 pts Linux

You will add two virtual disks to your Linux VM and practice both the legacy MBR partitioning workflow (fdisk) and the modern GPT workflow (parted). Understanding the differences between MBR and GPT is a foundational skill for any system administrator.

A.1 — Add Virtual Disks in Proxmox

  1. In the Proxmox web interface, select your Linux VM and navigate to Hardware.
  2. Click Add → Hard Disk. Set the size to 5 GiB, use the default storage pool, and click Add. This will become /dev/sdb (or the next available device).
  3. Repeat step 2 to add a second 5 GiB disk. This will become /dev/sdc.
  4. Start (or reboot) the VM. Log in via SSH or the Proxmox console.
  5. Verify both disks are visible: 
    lsblk
sudo fdisk -l | grep "Disk /dev/sd"
If the new disks do not appear after adding them in Proxmox, try a full VM reboot rather than just a rescan. You can also run echo "- - -" | sudo tee /sys/class/scsi_host/host*/scan to trigger a bus rescan without rebooting.

A.2 — MBR Partitioning with fdisk (Disk 1: /dev/sdb)

Use fdisk to create two primary partitions on /dev/sdb: one 2 GiB partition and one using the remaining space.

  1. Launch fdisk on the first new disk: 
    sudo fdisk /dev/sdb
  2. At the fdisk prompt, press m to view help, then press o to create a new MBR (DOS) partition table.
  3. Press n → p (primary) → partition number 1 → accept default first sector → type +2G for the last sector.
  4. Press n → p → partition number 2 → accept default first sector → accept default last sector (uses remaining space).
  5. Press p to print the partition table and verify.
  6. Press w to write and exit.
  7. Run lsblk /dev/sdb and sudo fdisk -l /dev/sdb to confirm the partitions.
After writing with w, the kernel may not immediately see the new partition table. Run sudo partprobe /dev/sdb to notify the kernel, then re-run lsblk.

A.3 — GPT Partitioning with parted (Disk 2: /dev/sdc)

Use parted to create a GPT partition table on /dev/sdc with two partitions: one 2 GiB and one using the remaining space.

  1. Launch parted on the second disk: 
    sudo parted /dev/sdc
  2. At the (parted) prompt, create a GPT label: 
    mklabel gpt
  3. Create the first partition (1 MiB to 2049 MiB, leaving 1 MiB alignment at the start): 
    mkpart primary ext4 1MiB 2049MiB
  4. Create the second partition using the remaining space: 
    mkpart primary ext4 2049MiB 100%
  5. Print the partition table: 
    print
  6. Exit parted: 
    quit
  7. Verify with lsblk /dev/sdc and sudo parted /dev/sdc print.
If parted warns about disk alignment, you can acknowledge with I (Ignore) during interactive use. For production systems, always align partitions to 1 MiB boundaries for optimal performance on modern drives.

Part B: Software RAID Implementation 8 pts Linux

You will create a software RAID 1 (mirroring) array using mdadm on two new virtual disks, format it, mount it, simulate a disk failure, and perform a rebuild. This mirrors a real-world scenario where a drive fails and must be replaced.

New Disks Required: Parts B and C each need dedicated disks. Add two more 5 GiB disks in Proxmox for Part B (these will become /dev/sdd and /dev/sde, or the next available devices). Do NOT reuse the disks from Part A.

B.1 — Add Disks and Install mdadm

  1. In Proxmox, add two additional 5 GiB virtual disks to your Linux VM (as described in Part A.1). Note the device names shown by lsblk (e.g., /dev/sdd, /dev/sde).
  2. Install mdadm if not already present: 
    # Ubuntu/Debian:
sudo apt update && sudo apt install -y mdadm

# Rocky Linux / RHEL:
sudo dnf install -y mdadm
  3. Verify installation: 
    mdadm --version

B.2 — Create the RAID 1 Array

  1. Create the RAID 1 array (replace sdd and sde with your actual device names): 
    sudo mdadm --create /dev/md0 \
  --level=1 \
  --raid-devices=2 \
  /dev/sdd /dev/sde
    Type y and press Enter when prompted to confirm.
  2. Monitor the sync progress: 
    watch -n 2 cat /proc/mdstat
    Wait for the sync to complete (the progress bar will reach 100%). Press Ctrl+C to exit watch.
  3. Check the array details: 
    sudo mdadm --detail /dev/md0
On small 5 GiB virtual disks, the initial sync usually completes in under a minute. On real hardware with large disks, this can take hours.

B.3 — Format, Mount, and Test

  1. Create an ext4 filesystem on the RAID array: 
    sudo mkfs.ext4 /dev/md0
  2. Create a mount point and mount the array: 
    sudo mkdir -p /mnt/raid1
sudo mount /dev/md0 /mnt/raid1
  3. Write a test file to confirm the filesystem is working: 
    echo "RAID 1 test file - CSC152 Lab 10" | sudo tee /mnt/raid1/testfile.txt
cat /mnt/raid1/testfile.txt
  4. Check available space: 
    df -h /mnt/raid1

B.4 — Simulate a Drive Failure and Rebuild

  1. Mark one disk as faulty (simulating a hardware failure): 
    sudo mdadm --manage /dev/md0 --fail /dev/sde
  2. Check the array status — it should now show a degraded state: 
    sudo mdadm --detail /dev/md0
cat /proc/mdstat
  3. Remove the failed disk from the array: 
    sudo mdadm --manage /dev/md0 --remove /dev/sde
  4. Simulate replacing the drive by adding the same device back (in a real scenario this would be a new physical disk): 
    sudo mdadm --manage /dev/md0 --add /dev/sde
  5. Monitor the rebuild: 
    watch -n 2 cat /proc/mdstat
    Wait for the rebuild to complete, then press Ctrl+C.
  6. Verify the array is healthy again: 
    sudo mdadm --detail /dev/md0
During the degraded state, your data is still accessible — RAID 1 continues to operate with one disk. However, you have NO redundancy during this window. Always rebuild as quickly as possible in production.

B.5 — Save Configuration and Add to /etc/fstab

  1. Save the RAID configuration to mdadm.conf so the array is reassembled automatically at boot: 
    # Ubuntu/Debian:
sudo mdadm --detail --scan | sudo tee -a /etc/mdadm/mdadm.conf
sudo update-initramfs -u

# Rocky Linux / RHEL:
sudo mdadm --detail --scan | sudo tee -a /etc/mdadm.conf
  2. Get the UUID of the RAID filesystem: 
    sudo blkid /dev/md0
  3. Add a persistent mount entry to /etc/fstab. Open the file with your preferred editor and add a line similar to (use your actual UUID): 
    UUID=<your-uuid-here>  /mnt/raid1  ext4  defaults,nofail  0  2
    The nofail option prevents the system from halting at boot if the RAID array is unavailable.
  4. Test the fstab entry without rebooting: 
    sudo umount /mnt/raid1
sudo mount -a
df -h /mnt/raid1
Always use nofail in /etc/fstab for RAID and LVM volumes in a lab environment. A typo or missing device can prevent the system from booting, requiring rescue mode to fix.

Part C: LVM Configuration 8 pts Linux

Logical Volume Management (LVM) provides a flexible abstraction layer over physical storage, allowing you to resize volumes, add disks, and manage storage pools without unmounting filesystems. You will create a volume group from two new disks, carve out three logical volumes, format and mount them, and then extend one volume online.

New Disks Required: Add two more 5 GiB disks in Proxmox for Part C (e.g., /dev/sdf and /dev/sdg). Do NOT reuse disks from Parts A or B.

C.1 — Install LVM Tools and Prepare Disks

  1. Install LVM2 if not already present: 
    # Ubuntu/Debian:
sudo apt install -y lvm2

# Rocky Linux / RHEL:
sudo dnf install -y lvm2
  2. Verify the new disks are visible: 
    lsblk | grep -E "sdf|sdg"

C.2 — Create Physical Volumes

  1. Initialize both disks as LVM physical volumes: 
    sudo pvcreate /dev/sdf /dev/sdg
  2. Verify the physical volumes: 
    sudo pvs
sudo pvdisplay

C.3 — Create a Volume Group

  1. Create a volume group named vg_lab10 from both physical volumes: 
    sudo vgcreate vg_lab10 /dev/sdf /dev/sdg
  2. Verify the volume group: 
    sudo vgs
sudo vgdisplay vg_lab10

C.4 — Create Three Logical Volumes

  1. Create the first logical volume (2 GiB, for web data): 
    sudo lvcreate -L 2G -n lv_web vg_lab10
  2. Create the second logical volume (2 GiB, for database data): 
    sudo lvcreate -L 2G -n lv_db vg_lab10
  3. Create the third logical volume (1 GiB, for logs): 
    sudo lvcreate -L 1G -n lv_logs vg_lab10
  4. Verify all three logical volumes: 
    sudo lvs
sudo lvdisplay vg_lab10

C.5 — Format, Mount, and Add to /etc/fstab

  1. Create ext4 filesystems on all three logical volumes: 
    sudo mkfs.ext4 /dev/vg_lab10/lv_web
sudo mkfs.ext4 /dev/vg_lab10/lv_db
sudo mkfs.ext4 /dev/vg_lab10/lv_logs
  2. Create mount points: 
    sudo mkdir -p /mnt/web /mnt/db /mnt/logs
  3. Mount all three volumes: 
    sudo mount /dev/vg_lab10/lv_web  /mnt/web
sudo mount /dev/vg_lab10/lv_db   /mnt/db
sudo mount /dev/vg_lab10/lv_logs /mnt/logs
  4. Verify all mounts: 
    df -h /mnt/web /mnt/db /mnt/logs
  5. Add persistent entries to /etc/fstab using the device mapper paths (or UUIDs from blkid): 
    /dev/vg_lab10/lv_web   /mnt/web   ext4  defaults,nofail  0  2
/dev/vg_lab10/lv_db    /mnt/db    ext4  defaults,nofail  0  2
/dev/vg_lab10/lv_logs  /mnt/logs  ext4  defaults,nofail  0  2
  6. Test the fstab entries: 
    sudo umount /mnt/web /mnt/db /mnt/logs
sudo mount -a
df -h /mnt/web /mnt/db /mnt/logs

C.6 — Online Volume Extension

One of LVM's most powerful features is the ability to extend a logical volume and its filesystem while it is mounted and in use — no downtime required.

  1. Check the current free space in the volume group: 
    sudo vgs vg_lab10
  2. Extend lv_web by 1 GiB: 
    sudo lvextend -L +1G /dev/vg_lab10/lv_web
  3. Resize the ext4 filesystem to use the new space (the volume must be mounted): 
    sudo resize2fs /dev/vg_lab10/lv_web
  4. Verify the new size: 
    df -h /mnt/web
sudo lvs vg_lab10
resize2fs works on ext4 filesystems while mounted (online resize). If you are using XFS, use xfs_growfs /mnt/web instead. Do NOT run resize2fs on an unmounted XFS volume — they use different tools.
You can also use lvextend -r -L +1G /dev/vg_lab10/lv_web — the -r flag automatically runs resize2fs (or xfs_growfs) after extending, combining steps 2 and 3 into one command.

Part D: Windows Storage Spaces 8 pts Windows Server

Windows Storage Spaces is Microsoft's software-defined storage solution, providing pooled storage, resiliency (mirroring/parity), and thin provisioning. You will create a storage pool and virtual disk using both the Server Manager GUI and PowerShell on your Windows Server 2022 VM.

New Disks Required: Add at least three 5 GiB virtual disks to your Windows Server VM in Proxmox. Storage Spaces requires a minimum of two disks for a mirror; three disks are recommended so you can observe the pool capacity. Bring the VM online and verify the disks appear in Disk Management before proceeding.

D.1 — Add Virtual Disks and Initialize in Disk Management

  1. In Proxmox, add three 5 GiB virtual disks to your Windows Server VM (same process as the Linux VMs).
  2. Start the Windows Server VM and log in as Administrator.
  3. Open Disk Management (right-click Start → Disk Management). The new disks will appear as "Unknown / Not Initialized."
  4. Right-click each new disk and select Initialize Disk. Choose GPT as the partition style and click OK.
  5. The disks should now show as "Online" with unallocated space. Do NOT create volumes — Storage Spaces will manage the disks directly.
Storage Spaces requires disks to be unformatted and unpartitioned (raw). If you accidentally created a partition, you can delete it in Disk Management before adding the disk to a pool.

D.2 — Create a Storage Pool via Server Manager GUI

  1. Open Server Manager. In the left panel, click File and Storage Services → Storage Pools.
  2. In the Storage Pools panel, click Tasks → New Storage Pool… to launch the wizard.
  3. On the Before You Begin page, click Next.
  4. Name the pool Lab10Pool and click Next.
  5. Select all three new physical disks to include in the pool and click Next.
  6. Review the summary and click Create. Wait for the pool to be created, then click Close.

D.3 — Create a Virtual Disk via Server Manager GUI

  1. With Lab10Pool selected, click Tasks → New Virtual Disk… in the Virtual Disks panel.
  2. Select Lab10Pool as the storage pool and click OK.
  3. Name the virtual disk Lab10VDisk and click Next.
  4. Choose Mirror as the storage layout (requires at least 2 disks) and click Next.
  5. Select Thin provisioning to allow the virtual disk to grow as data is written, then click Next.
  6. Set the virtual disk size to 8 GiB and click Next.
  7. Review the summary and click Create. Leave the "Create a volume when this wizard closes" checkbox selected and click Close.
  8. The New Volume Wizard will launch automatically. Format the volume as NTFS, assign drive letter V:, and label it Lab10Volume. Complete the wizard.
If the New Volume Wizard does not launch automatically, open Disk Management, find the new virtual disk (it will appear as a new disk with unallocated space), right-click, and select New Simple Volume.

D.4 — PowerShell Verification and Commands

Now verify your Storage Spaces configuration using PowerShell cmdlets and capture the output for your lab report.

  1. Open Windows PowerShell as Administrator (right-click Start → Windows PowerShell (Admin)).
  2. List all storage pools: 
    Get-StoragePool | Format-List FriendlyName, OperationalStatus, HealthStatus, Size, AllocatedSize
  3. List all virtual disks: 
    Get-VirtualDisk | Format-List FriendlyName, ResiliencySettingName, OperationalStatus, HealthStatus, Size, FootprintOnPool
  4. List all volumes: 
    Get-Volume | Where-Object {$_.DriveLetter -eq "V"} | Format-List *
  5. List the physical disks in the pool: 
    Get-PhysicalDisk | Format-Table FriendlyName, CanPool, OperationalStatus, HealthStatus, Usage, Size -AutoSize
You can also run Get-StoragePool -FriendlyName "Lab10Pool" | Get-PhysicalDisk to list only the disks that belong to your specific pool.

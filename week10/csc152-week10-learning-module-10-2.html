Introduction — Why Partitioning Matters

▼

Before any operating system can store data on a physical or virtual disk, two preparatory steps must happen: partitioning and formatting. Partitioning divides the raw disk into logical regions; formatting lays down a file system structure inside each region so the OS can track files, directories, permissions, and metadata.

Get either step wrong and the consequences range from a disk the OS silently ignores to data that is inaccessible after a reboot. In production environments, poor partitioning decisions made at install time can be expensive to undo — which is why administrators plan partition layouts before touching a disk.

🔗 Connection to the Rest of Week 10
Correct partitioning is a prerequisite for the work ahead. Module 10.3 (RAID with mdadm) and Module 10.4 (LVM) both require properly partitioned disks — specifically, partitions flagged as Linux RAID or LVM type. Lab 10 Part A asks you to partition two 10 GB virtual disks using both fdisk and parted. This module gives you the knowledge to do that confidently.

MBR vs. GPT — Partition Table Types

▼

Every disk begins with a partition table — a data structure written at the start of the disk that records how the disk is divided. Two standards are in common use: the legacy Master Boot Record (MBR) and the modern GUID Partition Table (GPT).

Deep Dive: MBR

Master Boot Record was introduced with IBM PC DOS in 1983. The partition table is stored in the first 512-byte sector of the disk alongside the boot code. Its age shows in its limitations:

  • Maximum disk size: 2 TB (32-bit LBA addressing)
  • Maximum primary partitions: 4
  • Workaround: one primary partition can be designated extended, which can contain multiple logical partitions — but this adds complexity
  • No redundancy: if the single MBR sector is corrupted, the entire partition table is lost
  • Boot mode: used with legacy BIOS (not UEFI)

Deep Dive: GPT

GUID Partition Table is part of the UEFI specification and was designed to replace MBR. Every partition gets a globally unique identifier (GUID). Key improvements:

  • Maximum disk size: 9.4 ZB (64-bit LBA — effectively unlimited for current hardware)
  • Maximum partitions: 128 (on most implementations)
  • Redundancy: a backup copy of the partition table is stored at the end of the disk; corruption of the primary header is recoverable
  • CRC checksums protect partition table integrity
  • Boot mode: required for UEFI boot; compatible with modern Linux and Windows
⚠ Lab 10 Note
Lab 10 Part A asks you to create a GPT partition table on the second virtual disk using parted. The first disk uses fdisk, which defaults to MBR on disks under 2 TB. Be sure you understand which tool creates which partition table type before you start.

Side-by-Side Comparison

Attribute MBR GPT

💡 Rule of Thumb
For any new server disk deployment in 2026, default to GPT. Use MBR only when you have a specific legacy compatibility requirement or are working with hardware that predates UEFI.

Linux Partitioning Tools — fdisk, parted, and gdisk

▼

Block Device Naming Conventions

Before using any partitioning tool, you need to identify the correct device. Linux represents disks as block device files under /dev/. The naming scheme depends on the interface:

Device Pattern Interface Examples Partition Naming
/dev/sdX SATA, SAS, USB, SCSI /dev/sda, /dev/sdb /dev/sda1, /dev/sda2
/dev/nvmeNnM NVMe (PCIe) /dev/nvme0n1, /dev/nvme1n1 /dev/nvme0n1p1, /dev/nvme0n1p2
/dev/vdX VirtIO (KVM/QEMU virtual disks) /dev/vda, /dev/vdb /dev/vda1, /dev/vda2
/dev/xvdX Xen virtual disks (AWS older instances) /dev/xvda /dev/xvda1

To list all block devices on your system with their sizes and partition layout:

lsblk                     # Tree view of all block devices
lsblk -f                  # Include filesystem type and UUID
sudo fdisk -l             # Detailed disk and partition info

fdisk — The Classic Interactive Partitioner

fdisk is the traditional Linux disk partitioning utility. It works interactively through a menu-driven interface. Modern versions (util-linux 2.28+) support GPT, but fdisk is most commonly used for MBR disks under 2 TB.

Starting fdisk

sudo fdisk /dev/sdb       # Open /dev/sdb for editing

Key fdisk Commands (type at the Command (m for help): prompt)

KeyAction
mShow help menu
pPrint current partition table
nAdd a new partition
dDelete a partition
tChange partition type (e.g., fd = Linux RAID, 8e = Linux LVM)
wWrite changes and exit (commits to disk)
qQuit without saving

Typical fdisk Workflow — Create a Single Partition (Lab 10 Style)

# 1. Open the disk
sudo fdisk /dev/sdb

# At the fdisk prompt:
# n  → new partition
# p  → primary
# 1  → partition number 1
# [Enter] → accept default first sector
# [Enter] → accept default last sector (uses entire disk)
# t  → change type
# fd → Linux RAID auto (required for mdadm in Module 10.3)
# p  → verify the table looks correct
# w  → write and exit

# Verify the new partition
lsblk /dev/sdb
📌 Important
fdisk changes are held in memory until you type w to write them. If you make a mistake, type q to quit without saving and start over. Nothing is committed to disk until w.

parted — The GPT-Capable Powerhouse

parted (GNU Parted) supports both MBR and GPT partition tables and can handle disks larger than 2 TB. Unlike fdisk, parted writes changes immediately — there is no "quit without saving." It can be used interactively or with non-interactive (scripted) commands.

Non-Interactive Usage (Scripting-Friendly)

# Create a GPT partition table on /dev/sdc
sudo parted /dev/sdc --script mklabel gpt

# Create a single partition using the entire disk
sudo parted /dev/sdc --script mkpart primary 0% 100%

# Set the LVM flag on partition 1 (for Module 10.4)
sudo parted /dev/sdc --script set 1 lvm on

# Verify
sudo parted /dev/sdc print

Interactive Usage

sudo parted /dev/sdc       # Enter interactive mode

# At the (parted) prompt:
# print        → show current layout
# mklabel gpt  → create GPT table (DESTRUCTIVE — erases existing table)
# mkpart primary 0% 100%  → single partition, full disk
# set 1 lvm on → set LVM flag
# print        → verify
# quit         → exit (changes already written)

Common parted Partition Flags

FlagPurpose
bootMarks partition as bootable (MBR) or EFI system partition (GPT)
lvmMarks partition for use by LVM (Module 10.4)
raidMarks partition for use by software RAID (Module 10.3)
swapMarks partition as swap space
⚠ Caution — Immediate Writes
Unlike fdisk, parted writes every change immediately. There is no undo within a session. Always double-check your device name before running mklabel — it will erase the existing partition table instantly.

gdisk — GPT-Focused fdisk Replacement

gdisk (GPT fdisk) is designed specifically for GPT disks. Its interface is nearly identical to fdisk, making it easy to learn if you already know fdisk. It is the preferred interactive tool when working exclusively with GPT partition tables.

Starting gdisk

sudo gdisk /dev/sdb       # Open /dev/sdb for GPT editing

Key gdisk Commands

KeyAction
?Show help menu
pPrint current partition table
nAdd a new partition
dDelete a partition
tChange partition type (uses GUID codes, e.g., 8300 = Linux filesystem, 8e00 = Linux LVM, fd00 = Linux RAID)
iShow partition information (including GUID)
wWrite changes and exit
qQuit without saving

gdisk vs. fdisk — When to Use Each

ScenarioRecommended Tool
New disk, GPT required, interactive workflowgdisk
New disk, GPT required, scripted/automatedparted
Legacy MBR disk, interactive workflowfdisk
Any disk type, scripted automationparted
Disk > 2 TBgdisk or parted (GPT only)

After Partitioning: Inform the Kernel

After modifying a partition table on a disk that is already in use (mounted partitions), the kernel may not immediately see the changes. Use one of these commands to refresh the kernel's partition table view:

sudo partprobe /dev/sdb   # Re-read partition table (preferred)
sudo udevadm settle        # Wait for udev to process device events

File Systems and Formatting — ext4, xfs, and btrfs

▼

A partition is just raw space — a contiguous block of sectors with no structure. Formatting writes a file system onto that space: a data structure that tracks where files begin and end, stores metadata (permissions, timestamps, ownership), manages free space, and handles crash recovery. Choosing the right file system for a workload is an important administrative decision.

ext4 — The Linux Workhorse

ext4 (Fourth Extended File System) is the default file system for most Linux distributions, including Ubuntu Server. It is mature, well-tested, and supported everywhere.

  • Max file size: 16 TB
  • Max volume size: 1 EB (exabyte)
  • Journaling: Yes — tracks pending changes to recover from crashes
  • Extents: Contiguous block allocation reduces fragmentation
  • Delayed allocation: Improves write performance by batching small writes
  • Best for: General-purpose server workloads, OS partitions, home directories, databases
# Format /dev/sdb1 as ext4
sudo mkfs.ext4 /dev/sdb1

# Format with a label (useful for identification)
sudo mkfs.ext4 -L "datastore" /dev/sdb1

# View ext4 filesystem information after formatting
sudo tune2fs -l /dev/sdb1 | grep -E "Filesystem|Block|UUID"

xfs — High-Performance Parallel I/O

xfs was developed by SGI for high-performance workloads and is the default file system for Red Hat Enterprise Linux and Rocky Linux. It excels at large files and parallel I/O.

  • Max file size: 8 EB
  • Max volume size: 8 EB
  • Journaling: Yes (metadata journal)
  • Allocation groups: Divides the file system into parallel regions, enabling concurrent I/O
  • Online resizing: Can grow while mounted (cannot shrink)
  • Best for: Large files, high-throughput workloads, media storage, NFS exports, Rocky Linux deployments
  • Caution: Cannot be shrunk — plan capacity carefully
# Format /dev/sdb1 as xfs
sudo mkfs.xfs /dev/sdb1

# Format with a label
sudo mkfs.xfs -L "mediastore" /dev/sdb1

# View xfs filesystem information
sudo xfs_info /dev/sdb1

btrfs — A Brief Mention

btrfs (B-Tree File System) is a modern copy-on-write file system with built-in snapshot, RAID, and subvolume capabilities. It is the default on openSUSE and is used in some Fedora configurations. While powerful, it is more complex to administer and less universally supported than ext4 or xfs. It is not covered in depth this week but is worth knowing exists.

Choosing the Right File System

Workload / Scenario Recommended FS Reason
Ubuntu Server OS partition ext4 Distribution default; excellent compatibility and tool support
Rocky Linux OS partition xfs RHEL/Rocky default; excellent for enterprise workloads
General-purpose data partition ext4 Mature, reliable, resizable in both directions
Large media or database files xfs Better performance with large files and parallel I/O
NFS export volume xfs Handles concurrent NFS client I/O efficiently
LVM logical volumes (Module 10.4) ext4 or xfs Both work well; ext4 allows shrinking, xfs does not
💡 Key Difference to Remember
ext4 can be shrunk (with resize2fs after unmounting); xfs cannot be shrunk — only grown. If you anticipate needing to reduce a partition's file system size later (common in LVM scenarios), choose ext4.

Persistent Mounts — /etc/fstab, UUIDs, and the mount Command

▼

Formatting a partition creates a file system, but the OS still needs to be told where to attach it in the directory tree. This is called mounting. A mount made with the mount command alone is temporary — it disappears after a reboot. To make a mount survive reboots, you add an entry to /etc/fstab.

The mount Command

# Create a mount point directory first
sudo mkdir -p /mnt/data

# Mount /dev/sdb1 at /mnt/data
sudo mount /dev/sdb1 /mnt/data

# Verify the mount
df -h /mnt/data
mount | grep /mnt/data

# Unmount when done
sudo umount /mnt/data

Why UUIDs Instead of Device Names?

Device names like /dev/sdb1 are not stable. If you add a new disk or change the order disks are detected, /dev/sdb might become /dev/sdc — and your /etc/fstab entry would point to the wrong device or fail entirely. UUIDs (Universally Unique Identifiers) are assigned to the file system at format time and remain constant regardless of device enumeration order.

# Find the UUID of a partition (three equivalent methods)
sudo blkid /dev/sdb1
lsblk -f
ls -l /dev/disk/by-uuid/

Anatomy of /etc/fstab

Each non-comment line in /etc/fstab has exactly six fields separated by whitespace. Every field matters — a typo can prevent the system from booting.

# <device/UUID> <mount point> <type> <options> <dump> <pass>
UUID=a1b2c3d4-e5f6-7890-abcd-ef1234567890 /mnt/data ext4 defaults 0 2
Field 1: Device (UUID)
Field 2: Mount point
Field 3: FS type
Field 4: Options
Field 5: dump
Field 6: pass

Field-by-Field Reference

Field Name Description Common Values
1 Device What to mount. UUID= is strongly preferred over /dev/sdX. UUID=…, LABEL=name, /dev/sdb1
2 Mount point Directory where the file system will be attached. Must exist. /mnt/data, /var/log, none (swap)
3 Type File system type. ext4, xfs, swap, nfs, auto
4 Options Comma-separated mount options. defaults, ro, noexec, nofail, _netdev
5 dump Whether the dump backup utility should back up this FS. Almost always 0. 0 (disabled), 1 (enabled)
6 pass Order for fsck file system check at boot. Root FS = 1; others = 2; skip = 0. 0, 1 (root only), 2

Common Mount Options Explained

  • defaults — equivalent to rw,suid,dev,exec,auto,nouser,async
  • nofail — boot continues even if this device is absent (important for non-critical data disks)
  • ro — mount read-only
  • noexec — prevent execution of binaries on this partition (security hardening)
  • _netdev — wait for network before mounting (required for NFS, iSCSI)

Testing an fstab Entry Without Rebooting

# Mount all entries in /etc/fstab that are not yet mounted
sudo mount -a

# If no errors appear, your fstab entry is syntactically correct
# Verify the mount appeared
df -h
⚠ A Bad fstab Can Prevent Booting
Always test with sudo mount -a before rebooting. A syntax error or wrong UUID in /etc/fstab can drop the system into emergency mode at boot. If that happens, you will need to boot from rescue media or use single-user mode to fix it. The nofail option on non-root partitions is a safety net — the system will boot even if that device is missing.

Windows Partitioning — Disk Management & PowerShell

▼

Windows Server provides two approaches to disk partitioning and formatting: the Disk Management GUI (familiar to most Windows users) and PowerShell (essential for scripted, automated, or remote disk management). Both accomplish the same tasks; PowerShell is preferred in production environments for repeatability and documentation.

Disk Management GUI

Open Disk Management: Server Manager → Tools → Computer Management → Disk Management, or run diskmgmt.msc.

Initializing a New Disk

  1. New disks appear as Unknown / Not Initialized at the bottom of the Disk Management window.
  2. Right-click the disk label → Initialize Disk.
  3. Choose GPT (recommended) or MBR → OK.

Creating a New Volume

  1. Right-click the unallocated space → New Simple Volume.
  2. Follow the wizard: set size, assign a drive letter, choose file system (NTFS or ReFS), set allocation unit size, enter a volume label.
  3. Click Finish — Windows formats the volume immediately.

PowerShell — Disk Management Cmdlets

The Storage module provides PowerShell cmdlets for complete disk management. These commands are idempotent and scriptable — ideal for deploying consistent disk configurations across multiple servers.

# List all disks and their status
Get-Disk

# Initialize a new disk (Disk 1) with GPT
Initialize-Disk -Number 1 -PartitionStyle GPT

# Create a new partition using all available space
# AssignDriveLetter auto-assigns the next available letter
New-Partition -DiskNumber 1 -UseMaximumSize -AssignDriveLetter

# Format the partition as NTFS with a label
# (replace E: with the letter that was assigned)
Format-Volume -DriveLetter E -FileSystem NTFS -NewFileSystemLabel "DataDisk" -Confirm:$false

# Verify the result
Get-Volume -DriveLetter E

Windows File Systems: NTFS and ReFS

Attribute NTFS ReFS
Full name New Technology File System Resilient File System
Introduced Windows NT 3.1 (1993) Windows Server 2012
Max volume size 256 TB (practical) 35 PB
Data integrity Journaling Copy-on-write + checksums (stronger)
Boot volume Yes No
Best for OS drives, general-purpose volumes, application data Large data volumes, Storage Spaces, Hyper-V VHD storage
📌 Default to NTFS
For most Windows Server scenarios — including Lab 10 — use NTFS. ReFS is appropriate for large-scale data storage with Storage Spaces (covered in Module 10.5) but is not suitable as a boot volume and has limited tool support compared to NTFS.

Mini-Case — "The Unmounted Data Drive"

▼

A junior administrator has added a new 500 GB data disk to a Rocky Linux server, partitioned it with parted (GPT, single xfs partition), and formatted it. She then edited /etc/fstab to make the mount persistent, but after a reboot the disk is not mounted at /mnt/appdata.

She runs sudo mount -a and gets:

mount: /mnt/appdata: can't read superblock.

The Faulty /etc/fstab Entry

Here is the relevant section of /etc/fstab. The UUID shown is correct (verified with blkid). Find the error(s).

# <device> <mount> <type> <options> <dump> <pass>
UUID=9f3a1c2d-84b7-4e5f-a012-bc3456789def /mnt/appdata ext4 defaults 0 2

The disk was formatted with xfs, but the fstab entry specifies ext4. Additionally, the mount point directory /mnt/appdata does not yet exist on the system.

Your Tasks

  1. Identify all errors in the fstab entry above.
  2. Write the corrected fstab entry.
  3. List the commands needed to fully resolve the problem (create the directory, fix fstab, test the mount).
✓ Solution

Error 1 — Wrong file system type: The partition was formatted as xfs but fstab specifies ext4. Linux uses the type field to select the correct kernel driver. When the ext4 driver tries to read an xfs superblock, it fails — hence "can't read superblock."

Error 2 — Missing mount point directory: /mnt/appdata must exist as a directory before the mount can succeed. If it does not exist, mount will fail with a "mount point does not exist" error.

Corrected fstab entry:

# <device> <mount> <type> <options> <dump> <pass>
UUID=9f3a1c2d-84b7-4e5f-a012-bc3456789def /mnt/appdata xfs defaults 0 2

Commands to fully resolve the problem:

# Step 1: Create the missing mount point
sudo mkdir -p /mnt/appdata

# Step 2: Edit /etc/fstab — change ext4 to xfs
sudo nano /etc/fstab

# Step 3: Test the corrected fstab without rebooting
sudo mount -a

# Step 4: Verify the mount succeeded
df -h /mnt/appdata

Lesson: Always match the file system type in fstab to the actual file system on the partition. Use lsblk -f or blkid to confirm the type before writing the fstab entry. And always test with mount -a before rebooting.

§0 – Objectives & Context

Why this module matters: In professional IT environments, problems rarely confine themselves to a single service or layer. A user who "can't log in" might be experiencing a network issue, a DNS failure, an AD replication problem, or a simple account lockout. This module trains you to think systematically across layers—the skill that separates experienced administrators from beginners.

Module Objectives

  • Apply structured troubleshooting methodologies (bottom-up, top-down, divide-and-conquer) to multi-layer problems
  • Identify service dependencies and trace failures across system layers
  • Select appropriate diagnostic commands and log sources for each layer of the infrastructure stack
  • Correlate symptoms to root causes using logical elimination

Connection to Active Work

The troubleshooting skills in this module directly support your Project 3 work (due Monday, May 18, 2026). If you encounter issues with your multi-server infrastructure, the diagnostic pathways here will help you resolve them efficiently. These scenario types also appear on the Final Exam (Week 17).

§1 – Troubleshooting Methodology Review

The OSI/TCP-IP Model as a Diagnostic Framework

The OSI and TCP/IP models aren't just theoretical constructs—they provide a structured way to isolate where a problem exists. When a service fails, the issue must exist at some layer. By systematically testing each layer, you eliminate possibilities until the root cause is revealed.

OSI Layer Focus Area Key Diagnostic Tools
1 – Physical Cables, NICs, link status ip link, Get-NetAdapter, link lights
2 – Data Link MAC addressing, VLANs, switching arp, Get-NetNeighbor, switch port status
3 – Network IP addressing, routing, firewalls ping, traceroute/tracert, ip route
4 – Transport Ports, TCP/UDP connectivity Test-NetConnection, ss/netstat, telnet
5–7 – Session/Presentation/Application Service status, authentication, app logic systemctl, Get-Service, dcdiag, logs

Three Troubleshooting Approaches

Bottom-Up Top-Down Divide-and-Conquer ───────── ──────── ────────────────── Layer 7 ▼ Layer 6 │ Layer 5 │ Layer 4 │ ──▶ Start Here (L3/L4) Layer 3 │ │ Layer 2 │ │ Layer 1 ▲ ▼ Go up or down based │ on results Start Here Start Here
Bottom-Up Approach

Start at Layer 1 (Physical) and work upward.

  • When to use: When you have no information about where the problem might be, or when you suspect a physical/network issue (new installation, cable changes, hardware replacement).
  • Process: Verify physical connectivity → check link status → verify IP configuration → test network reachability → verify service status → test application functionality.
  • Advantage: Thorough; won't miss lower-layer issues.
  • Disadvantage: Can be slow if the problem is at a higher layer.
Top-Down Approach

Start at Layer 7 (Application) and work downward.

  • When to use: When users report a specific application or service failure and the underlying network was recently working. Good for service configuration issues.
  • Process: Check application/service status → verify service configuration → test transport connectivity → verify network routing → check physical layer.
  • Advantage: Fast if the problem is a service misconfiguration.
  • Disadvantage: May waste time testing services when the real issue is a disconnected cable.
Divide-and-Conquer Approach

Start in the middle (typically Layer 3/4) and go up or down based on results.

  • When to use: When you have experience with the environment and can make an educated guess about the problem area. Most efficient for experienced administrators.
  • Process: Test network connectivity (ping/traceroute) → if that works, go up to services; if it fails, go down to physical/data link.
  • Advantage: Fastest on average; eliminates half the stack immediately.
  • Disadvantage: Requires experience to choose the right starting point.
Pro Tip: In practice, experienced admins often use divide-and-conquer as their default. A quick ping to the target server immediately tells you whether the problem is below Layer 3 (ping fails) or above it (ping succeeds but the service doesn't work).

§2 – Scenario 1: "Users Can't Log In"

🔒 Scenario 1: Active Directory Authentication Failure
Reported Symptom: Multiple users in the Accounting department report they cannot log in to their domain-joined Windows 10 workstations. They receive the error: "The trust relationship between this workstation and the primary domain failed" or "There are currently no logon servers available to service the logon request." Users in other departments are unaffected.

Recommended Approach: Divide-and-conquer — start with network connectivity to the domain controller, then branch up or down.

Diagnostic Pathway

Step 1: Verify Network Connectivity (Layer 3)

Goal: Determine if the affected workstations can reach the domain controller at the network level.

# From an affected workstation:
ping 192.168.1.10    # DC IP address
ping dc01.corp.local # DC hostname

If ping to IP succeeds but hostname fails: DNS issue → go to Step 2.

If ping to IP fails: Network/physical issue → check cables, switch ports, VLAN assignments for the Accounting subnet.

If both succeed: Network is fine → go to Step 3 (service layer).

Step 2: Verify DNS Resolution (Layer 7 – DNS Service)

Goal: Confirm workstations can resolve the domain controller and AD-related DNS records.

# Test basic DNS resolution:
nslookup dc01.corp.local
nslookup corp.local

# Test AD-specific SRV records:
nslookup -type=SRV _ldap._tcp.dc._msdcs.corp.local
nslookup -type=SRV _kerberos._tcp.corp.local

If DNS fails: Check the workstation's DNS settings — are they pointing to the DC? Check if the DNS Server service is running on the DC.

# On the DC (PowerShell):
Get-Service DNS
Resolve-DnsName corp.local

Common cause: Workstations received a DHCP lease with incorrect DNS server (e.g., pointing to the router instead of the DC).

Step 3: Verify AD Domain Services Status (Layer 7 – AD Service)

Goal: Confirm that Active Directory Domain Services is running and healthy.

# On the Domain Controller (PowerShell):
Get-Service NTDS
Get-Service Netlogon
Get-Service KDC

# Run AD diagnostics:
dcdiag /v

# Check replication (if multiple DCs):
repadmin /replsummary

If services are stopped: Restart them and check Event Viewer for why they stopped.

If dcdiag shows errors: Focus on the specific test that failed (DNS, replication, SYSVOL).

Step 4: Check Account-Level Issues (Application Layer)

Goal: Determine if specific user accounts are locked out or disabled.

# PowerShell – Check account status:
Get-ADUser -Identity jsmith -Properties LockedOut,Enabled,LastLogonDate
Search-ADAccount -LockedOut | Select-Object Name,SamAccountName

# Check if it's all Accounting users:
Get-ADUser -Filter * -SearchBase "OU=Accounting,DC=corp,DC=local" `
  -Properties LockedOut | Where-Object {$_.LockedOut -eq $true}

If accounts are locked: Investigate why — check security logs for failed login attempts (possible brute force or a service using old credentials).

# Splunk search for failed logins:
index=wineventlog EventCode=4740 OR EventCode=4625
| stats count by Account_Name, src_ip
| sort -count
Step 5: Resolution and Documentation

Common root causes for this scenario:

  • DNS misconfiguration: Workstations pointed to wrong DNS server after a DHCP scope change.
  • DC service failure: The Netlogon service crashed after a patch was applied.
  • Mass account lockout: A shared service account with an expired password caused cascading lockouts.
  • Network segmentation: A new firewall rule blocked Kerberos (TCP/UDP 88) or LDAP (TCP 389) traffic from the Accounting VLAN.

After resolution: Document the root cause, the fix applied, and any preventive measures (e.g., monitoring alert for Netlogon service status).

§3 – Scenario 2: "New PCs Not Getting Addresses"

🌐 Scenario 2: DHCP Failure
Reported Symptom: Five new workstations deployed in the Engineering lab are receiving 169.254.x.x (APIPA) addresses instead of valid IPs from the DHCP server. Existing machines in the same room still have valid addresses and are working normally.

Recommended Approach: Bottom-up — new hardware deployment suggests starting with physical/network layer verification.

Diagnostic Pathway

Step 1: Verify Physical/Virtual Network Connectivity (Layer 1–2)

Goal: Confirm the new PCs are physically connected and on the correct network segment.

# Check link status (Windows):
Get-NetAdapter | Select-Object Name, Status, LinkSpeed

# Check link status (if VMs):
# Verify virtual switch assignment in Hyper-V or VMware
Get-VMNetworkAdapter -VMName "ENG-PC01" | Select-Object SwitchName, Status

Key questions:

  • Are the link lights active on the NIC and switch port?
  • Are the new PCs connected to the same switch/VLAN as existing working machines?
  • If VMs: Are they attached to the correct virtual switch?

If link is down: Check cable, switch port, or virtual switch configuration.

If link is up: Proceed to Step 2.

Step 2: Verify DHCP Service Status (Layer 7 – Service)

Goal: Confirm the DHCP server is running and operational.

# On the DHCP server (Windows):
Get-Service DHCPServer

# On the DHCP server (Linux):
systemctl status isc-dhcp-server
# or
systemctl status dhcpd

# Check DHCP server event logs (Windows):
Get-WinEvent -LogName "Microsoft-Windows-Dhcp-Server/Operational" -MaxEvents 20

If service is stopped: Start it and check logs for why it stopped.

If service is running: Proceed to Step 3.

Step 3: Check DHCP Scope and Available Addresses

Goal: Determine if the DHCP scope has available addresses to lease.

# Windows DHCP Server (PowerShell):
Get-DhcpServerv4Scope | Select-Object ScopeId, Name, State, StartRange, EndRange
Get-DhcpServerv4ScopeStatistics | Select-Object ScopeId, Free, InUse, PercentageInUse

# Linux (check lease file):
cat /var/lib/dhcp/dhcpd.leases | grep -c "binding state active"

If scope is exhausted (0 free): Expand the scope range, shorten lease duration, or reclaim unused leases.

If scope has addresses available: The problem is that DHCP Discover packets aren't reaching the server → proceed to Step 4.

Step 4: Check Firewall Rules and DHCP Traffic

Goal: Verify that DHCP broadcast traffic (UDP 67/68) is not being blocked.

# Check Windows Firewall on the DHCP server:
Get-NetFirewallRule | Where-Object {$_.DisplayName -like "*DHCP*"}

# Check pfSense/Linux firewall logs for blocked UDP 67/68:
# In Splunk:
index=firewall dest_port=67 OR dest_port=68 action=blocked
| table _time, src_ip, dest_ip, action

# On Linux firewall:
iptables -L -n -v | grep -E "67|68"

If traffic is being blocked: Add firewall rules to permit DHCP traffic.

If traffic is not blocked: Proceed to Step 5.

Step 5: Check DHCP Relay Agent (If Cross-Subnet)

Goal: If the new PCs are on a different subnet from the DHCP server, verify the relay agent is forwarding requests.

# On the router/relay agent:
# Check if ip helper-address is configured (Cisco):
show running-config interface vlan 20
# Look for: ip helper-address 192.168.1.5

# On pfSense: Services → DHCP Relay
# Verify the relay interface and destination server IP

# Test by capturing traffic on the DHCP server:
# Windows:
netsh trace start capture=yes tracefile=dhcp.etl
# Linux:
tcpdump -i eth0 port 67 or port 68 -n

Key insight: DHCP uses broadcast. If the new PCs are on a VLAN that doesn't have a relay agent pointing to the DHCP server, broadcasts will never cross the router.

Common cause for this scenario: New PCs were placed on a new VLAN (VLAN 20) but the router's DHCP relay (ip helper-address) was only configured for the original VLAN (VLAN 10). Existing machines work because they're on VLAN 10.

Step 6: Resolution and Documentation

Common root causes for this scenario:

  • VLAN mismatch: New PCs on a different VLAN without a corresponding DHCP scope or relay agent.
  • Scope exhaustion: All available IPs are leased (check for orphaned leases from decommissioned machines).
  • MAC filtering: DHCP server configured to only serve known MAC addresses (reservation-only mode).
  • Virtual switch misconfiguration: VMs attached to the wrong virtual network (internal vs. external switch).

After resolution: Verify all new PCs receive addresses, document the VLAN/scope configuration, and consider adding a Splunk alert for DHCP scope utilization above 90%.

§4 – Scenario 3: "Splunk Not Receiving Logs"

📊 Scenario 3: Forwarder Communication Failure
Reported Symptom: You notice that your Splunk dashboard for the Linux web server (web01) has stopped updating. The last event received was 3 hours ago. The Splunk Enterprise (indexer) instance on splunk01 is accessible via the web UI and other sources are still sending data normally.

Recommended Approach: Top-down — the Splunk indexer is working (other sources are fine), so the problem is likely at the forwarder or the path between forwarder and indexer.

Diagnostic Pathway

Step 1: Verify Universal Forwarder Service Status

Goal: Confirm the Splunk Universal Forwarder is running on web01.

# SSH to web01 and check the forwarder service:
sudo systemctl status SplunkForwarder

# Or check the Splunk process directly:
ps aux | grep splunkd

# Check forwarder internal logs:
sudo tail -50 /opt/splunkforwarder/var/log/splunk/splunkd.log

If service is stopped: Start it with sudo systemctl start SplunkForwarder and check logs for why it stopped (disk full, crash, manual stop).

If service is running: Proceed to Step 2.

Step 2: Verify Network Connectivity to Indexer (Layer 3–4)

Goal: Confirm the forwarder can reach the indexer on the receiving port.

# Test basic connectivity:
ping splunk01.corp.local

# Test the specific receiving port (9997):
# Using netcat:
nc -zv splunk01.corp.local 9997

# Or using bash:
echo > /dev/tcp/splunk01.corp.local/9997 && echo "Port open" || echo "Port closed"

# Check DNS resolution:
nslookup splunk01.corp.local

If ping fails: Network issue between web01 and splunk01 → check routing, interface status.

If ping succeeds but port 9997 is closed: Firewall blocking or receiving port not enabled → proceed to Step 3.

If port 9997 is open: Network path is fine → proceed to Step 4 (configuration).

Step 3: Check Firewall Rules for Port 9997

Goal: Verify that no firewall (host-based or network) is blocking TCP 9997.

# On web01 (outbound):
sudo iptables -L OUTPUT -n -v | grep 9997

# On splunk01 (inbound):
sudo iptables -L INPUT -n -v | grep 9997

# On pfSense/network firewall – check via Splunk:
index=firewall dest_port=9997
| table _time, src_ip, dest_ip, action
| sort -_time

# On splunk01 – verify the receiving port is actually listening:
sudo ss -tlnp | grep 9997

If firewall is blocking: Add a rule to allow TCP 9997 from web01 to splunk01.

If port 9997 is not listening on splunk01: The receiving port is not enabled in Splunk → enable it via Splunk Web (Settings → Forwarding and Receiving → Receive Data → Add New on port 9997).

Step 4: Verify inputs.conf on the Forwarder

Goal: Confirm the forwarder is configured to monitor the correct log files.

# Check inputs.conf:
sudo cat /opt/splunkforwarder/etc/system/local/inputs.conf

# Expected content (example):
# [monitor:///var/log/syslog]
# disabled = false
# index = linux
# sourcetype = syslog

# [monitor:///var/log/apache2/access.log]
# disabled = false
# index = web
# sourcetype = access_combined

Common issues:

  • A monitored file path was changed (e.g., log rotation moved the file).
  • disabled = true was set accidentally.
  • The Splunk forwarder user doesn't have read permission on the log file.
# Check permissions:
ls -la /var/log/syslog
ls -la /var/log/apache2/access.log

# Verify the splunk user can read:
sudo -u splunk cat /var/log/syslog | head -5
Step 5: Verify outputs.conf Targeting

Goal: Confirm the forwarder knows where to send data.

# Check outputs.conf:
sudo cat /opt/splunkforwarder/etc/system/local/outputs.conf

# Expected content:
# [tcpout]
# defaultGroup = default-autolb-group
#
# [tcpout:default-autolb-group]
# server = splunk01.corp.local:9997
#
# [tcpout-server://splunk01.corp.local:9997]

Common issues:

  • Server hostname is misspelled or uses an old IP that changed.
  • Wrong port number specified.
  • SSL/TLS mismatch between forwarder and indexer (if encryption is configured).

After fixing any configuration: Restart the forwarder:

sudo /opt/splunkforwarder/bin/splunk restart
Step 6: Verify Data Flow and Resolution

Goal: Confirm data is now flowing after the fix.

# On the Splunk indexer web UI, search:
index=linux host=web01 earliest=-15m
| stats count by sourcetype

# Check the forwarder's internal metrics:
index=_internal host=web01 source=*metrics.log group=tcpout_connections
| table _time, destHost, destPort, status

Common root causes for this scenario:

  • Forwarder service crashed: Disk filled up on web01, causing splunkd to stop.
  • Firewall rule change: A security hardening script added iptables rules that blocked outbound 9997.
  • DNS change: splunk01's IP changed but DNS wasn't updated, and outputs.conf used the hostname.
  • Permission change: A system update changed log file permissions, preventing the forwarder from reading them.

§5 – Scenario 4: "Group Policy Not Applying"

📋 Scenario 4: GPO Failure
Reported Symptom: You created a new Group Policy Object (GPO) called "Accounting-Drive-Map" that should map a network drive (H:) for all users in the Accounting OU. Users report that the drive does not appear after logging in and running gpupdate /force. Other GPOs linked to the same OU (like the desktop wallpaper policy) are working fine.

Recommended Approach: Top-down — GPOs are application-layer, and since other policies in the same OU work, the infrastructure (network, DNS, AD) is likely fine. Focus on the policy configuration itself.

Diagnostic Pathway

Step 1: Analyze gpresult Output

Goal: Determine whether the GPO is being applied, denied, or not seen at all.

# Run on the affected user's workstation:
gpresult /r

# For detailed HTML report:
gpresult /h C:\Temp\gpreport.html

# PowerShell equivalent:
Get-GPResultantSetOfPolicy -ReportType Html -Path C:\Temp\gpreport.html

What to look for:

  • Is "Accounting-Drive-Map" listed under Applied GPOs? → Policy is applying but the setting may be misconfigured.
  • Is it listed under Denied GPOs? → Security filtering or WMI filter is blocking it → Step 3 or 4.
  • Is it not listed at all? → Not linked to the correct OU, or not reaching the client → Step 2.
Step 2: Verify OU Targeting and GPO Link

Goal: Confirm the GPO is linked to the correct OU and the user/computer is in that OU.

# Verify where the user object resides:
Get-ADUser -Identity jsmith -Properties DistinguishedName | Select-Object DistinguishedName

# Expected: CN=John Smith,OU=Accounting,DC=corp,DC=local

# Verify GPO links on the Accounting OU:
Get-GPInheritance -Target "OU=Accounting,DC=corp,DC=local"

# Check if the GPO is linked and enabled:
(Get-GPO -Name "Accounting-Drive-Map").GpoStatus

Common issues:

  • GPO is linked to the wrong OU (e.g., linked to "Accounting Computers" but it's a User Configuration policy, so it needs to be where the user objects are).
  • GPO link is disabled.
  • User object was moved to a different OU and no one updated the GPO link.
Step 3: Check Security Filtering

Goal: Verify the user/group has permission to apply the GPO.

# Check security filtering on the GPO:
(Get-GPO -Name "Accounting-Drive-Map").GetSecurityInfo()

# Or in PowerShell:
Get-GPPermission -Name "Accounting-Drive-Map" -All |
  Select-Object Trustee, Permission, Inherited

Key requirement: The user (or a group they belong to) must have both "Read" and "Apply Group Policy" permissions.

Common mistake: When you change security filtering from "Authenticated Users" to a specific group, you must also ensure "Authenticated Users" (or "Domain Computers") retains Read permission. Without Read permission, the computer can't even download the GPO to evaluate it.

# Fix: Add "Domain Computers" with Read permission:
Set-GPPermission -Name "Accounting-Drive-Map" -TargetName "Domain Computers" `
  -TargetType Group -PermissionLevel GpoRead
Step 4: Check WMI Filters

Goal: Determine if a WMI filter is preventing the GPO from applying.

# Check if a WMI filter is linked to the GPO:
(Get-GPO -Name "Accounting-Drive-Map").WmiFilter

# If a WMI filter exists, test it manually on the workstation:
Get-WmiObject -Query "SELECT * FROM Win32_OperatingSystem WHERE Version LIKE '10%'"

If the WMI query returns no results on the target machine: The filter is blocking the GPO. Either fix the WMI query or remove the filter if it's not needed.

Step 5: Check AD Replication and DNS SRV Records

Goal: If the GPO was recently created, verify it has replicated to the DC that the workstation is authenticating against.

# Check replication status:
repadmin /replsummary
repadmin /showrepl

# Verify which DC the workstation is using:
nltest /dsgetdc:corp.local

# Check DNS SRV records for domain controllers:
nslookup -type=SRV _ldap._tcp.dc._msdcs.corp.local

# Force replication (if needed):
repadmin /syncall /AdeP

If replication is failing: The GPO exists on one DC but not the one the client is using. Fix replication first.

Also check SYSVOL replication: GPO files are stored in SYSVOL. If DFSR/FRS replication is broken, the policy files won't be available on all DCs.

# Check SYSVOL share accessibility from the workstation:
dir \\corp.local\SYSVOL\corp.local\Policies\
Step 6: Resolution and Documentation

Most likely root cause for this specific scenario: Security filtering was changed from "Authenticated Users" to "Accounting-Users" group, but "Domain Computers" Read permission was removed. The computer cannot download the GPO, so it never evaluates the user settings.

Fix applied:

# Restore Read permission for computers:
Set-GPPermission -Name "Accounting-Drive-Map" -TargetName "Domain Computers" `
  -TargetType Group -PermissionLevel GpoRead

# Force policy update on affected workstation:
gpupdate /force

# Verify:
gpresult /r | findstr "Accounting-Drive-Map"

Lesson learned: When modifying GPO security filtering, always ensure the computer accounts can still read the GPO, even if they shouldn't apply it. This is a well-known GPO troubleshooting gotcha.

§6 – Knowledge Check (Ungraded)

Instructions: For each mini-scenario below, select the most likely root cause and consider what your first diagnostic step would be. Click "Check Answer" to see the explanation. This is ungraded practice for the Final Exam.

1. A user reports they can browse the internet but cannot access any shared folders on the file server (fs01.corp.local). Pinging fs01 by IP address (192.168.1.20) succeeds, but ping fs01.corp.local fails. What is the most likely issue?

✓ Correct! Since ping-by-IP works but ping-by-hostname fails, this is clearly a DNS resolution issue. The first diagnostic step would be nslookup fs01.corp.local to confirm, then check the workstation's DNS server settings. The user can browse the internet because external DNS (via the router/ISP) may still work, but internal Active Directory DNS records are not resolving.
✗ Not quite. The correct answer is B – DNS resolution is failing. The key clue is that ping-by-IP succeeds (so network connectivity is fine and the server is up) but ping-by-hostname fails. This isolates the problem to name resolution. Your first step: nslookup fs01.corp.local.

2. After applying security hardening changes to a Linux server, your Splunk dashboard shows the server stopped sending logs 30 minutes ago. The server is still reachable via SSH. What should you check first?

✓ Correct! The timing correlation is the key clue: logs stopped right after "security hardening changes." Hardening scripts commonly add restrictive firewall rules. Since SSH (port 22) still works but Splunk forwarding stopped, the most likely cause is new iptables rules blocking outbound TCP 9997. First step: sudo iptables -L OUTPUT -n -v | grep 9997. (Checking the forwarder service status would also be reasonable, but the timing correlation with firewall hardening makes iptables the best first check.)
✗ Not quite. The correct answer is B – Check iptables rules for port 9997. The critical clue is the timing: logs stopped immediately after security hardening. Hardening often includes adding restrictive firewall rules. Since the server is still reachable via SSH (port 22 is allowed), but Splunk data stopped flowing, the most likely cause is new outbound firewall rules blocking TCP 9997.

3. A newly created GPO that enforces a screen lock timeout is not applying to computers in the "Servers" OU. Running gpresult /r on an affected server shows the GPO under "Denied GPOs" with the reason "Inaccessible" or "Access Denied." Other GPOs are applying normally. What is the most likely cause?

✓ Correct! "Inaccessible" or "Access Denied" in gpresult means the computer cannot read the GPO at all. This happens when security filtering is modified and "Authenticated Users" or "Domain Computers" loses Read permission. The fix: add "Domain Computers" with GpoRead permission to the GPO. This is one of the most common GPO troubleshooting issues in production environments.
✗ Not quite. The correct answer is B – Security filtering removed Read permission. The "Inaccessible" status in gpresult specifically indicates the computer cannot read the GPO files. This occurs when security filtering is changed and the computer account's Read permission is inadvertently removed. Other GPOs work because they still have "Authenticated Users" with Read access.

4. A workstation that was working fine yesterday now cannot log in to the domain. The error says "no logon servers available." You can ping other workstations on the same subnet by IP, but cannot ping the domain controller by IP address. What troubleshooting approach and first step would you choose?

✓ Correct! Since you can ping other devices on the same subnet (Layer 1–2 is fine locally) but cannot ping the DC by IP (Layer 3 failure to the DC specifically), the issue is in the network path to the DC. Using tracert to the DC's IP will show where the path breaks—possibly a router issue, a VLAN problem, or the DC itself being down. This is divide-and-conquer: you've already established local connectivity works, now you're tracing the specific failing path.
✗ Not quite. The correct answer is D – Divide-and-conquer with tracert to the DC. Key clues: local subnet pings work (so the workstation's physical connection and local network are fine), but the DC is unreachable by IP. This means the problem is in the network path to the DC. tracert will reveal where packets are being dropped—a router, firewall, or the DC itself may be down.

§7 – Summary & What's Next

Key Troubleshooting Principles

  • Choose your approach based on available information: Use bottom-up when you have no clues, top-down when you suspect a service issue, and divide-and-conquer when you can make an educated guess.
  • Follow the dependencies: Services depend on other services. AD needs DNS. DHCP needs network connectivity. Splunk forwarders need network access to the indexer. Always verify the lower layers before blaming the higher ones.
  • Correlate timing: "When did it stop working?" and "What changed?" are the two most powerful troubleshooting questions. Changes (patches, firewall rules, configuration edits) are the #1 cause of outages.
  • Use logs: Splunk, Event Viewer, journalctl, and application logs contain the evidence. Learn to search them efficiently.
  • Document everything: Record what you tried, what you found, and what fixed it. Future-you (and your colleagues) will thank you.

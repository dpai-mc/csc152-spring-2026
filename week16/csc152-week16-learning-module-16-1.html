§1 – The Infrastructure Stack

Throughout this course, you've built expertise in individual technologies. Now it's time to see them as layers of a unified stack. Every production environment—from a small office to a global data center—relies on these layers working together.

Six-Layer Model

1 Virtual / Physical Weeks 1–2, 13 · Hardware, hypervisors, VMs
2 Operating System Weeks 3–5 · Linux & Windows Server, CLI
3 Network Services Weeks 8–9 · IP config, DNS, DHCP, firewalls
4 Directory / Identity Weeks 6–7, 11–12 · AD DS, LDAP, users, GPO
5 Security Weeks 14–15 · Hardening, encryption, backup
6 Monitoring Weeks 9, 12–16 · Splunk, logs, alerts, dashboards
Layer 1: Virtual / Physical — Details

The foundation of every infrastructure. Without functional hardware or a hypervisor, nothing above can exist.

  • Physical: Server form factors (tower, rack, blade), CPU, RAM, storage, NICs, power/cooling (Weeks 1–2)
  • Virtual: Type-1 hypervisors (Proxmox in our lab), VM creation, virtual networking and storage (Week 13)

Key insight: A failure at this layer (disk crash, hypervisor panic) cascades upward through every other layer.

Layer 2: Operating System — Details

The OS provides the platform on which all services run. Both Linux and Windows Server were covered.

  • Linux: Ubuntu Server / Rocky Linux installation, file system hierarchy, package management (Week 3)
  • Windows Server: Installation, Server Manager, roles and features, remote management (Week 4)
  • CLI: Bash and PowerShell fundamentals, scripting, SSH/PS Remoting (Week 5)

Key insight: OS misconfiguration (wrong hostname, broken networking stack) prevents higher-layer services from starting.

Layer 3: Network Services — Details

Network services allow machines to communicate and discover each other.

  • IP addressing: Static/dynamic, IPv4/IPv6, interface configuration (Week 8)
  • DNS: Name resolution, zones, forward/reverse lookups (Week 9)
  • DHCP: Automatic address assignment, scopes, reservations (Week 9)
  • Firewalls: iptables, Windows Firewall, pfSense, port/protocol rules (Week 9)

Key insight: DNS is the single most critical network service—Active Directory, Splunk forwarders, and almost every application depend on it.

Layer 4: Directory / Identity — Details

Centralized identity management enables single sign-on, policy enforcement, and organized resource access.

  • Users & Groups: Local account management, privilege models (Weeks 6–7)
  • Active Directory: Domains, OUs, Group Policy Objects, domain-joined machines (Week 11)
  • LDAP / OpenLDAP: Directory Information Tree, schema, authentication integration (Week 12)

Key insight: AD DS requires functioning DNS (SRV records) and network connectivity to replicate between domain controllers.

Layer 5: Security — Details

Security is a cross-cutting concern applied at every layer, but Week 14–15 focused on dedicated hardening and recovery.

  • Hardening: Minimizing attack surface, patch management, SSH/TLS configuration (Week 14)
  • Access Control: NTFS ACLs, Linux permissions, GPO security baselines (Weeks 6–7, 11, 14)
  • Backup & DR: Full/incremental/differential backups, RTO/RPO, recovery procedures (Week 15)

Key insight: Security without monitoring is "hope-based security." You need Layer 6 to validate that controls are working.

Layer 6: Monitoring — Details

Monitoring closes the loop: it tells you whether everything below is healthy and alerts you when it isn't.

  • Splunk Enterprise: Centralized log collection, indexing, SPL searches (Weeks 12–13)
  • Universal Forwarders: Agents on Linux/Windows sending logs to the indexer (Week 13)
  • Dashboards & Alerts: Visual monitoring, threshold-based alerting, security use cases (Weeks 14–16)
  • Syslog: pfSense and Linux rsyslog forwarding (Week 9)

Key insight: Splunk forwarders depend on network connectivity (Layer 3) and a running OS (Layer 2). If the network is down, logs stop flowing.

Understanding the Dependencies
  • DNS → Active Directory: AD DS uses DNS SRV records for domain controller location, replication partner discovery, and site-aware authentication. Without DNS, clients cannot find a DC to log in.
  • DHCP → DNS: DHCP can dynamically register A and PTR records in DNS (Dynamic DNS). Clients receiving addresses via DHCP rely on DNS to resolve domain resources. DHCP scopes often include DNS server addresses.
  • AD → Group Policy: GPOs are stored in SYSVOL and linked to OUs within AD. If AD replication fails or the DC is unreachable, GPOs cannot be downloaded or applied to clients.
  • Network → Splunk: Universal Forwarders transmit log data over TCP (typically port 9997). If network connectivity between a forwarder and the indexer is broken—whether by a cable fault, firewall rule, or routing issue—logs stop flowing and monitoring goes blind.

§3 – Cross-Platform Administration

Enterprise environments rarely run a single OS. Your course infrastructure includes both Linux and Windows servers working together. This section reviews how they coexist and how you administer across platforms.

Linux & Windows Coexistence

In a typical mixed environment:

  • Windows Server handles Active Directory, Group Policy, and often DNS/DHCP for domain-joined clients.
  • Linux servers run web services (Apache/Nginx), databases (MySQL/PostgreSQL), monitoring (Splunk), and network appliances (pfSense).
  • Both platforms need to share identity (users log in once) and share files (cross-platform file access).

The challenge: Windows uses NTLM/Kerberos authentication and SMB for file sharing, while Linux uses PAM, LDAP, and NFS. Bridging tools make them interoperate.

Samba & Winbind

Samba is an open-source implementation of the SMB/CIFS protocol that lets Linux servers:

  • Join an Active Directory domain as a member server
  • Serve file shares accessible to Windows clients
  • Act as a print server for Windows clients

Winbind (part of the Samba suite) bridges AD identity into Linux:

  • Maps AD users/groups to Linux UIDs/GIDs
  • Allows Linux to authenticate users against AD via PAM
  • Enables getent passwd to list AD users on Linux

Result: A user with an AD account can log into Linux servers and access Samba file shares—all with a single set of credentials.

Bash + PowerShell Across Platforms

As a systems administrator, you'll often need to automate tasks on both platforms:

  • Bash: Native on Linux; available on Windows via WSL or Git Bash. Best for Linux file operations, service management, text processing.
  • PowerShell: Native on Windows; available on Linux as PowerShell Core (pwsh). Best for AD management, Windows service control, .NET integration.

Common cross-platform pattern:

# Bash: Check if a service is running on Linux
systemctl is-active --quiet splunkd && echo "Splunk OK" || echo "Splunk DOWN"

# PowerShell: Check if a service is running on Windows
if ((Get-Service -Name SplunkForwarder).Status -eq 'Running') {
    Write-Output "Splunk OK"
} else { Write-Output "Splunk DOWN" }

Tip: For Project 3 and beyond, consider which tool is best for each task rather than forcing one shell to do everything.

§4 – Mini-Case: New Branch Office

Scenario

Mendocino Brewing Company is opening a new branch office in Ukiah. The IT team needs to deploy the following services for the branch:

  • A Windows Server VM on the branch hypervisor (Proxmox)
  • DNS (integrated with the existing domain)
  • DHCP for automatic client addressing
  • Active Directory (additional domain controller for the branch site)
  • Group Policy objects for branch-specific settings
  • Splunk Universal Forwarder sending logs to HQ's indexer
  • Firewall rules on pfSense allowing necessary traffic

Your task: Determine the correct deployment order. Think about which services depend on others being functional first.

Reveal: Correct Deployment Order & Reasoning
  1. Hypervisor / VM creation — Nothing runs without compute resources (Layer 1).
  2. Windows Server OS installation — The platform for all Windows-based services (Layer 2).
  3. Network configuration & firewall rules — Static IP, default gateway, pfSense rules allowing AD replication (ports 389, 636, 88, 445, 135, 49152–65535) and DNS (53). Without network connectivity, no service can communicate (Layer 3).
  4. DNS — Must be functional before promoting to DC. AD requires DNS SRV records. Configure the server to point to HQ's DNS first, then add the local DNS role (Layer 3).
  5. Active Directory (DC promotion) — Requires working DNS and network connectivity to replicate from HQ's existing DC (Layer 4).
  6. DHCP — Can now be configured with DNS server addresses pointing to the new local DC. Depends on DNS being available for dynamic updates (Layer 3, but configured after DNS/AD for proper integration).
  7. Group Policy — Created/linked after the DC is operational and OUs exist for the branch (Layer 4).
  8. Splunk Universal Forwarder — Installed last; requires network connectivity to HQ's indexer (port 9997 allowed through firewall) and a running OS to host the agent (Layer 6).
Key Principle: Always deploy from the bottom of the stack upward. Each layer depends on the layers below it being stable and functional.

§7 – Summary & What's Next

Key Takeaways

  • Infrastructure is a stack of interdependent layers—failures cascade upward.
  • DNS is the most critical network service; it underpins Active Directory, DHCP dynamic updates, and service discovery.
  • Always deploy services bottom-up: physical/virtual → OS → network → directory → security → monitoring.
  • Cross-platform environments require bridging tools (Samba/Winbind) and multi-shell fluency (Bash + PowerShell).
  • Monitoring (Splunk) validates that all lower layers are functioning correctly.

§1 – Defense in Depth: The Complete Picture

Defense in depth is a security strategy that employs multiple, independent layers of protection. If one layer fails, the next layer continues to protect the system. Throughout this course, you have implemented controls at every layer:

Layer 1: Physical Security — Data center access controls, locked server rooms. In our lab environment, this corresponds to the hypervisor host security and vSphere access restrictions discussed in Weeks 1–2.
Layer 2: Network Segmentation — VLANs, subnets, and isolated network zones. You configured pfSense interfaces and network segments in Weeks 3–4 to separate management, server, and client traffic.
Layer 3: Firewalls & Perimeter Controls — Stateful packet filtering and rule sets. pfSense firewall rules (Weeks 4–5) and Windows Firewall with Advanced Security (Week 10) control traffic flow between zones.
Layer 4: OS Hardening — Minimizing attack surface through service reduction, patching, and secure configuration. Linux hardening (Weeks 6–7) and Windows Server hardening via Group Policy (Weeks 9–10) reduce vulnerabilities on each host.
Layer 5: Authentication & Authorization — Strong passwords, account lockout policies, least-privilege access. Active Directory policies (Weeks 9–11), SSH key-based authentication (Week 7), and sudo configuration enforce identity controls.
Layer 6: Monitoring & Alerting — Continuous visibility into system activity. Splunk deployment (Weeks 13–15) provides log aggregation, correlation, and real-time alerting — the layer that validates all other controls are working.

Why Layers Matter

No single control is perfect. A firewall rule might be misconfigured; an OS patch might be delayed; a password might be compromised. Defense in depth ensures that an attacker must defeat multiple independent controls to achieve their objective. Monitoring (Layer 6) is uniquely important because it provides visibility — without it, you cannot know whether other layers have been breached.

§3 – Monitoring as a Security Control

Splunk as a SIEM

A Security Information and Event Management (SIEM) system collects, normalizes, correlates, and analyzes security-relevant log data from across an infrastructure. Splunk fulfills this role in our course environment by:

  • Collecting — Universal Forwarders on Linux and Windows hosts send syslog, Windows Event Logs, firewall logs, and application logs to the Splunk indexer.
  • Normalizing — Splunk's field extraction and Common Information Model (CIM) map disparate log formats into searchable, consistent fields.
  • Correlating — Correlation searches combine events from multiple sources (e.g., firewall denies + authentication failures from the same source IP) to identify attack patterns.
  • Alerting — Saved searches trigger alerts when thresholds are exceeded, enabling rapid incident response.

The Security Lifecycle

Monitoring completes the security lifecycle by providing the Detect and Respond phases:

  1. Prevent — Firewalls, hardening, authentication controls (Layers 1–5)
  2. Detect — Splunk identifies anomalies, policy violations, and attack indicators
  3. Respond — Alerts trigger investigation; automated actions (e.g., account lockout) contain threats
  4. Recover — Restore services; update controls based on lessons learned
  5. Review — Post-incident analysis; refine detection rules and hardening policies

Alerts and Correlation Searches

In Splunk, a saved search runs on a schedule and evaluates results against trigger conditions. When the condition is met (e.g., more than 5 failed logins in 5 minutes from a single IP), the alert fires and can:

  • Send an email notification to the security team
  • Log the alert to a summary index for trending
  • Trigger a webhook or script for automated response
  • Create a notable event in an incident review dashboard

A correlation search goes further by joining events across multiple data sources — for example, correlating a VPN login from an unusual geography with subsequent privilege escalation events on a domain controller.

§4 – Splunk Security Use Cases (SPL Examples)

Below are example SPL searches relevant to our course infrastructure. Expand each use case to see the query and explanation.

Failed Login Detection (Linux)

Detect multiple failed SSH login attempts from a single source IP within a 5-minute window:

index=linux_logs sourcetype=syslog "Failed password"
| rex field=_raw "from (?<src_ip>\d+\.\d+\.\d+\.\d+)"
| bin _time span=5m
| stats count as failed_attempts by src_ip, _time
| where failed_attempts > 5
| sort -failed_attempts

How it works: This search filters syslog events containing "Failed password," extracts the source IP using regex, bins events into 5-minute windows, counts failures per IP per window, and surfaces any IP exceeding 5 failures.

Failed Login Detection (Windows)

Detect failed logon events (Event ID 4625) on Windows domain controllers:

index=windows_logs sourcetype=WinEventLog:Security EventCode=4625
| bin _time span=5m
| stats count as failed_attempts by src_ip, Account_Name, _time
| where failed_attempts > 5
| table _time, src_ip, Account_Name, failed_attempts
| sort -failed_attempts

How it works: Windows logs failed logon attempts as Event ID 4625. This search aggregates them by source IP and target account, flagging brute-force patterns.

Service Stop/Start Events

Monitor critical service state changes on both platforms:

index=linux_logs sourcetype=syslog ("Started" OR "Stopped") 
  (sshd OR apache2 OR named OR splunkd)
| table _time, host, _raw
index=windows_logs sourcetype=WinEventLog:System 
  (EventCode=7036 OR EventCode=7045)
| table _time, host, ServiceName, Message
| sort -_time

Why it matters: Unexpected service stops may indicate an attacker disabling security controls (e.g., stopping the firewall service or audit daemon). New service installations (7045) may indicate persistence mechanisms.

Firewall Deny Logs

Aggregate blocked connection attempts from pfSense firewall logs:

index=firewall_logs sourcetype=pfsense action=block
| stats count as blocked_attempts by src_ip, dest_ip, dest_port
| where blocked_attempts > 20
| sort -blocked_attempts
| head 20

How it works: This identifies source IPs generating a high volume of blocked connections — potential port scans or brute-force attempts being stopped at the perimeter.

Account Creation/Modification Alerts

Detect new account creation or group membership changes in Active Directory:

index=windows_logs sourcetype=WinEventLog:Security 
  (EventCode=4720 OR EventCode=4722 OR EventCode=4732 OR EventCode=4728)
| table _time, host, Account_Name, EventCode, Message
| sort -_time

Event codes:

  • 4720 — User account created
  • 4722 — User account enabled
  • 4732 — Member added to a local group
  • 4728 — Member added to a global security group

Unauthorized account creation or privilege escalation (adding a user to Domain Admins) is a critical indicator of compromise.

Linux Account & Sudo Monitoring

Detect new user creation and sudo usage on Linux systems:

index=linux_logs sourcetype=syslog ("useradd" OR "usermod" OR "groupadd")
| table _time, host, _raw
| sort -_time
index=linux_logs sourcetype=syslog source="/var/log/auth.log" "sudo:"
  NOT "pam_unix(sudo:session)"
| rex field=_raw "sudo:\s+(?<sudo_user>\S+)"
| rex field=_raw "COMMAND=(?<command>.+)"
| table _time, host, sudo_user, command
| sort -_time

Why it matters: Monitoring sudo usage and account changes on Linux provides visibility equivalent to Windows audit policies for privilege escalation detection.

§5 – Mini-Case: Detecting a Brute Force Attack

Scenario

An external attacker targets the SSH service on your Linux web server. They use an automated tool to attempt thousands of username/password combinations. Let's trace how each defense-in-depth layer responds:

1
Firewall (Layer 3) — First Contact
The pfSense firewall permits inbound SSH (port 22) to the web server per the rule set. However, it logs every connection attempt. The volume of connections from a single external IP begins generating firewall log entries forwarded to Splunk.
2
OS Hardening (Layer 4) — Rate Limiting
The SSH daemon is configured with MaxAuthTries 3, disconnecting the attacker after 3 failed attempts per session. The attacker's tool reconnects repeatedly, but each session is limited. Additionally, fail2ban monitors /var/log/auth.log and bans the source IP after 5 failures within 10 minutes by adding a DROP rule to iptables.
3
Authentication (Layer 5) — Account Lockout
PAM's pam_faillock module locks the targeted account after the configured threshold (e.g., 5 failures). Even if the attacker guesses the correct password after lockout, authentication is denied until the lockout period expires or an administrator intervenes.
4
Monitoring (Layer 6) — Splunk Alert Fires
The saved search for failed SSH logins detects >5 failures from the attacker's IP within a 5-minute window. The alert fires, sending a notification to the security team and logging a notable event. The correlation search also links the firewall deny logs (from fail2ban's block) with the authentication failures, confirming the attack pattern.
5
Response — Incident Handling
The administrator reviews the Splunk alert, confirms the brute-force pattern, and takes additional action: adds a permanent firewall block for the attacker's IP range on pfSense, verifies no successful logins occurred, and documents the incident. The Splunk dashboard is updated to track the attacker's IP for future correlation.

Key Takeaway

No single layer stopped the attack alone. The firewall permitted SSH traffic (by design — it's a legitimate service). OS hardening slowed the attacker. Authentication controls prevented credential compromise. But it was monitoring that provided the visibility to detect, confirm, and respond to the threat. This is defense in depth in action.

§6 – Knowledge Check (Ungraded)

Test your understanding of the concepts covered in this module. Select your answer and check the feedback.

1. A system administrator configures pam_faillock to lock accounts after 5 failed login attempts. Which defense-in-depth layer does this control belong to?

Correct! Account lockout policies are an authentication/authorization control (Layer 5). They enforce identity security by preventing unlimited password guessing, regardless of how strong or weak the password is.
Not quite. While pam_faillock is configured at the OS level, its function is to enforce authentication policy — specifically account lockout. This places it in Layer 5 (Authentication & Authorization). OS Hardening (Layer 4) focuses on reducing attack surface through service minimization and secure configuration, not identity controls.

2. Which Splunk search would best detect unauthorized account creation in an Active Directory environment?

Correct! Event ID 4720 is logged when a new user account is created in Active Directory. Monitoring for this event allows you to detect unauthorized account creation, which is a common persistence technique used by attackers after initial compromise.
Not quite. The correct answer is EventCode=4720, which specifically logs user account creation events. EventCode=4625 detects failed logons (brute force), 7036 detects service state changes, and firewall blocks detect network-level denials — none of these would reveal a new account being created.

3. In the brute-force mini-case scenario, what is the primary role of the monitoring layer (Splunk)?

Correct! Monitoring's primary role is detection and visibility — it correlates firewall logs with authentication failures, confirms the attack pattern, and alerts the security team so they can respond. Blocking is done by the firewall (Layer 3), lockout by authentication controls (Layer 5), and password complexity by policy (Layer 5).
Not quite. While blocking (firewall/fail2ban), account lockout (PAM), and password policy are all important, they belong to other layers. Splunk's role as the monitoring layer is to provide visibility — detecting the attack pattern by correlating events from multiple sources and alerting the team to respond. Without monitoring, the other layers might stop the attack, but you would never know it happened.

4. Match the security control to its correct defense-in-depth layer. A Windows Group Policy that sets the minimum password length to 14 characters belongs to:

Correct! Password policies — even when delivered via Group Policy — are authentication and authorization controls (Layer 5). They strengthen the identity verification process by requiring credentials that are harder to guess or crack.
Not quite. Although Group Policy is an OS management tool (which might suggest Layer 4), the function of a password length policy is to strengthen authentication. We classify controls by their security function, not their delivery mechanism. Password policies belong to Layer 5 – Authentication & Authorization.

§7 – Summary & What's Next

Key Takeaways

  • Defense in depth uses multiple independent layers so that no single point of failure compromises the entire system.
  • Hardening on both Linux and Windows follows the same principles — minimize services, enforce strong authentication, configure host firewalls, and enable audit logging — but uses platform-specific tools.
  • Monitoring (Splunk as SIEM) is the layer that ties everything together: it provides the visibility to detect when preventive controls fail and enables timely response.
  • SPL searches for security use cases follow a pattern: filter relevant events → extract key fields → aggregate by meaningful dimensions → apply thresholds → surface actionable results.
  • A real attack traverses multiple layers — each layer slows, limits, or stops the attacker, and monitoring ensures the security team knows what happened.

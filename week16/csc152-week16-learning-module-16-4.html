§3 – Practice Scenarios

Work through each scenario below. Formulate your answer before revealing the model response. These mirror the style and complexity of Final Exam scenario questions.

Scenario 1 – OS Fundamentals & Virtualization

A small company is deploying a new Proxmox server to host 4 VMs: a file server, a DNS server, a web server, and a monitoring server. The physical server has 64 GB RAM and 8 CPU cores. The IT manager asks you to recommend a minimum resource allocation for each VM and explain why a Type 1 hypervisor was chosen over a Type 2.

Your task: Provide a resource allocation table and a brief justification for the hypervisor choice.

Resource Allocation (example):

VMRAMvCPUsRationale
File Server16 GB2Caches frequently accessed files in RAM
DNS Server4 GB1Lightweight; mostly responds to queries
Web Server8 GB2Handles concurrent connections
Monitoring (Splunk)16 GB2Indexing and search operations are memory-intensive
Reserved for host~20 GB1Proxmox host overhead and buffer

Hypervisor justification: A Type 1 (bare-metal) hypervisor like Proxmox runs directly on hardware without a host OS, providing better performance, lower overhead, and direct hardware access. Type 2 hypervisors run atop a host OS, adding latency and consuming resources for the host OS itself—unsuitable for production server workloads.

Scenario 2 – User/Group Administration & File Systems

You manage a Linux server with three departments: Engineering (5 users), Marketing (3 users), and Finance (4 users). A shared project directory /data/project-alpha must be accessible to all Engineering and Marketing users with read/write, but Finance should have no access. The directory is owned by root:engineering.

Your task: Describe how you would configure permissions (including ACLs if needed) to meet this requirement.

Solution using Linux ACLs:

# Set base permissions: owner=root (rwx), group=engineering (rwx), other=none
chmod 770 /data/project-alpha

# Add ACL for marketing group
setfacl -m g:marketing:rwx /data/project-alpha

# Set default ACL so new files inherit permissions
setfacl -d -m g:engineering:rwx /data/project-alpha
setfacl -d -m g:marketing:rwx /data/project-alpha
setfacl -d -m o::--- /data/project-alpha

# Verify
getfacl /data/project-alpha

Explanation: Standard Linux permissions only support one owning group. Since two groups need rwx access, we use POSIX ACLs via setfacl. The default ACL entries ensure that newly created files within the directory inherit the correct permissions. Finance has no entry, so they receive "other" permissions (none).

Scenario 3 – Network Services

Users on the 10.0.50.0/24 (Client) subnet report they cannot resolve internal hostnames. The DNS server (10.0.10.5) is on the Server subnet. The pfSense firewall separates the two subnets. You confirm that clients can ping 10.0.10.5 successfully.

Your task: List your troubleshooting steps in order and identify the most likely cause.

  1. Verify client DNS configuration: cat /etc/resolv.conf (Linux) or Get-DnsClientServerAddress (Windows) – Confirm clients point to 10.0.10.5.
  2. Test DNS query directly: nslookup hostname.domain.local 10.0.10.5 – If this fails but ping works, the issue is DNS-specific.
  3. Check firewall rules: Verify pfSense allows UDP/TCP port 53 from 10.0.50.0/24 to 10.0.10.5. This is the most likely cause – ping (ICMP) is allowed but DNS (port 53) is blocked.
  4. Verify DNS service status: On the DNS server, confirm the service is running and listening on port 53.
  5. Check DNS zone configuration: Ensure the forward lookup zone exists and contains the expected records.

Most likely cause: A missing or misconfigured firewall rule blocking port 53 (DNS) traffic from the Client subnet to the Server subnet. ICMP (ping) passes because it uses a different protocol/port that has a separate allow rule.

Scenario 4 – Directory Services & Centralized Management

You created a GPO named "Restrict USB Storage" and linked it to the "Workstations" OU. Users report that USB drives still work on their machines. You run gpresult /r on an affected workstation and see the GPO listed under "The following GPOs were not applied because they were filtered out" with reason "Denied (Security Filtering)".

Your task: Explain the cause and how to fix it.

Cause: Since the GPO targets computer settings (USB restriction is a computer configuration policy), the computer objects need "Read" and "Apply Group Policy" permissions on the GPO. By default, GPOs grant these to "Authenticated Users." If security filtering was changed to target a specific group (e.g., a user group instead of a computer group), the computer objects don't have permission to read and apply the GPO.

Fix:

  1. Open Group Policy Management Console.
  2. Select the "Restrict USB Storage" GPO → Delegation tab → Advanced.
  3. Ensure "Domain Computers" (or "Authenticated Users") has Read permission.
  4. In Security Filtering, add the computer group (or "Domain Computers") that should receive the policy.
  5. Run gpupdate /force on affected workstations and verify with gpresult /r.

Key concept: Computer Configuration policies require the computer object to have Read + Apply Group Policy permissions, not just the logged-in user.

Scenario 5 – Security, Monitoring & Log Analysis

Your Splunk dashboard shows a sudden spike of 500+ failed SSH login attempts against your Linux server (10.0.10.10) within 15 minutes, all from IP 10.0.50.200. The attempts target multiple usernames.

Your task: (a) Write an SPL search to identify this activity, (b) identify the attack type, and (c) describe two immediate mitigation steps.

(a) SPL Search:

index=linux sourcetype=syslog "Failed password"
| stats count by src_ip, dest_ip
| where count > 100
| sort -count

(b) Attack type: Brute force (or credential stuffing) attack – automated attempts using multiple usernames to guess valid credentials.

(c) Immediate mitigation steps:

  1. Block the source IP at the firewall: Add a pfSense rule to deny all traffic from 10.0.50.200, or on the Linux host: sudo ufw deny from 10.0.50.200
  2. Implement fail2ban: Configure fail2ban to automatically ban IPs after a threshold of failed attempts (e.g., 5 failures within 10 minutes = 30-minute ban).
  3. Additional (bonus): Verify no accounts were compromised by checking /var/log/auth.log for any successful logins from that IP; enforce SSH key-only authentication to eliminate password-based attacks entirely.

Scenario 6 – Cross-Domain Integration

After a power outage, your infrastructure comes back online but users cannot log in to domain-joined workstations. DNS is working, DHCP is assigning addresses, but dcdiag reports the domain controller's SYSVOL share is not accessible. Splunk shows the DC's event log has Event ID 13508 (FRS replication issues).

Your task: Explain why users cannot log in and outline the recovery steps.

Why users cannot log in: Domain authentication requires access to the SYSVOL and NETLOGON shares on the domain controller. SYSVOL stores Group Policy templates and logon scripts. If SYSVOL is unavailable (due to FRS/DFSR replication failure after unclean shutdown), the DC cannot process authentication requests properly, and workstations cannot retrieve logon policies.

Recovery steps:

  1. Verify SYSVOL share status: net share on the DC – check if SYSVOL and NETLOGON are listed.
  2. Check replication service: Get-Service DFSR (or NTFRS for older systems) – restart if stopped.
  3. Force replication: repadmin /syncall /AdeP
  4. If SYSVOL is corrupt: Perform an authoritative or non-authoritative DFSR restore depending on whether this is the only DC or one of many.
  5. Verify recovery: dcdiag /test:sysvolcheck and net share to confirm SYSVOL is accessible, then test user login.

You'll work with your personal Proxmox hypervisor to create three virtual machines with different hardware configurations, then use monitoring tools to assess their performance characteristics.

    üí° Pro Tip: Resource Planning Strategy

    With limited resources (6 vCPUs, 24 GB RAM, 200 GB storage), think about workload priorities. The database server typically needs the most resources for performance. The file server needs more storage but can work with fewer CPU cores. The web server needs balanced resources. A common approach: allocate 50% of resources to your highest-priority workload, then divide the remainder between the other two VMs.

    Part B: Server Resource Monitoring 10 points

    Monitor server resources using built-in tools and analyze performance metrics.

    Task Overview

    Install an operating system on at least one of your virtual machines (Ubuntu Server or Rocky Linux recommended) and use various monitoring tools to assess resource utilization. You'll capture baseline metrics and analyze performance characteristics.

    Linux Monitoring Commands

    Use these commands to gather system information and monitor resources. Run each command and document the output.

    CPU Information

    lscpu

    Displays CPU architecture, cores, threads, and virtualization capabilities

    Memory Usage

    free -h

    Shows total, used, and available memory in human-readable format

    Storage Devices

    lsblk

    Lists all block devices and their mount points

    Disk Usage

    df -h

    Reports filesystem disk space usage

    Real-Time Monitoring

    top

    Interactive process viewer showing CPU and memory usage (press 'q' to quit)

    I/O Statistics

    iostat -x 1 5

    Extended disk I/O stats, updated every 1 second for 5 iterations

    üìù Note: If iostat is not installed, you can install it using: sudo apt install sysstat (Ubuntu/Debian) or sudo dnf install sysstat (Rocky Linux/RHEL)

    Required Documentation for Part B

    1. System Specifications Report:
      • Output from lscpu showing CPU details
      • Output from free -h showing memory allocation
      • Output from lsblk and df -h showing storage
    2. Performance Monitoring Screenshots:
      • Screenshot of top command showing running processes
      • Screenshot of iostat output
    3. Baseline Measurements Table: Create a table documenting:
      • Total CPU cores and current utilization percentage
      • Total RAM, used RAM, and available RAM
      • Total storage capacity and used space
      • Average disk I/O operations per second (IOPS)
    4. Analysis Report: Write 2-3 paragraphs (200-300 words total) discussing:
      • What the monitoring data reveals about current resource usage
      • Whether any resources are under-utilized or approaching capacity
      • How you would identify a bottleneck using these tools
      • What additional monitoring might be helpful

    üí° Pro Tip: Understanding the Output

    When interpreting monitoring data, focus on patterns rather than single snapshots. High CPU usage isn't necessarily bad if the server is actively processing work. Similarly, Linux typically uses most available RAM for caching (this is good!). Look at the "available" memory rather than "free" memory to understand true availability.

    Common Mistakes to Avoid

    ‚ùå Missing Screenshots: Every configuration and monitoring activity requires visual proof. Screenshots must be clear, properly cropped, and include relevant portions of the screen (including VM names or timestamps when applicable).
    ‚ùå Calculator-Only Calculations: Using just the calculator results without showing manual work will result in point deductions. You must demonstrate understanding of the formulas and conversion factors.
    ‚ùå Unrealistic Hardware Choices: Allocating resources that exceed your Proxmox limits (6 vCPUs, 24 GB RAM, 200 GB storage total) or allocating inappropriately shows lack of planning. For example, giving one VM 20 GB of RAM leaves almost nothing for the other two VMs. Similarly, a simple web server doesn't need 4 vCPUs, and a database server shouldn't get only 512MB RAM.
    ‚ùå Generic Justifications: "This seems like enough" or "I picked this because it's in the middle" aren't adequate justifications. Explain your reasoning based on workload characteristics and requirements.
    ‚ùå Ignoring Monitoring Data: If your Part C upgrade recommendation doesn't reference specific findings from Part B, you're not connecting the dots. Use your monitoring data to justify upgrade decisions.
    ‚ùå Missing Units: Always include units in your calculations (W, VA, BTU/hr, GB, GHz, etc.). Numbers without units are meaningless in technical documentation.
